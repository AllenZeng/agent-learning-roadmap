# 用户说「你先试试」——模糊需求下任务完成的工程思考

> 模型没有真正的「完成感」，用户也说不清「怎么样算完成」。在这种双重模糊下，Agent 系统如何保障任务质量？这不仅是工程问题，也是产品设计问题。

---

## 从一次对话说起

```
用户：帮我把这段代码优化一下。
模型：好的。（开始大改，重构了整个模块，引入新抽象，改了接口签名...）
用户：等等，我只是想让你把变量名改得更可读一点！
```

这个场景每天都在发生。根因很简单：**用户脑子里的「优化」和模型理解的「优化」不是一回事。** 用户说不清他要什么（模糊需求），模型也不会追问（闷头执行），结果就是跑偏、返工、不信任。

这暴露了 Agent 系统中最难解决的一类问题：**任务完成的判定到底由谁来做、怎么做？** 模型没有感知，用户说不清标准——两头都模糊。

本文讨论在这种双重模糊下，工程和产品层面可以做什么。

---

## 一、先承认：这个问题目前没有完美的解

在展开具体方案之前，先诚实地说清楚边界。

**当前技术条件下，没有任何 Agent 系统能自动判断「任务是否真的完成了」。** 因为：

- 模型只会做统计预测，不会「知道自己不知道什么」
- 用户的真实意图是大脑里的隐性知识，不可能完全提取
- 「完成」是一个语义判断，不是二进制条件——即使用户自己，也经常在看到结果后才意识到「这不是我想要的」

所以下面的所有讨论，目标不是「完美解决」，而是**在双重模糊下尽可能提高对齐概率、降低跑偏成本**。

---

## 二、两层保障：信号层 + 交互层

把问题拆成两个维度：

```
          系统侧（信号层）              用户侧（交互层）
          ────────────────            ────────────────
做什么：   收集客观信号，兜底质量        让用户在过程中校准方向
怎么做：   测试、lint、schema、权限校验   计划确认、过程透明、证据呈现
局限：     只能验证形式，不能验证语义     确认有成本，用户不一定仔细看
```

**信号层解决「系统能自动验证什么」，交互层解决「用户怎么参与判定」。** 两层配合，覆盖各自的盲区。

---

## 三、信号层：系统默默做的事

信号层的设计原则：**用户不需要知道它们的存在，但它们始终在后台兜底。**

### 3.1 验证原语——原子化的信号收集能力

系统不预设「某个任务怎么验证」，而是提供一套与任务无关的验证原语。不同任务组合不同的原语：

```
文件系统原语：
  file_exists(path)
  file_changed(path)
  file_contains(path, pattern)

代码质量原语：
  test_run(target, requirement: all_passing)
  lint_check(target, requirement: zero_new_issues)
  type_check(target)

结构原语：
  schema_validates(output, schema)
  field_present(output, field_list)

过程原语：
  step_completed(step_index)
  no_infinite_loop(history, threshold)

引用原语：
  source_exists(citation)
  source_confirms(citation, claim)
```

这些是系统内置的、确定性的能力。它们不替代对任务的理解，只提供客观信号——测试过了就是过了，lint 红了就是红了。

### 3.2 对比基线：不定义「好」，只保证「不比之前差」

不追问用户什么叫好，而是自动取当前状态作为基线：

```
代码优化场景：
  基线 = 当前测试通过数 + lint 结果
  底线 = 你的修改不能破坏基线

Bug 修复场景：
  基线 = 当前失败的测试
  底线 = 那个测试变绿，其他测试不变红

文档创建场景：
  基线 = 无（首次创建）
  底线 = 所有声称的代码示例可运行
```

**好处**：不需要用户声明，系统自动拿到基线数据。**局限**：只适合有客观基线的任务。

### 3.3 假完成检测

模型最常见的几种「假完成」，系统需要识别：

| 假完成模式 | 模型表现 | 系统检测 |
|-----------|---------|---------|
| 提前收工 | 做了几步就汇报完成 | 比对声明步骤和实际执行步骤 |
| 回避困难 | 跳过复杂部分，只做简单的 | 检查每个子任务是否都有产出 |
| 幻觉完成 | 报告做了 A，实际没做 | 验证文件状态是否真的变更了 |
| 降级冒充 | 做不了原需求，偷偷换成能做的 | 比对输出和原始需求的覆盖度 |
| 兜圈子放弃 | 反复失败后静默跳过 | 检测重复操作 + 无产出循环 |

### 3.4 信号层的边界

信号层能保证的：形式正确、不引入已知破坏、操作在权限内。信号层不能保证的：任务是否符合用户的真实意图、输出是否真的有价值。

**信号层是底线，不是天花板。** 它防止最坏的情况（搞坏代码库、编造事实、越权操作），但不保证用户满意。

---

## 四、交互层：让用户在过程中校准方向

信号层解决不了的，需要交互层来补。核心原则：**不让用户当规范制定者，让用户当方向校准者。**

### 4.1 为什么「前置声明完成标准」在产品上失败

```
用户来找 Agent 帮忙，就是因为：
  - 需求是模糊的（「帮我优化一下这段代码」）
  - 标准是未知的（他不知道什么叫优化好了）
  - 期望是在看的过程中调整的（「我说的优化不是这个意思」）

然后你让他在一开始就填：
  ☐ 关联测试全部通过
  ☐ 无新增 lint 问题
  ☐ 变更摘要格式完整

→ 用户：「我要知道这些，还找你干嘛？」
```

这是根本性的交互悖论：**完成条件越精确，越能保证质量；但要求用户给精确条件，本身就违背了 Agent 要解决的「模糊需求处理」问题。** Claude Code 没有做前置声明，这个产品选择是对的。

### 4.2 计划确认：让用户做判断，而非做规范

更好的模式是模型把模糊需求翻译成具体计划，展示给用户，由用户判断方向对不对：

```
用户（脑子里的）：我想要一个干净点的代码结构
        ↓ 模型翻译
模型（显性输出）：我计划：
                1. 提取 user_service.py 中重复的数据库连接逻辑
                2. 统一错误处理模式
                3. 移除不再使用的 import
                确认方向？

用户：「嗯，就是我想要的，做吧」
```

关键区别：**模型承担了「把模糊变具体」的认知负担，用户只承担「判断对不对」的负担。** 认知科学里这叫 Recognition over Recall——识别比回忆容易得多。

### 4.3 计划确认的设计要点

**确认的是意图和方向，不是操作细节：**

```
不好的（太细）：
  「第 1 步：读 user_service.py，第 2 步：定位第 23 行，第 3 步：将 foo 改为 bar...
   确认？」→ 用户烦了，不看就点确认

好的（恰好的粒度）：
  「我计划做三件事：
   1. 提取重复的数据库连接代码
   2. 统一异常处理
   3. 移除无用 import
   确认方向？」→ 用户扫一眼就能判断
```

**分级确认——风险越高，确认越重：**

| 风险等级 | 确认方式 | 例子 |
|---------|---------|------|
| 低风险 | 不确认，事后通知 | 格式化代码、补充注释 |
| 中风险 | 确认计划，执行后通知 | 重构函数、更新依赖 |
| 高风险 | 确认计划 + 执行前逐步确认 | 数据库迁移、删除文件 |

**允许模糊确认——用户可以不精确：**

「方向差不多，你先做吧，做出来再看看」——这也是有效的确认。不应该强迫用户做更精确的判断。确认是梯度，不是二元的通过/驳回。

### 4.4 过程透明 + 随时纠偏

计划确认之后，执行过程也要透明：

```
「正在处理 user_service.py（第 1/3 步）...」
「提取了 3 处数据库连接代码，合并为 1 个工具函数」
「统一了 5 处异常处理，现在都走 AppError」
```

这给了用户第二个纠偏机会——如果看到中间产物不对，可以说「等等，这个方向开始偏了」。比跑完才发现省时间。

### 4.5 证据呈现，而非结论

任务完成后，不报告「已完成」，而是展示**可验证的证据**：

```
不好的：
  「任务已完成，所有测试通过。」（用户：你说的，我怎么信？）

好的：
  「修改总结：
   ✅ 修改了 2 个文件
   ✅ 测试 8→8 全部继续通过（无回归）
   ✅ 无新增 lint issue
   📝 变更摘要：提取重复连接代码，统一异常处理
   要再看看吗？」
```

**给证据不给结论，把最终判断留给用户。** 这是对「用户说不清标准」的承认——系统不替你判断，只给你判断所需的材料。

## 五、如何覆盖不同任务场景：一个统一模型

回到工程核心问题：系统不需要理解所有任务，只需要提供可组合的能力，让不同类型的任务匹配不同的验证和交互策略。

### 5.1 四层框架

```
第一层：理解对齐（交互层）
  模型拆解需求 → 显性展示理解 → 用户确认方向
  解决：「你是不是真的理解我了？」

第二层：过程透明（交互层）
  执行中展示进度和中间结果 → 用户随时纠正
  解决：「你正在做的是不是我想要的？」

第三层：信号兜底（信号层，后台运行）
  测试/类型检查/lint/权限校验 → 确定性信号自动修正
  解决：「不依赖用户判断的客观质量底线」

第四层：证据呈现（交互层）
  展示做了什么、改了什么、验证了什么
  解决：「我怎么知道你真的做了？」
```

用户不需要当规范制定者，也不需要当验收工程师。四层之中，用户只参与最轻松的事：**看一眼方向对不对（第一层）、发现偏了及时说（第二层）、最后看一眼证据确认（第四层）。** 认知负担被压缩到最小。

### 5.2 不同任务类型适配

| 任务类型 | 计划确认 | 信号兜底重点 | 证据呈现重点 |
|---------|---------|------------|------------|
| 代码变更 | 中风险：确认方向 | 测试、lint、类型检查 | diff、测试报告、变更摘要 |
| 信息查询 | 低风险：不确认 | 引用可检索、来源可验证 | 引用溯源、检索覆盖度 |
| 文档写作 | 低风险：确认大纲 | 代码示例可运行、链接有效 | 结构完整度、章节覆盖 |
| 系统操作 | 高风险：逐步确认 | 权限检查、影响范围分析 | 操作记录、回滚方案 |
| 创意任务 | 不确认：快速出第一版 | 无（没有客观基线） | 迭代记录、各版本对比 |

注意：创意类任务（写诗、起名、头脑风暴）的信号层几乎为空——因为没有客观质量基线。这类任务更依赖交互层的快速迭代：「先来一版看看」比「确认方向再做」更有效。

## 六、Claude Code 实际是怎么做的：一个务实参照

Claude Code 没有显式的完成保障系统，但它有几个关键设计在实际运行中覆盖了大部分问题：

**1. 工具执行结果作为隐式信号**

模型调用 `npm test`，测试输出本身就是信号——失败了就继续修，通过了就继续往下。模型不需要「知道」完成了，它只需要读输出文本然后决定下一步。

**2. 可中断 + 用户反馈作为兜底**

用户可以在任何时候打断、纠正。这从架构上承认了系统不可能预判所有任务怎么算完成。用户的「等等，不对」就是最直接的完成判定信号。

**3. 权限分级作为隐式门控**

高权限操作弹出确认框——这个确认本身就是一次「微验收」，用户检查后批准才执行。

**4. 缺少但可以补的**

Claude Code 没有显式的完成条件验证、没有假完成检测、没有独立的验收步骤。这些是当前 Agent 工具普遍欠缺的，也是未来可以演进的方向。

---

## 七、最诚实的结论

关于「模糊需求下如何保障任务完成」，目前不存在一劳永逸的方案。能做的是：

1. **不要逼用户做规范——让他们做判断。** 计划确认比需求问卷有效一百倍。
2. **系统默默兜底——用户不需要知道你在兜底。** 测试、lint、权限、收敛检测都在后台跑，不增加用户认知负担。
3. **给证据不给结论。** diff、测试报告、变更摘要——这些是让用户判断的材料，不是替用户判断的结论。
4. **承认很多时候判定权就在用户手里。** Claude Code 的选择是对的——把过程透明化，让用户随时介入，最终的完成判定由用户做。这不是设计的缺陷，是对问题本质的承认。

**也许这个领域最终要解决的问题不是「系统如何自动判定完成」，而是「如何让用户更轻松、更准确地做判断」。** 过程中的方向确认、中间结果的可视化、最终证据的结构化呈现——这些交互设计的重要性，不亚于底层的验证机制。
