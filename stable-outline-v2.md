# Agent 学习课程 · 稳定大纲 V2

## 七阶段结构 & 与 V1（六阶段）的映射关系

---

## 设计思路：为什么从六阶段调整为七阶段

V1的问题：
- 阶段一（基础认知）内容过重：既讲历史渊源、又讲范式演进、又讲产品拆解，认知负荷太大
- "历史演进"和"动手实践"混在一起，学习者不清楚什么时候该理解、什么时候该动手
- 缺少一个纯粹的"入门引导"阶段——先建立感性认识，再讲原理

V2的改进：
- 将 V1 阶段一拆分为两个阶段：**初识Agent**（感性认知）→ **Agent演进**（理性理解）
- 阶段更聚焦：每个阶段有单一、明确的目标
- 节奏更合理：先看现象 → 再学原理 → 再动手 → 再深入 → 再产品化 → 再实战 → 再追前沿

## 第一阶段：初识Agent —— 建立感性认知

**定位**：不需要写代码，不需要懂原理。先"看见"Agent，建立直觉。
**学时**：1 周
**前置**：使用过 ChatGPT 或类似产品即可

### 课程内容

1. **什么是 Agent？**
   - 一句话定义：能自主使用工具完成任务的 AI 系统
   - 与 Chatbot 的区别：Chatbot 只能"说"，Agent 能"做"
   - 与 Workflow 的区别：Workflow 是固定流水线，Agent 是自己决定怎么做
   - 最小定义公式：`Agent = LLM + Tool Use + 循环决策`

2. **为什么现在谈 Agent？**
   - GPT-3.5/ChatGPT 让大众体验到了 LLM 的能力
   - 但人们很快发现：光"说"不够，需要"做"
   - Agent 是 LLM 能力从"对话"到"行动"的自然延伸

3. **体验 3-5 款 Agent 产品**
   - Claude Code：终端中的编程助手
   - Cursor：IDE 中的 AI 编程搭档
   - Devin/OpenHands：自主软件工程师
   - GPT-Researcher：自主研究助手
   - 体验目标：带着问题去用——"它什么时候自己做决定？什么时候问我？什么时候做错了？"

4. **产品拆解模板**
   - 9 维度观察框架：用户输入 / 规划时机 / 工具调用 / 用户确认 / 失败恢复 / 类型判断 / 惊喜与失望 / 改进建议
   - 用模板分析至少 3 款产品

### 目标 & 验收

- 能不看资料说出 Agent 和 Chatbot/Workflow 的关键区别
- 能用产品拆解模板完成 3 份分析报告，每份覆盖全部 9 个维度
- 对"Agent 能做什么、做不好什么"有自己的直观理解

## 第二阶段：Agent 当前的演进 —— 理解范式从何而来

**定位**：不写代码，但深入理解 Agent 四大范式的来龙去脉。为阶段三的动手做理论准备。
**学时**：1-2 周
**前置**：完成阶段一，对 Agent 有感性认知

### 课程内容

1. **Agent 概念的历史渊源**
   - 强化学习中的 Agent（Bellman 1957 → DQN 2013 → AlphaGo 2016）
   - 软件 Agent（Wooldridge & Jennings 1994，BDI 模型）
   - Russell & Norvig 四种 Agent 类型：反射型 → 模型型 → 目标驱动 → 效用驱动
   - 传统 Agent 的根本痛点：封闭规则空间，无法泛化

2. **LLM 带来的范式革命**
   - GPT-3 的涌现能力（2020）——模型"长出了"推理的种子
   - ReAct（2022.10）——首次将推理和行动融合为循环
   - Toolformer（2023.2）——让模型自监督地学会用工具
   - 关键转折：从"LLM 作为聊天机器人"到"LLM 作为自主决策核心"

3. **核心范式一：Tool Use / Function Calling**
   - 之前：LLM 只能生成文本
   - 转折：ChatGPT Plugins（2023.3）→ OpenAI Function Calling（2023.6）→ JSON Schema 标准化
   - 现在：MCP 协议让工具定义跨框架通用
   - 思考：如何让 LLM 知道"什么时候该用工具、用哪个、怎么填参数"？

4. **核心范式二：Planning（规划）**
   - 之前：LLM 一次只能处理一个简单问题
   - 演进：CoT（2022）→ Tree-of-Thoughts（2023）→ Plan-and-Solve（2023）
   - 核心矛盾：规划能力和执行能力往往是此消彼长的
   - 未来：分层规划、动态重规划

5. **核心范式三：Reflection（反思）**
   - 之前：模型输出后无法自我修正
   - 演进：Self-Refine（2023）→ Reflexion（2023）
   - 关键洞察：Reflection 不是模型架构的能力，而是系统设计的能力
   - 四种实现方案：Multi-Pass Feedback / Self-Consistency / PRM / 内部探针

6. **核心范式四：Multi-Agent（多智能体协作）**
   - 之前：单个 Agent 的能力有上限
   - 演进：AutoGen（2023）→ CrewAI（2024）→ 动态 Agent 团队
   - 四种架构：层级式 / 对话式 / 辩论式 / 发布-订阅式

### 目标 & 验收

- 能用"之前有什么问题 → 提出者怎么想的 → 带来了什么变化"的框架讲清楚每个范式
- 能判断一篇 Agent 论文或产品公告属于哪个范式
- 对每个范式的适用场景和局限性有自己的判断

## 第三阶段：Agent 底层核心 —— 五大支柱

**定位**：开始动手写代码。理解并实现支撑 Agent 的五大核心技术。
**学时**：3-6 周
**前置**：完成阶段一和阶段二，熟悉 Python

### 课程内容

1. **第一支柱：LLM 原理——Agent 的大脑**
   - Transformer 架构（Self-Attention / Multi-Head Attention）
   - Tokenization（BPE 原理、对工具调用的影响）
   - Context Window（Lost in the Middle、注意力 O(n²)）
   - 采样策略（Temperature / Top-p / Top-k）

2. **第二支柱：Tool Use——让 Agent 能"做"**
   - JSON Schema 定义工具
   - 工具描述的黄金法则
   - 工具选择的核心问题（时机 / 选择 / 参数）

3. **第三支柱：RAG——给 Agent 接入知识**
   - Embedding 与向量检索
   - Chunking 策略
   - 混合检索 + Rerank

4. **第四支柱：Memory——让 Agent 不再"转瞬即忘"**
   - 短期记忆：上下文窗口管理（滑动窗口 / 摘要压缩 / 重要性过滤）
   - 长期记忆：跨会话持久化（最简 JSON 实现）
   - 三层上下文架构：热数据 / 温数据 / 冷数据

5. **第五支柱：Prompt Engineering——Agent 的"编程语言"**
   - System Prompt 设计
   - Few-shot Prompting
   - Chain-of-Thought
   - 结构化输出

6. **ReAct：五大支柱的联合运用**
   - ReAct 循环实现（Thought → Action → Observation）
   - 停止条件设计
   - 循环检测与防护

7. **桥梁项目：文件整理 Agent**
   - 先踩坑，再学架构
   - 带复盘模板，为阶段四做准备

### 目标 & 验收

- 实现一个 CLI Agent，支持 3+ 个工具，具备跨会话记忆
- 10 条测试任务成功 ≥ 7 条
- 能按 tool selection / prompt / schema / 上下文管理 / memory 分类失败原因
- RAG 检索 Top-5 召回率 ≥ 60%
- System Prompt 迭代有可验证的改进

## 第四阶段：Agent 架构深入 —— 从能用到好用

**定位**：你的 Agent 已经能跑了，但它会在某些场景下死循环、遗忘、越界、无法调试。本阶段系统解决这些问题。
**学时**：4-8 周
**前置**：完成阶段三，手上有一个可运行的 Agent

### 课程内容

1. **Harness（运行时引擎）**
   - 三层架构：驱动层 / 控制层 / 管理层
   - Harness vs Orchestration 的区别
   - 主流框架的 Harness 设计对比

2. **Orchestration（编排调度）**
   - 四种模式：Chain / Router / Plan-Execute / ReAct Loop / Graph
   - LangGraph 的 StateGraph 设计哲学
   - 模式选择决策树

3. **Memory 架构深入**
   - 向量数据库选型
   - MemGPT 虚拟内存思想
   - 记忆衰减与更新策略
   - 多 Agent 共享记忆

4. **Evaluation（评测）**
   - 端到端评测 vs 步骤级评测
   - LLM-as-Judge
   - 评测集构建方法论
   - 为什么要从阶段三就开始评测

5. **Guardrails（安全护栏）**
   - 输入护栏：Prompt Injection 防御
   - 输出护栏：工具参数校验、事实性校验
   - Human-in-the-Loop：五级审批策略

6. **Observability（可观测性）**
   - Tracing 架构
   - 关键指标：延迟分解、Token 消耗、错误分类
   - 调试与决策归因

### 目标 & 验收

- 将阶段三的"裸循环"重构为三层 Harness 架构
- 编排对比报告中有具体数据支撑
- 评测框架自动化运行，给出每个任务的 pass/fail 和失败原因
- Guardrails 至少拦截 1 次真实的越界行为
- Tracing 数据能在 5 分钟内定位一个已知 bug

## 第五阶段：Agent 走向生产环境 —— 从能用到能上线

**定位**：你的 Agent 功能正确了，但用户会觉得它慢、贵、不安全、不靠谱。本阶段解决"产品化"的问题。
**学时**：4-6 周
**前置**：完成阶段四，Agent 功能完整

### 课程内容

1. **交互设计**
   - 流式输出（SSE 原理）
   - 中断与恢复（Checkpoint 机制）
   - 渐进式信息披露
   - 用户确认节点设计

2. **可靠性工程**
   - 重试策略（指数退避 + 智能重试）
   - 降级机制（工具降级 / 模型降级 / 功能降级）
   - 超时处理（分层超时）
   - Rate Limit 管理

3. **成本控制**
   - 模型选择策略（路由器模式 / 级联策略）
   - Prompt Caching 原理与实践
   - 上下文窗口经济性

4. **产品评价体系**
   - 六大核心指标：任务完成率 / 平均步数 / 用户干预率 / Token 效率 / 满意度 / 留存率
   - Agent A/B 测试的特殊挑战

5. **安全合规**
   - Prompt Injection 防御体系
   - 数据脱敏与审计日志
   - GDPR / SOC2 合规要点

6. **失败模式清单**
   - 工具调用类：选择错误 / 参数错误 / 死循环
   - 上下文管理类：遗漏约束 / RAG 误导
   - 决策逻辑类：过早执行 / 权限不明

### 目标 & 验收

- 流式输出延迟感知 ≤ 500ms
- 常见故障可自动恢复
- 单任务平均 Token 消耗降低 ≥ 20%
- 失败模式清单覆盖 ≥ 5 种类型

## 第六阶段：Agent 项目实战 —— 从学到到做到

**定位**：回归实践。用一个完整的端到端项目，将前五个阶段的知识整合输出。
**学时**：2-4 周（根据项目深度调整）
**前置**：完成阶段一到阶段五

### 三个递进项目

| 项目 | 难度 | 合格版 | 进阶版 | 产品版 |
|------|------|--------|--------|--------|
| **个人知识助手** | 入门 | 检索+回答，2工具，正确率≥60% | 混合检索+Rerank，多轮记忆，流式输出 | Web界面，多用户，正确率≥90% |
| **代码审查 Agent** | 中级 | 读取diff→输出review | 调用测试/静态分析，置信度标注 | GitHub App自动触发，行内评论，误报反馈 |
| **领域 Agent 产品** | 高级 | 核心功能可用，5场景 | 20+场景，有评测报告 | 有真实用户反馈，监控告警，持续迭代 |

每个项目都附带设计文档模板、复盘模板和评测框架。

### 目标 & 验收

- 项目代码完整可运行（GitHub 仓库）
- 有设计文档（架构 + 技术选型理由）
- 有复盘报告（问题 + 解决方案 + 重做改进）
- 有评测报告（用阶段四学的评测框架）

## 第七阶段：Agent 当前最新的发展趋势 —— 保持敏锐

**定位**：了解正在发生什么，但主线精力不在此。每周 2-4 小时浏览即可。
**学时**：持续跟进
**前置**：完成阶段三以上的学习

> ⚠️ **重要定位**：本阶段是"跟进生态"，不是"主线必修"。建议在完成阶段三到阶段五之前，不要在这里投入过多时间。

### 课程内容

1. **MCP / A2A——Agent 的标准化协议**
   - MCP：Agent 与工具的通信协议（Anthropic 2024.11）
   - A2A：Agent 与 Agent 的通信协议（Google 2025.4）
   - 主线关联度：⭐⭐⭐⭐⭐（必修，至少接入体验）

2. **Computer Use——让 Agent 操作计算机**
   - GUI Agent（截屏分析 + 元素定位 + 操作执行）
   - 浏览器自动化（Playwright + LLM）
   - 主线关联度：⭐⭐⭐（了解原理 + 跑 Demo）

3. **Multi-Agent 系统**
   - 四种架构模式深入
   - 框架对比：AutoGen vs CrewAI vs LangGraph
   - 主线关联度：⭐⭐⭐⭐（当单 Agent 准确率遇瓶颈时深入）

4. **Code Agent**
   - 里程碑：Copilot(2021) → SWE-bench(2023) → Devin(2024) → Claude Code(2025)
   - 主线关联度：⭐⭐⭐⭐⭐（日常使用就是学习）

5. **RL for Agent**
   - RLHF → GRPO → DeepSeek-R1
   - Agent RL 的特殊挑战（奖励稀疏、探索空间大、环境非平稳）
   - 主线关联度：⭐⭐（了解概念，等有大量使用数据后深入）

### 主线 vs 支线速查

| 主题 | 与主线关联 | 何时深入 | 建议深度 |
|------|----------|---------|---------|
| MCP / A2A | ⭐⭐⭐⭐⭐ | 需要标准化工具调用时 | 必修 |
| Computer Use | ⭐⭐⭐ | 有操作无API软件的需求时 | 了解 + Demo |
| Multi-Agent | ⭐⭐⭐⭐ | 单Agent准确率遇瓶颈时 | 理解四种模式 |
| Code Agent | ⭐⭐⭐⭐⭐ | — | 日常深度使用 |
| RL for Agent | ⭐⭐ | 有大量使用数据时 | 了解概念 |

```
阶段一（初识Agent）
    │
    v
阶段二（Agent演进）
    │
    v
阶段三（底层核心·五大支柱）←──────── 动手起点
    │
    v

    v                         │

    v
```

---

> **这个大纲的稳定性来自三条原则**：
> 1. 每个阶段有单一、明确的目标（认知 → 理解 → 动手 → 深入 → 产品化 → 实战 → 跟进）
> 2. 阶段之间顺序依赖清晰，不会让学习者在"该做什么"上迷茫
> 3. 五大支柱作为课程的核心锚点，其余内容围绕它展开
>
> 这份大纲应该经得起 1-2 年的考验。具体课程内容（框架名称、协议版本、论文引用）可以随技术发展更新，但阶段结构不需要频繁调整。
