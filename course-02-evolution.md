# 课程二：Agent 范式演进

## 课程导言

课程一让你从产品体验出发，看见了 Agent 和 Chatbot、Workflow 的区别。你已经知道，一个产品是否具备 Agent 特征，关键不在于它是否用了 LLM，而在于：

```text
谁决定下一步？
```

课程二要回答更深一层的问题：

```text
为什么 Agent 会演进成今天这种形态？
```

学习 Agent 最容易踩的坑，是一上来就问“某个框架怎么用”。这样会很快接触到一堆名词：Tool Use、Function Calling、RAG...。每个词看起来都重要，但你很难判断它们之间的关系，也不知道什么时候该用、什么时候不该用。

更好的学习方式是反过来：

```text
先问系统遇到了什么问题，再看新能力为什么出现。
```

例如：

- 工具调用出现，是因为应用场景要求 LLM 从回答问题走向连接外部能力。
- ReAct 重要，是因为它把一次性回答变成了可反馈的行动循环。
- Planning 出现，是因为复杂任务不能只靠一步回答完成。
- Reflection 出现，是因为模型和工具都会失败，系统需要纠错机制。
- Multi-Agent 出现，是因为单 Agent 在角色、上下文、工具数量和并行任务上会遇到瓶颈。

本课是“范式理解层”。它不要求你立刻实现完整 Agent Runtime，也不要求你记住每篇论文的细节。它的目标是帮你建立判断力：当你看到一个新 Agent 技术或产品时，能问出正确的问题：

- 它解决的是哪个旧问题？
- 它把什么能力放进了系统？
- 它让哪些任务变得更容易？
- 它又带来了哪些新的工程问题？

带着这套问题意识进入课程三，你会更容易理解为什么一个最小 Agent 需要 LLM 决策、工具交互、状态管理和循环控制，而不是只需要一个“更聪明的模型”。

---

## 学习目标

学完本课，你将能够：

1. **解释传统 Agent 为什么没有大规模进入普通应用**：理解规则建模、状态建模、计划空间和开放环境之间的矛盾。
2. **说明 LLM 如何改变 Agent 的决策核心**：知道从“人工写规则”到“模型理解目标”的转变意味着什么，也知道它没有自动解决可靠性问题。
3. **讲清工具调用为什么出现**：从“LLM 会说但不会做”的问题出发，理解 Toolformer、ChatGPT Plugins 和 Function Calling 分别解决什么。
4. **解释 ReAct 为什么是理解 Agent 循环的经典框架**：能说清 Thought、Action、Observation 如何组成反馈循环。
5. **区分模型内部推理和系统层纠错机制**：知道“让模型多想想”和“让系统能发现并修正错误”不是同一件事。
6. **说明 Planning、Reflection、Multi-Agent 的出现背景**：能用具体任务解释这些范式解决的问题和带来的新问题。
7. **用“问题 -> 解法 -> 新问题”分析任何 Agent 能力**：面对新框架、新论文或新产品时，不停留在名词层面。

## 第一章：传统 Agent 的历史背景

“Agent”不是 LLM 时代才出现的概念。早在人工智能和软件工程中，人们就在研究一个系统如何感知环境、做出决策、采取行动，并根据结果调整自己。

### 1.1 没有 LLM 时，怎么做一个“会办事”的系统

先想一个没有 LLM 的时代会遇到的问题。

你想做一个“自动出差助理”。它需要根据你的日程、预算、偏好和公司政策，帮你安排航班和酒店。

这个系统至少要处理：

- 你什么时候出发？
- 哪些航班符合预算？
- 酒店离会议地点多远？
- 公司报销政策允许什么价位？
- 如果航班延误，是否要重新安排酒店入住时间？
- 是否需要在付款前让用户确认？

如果没有 LLM，这个系统通常要靠工程师提前写规则：

```text
如果会议在上午 10 点之前，优先选择前一天晚上到达。
如果航班价格超过预算 20%，需要用户确认。
如果酒店距离会议地点超过 3 公里，降低推荐优先级。
...
```

规则少的时候，这种系统还可以工作。可是一旦需求变复杂，规则会迅速膨胀。用户说一句“我不想太折腾，但也别太贵”，系统就很难处理，因为“太折腾”和“太贵”不是简单的固定字段。

这就是传统 Agent 面临的核心困境：

```text
系统需要像人一样围绕目标做判断，但工程上又必须把世界拆成明确规则和状态。
```

### 1.2 Agent 的基本问题从来没有变

无论有没有 LLM，Agent 要解决的基本问题都很稳定：

```text
感知环境 -> 判断状态 -> 选择行动 -> 执行动作 -> 接收反馈 -> 调整下一步
```

例如出差助理：

- 感知环境：读取日程、航班、酒店、预算。
- 判断状态：知道当前还没有订票，会议时间固定，预算有限。
- 选择行动：先查航班，还是先查酒店？
- 执行动作：调用航班搜索 API。
- 接收反馈：发现下午航班便宜但到达太晚。
- 调整下一步：改查前一天晚上的航班。

这个循环本身并不新。困难在于：在开放现实世界里，环境状态太复杂，用户目标太模糊，可能行动太多，反馈也不总是清晰。

传统 Agent 方法的努力方向，就是想办法把这些东西形式化。

### 1.3 强化学习 Agent、软件 Agent、BDI 和经典 Agent 分类

传统 Agent 里有几类重要思路。

#### 强化学习 Agent

强化学习可以先理解成“在环境中不断试错的决策系统”。

比如一个游戏 AI：

- 它看到当前画面和分数。
- 它决定向左、向右、攻击还是防守。
- 如果得分变高，说明这一步可能有用。
- 如果失败或被击败，说明这一步可能不合适。
- 经过大量尝试后，它逐渐学会什么状态下该做什么。

这类 Agent 在游戏、机器人控制、推荐策略等场景中很重要，因为这些场景可以定义状态、动作和奖励。

但它也有明显限制：

- 需要大量试错。
- 奖励函数很难设计。
- 开放语言任务不容易转成明确状态和动作。
- 用户目标经常不是一个简单分数能表达的。

#### 软件 Agent

软件 Agent 更接近日常工程系统。它们可以自动监控、触发任务、执行脚本、同步数据。

例如：

- 邮件过滤器自动分类邮件。
- 监控系统发现服务异常后自动重启。
- 数据同步程序定时把一个系统的数据写入另一个系统。

这些系统能“自动做事”，但通常只适合规则明确、边界稳定的任务。一旦用户目标变成自然语言，它们就很难理解。

#### BDI 模型

BDI 是 Belief、Desire、Intention 的缩写，可以用“一个靠谱助理如何做事”来理解。

假设你让助理安排一次出差：

- Belief：他知道你下周三有会议、预算有限、晚上不想太晚到酒店。
- Desire：他的目标是让你顺利参加会议，并尽量减少路上折腾。
- Intention：他决定先查航班，再比较酒店，最后把方案发给你确认。

BDI 的价值在于，它提醒我们：Agent 不只是被动响应输入，还应该围绕目标形成计划。

但 BDI 的工程实现依然困难。系统需要明确定义世界状态、目标、规则和计划空间。当任务进入开放自然语言世界，建模成本会急剧上升。

#### Russell & Norvig 的 Agent 分类

经典人工智能教材里会把 Agent 分成几类：

| 类型 | 大致含义 | 局限 |
|---|---|---|
| 简单反射 Agent | 根据当前输入直接反应 | 不处理复杂历史和目标 |
| 基于模型的反射 Agent | 维护对世界状态的内部模型 | 世界模型难以完整构建 |
| 基于目标的 Agent | 根据目标选择行动 | 目标和行动空间需要建模 |
| 基于效用的 Agent | 在多个方案中计算偏好 | 效用函数很难设计 |
| 学习型 Agent | 根据经验改进行为 | 学习信号和泛化能力难控制 |

这些分类今天仍然有帮助，因为它们描述的是 Agent 系统的基本问题，而不是某个时代的具体实现。

### 1.4 封闭环境里可以工作，开放任务里很难扩展

传统 Agent 并不是失败的技术。它们在很多封闭环境中非常有效。

例如：

- 游戏环境：状态、动作、奖励都比较明确。
- 工业控制：传感器、控制器、反馈信号有清晰定义。
- 企业自动化：流程稳定，规则可以提前写好。
- 运维脚本：触发条件和处理动作比较固定。

这些场景的共同点是：

```text
世界可以被建模，行动空间可以被枚举，反馈信号可以被定义。
```

但普通用户日常任务不是这样。

用户会说：

```text
帮我看看这个项目最近为什么变慢了。
```

这句话背后可能涉及代码、日志、数据库、网络请求、产品改动、部署环境、用户行为。你很难提前写完所有规则。

所以传统 Agent 的根本限制不是“没有智能”，而是：

```text
它们很难低成本地理解开放语言目标，并把目标转成可执行决策。
```

### 1.5 开放语言世界需要新的决策核心

传统 Agent 留下了一个重要问题：

```text
如果用户目标无法完全规则化，Agent 的决策核心应该是什么？
```

在 LLM 出现之前，这个问题很难回答。工程师要么继续写规则，要么为特定领域训练模型，要么把任务限制在非常窄的范围里。

LLM 出现后，情况发生了变化。它让系统第一次具备了较强的自然语言理解、常识推理和任务解释能力。于是 Agent 的核心问题从：

```text
如何把所有规则写出来？
```

变成了：

```text
如何让模型根据目标和上下文判断下一步？
```

这就是第二章要讲的范式变化。

## 第二章：LLM 带来的范式变化

LLM 没有让 Agent 的所有问题消失，但它改变了 Agent 最关键的一层：决策核心。

### 2.1 用户用自然语言表达目标，系统却只认识规则

想象你在一个代码项目里说：

```text
帮我看看为什么登录接口最近变慢了，尽量找出原因。
```

这不是一个固定按钮能表达的任务。系统需要理解：

- “登录接口”可能对应哪些文件、路由、服务或日志？
- “最近”意味着要看时间范围、提交记录或监控数据。
- “变慢”可能是数据库、缓存、网络、代码逻辑或第三方服务导致的。
- “找出原因”不是简单生成答案，而是需要逐步排查。

如果用传统 Workflow 做，它必须提前定义流程：

```text
读取指标 -> 查询日志 -> 查询数据库 -> 分析最近提交 -> 输出报告
```

这适合一部分场景。但如果用户下一次问的是“帮我把这个模块重构一下”或“帮我评估这个 PR 有没有风险”，固定流程又不够了。

问题在于：

```text
真实任务的目标表达是开放的，但传统系统的决策方式是封闭的。
```

### 2.2 Agent 的决策核心发生了变化

LLM 带来的最大变化，是它可以在自然语言层面承担一部分决策工作。

过去系统要靠工程师写：

```text
如果用户问性能问题，就先查指标。
如果用户问测试失败，就先运行测试。
如果用户问代码重构，就先读相关文件。
```

LLM 时代，系统可以把当前目标、已有上下文、可用工具和历史反馈交给模型，让模型判断：

```text
下一步最有价值的动作是什么？
```

这不是说模型一定判断正确，而是决策机制变了。

| 维度 | 传统系统 | LLM Agent |
|---|---|---|
| 目标理解 | 依赖预定义字段和意图分类 | 直接理解自然语言目标 |
| 下一步选择 | 由流程或规则决定 | 由模型结合上下文判断 |
| 任务覆盖 | 窄而稳定 | 更开放、更通用 |
| 失败模式 | 规则覆盖不到 | 模型误判、幻觉、不稳定 |
| 工程重点 | 写规则和流程 | 管理上下文、工具、状态和控制边界 |

所以 LLM 不是简单“增强了 Chatbot”。它让 Agent 有机会从“规则驱动”走向“目标驱动”。

### 2.3 让 LLM 承担目标理解和下一步判断

在 LLM Agent 中，模型通常承担几类职责：

#### 目标理解

用户不会总是给出清晰任务说明。模型需要把模糊目标转成可操作目标。

例如：

```text
用户说：这个项目有点乱，帮我整理一下。
```

模型要判断这可能意味着：

- 先查看目录结构。
- 找出重复或废弃文件。
- 理解主要模块。
- 给出整理建议。
- 在得到确认后再修改文件。

#### 上下文解释

Agent 不只看用户一句话，还要看：

- 当前文件。
- 历史对话。
- 工具返回结果。
- 错误日志。
- 任务进度。

模型需要把这些信息合起来，判断当前状态。

#### 下一步决策

Agent 最关键的问题是：

```text
现在应该做什么？
```

下一步可能是：

- 回答用户。
- 继续读取文件。
- 调用搜索。
- 执行测试。
- 请求用户确认。
- 停止任务。

#### 结果解释

工具返回的数据通常不是最终答案。模型还要解释结果，判断它是否足够支持下一步。

例如测试失败后，模型需要从报错里判断：

- 是语法错误？
- 是断言失败？
- 是环境依赖缺失？
- 是测试本身过期？

### 2.4 从固定流程走向动态决策

LLM 让很多过去很难产品化的任务变得可能。

例如：

```text
帮我调研某个行业的最新趋势。
```

Agent 可以：

1. 拆解问题。
2. 搜索资料。
3. 判断哪些来源可信。
4. 阅读页面。
5. 提炼观点。
6. 补充遗漏信息。
7. 生成报告。

这个过程不是一个完全固定的流程。搜索结果不同，下一步就会不同；发现资料不足，Agent 需要继续查；发现信息冲突，Agent 需要对比来源。

这就是 LLM Agent 和普通 Workflow 的差异：

```text
Workflow 适合步骤稳定的任务。
Agent 适合目标明确但路径不完全确定的任务。
```

### 2.5 会判断不等于能可靠执行

LLM 成为决策核心之后，新问题马上出现：

| 新问题 | 表现 | 后续需要的能力 |
|---|---|---|
| 事实不足 | 模型不知道实时信息、私有数据或当前系统状态 | 工具调用、RAG、数据库连接 |
| 无法行动 | 模型只能输出文本，不能真实执行任务 | Tool Use、Function Calling、Runtime |
| 状态不稳 | 多步任务中忘记前面做过什么 | State、Memory、Trace |
| 循环失控 | 一直查资料、重复调用工具、迟迟不停止 | Loop Control、Stop Condition |
| 错误传播 | 前一步错了，后面继续基于错误推进 | Reflection、Evaluation、Guardrails |
| 风险动作 | 模型可能误删文件、误发消息、越权操作 | Permission、Human-in-the-loop |

这说明：

```text
LLM 是 Agent 的决策核心，但不是完整 Agent 系统。
```

为了让模型从“会判断”走向“能做事”，下一步必须连接外部能力。这就引出了工具调用。

## 第三章：工具调用为什么出现

随着应用场景从“回答问题”走向“完成任务”，实际需求要求 LLM 连接外部能力，而不能只依赖参数知识和文本建议。

它需要连接：

- 实时信息。
- 私有数据。
- 计算能力。
- 文件系统。
- 数据库。
- 业务系统。
- 外部 API。

### 3.1 LLM 会说，但不会做

早期 LLM 很像一个聪明但“没有手”的人。

你说：

```text
帮我检查这个项目为什么测试失败。
```

它可能回答：

```text
你可以先运行测试命令，查看错误日志，再定位相关文件。
```

这段建议可能是对的，但它没有真的运行测试，也没有读取日志。

你说：

```text
帮我确认这个订单是否已经退款。
```

它可能回答：

```text
你可以去订单系统查询退款状态。
```

这仍然只是建议。真正有用的系统应该能在权限允许的情况下调用订单 API，读取真实状态，再告诉你结果。

这就是工具调用出现的直接背景：

```text
用户要的不是“告诉我怎么做”，而是“在可控边界内帮我做”。
```

### 3.2 回答能力必须连接外部能力

LLM 的参数知识有几个天然限制：

- **不实时**：模型训练完成后，不会自动知道最新事件、最新库存、最新订单状态。
- **不私有**：模型不知道你公司的数据库、项目文件、内部文档。
- **不精确执行**：模型可以描述计算步骤，但不适合替代计算器、解释器或数据库。
- **不改变世界**：模型输出文本不会自动发邮件、改文件、创建工单或调用业务系统。

所以 Agent 系统必须把两类能力分开：

```text
模型负责理解目标和判断下一步。
工具负责获取事实、执行动作和返回结果。
```

这条边界非常重要。模型不应该假装自己查过数据库，也不应该假装自己运行过代码。它应该生成工具调用请求，由 Runtime 执行真实动作，再把结果返回给模型。

### 3.3 Toolformer：从研究上证明模型可以学习使用工具

#### 背景

在工具调用成为常见产品能力之前，一个关键研究问题是：

```text
语言模型能不能学会什么时候需要工具？
```

如果模型每次都调用工具，成本很高，也会制造噪音。如果模型该调用时不调用，就会继续幻觉或算错。

#### 研究思路

Toolformer 关注的问题是：

```text
模型能不能自己学会何时调用 API、传什么参数，以及如何使用返回结果？
```

它不是为每个任务人工写规则，而是用少量示例引导模型生成候选 API 调用，再筛选出真正有助于后续预测的调用样本，用这些样本继续训练模型。

换句话说，它把“使用工具”变成一种可学习的语言行为。

#### 实现方式

Toolformer 把工具调用写进文本序列里，让模型学习：

- 什么时候插入 API 调用。
- 应该调用哪个工具。
- 参数应该怎么写。
- 工具返回结果如何影响后续生成。

论文中使用的工具包括计算器、问答系统、搜索、翻译和日历等。

#### 产生的效果

Toolformer 证明了一件重要的事：

```text
工具使用不只是外部系统给模型加的能力，也可以成为模型学习到的行为模式。
```

这对 Agent 很关键。因为 Agent 的工具调用不能只靠硬编码规则，模型本身也要能判断“我现在需要外部信息”。

#### 新问题和未来方向

Toolformer 更偏研究范式，不是普通开发者最直接接触的接口。它没有完整解决：

- 工具权限如何控制。
- 工具执行失败如何处理。
- 工具结果是否可信。
- 工具调用如何被记录和调试。
- 高风险动作是否需要用户确认。

这些问题会在产品和工程层面继续展开。

### 3.4 ChatGPT Plugins：把“AI 可以做事”变成产品体验

#### 背景

用户对 LLM 的期待很快从“回答问题”变成“帮我完成任务”。

例如：

- 查实时新闻。
- 查机票和酒店。
- 订餐。
- 读取公司知识库。
- 运行计算。
- 调用第三方服务。

这些任务需要真实服务，而不是只生成建议。

#### 产品探索

ChatGPT Plugins 的意义在于，它把工具调用放进了普通用户能感知的产品体验里。

用户可以在 ChatGPT 中启用插件。插件把自己的能力、接口和使用方式描述给模型，模型再根据用户目标决定是否调用。

#### 实现方式

一个插件通常需要两类信息：

- 面向模型的能力描述：这个插件能做什么，什么时候适合用。
- 面向系统的接口描述：有哪些 API、参数是什么、认证方式是什么。

这让模型可以在对话中看到可用工具，并在需要时发起调用。

早期插件形态包括：

- 浏览器：访问最新网页信息。
- 代码解释器：运行代码和处理文件。
- 检索插件：查询私有或组织知识。
- 第三方服务插件：调用外部业务系统。

#### 产生的效果

Plugins 改变了大众对 LLM 的直觉：

```text
AI 不再只是生成文本，而是可以连接外部世界。
```

这一步非常重要。它让用户第一次在通用聊天入口里看到“模型 + 工具”的组合能力。

#### 新问题和未来方向

Plugins 也暴露了很多工程问题：

- 工具需要被准确描述给模型。
- 工具调用需要权限和安全边界。
- 用户需要知道模型什么时候在调用外部服务。
- 第三方工具可能带来隐私、越权和供应链风险。
- 工具返回内容可能诱导模型执行错误动作。

所以 Plugins 的意义不在于它是唯一形态，而在于它把“LLM 连接外部世界”推向了大规模产品体验。

### 3.5 Function Calling：工具调用的工程化接口

#### 背景

Plugins 更偏 ChatGPT 产品体验。开发者真正需要的是一种稳定接口，把模型接入自己的函数、数据库、内部系统和外部 API。

例如你在自己的应用里希望模型能：

- 根据用户请求创建订单。
- 从文本里抽取结构化字段。
- 查询数据库。
- 调用 CRM 系统。
- 触发内部工作流。

这需要比“插件商店”更底层、更可控的接口。

#### 工程沉淀

Function Calling 把工具调用抽象成一条清晰边界：

```text
模型负责决定要不要调用函数，以及生成结构化参数。
Runtime 负责验证参数、执行函数，并把真实结果返回给模型。
```

这一步让工具调用从“产品能力”变成了“开发者接口”。

#### 实现方式

基本流程是：

```text
开发者定义工具名称、描述和参数 Schema
    -> 模型根据用户目标决定是否调用工具
    -> 模型生成结构化参数
    -> Runtime 校验参数
    -> Runtime 执行真实函数或 API
    -> 工具结果返回给模型
    -> 模型继续判断下一步或生成最终回答
```

这里的关键不是“模型直接执行函数”。模型只是生成结构化调用意图，真正执行发生在你的系统里。

#### 产生的效果

Function Calling 让很多应用可以稳定地把自然语言连接到业务系统。

它可以用于：

- 把自然语言转换成 API 调用。
- 把用户请求转换成数据库查询。
- 从文本中抽取结构化数据。
- 在 Agent 循环中稳定调用工具。
- 把用户目标接入已有业务流程。

#### 新问题和未来方向

Function Calling 让接口更清晰，但没有消除复杂度。开发者仍然要处理：

- Schema 设计是否准确。
- 参数校验和重试。
- 工具结果是否可信。
- 权限如何控制。
- 高风险动作是否需要用户确认。
- 模型版本变化是否影响调用行为。
- Trace、Evaluation 和 Guardrails 如何覆盖工具链路。

### 3.6 工具调用是一整套系统机制

工具调用让 LLM 从“会说”走向“会做”，但它马上带来新的工程问题。

| 新问题 | 表现 | 后续课程 |
|---|---|---|
| 工具定义问题 | 模型不知道工具能做什么，或误解工具边界 | 课程四 |
| 工具选择问题 | 该用不用，不该用乱用，或选错工具 | 课程四 |
| 参数生成问题 | 参数类型错误、遗漏必填项、编造不存在对象 | 课程四 |
| 执行失败问题 | API 超时、权限不足、结果太长、外部服务错误 | 课程四 / 课程六 |
| 安全问题 | 误删文件、越权查询、发送错误消息、触发真实世界影响 | 课程四 / 课程六 / 课程七 |
| 可观测问题 | 用户和开发者不知道模型为什么调用这个工具 | 课程六 / 课程七 |

所以工具调用不是“接一个 API”这么简单。它是一整套系统机制。

工具调用解决了“LLM 不能行动”的问题，但还没有解决“什么时候行动、行动后如何继续”的问题。这就引出 ReAct。

## 第四章：ReAct 的意义

ReAct 是理解 Agent 运行循环的经典框架。它的核心不是某个框架 API，而是一种运行方式：

```text
Reasoning + Acting
```

也就是把推理和行动交织在一起。

### 4.1 一次性回答无法处理变化中的任务

假设你问 Agent：

```text
帮我找出这个项目测试失败的原因。
```

一个只会一次性回答的模型可能说：

```text
你应该先运行测试，然后查看错误日志，再检查相关代码。
```

这仍然是建议，不是解决问题。

一个只会调用工具但不会推理的系统可能机械执行：

```text
运行测试 -> 输出一大段日志 -> 结束
```

它拿到了信息，但没有判断下一步该看哪个文件，也没有解释错误原因。

真实任务需要的是：

1. 先判断当前缺什么信息。
2. 采取行动获取信息。
3. 观察结果。
4. 根据新信息继续判断。
5. 直到足够回答或应该停止。

### 4.2 推理、行动和观察不能分开

在 ReAct 之前，我们可以把 LLM 的能力粗略分成两类：

- 推理：模型在文本里分析问题。
- 行动：模型输出某个动作或调用某个工具。

问题是，两者经常分开。

如果只有推理：

- 模型无法接触最新事实。
- 模型容易基于错误假设继续推理。
- 模型不知道外部环境真实状态。

如果只有行动：

- 系统不知道为什么调用这个工具。
- 工具结果不会自然转化为下一步策略。
- 多步任务缺少连贯判断。

Agent 需要的是一个反馈循环：

```text
判断 -> 行动 -> 观察 -> 再判断
```

### 4.3 ReAct 把判断和行动放进同一个循环

一个简化的 ReAct 过程如下：

```text
Thought: 我需要知道测试具体失败在哪里。
Action: RunTests["npm test"]
Observation: 有 2 个测试失败，错误指向 auth/session.spec.ts。

Thought: 错误和 session 过期逻辑有关，我应该读取相关实现。
Action: ReadFile["src/auth/session.ts"]
Observation: 代码中 expiresAt 使用了秒级时间戳，但测试传入毫秒级时间戳。

Thought: 现在可以解释失败原因，并给出修复建议。
Final Answer: 测试失败是因为时间单位不一致...
```

这个循环有三个关键点：

- Thought 让模型表达当前判断。
- Action 让模型影响外部环境或获取外部信息。
- Observation 让工具结果回到上下文，影响下一步判断。

实际产品不一定会把 Thought 展示给用户。重要的是系统内部要有类似结构：

```text
模型判断下一步 -> 工具执行 -> 结果回填 -> 模型继续判断
```

### 4.4 Agent 有了最小运行模式

ReAct 重要，不是因为所有产品都必须显式输出 Thought，而是因为它清楚展示了 Agent 从 Chatbot 走向 Agent 的最小运行模式。

它把一次性回答变成了：

```text
判断下一步 -> 调用工具或行动 -> 接收观察 -> 继续判断 -> 停止
```

这正是课程三要实现的最小 Agent 闭环。

ReAct 的意义包括：

- 将推理、行动、观察放入同一个循环。
- 让 Agent 可以根据外部反馈调整下一步。
- 给出了最小运行时雏形。
- 暴露出后续工程问题。

你可以把 ReAct 理解成 Agent 的“最小行为语法”。它不等于生产系统，但它让我们看清生产系统要管理什么。

### 4.5 循环本身也需要被管理

ReAct 本身是范式，不是生产系统。

它会引出很多后续问题：

| 问题 | 说明 | 后续课程 |
|---|---|---|
| 停止条件 | Agent 什么时候应该结束？ | 课程三 / 课程六 |
| 循环控制 | 如果一直调用同一工具怎么办？ | 课程三 / 课程六 |
| 工具失败 | 工具超时、报错、返回无关结果怎么办？ | 课程四 / 课程六 |
| 上下文膨胀 | Observation 越来越多，上下文放不下怎么办？ | 课程六 |
| 状态管理 | 哪些信息进入下一轮？哪些只写入日志？ | 课程三 / 课程六 |
| 用户控制 | 高风险动作前是否需要确认？ | 课程四 / 课程七 |

所以 ReAct 是理解 Agent 的起点，不是工程落地的终点。

## 第五章：内部推理能力与系统纠错机制

进入 Planning 和 Reflection 之前，需要先澄清一个常见混淆：

```text
模型更会推理，不等于系统更可靠。
```

### 5.1 让模型“多想想”为什么仍然不够

假设 Agent 要修改一段代码。它先读取文件，然后给出修改方案，最后编辑代码。

如果模型在第一步误解了需求，后面可能全部建立在错误理解上。

你可能会想：

```text
那让模型多思考几步不就行了吗？
```

这有帮助，但不够。

原因是很多错误不是靠“想得更久”就能发现的。例如：

- 代码是否真的能编译，需要运行编译器。
- 测试是否通过，需要运行测试。
- API 是否可用，需要真实调用。
- 数据库查询是否正确，需要连接真实数据库。
- 用户是否接受某个方案，需要用户确认。

模型内部推理再强，也不能替代外部验证。

### 5.2 模型内部推理和系统外部反馈是两层能力

模型内部推理指的是模型在生成答案或决策前，能更好地处理复杂问题、拆分步骤、比较路径、发现局部矛盾。

它影响：

- LLM Decision 的质量。
- 计划的合理性。
- 工具选择的准确性。
- 输出答案的逻辑性。

但它仍然发生在模型内部。

系统外部反馈指的是运行时从模型外部收集信号，判断当前结果是否真的可用。

例如：

- 编译器报错。
- 单元测试失败。
- JSON Schema 校验失败。
- 工具返回权限错误。
- 用户驳回模型计划。
- 检索结果与回答不一致。

这两层能力的区别可以概括为：

| 维度 | 模型内部推理 | 系统外部反馈 |
|---|---|---|
| 发生位置 | 模型生成过程中 | Agent Runtime 中 |
| 依据 | 上下文和模型能力 | 工具结果、验证器、用户反馈 |
| 优点 | 快、灵活、泛化强 | 更接近真实世界 |
| 局限 | 可能自洽但错误 | 需要工程设计和成本 |

### 5.3 把可验证反馈放进运行时

可靠 Agent 不能只依赖模型“自己觉得对”。它需要在运行时引入可验证反馈。

常见方式包括：

#### Schema 校验

如果模型需要生成结构化参数，就用 Schema 检查：

```text
字段是否存在？
类型是否正确？
枚举值是否合法？
```

#### 工具执行反馈

工具执行后，把真实结果返回给模型：

```text
成功、失败、超时、权限不足、结果为空、结果过长。
```

#### 测试和验证器

对于代码任务，测试比模型自评更可信。

```text
模型说“修好了”不算，测试通过才算。
```

#### 用户确认

对于高风险动作，用户确认是重要反馈。

例如：

- 删除文件。
- 发送邮件。
- 创建订单。
- 修改生产配置。

### 5.4 错误可以被发现、记录和处理

一旦系统把反馈放进运行时，Agent 就可以从“生成答案”走向“处理任务”。

例如代码修复流程：

```text
模型提出修改
    -> 编辑文件
    -> 运行测试
    -> 测试失败
    -> 把错误日志回填给模型
    -> 模型重新定位问题
    -> 再次修改
    -> 测试通过或达到停止条件
```

这时系统不再只是相信模型，而是在用真实信号约束模型。

这也是后续 Reflection、Evaluation、Guardrails 的基础。

### 5.5 纠错机制必须依赖真实信号

纠错机制听起来很美好，但如果反馈信号不可靠，它也会出问题。

常见问题包括：

- 模型自评过于乐观。
- 评价标准太模糊。
- 工具结果不完整。
- 测试覆盖不足。
- 重试次数过多导致成本失控。
- 系统不断修正但没有收敛。

所以未来的 Agent Runtime 不会只依赖“让模型反思一下”。更可靠的方向是：

```text
模型反思 + 工具验证 + 测试反馈 + 用户确认 + 可观测 Trace
```

这为第七章的 Reflection 做准备。但在讲 Reflection 之前，我们先看另一个问题：复杂任务为什么需要 Planning。

## 第六章：Planning 的出现背景

Planning 解决的是长任务结构问题。

不是所有任务都适合一步一步临场发挥。有些任务从一开始就需要整体结构。

### 6.1 复杂任务不是一步回答，而是一串依赖关系

假设你让 Agent 做一件事：

```text
帮我调研 3 个竞品，比较它们的定位、功能、价格和增长策略，最后给出我们产品的建议。
```

如果 Agent 直接开始搜索，很容易出现几种问题：

- 只查了第一个竞品就开始写结论。
- 查了很多资料但没有统一比较维度。
- 忘记价格和增长策略。
- 引用来源混乱。
- 结论和前面调研材料脱节。

这不是因为模型不会写字，而是任务本身有结构：

```text
确定竞品 -> 定义比较维度 -> 搜集资料 -> 整理事实 -> 横向比较 -> 形成建议
```

每一步都依赖前一步。

### 6.2 没有计划，Agent 容易局部最优和中途漂移

ReAct 擅长动态探索，但如果任务很长，单纯依赖“下一步判断”会有风险。

常见问题包括：

#### 局部最优

Agent 看到一个容易处理的信息，就围绕它展开，忽略全局目标。

例如只找到一个竞品的资料，就开始过度分析。

#### 遗漏约束

长任务有多个要求。模型在后续步骤中可能忘记某些约束。

例如用户要求“价格、功能、增长策略”，最后只写了功能。

#### 步骤顺序错误

有些任务必须先定义标准，再搜集资料。否则资料会不可比。

#### 难以验收

没有计划，就很难判断任务做到哪一步、还差什么。

Planning 的核心价值是：

```text
先形成任务结构，再推进执行。
```

### 6.3 从 CoT、ToT 到 Plan-and-Execute

Planning 不是一个单一技术，而是一组让模型更好组织复杂任务的方法。

#### Chain of Thought

Chain of Thought 让模型显式或隐式地分步思考。

它解决的是：

```text
不要直接跳到答案，先拆解中间推理。
```

在 Agent 系统中，CoT 提醒我们：复杂任务需要中间步骤，而不是一次生成最终答案。

#### Tree of Thoughts

Tree of Thoughts 进一步把思考从“一条链”扩展为“多条候选路径”。

它关注的问题是：

```text
如果有多种解法，模型能不能比较不同路径？
```

这对开放任务很重要。比如写市场分析报告，可以从用户群、商业模式、产品功能、渠道策略等不同路径进入。

#### Plan-and-Execute

Plan-and-Execute 更接近 Agent 工程。

它通常分成两层：

```text
Planner: 先生成任务计划。
Executor: 按计划执行每一步，并把结果返回。
```

例如：

```text
Plan:
1. 确定 3 个主要竞品。
2. 为每个竞品收集官网、价格页和新闻资料。
3. 按定位、功能、价格、增长策略整理表格。
4. 对比差异。
5. 给出建议。
```

执行时，Agent 可以逐步完成，并在每一步后记录状态。

### 6.4 复杂任务有了结构化推进方式

Planning 带来的直接效果是：

- 任务目标更清晰。
- 步骤依赖更明确。
- 中间产物更容易检查。
- 用户更容易知道 Agent 在做什么。
- 系统更容易判断任务是否完成。

它特别适合：

| 场景 | 为什么需要 Planning |
|---|---|
| 调研报告 | 需要统一维度和资料来源 |
| 代码重构 | 需要先理解影响范围，再分步修改 |
| 数据分析 | 需要先定义指标，再处理数据 |
| 项目迁移 | 需要顺序、回滚和验证 |
| 多系统操作 | 需要明确依赖和风险点 |

### 6.5 计划需要被执行、检查和重规划

Planning 也会带来新问题。

#### 计划可能一开始就是错的

模型生成的计划看起来完整，但可能遗漏关键步骤。

#### 计划可能执行不下去

执行过程中工具失败、资料缺失、权限不足，原计划就需要调整。

#### 计划可能过度复杂

简单任务如果强行 Planning，会浪费成本和时间。

#### 计划和执行可能脱节

Planner 写了一个漂亮计划，Executor 实际执行时却走偏。

所以未来更可靠的 Planning 不是“一次性写完计划”，而是：

```text
计划 -> 执行 -> 检查 -> 重规划 -> 继续执行
```

这也解释了为什么课程五会继续讲 Chain、Router、ReAct Loop、Plan-Execute、Graph。不同任务需要不同组织方式。

## 第七章：Reflection 的出现背景

Reflection 解决的是错误修正问题。

在多步任务里，错误不是偶然情况，而是常态。模型会误解，工具会失败，资料会冲突，用户需求也可能变化。

### 7.1 多步任务中的小错误会被放大

假设 Agent 正在写一份竞品分析。

第一步，它把 A 公司和 B 公司混淆了。

第二步，它基于混淆后的信息做功能对比。

第三步，它根据错误对比给出产品建议。

最后报告看起来结构完整，但核心事实错了。

这类问题很危险，因为它不是简单的语法错误，而是“错误沿着任务链传播”。

代码任务里也一样：

- 读错需求。
- 改错文件。
- 测试没跑全。
- 看到一个错误就以为是根因。
- 修复 A 问题时引入 B 问题。

如果系统没有纠错机制，Agent 就会把错误一路带到最终输出。

### 7.2 Agent 需要从结果中学习当前这次任务

这里的“学习”不是指重新训练模型，而是指在当前任务中根据反馈调整行为。

人类做复杂任务时也会这样：

- 写完一段代码后运行测试。
- 写完一份报告后检查事实来源。
- 做完一个方案后请同事 review。
- 发现方向错了就回退重来。

Agent 也需要类似机制。

Reflection 的核心问题是：

```text
系统如何发现当前输出有问题，并把这个发现转化为下一轮改进？
```

### 7.3 执行、评价、修正的反馈回路

一个典型 Reflection 回路是：

```text
执行任务
    -> 产生中间结果或最终结果
    -> 评价结果
    -> 发现问题
    -> 生成修正策略
    -> 重新执行或局部修改
```

评价信号可以来自不同来源。

#### 模型自评

让模型检查自己的输出是否遗漏需求、是否逻辑矛盾。

优点是成本低、适用范围广。

缺点是模型可能自我确认，发现不了自己不知道的错误。

#### 工具验证

用测试、编译、Schema 校验、检索对齐等方式检查结果。

优点是更接近真实信号。

缺点是需要工程建设。

#### 用户反馈

用户直接指出问题或调整目标。

优点是最符合实际需求。

缺点是不能每一步都打断用户。

#### 专门评估器

用另一个模型或规则系统评估输出。

优点是可以分离生成和评价。

缺点是评估器本身也可能错。

### 7.4 系统从“一次生成”走向“可迭代改进”

有了 Reflection，Agent 可以处理更多真实任务中的不确定性。

例如代码修复：

```text
修改代码
    -> 运行测试
    -> 测试失败
    -> 分析失败原因
    -> 修改策略
    -> 再次运行测试
```

例如报告写作：

```text
生成初稿
    -> 检查是否覆盖所有维度
    -> 发现缺少价格信息
    -> 补充检索
    -> 更新报告
```

这让 Agent 不再只是“一次性输出”，而是能基于反馈迭代。

### 7.5 反思不能变成自我安慰

Reflection 最大的误区是把它理解成：

```text
让模型再评价一下自己。
```

这太弱了。

真正有价值的 Reflection 应该尽量依赖真实反馈：

- 测试结果。
- 编译结果。
- 工具返回。
- 用户确认。
- 数据校验。
- 多来源事实对齐。

Reflection 也会带来新问题：

| 新问题 | 表现 |
|---|---|
| 无限修正 | Agent 一直觉得还能改，不停止 |
| 成本上升 | 每次生成后都评估和重试 |
| 评价不可靠 | 自评或评估器本身判断错误 |
| 目标漂移 | 修着修着偏离用户原始目标 |
| 过度自信 | 模型把错误解释得很合理 |

所以未来的 Reflection 会越来越依赖可观测运行时：

```text
Trace + Evaluation + Tool Feedback + Human Review
```

## 第八章：Multi-Agent 的出现背景

Multi-Agent 解决的是单 Agent 的扩展问题。

当任务变复杂，一个 Agent 同时承担所有职责，会很快变得难以管理。

### 8.1 一个 Agent 很快会变成“什么都管”的巨型系统

假设你想让一个 Agent 帮你完成产品发布：

- 分析需求。
- 设计方案。
- 写代码。
- 写测试。
- 做安全检查。
- 写文档。
- 准备发布说明。
- 监控上线结果。

如果全部交给一个 Agent，它需要同时扮演：

- 产品经理。
- 架构师。
- 前端工程师。
- 后端工程师。
- 测试工程师。
- 安全工程师。
- 技术写作者。
- 运维工程师。

这会带来几个问题：

- 上下文太大。
- 工具太多。
- 角色目标冲突。
- 难以并行。
- 输出标准不一致。

### 8.2 复杂任务需要角色、上下文和工具边界

人类团队不会让一个人同时负责所有事情。不是因为一个人绝对做不到，而是因为分工可以降低复杂度。

Agent 系统也类似。

一个代码实现 Agent 需要关注：

- 当前文件怎么改。
- 测试怎么跑。
- 类型错误怎么修。

一个代码审查 Agent 需要关注：

- 有没有 bug。
- 有没有安全风险。
- 有没有缺测试。
- 有没有架构退化。

一个文档 Agent 需要关注：

- 用户能不能看懂。
- 概念是否准确。
- 示例是否完整。

这些角色需要不同上下文、不同工具、不同评价标准。

Multi-Agent 的核心问题是：

```text
如何把一个复杂任务拆给多个有边界的 Agent 协作？
```

### 8.3 角色分工、管理者-执行者和专家协作

Multi-Agent 有几种常见形态。

#### 角色分工

给不同 Agent 定义不同角色。

例如：

- Research Agent：收集资料。
- Planning Agent：制定计划。
- Coding Agent：执行代码修改。
- Review Agent：检查风险。
- Writer Agent：整理文档。

每个 Agent 的目标更窄，上下文更聚焦。

#### 管理者-执行者

一个 Manager Agent 负责拆解任务和分配工作，多个 Worker Agent 负责执行具体子任务。

例如：

```text
Manager: 这个需求分成 API、前端、测试、文档四块。
Worker A: 修改 API。
Worker B: 修改前端。
Worker C: 补测试。
Worker D: 写文档。
Manager: 汇总结果并检查一致性。
```

#### 专家协作

不同 Agent 从不同角度评估同一方案。

例如：

- 性能专家看性能风险。
- 安全专家看权限和数据风险。
- 产品专家看用户价值。
- 工程专家看实现复杂度。

这适合决策质量要求高的任务。

### 8.4 能力可以扩展，任务可以并行

Multi-Agent 的好处包括：

- 角色更清晰。
- 上下文更可控。
- 工具权限更容易隔离。
- 多个子任务可以并行。
- 复杂任务可以分阶段验收。
- 审查和执行可以分离。

例如一个产品发布流程可以变成：

```text
需求分析 Agent
    -> 方案设计 Agent
    -> 实现 Agent
    -> Review Agent
    -> 文档 Agent
    -> 发布检查 Agent
```

这比一个 Agent 从头做到尾更容易管理。

### 8.5 协作本身也会制造复杂度

Multi-Agent 并不是万能解法。

它会带来新的问题：

| 问题 | 表现 |
|---|---|
| 协调成本 | Agent 之间需要传递状态和结果 |
| 责任不清 | 出错时不知道哪个 Agent 负责 |
| 信息丢失 | 子任务结果被汇总时丢掉关键细节 |
| 冲突处理 | 不同 Agent 给出相反建议 |
| 成本上升 | 多个 Agent 调用模型和工具 |
| 调试困难 | Trace 分散在多个执行单元里 |

所以未来更实用的 Multi-Agent 方向，不是“越多 Agent 越好”，而是：

```text
角色清晰、边界明确、通信可追踪、结果可验证。
```

很多生产系统会把 Multi-Agent 和 Graph、Workflow、Human Review 结合起来，而不是让一群 Agent 自由聊天。

## 第九章：范式演进总览

本课的核心不是记住概念，而是理解 Agent 能力为什么一步步长出来。

### 9.1 一张表看完整条演进线

| 阶段 | 问题代入 | 解决方案 | 产生的效果 | 新问题和未来方向 |
|---|---|---|---|---|
| 传统 Agent | 系统需要感知、决策和行动 | RL、软件 Agent、BDI、经典 Agent 分类 | 在封闭环境里可控执行 | 开放语言任务建模成本高 |
| LLM 决策核心 | 用户用自然语言表达开放目标 | 让 LLM 理解目标和判断下一步 | 从规则驱动走向目标驱动 | 会判断不等于能执行 |
| 工具调用 | LLM 会说但不会做 | Toolformer、Plugins、Function Calling | 连接外部信息和真实动作 | 权限、失败、安全、Trace |
| ReAct | 一次性回答无法处理动态任务 | Thought / Action / Observation 循环 | Agent 有了最小运行模式 | 循环需要停止和控制 |
| 内部推理与系统纠错 | 模型多想想仍可能错 | 外部反馈、验证器、测试、用户确认 | 错误可以被发现和处理 | 反馈信号必须可靠 |
| Planning | 长任务有依赖关系 | CoT、ToT、Plan-and-Execute | 复杂任务结构化推进 | 需要执行检查和重规划 |
| Reflection | 多步错误会传播 | 执行、评价、修正回路 | 系统可迭代改进 | 不能只靠模型自评 |
| Multi-Agent | 单 Agent 角色和上下文过载 | 角色分工、管理者-执行者、专家协作 | 能力扩展和并行执行 | 协调、成本、责任边界 |

### 9.2 如何分析一个新的 Agent 技术

以后你看到一个新的 Agent 框架、论文或产品功能，可以用这组问题判断它的位置：

#### 第一步：它解决什么旧问题？

不要先问“它是不是先进”，先问：

- 它解决工具调用问题吗？
- 它解决长任务规划问题吗？
- 它解决状态管理问题吗？
- 它解决错误反馈问题吗？
- 它解决多 Agent 协作问题吗？

#### 第二步：它把能力放在哪一层？

Agent 系统至少有这些层：

- 模型层：模型本身更会推理或更会调用工具。
- 接口层：模型如何表达工具调用和结构化输出。
- Runtime 层：系统如何执行、记录、控制和恢复。
- 产品层：用户如何感知、确认和信任 Agent。
- 组织层：多个 Agent 或人类如何协作。

不同技术解决的是不同层的问题。

#### 第三步：它带来什么新问题？

任何能力增强都会带来新问题。

例如：

- 工具越多，选择越难。
- 计划越复杂，执行越容易脱节。
- Agent 越多，协调越困难。
- 自动化越强，安全边界越重要。

能看见新问题，才说明你真正理解了旧方案。

### 9.3 常见误区

#### 误区一：把 Agent 当成更强 Chatbot

Chatbot 的核心是回答。Agent 的核心是围绕目标持续决策和执行。

#### 误区二：以为加了工具就是 Agent

工具调用只是 Agent 的一部分。没有状态管理、循环控制和反馈处理，工具调用很容易变成一次性 API 包装。

#### 误区三：以为 Planning 越复杂越好

简单任务不需要复杂计划。过度 Planning 会增加成本和失败点。

#### 误区四：以为 Reflection 就是让模型自我批评

真正有价值的 Reflection 应该尽量依赖外部真实信号。

#### 误区五：以为 Multi-Agent 一定比 Single-Agent 强

Multi-Agent 只有在角色边界清晰、通信可追踪、结果可验证时才有价值。

## 下一课衔接

课程二解释了 Agent 范式为什么这样演进。你已经看到：

- 传统 Agent 提出了“感知、决策、行动、反馈”的基本问题。
- LLM 让自然语言目标理解和动态决策成为可能。
- 工具调用让模型连接外部能力。
- ReAct 给出了最小行动循环。
- Planning、Reflection、Multi-Agent 分别处理复杂任务、错误修正和能力扩展。

课程三会把这些概念收束成一个更具体的问题：

```text
一个最小但完整的 Agent，到底由什么组成？
```

你将从概念演进进入系统结构，学习如何把：

```text
LLM 决策 + 工具/环境交互 + 状态管理 + 循环控制
```

组合成一个最小可运行的 Agent 闭环。
