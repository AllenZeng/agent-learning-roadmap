# 课程二：Agent 范式演进

## 课程导言

课程一让你从产品体验出发，看见了 Agent 和 Chatbot、Workflow 的区别。你已经知道，一个产品是否具备 Agent 特征，关键不在于它是否用了 LLM，而在于：**谁决定下一步**。

课程二要回答更深一层的问题：

```text
为什么 Agent 会演进成今天这种形态？
```

这门课不按概念堆砌来讲，也不把 ReAct、Function Calling、Planning等当作平铺的模块清单。它采用一条更适合学习者理解的主线：

```text
之前有什么问题 -> 新方法解决了什么 -> 新方法又带来了什么新问题
```

你会看到，Agent 的每一种重要能力都不是凭空出现的：

- 工具调用出现，是因为应用场景要求 LLM 从回答问题走向连接外部能力。
- ReAct 重要，是因为它把一次性回答变成了可反馈的行动循环。
- Planning 出现，是因为复杂任务不能只靠一步回答完成。
- Reflection 出现，是因为模型和工具都会失败，系统需要纠错机制。
- Multi-Agent 出现，是因为单 Agent 在角色、上下文、工具数量和并行任务上会遇到瓶颈。

本课是“范式理解层”。它解释概念为什么出现，不要求你先掌握全部实现。如果你更想先动手，可以先学课程三，做完最小 Agent 闭环后再回来读本课。

---

## 学习目标

完成本课程后，你应该能够：

- 解释传统 Agent 为什么很少进入普通用户的日常应用场景。
- 说明 LLM 出现后，Agent 的决策核心发生了什么变化。
- 讲清 Toolformer、Plugins、Function Calling 分别解决了什么问题。
- 解释 ReAct 为什么适合作为理解 Agent 运行循环的经典框架。
- 区分模型内部推理能力和系统层 Reflection 机制。
- 说明 Planning、Reflection、Multi-Agent 分别是在什么背景下出现的。
- 用“问题 -> 解法 -> 新问题”的框架分析一个 Agent 技术或产品公告。

## 第一章：传统 Agent 的历史背景

“Agent”不是 LLM 时代才出现的概念。在人工智能和软件工程中，Agent 已经存在了几十年。

早期 Agent 关注的是：

```text
一个系统如何感知环境、做出决策、采取行动，并根据结果调整行为？
```

这个问题本身并不新。真正的新变化，是 LLM 让 Agent 的“决策核心”发生了变化。

### 1.1 强化学习 Agent

先不要被“强化学习”这个词吓到。你可以先把它想成一个不断试错的游戏玩家。

比如一个打游戏的 AI：
- 它看到当前画面和分数。
- 它决定下一步是向左、向右、攻击还是防守。
- 如果得分变高，说明这一步可能是好选择。
- 如果失败或被击败，说明这一步可能不合适。
- 经过大量尝试后，它逐渐学会“什么状态下该做什么”。

在强化学习中，这个不断与环境交互、根据反馈调整策略的决策实体，就叫 Agent。

典型循环是：

```text
观察状态 -> 选择行动 -> 获得奖励或惩罚 -> 更新策略 -> 再次行动
```

这种 Agent 在游戏、机器人控制、推荐策略等环境中很有价值。AlphaGo、Atari 游戏智能体、机器人控制系统，都可以放在这个脉络里理解。

但它有一个明显限制：环境、状态、动作、奖励函数通常需要被清晰定义。

也就是说，它适合“规则空间明确”的任务，不适合真实世界里开放、模糊、自然语言驱动的任务。

### 1.2 软件 Agent

软件工程领域也长期讨论 Agent。例如自动邮件过滤、日程助手、信息检索助手、自动化脚本系统，都曾被称为 software agent。

这类 Agent 通常具备一些特征：

- 自治性：能在一定范围内自行运行。
- 反应性：能响应外部事件。
- 主动性：能根据目标触发行动。
- 社交性：能与其他系统或用户交互。

问题在于，很多软件 Agent 本质上仍然依赖规则和固定流程。用户必须用系统能理解的方式表达需求，开发者必须提前写好大量规则。

### 1.3 BDI 模型

BDI 可以先用“一个靠谱助理如何做事”来理解。

假设你让助理帮你安排一次出差：
- 他知道你下周三有会议、预算不高、晚上不想太晚到酒店。
- 他知道目标是“顺利参加会议，并尽量减少路上折腾”。
- 他决定先查航班，再比较酒店，最后把两个方案发给你确认。

这里面其实有三层东西：

| 概念 | 含义 |
|---|---|
| Belief | 它认为当前世界是什么样的，例如时间、预算、限制条件 |
| Desire | 它想达成什么目标，例如完成出差安排 |
| Intention | 它当前承诺执行哪条计划，例如先查航班、再查酒店 |

BDI 就是 Belief、Desire、Intention 的缩写。它试图让 Agent 具备类似“心智状态”的结构。它能帮助我们理解：一个 Agent 不只是被动反应，还可能围绕目标形成计划。

但 BDI 的工程实现依然困难。系统需要明确定义世界状态、目标、规则和计划空间。当任务进入开放的自然语言世界，这套建模成本会急剧上升。

### 1.4 Russell & Norvig 的 Agent 类型

《Artificial Intelligence: A Modern Approach》中常见的 Agent 分类包括：

| 类型 | 工作方式 | 局限 |
|---|---|---|
| 简单反射型 Agent | 根据当前输入直接触发规则 | 没有记忆，无法处理需要历史信息的任务 |
| 基于模型的反射型 Agent | 维护内部状态模型 | 仍然偏反应式，目标能力有限 |
| 目标驱动型 Agent | 根据目标选择行动 | 需要明确目标和搜索空间 |
| 效用驱动型 Agent | 根据效用函数权衡行动 | 效用函数难以设计 |

这些分类到今天仍然有价值，因为它们说明了 Agent 的基本问题从来没有变：

```text
感知什么 -> 记住什么 -> 追求什么 -> 选择什么行动
```

变化的是，LLM 让“理解目标”和“生成下一步行动”这两件事有了新的可能。

### 1.5 传统 Agent 的根本限制

传统 Agent 很少成为普通用户每天使用的大众产品，核心原因不是“Agent 这个想法不重要”，而是它们太依赖封闭规则空间。

主要限制包括：

- **规则空间封闭**：系统只能处理设计者预先考虑过的状态和动作。
- **泛化能力弱**：一个为下棋设计的 Agent 很难迁移到订机票、写代码、整理文档。
- **自然语言理解弱**：用户必须用系统能识别的格式表达需求。
- **工程成本高**：每个领域都要重新定义状态、动作、规则、异常处理。
- **用户入口不自然**：普通用户很难直接把开放目标交给系统。

所以，传统 Agent 更像在固定轨道上运行的系统。它可以稳定、高效，但难以处理开放目标。

## 第二章：LLM 带来的范式变化

LLM 出现后，Agent 的核心变化不是“多了一个聊天界面”，而是决策核心发生了变化。

过去是：

```text
人类写规则 -> 系统按规则执行
```

现在变成：

```text
用户给目标 -> 模型理解目标 -> 模型生成下一步 -> 系统执行并反馈
```

### 2.1 从“人工写规则”到“模型理解目标”

传统系统需要开发者提前写清楚：

- 用户可能说什么。
- 用户意图有哪些。
- 每种意图对应什么流程。
- 每一步失败时如何处理。

LLM 改变了这一点。它可以直接处理自然语言，把模糊目标转成下一步行动建议。

例如用户说：

```text
帮我看看这个项目为什么测试失败，并给出修复建议。
```

传统系统需要预先定义大量流程。LLM Agent 则可能自己判断：

```text
先读取项目结构 -> 再运行测试 -> 再分析报错 -> 再定位相关文件 -> 再提出修复
```

这就是决策核心的变化。

### 2.2 从“固定流程”到“动态决策”

固定 Workflow 的特点是：

```text
步骤 A -> 步骤 B -> 步骤 C -> 结束
```

LLM Agent 的特点是：

```text
根据当前上下文和反馈，决定下一步做什么
```

两者的区别不是哪个更高级，而是适用场景不同。

| 场景 | 更适合 Workflow | 更适合 Agent |
|---|---|---|
| 步骤稳定 | 是 | 不一定 |
| 规则明确 | 是 | 不一定 |
| 目标开放 | 不一定 | 是 |
| 中途需要动态判断 | 不一定 | 是 |
| 风险高且必须可控 | Workflow 更容易兜底 | Agent 需要额外控制机制 |

LLM 让动态决策成为可能，但动态决策也带来不确定性。这就是后面课程为什么要讲状态管理、循环控制、工具权限、Evaluation 和 Guardrails。

### 2.3 从“只回答”到“能调用外部能力”

LLM 本身擅长语言理解和生成，但它有天然限制：

- 不知道实时信息。
- 不知道私有数据。
- 不能直接查询数据库。
- 不能直接运行代码。
- 不能直接修改文件。
- 不能直接操作业务系统。

如果 LLM 只停留在文本生成，它更像 Chatbot。Agent 的关键一步，是让 LLM 能连接外部能力。

这就引出了工具调用。

## 第三章：工具调用为什么出现

随着应用场景从“回答问题”走向“完成任务”，实际需求要求 LLM 连接外部能力，而不能只依赖参数知识和文本建议。

它需要连接：

- 实时信息。
- 私有数据。
- 计算能力。
- 文件系统。
- 数据库。
- 业务系统。
- 外部 API。

### 3.1 问题：LLM 被困在对话框里

早期 LLM 很像一个聪明但“没有手”的人。

它可以告诉你：

```text
你应该先运行测试，再查看错误日志。
```

但它不能真的运行测试，也不能读取日志。

它可以告诉你：

```text
你可以查询订单系统确认退款状态。
```

但它不能真的查询订单系统。

这导致一个明显问题：

```text
LLM 会说，但不会做。
```

工具调用就是为了解决这个问题。

### 3.2 Toolformer：让模型学会何时调用工具

**背景**：早期 LLM 已经能通过提示完成很多任务，但在算术、事实查询、日期计算、检索等问题上，单靠参数知识并不可靠。更合理的方向是：让模型在需要时调用专门工具。

**研究**：Toolformer 关注的问题是：

```text
模型能不能自己学会何时调用 API、传什么参数，以及如何使用返回结果？
```

它的做法不是人工为每个任务写规则，而是用少量示例引导模型生成候选 API 调用，再筛选出真正有助于后续预测的调用样本，用这些样本继续训练模型。

**实现**：Toolformer 把工具调用写进文本序列里，让模型学习：

- 什么时候插入 API 调用。
- 应该调用哪个工具。
- 参数应该怎么写。
- 工具返回结果如何影响后续生成。

论文中使用的工具包括计算器、问答系统、搜索、翻译和日历等。

**效果**：Toolformer 证明了一件重要的事：工具使用不只是外部系统给模型加的能力，也可以成为模型学习到的行为模式。模型如果能正确调用工具，就能在一些任务上弥补自身参数知识和计算能力的不足。

**带来的问题**：Toolformer 更偏研究范式，不是普通应用开发者直接使用的接口。它也没有完整解决工具权限、执行失败、安全边界、结果可信度和可观测性等工程问题。

### 3.3 ChatGPT Plugins：让普通用户第一次看到“AI 可以做事”

**背景**：用户对 LLM 的期待很快从“回答问题”变成“帮我完成任务”。例如查实时信息、订餐、订票、处理文件、查询企业知识库。这些任务需要连接真实服务，而不是只生成一段建议。

**研究 / 产品探索**：ChatGPT Plugins 把工具调用放进了普通用户能感知的产品体验里。用户可以在 ChatGPT 中启用插件，插件把自己的能力、接口和使用方式描述给模型，模型再根据用户目标决定是否调用。

**实现**：一个插件通常需要两类信息：

- 面向模型的能力描述：这个插件能做什么，什么时候适合用。
- 面向系统的接口描述：有哪些 API、参数是什么、认证方式是什么。

这让模型可以在对话中看到可用工具，并在需要时发起调用。早期插件形态包括浏览器、代码解释器、检索插件，以及第三方服务插件。

**效果**：Plugins 改变了大众对 LLM 的直觉：AI 不再只是生成文本，而是可以访问最新信息、运行计算、检索私有数据，甚至调用第三方业务服务。

**带来的问题**：Plugins 也把很多 Agent 工程问题暴露了出来：

- 工具需要被描述给模型。
- 工具调用需要权限和安全边界。
- 用户需要知道模型什么时候在调用外部服务。
- 第三方工具可能带来安全、隐私和越权风险。
- 工具返回的内容可能诱导模型执行错误动作。

所以 Plugins 的意义不在于它是今天唯一的工具形态，而在于它第一次把“LLM 连接外部世界”变成了大规模产品体验。

### 3.4 Function Calling：工具调用的工程化接口

**背景**：Plugins 更偏 ChatGPT 产品体验，但开发者需要一种更稳定的方式，把模型接入自己的函数、数据库、内部系统和外部 API。

**研究 / 工程沉淀**：Function Calling 把插件探索中的关键经验沉淀成 API 接口：

```text
模型负责决定要不要调用函数，以及生成结构化参数。
Runtime 负责验证参数、执行函数，并把真实结果返回给模型。
```

这一步很重要。模型不应该假装自己已经查过数据库，也不应该假装自己已经运行了代码。它应该生成结构化调用请求，由确定性系统执行。

**实现**：基本流程是：

```text
开发者定义工具名称、描述和参数 Schema
    -> 模型根据用户目标决定是否调用工具
    -> 模型生成结构化参数
    -> Runtime 执行真实函数或 API
    -> 工具结果返回给模型
    -> 模型继续判断下一步或生成最终回答
```

**效果**：Function Calling 让工具调用从“产品插件”变成了开发者可以直接集成的工程接口。它可以用于：

- 把自然语言转换成 API 调用。
- 把用户请求转换成数据库查询。
- 从文本中抽取结构化数据。
- 在 Agent 循环中稳定调用工具。

**带来的问题**：Function Calling 让接口更清晰，但没有消除工程复杂度。开发者仍然要处理：

- Schema 设计是否准确。
- 参数校验和重试。
- 工具结果是否可信。
- 高风险动作是否需要用户确认。
- 模型版本变化是否影响调用行为。
- Trace、Evaluation 和 Guardrails 如何覆盖工具链路。

这个责任边界会贯穿后续课程：

- 课程三会把工具调用放进最小 Agent 闭环。
- 课程四会深入工具定义、工具选择、参数生成、执行回填和权限安全。
- 课程六会讲如何把工具调用纳入 Harness、Trace、Evaluation 和 Guardrails。

### 3.5 工具调用带来的新问题

工具调用让 LLM 从“会说”走向“会做”，但它立刻带来新的工程问题。

| 新问题 | 表现 | 后续课程 |
|---|---|---|
| 工具定义问题 | 模型不知道工具能做什么，或误解工具边界 | 课程四 |
| 工具选择问题 | 该用不用，不该用乱用，或选错工具 | 课程四 |
| 参数生成问题 | 参数类型错误、遗漏必填项、编造不存在对象 | 课程四 |
| 执行失败问题 | API 超时、权限不足、结果太长、外部服务错误 | 课程四 / 课程六 |
| 安全问题 | 误删文件、越权查询、发送错误消息、触发真实世界影响 | 课程四 / 课程六 / 课程七 |
| 可观测问题 | 用户和开发者不知道模型为什么调用这个工具 | 课程六 / 课程七 |

所以工具调用不是“接一个 API”这么简单。它是一整套系统机制。

## 第四章：ReAct 的意义

ReAct 是理解 Agent 运行循环的经典框架。

它的核心不是某个框架 API，而是一种运行方式：

```text
Reasoning + Acting
```

也就是把推理和行动交织在一起。

### 4.1 ReAct 之前的问题

在 ReAct 之前，LLM 常见能力可以分成两类：

- 推理：模型在文本里一步步分析问题。
- 行动：模型输出某个动作或调用某个工具。

问题是，两者经常是分开的。

如果只有推理，模型无法接触外部事实，容易幻觉。

如果只有行动，模型缺少对行动目的、当前状态和下一步策略的解释。

ReAct 的核心贡献是把它们放在同一个循环里。

### 4.2 ReAct 循环长什么样

一个简化的 ReAct 过程如下：

```text
Thought: 我需要知道这个问题是否依赖最新信息。
Action: Search["某个主题的最新进展"]
Observation: 搜索结果显示有三条相关资料。

Thought: 第一条是官方文档，可信度更高，我应该先阅读它。
Action: Open[official_document]
Observation: 文档说明了核心限制和使用方式。

Thought: 现在可以结合资料回答用户问题。
Final Answer: ...
```

这个循环有三个关键点：

- Thought 让模型表达当前判断。
- Action 让模型影响外部环境或获取外部信息。
- Observation 让工具结果回到上下文，影响下一步判断。

### 4.3 ReAct 为什么重要

ReAct 重要，不是因为所有产品都必须显式输出 Thought，而是因为它清楚展示了 Agent 从 Chatbot 走向 Agent 的最小运行模式。

它把一次性回答变成了：

```text
判断下一步 -> 调用工具或行动 -> 接收观察 -> 继续判断 -> 停止
```

这正是课程三要实现的最小 Agent 闭环。

ReAct 的意义包括：

- 将推理、行动、观察放入同一个循环。
- 让 Agent 可以根据外部反馈调整下一步。
- 给出了最小运行时雏形。
- 暴露出后续工程问题。

### 4.4 ReAct 引出的工程问题

ReAct 本身是范式，不是生产系统。

它会引出很多后续问题：

| 问题 | 说明 | 后续课程 |
|---|---|---|
| 停止条件 | Agent 什么时候应该结束？ | 课程三 / 课程六 |
| 循环控制 | 如果一直调用同一工具怎么办？ | 课程三 / 课程六 |
| 工具失败 | 工具超时、报错、返回无关结果怎么办？ | 课程四 / 课程六 |
| 上下文膨胀 | Observation 越来越多，上下文放不下怎么办？ | 课程六 |
| 状态管理 | 哪些信息进入下一轮？哪些只写入日志？ | 课程三 / 课程六 |
| 用户控制 | 高风险动作前是否需要确认？ | 课程四 / 课程七 |

所以 ReAct 是理解 Agent 的起点，不是工程落地的终点。

## 第五章：内部推理能力与系统纠错机制的区别

进入 Planning 和 Reflection 之前，需要先澄清一个常见混淆：

```text
模型更会推理，不等于系统更可靠。
```

### 5.1 模型内部推理能力是什么

模型内部推理能力指的是模型在生成答案或决策前，能更好地处理复杂问题、拆分步骤、比较路径、发现局部矛盾。

它影响的是：

- LLM Decision 的质量。
- 计划的合理性。
- 工具选择的准确性。
- 输出答案的逻辑性。

这种能力很重要，但它仍然发生在模型内部。

### 5.2 系统纠错机制是什么

系统纠错机制指的是运行时在模型外部构建验证、反馈和重试机制。

例如：

- 代码执行失败。
- 单元测试不通过。
- JSON Schema 校验失败。
- 工具返回权限错误。
- 用户驳回模型计划。
- 检索结果与回答不一致。

这些外部信号比模型自我感觉更可靠。

### 5.3 为什么不能只依赖“让模型多想想”

模型可能：

- 自信地解释一个错误答案。
- 在反思中引入新的错误。
- 用更长的推理文本包装错误判断。
- 在缺少外部反馈时无法知道自己错在哪里。

因此，一个可靠 Agent 不应把可靠性完全寄托在模型“更聪明”上。

更稳妥的原则是：

```text
模型负责提出判断，系统负责提供反馈和约束。
```

这也是 Reflection 为什么应该被理解为系统设计，而不是模型天赋。

## 第六章：Planning 的出现背景

Planning 出现，是因为复杂任务无法一步完成。

### 6.1 问题：一次性回答不适合复杂任务

简单任务可以直接回答：

```text
解释一下什么是 API。
```

复杂任务通常不适合直接回答：

```text
帮我准备一场 30 分钟的 Agent 产品开发分享，包括大纲、案例、讲稿和练习。
```

这个任务需要：

- 理解目标。
- 拆解阶段。
- 决定材料结构。
- 选择案例。
- 生成大纲。
- 生成讲稿。
- 检查完整性。

如果模型直接生成最终答案，很容易遗漏步骤或前后不一致。

### 6.2 从 CoT 到 ToT 再到 Plan-and-Execute

Planning 的相关方法可以按复杂度理解：

| 方法 | 解决的问题 | 风险 |
|---|---|---|
| Chain-of-Thought | 让模型逐步推理，而不是直接给答案 | 沿一条路径走到底，早期错误会传导 |
| Tree of Thoughts | 探索多条推理路径，并进行选择或回溯 | 成本高、实现复杂 |
| Plan-and-Execute | 先制定计划，再按计划执行 | 计划可能不可执行，执行中可能脱节 |
| Replanning | 根据执行结果调整计划 | 需要状态管理和停止条件 |

这些方法的共同目的，是让模型在复杂任务中不只是“马上输出答案”，而是先组织问题。

### 6.3 Planning 适合什么场景

Planning 适合：

- 步骤多。
- 依赖关系强。
- 目标需要拆解。
- 用户希望先看到计划。
- 执行过程中可能需要阶段性确认。

例如：

- 写一份研究报告。
- 修复复杂 bug。
- 设计一个小型产品方案。
- 整理一批文档并生成总结。
- 做项目发布准备。

### 6.4 Planning 带来的新问题

Planning 并不总是好事。

它会带来：

- 计划看起来合理但不可执行。
- 计划过细，增加 token 和时间成本。
- 任务本来简单，却被过度拆解。
- 环境变化后，原计划不再适用。
- 用户看计划看累了，反而降低体验。

所以课程五会进一步讨论 Planning / Workflow Patterns 的选择：什么时候用 Chain，什么时候用 Router，什么时候用 Plan-and-Execute，什么时候用 Graph。

## 第七章：Reflection 的出现背景

Reflection 出现，是因为模型和工具调用都会失败。

### 7.1 问题：错误会在多步任务中传播

在单轮问答中，模型错了，用户可以直接指出。

在多步 Agent 任务中，一个早期错误可能传导到后面：

```text
工具参数填错 -> 查询结果为空 -> 模型误以为没有相关数据 -> 输出错误结论
```

如果系统没有纠错机制，Agent 会沿着错误路径继续执行。

### 7.2 Reflection 解决什么

Reflection 的目标不是让模型“自我觉醒”，而是让系统在失败后有机会修正。

它可以用于：

- 执行后自检。
- 根据测试结果重试。
- 根据工具错误修正参数。
- 对输出做批判。
- 对失败进行分类。
- 记录经验，避免下次重复犯错。

### 7.3 Reflection 应该依赖什么信号

Reflection 最好由外部信号触发：

| 信号 | 可靠性 | 示例 |
|---|---|---|
| 规则校验 | 高 | JSON 不合法、字段缺失、数值越界 |
| 测试结果 | 高 | 单元测试失败、编译失败 |
| 工具错误 | 中高 | 权限不足、参数不合法、网络失败 |
| 用户反馈 | 高 | 用户驳回计划或指出事实错误 |
| 模型自评 | 不稳定 | 模型自己觉得答案可能不完整 |

模型自评可以辅助，但不应成为唯一依据。

### 7.4 Reflection 带来的新问题

Reflection 也有成本和风险：

- 反思增加延迟和 token 成本。
- 模型可能自信地反思错误。
- 反思可能引入新错误。
- 无限反思会导致任务无法结束。
- 对高风险任务，模型自评不能替代人工确认或确定性校验。

因此，Reflection 需要配合：

- 触发条件。
- 重试上限。
- 失败分类。
- 停止条件。
- 外部验证信号。

## 第八章：Multi-Agent 的出现背景

Multi-Agent 出现，是因为单 Agent 在复杂任务中会遇到明确瓶颈。

### 8.1 单 Agent 的瓶颈

单 Agent 常见问题包括：

- 角色混淆：既要规划、执行、审查，又要写最终答案。
- 上下文拥挤：所有信息都塞进一个上下文窗口。
- 工具数量过多：几十个工具放在一起，模型选择困难。
- 难以并行：多个独立子任务只能串行执行。
- 审查不足：同一个模型既当执行者又当评审者。

这些问题会让单 Agent 的能力边界变得明显。

### 8.2 Multi-Agent 解决什么

Multi-Agent 的基本思路是分工。

常见模式包括：

| 模式 | 机制 | 适合场景 |
|---|---|---|
| Supervisor | 一个主管 Agent 分配任务，多个 Worker 执行 | 子任务边界清晰 |
| Debate | 多个 Agent 从不同角度提出观点并互相质疑 | 需要多角度判断 |
| Group Chat | 多个 Agent 在共享对话中协作 | 开放探索任务 |
| Graph | 用图结构定义协作和状态流转 | 多路径、可回放流程 |
| Reviewer | 一个 Agent 执行，另一个 Agent 审查 | 代码审查、文档审查、方案评估 |

Multi-Agent 的价值在于：

- 角色更清晰。
- 可以并行处理。
- 可以互相审查。
- 可以降低单个上下文的压力。

### 8.3 Multi-Agent 带来的新问题

Multi-Agent 也很容易变成复杂度陷阱。

常见风险包括：

- 通信成本高。
- 多个 Agent 互相放大错误。
- 角色边界不清。
- Debug 难度上升。
- 评测成本增加。
- 用户更难理解系统为什么这样做。

所以 Multi-Agent 不应该被过早引入。

不要用 Multi-Agent 的典型信号：

- 单 Agent 加清晰工具和上下文管理已经能完成任务。
- “多角色”只是 Prompt 名字不同，没有真实分工。
- 任务无法并行，多个 Agent 只会增加协调成本。
- 你还没有 trace、评测和失败分类能力。

课程五会把 Multi-Agent 放在场景增强能力中讨论；课程六会进一步区分传统 Multi-Agent 和“子代理 / 任务委托”这种运行时架构模式。

## 第九章：范式演进总览

把本课内容合在一起，可以得到一张演进图：

```text
传统 Agent
    问题：规则空间封闭、泛化弱、自然语言理解弱
    ->
LLM 作为决策核心
    解决：能理解开放目标，生成下一步
    新问题：只能说话，不能连接外部世界
    ->
Tool Use / Plugins / Function Calling
    解决：连接实时信息、私有数据、计算能力和外部系统
    新问题：工具选择、参数生成、权限、安全、执行失败
    ->
ReAct
    解决：把推理、行动、观察放入同一个循环
    新问题：停止条件、循环控制、状态管理、上下文膨胀
    ->
Planning
    解决：复杂任务需要拆解和阶段性目标
    新问题：计划不可执行、过度规划、计划和执行脱节
    ->
Reflection
    解决：模型和工具失败后需要纠错
    新问题：自信地反思错误、成本增加、无限反思
    ->
Multi-Agent
    解决：单 Agent 角色混淆、上下文拥挤、难以并行和缺少审查
    新问题：通信成本、错误放大、调试和评测复杂度
```

本课最重要的结论是：

```text
Agent 能力的演进不是能力堆叠，而是问题驱动。
```

每引入一种新能力，都要同时问：

- 它解决了什么问题？
- 它增加了什么复杂度？
- 这个复杂度在当前场景中是否值得？
