# 课程二：Agent 范式演进

---

## 课程导言

课程一让你"看见"了 Agent——通过亲身体验和产品拆解，你建立了一个感性认知：Agent 和 Chatbot 不一样，它真的能"做"一些事情。

课程二要回答的问题是：**Agent 为什么会是今天这个样子？**

你会发现，Agent 的每一个核心特征——用工具、做规划、自我纠错、多角色协作——都不是凭空设计出来的。它们背后各有一个清晰的故事：**之前有什么问题 → 新方法解决了什么 → 新方法又带来了什么新问题**。

这门课按"问题演进"来组织，而非按"概念分类"来罗列。你不只是在学"有哪些能力"，而是在学"这些能力为什么会出现、什么时候该用、什么时候不该用"。

**本课是范式理解层，解释概念为什么出现，不要求先掌握全部实现。** 如果你更想先动手，可以先跳到课程三做完最小闭环后再回来——那时候你会有更具体的疑问，本课的回答也会更加有力。

---

## 学习目标

学完本课，你将能够：

1. **讲清 Agent 范式的演进逻辑** — 每一次新方法的出现，都是在解决前一阶段暴露的什么问题
2. **理解 ReAct 为什么是 Agent 的基石框架** — 它如何将推理、行动、观察编织成同一个循环
3. **区分"模型能力"和"系统机制"** — 模型的思考/推理能力 ≠ 系统的可靠性，Reflection 是系统设计而非模型天赋
4. **用"问题→解法→新问题"框架分析任何 Agent 技术或产品公告** — 不被术语和热度带着走

## 第一章：传统 Agent 的前世今生

在谈 LLM Agent 之前，你需要知道："Agent"这个词在 AI 领域已经存在了几十年。理解它从哪里来，才能真正理解 LLM 改变了什么。

### 1.1 LLM 出现之前的 Agent

**强化学习中的 Agent**

在强化学习（Reinforcement Learning）框架中，Agent 是一个与环境持续交互的决策实体。它的工作模式是一个经典循环：

```text
观察环境状态 → 选择行动 → 接收奖励/惩罚 → 更新策略 → 再次观察
```

这个框架的数学基础可以追溯到 Bellman 1957 年的动态规划最优性原理。1989 年 Watkins 提出 Q-learning，2013 年 DeepMind 用 DQN 玩 Atari 游戏，2016 年 AlphaGo 击败李世石——这些都是 RL Agent 的高光时刻。

RL Agent 的核心特征是：**在封闭环境中通过试错学习最优策略**。它在游戏、机器人控制等「状态空间和动作空间可穷举或可连续建模」的场景中表现极强，但状态和动作都需要人工预先定义。

**软件 Agent 与 BDI 模型**

20 世纪 90 年代，软件工程领域兴起了"软件 Agent"的研究。1994 年，Wooldridge 和 Jennings 定义了软件 Agent 的两个层次：

- **弱定义**：自治性、社交能力、反应性、主动性
- **强定义（BDI 模型）**：在弱定义基础上增加 Belief（信念）、Desire（愿望）、Intention（意图）——Agent 不只是反应，还有自己的"心智状态"

1995 年，Pattie Maes 在 MIT 媒体实验室开发了能帮用户调度会议、过滤邮件的"自主 Agent"。这些 Agent 运行在封闭的、规则明确的环境中。

**Russell & Norvig 的 Agent 分类**

Stuart Russell 和 Peter Norvig 在《人工智能：一种现代方法》中将 Agent 分为四种基本类型，代表了从简单到复杂的演进：

| 类型 | 工作机制 | 局限 |
|------|---------|------|
| 简单反射型 | 基于当前感知直接行动（if-then） | 没有记忆，无法处理需要历史信息的任务 |
| 基于模型的反射型 | 维护内部状态模型，能处理部分可观测环境 | 只能反应，不能主动追求目标 |
| 目标驱动型 | 知道目标状态，能规划从当前到目标的行动序列 | 无法在多个目标间做权衡 |
| 效用驱动型 | 用效用函数量化不同状态的"好坏"，选择最优 | 效用函数的设计本身极其困难 |

### 1.2 传统 Agent 的根本限制

无论是 RL Agent、软件 Agent 还是规则驱动的目标型 Agent，它们共享一个致命局限：**只能在封闭的、预定义的规则空间内运行。**

这意味着三件事：

1. **规则依赖**：所有"感知→行动"的映射必须由人类专家预先定义。遇到规则之外的情况，Agent 要么崩溃，要么做出无意义的反应。
2. **泛化能力缺失**：一个会下围棋的 Agent 不会下象棋。每个 Agent 都是为特定任务定制的，无法迁移。
3. **自然语言理解薄弱**：即使设计了 NLP 接口的 Agent，也只能处理有限的、结构化的命令模式，无法真正理解自然语言中的模糊意图。

**这就是传统 Agent 为什么很少进入普通用户的日常应用场景**——构建一个传统 Agent 需要穷举定义所有可能的状态、动作和规则，在真实世界的开放性任务中，这几乎不可行。

用一个比喻：传统 Agent 像地铁——轨道是铺好的，它只能在轨道上跑，跑得快且稳定，但永远无法开到没有铺轨的地方。

## 第二章：LLM 带来的范式革命

### 2.1 决策核心变了：从"人工写规则"到"模型理解目标"

2020 年 GPT-3 发布后，研究者注意到了一个现象：当模型规模超过某个临界点后，它突然展现出了一些在设计时未曾预料的能力——这就是**涌现能力（Emergent Abilities）**。少样本推理、逐步推理、代码生成——这些能力不是工程师"塞进去"的，而是从海量数据和参数规模中"长出来"的。

这引发了一个根本性的问题：**如果一个模型已经具备了通用推理的"种子"，它能不能成为一个真正的自主决策核心？**

LLM 在三个维度上改变了 Agent 的可能性：

| 维度 | 传统 Agent | LLM Agent |
|------|-----------|-----------|
| 决策方式 | 人工编写规则 | 模型理解目标并生成下一步 |
| 流程特征 | 固定流程 | 动态决策 |
| 能力边界 | 只执行预设动作 | 能通过工具调用连接外部能力 |

换句话说，LLM 让 Agent 有了一个**通用的大脑**——它不一定在每个具体任务上比专用系统强，但它的泛化能力是传统 Agent 无法企及的。你不再需要为每个任务单独建造一条轨道。

但这里有一个关键认知：**"通用的大脑"也意味着"不可预测的大脑"。** LLM 的决策是不确定的——同一个目标，每次执行可能选择不同的路径。这既是灵活性的来源，也是工程复杂性的来源。

### 2.2 ReAct：推理与行动首次交织

2022 年 10 月，Google Research 和普林斯顿大学联合发表了《ReAct: Synergizing Reasoning and Acting in Language Models》。这篇论文在 Agent 发展史上具有里程碑意义。

**之前的问题**：当时 LLM 可以在 Prompt 引导下进行逐步推理（Chain-of-Thought），也可以在特定设置下使用工具。但这两条能力是**分离**的——要么让模型"推理"，要么让模型"行动"。推理无法从行动结果中获取新信息，行动缺乏推理的指导。

**核心洞察**：人类解决复杂问题从来不是单纯的"推理"或单纯的"行动"，而是两者的交织——先想一想，查一下资料，发现新信息后调整思路，再尝试新的方向。

**ReAct 的工作方式**：将任务过程建模为 Thought → Action → Observation 的交替循环：

```text
Thought: 我需要知道 2024 年奥运会在哪个城市举办
Action: Search["2024 Summer Olympics host city"]
Observation: 2024 年夏季奥运会将在法国巴黎举办
Thought: 现在我知道是巴黎，需要查巴黎近期的天气
Action: Weather[Paris]
Observation: 巴黎晴，15-22°C
Thought: 已获得所有需要的信息，可以生成回答了
```

这个看似简单的交替模式，实现了 LLM 能力的质变：**模型不是在"孤岛"上推理，而是在与外部世界的持续交互中推理。** 每一次观察都可能改变它的思考方向，每一次思考都可能触发新的行动。

**ReAct 的历史地位：**

1. 它第一次清晰展示了 Agent 如何从"一次性回答"走向"**持续执行任务**"
2. 它给出了 Agent **最小运行时雏形**：判断下一步 → 调用工具 → 接收结果 → 继续或停止
3. 它**引出了后续所有工程问题**：停止条件怎么设？循环失控怎么办？工具失败了如何处理？上下文如何管理？

可以说，ReAct 不是 Agent 的终点，而是 Agent 工程化的起点。课程三的最小 Agent 闭环，本质上就是工程化一个 ReAct 循环。

## 第三章：工具调用为什么出现

### 3.1 问题：LLM 被关在对话框里

LLM 刚出现时，它能做的最多就是"说话"——你问它答，问什么答什么。它可以告诉你"应该怎么分析数据"，但它不能帮你分析数据；它可以描述一个 bug 的可能原因，但它不能帮你修。

这种"思行分离"的状态是限制 LLM 应用场景的最大瓶颈。企业想用它做客服，但它查不到订单状态；想用它做助手，但它不能帮你订机票；想用它做分析，但它无法读取实时数据。

**工具调用的本质需求很朴素：让模型能"动手"。**

### 3.2 ChatGPT Plugins：第一次全民体验

2023 年 3 月，OpenAI 发布了 ChatGPT Plugins。技术原理不复杂——每个 Plugin 暴露 API 端点和描述，ChatGPT 根据用户问题判断调用哪个、传什么参数。

但它在心理层面的意义远超技术层面：**它第一次让普通用户看到，AI 不仅可以"说"，还可以"做"。** 订餐、查机票、搜索论文、生成图表——都在一个聊天窗口里完成。

Plugins 的问题也很明显：需要手动安装，Plugin 之间难以协作，没有统一的工具描述标准。

### 3.3 Function Calling：工具调用的工程化

2023 年 6 月，OpenAI 在 GPT-4 和 GPT-3.5-turbo 上原生支持了 Function Calling。这不是新功能，而是一次**接口的标准化**。

核心机制：

```text
1. 开发者定义函数的 JSON Schema（名称、描述、参数及类型）
2. 把这些定义随请求发给模型
3. 模型决定"要不要调工具、调哪个、参数是什么"→ 返回结构化 function_call
4. 开发者在实际环境中执行这个函数
5. 执行结果传回模型
6. 模型根据结果继续推理或给出最终答案
```

这里有一个容易被忽视的关键设计：**为什么是开发者执行函数，而不是模型直接"假装"执行？**

答案在于**真实性和安全性**。模型的输出不可靠——它可能算错数学、编造订单号。让模型"假装"执行等于放弃外部验证。而把执行权留在开发者手中，意味着：数据库查询的结果总是准确的，危险操作（转账、删除）可以有权限控制，每次工具调用都在监控和审计之下。

**这个设计体现了一个根本性原则：模型负责决策（what to do），环境负责执行（how to do）**——让 LLM 专注于它擅长的语义理解和决策，把精确执行交给确定性代码。这个原则后来被几乎所有 Agent 框架继承。

**Toolformer 的补充视角**：几乎同期，Meta AI 的 Toolformer（2023 年 2 月）从另一个方向证明了——LLM 可以通过微调学会"自主判断"何时该调用工具，而不需要依赖外部 Prompt 工程来"强迫"它。这为工具调用成为模型的"内化能力"而非"外挂能力"提供了支撑。

### 3.4 工具调用带来的新问题

工具调用解决了"会说话但不会动手"的问题，但它立刻带来了新问题：

| 新问题 | 表现 | 将在哪里解决 |
|--------|------|------------|
| 工具选择错误 | 模型不知道该用哪个工具，或明明有更合适的工具却选了别的 | 课程四：工具机制 |
| 参数填错 | 参数类型不对、值不合法、遗漏必填项 | 课程四：工具机制 |
| 执行失败 | 工具超时、返回错误、结果太长淹没上下文 | 课程四：工具机制 |
| 越权调用 | 模型调用了不该调用的工具，造成真实世界影响 | 课程四 + 课程六 + 课程七 |
| 工具数量膨胀 | 工具太多时模型选择困难，上下文也装不下所有工具描述 | 课程四：MCP + Skill |
| 循环依赖 | 工具 A 的输出不够好 → 再调工具 B → 还是不够 → 再调 A⋯⋯ | 课程六：循环控制 |

**核心认知**：工具调用不是"接入就完事了"，它是一整套需要设计的系统机制——选择、执行、权限、复用、可观测。这就是为什么课程四整门课只讲工具机制。

## 第四章：Planning 的出现背景

### 4.1 问题：复杂任务无法一步完成

早期的 LLM 使用模式是"用户提问→模型直接回答"。这对于"什么是光合作用"有效，但对于"帮我规划去京都的 5 天行程，预算 1 万元"——直接生成的答案质量很差。

根本原因在于：LLM 的本质是"下一个 Token 预测器"，它天然是**逐字生成**而不是**整体规划**的。没有机会"退后一步"看全局，很容易在半路跑偏。

**Planning 解决的核心问题：复杂任务需要先拆解、再执行、阶段性调整，而不是一步到位。**

### 4.2 CoT → ToT → Plan-and-Solve

Planning 能力的演进是一条清晰的路径：

**Chain-of-Thought（2022 年 1 月）**：在 Prompt 中加入"让我们一步步思考"，模型在复杂推理任务上的表现显著提升。原理是——模型生成每个 Token 时，前面的 Token 都会影响注意力分布。如果前面的 Token 包含推理过程，最终答案就更可能正确。

但 CoT 的问题在于：它沿一条路径走到底。如果第一步就走偏了，剩下的步骤只会让错误越来越大。

**Tree-of-Thoughts（2023 年 5 月）**：同时探索多条推理路径，在每个节点评估候选步骤，必要时回溯。在"24 点游戏"和创意写作等需要系统性探索的任务上表现出色，但 Token 消耗是 CoT 的 10 倍以上。

**Plan-and-Solve**：在执行任何具体步骤之前，先制定一个完整计划，然后按计划逐步执行。这和人类写作的逻辑一致——先列大纲再动笔，修改大纲的成本远低于修改草稿。

关键设计选择：

| 模式 | 适用场景 | 风险 |
|------|---------|------|
| 一步推理 | 简单问答 | 复杂任务质量差 |
| CoT（链式） | 需要逐步推理但路径明确的 | 走偏了无法回头 |
| ToT（树状） | 需要探索多条路径的 | Token 成本高 |
| Plan-Execute | 复杂任务，可先拆解的 | 计划可能不可执行 |
| ReAct Loop | 需要动态工具选择和反馈修正的 | 步骤不稳定 |

### 4.3 Planning 引入的新问题

Planning 解决了"复杂任务一步做不完"的问题，但引入了新的复杂性：

1. **计划看起来合理但不可执行**：模型不了解真实环境约束，制定的计划可能过于理想化
2. **计划过细，浪费 Token**：简单任务也被拆成十几步，得不偿失
3. **执行与计划脱节**：环境变化后，原计划不再适用，但模型还在按原计划执行
4. **验证困难**：你怎么判断一个计划好不好？只有执行完才知道——而执行成本可能很高

**核心认知**：Planning 不是越强越好。任务简单时不规划直接做；任务复杂时才先规划后执行。是否引入 Planning，应该由任务复杂度决定，而不是由"能力列表"决定。

## 第五章：Reflection 的出现背景

### 5.1 问题：模型和工具调用都会失败

Agent 在执行任务时，失败无处不在：

- 模型可能理解错用户意图
- 工具可能返回错误或无关结果
- 参数可能填错导致调用失败
- 多步任务中，前面的错误会传导到后面

**Reflection 解决的核心问题：系统需要能发现错误、分类错误、修正错误或调整策略，而不是一条路走到黑。**

### 5.2 Reflection 是系统机制，不是模型天赋

这是本课最重要的认知分界线之一。模型可以"想得更久"（通过增加推理 Token），但这不等于系统就更可靠。

**模型的思考/推理能力**：影响 LLM Decision 的质量——更好的推理 → 更好的下一步判断。但模型无法自己验证自己的判断是否正确。

**Reflection 是系统层设计**：通常依赖外部信号来触发纠错——

- 自检（Self-check）：模型检查自己的输出是否满足约束
- 评审（Critique）：另一个模型或同一模型以"审查者"身份评价输出
- 测试结果：代码执行失败、断言不通过
- 规则校验：输出格式不对、数值越界
- 用户驳回：用户说"不对，重新来"

**一个关键区分**：

```text
"让模型多想想" → 提升单次决策质量（模型能力）
"模型输出后让系统检查、必要时重试" → 增加纠错机制（系统设计）
```

不能把"模型想得更久"等同于"系统一定更可靠"——模型完全可能自信地反思错误，或者在反思中引入新错误。

**Reflection 的正确设计姿势**：

- 被外部信号触发，而不是默认每步都反思
- 有重试上限和停止条件，避免无限反思
- 高风险任务中，模型自评不能替代测试、规则校验或人工确认

### 5.3 Reflection 引入的新问题

1. **模型自信地反思错误**：没有外部验证的反思可能是"用新错误覆盖旧错误"
2. **反思增加成本但没提升质量**：思考 Token 花了很多，答案没变好
3. **无限反思**：永远觉得自己能做得更好，任务永远不结束
4. **延迟增加**：每步都反思一遍，用户体验变差

## 第六章：Multi-Agent 的出现背景

### 6.1 问题：单 Agent 的瓶颈在哪

当任务复杂度继续提升时，单 Agent 架构开始暴露出几个明确的瓶颈：

1. **角色混淆**：一个 Agent 既要理解用户意图，又要查资料，又要写代码，又要做质量检查——同一个 System Prompt 里塞入太多职责，模型容易混乱
2. **上下文拥挤**：所有工具描述、历史对话、中间结果全部挤在一个上下文窗口里，关键信息被淹没
3. **工具数量过多**：当可用工具达到几十个甚至上百个时，单个模型在"选择正确工具"上的准确率显著下降
4. **难以并行**：单 Agent 的 ReAct 循环本质上是串行的——一次只能做一件事。但如果任务中有多个独立子任务，让它们并行执行更高效
5. **审查缺失**：模型的输出由同一个模型自己审查——既当球员又当裁判，审查效果有限

**Multi-Agent 解决的核心问题：通过角色分工、并行处理和互相审查，突破单 Agent 的能力天花板。**

### 6.2 常见协作架构

| 架构 | 工作机制 | 适用场景 |
|------|---------|---------|
| Supervisor | 一个"主管"Agent 分配任务给多个"工人"Agent，汇总结果 | 任务可明确拆分为多个子任务 |
| Debate | 多个 Agent 从不同立场讨论，通过辩论逼近更优答案 | 需要多角度审视的决策问题 |
| Group Chat | 多个 Agent 在共享上下文中自由交流 | 需要灵活协作的开放性问题 |
| Graph | 用有向图定义 Agent 之间的协作关系和数据流 | 复杂多步骤、多分支流程 |

### 6.3 Multi-Agent 引入的新问题

**Multi-Agent 是高级能力，也是复杂度陷阱。** 它在解决单 Agent 瓶颈的同时，引入了自己的问题：

1. **通信成本高**：Agent 之间需要来回传递信息，Token 消耗成倍增长
2. **角色边界不清**：两个 Agent 做了同一件事，或关键步骤没有人负责
3. **错误放大**：一个 Agent 的错误输出被另一个 Agent 当作事实输入，层层放大
4. **调试难度上升**：出问题了，你不知道是哪个 Agent 在哪个环节出的错——5 个 Agent 各有各的 trace，根因定位极其困难

**不要过早引入 Multi-Agent 的明确信号：**

- 单 Agent 加清晰工具和上下文管理已经能完成任务
- "多角色"只是 Prompt 命名不同，没有真实分工
- 任务无法并行，多 Agent 只会增加协调成本
- 你还没有 trace、评测和失败分类能力，无法调试多 Agent 行为

**核心认知**：Multi-Agent 不是"高级版 Agent"，而是单 Agent 遇到明确瓶颈后的**针对性解决方案**。如果你能说清单 Agent 哪里不够用了，再考虑引入多 Agent。否则，它带来的复杂度很可能超过它解决的问题。

## 第七章：范式全景图

回顾整条演进脉络，每一次新方法的出现，都是为了解决前一阶段暴露出来的问题：

```text
「LLM 只能说话，不能做事」
    → Tool Use / Function Calling：让模型能调用外部工具
    → 新问题：工具选错、参数填错、执行失败、越权调用

「复杂任务一步完成不了，需要多步」
    → Planning / Workflow Patterns：先拆解，再执行，分阶段推进
    → 新问题：计划不可执行、过细浪费 Token、环境变化后脱节

「模型和工具调用都会失败，没有纠错机制」
    → Reflection：系统层自检、评审、重试、纠错
    → 新问题：自信地反思错误、无限反思、增加成本但没提升质量

「单 Agent 角色混淆、上下文拥挤、无法并行、缺少审查」
    → Multi-Agent：角色分工、互相审查、并行处理
    → 新问题：通信成本高、错误放大、调试困难
```

**三条核心原则，来自整条演进脉络**：

1. **每一次能力增强都带来新的工程问题**。不存在"加了这个能力就完美了"的魔法。Planning 让任务能做对，但也可能让简单任务过度复杂化；Reflection 能纠错，但也可能陷入无限反思。
2. **不是所有 Agent 都需要所有能力**。RAG、Memory、Planning、Reflection、Multi-Agent 都是可选的场景增强——它们围绕最小 Agent 闭环展开，而不是取代它。判断什么时候该用、什么时候不该用，比会用更重要。
3. **模型能力和系统机制是两回事**。模型负责理解和决策，系统负责执行、校验、权限、恢复和观测。不要把可靠性寄托在"模型更聪明"上——用确定性的工程机制来保证下限，让模型去提升上限。

## 参考资料

### 关键论文

- **[ReAct: Synergizing Reasoning and Acting in Language Models](https://arxiv.org/abs/2210.03629)**（Yao et al., 2022）— 将推理和行动首次放入同一循环，定义了 Agent 的最小运行时雏形
- **[Toolformer: Language Models Can Teach Themselves to Use Tools](https://arxiv.org/abs/2302.04761)**（Schick et al., 2023）— 证明了 LLM 可以通过微调自主学会何时调用工具
- **[Chain-of-Thought Prompting Elicits Reasoning in Large Language Models](https://arxiv.org/abs/2201.11903)**（Wei et al., 2022）— 发现"让我们一步步思考"能显著提升推理质量
- **[Tree of Thoughts: Deliberate Problem Solving with Large Language Models](https://arxiv.org/abs/2305.10601)**（Yao et al., 2023）— 将推理从"链式"扩展到"树状"，支持多路径探索和回溯
- **[Reflexion: Language Agents with Verbal Reinforcement Learning](https://arxiv.org/abs/2303.11366)**（Shinn et al., 2023）— 将反思机制系统化，让 Agent 从失败经验中学习

### 推荐阅读

- **[Building Effective Agents](https://www.anthropic.com/engineering/building-effective-agents)** — Anthropic 关于"何时用 Agent、何时用 Workflow"的工程视角
- **[A Survey of Large Language Model based Agent](https://arxiv.org/abs/2308.11432)**（Xi et al., 2023）— LLM Agent 综述，覆盖架构、应用和评测

### 值得关注的技术动态

- OpenAI Function Calling / Tools API 的更新（关注参数定义、并行调用、强制调用等能力演进）
- Anthropic Tool Use / MCP 的进展（关注工具协议标准化和跨模型互操作）
- LangGraph / CrewAI / AutoGen 等框架的架构变化（关注 Multi-Agent 协作模式的工程实践）

### 术语速查

| 术语 | 简要解释 | 本课程定位 |
|------|---------|-----------|
| ReAct | Reasoning + Acting，将推理和行动交织在同一循环中 | Agent 最小运行时的经典范式 |
| CoT | Chain-of-Thought，逐步推理 | Planning 能力的起点 |
| ToT | Tree-of-Thoughts，树状多路径推理 | CoT 的演进，探索多条路径 |
| Function Calling | 模型返回结构化函数调用指令 | 工具调用的工程化接口 |
| MCP | Model Context Protocol，工具接入协议 | 工具定义的标准化方案 |
| Reflection | 系统层的自我纠错机制 | 系统设计，不是模型天赋 |
| Multi-Agent | 多 Agent 角色分工与协作 | 单 Agent 瓶颈的针对性解决方案 |
