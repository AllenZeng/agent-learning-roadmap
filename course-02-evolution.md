# 课程二：Agent 核心能力的演进

---

## 课程导言

课程一是让你"看见"Agent——通过产品体验和拆解，你对 Agent 有了感性的认知，知道它长什么样、能做什么、和传统软件有什么不同。

课程二是让你"理解"Agent——把四种核心能力的来龙去脉彻底搞清楚。Tool Use 是怎么从"模型只能说话"变成"模型能做事"的？Planning 为什么不能只是一条路走到底？Reflection 的自我纠错到底靠不靠谱？Multi-Agent 是"人多力量大"还是"三个和尚没水喝"？

这些问题背后的答案，构成了 Agent 设计的底层逻辑。当你理解了这些"为什么"，课程三开始动手实现时，你写的每一行代码都会有清晰的意图——你知道自己在解决什么问题，也知道这个解决方案是从哪里演化而来的。

这就是本课程的定位：**为实践做理论准备**。

**前置**：完成课程一（初识Agent），对 Agent 有感性认知

## 第一章：从概念到范式 —— Agent 不是一夜之间出现的

Agent 不是一夜之间出现的。它的四个核心能力——Tool Use、Planning、Reflection、Multi-Agent——各有各的诞生故事。这些故事串在一起，就是一条清晰的演进脉络：**从"只能想"到"能做事"，从"单步回答"到"多步规划"，从"错了就错了"到"错了能改"，从"单打独斗"到"团队协作"**。

理解这些故事，你就能理解为什么今天的 Agent 是这样设计的——每一个设计选择，都是对某个具体问题的回应。当你自己动手构建 Agent 时，这些故事会告诉你：什么情况下该用什么能力，什么情况下不该用。

### 1.1 Agent 概念的历史渊源

#### 1.1.1 在 LLM 出现之前：AI 领域的 Agent 概念

"Agent"这个词在计算机科学中的历史远比大语言模型（LLM）古老。要理解今天的 AI Agent，我们必须先回到没有 LLM 的年代，看看那时的研究者们是如何思考"智能体"这个问题的。

**强化学习中的 Agent**

在强化学习（Reinforcement Learning, RL）的经典框架中，Agent 是一个与环境交互的决策实体。它的工作方式可以概括为一个循环：**观察环境状态（State）→ 选择行动（Action）→ 接收奖励（Reward）→ 更新策略**。

这个框架最早可以追溯到 20 世纪 50 年代的行为心理学，特别是 Skinner 的操作性条件反射理论。1957 年，Bellman 提出了动态规划中的最优性原理；1989 年，Watkins 提出了 Q-learning 算法。DeepMind 在 2013 年用 DQN 玩 Atari 游戏、2016 年 AlphaGo 击败李世石，都是这种 Agent 形态的高光时刻。

**软件工程中的 Agent**

在软件工程领域，"软件 Agent"的概念在 20 世纪 90 年代兴起。1994 年，Wooldridge 和 Jennings 提出了软件 Agent 的两个核心特征：

- **弱定义（Weak Notion）**：自治性（Autonomy）、社交能力（Social Ability）、反应性（Reactivity）、主动性（Pro-activeness）
- **强定义（Strong Notion）**：在弱定义基础上，增加了信念（Belief）、愿望（Desire）、意图（Intention）等心智状态——这就是著名的 BDI 模型

1995 年，Pattie Maes 在 MIT 媒体实验室开发了"自主 Agent"：能够代表用户执行任务的软件，比如会议调度 Agent、邮件过滤 Agent。这些 Agent 的特点是运行在封闭的、规则明确的环境中。

**Russell & Norvig 的经典分类**

Stuart Russell 和 Peter Norvig 在《人工智能：一种现代方法》（Artificial Intelligence: A Modern Approach）中，将 Agent 分为四种基本类型：

1. **简单反射型 Agent（Simple Reflex Agent）**：基于当前感知直接选择行动，遵循"条件-动作"规则。像一个恒温器——"如果温度低于 20 度，就开启加热"。

2. **基于模型的反射型 Agent（Model-based Reflex Agent）**：维护一个内部状态模型，能够处理部分可观测的环境。因为它"记得"之前发生过什么，所以能做出更合理的决策。

3. **目标驱动 Agent（Goal-based Agent）**：不仅知道当前状态，还知道"目标状态"，并能规划从当前到目标的行动序列。比如导航软件，不仅知道你在哪，还知道你要去哪，并为之规划路线。

4. **效用驱动 Agent（Utility-based Agent）**：在多个可达目标之间做权衡，选择"最优"的那个。它引入了一个"效用函数"来量化不同状态的"好坏程度"。

这四种类型代表了从简单到复杂、从反应到推理的演进路径。

**传统 Agent 的根本痛点**

然而，无论是强化学习 Agent、软件 Agent，还是基于规则的目标驱动 Agent，都存在一个共同的、致命的局限：**它们只能在封闭的、预定义的规则空间内运行**。

传统 Agent 就像下棋程序——它可以下出人类难以企及的妙手，但它永远无法理解"下棋"这件事的意义，也无法在棋盘被掀翻时知道该怎么办。它的智能是被"硬编码"进规则和策略中的，无法迁移到任何其他任务上。

更具体地说，传统 Agent 面临三个核心痛点：

- **规则依赖**：必须由人类专家预先定义所有可能的"感知-行动"映射。遇到规则之外的情况，Agent 要么崩溃，要么做出无意义的反应。
- **泛化能力缺失**：一个会下围棋的 Agent 不会下象棋，一个会调度的 Agent 不会写邮件。每个 Agent 都是为了特定任务定制的。
- **自然语言理解薄弱**：传统 Agent 无法真正理解人类的自然语言指令。即使是设计了 NLP 接口的 Agent，也只能处理有限的、结构化的命令模式。

这三个痛点指向同一个问题：**我们需要的不是一个"更聪明的专用 Agent"，而是一个能够理解任意指令、自主推理、灵活应变的通用决策核心。** 而这个核心，最终以 LLM 的形式到来了。

### 1.2 LLM 带来的范式革命

#### 1.2.1 涌现能力：GPT-3 敲开了新世界的大门

2020 年，OpenAI 发布了 GPT-3。最初，人们的关注点集中在它的文本生成能力上——写文章、翻译、答题。但很快，研究者注意到了一些"奇怪"的现象：

GPT-3 在没有专门训练的情况下，展现出了一定的**少样本推理能力**（Few-shot Reasoning）。你只需要给它几个例子，它就能按照例子的模式完成新的任务。更令人惊讶的是，它似乎能够进行某种形式的"链式推理"——当你说"让我们一步步思考"时，它的答案质量显著提升。

这就是**涌现能力（Emergent Abilities）**——当模型规模超过某个临界点后，突然获得了一些在设计时未曾预料的能力。这些能力不是工程师"塞进去"的，而是从海量数据和参数规模中"长出来"的。

这引发了一个关键思考：**如果一个模型已经具备了通用推理的"种子"，我们有没有可能把它发展成一个真正的自主决策核心？**

#### 1.2.2 ReAct：让模型同时"想"和"做"

2022 年 10 月，Google Research 和普林斯顿大学联合发表了论文《ReAct: Synergizing Reasoning and Acting in Language Models》。这篇论文在 Agent 发展史上具有里程碑意义。

**发表前的问题状态**：当时，研究者已经发现 LLM 可以通过 Chain-of-Thought（思维链）进行逐步推理，也可以在一些特定任务中使用工具。但这两条能力是分离的——要么让模型"推理"（像 CoT 那样），要么让模型"行动"（像使用 Wiki API 那样）。推理和行动之间没有交互：推理无法从行动结果中获取新信息，行动也缺乏推理的指导。

**核心洞察**：ReAct 团队观察到，人类解决复杂问题的方式从来不是单纯的"推理"或单纯的"行动"——而是两者的交织。你遇到一个问题，先思考一下，然后尝试查资料，发现新信息后调整思路，再尝试另一个方向……推理和行动互为输入输出，形成一个动态循环。

**ReAct 的工作方式**：

ReAct 将任务解决过程建模为"思考-行动-观察"的交替序列：

```
Thought: 我需要知道 2024 年奥运会在哪个城市举办
Action: Search["2024 Summer Olympics host city"]
Observation: 2024 年夏季奥运会将在法国巴黎举办
Thought: 现在我知道了是巴黎，我需要知道巴黎现在的天气
Action: Weather[Paris]
Observation: 巴黎今天晴，15-22°C
Thought: 我已经获得了所有需要的信息，可以回答了
Answer: 2024 年奥运会在巴黎举办，今天天气晴朗……
```

这个看似简单的交替模式，实际上实现了 LLM 能力的质变：**模型不是在"孤岛"上推理，而是在与外部世界的持续交互中推理**。每一次观察都可能改变它的思考方向，每一次思考都可能触发新的行动。

**带来的变化**：ReAct 模式将 LLM 从一个"回答机器"变成了一个"问题解决器"。在 HotpotQA（多跳问答）和 Fever（事实验证）等基准测试上，ReAct 显著优于单纯的 CoT 或单纯的 Action 方法。

#### 1.2.3 Toolformer：让模型自己学会用工具

在 ReAct 之后，2023 年 2 月，Meta AI 发表了 Toolformer，从另一个角度推动了 Agent 的发展。

**核心问题**：ReAct 方法虽然在思路上有突破，但它依赖精心设计的提示词（Prompt）来引导模型调用工具。这意味着：工具的调用逻辑是"外挂"的，模型本身并没有内化"使用工具"这一能力。而且，不同的工具需要不同的提示词设计，难以规模化。

**Toolformer 的创新思路**：与其教模型怎么用工具，不如让模型**自己学会**什么时候该用工具。具体做法是：

1. 在一个大规模文本语料中，自动标注出"如果在这里调用 API，结果会更好"的位置
2. 将这些标注转换为自然语言格式的训练样本
3. 用这些样本对预训练 LLM 进行微调
4. 微调后的模型在生成文本时，会自动在合适的位置插入 API 调用，并用返回结果继续生成

**思考路径**：Toolformer 团队的关键洞察是——"使用工具"本质上是一种语言模式。既然模型是通过学习语言模式来获得能力的，那为什么不让"使用工具"也成为语言模式的一部分呢？这个思路非常优雅，它避免了对模型架构的任何修改，纯粹通过数据驱动的方式让模型获得了工具使用能力。

**带来的变化**：Toolformer 证明了 LLM 可以**自主决定**何时调用工具，而不需要外部的提示词工程来"强迫"它。这为后来的 Function Calling 和 Tool Use 能力奠定了思想基础。

#### 1.2.4 关键转折：从"说话"到"做事"

把 ReAct 和 Toolformer 放在一起看，我们会发现 2022-2023 年间发生了一次根本性的认知转变：

| 维度 | 旧范式（2022 年之前） | 新范式（2022 年后） |
|------|---------------------|-------------------|
| LLM 的角色 | 聊天机器人 / 文本生成器 | 自主决策核心 |
| 能力边界 | 语言空间内的推理 | 语言+行动空间的联合推理 |
| 与外部世界的关系 | 隔离的、被动的 | 交互的、主动的 |
| 任务类型 | 单轮问答、生成 | 多步规划、工具调用、环境交互 |
| 评估标准 | 文本质量 | 任务完成率、决策最优性 |

这个转变不只是技术上的，更是哲学上的。它意味着我们开始用另一种眼光看待 LLM——不再把它当作一个"更聪明的聊天机器人"，而是把它看作一个**通用推理引擎**，一个能够像人类一样"感知-思考-行动"的自主实体。

正是这次认知转变，催生了 Agent 的四大核心能力。接下来，我们逐一追溯它们的演进历程。

## 第二章：四大核心能力的演进

### 2.1 Tool Use / Function Calling —— 让 LLM 能够"动手"

#### 2.1.1 之前的状态：只能"说"，不能"做"

在 Function Calling 出现之前，LLM 本质上是一个"只能说话的人"。它可以告诉你"应该怎么做"，但它自己做不到——它无法查询实时数据、无法发送邮件、无法操作数据库、无法控制任何外部系统。

这导致了什么？一个非常尴尬的场景：你问模型"天气怎么样"，它礼貌地回答你"我无法获取实时天气信息，建议您查看天气应用"。模型有知识（知道天气可以被查询），有推理能力（知道你这个问题的意图），但没有行动力。

这种"思行分离"的状态是限制 LLM 应用场景的最大瓶颈。企业想用它做客服，但它查不到订单状态；想用它做助手，但它不能帮你订机票；想用它做分析，但它无法读取实时数据。

#### 2.1.2 第一次全民体验：ChatGPT Plugins（2023 年 3 月）

2023 年 3 月，OpenAI 发布了 ChatGPT Plugins。这是大众用户第一次直观体验到"LLM 使用工具"意味着什么——ChatGPT 可以帮你订餐、查机票、搜索学术论文、生成图表。

Plugins 背后的技术原理并不复杂：每个 Plugin 暴露一组 API 端点和对应的描述，ChatGPT 根据用户的问题，判断应该调用哪个 Plugin 的哪个端点，并从对话上下文中提取参数。

但它的历史意义在于：**它在心理层面打破了"LLM 只是聊天工具"的认知边界**。用户第一次看到，同一个对话界面里，AI 不仅在对答，还在真正地"做事"。

Plugins 也有明显的问题：依赖手动选择安装，用户需要事先知道有哪些 Plugin 可用；Plugin 之间的协作困难（如果你想先搜索航班再预订，需要两个不同的 Plugin 配合）；缺乏统一的工具描述标准。

#### 2.1.3 标准化时刻：OpenAI Function Calling（2023 年 6 月）

2023 年 6 月，OpenAI 在 GPT-4 和 GPT-3.5-turbo 上原生支持了 Function Calling。这不是一个新功能，而是一次**接口的标准化**。

Function Calling 的核心机制：

1. 开发者定义函数的 JSON Schema（函数名、描述、参数列表及类型）
2. 把这些定义放在 API 请求中发给模型
3. 模型决定"是否要调用工具"以及"调用哪个工具、什么参数"
4. 模型返回一个结构化的 function_call 对象，而不是纯文本
5. 开发者在实际环境中执行这个函数
6. 把执行结果传回给模型
7. 模型根据结果继续推理或给出最终答案

**这里有一个值得仔细思考的设计问题**：既然模型只是返回一个 JSON 说要调用什么函数，那为什么要让开发者去真正执行这个函数，而不是让模型直接"假装"执行？

答案在于**真实性和安全性**。模型的输出是不可靠的——它可能算错一个数学题，可能编造一个不存在的订单号。让模型"假装"执行等于放弃了所有外部验证的机会。而把执行权留在开发者手中，意味着：
- 数据库查询的真实结果总是准确的
- 对危险操作（如转账、删除）可以有权限控制
- 每次工具调用都在开发者的监控和审计之下

**设计哲学**：Function Calling 的设计体现了一个关键取舍——**模型负责决策（what to do），环境负责执行（how to do）**。这种分工让 LLM 专注于它擅长的语义理解和决策，把精确执行交给确定的代码。这个设计哲学后来被几乎所有的 Agent 框架所继承。

#### 2.1.4 工具定义标准的演进：从 JSON Schema 到 MCP

随着 Function Calling 的普及，一个新问题浮现了：每个模型提供商（OpenAI、Anthropic、Google 等）都有自己的工具定义格式。开发者如果同时使用多个模型，就需要维护多套工具描述。这就像每个品牌的手机都有自己的充电接口——对生态的发展是巨大的阻力。

**JSON Schema** 成为了事实上的通用标准——大多数提供商都接受以 JSON Schema 格式描述函数的参数。但工具的"接口格式"只是问题的一部分，更大的挑战在于工具的**发现**和**连接**。

2024 年底，Anthropic 发布了 **MCP（Model Context Protocol，模型上下文协议）**，试图解决一个更根本的问题：如何让 Agent 连接到任意数据源和工具，而不需要每次都重新集成？

MCP 的设计灵感来自于 LSP（Language Server Protocol，语言服务器协议）——LSP 让任意编辑器都能支持任意编程语言的智能提示，而 MCP 的目标是让任意 Agent 都能使用任意工具/数据源。MCP 采用客户端-服务器架构，定义了标准化的数据模型：Resources（数据）、Prompts（提示模板）、Tools（可执行函数）。这意味着工具的定义和实现首次有了一个**跨模型的统一标准**。

#### 2.1.5 Tool Use 问题的三个子问题

回到工具使用本身，我们可以将这个问题分解为三个子问题，这有助于我们理解未来的发展方向：

1. **工具选择（Tool Selection）**：当有 100 个工具可用时，Agent 如何判断该用哪个？目前的方案包括语义匹配、分层索引、以及让模型直接做选择（但很贵且受上下文窗口限制）。

2. **参数填充（Parameter Filling）**：Agent 需要从对话上下文和工具描述中"猜"出函数需要的参数。难点在于参数的值可能来自历史对话、可能需要多步推理才能确定、可能缺失需要追问用户。

3. **工具组合（Tool Composition）**：复杂任务往往需要按顺序调用多个工具——先搜索信息，再分析，再生成报告，再发送邮件。这实际上是一个**规划问题**。

**未来方向**：动态工具发现（Agent 在运行时根据任务需求搜索和绑定新工具），以及工具使用经验的跨任务迁移（Agent 记住"上次遇到这类问题时，这个工具组合效果很好"）。

> **一句话总结**：Tool Use 解决的核心问题是——**让 LLM 从"只能给建议"变成"能真正执行"，通过"模型决策、环境执行"的分工，在保持灵活性的同时确保了安全性和可靠性。**

### 2.2 Planning —— 让 Agent 能够"思考在前"

#### 2.2.1 之前的状态：一步到位的回答模式

早期的 LLM 使用模式非常简单：用户提问 → 模型直接回答。这种模式对于简单问题（"什么是光合作用"）是有效的，但对于需要多步推理的复杂问题（"规划一个去京都的 5 天行程，预算 1 万元"），直接用"直觉"生成答案的质量很差。

用一个比喻来说：这就像让一个学生一听到题目就立刻写出最终答案，中间不允许打草稿、不允许查资料、不允许修改。任何有点难度的数学题在这种模式下都很难做对。

**根本问题**：LLM 的本质是一个"下一个 Token 预测器"，它天然是**逐字生成**的，而不是**整体规划**的。当你让它直接回答时，它没有机会"后退一步"审视整体结构，而是一路向前冲。

#### 2.2.2 Chain-of-Thought（2022 年 1 月）："让我们一步步思考"

2022 年 1 月，Google Research 发表了 Chain-of-Thought（CoT）论文。这篇论文的核心发现简单而深刻：**在提示词中加入"让我们一步步思考"（Let's think step by step），模型在复杂推理任务上的表现显著提升。**

CoT 背后的原理：模型在生成每个 Token 时，前面的所有 Token 都会影响它的注意力分布。如果前面的 Token 中包含"推理过程"——比如"首先，我们需要计算总价 = 单价 x 数量 = 5 x 3 = 15"——那么模型在生成最终答案时，就能"看到"这些中间推理步骤，从而使最终答案更可能正确。

换句话说，CoT 不是让模型"变得更聪明"，而是**改变了模型生成答案时的上下文窗口**——给予它更多的"思考空间"。

**Zero-shot CoT** 很快被发现：你不一定需要给模型展示解题样例（Few-shot），有时候只需要在 Prompt 末尾加上一句"Let's think step by step"，就能触发模型进行步骤式推理。这说明"逐步推理"这种模式是模型在预训练过程中从海量文本中学到的，CoT Prompt 只是在"唤醒"这个能力。

**局限性**：CoT 让模型"一步步思考"，但每一步往哪个方向走，模型没有太多选择——它基本上是按"最可能的路径"走下去的。如果第一步就走偏了，剩下的步骤只会让错误越来越大。

#### 2.2.3 Tree-of-Thoughts（2023 年 5 月）：探索多条推理路径

Tree-of-Thoughts（ToT）是 CoT 的演进，由 Shunyu Yao（同时也是 ReAct 的作者之一）和普林斯顿团队在 2023 年 5 月提出。

**核心改进**：CoT 沿一条路径走到底，ToT 则同时探索多条可能的推理路径，就像下棋时"多想几步"不同走法。

**工作机制**：
1. **生成候选步骤**：在每一个推理节点，生成多个可能的"下一步"
2. **评估**：对每个候选步骤进行打分，判断它是否有希望通向正确答案
3. **搜索策略**：可以选择广度优先（BFS）或深度优先（DFS）
4. **回溯**：如果发现某条路径走不通，可以回到之前的节点，换一条路

ToT 在需要系统性探索的问题上表现出色，比如"24 点游戏"和创意写作。但它的代价也很明显：需要多次调用 LLM，Token 消耗是 CoT 的 10 倍以上，实际应用中的性价比值得商榷。

#### 2.2.4 Plan-and-Solve：人类式的"先计划，后执行"

Plan-and-Solve 策略更进一步：在执行任何具体步骤之前，先制定一个**完整的计划**，然后按计划逐步执行。

**为什么先做计划比逐步探索更好？**

想象你要写一篇论文。有两种做法：
- **逐步探索**：先不管整体结构，从"看起来最重要"的部分开始写，写完后看看下一步该写什么
- **先计划**：先花时间梳理出整体大纲，确认逻辑自洽后，再开始动笔

有经验的写作者都知道，后者效率高得多。因为在整体框架下，你知道每一部分的写作边界和目标，不会写着写着就跑偏。更重要的是，**计划的成本远低于执行的成本**——修改大纲花 10 分钟，修改已经写好的草稿可能要花几小时。

Plan-and-Solve 把这个直觉应用到了 LLM 上：先分析任务，生成一个分步计划；再按计划逐步执行，每一步根据上一步的结果调整后续步骤（但整体框架不变）。

#### 2.2.5 关键矛盾：规划能力与执行能力

在 Agent 的规划问题上，有一个重要的但少有人讨论的矛盾：**模型的规划能力和执行能力往往是此消彼长的。**

为什么？因为"制定一个好计划"和"执行好每个步骤"需要不同的能力——前者需要全局视野和结构思维，后者需要细节专注和精确性。在 LLM 中，这两者共享同一个参数空间，提升一方的能力有时会损害另一方。一个过度规划的模型可能会把简单任务复杂化，而一个不善于规划的模型可能会在复杂任务中迷失方向。

**未来方向**：
- **分层规划**：高层次的"元规划器"负责分解大任务，低层次的"执行器"负责具体完成每个子任务
- **动态重规划**：在执行过程中监测"计划是否还适用"，如果发现偏离预期，自动触发重新规划
- **规划能力的分离训练**：未来可能会出现专门为规划优化的模型，与执行模型配合使用

> **一句话总结**：Planning 解决的核心问题是——**让 LLM 从"一步到位的直觉回答"变成"先想清楚再动手的系统性思考"，通过引入中间推理步骤来提升复杂任务的解决质量。**

### 2.3 Reflection —— 让 Agent 能够"自我纠错"

#### 2.3.1 之前的状态：模型输出后无法自我修正

在 Reflection 技术出现之前，LLM 的文本生成是**单向的、不可逆的**。一旦模型输出了内容，就再也没有机会"收回"或"修改"——除非人类介入，指出错误，并要求重新生成。

这导致了一个很常见的问题：即使模型"知道"自己的输出有问题，也无法纠正。比如，你问模型一个复杂计算，它给出一个错误答案，但如果你追问"你确定吗？再检查一下"，它可能会在第二次回答时做对——这说明它其实有能力判断对错，只是缺少一个"触发自我检查"的机制。

**痛点**：我们需要一种方法，让模型能够在输出过程中**自我监控**，发现问题后**自我纠正**，而不依赖人类的外部反馈。这在 Agent 场景下尤为重要——没有人会守在旁边检查 Agent 的每一步输出，Agent 必须具备某种形式的"自知之明"。

#### 2.3.2 Self-Refine（2023 年 3 月）：迭代式自我改进

2023 年 3 月，卡内基梅隆大学和 Allen AI 发表了 Self-Refine 论文。它的核心思想可以概括为：**用同一个 LLM 既当"作者"又当"编辑"**。

Self-Refine 的工作流程是一个迭代循环：

1. 生成初始输出（作者角色）
2. 对初始输出进行批评，指出问题（评论家角色）
3. 根据批评意见改进输出（作者+编辑角色）
4. 重复 2-3，直到输出质量满足要求或达到最大迭代次数

**设计巧妙之处在于角色切换**。同一个模型在 Prompt 的引导下，可以扮演完全不同的认知角色。"作为作者"时，它专注于生成内容；"作为评论家"时，它切换到批判性思维模式。这种角色的切换比"请检查你的答案"更有效，因为它明确分开了"创造"和"批判"这两种不同的心理模式——而这两种模式混在一起正是导致"自我审查"困难的原因。

**效果和局限**：Self-Refine 在对话生成、代码编写、数学推理等任务上都有显著提升（通常是 5-15% 的准确性提升）。但它也有局限：如果模型的初始输出就包含"系统性错误"（而非偶然失误），那么自我批评可能只是强化了同一种错误的逻辑——毕竟批评者和作者用的是同一个模型，共享同样的知识盲区。

#### 2.3.3 Reflexion（2023 年 6 月）：在 Agent 循环中嵌入反思

Reflexion 由 Noah Shinn 等人在 2023 年 6 月发表。它将反思机制从一个"独立的迭代过程"升级为 **Agent 决策循环的内置组件**。

**工作机制**：Reflexion 的 Agent 循环在传统的"行动-观察"之间插入了一个**反思步骤**：

```
任务 → 推理 → 行动 → 观察 → 反思 → 记忆 → 推理 → 行动 → ...
```

其中"反思"步骤会分析上一次行动的结果：
- 成功了？——为什么成功的？有什么经验可以记住？
- 失败了？——为什么失败？如果是"幻觉"导致的错误，那下次应该怎么做？
- 部分成功？——哪些部分做对了，哪些需要调整策略？

反思的结果不是一次性使用的，而是被存储到**情景记忆（Episodic Memory）**或**语义记忆（Semantic Memory）**中，供未来的推理步骤参考。这意味着 Agent 会从自己的错误中学习，变得越来越好。

**设计灵感**：Reflexion 深受人类"元认知（Metacognition）"研究的启发。元认知的核心是"对自己的认知过程进行监控和调节"——你不会只是机械地完成任务，而是会不时退一步想："我做得怎么样？这个方法对吗？是不是该换个思路？"更重要的是，Reflexion 引入了**长期记忆**——之前的反思经验不会被丢弃，而是成为 Agent"经验库"的一部分。

#### 2.3.4 核心机制：让 Agent"知道自己不知道"

Reflection 技术的深层本质是什么？它是一个更根本的问题的外显——**如何让 Agent 具备"知道自己不知道"的能力？**

在认知心理学中，这被称为"元认知监控"（Metacognitive Monitoring）。对人类来说，我们天然有一种"知道感"（Feeling of Knowing）——听到一个问题时，我们能快速判断自己"知道"还是"不知道"。即使不能立刻答对，我们也大概知道自己答得对不对。

而 LLM 缺少这种能力。LLM 的逐 Token 生成机制让它"不知道自己在说什么"——它只是在生成概率最高的文本序列，并没有任何关于"自己说的话是对是错"的元认知。

Reflection 类技术正在尝试弥补这个缺陷。通过让模型对自身输出进行结构化评估，它实际上在建立一个**元认知层**——这个层监视着"下层"的推理和行动，在检测到异常时触发修正。

#### 2.3.5 反思的代价与未来方向

反思并不总是有益的。过度反思会导致：
- **分析瘫痪**：Agent 陷入"行动→反思→发现小问题→修正→再反思→又发现不同问题→再修正……"的循环，永远无法完成任务
- **Token 浪费**：每一次反思都是一次（或多次）LLM 调用。如果 50% 的 Token 都花在"反思"而非"行动"上，效率就非常低下

**未来方向**：关键不是"多反思"，而是"在正确的时间反思正确的问题"：
- 自适应反思频率：简单任务少反思或不反思，复杂任务增加反思频率
- 反思触发条件：不等到出现了明显失败再反思，而是根据"不确定性信号"（如模型输出的内部置信度、多个推理路径的一致性程度）来预先触发反思
- 结构化反思模板：引导模型按照固定维度检查（正确性、完整性、效率、安全性），而非自由发挥

> **一句话总结**：Reflection 解决的核心问题是——**让 Agent 从"错了也不知道"变成"能发现错误并自我纠正"，通过建立元认知层，使 Agent 具备人类式的"自知之明"。**

### 2.4 Multi-Agent —— 让多个 Agent 像团队一样协作

#### 2.4.1 之前的状态：单 Agent 的能力天花板

单 Agent 系统的能力边界很快暴露出来。即使一个 Agent 同时具备工具使用、规划和反思能力，它也面临几个根本限制：

- **上下文窗口有限**：一个 Agent 处理复杂任务时，需要把任务描述、工具列表、历史交互、反思记录全部放进上下文窗口。当任务非常复杂时，窗口会被塞满，导致"遗忘"问题
- **能力维度单一**：一个模型（一个参数空间）在规划和执行、创造和批判、分析和综合之间需要做 Trade-off。让同一个模型同时擅长所有这些维度是很难的
- **缺乏并行性**：单 Agent 一次只能做一件事。而很多大型任务中，多个子任务是可以并行的

#### 2.4.2 AutoGen（2023 年 8 月）和 CrewAI（2024 年）：多 Agent 框架的兴起

**AutoGen** 由微软研究院于 2023 年 8 月发布。它的核心设计是**通过多 Agent 对话来协作完成任务**。

在 AutoGen 中，你可以定义多个 Agent，每个有不同的角色和能力。它们通过一个**对话管道**进行通信——一个 Agent 发送消息，另一个 Agent 回复，形成多轮对话。完成一个任务的过程，就是这些 Agent 在对话中逐步达成共识、协同推进的过程。

典型的 AutoGen 角色配置：
- **UserProxyAgent**：代表人类用户，可以执行代码，在必要时请求人类输入
- **AssistantAgent**：基于 LLM 的通用助手，负责推理、生成、回答问题
- **多个专用 Agent**：分别负责不同的子任务（检索、分析、编程等）

**CrewAI** 在 2024 年初发布，借鉴了"团队协作"的隐喻。它的核心理念是 Role-based Collaboration（基于角色的协作）：
- **定义角色（Role）**：每个 Agent 有明确的角色、目标和背景故事
- **分配任务（Task）**：任务被分配给有匹配角色的 Agent，可以有依赖关系
- **工具绑定**：每个 Agent 绑定一组特定工具，与它的角色匹配
- **协作流程**：定义 Agent 之间的协作模式——顺序执行还是并行执行

**思考过程**：Multi-Agent 框架的设计者们在思考一个根本问题：**人类社会的知识工作是如何组织的？** 答案是"分工协作"——不是每个人都什么都会，而是让擅长写的人写，擅长算的人算，擅长审的人审，然后把各自的产出组装成最终结果。

这种分工解决了单 Agent 的上下文窗口问题（每个 Agent 只需要关心自己的子任务），也解决了能力维度的冲突（不同的 Agent 可以使用不同的模型或不同的系统提示来强调不同的能力）。但挑战也非常明显：**Agent 之间的沟通效率和任务分配的公平性。**

#### 2.4.3 角色分工的经典模式

在 Multi-Agent 系统中，下面几种角色分工模式最为常见：

**Planner + Executor + Reviewer 三元组**：

| 角色 | 职责 | 需要的能力 |
|------|------|-----------|
| Planner | 分解任务为子任务，制定执行计划 | 结构思维、全局视野 |
| Executor | 按计划逐步执行每个子任务 | 执行精度、工具使用能力 |
| Reviewer | 审核执行结果，发现问题提出修改意见 | 批判性思维、细节关注 |

这个三元组将规划、执行和质量的关注点分离，避免了"既当运动员又当裁判"的认知冲突。

**Manager + Workers 层级模式**：Manager 接收任务，分解后分发给 Worker；Worker 完成自己的部分后汇报给 Manager；Manager 整合结果，决定是提交最终结果还是让某个 Worker 返工。

**Debate 辩论模式**：多个 Agent 独立解决同一个问题，各自展示方案和推理过程，互相质疑和辩论，最终通过投票或仲裁达成一致。辩论模式在需要"最优解"的问题上表现特别好，因为多个视角的碰撞更容易发现单一视角的盲点。

#### 2.4.4 核心挑战：通信效率、任务分配、共识达成

Multi-Agent 系统不是"人多力量大"那么简单。当 Agent 数量增加时，新的问题随之而来：

**通信效率**：如果有 N 个 Agent，它们之间的通信链路是 O(N^2) 级别的。如果不加设计，Agent 之间会产生大量的"闲聊"。一个好的 Multi-Agent 系统需要**高效的通信协议**：什么时候该沟通？沟通什么内容？以什么格式？谁对谁说话？

**任务分配的公平性**：当一个复杂任务需要多个 Agent 时，谁来决定谁做什么？如果由中央"调度器"分配，可能出现负载不均。如果在 Agent 之间进行协商，协商本身也需要时间和 Token。

**共识达成**：当不同 Agent 给出矛盾的建议时，应该信谁？简单的投票（少数服从多数）在复杂决策中并不总是最优的——有时候少数派掌握关键信息。如何在不牺牲效率的前提下，实现更智能的共识机制？

#### 2.4.5 未来：从固定角色到动态角色分配

当前的 Multi-Agent 系统大多数是**预定义角色**：在设计阶段就确定了哪个 Agent 负责什么。这带来了灵活性的损失——任务的性质是多样的，固定的角色分工很难总是最优。

**动态角色分配**是未来的方向：
- Agent 在任务开始时根据自身能力和任务需求"竞聘"角色
- 当一个 Agent 完成自己的任务后，可以"转岗"去帮助其他 Agent
- 系统能根据任务的实时进展自动调整角色配置——发现瓶颈在哪里，就增派资源

这种动态性让 Multi-Agent 系统更像是"自组织的团队"而非"固定的流水线"。当然，它也带来了新的技术挑战：如何让 Agent 理解自己的能力和局限（自我认知），以便在角色竞聘时做出合理选择？

> **一句话总结**：Multi-Agent 解决的核心问题是——**单 Agent 的能力天花板，通过"分工协作"打破上下文窗口、能力维度和并行性的限制，但随之而来的是通信效率和共识达成的新挑战。**

## 代码阅读过渡：从理论到实践的桥梁

在进入课程三之前，我们做一件事：**读一段真实的 Agent 源码**。这不是要你现在就理解每一行，而是让你带着课程二学到的能力概念去看——"这段代码对应的是哪种能力？为什么这样设计？"

### 推荐阅读：smolagents 的核心循环

[HuggingFace smolagents](https://github.com/huggingface/smolagents) 的核心 Agent 循环大约 200 行，是目前最简洁的 ReAct Agent 实现之一。它的特点是 **code-as-action**——Agent 不输出 JSON，而是直接生成 Python 代码作为"行动"。这种设计让工具调用变成了自然的代码执行。

**带着这些问题去读源码**：

1. **找到 ReAct 循环**：Thought → Action → Observation 三个环节在代码中分别对应什么？
2. **找到 Tool Use 能力**：工具是如何定义、注册、被调用的？
3. **找到 Planning 的影子**：Agent 如何决定"下一步做什么"？
4. **找到 Reflection 的痕迹**：如果代码执行出错了，Agent 如何处理？
5. **找到停止条件**：什么情况下 Agent 认为"任务完成了"？

**阅读方法**：

1. 先看 `README.md` 了解项目概况（5 分钟）
2. 找到核心 Agent 类（`CodeAgent` 或类似名称），定位 `run()` 方法（10 分钟）
3. 对照课程二的能力概念，在源码旁做标注（15 分钟）
4. 写下你的理解和疑问，作为课程三动手实现的参考

> 如果你读的是 Python 版本，重点关注 `src/smolagents/agents.py` 中的 `run()` 方法和 `_step()` 方法。

读完源码后，你应该能回答一个核心问题：**这个 Agent 的"思考-行动"循环和你预想的有什么不同？** 把这个答案留到课程三——当你在课程三中实现自己的 Agent 时，对比会非常有意思。
