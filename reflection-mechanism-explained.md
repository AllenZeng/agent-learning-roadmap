# Reflection 机制详解：LLM 如何对自身输出进行验证和评估

---

## 问题的本质

LLM 是**自回归解码**：给定上文 $x_{1:t-1}$，模型预测下一个 token 的概率分布 $P(x_t | x_{1:t-1})$，采样出 $x_t$，然后 $x_t$ 加入上下文，再预测 $x_{t+1}$……整个过程是**单向的、不可回溯的**。Token 一旦生成，就"写入历史"，没有原生的"删除键"或者"等等让我检查一下"的机制。

那么，"Reflection"（反思/自我验证）是怎么实现的？

---

## 关键洞察：Reflection 不是模型架构的能力，而是系统设计的能力

Reflection 不是让 LLM"顿悟"然后说自己错了。它是通过在**模型外部**构建一个反馈循环来模拟反思的。具体有四种主流方案：

---

### 方案一：多轮反馈式（Multi-Pass Feedback）—— 最主流

**核心思路**：把"生成"和"评价"拆成两次独立的 LLM 调用。第一次生成结果，第二次把结果喂回去让模型评价。

```
# 第一轮：生成
Round 1: "请写一个函数，计算斐波那契数列的第 n 项"
Model → def fib(n): return fib(n-1) + fib(n-2)  # 忘了 base case

# 第二轮：评价（把上一次的输出作为输入的一部分）
Round 2: "你写了这段代码：def fib(n): return fib(n-1) + fib(n-2)
         请检查这个代码有什么问题"
Model → "这个代码缺少 base case，会导致无限递归。正确的写法是……"
```

这就是 **Self-Refine（2023）** 的做法：

```
Generate → Critique → Refine → Critique → Refine → ... → Final Output
```

每次迭代中，模型看到的上下文是：`原始任务 + 上次输出 + "请指出问题并改进"`。模型基于这个**扩展后的上下文**重新生成——它不是在"修改"之前的 token，而是在新的生成中基于更多信息做出更好的预测。

**Reflexion（2023）** 在此基础上更进一步，把评价结果持久化为"长期记忆"：

```
Attempt → Failure → Reflect（生成"口头反思"并存储） → 下次尝试时加载反思 → Attempt again
```

例如：
- Attempt 1：Agent 写了一个 SQL 查询，返回空结果
- Reflect：模型被要求分析"为什么查询失败"，输出一条反思记录："这个数据库的表名使用复数形式，应该查询 `users` 而不是 `user`"
- Attempt 2：反思记录被注入 System Prompt，Agent 这次正确地写了 `SELECT * FROM users`

**关键点**：Reflection 发生时，并不是模型在"回看"自己之前的 token——而是**反思的内容作为新的 token 被添加到上下文中**，影响后续 token 的生成。

---

### 方案二：并行采样 + 投票/一致性检查

**核心思路**：既然单次生成不可回溯，那就生成多次，用统计学方法挑最好的。

**Self-Consistency（2022）**：
```
同一个问题，用不同的随机种子生成 5 个答案
→ 对 5 个答案做多数投票（或让 LLM 从中挑选最佳）
→ 选出的答案往往比单次生成的更可靠
```

**Self-CheckGPT（2023）**：
```
同一个问题生成多个回答
→ 观察各回答之间的不一致之处
→ 不一致之处很可能就是幻觉所在
```

这里的"反思"不是单个模型在检查自己的输出，而是利用**生成多样性**来找问题。

---

### 方案三：过程奖励模型（Process Reward Model, PRM）

**核心思路**：训练一个独立的"裁判模型"，它不生成内容，只负责打分。

这来自 OpenAI 在 2023 年发布的 **"Let's Verify Step by Step"** 论文：

- 传统方法（Outcome Reward Model, ORM）：只看最终结果对不对，对了+1，错了0
- PRM：**对每一步都打分**。即使最终结果错了，中间正确的步骤也能得到正向奖励；最终对了但中间有歪打正着的步骤，也会被惩罚

训练过程需要人类标注者对每个中间步骤标注正确/错误。有了 PRM 后，可以为 Beam Search 或 Tree Search 提供**逐步骤的指导信号**——模型在生成每一步时都有"这个方向对不对"的反馈。

```
Step 1: "我们需要先算总人数" → PRM: 0.95 ✓
Step 2: "总人数 = 120 + 80 = 200" → PRM: 0.92 ✓  
Step 3: "然后除以天数 200 / 5 = 40" → PRM: 0.88 ✓
Step 4: "所以每天 40 人" → PRM: 0.91 ✓

vs

Step 1: "我们需要先算总人数" → PRM: 0.95 ✓
Step 2: "总人数 = 120 × 80 = 9600" → PRM: 0.12 ✗  ← 这里就发现问题了
```

### 方案四：模型内部的"自知之明"

一个有意思的研究发现：LLM 在生成 token 时，其**内部隐藏状态**往往已经"知道"即将生成的 token 可能有问题，只是架构限制了它无法"刹车"。

**关键论文**：《Language Models (Mostly) Know What They Know》（2022，Anthropic）

研究发现：如果你在模型生成后问它"你确定吗？"，模型对正确和错误回答的内部表征是不同的——模型内部的某个方向上，区分了"我知道"和"我不确定"。但自回归机制不允许模型在 token 级别回溯。

**后续工作尝试利用这一点**：
- 训练一个"不确定性探针"（Probe），读取隐藏状态来判断当前生成的可靠性
- 当不确定信号超过阈值时，触发外部的 Reflection 循环

## 用一个统一框架来理解

```
┌──────────────────────────────────────────────────────────┐
│                    Agent 系统层面                        │
│                                                         │
│   ┌─────────┐    ┌─────────┐    ┌──────────────┐       │
│   │ 生成模块 │ → │ 评价模块 │ → │ 决策：通过/   │       │
│   │ (LLM)   │   │ (LLM /   │   │ 重试/修正/放弃│       │
│   │         │   │  PRM /   │   │              │       │
│   │         │   │  规则)    │   └──────┬───────┘       │
│   └─────────┘   └─────────┘          │               │
│        ↑                              │               │
│        └──────── 重试（带反馈）────────┘               │
│                                                         │
└──────────────────────────────────────────────────────────┘

关键点：LLM 只负责红框内的单次生成。
Reflection 是蓝框（评价）和绿框（决策）构成的外部循环。
```

## 总结

| 方案 | "反思"发生在哪里 | 谁来做判断 | 代价 |
|------|-----------------|-----------|------|
| Multi-Pass Feedback | LLM 外部（多轮调用） | LLM 自身（换个角色问） | Token 消耗翻倍 |
| Self-Consistency | 多次生成的统计 | 投票或 LLM 裁判 | Token 消耗 × N |
| Process Reward Model | 训练阶段 | 独立训练的裁判模型 | 需要大量人工标注 |
| 内部探针 | 隐藏状态 | 分类器/探针 | 需要训练数据，不够成熟 |

**最核心的一点**：Reflection 不是模型的"超能力"，而是工程师在模型外面搭了一个"检查-反馈-重试"的循环。模型还是那个逐 token 生成的模型，但我们通过**把它的输出再喂给它**、**让它看到自己说了什么**，来模拟反思的过程。这跟你让一个人"重新读一遍你刚才写的东西，看看有没有问题"是同一个道理。
