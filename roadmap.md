# Agent 学习路线图：从最小闭环到产品化系统

> 本文件为课程总纲。项目介绍、学习路径、设计原则、先修要求等请参阅 [README](README.md)。

核心公式：

```
Agent = Prompt（行为定义） + LLM 决策 + 工具/环境交互 + State（状态管理） + 循环控制
```

RAG、Memory、Planning、Reflection、Multi-Agent、MCP、Skill 都很重要，但它们不是同一层级的东西，也不是所有 Agent 应用的必选项。它们应该围绕最小 Agent 闭环展开：有些属于工具机制，有些属于场景增强，有些属于运行时架构，有些属于产品化能力。

---

## 课程设计分层思考

这套路线不是按概念热度排序，而是按学习者理解 Agent 产品开发时会遇到的认知层级来设计。

| 层级 | 对应课程 | 解决的问题 |
| --- | --- | --- |
| 感性认知层 | 课程一 | 先看见真实 Agent 产品，区分 Chatbot、Workflow 和 Agent |
| 范式理解层 | 课程二 | 理解 LLM、工具调用、ReAct、Planning、Reflection、Multi-Agent 为什么出现 |
| 最小闭环层 | 课程三 | 掌握 Agent 的最小组成：LLM 决策、工具/环境交互、状态管理、循环控制 |
| 工具机制层 | 课程四 | 理解工具调用如何变成可选择、可执行、可控、可复用、可审计的机制 |
| 场景增强层 | 课程五 | 判断什么时候需要 RAG / 外部知识接入、Memory、Planning / Workflow Patterns、Reflection、Multi-Agent |
| 运行时工程层 | 课程六 | 把最小闭环工程化为可追踪、可恢复、可评测、可治理的 Harness |
| 产品化运营层 | 课程七 | 解决用户信任、失败恢复、成本延迟、安全合规、指标反馈和持续迭代 |
| 综合实践层 | 课程八 | 通过项目把知识型、工具型、任务执行型 Agent 串联起来 |

```text
基础主线：感性认知 -> 范式演进 -> 最小 Agent 闭环
能力主线：工具机制 -> 场景增强能力
落地主线：运行时工程 -> 产品化实践 -> 项目实战
```

这种分层的目的，是避免把 RAG、Memory、MCP、Skill、Evaluation、Guardrails 等概念都平铺成“核心模块”，而是让学习者先建立最小闭环，再理解哪些能力是为了解决特定场景和工程问题而出现的。

## 课程一：初识 Agent

**定位**：建立感性认知，先看见 Agent 和普通 Chatbot / Workflow 的区别。

本课是感性认知层，不讲实现细节。它为课程二的范式理解和课程三的最小闭环提供产品观察样本。

### 核心问题

- Agent 到底是什么？
- 它和 Chatbot、Workflow 的区别在哪里？
- 一个产品在什么地方体现出 Agent 特征？
- 用户为什么会觉得 Agent 有用、失控、惊喜或不可靠？

### 核心内容

- **Agent 基本理解**：Agent 不是"更聪明的聊天机器人"，而是能够围绕目标持续决策和执行的系统。
- **Agent vs Chatbot vs Workflow**：
  - Chatbot 主要负责对话和回答。
  - Workflow 按固定流程执行。
  - Agent 会根据目标、状态和反馈决定下一步。
- **观察重点**：不要只看产品是否用了 LLM，而要看"谁决定下一步"。
- **主流产品体验**：
  - 国外产品：ChatGPT、Claude / Claude Code、Cursor、Perplexity、Replit Agent。
  - 国内产品：Kimi、豆包、通义千问、腾讯元宝、扣子 Coze。
  - 观察重点：它是否会自己拆解任务、调用工具、生成计划、读写文件、检索资料、跨应用行动；还是主要停留在问答、搜索或固定工作流。
- **产品体验观察框架**：
  - **任务入口**：用户输入的是开放目标、具体指令，还是普通问答？
  - **下一步决策权**：下一步由用户决定、固定流程决定，还是模型根据状态动态决定？
  - **行动能力**：产品能调用哪些工具或外部能力，例如搜索、代码、文件、浏览器、数据分析、工作流节点？
  - **过程可见性**：用户能否看到计划、步骤、工具调用、进度和中间结果？
  - **用户控制权**：产品在哪些节点请求确认，是否支持中断、修改计划、回退或接管？
  - **失败与恢复**：出错后是直接失败、重新尝试、换路径，还是请求用户帮助？
  - **Agent 成色判断**：它更像 Chatbot、Workflow，还是 Agent？哪些地方体现出 Agent 特征？

拆解模板：

| 维度 | 观察问题 | 记录方式 |
|---|---|---|
| 任务入口 | 用户输入的是目标、指令还是问答？ | 写出原始用户任务 |
| 下一步决策权 | 下一步由谁决定？ | 用户 / 固定流程 / 模型 |
| 行动能力 | 产品能调用哪些外部能力？ | 列出工具、数据源或工作流节点 |
| 过程可见性 | 用户能看到哪些步骤？ | 计划、工具调用、进度、中间结果 |
| 用户控制权 | 用户能否确认、中断、修改或接管？ | 标出可介入节点 |
| 失败与恢复 | 出错后如何处理？ | 重试、降级、澄清、失败退出 |
| Agent 成色 | 它更像哪一类产品？ | Chatbot / Workflow / Agent |

## 课程二：Agent 范式演进

**定位**：理解 Agent 为什么会出现，以及它解决了 LLM 的什么问题。

这门课按问题演进来理解 Agent：每一次新方法的出现，都是为了解决前一阶段暴露出来的问题。

本课是范式理解层，解释概念为什么出现，不要求先掌握全部实现。如果学习者更想先动手，可以先学课程三，做完最小闭环后再回来看本课。

### 核心问题

- 传统 Agent 为什么很少进入普通用户的日常应用场景？
- LLM 出现后，Agent 的决策核心发生了什么变化？
- Function Calling / Toolformer / Plugins 解决了什么问题？
- ReAct 为什么适合作为理解 Agent 运行循环的经典框架？
- Planning、Reflection、Multi-Agent 分别是在什么背景下出现的？

### 核心内容

- **传统 Agent 的历史背景**：
  - 强化学习 Agent
  - 软件 Agent
  - BDI 模型
  - Russell & Norvig 的 Agent 类型
  - 传统 Agent 的根本限制：封闭规则空间、泛化能力弱、工程成本高
- **LLM 带来的范式变化**：
  - 从"人工写规则"变成"模型理解目标并生成下一步"
  - 从"固定流程"变成"动态决策"
  - 从"只回答"走向"能调用外部能力"
- **工具调用为什么出现**：
  - 让 LLM 能连接实时信息、私有数据、计算能力和外部系统
  - 让工具调用成为模型能力的一部分
  - 让工具定义、参数生成和结果返回变得工程化
- **ReAct 的意义**：
  - 将推理、行动、观察放入同一个循环，清晰展示了 Agent 如何从"一次性回答"走向"持续执行任务"。
  - 给出了 Agent 最小运行时雏形：判断下一步 → 调用工具 → 接收结果 → 继续或停止。
  - 引出后续工程问题：停止条件、循环控制、工具失败处理、上下文和状态管理。
- **内部推理能力与系统纠错机制的区别**：
  - 模型的思考/推理能力影响 LLM Decision 的质量。
  - Reflection 是系统层设计，通常依赖自检、评审、测试结果或错误反馈来触发纠错。
  - 不应把"模型想得更久"等同于"系统一定更可靠"。
- **Planning 的出现背景**：
  - 复杂任务无法一步完成
  - 需要任务拆解、依赖排序和阶段性目标
- **Reflection 的出现背景**：
  - 模型和工具调用都会失败
  - 系统需要自检、重试、纠错和复盘
- **Multi-Agent 的出现背景**：
  - 单 Agent 面临角色混淆、上下文拥挤、工具数量过多、复杂任务难以并行等问题
  - 多 Agent 通过角色分工、互相审查和并行处理解决部分瓶颈

## 课程三：最小 Agent 闭环

**定位**：实现一个最小但完整的 Agent。

这门课是整条路线的技术起点。学习者必须先理解 Agent 最小闭环，后面所有能力都是围绕这个闭环扩展。

本课只解决"Agent 如何最小可运行"。工具的完整机制放到课程四，RAG / Memory / Planning 等增强能力放到课程五，工程化 Harness 放到课程六。

### 核心问题

- 一个最小 Agent 为什么不能只有 LLM？
- LLM 如何根据上下文和状态做下一步决策？
- 工具/环境交互如何产生反馈，并影响下一轮决策？
- 状态管理如何支撑多步任务，而不是只处理单轮问答？
- 循环控制如何判断继续、停止、重试或失败退出？

### 核心内容

#### 3.1 最小 Agent 组成

```text
Agent = LLM 决策 + 工具/环境交互 + 状态管理 + 循环控制
```

| 组成 | 学什么 | 为什么重要 |
|---|---|---|
| LLM 决策 | 模型选择、System Prompt、结构化输出、下一步判断 | 决定 Agent 如何理解目标并选择下一步 |
| 工具/环境交互 | Tool Use、Function Calling、API / 文件 / 数据库 / 浏览器 / 代码执行 | 决定 Agent 如何获取外部反馈或影响外部环境 |
| 状态管理 | 当前任务、历史消息、工具结果/环境反馈、错误信息 | 决定 Agent 是否能连续工作 |
| 循环控制 | 继续/停止判断、最大步数、超时、重复动作检测、失败退出 | 决定 Agent 如何多步执行且保持可控 |

#### 3.2 最小运行链路

```text
User Goal
    -> Context Assembly
    -> LLM Decision
    -> Tool / Environment Interaction
    -> Observation / Feedback
    -> State Update
    -> Continue or Stop
```

核心公式描述的是 Agent 的组成能力；运行链路描述的是这些能力在每一步如何协作。因此 `Context Assembly` 和 `Observation / Feedback` 不是额外组件，而是状态管理、LLM 决策、工具/环境交互在运行时的连接点。

#### 3.3 运行时循环总览

理解 Agent 时需要一张运行时地图。可以把最小 Agent 看成一个不断推进的控制循环：

```text
User Goal -> Context Assembly -> LLM Decision -> Interaction -> Execution
                ^                                               |
                | read                                          v
       +-----------------------+                    Observation / Feedback
       | State / Memory Store  |                                |
       | task state, history,  |                                v
       | optional persistence  | <----------------------   State Update
       +-----------------------+          write                 |
                                                                v
                                                         Continue or Stop
```

这张图的作用是展示主链路、状态读写和反馈闭环之间的关系。各环节的学习重点在 3.4 展开。

这里要特别区分：最小 Agent 必须有任务状态和循环控制，但不一定需要长期 Memory；RAG 也不是循环的必需节点，而是可以作为一种检索工具参与 Context Assembly 或 Interaction。

最小闭环背后的工程原则是：模型负责理解目标和做下一步判断，确定性基础设施负责工具执行、权限、安全、状态、恢复和观测。不要把可以确定性控制的问题全部交给模型自由发挥。

#### 3.4 沿运行链路展开学习

基于最小运行链路，后续学习不再按概念堆砌，而是逐步回答每个运行环节需要解决的问题。

| 运行环节 | 作用 | 学习内容 |
|---|---|---|
| State / Memory Store | 状态存储位置，不是主链路上的独立步骤；Context Assembly 从中读取，State Update 向其中写入 | 任务状态、历史记录、可选长期记忆、状态读写边界 |
| Context Assembly | 决定哪些信息进入模型上下文 | 用户目标、系统约束、历史消息、任务状态、工具结果/环境反馈、检索结果和记忆召回 |
| LLM Decision | 模型根据当前上下文判断下一步 | System Prompt、结构化输出、下一步判断、工具选择、低置信度处理 |
| Tool / Environment Interaction | 模型选择工具或环境动作，并生成必要参数 | 工具定义、参数生成、环境动作选择、权限判断、用户确认 |
| Execution | 运行时执行工具或环境动作 | 超时、重试、沙箱、异常处理、执行结果规范化 |
| Observation / Feedback | 把工具结果或环境反馈转成模型可理解的信息 | 结果格式化、错误归一化、摘要压缩、关键信息提取 |
| State Update | 更新任务状态、历史记录、错误信息和中间结果 | 状态写入时机、错误记录、中间结果保存、可选长期 Memory 写入 |
| Continue or Stop | 判断继续、停止、重试或失败退出 | 停止条件、最大步数、重复动作检测、失败退出、人工接管 |

## 课程四：工具机制

**定位**：理解 Agent 在运行时如何调用工具，以及如何把工具调用设计成可选择、可执行、可控、可复用、可审计的系统机制。

课程三已经说明了最小 Agent 闭环，这门课继续深入其中的工具调用部分。重点不是"接入更多能力"，而是理解工具调用为什么容易失败，以及系统如何通过工具定义、选择、执行、权限、协议、Skill 和 Human-in-the-loop 来解决这些问题。

本课讲工具机制本身，不讲完整 Runtime 架构，也不讲产品交互。运行时治理放到课程六，用户信任和产品体验放到课程七。

### 核心问题

- Agent 在运行时如何决定是否调用工具？
- 工具调用为什么经常失败？
- 如何让模型知道工具能做什么、什么时候该用？
- 如何让工具执行结果可靠地回到下一轮决策？
- 如何控制高风险工具调用？
- 如何把重复任务沉淀成可复用能力？

### 核心内容

#### 4.1 Agent 如何调用工具

工具调用不是一次孤立的 API 请求，而是 Agent Runtime 中的一段可控流程：

```text
LLM Decision
    -> Tool Selection
    -> Parameter Generation
    -> Permission Check
    -> Tool Execution
    -> Observation / Feedback
    -> State Update
```

每一步都可能失败：模型可能不知道该用哪个工具，参数可能填错，工具可能超时，结果可能太长，高风险动作可能需要用户确认。

| 工具调用中的问题 | 对应机制 | 学什么 |
|---|---|---|
| 模型不知道有哪些工具、什么时候该用 | 工具定义 | 名称、描述、参数 Schema、返回值、工具边界 |
| 模型选错工具或乱填参数 | 工具选择 | 工具路由、参数生成、强制/禁止工具调用、选择失败分类 |
| 工具执行可能失败或结果太长 | 执行与回填 | 参数校验、超时、重试、错误结构化、结果摘要 |
| 工具可能造成真实世界影响 | 权限与安全 | 风险分级、最小权限、审批、审计、回滚 |
| 工具接入方式不统一 | MCP | 工具发现、协议化接入、Client / Server、Tools / Resources / Prompts |
| 多个工具经常组合完成同类任务 | Skill | 工具组合、步骤经验、默认流程、失败处理、复用边界 |
| 模型不应独自决定高风险动作 | Human-in-the-loop | 确认、澄清、审批、接管、回滚、审计记录 |

#### 4.2 工具定义：让模型知道工具是什么

工具定义的目标是让模型知道"这个工具能做什么、什么时候该用、什么时候不该用"。

学习内容：

- 工具名、description、参数 Schema、返回值结构。
- 工具描述如何影响模型选择。
- 参数设计：类型、枚举、必填、默认值、约束。
- 工具粒度：原子工具 vs 组合工具。
- 工具设计原则：原子性、可重试性、可观测性、结构化错误、清晰边界。
- 粒度权衡：工具太细会增加步骤数，工具太粗会降低可控性和可解释性。
- 常见错误：描述太泛、参数模糊、返回值太长、多个工具边界重叠。

#### 4.3 工具选择：让模型知道什么时候用哪个工具

工具选择的核心问题是三件事：什么时候用、用哪个、参数怎么填。

学习内容：

- 模型自主选择、规则路由、混合路由。
- `tool_choice`、强制工具调用、禁止工具调用。
- 多工具候选时的冲突处理。
- 参数生成和参数修正。
- 选择失败类型：该用不用、不该用乱用、选错工具、参数猜错。

#### 4.4 执行与回填：让工具结果进入下一轮决策

工具执行不是模型完成的，而是 Runtime 完成的。模型生成 tool call，Runtime 校验、执行并把结果回填成 Observation / Feedback。

学习内容：

- Runtime 如何接收 tool call 并执行。
- 参数校验、超时、重试、幂等。
- 错误结构化：参数错误、权限错误、网络错误、业务错误。
- 结果压缩：长结果摘要、分页、截断、转资源。
- Observation / Feedback 如何影响下一轮决策。

#### 4.5 权限与安全：让工具调用可控

工具调用可能读写文件、查询数据库、发送消息、修改配置或触发外部动作，因此必须有权限和安全设计。

学习内容：

- 工具风险分级：只读、低风险写入、高风险写入、危险操作。
- 权限模型：用户授权、最小权限、作用域。
- 执行前校验：参数、目标对象、影响范围。
- 审批策略：自动执行、确认后执行、禁止执行。
- Deny-first：默认拒绝，只有明确允许的操作才自动执行。
- 权限冲突处理：更严格的拒绝规则应该覆盖更宽泛的允许规则。
- 渐进式授权：先从少量低风险工具自动执行开始，再随着信任逐步放权。
- 工具安全是子系统，不是几个 if 判断；复杂工具需要解析参数、识别危险模式、审计来源和限制副作用。
- 审计日志：谁发起、模型为什么调用、参数是什么、结果是什么。

#### 4.6 MCP：让工具接入标准化

MCP 是工具接入协议，不是 Agent 架构本身。它解决的是工具如何被发现、描述、调用和复用。

学习内容：

- MCP 解决什么问题：工具发现、参数暴露、资源访问、工具服务复用。
- MCP 和 Function Calling 的关系。
- MCP 的基本对象：Tools / Resources / Prompts。
- MCP Client / Server 的职责。
- 什么时候需要 MCP，什么时候普通函数工具就够了。
- MCP 更偏协议化接入和能力发现；Skill 更偏任务经验、默认流程和失败处理封装。

#### 4.7 Skill：把重复任务沉淀成能力包

Tool 是原子动作，Skill 是可复用能力包。Skill 可以组合多个工具，也可以封装某类任务的步骤经验、默认流程和失败处理。

学习内容：

- Skill 通常包含：说明、适用场景、工具组合、步骤经验、失败处理。
- Skill vs Tool：Tool 偏原子动作，Skill 偏任务能力。
- Skill vs Workflow：Skill 可被 Agent 灵活调用，Workflow 更偏固定流程。
- 好 Skill 的标准：边界清晰、输入输出明确、可组合、可回退。
- 示例：代码审查 Skill、数据分析 Skill、文档生成 Skill。

#### 4.8 Human-in-the-loop：让人参与高风险决策

Human-in-the-loop 不是单独的 Agent 能力，而是工具机制中的控制点。它用于处理高风险、低置信度、权限不足或连续失败的场景。

学习内容：

- 触发条件：高风险动作、低置信度决策、权限不足、连续失败。
- 介入方式：确认、驳回、修改参数、编辑计划、接管、回滚。
- 确认界面应该展示什么：动作、对象、影响范围、可撤销性。
- 人类反馈如何回到 Agent 状态里。
- 审计记录：确认人、确认时间、确认内容、执行结果。
- 产品侧如何让用户理解、确认和接管，详见课程七。

## 课程五：场景增强能力

**定位**：讲 RAG / 外部知识接入、Memory、Planning / Workflow Patterns、Reflection、Multi-Agent，但明确它们不是所有 Agent 都必须具备的"核心模块"，而是按场景选择的增强能力。

本课是场景判断层，不要求把每个能力都学到工程实现深水区。重点是判断什么时候该引入、什么时候不该引入、引入后增加什么复杂度。真正的运行时实现和治理问题放到课程六。

### 核心问题

- 什么情况下需要 RAG / 外部知识接入？
- 什么情况下需要 Memory？
- 什么情况下需要 Planning / Workflow Patterns？
- Reflection 是模型能力还是系统设计？
- Multi-Agent 是高级能力，还是复杂度陷阱？
- 什么情况下应该明确不引入某个增强能力？

### 能力地图

| 能力 | 什么时候需要 | 学什么 |
|---|---|---|
| RAG / 外部知识接入 | 需要外部知识、私有知识库、业务系统数据、长文档问答时 | Embedding、Chunking、Hybrid Search、Rerank、Web Search、数据库/API 查询、引用与事实性 |
| Memory | 需要跨轮、跨会话、跨任务延续时 | 短期记忆、长期记忆、用户偏好、任务状态、记忆更新 |
| Planning / Workflow Patterns | 任务复杂、步骤多、依赖关系强，或流程稳定但局部需要模型决策时 | Plan-and-Execute、任务拆解、动态重规划、Chain、Router、固定流程 + 局部 Agent |
| Reflection | 容易失败、需要自检和重试时 | Self-check、Critique、Retry、错误分类 |
| Multi-Agent | 需要角色分工、并行处理、互相审查时 | Supervisor、Debate、Group Chat、Graph 协作 |

本课的核心不是"会用所有能力"，而是能写出判断说明：

```text
当前场景的问题是什么 -> 哪个增强能力能解决 -> 它增加了什么复杂度 -> 为什么值得或不值得引入
```

### 5.1 RAG / 外部知识接入

RAG 不是 Agent 必需组件，而是外部知识接入的一种方式。外部知识接入还包括 Web Search、数据库查询、业务系统 API、Knowledge Graph 等。

适合引入外部知识接入的场景：

- 问答依赖私有文档。
- 需要引用来源。
- 模型训练数据不覆盖相关知识。
- 知识变化快，需要外部更新。
- 需要查询业务系统、数据库或实时网页。
- 需要降低幻觉。

主要内容：

- Embedding
- 向量数据库
- Chunking
- Metadata
- Hybrid Search
- Rerank
- Query Rewrite
- Web Search / 数据库查询 / 业务 API 查询
- 引用和溯源
- 失败模式：检索不到、检索错、查询错、引用错、模型忽略外部结果

不需要外部知识接入的典型信号：

- 任务主要依赖用户当下输入，不依赖外部资料。
- 需要的是格式转换、归纳、改写、分类，而不是事实查询。
- 外部数据质量低，接入后反而增加噪声。
- 业务风险不允许模型自由引用未经审核的数据源。

### 5.2 Memory：状态延续能力

Memory 不是简单地"存聊天记录"。它解决的是 Agent 在不同时间尺度上的状态延续问题。

常见类型：

- 短期记忆：当前会话上下文。
- 工作记忆：当前任务状态、中间结果、待办事项。
- 长期记忆：用户偏好、历史任务、稳定事实。
- 经验记忆：曾经失败或成功的策略。

需要关注：

- 什么时候写入
- 写入什么
- 如何更新
- 如何遗忘
- 如何避免错误记忆污染后续任务

实现关注点：

- Memory 和 RAG 的区别：RAG 偏外部知识召回，Memory 偏用户、任务和历史状态延续。
- 记忆存储形态：会话内状态、数据库、向量库、文件系统或业务系统记录。
- 记忆更新策略：显式确认后写入、低风险自动写入、定期清理和人工纠错。
- 记忆衰减和冲突处理：旧偏好不应永久覆盖新事实。

不需要 Memory 的典型信号：

- 每次任务都是独立的一次性请求。
- 用户不希望系统长期记住偏好或历史。
- 错误记忆的风险高于个性化收益。
- 任务状态可以由确定性业务系统保存，不需要模型记忆。

### 5.3 Planning / Workflow Patterns：复杂任务组织能力

Planning 适合多步骤、强依赖、需要长期目标管理的任务。Workflow Patterns 适合流程相对稳定，但某些步骤需要模型判断、工具选择或动态处理的任务。

常见模式：

- 一次性计划，然后执行。
- 边执行边重规划。
- 先粗计划，再逐步细化。
- 人类确认计划后执行。
- Chain：固定顺序执行。
- Router：按输入类型或状态分流。
- 固定流程 + 局部 Agent：主流程稳定，局部步骤交给模型动态决策。
- Graph：复杂状态机和多路径流程。

模式选择：

- 步骤稳定、分支少：优先 Chain。
- 输入类型差异大：优先 Router。
- 需要动态工具选择：优先 ReAct Loop。
- 任务复杂且可先拆解：优先 Plan-Execute。
- 状态多、分支多、需要回放：考虑 Graph。

典型风险：

- 计划看起来合理，但不可执行。
- 计划过细，浪费 token。
- 执行和计划脱节。
- 环境变化后不重规划。
- 固定流程过硬，无法处理异常分支。
- Graph 过复杂，调试和评测成本上升。

不需要 Planning / Workflow Patterns 的典型信号：

- 任务一步即可完成，或者两三步固定执行即可完成。
- 用户更希望直接得到结果，而不是先确认计划。
- 计划不可验证，只会增加看似合理的中间文本。
- 流程已经由确定性业务系统稳定承载。

### 5.4 Reflection：失败修正能力

Reflection 不是模型天然自带的可靠能力，而是系统设计出来的纠错机制。

常见用法：

- 执行后自检。
- 失败后分类错误。
- 对输出做批判。
- 根据测试结果重试。
- 对工具调用参数进行修正。
- 将外部验证信号纳入反思，例如测试失败、工具错误、规则校验失败、用户驳回。

典型风险：

- 模型自信地反思错误。
- 反思增加成本但没有提升质量。
- 无限反思导致任务变慢。

设计原则：

- Reflection 应该被外部信号触发，而不是默认每步都反思。
- Reflection 需要重试上限和停止条件。
- 对高风险任务，模型自评不能替代测试、规则校验或人工确认。

不需要 Reflection 的典型信号：

- 任务结果可以由确定性规则直接校验。
- 失败成本很低，简单重跑比反思更便宜。
- 模型没有足够反馈信号判断自己是否错了。
- 延迟和成本比质量提升更重要。

### 5.5 Multi-Agent：角色分工能力

Multi-Agent 不应该被过早引入。它适合在单 Agent 遇到明确瓶颈时使用。

适合场景：

- 需要多个专业角色。
- 需要并行处理多个子任务。
- 需要互相审查。
- 需要模拟讨论或辩论。

常见架构：

- Supervisor
- Group Chat
- Debate
- Graph
- 发布订阅

典型风险：

- 通信成本高。
- 角色边界不清。
- 多个 Agent 互相放大错误。
- Debug 难度上升。

不需要 Multi-Agent 的典型信号：

- 单 Agent 加清晰工具和上下文管理已经能完成任务。
- 多角色只是 Prompt 命名不同，没有真实分工。
- 任务无法并行，多个 Agent 只会增加协调成本。
- 你还没有 trace、评测和失败分类能力，无法调试多 Agent 行为。

需要区分的是：Multi-Agent 偏角色分工和协作；课程六会补充的子代理 / 任务委托更偏运行时架构，用于上下文隔离、独立探索和结果摘要。

## 课程六：Harness 运行时架构深入

**定位**：从"写一个 Agent"进入"设计 Agent Runtime"。

课程三关注最小闭环，课程四关注工具机制，课程五关注场景增强。课程六开始关注工程架构：如何让 Agent 稳定运行、可调试、可评测、可恢复、可治理。

课程三的运行时循环是概念模型，课程六的 Harness 是工程实现。换句话说，Harness 要把 `Context Assembly -> LLM Decision -> Interaction -> Execution -> Observation / Feedback -> State Update -> Stop` 这条循环变成可配置、可追踪、可恢复、可评测的系统。

### 核心问题

- Agent Runtime 由哪些工程环节组成？
- 每个环节最容易出现什么问题？
- 如何通过 Context Engineering、Orchestration、Checkpoint、Evaluation、Guardrails、Observability 解决这些问题？
- Harness 和 Orchestration 的边界是什么？
- 如何处理上下文压缩、会话恢复、任务委托和调试？
- 业界常见框架和平台分别在解决哪类工程问题？

### 核心内容

#### 6.1 运行时工程架构总览

可以把 Harness 理解为 Agent 的运行时引擎。它不是某一个具体框架，而是一组工程职责：驱动模型和工具、控制循环和状态、管理评测和观测。

| 层级 | 职责 |
|---|---|
| 驱动层 | 调用模型、执行工具、访问外部服务 |
| 控制层 | 管理循环、停止条件、重试、分支、状态迁移 |
| 管理层 | 会话、权限、配置、日志、评测、观测 |

对应到运行时工程环节，问题和解决机制可以这样看：

| 工程环节 | 常见问题 | 解决机制 / 业界实践 |
|---|---|---|
| Context Assembly | 上下文太长、遗漏约束、工具结果污染下一轮决策 | Context Engineering、上下文预算、信息分层、结果压缩、按需召回、上下文压缩 |
| Decision / Orchestration | 步骤组织混乱、循环失控、规划和执行脱节 | Chain、Router、ReAct Loop、Plan-Execute、Graph，常见框架如 LangGraph |
| Execution / State | 工具失败后无法恢复，中断后不能继续，状态不可回放 | Checkpoint、会话持久化、状态机、幂等执行、重试、分支执行 |
| Task Delegation | 子任务探索污染主上下文，复杂任务难以并行 | 子代理 / 任务委托、上下文隔离、摘要回传、权限继承 |
| Evaluation | 只看最终答案，无法判断过程是否正确 | 端到端评测、步骤级评测、工具调用评测、LLM-as-Judge，平台如 Braintrust、LangSmith、Ragas |
| Guardrails | 越权调用、Prompt Injection、危险参数、低置信度自动执行 | 输入/输出校验、工具参数校验、权限控制、Human-in-the-loop |
| Observability | 出错后不知道为什么、成本和延迟不可解释 | Trace、Token / Latency / Tool Error 指标、决策归因，平台如 Langfuse、LangSmith、Phoenix |

#### 6.2 Context Engineering：解决上下文组装问题

Context Engineering 不是简单写 Prompt，而是决定哪些信息在什么时候进入上下文。

常见问题：

- 历史消息、工具结果、RAG、Memory 全部塞进上下文，导致噪声过多。
- 关键约束被长上下文淹没。
- 工具结果太长，影响下一轮 LLM Decision。
- 记忆召回不准，反而污染当前任务。

解决思路和实践：

- 按信息类型分层：用户目标、系统约束、任务状态、工具结果、检索结果、错误信息。
- 设定上下文预算，优先保留高密度、高约束、高时效信息。
- 对长工具结果做摘要、截断、分页或转资源。
- RAG 和 Memory 按需召回，不默认注入全部内容。
- 将稳定约束放在稳定位置，避免被后续内容稀释。
- 在上下文接近上限前主动压缩，而不是等到超限后被动截断。
- 明确压缩保留规则：系统约束、用户目标、当前计划、关键工具结果和用户确认记录应优先保留。
- 记录压缩发生的时机和丢失风险，让用户或开发者能理解上下文为什么变化。
- 为连续压缩失败设置熔断条件，避免 Agent 在上下文管理上死循环。

#### 6.3 Orchestration：解决任务组织问题

Orchestration 解决的是任务如何被组织；Harness 负责 Agent 如何稳定运行。两者相关，但不等价。

常见问题：

- 所有任务都交给 ReAct Loop，导致步骤不稳定。
- 复杂任务没有计划，执行过程容易漂移。
- 固定 Workflow 太硬，遇到异常分支无法处理。
- 多路径任务状态复杂，难以调试。

解决思路和实践：

- Chain：适合固定步骤。
- Router：适合按输入类型或状态分流。
- ReAct Loop：适合需要动态工具选择和反馈修正的任务。
- Plan-Execute：适合先规划、再执行、再修正的复杂任务。
- Graph：适合状态机、多路径流程和可回放执行，典型框架是 LangGraph。

#### 6.4 Checkpoint、会话持久化与恢复：解决中断和失败恢复问题

Agent 任务经常是多步骤、长耗时、可中断的。没有 Checkpoint，失败后只能重跑，用户也很难接管。

常见问题：

- 工具执行到一半失败，任务状态丢失。
- 用户中断后无法恢复。
- 会话恢复后权限、上下文和用户意图边界不清。
- 重试导致重复写入、重复发送、重复扣费。
- 无法回放某次失败过程。

解决思路和实践：

- 保存关键状态：当前目标、计划、已完成步骤、工具结果、错误信息、用户确认记录。
- 对写操作做幂等设计。
- 支持任务恢复、回放和分支执行。
- 在关键节点设置 Checkpoint，而不是每个 token 都保存。
- 区分任务状态、会话历史、长期 Memory 和审计日志。
- 对会话记录采用 append-only 思路，优先保证可回放和可审计。
- Resume 时重新确认高风险权限，不默认继承上一轮授权。

#### 6.5 子代理 / 任务委托：解决上下文隔离和独立探索问题

这里的子代理不是课程五的传统 Multi-Agent 协作，而是一种运行时架构模式。它更像把某个子任务委托给隔离的执行单元，完成后只把摘要、结论或结构化结果返回主 Agent。

常见问题：

- 主 Agent 需要搜索大量文件或资料，工具输出污染主上下文。
- 一个复杂任务中有多个可独立探索的分支。
- 需要不同任务使用不同工具集或系统提示。
- 主 Agent 不应该继承子任务中的全部中间噪声。

解决思路和实践：

- 子代理只拿到完成子任务所需的最小上下文。
- 子代理权限不高于父 Agent，敏感操作仍需确认。
- 子代理返回摘要、证据和结构化结果，而不是完整过程日志。
- 设置委托深度和并发上限，避免递归调用失控。
- 用独立 trace 记录子代理过程，主 trace 只保留关键结论。

#### 6.6 Evaluation：解决效果不可验证问题

Agent 评测不能只看最终回答，还要看过程是否正确。

常见问题：

- 最终答案看起来对，但工具选择和中间步骤是错的。
- 没有固定任务集，无法判断改动是否变好。
- LLM-as-Judge 不稳定，评测结果难以复现。
- 只评测单轮回答，忽略多步任务完成率。

解决思路和实践：

- 构建任务集：覆盖成功路径、失败路径、边界条件和权限场景。
- 同时评测最终结果和步骤过程。
- 记录工具选择、参数正确性、步骤数、权限遵守、失败恢复。
- LLM-as-Judge 要配合规则、人工抽检和回归样例。
- 把线上失败案例纳入回归测试集。
- 区分产品指标和工程指标：任务完成率是结果指标，工具选择准确率、参数正确率、恢复成功率是诊断指标。
- 使用 Braintrust、LangSmith、Ragas 等平台管理评测和回归。

#### 6.7 Guardrails：解决越界和风险控制问题

Guardrails 应该嵌入运行时，而不是上线前补一个过滤器。

常见问题：

- 用户输入包含 Prompt Injection。
- 模型计划越权或调用危险工具。
- 工具参数指向错误对象或影响范围过大。
- 低置信度情况下继续自动执行。
- 输出包含隐私、合规或事实性问题。

解决思路和实践：

- 输入阶段：防 Prompt Injection、识别恶意请求。
- 决策阶段：限制越权计划。
- 工具调用前：校验参数、权限、风险等级。
- 工具调用后：校验结果是否可信。
- 输出阶段：校验事实性、隐私、合规。
- Human-in-the-loop：高风险动作前确认，低置信度决策时请求澄清，异常状态下允许人工接管。
- Prompt Injection 防御需要分层处理：用户输入、外部文件、工具结果、RAG 内容和最终输出都可能携带注入内容。
- 安全判断尽量依赖结构化参数和权限上下文，不只依赖模型生成的自然语言解释。

#### 6.8 Observability 与调试方法：解决不可调试问题

Agent 的可观测性要能解释"为什么这么做"，而不只是记录最终输出。

常见问题：

- 出错后不知道是哪一步失败。
- 成本和延迟无法拆解。
- 工具错误、模型错误、上下文错误混在一起。
- 用户在哪些节点频繁打断无法分析。
- 有 trace 但不知道如何从 trace 定位根因。

解决思路和实践：

- 记录完整 Trace：输入、上下文、模型输出、工具调用、Observation / Feedback、状态更新。
- 拆解核心指标：Token、Latency、Tool Error、Retry、Step Count、User Intervention。
- 对失败进行分类：工具选择、参数、权限、上下文、模型判断、外部服务。
- 从失败步骤反向定位：先看工具结果，再看参数，再看上下文，再看模型决策。
- 用同一状态快照回放，比较 Prompt、工具描述、路由策略或模型版本变化。
- 将最小复现场景沉淀为评测样例。
- 用 Langfuse、LangSmith、Phoenix 等工具做 trace、调试和决策归因。

进阶关注：

- Hooks / 扩展机制：在工具调用前后、权限请求、会话开始/结束、压缩前后等生命周期节点插入自定义逻辑。
- Streaming Architecture：把流式输出从 UX 能力提升为运行时能力，支持进度展示、中断、实时转向和工具结果交织展示。

## 课程七：Agent 产品化实践

**定位**：解决 Agent 从"技术能跑"到"用户愿意用、业务愿意上线、团队能够持续运营"之间的差距。

课程六解决 Runtime 是否稳定，课程七解决产品是否可信、业务是否可衡量、风险是否可治理、团队是否能持续迭代。

本课不重复讲工具、RAG 或 Harness 的实现细节，而是把前面能力转换成用户可理解、业务可验收、组织可运营的产品机制。

### 核心问题

- 用户为什么觉得 Agent 慢？
- 用户为什么不信任 Agent？
- Agent 失败时如何让用户继续掌控局面？
- 如何控制成本和延迟？
- 如何证明 Agent 比人工或传统 Workflow 更好？
- 如何让 Agent 的风险可审计？
- 如何把线上反馈变成下一轮改进？

### 核心内容

#### 7.1 产品化问题总览

Agent 产品化不是给 Demo 加一个界面，而是把用户体验、业务价值和风险治理放到同一个闭环里。

| 产品化问题 | 用户感受 | 对应机制 |
|---|---|---|
| Agent 慢 | 不知道它在干什么，等不下去 | 流式输出、进度展示、步骤解释、延迟拆解 |
| Agent 不可信 | 不敢让它自动执行 | Human-in-the-loop、可编辑计划、确认节点、回滚 |
| Agent 容易失败 | 出错后用户不知道怎么办 | 重试、降级、恢复、失败后的下一步建议 |
| Agent 太贵 | 单次任务成本不可控 | 模型路由、缓存、上下文裁剪、任务级成本上限 |
| Agent 有风险 | 可能泄露数据或越权操作 | 权限隔离、审批、审计、安全合规 |
| Agent 价值不清楚 | 不知道是否真的比人工或传统流程更好 | 任务完成率、用户干预率、满意度、留存、ROI |

#### 7.2 交互与信任

交互设计的目标不是让 Agent "看起来智能"，而是让用户知道它正在做什么、为什么这么做、什么时候可以介入。

学习内容：

- 流式输出、任务进度展示、当前步骤解释。
- 可编辑计划、确认、驳回、接管、回滚。
- 高风险动作前的 Human-in-the-loop。
- Progressive Trust：先让用户确认高风险动作，再随着成功经验逐步放宽低风险自动执行。
- 中断与恢复。
- 失败后的下一步建议。

#### 7.3 可靠性与恢复

Agent 失败不可避免，产品化重点是让失败可恢复、可解释、可继续。

学习内容：

- 重试策略。
- 指数退避。
- 工具降级。
- 模型降级。
- 超时控制。
- Rate Limit 管理。
- 幂等设计。
- 任务恢复。
- 失败分类和恢复策略。

#### 7.4 成本与延迟

成本和延迟同时影响用户体验和商业模型，不能只在上线后看账单。

学习内容：

- 模型路由。
- 小模型分类，大模型推理。
- 连续失败后升级模型，简单任务降级模型。
- Prompt Caching。
- 上下文裁剪。
- 工具结果压缩。
- RAG 检索数量控制。
- 任务级成本预算和成本上限。
- 失败重试成本上限。
- 延迟拆解：模型、工具、检索、排队、网络。

成本估算不需要一开始追求精确，但必须可感知：

```text
单次任务成本 = 模型输入/输出 token 成本 + 工具/API 成本 + 重试成本
```

例如一个 10 步任务，如果每步平均消耗 4000 token，总量就是约 40000 token；再乘以所选模型的单价，就能得到粗略成本。产品设计时要同时看平均成本、P95 成本和失败重试成本。

#### 7.5 安全、权限与合规

这一节承接课程四的权限机制和课程六的 Guardrails，重点是让 Agent 的风险可审计、可追责、可治理。

学习内容：

- Prompt Injection 防御。
- 数据脱敏。
- 权限隔离。
- 审计日志。
- 用户授权。
- 高风险动作审批。
- 人工确认记录、审批链路和回滚记录。
- 企业环境中的 GDPR、SOC2、数据驻留要求。

#### 7.6 产品指标与反馈闭环

Agent 产品不能只看 DAU 或调用次数。指标要能证明它是否真的完成任务、减少人工负担、提升用户信任。

学习内容：

- 北极星指标：任务完成率、用户满意度、人工替代或增强效果。
- 诊断指标：用户干预率、平均步骤数、工具调用成功率、失败恢复率。
- 成本指标：Token 成本、工具/API 成本、平均延迟、P95 延迟。
- 风险指标：越权拦截次数、高风险审批通过率、回滚次数。
- 用户反馈如何进入评测集和下一轮迭代。

#### 7.7 失败模式与产品复盘

失败模式不只是 bug list，而是产品复盘框架。每次失败都应该能归类、复现、评估影响，并进入后续改进。

常见类型：

- 用户意图理解错误。
- 工具选择、参数或执行失败。
- 上下文遗漏或错误记忆污染。
- 外部知识接入误导。
- 权限越界或审批缺失。
- 输出无法执行。
- 过早停止或死循环。
- 用户无法理解、无法接管或无法恢复。

## 课程八：项目实战与生态跟进

**定位**：用完整项目整合知识，同时建立持续跟进生态的习惯。

项目实战和生态趋势可以放到最后，不应该在初学阶段打散主线。

本课不是并列罗列三个 demo，而是用递进项目覆盖知识型、工具型、任务执行型 Agent。学习者可以只选一个项目做深，也可以按难度逐步推进。

### 8.1 项目实战

项目难度与递进关系：

| 项目 | 建议顺序 | 难度 | 建议时间 | 适合目标 |
|---|---|---|---|---|
| 个人知识助手 | 第一个项目 | 入门-中级 | 3-5 天 | 训练 RAG / 外部知识接入、引用、Memory |
| 代码审查 Agent | 第二个项目 | 中级 | 4-7 天 | 训练工具调用、文件读取、测试调用、行内反馈 |
| 个人任务执行 Agent | 第三个项目 | 中级-高级 | 7-10 天 | 训练 Planning / Workflow Patterns、状态管理、Human-in-the-loop、产品化 |

| 项目 | 重点训练 | 合格版 | 进阶版 | 产品版 |
|---|---|---|---|---|
| 个人知识助手 | RAG / 外部知识接入、Memory、引用、上下文管理 | 文档检索 + 回答 | 混合检索 + 多轮记忆 | 多用户、权限、评测、引用可信 |
| 代码审查 Agent | Tool Use、文件读取、测试调用、置信度、行内反馈 | 读取 diff 后输出 review | 调用测试和静态分析 | GitHub App、行内评论、误报反馈 |
| 个人任务执行 Agent | Planning / Workflow Patterns、工具机制、状态管理、Human-in-the-loop、产品化 | 支持学习计划、文档整理、项目发布准备 3 类任务，能拆解计划并调用本地工具 | 支持信息追问、分支流程、任务恢复、失败复盘 | 有任务看板、trace、成本/耗时统计、确认/接管机制和指标报告 |

### 8.2 项目场景定义

**个人知识助手**

使用场景：个人有一批学习笔记、PDF、网页摘录、项目文档，希望 Agent 能基于这些资料回答问题、给出引用，并在多轮对话中记住当前研究主题。

典型任务：

- "根据我这 5 篇文章，总结 Agent 工具调用的关键设计点。"
- "我之前整理过 RAG 的笔记，帮我找出和 Memory 相关的部分。"
- "回答这个问题时必须引用来源，不确定就说不知道。"

**代码审查 Agent**

使用场景：个人或小团队希望在提交代码前，让 Agent 读取 diff、理解改动意图、调用测试或静态分析，并输出可操作的 review 建议。

典型任务：

- "审查当前 Git diff，指出潜在 bug 和缺少的测试。"
- "运行相关测试，根据失败结果判断问题可能在哪里。"
- "把 review 结果按严重程度排序，并给出行内评论。"

**个人任务执行 Agent**

使用场景：个人输入一个目标，例如准备技术分享、整理学习计划、做项目发布准备，Agent 自动拆解任务、调用本地工具生成交付物，并在关键节点请求确认。

典型任务：

- "帮我制定一周 Agent 学习计划，并生成每天的任务清单。"
- "把这个目录下的零散笔记整理成一份 Markdown 文章草稿。"
- "为这个小项目做发布准备：检查 README、整理 changelog、生成发布 checklist。"

### 8.3 每个项目的交付物

- 可运行代码。
- 起始模板或最小骨架。
- 设计文档。
- 工具清单。
- 权限设计。
- 评测集。
- Trace 示例。
- 失败复盘。
- 成本估算。
- 下一步产品计划。

### 8.4 生态跟进

生态内容是持续关注，不是初学阶段的主线必修。

建议关注：

- MCP / A2A。
- Code Agent。
- Multi-Agent 框架。
- Agent RL。
- 新模型的工具调用、长上下文、结构化输出能力。
- Agent 框架和 Runtime 的演进。

跟进方式：

- 优先看官方文档和 release notes，确认能力边界和接口变化。
- 关注主流框架的 GitHub 示例、issue 和 changelog。
- 对论文和架构分析，重点看它解决了什么运行时问题，而不是照搬术语。
- 把新概念放回本路线的分层中判断：它属于工具机制、场景增强、运行时工程，还是产品化能力。

## 推荐学习顺序

不同角色不需要从同一个入口进入课程。学习顺序应围绕各自最需要建立的判断力来设计，但最终都要回到“最小闭环 -> 能力增强 -> 工程化 -> 产品化”的完整框架中。

### 产品经理 / 创业者

```text
课程一 -> 课程七 -> 课程三 -> 课程四 -> 课程五 -> 课程八
```

重点先建立产品直觉：什么场景适合 Agent，用户为什么愿意信任它，失败如何被产品机制吸收。随后补上最小闭环和工具机制，避免只停留在概念和案例层面。

适合目标：

- 判断 Agent 产品机会和边界
- 设计 Agent 产品形态、交互和指标
- 和工程团队讨论可行性、风险与迭代路径

### 应用工程师 / 全栈工程师

```text
课程三 -> 课程四 -> 课程六 -> 课程七
```

重点先掌握 Agent 如何跑起来，再理解工具机制和运行时架构。课程五可以按项目需要选择性学习，例如做知识问答时优先学 RAG / 外部知识接入，做复杂任务时优先学 Planning / Workflow Patterns。

适合目标：

- 实现能运行、能调试、能恢复的 Agent 应用
- 理解 Tool Use、状态管理、权限、安全、评测和观测
- 把 demo 推进到可上线的工程系统

### 算法 / AI 工程师

```text
课程二 -> 课程三 -> 课程五 -> 课程六 -> 课程八
```

重点理解 Agent 范式为什么从纯 Chatbot 走向工具调用、规划、反思和多 Agent，再把这些能力放回运行时闭环和工程评测中，而不是只研究单点算法。

适合目标：

- 理解 ReAct、Planning、Reflection、Multi-Agent 等范式的定位
- 判断哪些能力是模型能力，哪些是系统机制
- 将算法能力落到可验证的应用链路中

### 业务负责人 / 运营人员

```text
课程一 -> 课程七 -> 课程八 -> 课程五
```

重点先理解 Agent 能解决什么业务问题、不能解决什么问题，以及如何用指标和反馈判断效果。课程五用于补充不同业务场景的能力判断，例如知识密集场景、长期陪伴场景、流程执行场景分别需要什么增强能力。

适合目标：

- 识别适合 Agent 的业务流程和用户任务
- 定义验收标准、失败处理方式和运营指标
- 与产品、工程团队对齐需求边界

### 初学者 / 转型学习者

```text
课程一 -> 课程三 -> 课程二 -> 课程四 -> 课程五 -> 课程六 -> 课程七 -> 课程八
```

重点先形成感性认知，再做出最小闭环，然后回到课程二理解范式演进。这样不会在还没见过运行循环前就陷入 ReAct、Planning、Reflection 等概念。

适合目标：

- 系统建立 Agent 产品开发知识框架
- 从体验、原理、工程到产品化完整过一遍
- 通过项目实战形成可展示的作品
