## 7.1 "它替我做了决定，而我不在场"

知识助手 Agent 已经接入了 RAG、Memory、Context Engineering、Planning 和 Reflection。它能检索你的笔记、记住你的偏好、管理上下文、拆解任务、修正失败。你开始信任它。

然后有一天，你让它整理文件。

```text
你："帮我把 /tmp/logs 下超过 30 天的日志文件清理一下。"
Agent："好的。" → 调用 delete_file → 开始删除
```

你突然看到一个文件名闪过——`.env.backup`。那是你上周手动放进去的备份文件，不是日志。你来不及叫停，文件已经没了。

这不是 Agent "变坏了"。它按照你的指令执行了。问题在于：**"清理日志文件"这个任务，有一些后果是你没说的（env 备份不能删），而 Agent 不知道这些隐含约束。**

另一个场景：你在测试一个客服 Agent。用户说"我要退款"。

```text
用户："这个产品不好用，我要退款。"
Agent："非常抱歉给您带来了不好的体验。我已为您提交了全额退款申请，款项将在 3-5 个工作日内退回。"
```

你盯着屏幕，手心出汗。退款策略还没定义——金额上限、退款条件、审批流程，都还没做。Agent 替你做了决定。

这些场景指向同一个问题：**Agent 有能力执行某个操作，但不代表它应该自主执行。** 能力和权限之间有一道鸿沟——不是技术鸿沟，是判断力鸿沟。模型不知道你的业务规则、你的风险偏好、你不知道自己不知道的那些隐含约束。

这就是 Human-in-the-loop（HITL）要解决的问题：**在 Agent 的决策链路中，在正确的位置插入人类判断。**

## 7.2 "能做"不等于"该做"——能力与权限之间的鸿沟

先理清几个概念在 HITL 语境下的含义：

```text
能力（Capability）：Agent 在技术上能执行什么操作。
  例：Agent 可以调用 delete_file、可以调用 refund_api、可以发送邮件。

权限（Permission）：系统允许 Agent 自主执行什么操作。
  例：delete_file 被允许，但需要人类确认。refund_api 完全禁止自主调用。

判断（Judgment）：这个具体操作在当下是否合适。
  例：删除 /tmp/logs/access_2026.log 是合适的。删除 /tmp/logs/.env.backup 不是。
```

HITL 的本质是：**当 Agent 的能力覆盖了某个操作，但系统无法在代码层面穷举所有判断规则时，把判断权交还给人类。**

为什么不能在代码里写规则？因为：

- `.env.backup` 是"日志文件"吗？从命名规则看不是，但它确实在 /tmp/logs 目录下。Agent 没判断出来。
- "全额退款"合适吗？取决于你的退款政策、用户历史、订单金额……这不是几行 if-else 能覆盖的。

HITL 不是充分的安全机制，不能替代权限校验、输入过滤、工具隔离、审计日志和 Guardrails（课程六第七章会讲）。它更像防线中的一层**决策增强机制**——在 Agent 的自主判断力不足以覆盖风险时，引入人类的判断力作为增强。

## 7.3 HITL 的五种模式

HITL 不是简单的"弹个确认框"。根据人类介入的时机和深度，可以分为五种模式。

### 7.3.1 确认模式（Confirmation）

**什么时候用**：Agent 已经决定了要做什么，但操作有风险，需要人类点头。

```text
Agent："我将删除以下 12 个文件：
  - /tmp/logs/access_20260501.log (32MB, 45 天前)
  - /tmp/logs/error_20260515.log (8MB, 31 天前)
  - ……
  [确认执行] [取消] [修改范围]"
```

确认模式的三个设计要点：

**1. 呈现"后果"而不是"操作"**

```text
❌ 差："确认调用 delete_file？"
   人类不知道要删什么、删多少、有什么影响。

✅ 好："将删除 12 个文件，共 156MB。这些文件上次修改超过 30 天。
   注意：/tmp/logs 目录下还有 3 个非日志文件（.env.backup, config.json, README），
   它们不会被删除。"
```

**2. 提供中间选项，不只"是/否"**

```text
[确认全部] [只删 .log 文件] [让我逐个确认] [取消]
```

二选一让人类倾向于点"确认"（决策疲劳）。给中间选项提高了决策质量。

**3. 确认的上下文要完整但不冗长**

人类需要看到：要操作什么、为什么 Agent 认为该做、风险是什么、不操作会怎样。

### 7.3.2 澄清模式（Clarification）

**什么时候用**：Agent 不确定用户意图，需要人类明确。

```text
你："把最近的文章整理一下。"
Agent："'最近的文章'可能指：
  A. 最近 7 天写的文章（共 3 篇）
  B. 最近修改过的文章（共 8 篇）
  C. 最近打开过的文章（共 5 篇）
  你指的是哪种？"
```

澄清模式和确认模式的区别：确认是"我已经决定了，你点头"，澄清是"我需要你帮我决定"。

澄清模式的设计要点：

- **给出具体选项，而不是抛回开放式问题**。❌ "请明确你的需求" ✅ "你是要 A、B 还是 C？"
- **每个选项附带数量或示例**，让人类能基于信息选择，而不是基于猜测。
- **提供"其他"选项**，让人类可以给出 Agent 没考虑到的答案。

### 7.3.3 接管模式（Takeover）

**什么时候用**：操作太复杂或太敏感，Agent 不应该自己执行，应该把控制权交还给人类。

```text
Agent："检测到数据库迁移脚本涉及以下操作：
  - 修改 users 表的 email 字段类型
  - 删除 deprecated_records 表
  
  这类操作需要人工执行。我已准备好迁移脚本：
  /tmp/migration_20260629.sql
  
  执行后请回复'已完成'，我会继续后续步骤。"
```

接管和确认的本质区别：确认是 Agent 执行、人类许可。接管是**人类执行、Agent 等待**。Agent 退到辅助角色——它准备、它说明、它等待，但不操作。

接管模式的适用场景：
- 数据库 schema 变更
- 生产环境部署
- 涉及合规审批的操作
- 任何"做错了无法简单回滚"的操作

### 7.3.4 审核模式（Review）

**什么时候用**：Agent 完成了产出，但需要人类检查质量或在多个方案中选择。

```text
Agent："技术方案已完成，共 4 个部分：
  1. 架构概述（已生成）
  2. API 设计（已生成）
  3. 数据库设计（已生成）
  4. 安全考虑（已生成，标注了 3 个待确认点）
  
  [查看全文] [逐节审核] [修改某节] [通过]"
```

审核模式的核心是要让人类能**快速定位需要关注的点**。Agent 应该主动标注：
- 它不确定的地方
- 有多种合理选择的地方
- 它建议但需要人类拍板的地方

不要让人类通读全文才能做判断。**标注不确定性是审核模式最有价值的输出。**

### 7.3.5 教学反馈模式（Teaching Feedback）

**什么时候用**：Agent 做错了或做得不够好，人类给出修正，Agent 从修正中学习。

```text
Agent："我为你生成了发布 checklist。"
你："第 3 项漏了数据库备份步骤。第 6 项的检查命令不对，应该用 pytest 而不是 unittest。"
Agent："已修正。我注意到你在这个项目中一直使用 pytest。我将在 Memory 中更新：
  '项目 test_runner 偏好：pytest（非 unittest）'
  以后生成 checklist 时也会自动包含数据库备份步骤。"
```

教学反馈是 HITL 和 Memory 的交汇点。人类的每一次修正，都应该被捕获为可复用的知识。有三个层次：

| 层次 | 学到什么 | 影响范围 |
|---|---|---|
| **即时修正** | 这次改对了 | 仅当前任务 |
| **偏好更新** | 用户喜欢 pytest | 当前项目的后续任务 |
| **模式学习** | "发布 checklist 必须包含数据备份" | 所有项目的发布类任务 |

教学反馈不要过度承诺"自动学习"。V0 只做到"即时修正"就够了。偏好更新和模式学习需要 Memory 系统的配合（见第三章）。

### 7.3.6 五种模式对比

| 模式 | 人类角色 | Agent 角色 | 介入时机 | 典型场景 |
|---|---|---|---|---|
| **确认** | 许可者 | 执行者 | 执行前 | 删除文件、发送邮件、Git push |
| **澄清** | 定义者 | 执行者 | 理解后 | 模糊需求、多义指令 |
| **接管** | 执行者 | 辅助者 | 执行时 | 数据库迁移、生产部署 |
| **审核** | 审查者 | 创作者 | 产出后 | 代码审查、方案审核 |
| **教学反馈** | 教练 | 学习者 | 错误后 | 输出不符合预期、偏好变化 |

## 7.4 设计决策：什么该问、多久问一次、怎么呈现

### 7.4.1 风险分级：哪些操作需要人类介入

不是所有操作都需要 HITL。分级依据：**这个操作的不可逆性和后果严重程度。**

| 风险等级 | 操作特征 | HITL 模式 | 示例 |
|---|---|---|---|
| **低** | 只读、无副作用、可重复 | 无需介入 | 读文件、搜索代码、生成文本 |
| **中** | 写入但可回滚、影响范围小 | 确认（可批量） | 创建/编辑文件、发送草稿 |
| **高** | 写入且难回滚、影响外部系统 | 逐项确认 | 删除文件、Git commit、API 写入 |
| **关键** | 不可逆、涉及合规/资金/安全 | 接管或逐项确认+审计 | 数据库迁移、退款、部署、权限变更 |

分级不是一劳永逸的。一个操作的风险等级会随上下文变化：
- `delete_file` 删除临时文件 → 中风险
- `delete_file` 删除 `~/.ssh/` 下的文件 → 关键风险
- `send_email` 发送草稿给同事 → 中风险
- `send_email` 群发给 1000 个用户 → 高风险

所以风险分级不能只基于工具名称，要结合参数和上下文来判断。

### 7.4.2 频率控制：别太烦，也别太放心

HITL 设计中最难把握的不是"什么该问"，而是"多久问一次"。

**太频繁的后果**：人类变成"确认机器人"。每步都要点确认，用户很快会养成肌肉记忆——看都不看就点"确认"。HITL 形同虚设。

**太稀疏的后果**：人类失去对 Agent 正在做什么的感知。等发现问题时，Agent 已经做了十个不可逆的操作。

频率控制的几个策略：

**策略一：批次确认**

```text
❌ 单个确认：
  "确认删除 access_20260501.log？" [确认]
  "确认删除 error_20260515.log？" [确认]
  "确认删除 debug_20260520.log？" [确认]
  …（12 次）

✅ 批次确认：
  "将删除以下 12 个文件，共 156MB。其中 11 个为 .log 文件，1 个为 .txt 文件。
   [确认全部] [只删 .log 文件] [逐个确认] [取消]"
```

**策略二：信任累积**

如果用户连续 5 次对同类操作确认通过，可以问："是否信任我对同类文件的操作，不再逐项确认？"

**策略三：摘要式回溯确认**

不是执行前确认每步，而是执行后给一个摘要，让用户确认整体方向：

```text
"上一阶段完成：已收集 8 篇相关文档，共 32KB。准备进入写作阶段。
 [继续] [查看文档列表] [调整方向]"
```

**策略四：基于会话的信任衰减**

同一次会话中，Agent 对用户偏好的判断会越来越准。但跨会话后，信任应该重置——因为任务可能不同了。

### 7.4.3 上下文呈现：让人类能做快速判断

人类做 HITL 决策时，同样面临信息不足的问题。你弹一个确认框："Agent 要执行 refund 操作，确认？"——就这一句话，人类怎么判断？

好的 HITL 设计在请求人类决策时，会给出一个**精简但完整的决策上下文**：

```text
Agent 请求执行退款操作：

▸ 用户：张三 (ID: 12847)
▸ 订单：ORD-20260629-0042
▸ 金额：¥299.00（全额）
▸ 原因：用户反馈"产品功能与描述不符"
▸ 用户历史：注册 2 年，此前 0 次退款申请
▸ 订单状态：已支付，未发货

Agent 判断：符合退款条件（7 天无理由，未发货）

[批准退款] [拒绝并说明原因] [转为人工处理]
```

关键设计原则：

1. **突出异常**：如果"此前 0 次退款申请"变成"此前 5 次退款申请"，应该用不同颜色/标记突出
2. **Agent 的判断依据透明化**：不仅告诉人类 Agent 想做什么，还告诉人类 Agent 为什么这么判断
3. **提供上下文相关的选项**：不只"是/否"，还有"转为人工处理"这类中间选项

## 7.5 从反馈中学习

HITL 每次介入都是一次标注。人类说"对"或"不对"，这些信号如果只用于当前决策，价值就浪费了。

三个层次的学习：

**层次一：即时应用（Always）**

人类的决定影响当前任务的继续方向。"驳回退款"→ Agent 停止退款流程，转告用户。

**层次二：偏好更新（与 Memory 协作）**

人类反复确认或驳回同类操作时，更新 Memory：

```text
"用户偏好更新：对 /tmp/logs 目录下的文件清理操作，用户已连续 3 次确认同类操作。
下次可以降低确认频率（批次确认即可）。"
```

**层次三：策略调整（需要人工审查）**

从 HITL 数据中分析：哪些操作的确认通过率太低（说明 Agent 判断不准）？哪些操作用户从来不驳回（可能不需要 HITL）？

```text
HITL 数据分析（过去 30 天）：
- read_file 确认：120 次，通过率 100%  → 考虑移除确认
- delete_file 确认：45 次，通过率 89%  → 保持确认
- refund 确认：8 次，通过率 50%       → 确认不够，考虑改为接管模式
- send_email 确认：30 次，通过率 97%  → 可降低为中风险操作
```

这个分析不应该自动执行——人类应该审查并决定是否调整策略。HITL 数据的价值在于**让策略调整有依据**，而不是让系统自己改策略。

## 7.6 进化路线：从"全确认"到"精准介入"

| 阶段 | 做了什么 | 适用场景 |
|---|---|---|
| **V0：全手动** | 高风险操作全部硬编码禁止，人类手动执行后 Agent 继续 | 原型、内部工具 |
| **V1：关键操作确认** | 对写操作统一弹确认框（是/否） | 第一个有用户的版本 |
| **V2：风险分级 + 批次确认** | 按风险等级使用不同 HITL 模式，同类操作支持批次确认 | 用户开始频繁使用 |
| **V3：上下文呈现 + 学习** | 确认框展示决策上下文；驳回的偏好写入 Memory | 用户信任建立期 |
| **V4：基于数据的策略调整** | 用 HITL 通过率数据调整风险分级和 HITL 模式 | 成熟产品、持续优化 |

大多数项目从 V1 开始就够了。V0 太保守（Agent 做不了任何有用的事），V4 需要有足够的用户量和使用数据才有意义。

## 7.7 HITL 的五个反模式

**反模式一：每步都确认**

在所有工具调用前都弹确认框。结果：用户 30 秒内点了 15 次"确认"，一个都没看。HITL 形同虚设。

**正确做法**：只在高风险和关键操作上介入。低风险操作让它自动执行。

**反模式二：确认框信息不足**

```text
"Agent 要执行 write_file，确认？"
```

用户不知道要写什么、写到哪、为什么写、有什么影响。这种确认框没有给用户任何判断依据。

**正确做法**：给出文件路径、变更摘要、Agent 的判断依据。

**反模式三：HITL 当作安全机制**

用 HITL 来防止 Prompt Injection 或越权操作。"反正有人确认，不用做输入校验。"

**正确做法**：HITL 是决策增强，不是完整的安全方案。安全需要权限、校验、隔离、审计和 Guardrails 一起工作。人类也会犯错——攻击者可以设计让人类倾向于"确认"的上下文。

**反模式四：没有超时处理**

Agent 等待人类确认，人类去开会了。Agent 一直等。

**正确做法**：设定超时。超时后的行为取决于操作风险——低风险操作可以自动继续，高风险操作应该安全中止。

**反模式五：所有用户一视同仁**

新手用户和资深用户看到同样的确认频率。

**正确做法**：允许用户调整 HITL 级别。"开发者模式"可以降低确认频率，"安全模式"提高确认频率。让用户自己控制他们愿意承担的自主程度。

## 7.8 什么时候不需要 HITL

1. **低风险的纯信息型 Agent**：只回答问题、不执行操作的 Agent，没有工具副作用风险，通常不需要在每次回答前加入 HITL。但如果输出会影响医疗、法律、财务、安全、合规等高风险决策，仍然需要人工审核或明确免责声明。
2. **只读操作**：如果 Agent 的所有工具都是只读的（读文件、搜索、查询），没有副作用。HITL 的价值几乎为零。
3. **可逆、低影响的个人操作**：只有你自己用的 Agent，且操作可回滚、不涉及密钥、隐私、生产系统或外部发送。HITL 可以降低频率，但不应因为"个人使用"就默认放开所有动作。
4. **有完善的自动化校验**：如果工具执行后有自动化的测试、lint、类型检查作为安全网，HITL 的频率可以大幅降低。自动化校验在可检测性质上比人类确认更稳定，但不能替代业务意图、合规判断和用户影响评估。

## 可运行示例

完成本章学习后，可以对照运行课程五 05-07 的本地 Human-in-the-loop 示例：

- [课程五 05-07 Human-in-the-loop 示例](../examples/course-05-07-human-in-the-loop/README.md)

该示例围绕文件清理、退款处理和发布文档生成三个场景，用 Python 和 Node.js 双版本演示 HITL 的五种介入模式：确认、澄清、接管、审核和教学反馈。你会看到同一个 `delete_file` 操作如何因为路径、可逆性和影响范围不同，被判定为中风险、高风险或关键风险；也会看到 Agent 在请求人类判断时如何呈现后果、异常项、判断依据和中间选项。

示例还会生成 `hitl_audit.jsonl` 和 `hitl_memory.json`，用来观察两件事：第一，HITL 决策本身应该被审计，而不是只停留在弹窗层；第二，人类的审核和纠正可以转化为可复用的 Memory，例如"发布 checklist 必须包含数据库备份步骤"。这对应本章 7.4 的风险分级与上下文呈现，也对应 7.5 的反馈学习。

示例是教学实现，不是完整生产安全框架：风险评估使用规则实现，退款和数据库迁移不会调用外部系统；运行时需要手动选择 HITL 决策，直接按 Enter 会采用默认选项，方便你连续观察完整流程。真实系统仍然需要权限校验、工具隔离、输入过滤、审计日志和 Guardrails 共同配合。

cd examples/course-05-07-human-in-the-loop/python

python3 -m unittest test_hitl_demo.py

# Node.js 版本
