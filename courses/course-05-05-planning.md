## 5.1 任务变长后 Agent 开始漂移

回到 1.1 的知识助手。你给它一个多步骤任务：

```text
帮我做发布准备：检查 README 完整性、跑一遍测试、整理 changelog、生成 release checklist。
```

Agent 开始工作。第一步，它读了 README，写了一段评论。第二步，它……开始去翻源码核实 README 里提到的一个细节。第三步，它又去搜了相关的 GitHub issue。五分钟后你回来检查，发现它偏离原始目标，不知道在做什么。你翻 trace 才发现：它在第 4 步之后就再也没碰过任何和"发布"相关的操作。

原因是：**裸 ReAct 循环没有"全局任务结构"的概念。** 它每步都在做局部最优的决策，但局部最优之和不一定能到达目标。就像一个没有导航的司机，每个路口都选看起来对的方向，但可能永远到不了目的地。

任务越长，越需要一种机制来回答：

- 要做哪些步骤？步骤之间的依赖关系是什么？
- 哪些步骤可以并行执行？
- 哪些步骤执行前需要用户确认？
- 某一步失败后，是重试、跳过、重规划还是停止？

这就是 Planning / Workflow Patterns 要解决的核心问题。

**在本章中，我们将用一个贯穿全章的"发布助手"场景来具体演示。** 这个场景定义如下：

```text
场景：发布助手 Agent

用户：一位维护着开源项目的开发者
典型任务：软件版本发布准备（检查 README、运行测试、整理 changelog、生成 checklist）
复杂度：4 个步骤，步骤间存在依赖关系（changelog 必须在测试通过后生成，
        checklist 依赖所有前置步骤完成）
挑战：
  - 步骤不是完全独立的——有先后依赖
  - 测试可能失败——需要决定重试还是跳过
  - README 可能缺少必要章节——需要决定是否阻塞发布
  - 初次使用者可能说不清楚所有步骤——需要 Agent 自己补全

Planning 需求：
  - 步骤之间的依赖关系需要被明确建模
  - 某一步失败后需要明确的处理策略
  - 执行前用户可能需要确认计划
```

后续每个 Planning 模式，我们都会用这个场景来演示——同样的任务，不同的组织方式，效果完全不同。

## 5.2 为什么只会 ReAct 还不够：从临场判断到显式流程

Planning 在 Agent 领域不是一开始就有的。回顾一下演进：

**2022 年 10 月，ReAct 论文（Yao et al.）。** 论文的核心发现是：让 LLM 在"推理"和"行动"之间交替循环，能大幅提升复杂任务的完成率。但 ReAct 的模式是"每一步临场判断"——模型看到当前 Observation，决定下一步 Action。这对 3-5 步的任务有效，但步数一多就暴露了问题：模型没有全局视野，容易漂移。

**2023 年初，Plan-and-Execute 模式出现。** 思路很直接：让模型先把整个任务拆成执行计划，用户确认后再逐步执行。这解决了 ReAct 的"没有全局视图"问题，但引入了新问题——计划可能不可执行（模型列出的步骤在实际环境中无法完成），或者环境变化后原计划不再适用。

**2023 年下半年，图结构（Graph）思想兴起。** LangGraph、CrewAI 等框架开始把任务建模成节点和边的图——每个节点是一个动作或判断，边代表状态迁移，失败后可以跳回特定节点重试。图结构不是 Plan-and-Execute 的唯一演进方向，而是把一维步骤列表扩展成显式控制流：它可以表达分支、回溯、并行路径、人工确认点，也可以承载 ReAct、Router 或 Plan-Execute 这类模式。

**2024 年至今，Workflow Patterns 成为共识。** Anthropic 的"Building Effective Agents"文章把 Prompt Chaining、Routing、Parallelization、Orchestrator-Workers、Evaluator-Optimizer 总结为常见工作流模式；OpenAI Agents SDK 则把 handoff、guardrail、tracing、evals 放进 Agent 编排的基本工具箱；LangGraph 强调 durable execution、human-in-the-loop、persistence 和 time travel。它们共同指向同一个原则：大多数场景不需要一开始就上复杂 Graph，关键是根据任务形态选择最小够用的组织方式，并用 trace 和 eval 验证复杂度是否真的带来收益。

## 5.3 任务组织比单步决策更重要

ReAct Loop 擅长动态决策，但不擅长天然保证全局结构。

复杂任务通常有几种形态：

| 任务形态 | 例子 | 适合模式 |
|---|---|---|
| 步骤固定 | 读取文件 -> 摘要 -> 输出 | Chain |
| 输入类型差异大 | 问答、写作、代码审查走不同路径 | Router |
| 子任务相互独立 | 同时检查 README、测试、依赖漏洞 | Parallelization |
| 子任务数量不确定 | 代码迁移、跨文件重构、竞品调研 | Orchestrator-Workers |
| 输出需要反复打磨 | 文案优化、代码修复、计划质量审查 | Evaluator-Optimizer |
| 工具选择动态 | 需要边查边决定下一步 | ReAct Loop |
| 任务可先拆解 | 发布准备、调研报告、代码迁移 | Plan-Execute |
| 状态和分支复杂 | 工单处理、审批流、多阶段任务 | Graph |

但光看表格不够直观。我们用发布助手场景来做一个**同一任务的不同模式对比**：

```text
任务："准备 v1.2.0 版本发布"

┌─────────────────────────────────────────────────────────────────────────┐
│ Chain 模式处理：                                                         │
│   检查 README → 运行测试 → 整理 changelog → 生成 checklist               │
│   遇错即停。测试失败 → 后面全跳过。                                      │
│                                                                         │
│ Plan-Execute 模式处理：                                                   │
│   1. 生成计划（同上 4 步）                                               │
│   2. 用户确认                                                            │
│   3. 逐步执行。测试失败 → 重试 2 次 → 仍失败 → 重规划                    │
│      "测试失败原因是 test_memory_cleanup 断言错误，                      │
│       新的后续步骤：分析失败原因 → 修复 → 重新测试 → 继续"                │
│                                                                         │
│ Graph 模式处理：                                                          │
│   检查 README ──success──▶ 运行测试 ──success──▶ changelog ──▶ checklist │
│        │                      │                                         │
│        error                  error                                      │
│        ▼                      ▼                                         │
│   修复 README            重试测试                                        │
│   (修复后终止，         (重试成功→changelog                               │
│    人工检查)             重试失败→终止)                                   │
└─────────────────────────────────────────────────────────────────────────┘
```

**关键洞察**：同一个任务，三种模式的区别不在于"能不能完成"——正常流程下都能完成。区别在于**异常发生时**：
- Chain 直接放弃
- Plan-Execute 动态调整
- Graph 按预定义分支跳转

所以不要问"Planning 好不好"，而要问：

- 任务是否足够复杂，需要先拆解？
- 步骤是否稳定，可以固化成流程？
- 是否需要中途重规划？
- 用户是否需要确认计划？
- 失败后是否需要回到某个节点？

## 5.4 先选组织方式，再谈 Agent 能力

下面我们围绕发布助手场景，逐一展开每种模式的接口骨架、执行行为和适用边界。本章末尾的示例项目中有完整的可运行代码；正文里的代码片段是为了说明控制流，省略了部分工程细节。

### 5.4.1 Chain：固定顺序换来确定性，也牺牲恢复能力

Chain 是"一条路走到黑"——就像工厂流水线，每个工位按固定顺序加工，前一站没做完后面就等着。发布准备的四个步骤按固定顺序执行——先检查 README，再运行测试，然后整理 changelog，最后生成 checklist。没有任何分支，没有任何"如果……就……"。

```python
class ChainExecutor:
    """按固定顺序依次执行步骤。遇错即停，无分支，无重试。"""
    
    def execute(self, on_step_start=None, on_step_end=None):
        for i, step_name in enumerate(self.steps):
            tool = TOOL_REGISTRY.get(step_name)
            if not tool:
                return ChainResult(status="failed", failed_at=i, 
                                   error=f"未找到工具: {step_name}")
            
            result = tool()  # 执行步骤
            if result.status == StepStatus.ERROR:
                return ChainResult(status="failed", failed_at=i, 
                                   error=result.error)
            # 成功，继续下一步
        
        return ChainResult(status="completed", results=all_results)
```

**Chain 的问题不在正常路径上**——正常时四个步骤依次跑完，一切 OK。但在异常路径上：

```text
场景：运行测试失败

Chain 执行过程：
  ✅ 检查 README → 通过（缺少 Contributing 章节，但不算失败）
  ❌ 运行测试 → 失败！1 个测试用例未通过
  ⏭️ 整理 changelog → 跳过
  ⏭️ 生成 checklist → 跳过
  
最终报告："发布准备失败，第 2 步运行测试出错。"

用户的感受：我就因为一个测试失败，整个发布流程就停了？
而且那个失败的测试跟这次发布改动无关，是之前的遗留问题。
Chain 不会说"这个测试失败不阻塞发布，可以跳过"——它只会停止。
```

**Chain 的适用边界很窄也很清晰**：步骤完全确定、不会出意外、不需要任何动态决策。比如"读取配置文件 → 解析 → 校验 → 写入缓存"——这四步在正常情况下不会失败，失败就意味着系统出了问题，停止是对的。

Chain 看似简单，但它的失败往往被低估——因为"遇错即停"本身就是一个设计决策，不总是正确的：

| 失败表现 | 典型原因 | 修正方式 |
|---|---|---|
| 非关键步骤失败导致全流程中断 | Chain 不区分"阻塞性错误"和"非阻塞性警告"——README 少了个非关键章节和测试全红，在 Chain 眼里都是"失败" | 如果确实需要 Chain，在步骤内部区分 error 和 warning，warning 不阻断后续步骤 |
| 步骤顺序写死但实际依赖不同 | changelog 依赖"测试通过"，但 Chain 的步骤列表是扁平的，没有依赖建模——只能靠顺序隐含依赖 | 如果步骤间有复杂依赖，Chain 不适合；升级到 Plan-Execute 或 Graph |
| 同样的失败每次都要从头跑 | 第 3 步失败了，下次执行要从第 1 步重新开始——Chain 没有"从失败步骤继续"的概念 | 如果任务执行耗时长，考虑 Graph 的状态持久化能力 |
| 用户插手的代价高 | 执行到一半用户想跳过某一步，Chain 不支持——只能等它跑完或手动终止后从中间步骤重新拼接 | 如果用户交互频繁，至少加上步骤间确认点（这不是纯 Chain 了，但实用） |

### 5.4.2 Router：先分清任务类型，再选择执行路径

Chain 的"一条路走到黑"在步骤固定时是优点，但当用户输入从"发布准备"变成"帮我修个 bug"时——你不可能用同一套步骤。Agent 需要先判断"这是什么类型的任务"，再走对应的路径。这就是 Router。

Router 本质是"分类器 + 多条 Chain"。不同任务走不同路径：

```python
class RouterExecutor:
    ROUTES = {
        "release": ["检查 README", "运行测试", "整理 changelog", "生成 checklist"],
        "bugfix":  ["运行测试", "整理 changelog"],
        "docs":    ["检查 README"],
    }
    
    def classify(self, query: str) -> str:
        # 实际项目中这里是 LLM 调用，这里简化为关键词匹配
        if "发布" in query or "release" in query.lower():
            return "release"
        if "bug" in query.lower() or "修复" in query:
            return "bugfix"
        if "文档" in query or "readme" in query.lower():
            return "docs"
        return "release"  # 兜底
    
    def execute(self, query: str):
        category = self.classify(query)
        steps = self.ROUTES.get(category, self.ROUTES["release"])
        return ChainExecutor(steps).execute()
```

Router 的关键设计点：
- **分类准确度**：分类错了，后续全错。生产环境中不要只看"准确率百分比"，要看"错分一次的代价"。客服 FAQ 错分一次可能只是答错问题；退款、发版、生产操作错分一次就可能造成真实损失。高风险路径需要确认、拒识或人工接管。
- **兜底路径**：分类失败时必须有 default 路径。兜底不应该默认走"最完整"的路径，而应该走"最安全"的路径：解释无法判断、收集更多信息，或只执行只读检查。
- **路径内还是 Chain**：Router 只解决"选哪条路"，不解决"路上遇到坑怎么办"。路径内部仍然是遇错即停。

Router 的失败通常不出在"选对了路"的时候，而出在"选错了路"和"没有路可选"的时候：

| 失败表现 | 典型原因 | 修正方式 |
|---|---|---|
| 分类错误导致执行了错误的路径 | 用户说"帮我检查一下代码"，Router 分类为 bugfix 路径（运行测试+整理changelog），但用户只想做代码审查 | 分类器不只依赖关键词匹配；用 LLM few-shot 做意图分类，对模糊输入加一层"确认路由"的交互 |
| 兜底路径成为"万能垃圾箱" | 所有不确定的输入都走 default release 路径，导致 release 路径被过度触发，用户困惑为什么"帮我写个文档"也触发了发布流程 | 兜底路径应该是最安全的路径（如只做信息展示），不是最全的路径；对兜底路径的触发频率做监控 |
| 路径定义过时 | 上个月定义的 bugfix 路径只包含"运行测试+整理changelog"，但团队现在的 bugfix 流程还需要"更新版本号" | 路径定义需要跟着实际工作流演进；定期审查每个路径的使用频率和用户反馈 |
| 路径间没有共享步骤 | release 路径和 bugfix 路径都有"运行测试"，但各自独立定义——修改测试工具名需要改两次 | 提取公共步骤为可复用的 step 定义，Router 路径引用 step 名而非重复定义 |

### 5.4.3 Plan-Execute：先让计划可见，再允许失败后重规划

Router 解决了"选哪条路"，但路径内部仍是 Chain——遇错即停。发布准备中运行测试失败了怎么办？如果只是因为一个与本次发布无关的遗留测试失败就全盘停止，用户不会满意。你需要的不是固化的路径，而是"根据执行情况动态调整"的能力——先出计划、执行中遇错后重规划。

这是 Planning 最经典的实现。核心流程分四步：

```text
1. 生成计划：分析目标 → 拆解为结构化步骤（每步包含：名称、工具、描述、依赖、重试次数）
2. 用户确认：展示计划 → 用户修改或确认
3. 逐步执行：按依赖顺序执行 → 记录每步结果
4. 失败重规划：某步重试耗尽 → 基于当前状态生成新的后续步骤
```

```python
class PlanExecuteExecutor:
    def execute(self, goal, inject_failures=None):
        # 1. 生成计划
        plan = self.generate_plan(goal)  
        # plan.steps = [
        #   PlanStep("检查 README", tool="检查 README", depends_on=[]),
        #   PlanStep("运行测试", tool="运行测试", depends_on=[]),
        #   PlanStep("整理 changelog", tool="整理 changelog", depends_on=["运行测试"]),
        #   PlanStep("生成 checklist", tool="生成 checklist", 
        #            depends_on=["检查 README", "运行测试", "整理 changelog"]),
        # ]
        
        # 2. 逐步执行
        for step in plan.steps:
            # 检查依赖是否满足
            unmet = [d for d in step.depends_on if d not in plan.completed_steps]
            if unmet:
                # 依赖未满足 → 先插入依赖步骤
                continue
            
            result = execute_step(step, inject_failures)
            
            # 3. 失败处理
            if result.status == "error":
                step.retries += 1
                if step.retries < step.max_retries:
                    continue  # 重试
                else:
                    # 重试耗尽 → 触发重规划
                    new_steps = self.replan(goal, plan.completed_steps, 
                                            step.name, result.error)
                    plan.replace_remaining(new_steps)
        
        return plan.final_output
```

**重规划（replan）是 Plan-Execute 与 Chain 的核心差异。** 它的逻辑是：基于已完成的步骤 + 失败原因 + 可用工具，生成一组新的后续步骤。这不是"从零重新计划"——已完成的步骤不会重做。

```text
重规划示例（运行测试失败）：
  输入：
    已完成：["检查 README"]
    失败步骤："运行测试"
    错误信息："1 个测试失败: test_memory_cleanup_on_session_end"
  
  重规划输出（新步骤替换原计划的后续步骤）：
    步骤 3. 分析运行测试失败原因 — 重新运行测试以确认失败原因
    步骤 4. 整理 changelog — 继续执行
    步骤 5. 生成 checklist — 继续执行
```

**Plan-Execute 的常见陷阱**：重规划本身可能失败。如果 replan 生成的新步骤也不可执行，你需要一个退出条件——通常是最大重规划次数（比如最多重规划 2 次），超过后上报给人。示例项目中的 Plan-Execute 执行器也实现了这层保护：达到 `max_replan_count` 或生成重复重规划步骤时，会停止执行，避免陷入循环。

Plan-Execute 是四种模式中最灵活的，但灵活性也带来了独特的失败模式：

| 失败表现 | 典型原因 | 修正方式 |
|---|---|---|
| 重规划死循环 | replan 不区分"可恢复错误"（网络超时）和"不可恢复错误"（不在 git 仓库中），对所有失败一视同仁地重规划，且新计划与旧计划相同 | 错误分类：可恢复→重试，不可恢复→跳过并通知用户；设 max_replan_count=2；检测新步骤是否与前次相同 |
| 计划过于模糊，无法执行 | 模型生成的步骤如"确保一切正常"——没有具体工具、没有验收标准、没有输入输出 | 计划生成时要求每步必须绑定具体工具和验收标准；对不含工具名的步骤打回重新生成 |
| 用户确认后计划被篡改 | 执行过程中 Agent 检测到环境异常，自行插入了额外步骤，用户不知道计划变了 | 确认后的计划锁定为 immutable；任何偏离（包括插入步骤）都需要重新请求用户确认 |
| 计划生成太慢，用户等不及 | 复杂任务的计划生成调用了多次 LLM，耗时 30 秒以上，用户已经切到其他窗口 | 计划生成超时限制（如 10 秒）；先出粗略计划让用户确认方向，再细化 |
| 依赖关系建模错误 | changelog 声明依赖"运行测试"，但实际执行时"运行测试"还未完成 changelog 就开始了 | 执行引擎需要检查依赖步骤的完成状态，而非只检查步骤名是否存在 |

### 5.4.4 Graph：把异常路径提前画进状态机

Plan-Execute 的动态重规划很灵活，但这种灵活性也有代价——每次失败都要重新调用 LLM 生成新步骤，延迟高且结果不确定。如果你的发布流程每周执行一次、失败模式已经稳定（README 缺失→修复；测试失败→重试），你不需要每次都让 LLM 重新想"现在该怎么办"。你可以提前把答案画成一张地图——这就是 Graph。

Graph 把任务建模成有向图。与 Plan-Execute 的"一维列表 + 动态重规划"不同，Graph 的**所有可能路径在构建时已定义**：

```text
发布准备的 Graph 结构：

              ┌────────────┐
              │ 检查 README │
              └─────┬──────┘
                    │ success         error
                    ▼                 ▼
              ┌────────────┐    ┌──────────────┐
              │  运行测试   │    │ 修复 README   │
              └─────┬──────┘    └──────┬───────┘
                    │                  │ success
             success│ error            ▼
                    ▼     ▼       (终止: 人工检查)
              ┌─────────────┐
              │整理 changelog│
              └─────┬───────┘
                    │ success/error → (终止)
                    ▼
              ┌─────────────┐
              │生成 checklist│
              └─────┬───────┘
                    ▼
                  (完成!)
```

```python
class WorkflowGraph:
    def __init__(self):
        self.nodes: dict[str, Node] = {}
    
    def add_node(self, name, action, on_success=None, on_error=None, 
                 max_retries=1):
        self.nodes[name] = Node(name, action, on_success, on_error, max_retries)
    
    def run(self, entry, context):
        current = entry
        while current:
            node = self.nodes[current]
            result = node.action(context)
            
            if result.status == "error":
                node.retry_count += 1
                if node.retry_count < node.max_retries:
                    current = current  # 重试
                else:
                    current = node.on_error  # 走失败分支
            else:
                current = node.on_success  # 走成功分支
```

Graph 的真正威力在失败分支。正常运行时的路径是 `check_readme → run_tests → changelog → checklist`——和 Chain 一样。区别在异常时：

```text
Graph 错误处理（README 检查失败）：
  check_readme → 失败 → on_error 跳转 → fix_readme
  路径：check_readme → fix_readme → (终止)
  
  Chain 做不到：Chain 的步骤顺序是 ["检查 README", "运行测试", ...]，
  失败后只能停在第一步。

  Plan-Execute 也可以通过 replan 生成"修复 README"这类步骤，
  但它是运行时动态生成；Graph 的修复路径是预定义的——
  更可预测，也更依赖事先设计和测试。
```

**Graph 的建模成本**：你需要提前想清楚每个节点在所有情况下的跳转目标。对于发布准备这种 4 步任务，Graph 需要定义 6 个节点（包括错误处理节点）。对于更复杂的工单处理流程，节点数可能达到几十个——这时候 Graph 的维护成本就变得显著。

Graph 的威力在于预定义的分支，但它的失败也恰恰源于"预定义"——你只能处理你想过的异常：

| 失败表现 | 典型原因 | 修正方式 |
|---|---|---|
| 错误分支代码从未被测试过 | 主路径几乎不出错→错误分支 3 个月没触发过→第一次触发时 crash（写入了不存在的目录） | 定期注入故障强制触发错误分支（chaos test）；每个错误分支单独做单元测试 |
| 未预见的异常没有分支可走 | Graph 只为"测试失败"和"README 缺失"定义了分支，但"磁盘空间不足"出现时没有任何节点能处理 | 每个 Graph 保留一个 `on_unexpected` 兜底节点——默认行为是暂停并通知用户，而非静默失败 |
| 图膨胀到难以维护 | 最初 6 个节点，半年后变 30 个节点、50 条边，没人能说清全貌 | 子图封装（把"测试相关"的多个节点封装为一个子图）；定期审查死节点（3 个月未触发的节点考虑移除） |
| 状态持久化缺失导致无法恢复 | 执行到第 3 步时进程 crash，重启后不知道执行到哪了、结果是什么 | 每个节点执行后持久化状态（当前节点、已完成步骤、中间结果）；支持从失败节点恢复执行 |
| 过度工程化 | 一个"读取配置文件→打印输出"的三行脚本被建模为 5 节点 Graph | 用 5.4.5 的决策树自检：步骤是否完全固定且无异常分支？→ Chain 就够了 |

### 5.4.5 正常路径都能跑，失败时才看出模式差异

四种模式都讲完了。你可能在想：它们到底有多大区别？下面把同一个发布准备任务用四种模式各跑一次——一次正常情况，一次测试失败——让你亲眼看到区别在哪里。

把发布准备任务用四种模式各跑一次，对比实际表现：

```text
═══════════════════════════════════════════════════════════════════
任务："准备 v1.2.0 版本发布"（所有步骤正常）
═══════════════════════════════════════════════════════════════════

Chain:           ✅ 4/4 完成。路径: 检查README→运行测试→changelog→checklist
Router(发布):     ✅ 4/4 完成。路径同上。
Plan-Execute:    ✅ 4/4 完成。先展示计划，用户确认后执行。
Graph:           ✅ 4 个节点。路径: check_readme→run_tests→changelog→checklist

所有模式都能完成。此时看不出差异。

═══════════════════════════════════════════════════════════════════
任务："准备 v1.2.0 版本发布"（运行测试失败）
═══════════════════════════════════════════════════════════════════

Chain:           ❌ 第 1 步完成，第 2 步失败，第 3-4 步跳过。
                 最终输出: "发布准备失败"。

Plan-Execute:    ⚠️ 第 1 步完成，第 2 步失败 → 重试 1 次 → 仍失败
                 → 触发重规划 → 插入"分析失败原因"步骤
                 → 判断为非阻塞且用户确认跳过后，继续执行 changelog 和 checklist。
                 最终输出: "发布准备完成（测试失败已记录，待修复）。"

Graph:           ✅ check_readme✓ → run_tests✗ → 重试 → 仍失败
                 → on_error 跳转 → retry_tests✓ → changelog✓ → checklist✓
                 最终输出: "发布准备完成"。

三种模式对同一异常的处理完全不同：
- Chain 直接放弃
- Plan-Execute 动态生成应对策略
- Graph 执行预定义的失败恢复路径
```

**模式选择决策树**（实用判断流程）：

```text
你的任务步骤是完全固定的吗？
  ├─ 是 → 步骤会有异常需要分支处理吗？
  │      ├─ 不会 → Chain
  │      └─ 会 → 异常分支可以提前穷举吗？
  │             ├─ 可以 → Graph
  │             └─ 不可以 → Plan-Execute
  └─ 否 → 需要根据输入类型选不同路径吗？
         ├─ 是 → Router
         └─ 否 → Plan-Execute 或 ReAct Loop
```

> **记住**：不要把简单任务硬塞进 Graph。Graph 的问题不是"功能太强"，而是你要维护节点、边、状态、错误分支、恢复策略和测试覆盖；如果任务本来只有三步固定流程，这些成本不会自动带来收益。

### 5.4.6 不只四种模式：并行、编排工人、评估优化各有边界

前面四种模式已经足够解释本章的发布助手场景，但真实项目里还会遇到三种高频变体。它们不一定比 Chain、Router、Plan-Execute、Graph 更高级，只是解决的问题不同。

| 模式 | 解决的问题 | 发布助手中的例子 | 适用边界 |
|---|---|---|---|
| Parallelization | 子任务互不依赖，可以同时执行 | README 检查、依赖漏洞扫描、格式检查可以并行跑，最后汇总结果 | 只有当子任务之间没有写冲突、共享状态少、失败结果可独立解释时才适合 |
| Orchestrator-Workers | 子任务数量和类型运行前不确定，需要一个编排者动态分派 | 迁移项目文档时，编排者先扫描文件列表，再把不同文件分给多个 worker 处理 | 适合开放式任务；成本更高，需要 worker 输出格式统一、编排者能合并冲突 |
| Evaluator-Optimizer | 输出质量需要反复检查和修正 | 生成 release checklist 后，由 evaluator 检查是否漏掉版本号、回滚方案、发布权限，再让 optimizer 修订 | 适合质量可判定的产物；如果评价标准本身模糊，容易变成无意义循环 |

把它们放回发布助手场景，可以这样理解：

```text
用户："帮我做 v1.2.0 发布准备。"

Parallelization:
  同时执行 README 检查、测试、依赖漏洞扫描。
  等所有只读检查完成后，再决定是否生成 changelog。

Orchestrator-Workers:
  编排者发现项目里有 Python、Node.js、Docker 三类发布材料。
  分别派发给三个 worker 检查，再汇总为统一 checklist。

Evaluator-Optimizer:
  先生成 checklist。
  evaluator 检查 checklist 是否覆盖测试、版本号、文档、回滚、权限。
  如果缺项，optimizer 修订 checklist，最多迭代 2 次。
```

这三种模式常和前面的模式组合使用：Router 可以先判断任务类型，再进入 Parallelization；Plan-Execute 可以把某一步拆成 Orchestrator-Workers；Graph 可以把 Evaluator-Optimizer 固化成一个子图。组合的前提仍然一样：先用 trace 和 eval 证明简单模式不够，再增加复杂度。

## 5.5 不要一上来就画图：用故障和评测驱动升级

Planning / Workflow Patterns 可以这样迭代：

| 阶段 | 做什么 | 适合目标 | 不要过早做什么 |
|---|---|---|---|
| V0：裸 ReAct | 模型逐步决定工具，每步临场判断 | 探索任务是否真的需要多步骤 | 不要过早引入 Planning 框架——先确认裸 ReAct 确实不够用 |
| V1：Checklist | 让模型先列出步骤清单，但不强制执行 | 减少漏步骤，验证"是否有固定的步骤模式" | 不要过早要求结构化计划——先确认步骤清单确实有帮助 |
| V2：用户确认计划 | 执行前让用户修改计划，确认后执行 | 提升可控性，让用户参与决策 | 不要过早自动化——用户还无法说清步骤时，先保持手动 |
| V3：结构化 Plan | 每步绑定目标、工具、输入、验收标准 | 支持可追踪的执行和失败定位 | 不要过早追求形式完整——5 步够用就别生成 15 步 |
| V4：动态重规划 | 失败或环境变化时自动调整计划 | 支持复杂任务的自适应执行 | 不要过早信任重规划——先验证"错误分类"和"退出条件"是否可靠 |
| V5：Graph | 将任务节点、状态和分支固化为图结构 | 支持状态持久化、执行恢复、路径回放 | 不要过早固化为 Graph——等流程稳定运行至少一个月后再考虑 |

**每个阶段的升级信号是具体的、可观察的**，不是"系统应该更好"这种模糊感觉：

- V0→V1：用户说"你又漏了步骤"或"你应该先检查 X 再做 Y"——说明任务有固定模式，值得列清单。
- V1→V2：用户在执行过程中频繁打断 Agent（"不对，先别做那个"）——说明用户想要参与决策。
- V2→V3：同类任务执行超过 5 次，每次用户给出相似的反馈（"这一步的验收标准是什么"）——说明需要结构化。
- V3→V4：出现过"某一步失败后 Agent 硬继续或全盘放弃"的故障——说明需要重规划能力。
- V4→V5：任务每周至少执行一次，团队里有新人需要接手执行——此时 Graph 的可回放和可恢复价值才能兑现。

初学阶段建议做到 V2 或 V3。Graph 等到状态和分支真的复杂时再引入。它的代价不是抽象的"复杂"，而是你需要维护节点定义、边条件、状态持久化、恢复策略、错误分支测试和调试工具。

**不要只靠感觉升级模式，要用评测闭环判断。** 准备一组真实任务和失败注入任务，用不同模式各跑一次，再比较结果：

| 评测项 | 记录什么 | 为什么重要 |
|---|---|---|
| 任务完成率 | 最终是否达到用户目标 | 防止复杂模式只是"看起来更完整" |
| 关键步骤遗漏率 | 是否漏掉 README、测试、changelog、checklist 等必要步骤 | 验证 Planning 是否真的减少漏步骤 |
| 失败恢复率 | 注入测试失败、文件缺失、权限错误后是否给出正确处理 | Planning 的价值主要体现在异常路径 |
| 用户确认次数 | 执行前和执行中需要用户确认多少次 | 太少会失控，太多会打断工作流 |
| 平均耗时和成本 | LLM 调用次数、工具耗时、总延迟 | Plan-Execute 和 Graph 的收益必须覆盖成本 |
| 计划偏离次数 | 实际执行是否插入、删除、跳过已确认步骤 | 验证"确认后锁定"是否生效 |
| 可回放性 | 是否能从 trace 看清每步输入、输出、状态和错误 | 决定系统能否调试、审计和交接 |

一个简单的对比模板：

```text
任务集：发布准备 10 条，其中 3 条注入失败

模式              完成率  失败恢复率  平均耗时  用户确认次数  计划偏离次数  是否升级
Chain             70%     20%         低        0             0             否：异常路径太弱
Plan-Execute      90%     80%         中        1.4           0             是：收益覆盖成本
Graph             92%     90%         中        1.1           0             暂缓：建模成本高，任务还不稳定
```

**生产级 Planning 至少要记录这些信息**。否则计划看起来清晰，实际无法恢复、无法审计、无法调试：

| 字段 | 含义 |
|---|---|
| `goal` | 用户原始目标，避免执行中目标漂移 |
| `plan_version` | 计划版本；每次结构变化都生成新版本 |
| `steps` | 每步的名称、工具、输入、输出、依赖、验收标准 |
| `state` | `pending` / `running` / `completed` / `failed` / `skipped` |
| `approval` | 谁确认了哪个版本的计划，确认范围是什么 |
| `risk_level` | `read_only` / `modifies_files` / `external_side_effect` / `irreversible` |
| `requires_approval` | 是否需要人工确认才能执行，特别是写文件、提交、发布、删除、付款等操作 |
| `trace` | 每步工具调用、耗时、错误、重试次数和观察结果 |
| `recovery` | 失败后重试、跳过、重规划、人工接管或终止的规则 |

**设计 Planning 的决策顺序（以发布助手为例）：**

1. **先确认任务是否真的需要 Planning**：用 5.7 的判断标准自检——任务一步能完成？两三步固定流程已足够？如果答案是"是"，停在这里，不要引入 Planning。裸 ReAct 跑 5 次任务，观察是否出现"漏步骤、走偏、无法恢复"的症状。
2. **观察任务的自然形态**：任务步骤是固定的还是随输入变化的？失败后需要重试、跳过还是重规划？用户需要参与决策吗？这些观察决定了你该往哪个方向迭代——而不是一上来就选一个"高级"模式。
3. **从最简单的模式开始**：先 Chain（如果步骤固定且无异常）或 Router（如果按输入类型分流）。只有当你确实遇到了 Chain/Router 无法处理的异常时，才升级到 Plan-Execute 或 Graph。**模式升级的唯一合理理由是你遇到了具体故障，而不是"这个模式更高级"。**
4. **为每种模式定义退出条件**：Chain 在哪步失败时停止？Plan-Execute 最多重规划几次？Graph 的未预见异常往哪个兜底节点走？没有退出条件的 Planning 不是 Planning，是"Agent 在循环里出不来"。
5. **错误路径和主路径同等对待**：Graph 的失败分支如果只是"写完就忘了"，它们在生产环境第一次被触发时就是你最不想面对的时刻。定期注入故障强制触发，每个错误分支单独测试。
6. **计划一旦确认就锁定**：用户确认的是 4 步计划，执行的就不能是 5 步。任何偏离——不管是插入新步骤还是跳过——都需要重新确认。区分"微小调整"（重试延迟）和"结构变更"（插入/删除步骤），前者自动，后者确认。
7. **用真实任务做对比评测**：准备 3 个任务——一个全正常、一个中间步骤失败、一个需要重规划。用不同模式各跑一次，对比：步骤完成率、失败恢复率、用户干预次数、最终输出质量、成本和 trace 可读性。不要靠"感觉 Chain 够用了"做决定。

## 5.6 Planning 失败通常死在异常路径上

### 5.6.1 一眼定位：Planning 失败模式速查表

| 失败模式 | 表现 | 可能原因 | 修正方向 |
|---|---|---|---|
| 计划不可执行 | 步骤看似合理但无法落地 | 没有工具和环境约束 | 计划中绑定工具和验收标准 |
| 计划过细 | 生成大量无意义步骤 | 追求形式完整 | 限制步骤数和粒度 |
| 执行偏离计划 | 执行时忘记原计划 | 状态没有记录当前步骤 | 每步执行后更新状态 |
| 不会重规划 | 工具失败后硬继续 | 没有失败分支 | 增加重规划触发条件 |
| 过度工程化 | 小任务也上 Graph | 技术驱动 | 从 Chain / Router 开始 |
| 用户不买账 | 计划不符合用户真实目标 | 缺少确认节点 | 计划执行前让用户确认 |

### 5.6.2 三个故障故事：死循环、未测分支、擅改计划

速查表告诉你"可能是什么问题"，但实际调试时，症状和根因之间往往隔着几层。下面用三个发布助手场景中发生的故障，展示如何从症状定位到根因并修复。

---

**调试故事一：Plan-Execute 重规划陷入了死循环**

```text
症状：
用户说"帮我做发布准备"。Agent 生成计划，执行到"整理 changelog"时失败
（错误：GitError: 不在 git 仓库中）。重规划生成的新步骤中又有"整理 changelog"，
再次失败，再次重规划……用户等了 3 分钟，Agent 还在循环。

排查过程：
1. 查看 trace → changelog 步骤失败了 8 次，重规划了 6 次
2. 每次重规划都生成了相同的步骤："整理 changelog" + "生成 checklist"
3. 错误信息一直是 "GitError: 不在 git 仓库中"——环境问题，重试不可能解决
4. 重规划逻辑没有判断"失败原因是否可恢复"——对所有错误都一视同仁

根因：
- replan() 函数只看了"失败步骤名"，没分析"失败原因类型"
- GitError 是环境问题（不可恢复），但被当作可重试的错误处理
- 缺少最大重规划次数限制

修复：
1. 错误分类：可恢复错误（网络超时、资源暂时不可用）→ 重试；
   不可恢复错误（配置错误、环境问题、权限不足）→ 跳过并通知用户
2. 增加 max_replan_count=2：最多重规划 2 次，超过后终止并上报
3. 重规划时检查"新步骤是否与前次相同"——如果相同，说明 replan 没有
   产生有效替代方案，应该终止而不是继续
```

---

**调试故事二：Graph 的错误分支从未被测试过**

```text
症状：
"修复 README"节点在生产环境中第一次被触发时，
执行了 30 秒后 crash——它尝试写入一个不存在的目录。
但这个节点在图中已经存在了 3 个月。

排查过程：
1. 查看 Graph 定义 → fix_readme 节点的 action 是 lambda 写的临时代码
2. 这个节点从未在正常流程中被触发（因为"检查 README"几乎不会失败）
3. 开发时只测试了主路径（happy path），错误分支只在代码中存在，
   从未被集成测试覆盖

根因：
- Graph 的失败分支在构建时定义，但测试时几乎只覆盖主路径
- 错误分支代码是"写完就忘了"的代码——没有监控、没有告警
- 正常情况下这些节点永远不会被访问，但一旦被访问就是紧急情况

修复：
1. 注入测试：定期向 Graph 中注入错误，强制触发所有错误分支
   （例如：每周跑一次 chaos test，故意让 README 检查失败）
2. 错误分支节点也必须有完整的日志和监控
3. 每个错误分支节点单独做单元测试
```

---

**调试故事三：用户确认后 Agent 擅自改了计划**

```text
症状：
用户确认了 Agent 生成的 4 步发布计划。执行到第 3 步（"整理 changelog"）时，
Agent 发现 git log 中有一些未提交的改动，它自行插入了一个
"先提交未保存的改动"步骤，导致 changelog 内容包含了不该包含的改动。

排查过程：
1. 查看执行 trace → 第 3 步执行前，Agent 检测到 dirty working tree
2. Plan-Execute 的 execute() 方法中有一段逻辑：
   "如果检测到环境异常，自动插入修复步骤"
3. 用户确认的是 4 步计划，但实际执行了 5 步——用户不知道第 5 步的存在

根因：
- 计划确认后没有锁定：用户确认的是一个"快照"，但执行器把它当
  "参考模板"来用
- "自动插入步骤"的权限太高——它不应该在用户确认后还能修改计划

修复：
1. 计划确认后锁定：confirmed_plan 是 immutable 的，
   任何偏离都需要重新请求用户确认
2. 如果执行过程中发现需要额外步骤，暂停执行并通知用户：
   "发现环境异常（working tree 有未提交改动），
   建议先提交再继续。是否允许我插入这个步骤？"
3. 区分"微小调整"（如重试延迟时间）和"结构变更"（如插入新步骤）——
   前者自动处理，后者必须确认
```

---

**三个故事的共性**：

- **重规划不是银弹**——它需要搭配错误分类（可恢复 vs 不可恢复）和退出条件（最大重规划次数）。
- **错误路径值得和主路径一样认真对待**——Graph 的失败分支如果只是"写完就忘了"，它们在生产环境第一次被触发时就是你最不想面对的时刻。
- **计划一旦确认就应该锁定**。用户确认的是 4 步计划，执行的就不能是 5 步。任何偏离都需要重新确认。

## 5.7 不是每个多步任务都值得 Planning

以下场景不需要复杂 Planning：

- 任务一步就能完成。
- 两三步固定流程已经足够。
- 用户更想要快速结果，不需要先看计划。
- 计划无法验证，只是生成一段中间文本。
- 任务由确定性业务流程稳定承载。
- 引入计划后成本和延迟明显上升，但质量没有提升。
- 没有 trace 或评测数据，无法证明 Planning 真的减少了漏步骤、失败和返工。

实用判断：

```text
如果问题主要是"漏步骤、走偏、无法恢复"，考虑 Planning。
如果问题只是"某一步工具调用错了"，先修工具机制。
```

**回放中暴露的几个关键行为**：

1. **裸 ReAct 不是 Planning**：它没有全局任务视图，容易被局部信息带偏——Agent 在第 1 步就被 changelog 的内容吸引，偏离了"生成 checklist"的目标。
2. **Plan-Execute 的核心价值在于"计划可视化 + 失败决策点"**：用户能提前看到步骤，能在失败时做选择——"跳过这个非阻塞测试"是 Chain 做不到的。
3. **Graph 的价值在于"固化的可靠性"**：正常流程和异常流程都预定义好了，每次执行的行为可预测、可回放。但构建成本高——这个发布 Graph 花了团队一个下午——不该成为默认选项。
4. **迭代顺序很重要**：先裸 ReAct 验证可行性 → 加 Plan-Execute 提升可靠性 → 稳定后才考虑固化为 Graph。不要跳过中间步骤。跳过了，你就会在 Graph 里发现从未测试过的错误分支在生产环境第一次触发时 crash。

**四种模式的核心决策回顾：**

| 模式 | 核心决策 | 做错的表现 |
|---|---|---|
| Chain（5.4.1） | 步骤是否完全固定且无异常分支？ | 非关键步骤失败导致全流程中断；用户想跳过某一步但 Chain 不支持 |
| Router（5.4.2） | 分类是否足够准确？兜底路径是否安全？ | 分类错误导致执行了错误的路径；兜底路径成为"万能垃圾箱" |
| Plan-Execute（5.4.3） | 重规划的退出条件是否可靠？计划确认后是否锁定？ | 重规划死循环；用户确认后 Agent 擅自插入新步骤 |
| Graph（5.4.4） | 是否值得付出建模成本？错误分支是否和主路径一样被测试？ | 30 个节点没人能说清全貌；错误分支 3 个月未触发，首次触发时 crash |

**三个贯穿全链路的主线：**

1. **模式选择由场景驱动，不由"高级感"驱动**：Chain 不是"低级模式"，Graph 不是"高级模式"。如果一个任务的步骤完全固定且不会出意外，Chain 是最优解——它简单、可预测、零维护成本。复杂模式只有在 trace 和 eval 证明它能减少漏步骤、提升失败恢复率或改善交接审计时才值得引入。
2. **异常路径值得和主路径一样认真对待**：大部分 Planning 故障不出在主路径上——正常流程四种模式都能完成。故障出在异常发生时：Chain 直接放弃、Plan-Execute 陷入重规划循环、Graph 的错误分支从未被测试过。评测 Planning 系统时，不要只看"正常时能跑完吗"，要看"异常时怎么处理"。
3. **计划一旦确认就锁定，偏离需要重新确认**：Planning 的可信度建立在用户对计划的确认上。用户确认了 4 步计划，Agent 就不能执行 5 步——无论出于什么"善意"的理由（"我发现环境异常所以帮你多做了这一步"）。任何偏离都需要用户知晓和确认。

回到本章开头的问题——Agent 面对多步骤任务时开始漂移。以发布助手为例：用户给了 4 个步骤的发布任务，裸 ReAct 的 Agent 在第 1 步就开始深入探索 changelog 的细节，偏离了原始目标。Planning 的本质不是让 Agent "更聪明地选择下一步"，而是**在动手之前先建立全局任务结构**——哪些步骤、依赖关系是什么、失败后怎么办——然后在这个结构的约束下执行。结构越确定（Chain），灵活性越低但越可预测；结构越动态（Plan-Execute），灵活性越高但越需要退出条件。

> **这个故事还没完。** Planning 让 Agent 有了全局任务视图，发布准备不再漏步骤。但新问题来了：Agent 执行到"运行测试"这一步时测试失败了，TypeError 清清楚楚地打在日志里——Agent 看到了，却继续写 changelog，没有停下来判断下一步该重试、处理、降级还是停止。它缺少"反馈→分类→决策→处理或停止"的闭环。这就是下一章要讲的 Reflection。

### 本章速记

Planning 的本质不是让 Agent "更聪明地选择下一步"，而是在动手之前先建立全局任务结构——哪些步骤、依赖关系是什么、失败后怎么办——然后在这个结构的约束下执行。三个贯穿全章的主线：

**模式选择由场景驱动，不由"高级感"驱动。** Chain 不是低级模式，Graph 不是高级模式。如果一个任务的步骤完全固定且不会出意外，Chain 是最优解——简单、可预测、零维护成本。用 5.4.5 的决策树自检：步骤固定且无异常分支？→ Chain。需要按输入分流？→ Router。异常模式可穷举？→ Graph。否则 → Plan-Execute。升级模式前，先用 trace 和 eval 证明复杂度能换来更高的完成率、失败恢复率或可回放性。

**异常路径值得和主路径一样认真对待。** 大部分 Planning 故障不出在主路径上——正常流程四种模式都能完成。故障出在异常发生时：Chain 直接放弃、Plan-Execute 陷入重规划循环、Graph 的错误分支从未被测试过。评测 Planning 系统时，不要只看"正常时能跑完吗"，要看"异常时怎么处理"。定期注入故障强制触发错误分支，每个错误分支单独测试。

**计划一旦确认就锁定。** 用户确认了 4 步计划，Agent 就不能执行 5 步——无论出于什么"善意"的理由。任何偏离（插入、删除、跳过步骤）都需要重新请求用户确认。区分"微小调整"（重试延迟时间）和"结构变更"（插入/删除步骤）——前者自动处理，后者必须确认。

## 可运行示例

完成本章学习后，可以对照运行课程五 05-05 的本地 Planning 示例：

- [课程五 05-05 Planning / Workflow Patterns 示例](../examples/course-05-05-planning/README.md)

该示例围绕"发布助手"场景，用 Python 和 Node.js 双版本演示四种 Planning 模式：Chain（固定顺序）、Router（分类路由）、Plan-Execute（计划→执行→重规划）、Graph（节点图+条件边+状态机）。示例包括一个交互式 REPL，你可以在其中切换不同模式执行同一任务，注入失败观察重规划行为，以及对比四种模式对同一异常的不同处理。

示例是教学实现，不是完整生产框架：它用按 Enter 模拟用户确认，Graph 记录可回放路径但不做持久化恢复；Plan-Execute 已包含最大重规划次数和重复重规划检测，避免示例中的失败路径陷入循环。

# Python 版本
cd examples/course-05-05-planning/python
python3 planning_demo.py

# Node.js 版本
cd examples/course-05-05-planning/nodejs

> **四章回顾。** 你现在有了四个视角看 Agent：场景增强（第 1 章）定义了 Agent 在多轮交互中需要哪些额外能力；RAG（第 2 章）解决了"模型不知道的事去查资料"；Memory（第 3 章）解决了"跨会话的状态延续"；Planning（本章）解决了"多步骤任务的组织和执行"。但 Agent 面对复杂任务时还有一个问题没解决——**它看到了反馈信号，却继续原路径往下走，不会停下来决定下一步**。这就是下一章 Reflection 要回答的问题。
