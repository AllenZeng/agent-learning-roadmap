### 4.1 为什么 Agent 越接 RAG、Memory、Tool，反而越不稳定？

你做了一个知识库 Agent。

第一周，它表现很好。你给它接上了 Notion 文档（RAG），它能检索你的笔记回答问题了。你给它加了 Memory，它能记住你偏好简洁回答、代码示例用 Python 了。你给它配了几个工具——读文件、搜索代码库、执行 shell 命令——它也能自己查资料、跑脚本了。

你很满意。它越来越"能干"了。

第二周，问题开始出现。起初只是些小毛病，你以为是偶发：

- 你明明在 System Prompt 里写了"始终使用中文回复"，但它偶尔冒出一段英文。你心想"模型抽风，刷新就好"。
- 它查到了正确的文档，回答里却引用了三天前的旧结论。你觉得可能是 Memory 召回了过期内容。
- 某次它读完一个 2000 行的日志文件后，接下来 5 轮对话都围着日志里一个无关的 WARNING 打转，你问的是代码结构，它却在分析那个 WARNING 的根因。

你开始警觉了。

第三周，问题变成常态：

- System Prompt 里的输出格式约束、安全规则，它经常"忘记"。你不得不在每条消息末尾再重复一遍规则。
- 它明明有工具能查最新文档，却开始凭空猜测——因为上下文太长了，它"懒得"再调工具。
- 每轮回答越来越慢，token 消耗越来越大，但回答质量并没有变好。你看着账单，开始怀疑："加这些能力到底是在帮它，还是在害它？"

你翻看某次出错的 trace。在第 3 步，上下文里塞了这些东西：

- 系统提示词（1200 tokens）
- 用户目标（50 tokens）
- RAG 检索到的 3 个文档片段（共 4200 tokens）
- Memory 召回的 2 条偏好（300 tokens）
- 第 1 步的工具调用结果——一个 1500 行的日志文件（8000 tokens）
- 第 2 步的工具调用结果——代码库搜索结果（2500 tokens）
- 历史消息（1800 tokens）

总计约 18000 tokens。模型在第 15000 个 token 处做决策时，1200 tokens 之前的 System Prompt 约束在注意力机制中已经严重衰减。"使用中文回复"这条指令，模型"看见"了，但没有"注意到"。

**这不是模型突然变笨了。不是 RAG 的问题。不是 Memory 的问题。不是工具的问题。**

**是你把 Agent 的上下文变成了垃圾场。**

每引入一个新能力——RAG、Memory、Tool——你都在往上下文里加东西。但没有人停下来想过：这些东西的组织方式还合理吗？上下文里到底有多少是信号、多少是噪声？

这就是 Context Engineering 要解决的问题。

### 4.2 Context Engineering 到底在解决什么问题？

先退一步，想清楚一个根本问题：**模型到底能看见什么？**

模型没有眼睛。没有耳朵。它看不到你的代码仓库，看不到你的数据库，看不到你的文件系统。它每次推理时唯一能"看见"的东西，是你调用 API 时传进去的那一堆文本——**上下文（Context）**。

```text
模型当前一步的"直接视野" = 你传进去的上下文

模型不知道上下文之外有任何东西。
模型不会"想起来"你上次说过什么（除非在上下文里）。
模型不会"主动去查"什么（除非它决定调用工具，而工具结果又回到上下文里）。
```

但这里有一个关键区分，很多初学者会忽略：

**模型当前一步只能直接看到上下文，但 Agent 系统的完整视野远不止上下文。**

Agent 还可以通过以下方式扩展它的"间接视野"：

- 工具：按需读取文件、查询数据库、搜索代码库——信息不在上下文里，但可以通过工具随时获取
- 外部状态：scratchpad 文件、runtime state、数据库游标——任务进度和中间结果存在上下文之外
- Memory store：长期记忆不常驻上下文，只在需要时召回
- 文件系统：大文件、日志、配置文件不进入上下文，只保留索引
- 子 Agent：把大任务拆出去，子 Agent 在自己的上下文里深挖，只返回浓缩结果

**Context Engineering 的任务不是"把所有视野都塞进模型"，而是决定"当前这一步让模型看见什么"。**

更准确地说：

> Context Engineering 不是在写 Prompt。它是在 Agent 每一步决策前，动态管理"哪些信息进入模型、哪些信息留在外部状态、哪些信息被压缩、哪些信息交给子 Agent 隔离处理、哪些信息需要评测验证"的工程体系。

Prompt Engineering 关注"怎么把指令写清楚"；
RAG 关注"从知识库查什么"；
Memory 关注"跨轮次记什么"；
**Context Engineering 关注"在每一步决策前，模型到底应该看见什么"。**

这四个的关系是：

```text
RAG 负责"查什么"   ──→ 信息生产者
Memory 负责"记什么" ──→ 信息生产者
工具负责"执行什么"  ──→ 信息生产者（工具结果也是上下文的一部分）

Context Engineering 负责"怎么组织这些生产者的产出" ──→ 信息组织者
```

没有组织者，生产者越多越乱。这不是效率问题，是正确性问题——模型真的会"看不到"被淹没在上下文里的关键信息。

随着课程二到课程四引入的能力越来越多，上下文的信息源也在增加：

```text
V0（最小闭环）：
  上下文 = System Prompt + 用户消息 + 历史消息 + 工具结果

V1（接入 RAG）：
  上下文 = System Prompt + RAG 片段 + 用户消息 + 历史消息 + 工具结果

V2（接入 Memory）：
  上下文 = System Prompt + Memory 召回 + RAG 片段 + 用户消息 + 历史消息 + 工具结果

V3（接入 Planning）：
  上下文 = System Prompt + 执行计划 + Memory 召回 + RAG 片段 + 用户消息 + 历史消息 + 工具结果

每一步都在往上下文里加东西。如果不加组织，V3 不是更强的 Agent，而是更混乱的 Agent。
```

所以 Context Engineering 的核心命题不是"要不要往上下文里放东西"，而是：**在当前这一步，哪些信息必须进、哪些信息压缩后进、哪些信息留在外面按需获取、哪些信息应该清理——以及如何验证这些决策是对的。**

### 4.3 Agent 上下文从哪来？——七类来源与全景图

在讨论"怎么管理"之前，先把"管理什么"看清楚。Agent 每一步推理时，可能进入上下文的来源有七类：

```text
用户目标
  ↓
┌─────────────────────────────────────────────────────┐
│                  Context Sources                    │
│                                                     │
│  ① System Instructions     角色、约束、安全规则        │
│  ② User Message            当前用户输入               │
│  ③ Conversation History    历史对话（原文或摘要）      │
│  ④ RAG Knowledge           检索到的外部知识片段        │
│  ⑤ Memory                  召回的用户偏好和长期记忆    │
│  ⑥ Tool Definitions        可用工具的 schema 和描述   │
│  ⑦ Tool Results            工具调用的返回结果          │
│  ⑧ Runtime State           当前计划、进度、scratchpad │
│                                                     │
└─────────────────────────────────────────────────────┘
  ↓
┌─────────────────────────────────────────────────────┐
│               Context Operations                    │
│                                                     │
│  Write     → 把计划、笔记、中间状态写入外部状态           │
│  Select    → 从知识、记忆、工具、历史中选相关内容         │
│  Compress  → 压缩历史、工具结果、长文档、推理轨迹         │
│  Isolate   → 用子 Agent / sandbox 隔离大上下文         │
│  Assemble  → 按分层和预算组装最终 prompt               │
│  Evaluate  → 评测不同策略对成功率、成本、延迟的影响       │
│                                                     │
└─────────────────────────────────────────────────────┘
  ↓
┌─────────────────────────────────────────────────────┐
│                   Model Call                        │
│                                                     │
│  模型基于组装好的上下文，决定下一步：                     │
│  → 回答问题                                          │
│  → 调用工具                                          │
│  → 写入外部状态                                       │
│  → 启动子 Agent                                      │
│                                                     │
└─────────────────────────────────────────────────────┘
```

这张图的核心信息是：**Context Engineering 不是"怎么省 token"，而是 Agent 运行时的信息调度系统。** 你不是在管理一个静态的 Prompt 模板，而是在管理一个动态的信息流——每个来源都在持续产出信息，你需要一个调度层来决定哪些进入模型、哪些留在外面。

本章接下来的结构就是按这个全景图展开的：

- **4.4 入门三件套**：从分层、预算、选择开始——这是 Context Engineering 的基础版本
- **4.5 生产级四类手段**：写入、选择、压缩、隔离——这是完整的生产级工具箱
- **4.6 工具输出**：单独深入工具输出这个最容易爆炸的来源
- **4.7 评测**：如何验证你的上下文策略真的有效

### 4.4 入门三件套：分层、预算、选择

对初学者来说，Context Engineering 可以先从三件事入手：**分层、预算、选择**。这三个够覆盖 80% 的原型到 V1 场景。

但要先说清楚：**这三件是入门版本，不是完整方法论。** 生产级 Agent 还要继续引入压缩、外部状态、上下文隔离和评测（4.5-4.7 节）。

#### 4.4.1 分层：为什么不能把 RAG、Memory、工具结果混在一起？

不要把不同类型的信息混在一起扔给模型。用结构化的方式组织上下文，让模型能区分"这是规则"、"这是参考"、"这是当前状态"、"这是历史"。

**错误做法：**

```text
把所有信息平铺在 messages 数组里：
message[0]: system prompt
message[1]: user message
message[2]: RAG 结果 1
message[3]: RAG 结果 2
message[4]: Memory 召回
message[5]: 工具输出 1
message[6]: 历史消息 1
……
模型看到的是一大坨文本，无法区分哪些是指令、哪些是参考、哪些是历史。
当上下文变长，"Lost in the Middle"效应会让中间的信息被严重忽略。
```

**更好的做法：**

```text
┌─────────────────────────────────────────────────────────┐
│ Layer 0: 系统指令（System / Developer Instructions）      │  ← 最高优先级，永不裁剪
│ - 角色定义、行为约束、输出格式、安全规则                      │
├─────────────────────────────────────────────────────────┤
│ Layer 1: 用户目标 + 当前任务                               │  ← 高优先级
│ - 用户原始消息、确认后的计划、当前子任务                       │
├─────────────────────────────────────────────────────────┤
│ Layer 2: 当前计划 / Scratchpad 摘要                       │  ← 高优先级，动态更新
│ - 当前目标、已完成步骤、待办事项、已验证事实                   │
├─────────────────────────────────────────────────────────┤
│ Layer 3: 任务相关知识（RAG）                               │  ← 中优先级，按相关性过滤
│ - 检索到的文档片段、去重后的知识                             │
├─────────────────────────────────────────────────────────┤
│ Layer 4: 用户相关知识（Memory）                            │  ← 中优先级，按相关性和时效性过滤
│ - 用户偏好、历史决策、项目约定                               │
├─────────────────────────────────────────────────────────┤
│ Layer 5: 工具可用性（Tool Definitions）                    │  ← 中优先级，按当前任务筛选
│ - 当前步骤可能用到的工具 schema 和描述                       │
├─────────────────────────────────────────────────────────┤
│ Layer 6: 最近工具结果（Tool Results）                      │  ← 中优先级，动态变化
│ - 最近 N 步的工具调用与处理后的结果                          │
├─────────────────────────────────────────────────────────┤
│ Layer 7: 历史对话摘要                                     │  ← 低优先级，摘要化
│ - 更早的消息用摘要代替原文                                  │
├─────────────────────────────────────────────────────────┤
│ Layer 8: 外部状态索引（不进入上下文，但模型知道可以获取）       │  ← 不注入
│ - 完整文件路径、数据库行 ID、日志文件位置                     │
└─────────────────────────────────────────────────────────┘
```

每一层用明确的标记分隔，帮助模型的注意力机制定位信息：

```text
<system>
## 角色定义
你是个人知识助手……
## 行为约束
- 使用中文回复
- 不确定时明确说"不确定"
- 回答末尾标注信息来源
</system>

<user_goal>
帮我写一篇关于 Agent Context Engineering 的技术文章
</user_goal>

<current_plan>
## 当前进度
- 步骤 1/4：收集资料 ✅ 已完成
- 步骤 2/4：写大纲 ⏳ 等待用户确认
- 步骤 3/4：写正文 🔜
- 步骤 4/4：review 和修改 🔜

## 已验证事实
- 用户偏好技术文章先给大纲确认
- 目标读者是有 Agent 开发经验的工程师
- 字数要求 3000-5000 字
</current_plan>

<reference_knowledge>
## 相关笔记
[检索到的 3 篇笔记内容……]
</reference_knowledge>

<user_memory>
## 用户偏好
- 技术文章先给大纲确认
- 语气直接，不要营销化
- 代码示例使用 Python
</user_memory>

<tool_results>
## 文件搜索结果
文件搜索 "context window" 返回 12 个结果，其中 3 个与当前任务相关：
1. context_engineering_notes.md - 上下文分层策略 (2.3K tokens)
2. llm_attention_mechanism.md - 注意力机制与上下文位置 (1.8K tokens)
3. production_context_pipeline.py - 上下文管线代码 (0.9K tokens)

其余 9 个结果涉及测试和配置文件，已保存到索引，可通过 read_file 按需读取。
</tool_results>

<history_summary>
用户之前讨论了 Context Engineering 和 RAG 的区别。结论：RAG 负责查，CE 负责组织。
</history_summary>

<external_index>
以下内容未注入上下文，模型可通过工具按需获取：
- 完整日志文件：/tmp/task-log-20260629.txt (8500 tokens)
- 完整检索结果：12 条匹配的完整内容
</external_index>
```

这种分层方式的核心价值是**让关键信息占据注意力的优势位置**。模型对上下文开头和结尾的信息关注度最高（"Lost in the Middle"效应）。把行为约束放在开头、当前任务紧随其后、外部索引放在末尾，是最优的注意力分配。

关键设计决策：**RAG 和 Memory 要分开。** RAG 是任务相关知识——"这个项目的架构是什么"；Memory 是用户相关知识——"这个人偏好什么回答风格"。混在一起会让模型难以区分"事实"和"偏好"。

**Tool Definitions 和 Tool Results 也要分开。** 工具定义告诉模型"你能做什么"，工具结果告诉模型"上一步发生了什么"。一个是能力声明，一个是执行反馈。分开后，当上下文超限时可以先裁剪历史工具结果，而不影响模型对可用工具的理解。

#### 4.4.2 预算：128K 上下文为什么也不能随便塞满？

分层解决了"放在哪"的问题，预算解决"放多少"的问题。

即使模型支持 128K 或 200K token 的上下文窗口，你也不应该填满。原因有三个：

1. **注意力衰减**：上下文越长，模型对中间位置的关注度越低。100K token 不等于"100K 都有效"。研究表明，模型对上下文中间位置的信息召回率显著低于开头和结尾——这就是"Lost in the Middle"效应。

2. **延迟和成本**：上下文越长，每次调用越慢、越贵。每多 1000 token 的输入，延迟和成本都在线性甚至超线性增加。

3. **约束稀释**：System Prompt 的约束在长上下文中被大量中间内容稀释。你精心写的"使用中文回复"，在 80000 tokens 之后的位置几乎等于不存在。

给每一层设定 Token 预算：

| 层 | 预算范围（token） | 说明 |
|---|---|---|
| System Instructions | 1000-2000 | 核心约束，不参与裁剪 |
| 用户目标 + 当前计划 | 500-1000 | 保持完整 |
| RAG 结果 | 2000-4000 | 按相关性截断，标注截断点 |
| Memory 召回 | 500-1000 | 只召回最相关的 3-5 条 |
| Tool Definitions | 500-1500 | 按当前任务筛选相关工具 |
| 工具结果（最近 N 步） | 3000-5000 | 对长结果做摘要后再放入 |
| 历史消息 | 2000-3000 | 早期消息用摘要替代 |
| **总计** | **约 10000-16000** | 远低于模型上限，但信息密度高 |

> **重要提示：这不是固定模板。** 预算是任务类型、模型能力、成本约束、延迟要求共同决定的工程参数。不同场景的合理预算差异很大：

| 场景 | 上下文策略要点 |
|---|---|
| 客服 Agent | 短上下文 + 强规则 + 少量订单信息。总预算 3000-5000 token 足够 |
| 代码审查 Agent | 较大工具结果预算（需要看代码片段），但文件按需读取不预加载 |
| 研究调研 Agent | 子 Agent 并行探索 + 总结回传。主 Agent 上下文严格控制 |
| 个人助手 Agent | Memory selection 风险控制更重要——错误记忆的代价远大于多花 token |
| 企业知识库 Agent | 权限过滤 + 引用溯源优先于 token 优化 |

预算分配的核心原则：**模型本轮决策需要什么，就给什么。模型可能以后需要的，不一定现在给——转成外部索引，需要时再通过工具获取。**

#### 4.4.3 选择：上下文超限时，应该先丢掉谁？

预算定了，但实际运行时还是会超。工具返回了一个 10000 行的日志，RAG 召回了 20 个片段。必须做选择。

选择的核心原则：**按"模型决策对该信息的依赖程度"排序，而不是按"信息本身是否重要"排序。**

一条 Memory 可能很重要，但如果当前步骤不需要它，就不应该占用预算。

```text
裁剪优先级（从先裁到后裁）：

1. 最早的工具输出 → 用摘要替代原文
   "步骤 2 的文件读取返回了 config.py，内容已记录在状态中"

2. 相关性低的 RAG 片段 → 丢弃
   保留 score > 阈值的片段，标注已丢弃数量和原因

3. 早期历史消息 → 用摘要替代
   "前 5 轮对话讨论了 X，结论是 Y"

4. Memory 召回 → 只保留用户明确设定的偏好
   丢弃自动推断的、置信度低的、与当前任务无关的记忆

5. 历史工具结果 → 只保留摘要 + 索引
   完整结果保存到外部文件，模型需要时可再次通过工具读取

永远不裁：
- System Prompt 的核心约束（安全规则、输出格式、角色定义）
- 用户当前消息
- 当前任务的执行计划和进度
```

这里有一个重要的设计选择：**裁剪逻辑应该放在"注入上下文之前"，而不是放在"模型已经读了上下文之后"。** 上下文一旦进入模型的注意力窗口，就已经产生了计算成本和注意力稀释。在注入之前做选择，是对 Token 预算的主动管理，而不是事后的被动截断。

### 4.5 生产级四类手段：写入、选择、压缩、隔离

入门三件套（分层、预算、选择）能解决"怎么组织已有信息"的问题。但生产级 Agent 还需要解决四个更深的问题：

- **写入**：Agent 在执行过程中产生的中间状态放哪？不能都堆在消息历史里
- **压缩**：长对话、长工具结果、长执行轨迹怎么压成能用的摘要？
- **隔离**：遇到需要大量上下文深挖的子任务，主 Agent 亲自读还是派子 Agent？
- **选择**：从海量候选中挑出相关内容（4.4.3 的进阶版）

#### 4.5.1 写入上下文：不要只会读，也要会"记笔记"

长任务 Agent 不能只依赖聊天历史。聊天历史是流水账——每轮对话都记、顺序排列、不做结构化。当任务超过 10 步、跨越数小时甚至数天时，流水账式的历史会让模型迷失。

**需要的外部状态：**

- 当前目标和子任务拆解
- 已完成步骤和关键产出
- 待办事项和依赖关系
- 已验证的事实和结论
- 已排除的方案和原因
- 文件修改计划和进度
- 用户明确确认过的约束和决策
- 工具调用结果索引（完整结果在哪、关键发现是什么）

这些信息不需要每轮都全部进入上下文，但需要被系统保留，并在合适时选择性注入。

**错误做法：**

```text
每轮都把完整的历史工具结果和消息塞回上下文：
message[0]: system prompt
message[1]: 用户：帮我发布一个新版本
message[2]: 助手：[工具调用] 读取 README，返回 3000 tokens……
message[3]: 助手：[工具调用] 检查课程 1，返回 5000 tokens……
message[4]: 助手：[工具调用] 检查课程 2，返回 4500 tokens……
……
message[20]: 用户：确认发布
message[21]: 助手：[工具调用] 执行发布脚本……

到第 20 轮，上下文已经超过 50000 tokens。
模型开始"忘记"用户的发布要求，忽略 README 的约束，甚至重复已完成的步骤。
```

**更好的做法：**

```text
维护一个外部 Scratchpad / Runtime State：

{
  "goal": "发布 GitHub 阶段性版本 v0.5.0",
  "constraints": [
    "README 必须说明未完成章节",
    "license 使用 CC BY-SA 4.0",
    "发布前必须人工确认"
  ],
  "completed": [
    {"step": "检查 README", "result": "已更新，标注了未完成章节", "time": "2026-06-29 10:23"},
    {"step": "review 课程 1-4", "result": "通过，无需修改", "time": "2026-06-29 10:45"}
  ],
  "pending": [
    {"step": "review 课程 5-6", "depends_on": []},
    {"step": "补 examples 目录", "depends_on": ["review 课程 5-6"]},
    {"step": "生成 changelog", "depends_on": ["review 课程 5-6", "补 examples 目录"]}
  ],
  "evidence_index": {
    "README.md": "lines 1-80, 已确认包含未完成章节说明",
    "syllabus.md": "课程 5 章节索引已更新"
  },
  "decisions": [
    {"decision": "使用 CC BY-SA 4.0 而非 MIT", "reason": "用户明确要求", "confirmed_by": "user"}
  ]
}
```

每轮推理时，Scratchpad 中与当前步骤相关的部分被选择性注入上下文。不是全部注入——模型只需要知道"现在做到哪了、下一步要做什么、有哪些约束"，不需要看到完整的执行历史。

Scratchpad 可以通过工具写入文件，也可以通过 runtime state schema 保存。关键设计原则：**只有与当前决策相关的字段暴露给模型，其余字段保留在外部状态中。**

#### 4.5.2 选择上下文：从海量信息中挑出当前步骤需要的

选择是 Context Engineering 中使用频率最高的操作。它的核心问题是：**从 RAG 结果、Memory 召回、工具定义、历史消息中，挑出当前步骤真正需要的。**

这和 4.4.3 节的"裁剪"不同。裁剪是被动的——上下文超限了才裁。选择是主动的——在注入上下文之前就做筛选。

选择的核心策略：

**RAG 选择**（详见第二章）：
- 相关性分数阈值过滤
- 去重：相同内容的不同 chunk 只保留一个
- 冲突检测：当两个片段给出矛盾信息时，优先时间戳更新的或来源更权威的

**Memory 选择**（详见第三章）：
- 只召回与当前任务语义相关的记忆
- 时间衰减：越久远的记忆，需要越高的相关性分数才能被召回
- 用户明确设定的记忆（"记住，我偏好 X"）优先级高于自动推断的记忆

**工具选择**：
- 不是所有工具定义都每轮注入。当前步骤大概率用不到的工具，不注入其 schema
- 工具太多时，按类别分组，先让模型选类别，再注入该类别下的具体工具
- 工具描述重叠时，合并或删除冗余工具——模糊的工具描述本身就是上下文噪声

**历史选择**：
- 最近 3-5 轮保留原文
- 更早的只保留摘要
- 与当前任务无关的历史分支（用户岔开话题聊了几句别的）可以被裁剪

#### 4.5.3 压缩上下文：从聊天流水账到任务状态摘要

预算有限，不可能保留所有原文。压缩就是把长内容变成短摘要，同时保留对决策有用的信息。

三种压缩类型：

| 压缩类型 | 压缩对象 | 保留什么 | 丢弃什么 | 主要风险 |
|---|---|---|---|---|
| **历史压缩** | 多轮对话原文 | 用户目标、确认的结论、约束、关键决策 | 过程性对话、试探性工具调用、已废弃的方案 | 丢失细节导致模型重复提问或做出矛盾决策 |
| **工具压缩** | 长工具结果 | 关键发现、证据索引、下一步建议 | 完整原文、中间过程、冗余信息 | 摘要错误导致模型基于不完整信息做判断 |
| **任务压缩** | 执行轨迹 | 当前计划、已完成/待办、阻塞点、已验证事实 | 具体执行过程、已回退的操作、已解决的分支 | 任务状态漂移——摘要和实际状态不一致 |

**历史压缩示例：**

```text
原始历史（1500 tokens，5 轮对话）：
用户：查一下 context engineering 的论文
助手：[搜索工具] 找到了 8 篇……
用户：只看 2024 年以后的
助手：[过滤] 还剩 3 篇……
用户：第一篇讲了什么
助手：[读取] 这篇讨论了上下文窗口的注意力衰减……
用户：和我们的项目有什么关系
助手：[分析] 你们的上下文管线可以在组装阶段……

压缩后摘要（200 tokens）：
用户在研究 context engineering 相关论文，筛选到 2024 年后的 3 篇。
已讨论第 1 篇关于注意力衰减的内容，用户正在评估如何应用到自己的上下文管线。
关键决策：用户关注组装阶段的优化，暂不关注训练阶段的改进。
待确认：是否继续讨论剩余 2 篇。
```

**工具压缩示例：**

```text
原始工具结果（3000 tokens，grep 返回 120 行匹配）：
[120 行 grep 输出……]

压缩后（500 tokens）：
搜索 "rate_limiter" 在 8 个文件中找到 120 处匹配：
1. src/api/middleware.py - 23 处（限流器主实现）
   L45: class RateLimiter
   L78: def check_limit(self, key: str) -> bool
2. src/api/handlers.py - 15 处（使用限流器）
3-8. 其余 6 个文件共 82 处（测试、配置、文档）
完整结果索引：/tmp/grep-rate-limiter-20260629.txt
```

压缩不是你想象的一个 LLM 调用完事。压缩本身也会出错——摘要可能遗漏关键信息、可能错误理解原意、可能与当前任务状态不同步。所以压缩后一定要保留原始信息的索引，让模型在需要时可以回溯原文。

#### 4.5.4 隔离上下文：什么时候不该让主 Agent 亲自看完所有东西

这是目前比较新的工程实践。核心理念：**不要让一个主 Agent 吃下所有上下文。让子 Agent 在自己的上下文里做局部深挖，最后只把压缩后的结果返回主 Agent。**

每个子 Agent 可以消耗大量 token 深挖某个问题——读几十个文件、分析完整日志、检索大量文档——但只返回 1000-2000 tokens 的浓缩总结给主 Agent。主 Agent 的上下文始终保持轻量。

| 场景 | 不好的做法 | 更好的做法 |
|---|---|---|
| 代码库分析 | 主 Agent 读几十个文件，上下文迅速膨胀到 50K+ | 子 Agent 分模块分析，返回每个模块的结构摘要和风险点 |
| 市场调研 | 主 Agent 读取全部资料，不同主题的信息互相干扰 | 多个研究 Agent 分主题（竞品、用户需求、技术趋势）独立检索，各自返回结构化摘要 |
| Bug 排查 | 主 Agent 保留所有日志和堆栈，上下文被大量技术细节淹没 | 日志分析 Agent 只返回异常模式、时间线、可疑代码路径；主 Agent 基于摘要做判断 |
| 文档 Review | 主 Agent 读完整 50 页文档，注意力分散 | 文档 Agent 提取结构、风险、不一致、建议；主 Agent 只看结构化审查报告 |
| 大量数据探索 | 主 Agent 执行 20 次 SQL 查询，结果全部进入上下文 | 数据分析 Agent 独立探索，只返回关键洞察和可视化建议 |

子 Agent 的上下文隔离本质上是 Context Engineering 的一部分——你在决定"哪些信息不给主 Agent 看，而是交给另一个有自己上下文的执行单元处理"。这和后面 Multi-Agent 章节（第八章）有交集，但立足点不同：Context Engineering 视角关心的是"上下文预算的分配策略"，Multi-Agent 视角关心的是"多角色协作的组织模式"。

### 4.6 工具输出：为什么工具结果最容易把 Agent 带偏？

#### 4.6.1 为什么工具输出最容易爆炸

在所有上下文污染源中，工具输出是破坏力最强的。原因很简单：**你无法完全控制工具输出的大小。**

- System Prompt 是你写的，长度可控。
- RAG 结果是你检索的，你可以只取 top-K。
- Memory 召回是你控制的，你可以设置数量上限。

但工具输出——你让 Agent 读一个文件，它可能是 50 行，也可能是 5000 行。你让它搜索代码库，可能返回 3 个结果，也可能返回 300 个。你让它跑一个命令，stdout 可能只有一行，也可能有几万行。

而且工具输出一旦进入上下文，就会成为下一轮 LLM 调用的输入。模型基于这些（可能无用、可能过时、可能误导的）信息做下一步决策。一个臃肿的工具输出不仅浪费 Token，更危险的是——它可能引导模型在无关信息上钻牛角尖。

回到 4.1 的场景：Agent 读到 README 里的一个技术细节后，偏离了"发布准备"的目标。这就是工具输出污染上下文的经典案例。

#### 4.6.2 结果瘦身三策略

不同工具需要不同的瘦身策略：

**策略一：截断 + 标注（适合文件读取、日志查看）**

```text
原始输出：1500 行日志
处理后：
[前 50 行]
……
[日志中间 1350 行已省略，包含 23 条 INFO、5 条 WARNING]
……
[后 50 行]
总计 1500 行，已显示首尾各 50 行。如需查看完整日志，请指定行号范围。
```

关键：截断后必须标注"截断了什么"和"如何获取剩余内容"。否则模型可能基于不完整信息做错误推断。

**策略二：提取 + 摘要（适合代码搜索、文档检索）**

```text
原始输出：grep 返回 120 个匹配结果
处理后：
搜索 "context window" 在 8 个文件中找到 120 处匹配。按文件分组：

1. src/context/builder.py - 15 处匹配（上下文构建主文件）
   L42: def build_context(layers: List[ContextLayer]) -> str:
   L78: window_budget = self.calculate_budget(model_max_tokens)
   L156: truncated = self.truncate_by_priority(layers, remaining)

2. src/context/compressor.py - 8 处匹配（上下文压缩）
   L23: class ContextCompressor:
   L45: def compress_tool_output(self, output: str, max_tokens: int):

3-8. [其他 6 个文件共 97 处匹配，涉及测试和配置文件]
   使用 "grep -n 'context window' <file>" 查看具体行
```

关键：结构化呈现搜索结果，让模型能快速定位最相关的文件，而不是迷失在 120 行 grep 输出里。

**策略三：结构化 + 过滤（适合 API 调用、数据库查询）**

```text
原始输出：API 返回 200 条用户记录（JSON，每条 20 个字段）
处理后：
查询返回 200 条记录。按状态分组：
- active: 142 条
- inactive: 43 条
- suspended: 15 条

最近 5 条 active 记录的关键字段：
| id | name | email | last_login |
|----|------|-------|------------|
| 1  | ...  | ...   | ...        |

完整数据已写入临时文件 /tmp/query_result_20260629.json
需要进一步分析时，使用 read_file 工具读取指定字段。
```

关键：不要让模型"记住"大量结构化数据。保留摘要和索引，把完整数据放到模型可以通过工具再次访问的地方。

#### 4.6.3 工具返回不只是"短"，还要"可行动"

瘦身是第一步。但好的工具输出不只是短——它应该直接服务下一步决策。

**不好的工具返回：**

```text
搜索 "rate_limiter" 返回 120 行匹配结果：
src/api/middleware.py:45: class RateLimiter:
src/api/middleware.py:78:     def check_limit(self, key: str) -> bool:
src/api/middleware.py:102:    def reset_limit(self, key: str) -> None:
src/api/handlers.py:23:       limiter = RateLimiter(redis_client)
src/api/handlers.py:56:       if not limiter.check_limit(user_id):
……
[115 more lines]
```

模型需要自己解析 120 行，找出重点，决定下一步读哪个文件。每一步都是额外的推理成本和出错机会。

**更好的工具返回：**

```json
{
  "query": "rate_limiter",
  "total_matches": 120,
  "key_files": [
    {
      "file": "src/api/middleware.py",
      "role": "限流器主实现",
      "key_lines": [45, 78, 102, 156],
      "match_count": 23,
      "relevance": "核心"
    },
    {
      "file": "src/api/handlers.py",
      "role": "限流器使用方",
      "key_lines": [23, 56, 89],
      "match_count": 15,
      "relevance": "高"
    }
  ],
  "other_matches": "其余 6 个文件共 82 处匹配（测试、配置、文档）",
  "suggested_next_action": "read_file('src/api/middleware.py', start_line=40, end_line=160)"
}
```

模型拿到这个结果后，不需要在 120 行中自己找重点。它直接知道了"核心实现在哪、关键行号是什么、建议下一步读什么"。这减少了模型的二次判断负担，也降低了出错概率。

#### 4.6.4 工具本身也需要 Context Engineering

Context Engineering 不只管理工具结果，也管理工具定义本身。

工具描述、参数 schema、返回字段说明——这些都会消耗上下文。一个 Agent 如果有 50 个工具，光工具定义就可能占到 5000+ tokens。如果当前任务只需要其中 5 个，那另外 45 个工具的定义就是纯粹的上下文噪声。

而且工具定义重叠或模糊时，模型会出现工具选择错误——两个工具看起来都能做这件事，模型选了一个不合适的。

**工具定义的上下文管理策略：**

1. **按任务筛选工具**：不是所有工具都每轮注入。根据当前步骤的意图，只注入可能用到的工具定义。
2. **工具描述要精确区分**：如果两个工具功能重叠，要么合并、要么在描述中明确差异，"用 A 不用 B 的场景是什么"。
3. **工具太多时分组建模**：比如"文件操作类"、"代码搜索类"、"外部 API 类"。先让模型选择工具组，再注入该组内具体工具。
4. **为工具定义设定预算**：和工具结果一样，工具定义也消耗 token。为工具定义区域设置独立的 token 预算。

#### 4.6.5 清理，不只是压缩：Tool-result Clearing

压缩是把长内容变短。但有些工具结果不应该被压缩后保留——它们应该被清理。

以下类型的工具结果适合清理而非压缩：

- **一次性查询结果**：一次 SQL 查询的 500 行结果。当前步骤有用，但后续 20 轮都不需要。
- **完整原文**：读取了一个 5000 行的 HTML 页面。摘要足够，原文不应该常驻上下文。
- **重复结果**：同样的 grep 跑了 3 次（不同参数），保留最后一次即可。
- **大型二进制/日志**：完整日志文件 20000 行。上下文里只保留异常摘要 + 文件路径索引。

**清理策略：**

```text
1. 工具结果只在接下来 N 轮可见（如 N=3），超出的自动清理
2. 同类工具结果只保留最后一次（新的 grep 结果替换旧的）
3. 大型结果不进入 message history，而是保存到外部 store，上下文中只保留结果 ID / 文件路径
4. 跨轮仍然需要的结果——如用户确认过的决策——应该提取到 Scratchpad，而不是留在工具结果里
```

**清理 vs 压缩的选择：**

| 情况 | 用压缩 | 用清理 |
|---|---|---|
| 结果可能被后续步骤引用 | ✅ | ❌ |
| 结果包含用户确认的决策 | ✅（提取到 Scratchpad） | ❌ |
| 结果是一次性查询 | ❌ | ✅ |
| 结果超过 5000 tokens 且信息密度低 | ❌ | ✅ |
| 同类结果已有更新版本 | ❌ | ✅（清理旧版本） |

一个实用的经验法则：**如果工具结果对下一步决策有用，压缩后保留。如果只是中间过程的产物（"查一下看看"），用完后清理。**

#### 4.6.6 代码骨架：可插拔的结果处理器

不为每种工具写死处理逻辑。用一个可插拔的结果处理器，按工具注册不同的处理策略：

```python
from abc import ABC, abstractmethod
from dataclasses import dataclass, field
from typing import Optional

@dataclass
class ProcessedResult:
    """处理后的工具输出"""
    content: str                # 注入上下文的内容
    original_size: int          # 原始大小（tokens）
    processed_size: int         # 处理后大小（tokens）
    truncated: bool             # 是否被截断
    truncation_note: Optional[str] = None  # 截断说明（供模型理解）
    external_ref: Optional[str] = None     # 完整结果的外部存储路径

class ResultProcessor(ABC):
    """工具输出处理器基类"""
    @abstractmethod
    def process(self, raw_output: str, max_tokens: int) -> ProcessedResult:
        ...

class FileReadProcessor(ResultProcessor):
    """文件读取：保留首尾，中间截断"""
    def process(self, raw_output: str, max_tokens: int) -> ProcessedResult:
        lines = raw_output.split("\n")
        if len(lines) <= 100:
            return ProcessedResult(
                content=raw_output,
                original_size=len(raw_output),
                processed_size=len(raw_output),
                truncated=False
            )
        head = "\n".join(lines[:50])
        tail = "\n".join(lines[-50:])
        omitted = len(lines) - 100
        content = (
            f"{head}\n\n……\n[中间 {omitted} 行已省略]\n……\n\n{tail}\n\n"
            f"[文件共 {len(lines)} 行，已显示首尾各 50 行。"
            f"如需查看完整文件，请指定行号范围调用 read_file。]"
        )
        return ProcessedResult(
            content=content,
            original_size=len(raw_output),
            processed_size=len(content),
            truncated=True,
            truncation_note=f"文件共 {len(lines)} 行，已显示首尾各 50 行"
        )

class SearchResultProcessor(ResultProcessor):
    """搜索结果：结构化分组摘要"""
    def process(self, raw_output: str, max_tokens: int) -> ProcessedResult:
        # 分组、去重、按文件聚合……
        # 返回结构化的搜索结果摘要
        ...

class APIResultProcessor(ResultProcessor):
    """API 结果：统计摘要 + 关键样本 + 外部存储"""
    def process(self, raw_output: str, max_tokens: int) -> ProcessedResult:
        # 解析 JSON，提取统计信息，保留少量关键记录
        # 完整数据写入临时文件，返回文件路径作为 external_ref
        ...

class GenericProcessor(ResultProcessor):
    """通用处理器：简单截断"""
    def process(self, raw_output: str, max_tokens: int) -> ProcessedResult:
        char_limit = max_tokens * 4  # 粗略估算 1 token ≈ 4 chars
        if len(raw_output) <= char_limit:
            return ProcessedResult(
                content=raw_output,
                original_size=len(raw_output),
                processed_size=len(raw_output),
                truncated=False
            )
        truncated = raw_output[:char_limit]
        return ProcessedResult(
            content=(
                f"{truncated}\n\n"
                f"[输出已截断，原始长度 {len(raw_output)} 字符，"
                f"已截断至约 {max_tokens} tokens。如需完整内容，请使用更精确的查询参数。]"
            ),
            original_size=len(raw_output),
            processed_size=len(truncated),
            truncated=True,
            truncation_note=f"输出超出限制，已截断至约 {max_tokens} tokens"
        )

# 注册处理器
result_processors = {
    "read_file": FileReadProcessor(),
    "search_codebase": SearchResultProcessor(),
    "call_api": APIResultProcessor(),
    # 未注册的工具使用 GenericProcessor
}
default_processor = GenericProcessor()

def process_tool_output(tool_name: str, raw_output: str, max_tokens: int) -> ProcessedResult:
    """对工具输出进行处理后再注入上下文"""
    processor = result_processors.get(tool_name, default_processor)
    return processor.process(raw_output, max_tokens)
```

这个骨架的核心设计思路：

1. **每种工具独立处理逻辑**：文件读取和代码搜索的输出形态完全不同，不能一刀切
2. **处理结果透明**：告诉模型"原始大小、处理后大小、是否截断、完整结果在哪"，模型需要这些元信息来做判断
3. **截断说明是给模型看的**：`"文件共 1500 行，已显示首尾各 50 行"`——模型知道信息不完整，它会决定是继续基于现有信息推理，还是调用工具获取更多内容
4. **external_ref 支持清理语义**：完整结果保存在外部，上下文里只放摘要。后续轮次可以通过清理机制移除工具结果，只保留 external_ref 索引

### 4.7 如何评测上下文策略是否真的有效？

Context Engineering 不能只靠感觉——"我觉得分层之后回答变好了"不是工程语言。你需要用指标和数据来验证策略是否真的有效。

**核心评测指标：**

| 指标 | 说明 | 如何测量 |
|---|---|---|
| Task Success Rate | 任务完成率——Agent 是否达成了用户目标 | 人工标注或 LLM-as-Judge 评估 |
| Constraint Following Rate | 输出格式、安全约束、语言约束遵守率 | 自动检测（正则、格式校验）或人工标注 |
| Tool Call Accuracy | 工具选择是否正确、调用参数是否合理 | 对比 golden tool call 或人工评估 |
| Retrieval Usefulness | 注入的 RAG/Memory 内容是否被有效使用 | 检查模型回答是否引用了注入内容（citation match） |
| Context Utilization | 注入的内容中有多少真正影响了回答 | 对比有/无该内容的输出差异 |
| Token Cost | 输入 token、输出 token、总成本 | API 返回的 usage 字段 |
| Latency | 模型响应延迟（含上下文组装时间） | 端到端计时 |
| Context Drift Rate | 多轮后目标偏移比例——Agent 是否还在做最初要做的事 | 每 N 轮对比当前行为和初始目标 |
| Recovery Rate | 出错后能否恢复——遇到上下文相关问题后重新对齐的比例 | 标注"偏离"事件，统计恢复比例 |

**一个简单的 A/B 评测设计：**

不需要一开始就做全自动评测管线。固定 30 个代表性任务样本，手工跑四组对比：

```text
A 组：全量上下文（V0 baseline）
  所有信息原封不动注入，不做分层、不做预算、不做压缩

B 组：分层 + 预算（V3）
  上下文按 4.4.1 的分层结构组织，每层有预算上限

C 组：分层 + 预算 + 工具压缩（V4）
  在 B 组基础上，为高频工具注册结果处理器

D 组：分层 + 预算 + 工具压缩 + Scratchpad（V4+）
  在 C 组基础上，引入外部 Scratchpad 管理任务状态

比较维度：
- 完成率（任务是否成功完成）
- 平均输入 token 数
- 平均耗时
- 约束遵守率（如输出格式、语言要求）
- 人工评分（1-5 分，由你或同事评估回答质量）
```

跑完 30 个样本 × 4 组 = 120 次评测，你就会对"哪种策略在你的场景里真正有效"有数据支撑，而不是凭感觉调参。

评测的核心价值不是"证明我的策略最好"，而是**发现什么策略对你的任务类型确实有用、什么策略是过度设计**。你可能发现 B 组（分层+预算）已经把完成率从 70% 提到 90%，C 组（加工具压缩）提到 93%，但 D 组（加 Scratchpad）不升反降——因为你的任务很少超过 5 步，Scratchpad 的额外复杂度反而引入了新的失败模式。

这就是评测的意义：**从"我觉得应该加"变成"数据说需要加"。**

### 4.8 进化路线：从"全塞进去"到"上下文调度系统"

Context Engineering 不需要一步到位。和所有增强能力一样，它也应该从最小版本开始迭代：

| 阶段 | 做了什么 | 新增的能力维度 | 适用场景 |
|---|---|---|---|
| **V0：全塞进去** | 所有信息原封不动注入上下文，超限时用 tokenizer 截断 | 无 | 原型验证、单步简单任务 |
| **V1：加个预算** | 给每类信息设定最大 token 数，超出时简单截断 | 预算 | 信息源不超过 3 个的原型 |
| **V2：工具输出处理** | 为高频工具注册专门的结果处理器，长输出先压缩/清理 | 预算 + 工具压缩 | 工具调用频繁、输出容易爆炸 |
| **V3：分层 + 优先级** | 上下文结构化分层，超限时按优先级裁剪而非简单截断 | 分层 + 预算 + 选择 | 信息源 3+ 个，任务多步骤 |
| **V4：Scratchpad + Compaction** | 引入外部状态管理任务进度；对历史和工具结果做结构化压缩 | +写入 + 压缩 | 单任务超过 10 步，或跨越多轮 |
| **V5：子 Agent 隔离** | 大上下文子任务交给子 Agent 独立处理，只返回浓缩结果 | +隔离 | 需要深度分析大量文件的复杂任务 |
| **V6：Eval-driven Optimization** | 建立上下文策略的评测体系，用数据驱动策略选择和参数调优 | +评测 | 持续优化上下文质量的生产系统 |

大多数项目的合理目标是 **V3（分层+优先级）**。V4-V6 适合以下情况：

- 单任务经常超过 10 步 → 考虑 V4
- 经常需要深度分析大量代码/文档 → 考虑 V5
- 上下文质量直接影响核心业务指标 → 考虑 V6

一个重要的经验法则：**每当你往上下文里加一个新信息源，同时就加一段处理逻辑。** 不要只加生产者，不加组织者。

### 4.9 常见失败症状与救治

这些症状不会同时出现，但几乎每个从原型向生产演进的 Agent 项目都会遇到其中 2-3 个：

| 你的症状 | 业内常见说法 | 底层解释 | 救治方向 |
|---|---|---|---|
| **约束遗忘**：模型违反 System Prompt 中的明确约束 | Instruction Dilution | 约束在长上下文中被后续内容"淹没"，注意力权重被分散到大量无关 token 上 | 将关键约束在上下文末尾重复一次；缩短上下文总长度；将约束放在注意力最集中的开头位置 |
| **中间迷失**：模型对上下文中间位置的信息视而不见 | Lost in the Middle | 注意力机制对序列中间位置的关注度显著低于开头和结尾 | 将关键信息放在开头或结尾；对中间内容做摘要并前移到靠前位置；避免将重要信息埋在上下文中间 |
| **工具结果中毒**：模型被一次无关的工具输出引导跑偏，后续多轮都围着无关内容打转 | Context Poisoning | 工具输出未经过滤直接注入，大段无关内容占据了注意力的主导位置 | 为工具输出注册处理器；设定单次工具输出的 Token 上限；对大型结果使用"摘要 + 外部索引"模式；实施 tool-result clearing |
| **信息冲突**：RAG 片段和 Memory 召回给出了矛盾的偏好或事实 | Context Clash | 不同信息源缺乏去重和冲突解决机制，模型面对矛盾信息时随机选择或缝合 | 加入跨源去重逻辑；信息冲突时优先采用时间戳更新的或来源更权威的；当检测到冲突时显式标注"存在矛盾"让模型知晓 |
| **上下文膨胀**：每轮都追加工具结果，上下文逐渐逼近窗口上限 | Context Rot / Bloat | 历史工具输出未被压缩或清理，上下文像滚雪球一样越来越大，每轮都更慢更贵 | 对超过 N 步的工具输出做摘要替换；设置上下文使用率告警阈值（如 70%）；引入 tool-result clearing；将任务状态外移到 Scratchpad |
| **重复注入**：同一条知识同时出现在 RAG 结果和 Memory 中 | Context Redundancy | 信息源之间缺乏协调，重复内容占据宝贵的 token 预算却不增加信息量 | 在 Context Assembly 阶段做跨源去重；相同内容优先使用来源质量更高的版本；对重复内容标注"已在 [来源] 中出现" |

### 4.10 什么时候不需要过度设计？

Context Engineering 也遵循"按需引入"原则。以下情况不需要复杂的设计：

1. **单步简单任务**：用户问一句、模型答一句，没有工具调用、没有多轮历史。直接传 System Prompt + 用户消息就够了。

2. **信息源单一**：只有 System Prompt 和用户消息，没有 RAG、Memory、工具输出。没有"组织"的需求。

3. **上下文远低于模型上限**：总 Token 数不到模型上限的 30%，且任务简单、步数少。过度设计上下文管线是浪费工程资源。

4. **原型验证阶段**：还在验证"这个方向能不能 work"。V0 的"全塞进去"完全可以接受。Context Engineering 是优化手段，不是前提条件。

5. **短生命周期 Agent**：这个 Agent 只用几次就废弃了——比如一次性数据迁移脚本。不值得为它搭建上下文管线。

一个判断信号：**当你发现自己开始在 System Prompt 里写"请务必遵守第 X 条规则"时，说明上下文已经长到模型需要被"提醒"才能记住约束了。这是该引入 Context Engineering 的信号。**

反过来说，如果你的 Agent 现在跑得好好的——约束被遵守、工具结果没跑偏、用户没抱怨回答质量下降——那就继续用当前的上下文策略。**Context Engineering 是治病的手段，不是保健品。没问题的时候别乱吃。**

### 本章速记

> 上下文是模型当前一步决策的直接视野，而 Agent 的完整视野由上下文、外部状态、工具、长期记忆共同组成。Context Engineering 不是"把上下文写短一点"，而是在每一步决策前动态管理哪些信息进入模型、哪些留在外部状态、哪些被压缩、哪些交给子 Agent 隔离处理、哪些需要评测验证。入门从三件事开始：分层（每类信息各就各位，不要把 RAG、Memory、工具结果混在一起）、预算（128K 上下文也不能随便塞满，因为注意力衰减、延迟成本、约束稀释）、选择（超限时按模型对该信息的依赖程度裁剪，不重要的先出局）。生产级还需四类手段：写入（Scratchpad 管理任务状态，不要只依赖聊天历史）、选择（主动从海量信息中挑出当前步骤需要的）、压缩（历史压缩、工具压缩、任务压缩三种类型，各有风险和适用场景）、隔离（子 Agent 在自己的上下文深挖，只返回浓缩结果给主 Agent）。工具输出是最容易爆炸的上下文来源——不只做瘦身，还要让它"可行动"（直接服务下一步决策）、管理工具定义本身、区分注入和外部化、对一次性结果做清理而不只是压缩。用评测数据（完成率、约束遵守率、上下文利用率、成本延迟）而不是感觉来验证策略是否有效。进化从 V0（全塞）到 V6（eval-driven），但大多数项目停在 V3（分层+优先级）就够了。当你开始在 Prompt 里写"请务必遵守第 X 条规则"时，就该引入 Context Engineering 了。

> **四章回顾。** 你现在有了四个视角看 Agent：RAG 让它能获取外部知识，Memory 让它能记住用户和任务状态，Context Engineering 让多源信息在上下文中有序组织而非互相淹没。这三者共同构成了 Agent 的"上下文增强"能力——解决的是"决策所需的信息从哪来、怎么管"的问题。
>
> 但 Agent 面对复杂任务时还有另一个问题没解决：**信息有了，但多个步骤之间的执行顺序、依赖关系、失败恢复怎么组织？** 裸 ReAct 循环没有任务结构，多步骤任务容易漂移、出错后不知道从哪恢复。
>
> 这就是下一章 Planning 要回答的问题。
