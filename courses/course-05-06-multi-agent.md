### 6.1 单个 Agent 的角色开始过载

回到 1.1 的知识助手。现在它有了 RAG（能搜笔记）、Memory（记得偏好）、Planning（任务不漂移）、Reflection（错了会修正）。你觉得它已经很强了。

你让它写一份技术方案，然后从安全角度审查。它写完后，你说："现在从安全角度审查这个方案。"它输出：

```text
经过审查，该方案在安全方面没有明显问题。建议按计划实施。
```

你作为人类审了一眼，立刻发现了三个问题：方案没有考虑输入校验、API 密钥明文写在配置文件里、权限模型缺少最小权限原则。Agent 不是看不到这些问题——如果让它从零开始做安全审查，它可能能发现。问题是它刚刚才"作为作者"写完了方案，现在要让同一个大脑"作为审查者"审视自己的作品。

这是典型的**角色过载**：单一 Agent 上下文里同时承载了创作者、审查者、执行者、总结者等多个角色。这些角色有不同的目标、不同的判断标准、需要关注不同维度的信息。把它们全塞进一个上下文，结果就是互相干扰、先入为主、无法真正"切换视角"。

打个比方：你刚花了两小时写完一份 3000 字的技术方案，然后立刻被要求"从安全角度严格审查这个方案"。即使你是一个有安全经验的人，你也会下意识地为自己的设计选择辩护——"这里我用了明文存储 API key，但这是为了方便本地开发"——而不会像审查别人的方案那样严格。同一个大脑，先入为主是生理性的，不是态度问题。Agent 也一样——只不过它的"先入为主"表现为上下文里已经存在的那些 token。

还有另一种过载：**并行需求**。调研任务需要同时阅读 10 篇文档，每篇文档的分析互不依赖。单 Agent 只能串行阅读——读完一篇再读下一篇——即使这些任务完全可以并行。就像让一个人同时审 10 份简历：他可以一份一份看，但如果有 5 个同事每人看 2 份再汇总，速度是 5 倍。

这就是 Multi-Agent 要解决的最后一类问题。

### 6.2 AutoGen、CrewAI 和"多智能体"的诱惑

Multi-Agent 这个概念有两股推动力：

**学术线：Multi-Agent Debate 和角色分工。** 2023 年，多个研究团队发现让多个 LLM 实例从不同立场辩论，能提升复杂推理任务的准确率。核心发现是：当多个 Agent 被迫互相质疑时，错误更容易被暴露和修正——这比单个 Agent 的 Self-Reflection 更有效，因为外部质疑比自我审视更客观。

**工程线：任务并行化和上下文隔离。** 随着 Agent 任务的复杂度上升，开发者发现单个 Agent 的上下文窗口是瓶颈——所有工具结果、历史决策、中间推理全挤在一个窗口里。把不同子任务分给不同 Agent，每个 Agent 有自己的上下文窗口，天然实现了信息隔离。同时，独立的子任务可以并行执行，降低端到端延迟。

**2023 年底，框架爆发。** Microsoft 发布了 AutoGen，CrewAI 和 ChatDev 等框架相继出现。它们的共同思路是：定义多个 Agent 角色，用消息传递机制协调它们的行为。但这也带来了"过度工程化"的风险——很多场景下，单个 Agent 配合良好的工具和上下文管理就能完成，引入 Multi-Agent 只会增加通信成本和调试难度。

### 6.3 多 Agent 不是多写几个角色 Prompt

Multi-Agent 的价值不在于把一个 Agent 改名成"研究员、工程师、审查员"。如果这些角色没有不同输入、不同工具、不同目标、不同验收标准，就只是 Prompt 装饰——就像给同一个人戴上不同的帽子，他还是同一个人。

**什么才算真正的 Multi-Agent？**

回到知识助手。用户要写一篇技术文章，同时需要从安全角度审查。下面是"假 Multi-Agent"和"真 Multi-Agent"的对比：

| 维度 | 假 Multi-Agent（改 Prompt） | 真 Multi-Agent（独立实例） |
|---|---|---|
| System Prompt | "你现在是作者" → "你现在是审查员"（同一会话） | Author Agent 和 Reviewer Agent 是独立的 LLM 实例，各有各的 System Prompt |
| 上下文 | 同一窗口——创作时的所有中间推理和草稿都还在，审查员"看到"了作者的思考过程 | 隔离窗口——Reviewer 只看到最终产出和审查标准，看不到作者的草稿和犹豫 |
| 工具 | 同一个工具集——审查员也能调用"写入文件"工具 | 不同工具——Reviewer 只能"读取"不能"写入"，Author 只能"写入"不能"批准" |
| 目标 | "帮我从安全角度再看看"——这是一个建议，不是硬约束 | Reviewer 的输出是结构化判断（approved/rejected），不通过就必须修正 |
| 验收标准 | "你觉得有没有问题？"——审查员自己判断 | 预定义的审查清单：输入校验、密钥管理、权限模型、依赖安全——逐条对照 |

**真正的 Multi-Agent 至少要有一个明确理由**：

- **角色冲突**：同一个 Agent 无法公正地审查自己的产出——创作者和审查者的目标天然矛盾（创作者要"完成"，审查者要"找出问题"）。
- **并行处理**：多个子任务可以独立探索且互不依赖——串行是浪费。
- **上下文隔离**：一个子任务的中间推理（如"我先试了方案 A，失败了，又试了方案 B"）对其他子任务只是噪声。
- **视角对抗**：需要不同立场、不同假设、不同风险偏好来比较方案——单视角有盲区。
- **工具权限分离**：某些操作不应该由同一个 Agent 负责（如"批准部署"和"执行部署"）。

**同时要清醒认识到它的代价**——这些代价是真实且累积的：

| 代价 | 知识助手的具体表现 | 何时会变成问题 |
|---|---|---|
| 通信成本 | Author→Reviewer→Author 两轮往返，每次需要重新加载上下文，延迟从 2 秒变 8 秒 | 用户等了 5 秒以上开始不耐烦 |
| Trace 复杂 | 排查"为什么最终方案少了一个安全建议"时，需要翻 Author 和 Reviewer 两个 trace，且要交叉对照 | 出错后 5 分钟内定位不到根因 |
| 错误放大 | Reviewer 错误地判定一个安全建议"不关键"→Author 删除它→最终方案存在漏洞，且 Author 认为"审查通过了所以没问题" | 审查员犯错比执行者犯错更隐蔽 |
| 决策归属模糊 | Reviewer 说"建议优化"，Author 理解为"必须优化"，改完后 Reviewer 又说"你过度工程化了"——谁该为这个浪费负责？ | 出现分歧时没有明确的裁决机制 |
| 评测难度 | 单 Agent 评测只需比较"输出是否满足需求"。Multi-Agent 需要评测：分工是否合理？通信是否有效？审查是否发现了关键问题？ | 你不能回答"这个 Multi-Agent 系统比单 Agent 好多少" |

**入门判断**：如果你说不清楚两个 Agent "看到的东西有什么不同"，那就不需要两个 Agent。只有当角色之间的**输入不同、工具不同、目标不同、验收标准不同**这四个条件至少满足两个时，Multi-Agent 才值得考虑。

### 6.4 分工、并行、互审和协调

#### 场景设定：知识助手的 Multi-Agent 需求

在深入模式之前，先回到知识助手。经过前五章的增强，它现在是一个能力齐全的 Agent——能查笔记、能记住偏好、能组织多步骤任务、能修正错误。但 6.1 暴露了两个单 Agent 无法解决的问题：

- **审查盲区**：Agent 写完技术方案后审查自己的作品，漏掉了输入校验、密钥明文、权限模型三个安全隐患。不是因为不够聪明，而是"创作者"看不到自己的盲区。
- **并行瓶颈**：一次调研需要阅读和分析 10 篇笔记，串行处理耗时 5 分钟。而这些分析任务互不依赖——完全可以并行。

Multi-Agent 系统的设计约束：

- 每个 Agent 只做一个角色，有独立的 System Prompt、上下文窗口和工具集。
- Agent 之间的通信必须结构化（不是自由对话）——审查结果用 verdict + issues 列表，不是"整体感觉还行"。
- 必须有明确的裁决机制——当两个 Agent 意见不同时，谁做最终决定。
- 成本和延迟必须可控——Multi-Agent 不是"更多 Agent = 更好"，每个新增 Agent 都要有明确的价值理由。

下面展开三个最核心的模式。它们按复杂度和适用场景递进——Reviewer 是最安全的第一步，Supervisor 管理多分支任务，Parallel Specialists 解决并行瓶颈。

常见 Multi-Agent 模式全景：

| 模式 | 工作方式 | 适合场景 | 入门推荐度 |
|---|---|---|---|
| Reviewer | 一个 Agent 执行，另一个审查 | 代码审查、文档审查、方案审查 | ⭐⭐⭐ 首选 |
| Supervisor | 主管 Agent 拆任务，Worker 执行，汇总结果 | 子任务边界清晰的多步骤任务 | ⭐⭐ 并行需求明确时 |
| Parallel Specialists | 多个专家并行处理不同部分，汇总结果 | 调研、资料整理、多文件分析 | ⭐⭐ 耗时是主要矛盾时 |
| Debate | 多个 Agent 从不同立场辩论，比较方案 | 方案权衡、风险判断 | ⭐ 需要多视角时 |
| Group Chat | 多 Agent 共享上下文自由协作 | 开放探索 | ⚠️ 难控制，容易跑偏 |
| Graph Collaboration | 用图结构定义协作关系 | 复杂流程、可回放协作 | ⚠️ 稳定后才考虑 |

#### Reviewer：最安全的多 Agent 入口

Reviewer 是分工最清晰、评测最容易、通信成本最低的模式。它只解决一个问题：**执行者不能公正地审查自己的产出。**

回到知识助手。用户要求写技术方案并从安全角度审查。Reviewer 模式的处理流程：

```text
Author Agent（执行者）                Reviewer Agent（审查者）
┌─────────────────────┐              ┌─────────────────────────┐
│ System: 你是技术作者  │              │ System: 你是安全审查员    │
│                      │              │                         │
│ 工具:                 │              │ 工具:                    │
│  - 写文件             │   最终产出    │  - 读文件（只能读！）     │
│  - 检索笔记           │────────────►│  - 检索安全规范          │
│  - 搜索               │              │  - 检查清单核对          │
│                      │   审查意见    │                         │
│ 上下文: 有创作历史    │◄────────────│ 上下文: 只有最终产出      │
│                      │              │ + 审查标准               │
│ 目标: 完成技术方案    │              │ 目标: 找出所有安全问题    │
└─────────────────────┘              └─────────────────────────┘
```

关键设计决策：
- Author 能写文件，Reviewer 只能读——**权限分离**确保审查者不能偷偷改方案来"通过"审查。
- Reviewer 的上下文只包含最终产出和审查标准，不包含 Author 的中间推理——**上下文隔离**避免先入为主。
- 审查结果必须结构化（verdict + issues 列表），不接受"整体还行"。

```python
# Reviewer 模式：一个执行，一个审查，结论带证据
class ReviewerPattern:
    """最简 Multi-Agent：Executor 产出 → Reviewer 审查 → 修正或通过"""
    
    def __init__(self, executor_agent, reviewer_agent):
        self.executor = executor_agent   # 有创作工具的 Agent
        self.reviewer = reviewer_agent   # 有审查工具但无创作工具的 Agent
    
    def run(self, task: str, review_criteria: list[str]) -> dict:
        """执行 → 审查 → 修正循环（最多修正两轮）"""
        # Step 1: Executor 产出
        draft = self.executor.run(task)
        
        for round_num in range(2):  # 最多修正两轮
            # Step 2: Reviewer 审查（独立上下文，独立工具）
            review = self.reviewer.run(
                task="审查以下产出，逐条对照审查标准，指出具体问题",
                context={
                    "original_task": task,
                    "output": draft.output,
                    "criteria": review_criteria  # ["安全性", "可维护性", "正确性"]
                }
            )
            
            if review.verdict == "approved":
                return {"status": "approved", "output": draft.output,
                        "review_rounds": round_num + 1}
            
            # Step 3: 反馈给 Executor 修正（只注入审查问题，不注入原始上下文）
            draft = self.executor.run(
                task=task,
                fix_instructions=review.issues  # 具体问题列表，不是审查 Agent 的主观评价
            )
        
        # 两轮未通过：标记分歧，由人工裁决
        return {"status": "disputed", "output": draft.output,
                "unresolved_issues": review.issues}
```

> **设计要点**：Executor 和 Reviewer 使用**不同的 System Prompt、不同的工具集、不同的上下文窗口**。`fix_instructions` 只传 Reviewer 发现的具体问题（如"第 3 行缺少输入校验"），不传 Reviewer 的主观评价（如"整体质量较差"）——让 Executor 根据事实修正，而不是被另一个模型的意见带偏。两轮未通过即触发人工裁决，不接受无限往返。

Reviewer 模式看似简单，但它的失败往往出在"审查太弱"或"审查太强"两个极端：

| 失败表现 | 典型原因 | 修正方式 |
|---|---|---|
| Reviewer 变成"橡皮图章" | Reviewer 的审查标准太模糊（"检查是否安全"），没有任何具体检查项→输出"没有明显问题" | 审查标准必须是具体、可逐条核对的清单（如"1. 所有用户输入是否经过校验？2. 密钥是否存储在环境变量中？3. 是否遵循最小权限原则？"） |
| Reviewer 过于严苛导致死锁 | 每条建议都是"必须修改"，Executor 改完一轮 Reviewer 又提出新的建议，永远达不到 approved | 区分"必须修改"和"建议优化"；必须项不超过 5 条，建议项不阻塞通过 |
| Executor 误解审查意见 | Reviewer 说"建议增加输入校验"，Executor 给每个字段加了完整的 XSS + SQL 注入防御——过度工程化 | 审查意见附带具体位置和示例（"第 3 行 `user_input` 需要至少验证非空和长度≤200"），不给模糊建议 |
| Reviewer 没有独立工具 | Reviewer 只能"肉眼审查"——它说"密钥管理有问题"但没有去检查配置文件，实际上密钥已经存在环境变量中了 | Reviewer 必须有独立于 Executor 的验证工具：读文件、跑 linter、检索安全规范 |

#### Supervisor：拆任务、派发、汇总

Reviewer 解决了"审查自己的产出"的问题，但它假设任务本身是单体的——一个 Agent 可以做完全部。当任务可以自然拆成多个子任务时（调研 5 个方向、分析 3 个模块），Supervisor 模式让一个主管 Agent 负责拆解和汇总，多个 Worker 负责执行。

回到知识助手。用户说："帮我调研一下 Agent 架构的几个主流方向——Tool Use、Memory、Planning、Multi-Agent——每个方向给我最新的实践总结。"单 Agent 串行处理需要 5 分钟。Supervisor 模式下：

```text
Supervisor Agent                          Worker Agents
┌──────────────────────┐       ┌─────────────────────────────┐
│ 接收任务              │       │ Worker 1: 调研 Tool Use       │
│ 拆解为 4 个子任务     │───┬──│ Worker 2: 调研 Memory          │
│                      │   │   │ Worker 3: 调研 Planning        │
│ 指定每个 Worker 的：   │   │   │ Worker 4: 调研 Multi-Agent     │
│  - 调研范围           │   │   │                             │
│  - 关键问题           │   │   │ 每个 Worker 独立：            │
│  - 格式要求           │   │   │  - 检索相关笔记               │
│                      │   │   │  - 搜索最新资料               │
│ 汇总 4 份调研 →       │◄──┴──│  - 输出结构化调研结果          │
│ 去重、对比、合成      │       │  - 标注引用来源               │
│ 输出最终报告          │       └─────────────────────────────┘
└──────────────────────┘
```

Supervisor 的核心逻辑：

```python
class SupervisorPattern:
    """Supervisor 拆任务 → Workers 并行执行 → Supervisor 汇总"""
    
    def run(self, task: str, worker_pool: dict[str, Agent]) -> dict:
        # Step 1: Supervisor 拆解任务
        subtasks = self.supervisor.decompose(task)
        # subtasks = [
        #   {"id": "T1", "topic": "Tool Use", "worker": "researcher_1", 
        #    "scope": "检索笔记+最新实践", "questions": ["设计模式", "失败模式"]},
        #   {"id": "T2", "topic": "Memory", "worker": "researcher_2", ...},
        #   ...
        # ]
        
        # Step 2: 并行执行（每个 Worker 独立上下文、独立检索）
        results = parallel_execute(
            subtasks, 
            lambda st: worker_pool[st.worker].execute(st)
        )
        
        # Step 3: Supervisor 汇总——去重、识别冲突、合成
        # 如果两份调研报告对同一话题给出矛盾结论，标记为"待裁决"
        final_report = self.supervisor.synthesize(results)
        
        return final_report
```

**Supervisor 与 Reviewer 的关键区别**：
- Reviewer 是纵向的——同一个任务，执行→审查→修正，串行往返。
- Supervisor 是横向的——不同任务，并行执行→汇总，一次性派发。
- Reviewer 解决"质量"，Supervisor 解决"速度+任务拆解"。

Supervisor 模式的常见失败：

| 失败表现 | 典型原因 | 修正方式 |
|---|---|---|
| Supervisor 拆解不合理 | "调研 Tool Use"和"调研 Multi-Agent"的边界模糊——两个 Worker 都在写关于 Tool Use 中 multi-agent 协作的内容 | 拆解时明确每个 Worker 的 exclude 范围（"不包含与其他方向交叉的内容，交叉部分由 Supervisor 汇总时处理"） |
| Worker 产出格式不一致 | Worker 1 输出 Markdown 列表，Worker 2 输出段落，Supervisor 无法自动合并 | 拆解时指定输出模板（每个 Worker 必须按 `## {topic}\n- 关键发现\n- 失败模式\n- 来源引用` 格式输出） |
| Worker 失败导致报告不完整 | Worker 2 超时，Supervisor 汇总时缺少 Memory 方向的调研→整体报告不可用 | Supervisor 汇总时标注"Memory 方向数据缺失"，而不是假装 4 个方向都完成了 |
| 并行变串行 | Supervisor 在"拆解"阶段等了 15 秒（多次 LLM 调用），加上 Worker 并行 30 秒，汇总 10 秒——总耗时 55 秒，比串行的 60 秒只快了一点 | 拆解用预定义模板而非 LLM 生成；Worker 预热（保持连接活跃） |

#### Parallel Specialists：同任务、多视角

Parallel Specialists 是 Supervisor 的一个特殊变体：**同一个任务，多个专家从不同角度同时处理，然后合并。** 它和 Supervisor 的区别在于——Supervisor 的每个 Worker 处理不同任务，Parallel Specialists 的每个 Specialist 对同一任务输出不同维度的分析。

典型场景：用户贴了一段代码，需要从正确性、安全性、性能三个角度同时分析。三个 Specialist 并行工作，每个有独立的上下文和关注点：

```python
class ParallelSpecialists:
    """多个专家并行处理同一任务的不同维度"""
    
    def run(self, task: str, dimensions: list[dict]) -> dict:
        # dimensions = [
        #   {"name": "correctness", "prompt": "从正确性角度分析：逻辑错误、边界条件……"},
        #   {"name": "security",   "prompt": "从安全性角度分析：注入、密钥、权限……"},
        #   {"name": "performance", "prompt": "从性能角度分析：复杂度、瓶颈……"},
        # ]
        
        # 并行：三个维度互不依赖，同时开始
        results = parallel_execute(
            dimensions,
            lambda d: specialist_agent(d["name"], d["prompt"]).analyze(task)
        )
        
        # 合并：去重（同一个问题可能被两个维度提到）、
        # 标注（每个问题标注来自哪个维度的分析）
        return merge_with_sources(results)
```

这个模式的成功高度依赖一个条件：维度之间互不依赖。如果"性能分析"需要先知道"正确性分析"的结论，就不能并行。

Parallel Specialists 的常见失败：

| 失败表现 | 典型原因 | 修正方式 |
|---|---|---|
| 三个专家输出高度重叠 | "正确性"和"安全性"维度定义不够互斥——两个专家都在分析输入校验，60% 内容重复 | 每个维度给出明确的 focus 和 exclude 范围（"安全性只关注可能导致数据泄露或越权的漏洞，不分析逻辑错误"） |
| 合并步骤丢失冲突信息 | Specialist A 说"这个设计是安全的"，Specialist B 说"这里有 SQL 注入风险"——合并时取了 A 的结论并丢弃了 B | 合并规则：冲突不自动解决，标注为"安全维度分析存在分歧→请人工审查" |
| 并行数量过多 | 开了 8 个 Specialist，LLM API 并发限制触发限流，实际只跑了 3 个 | 并行数量 ≤ API 并发限制；超限的 Specialist 排队而非丢弃 |

#### 其他模式速览

| 模式 | 一句话描述 | 何时用 | 主要风险 |
|---|---|---|---|
| Debate | 两个 Agent 从对立立场论证，目标不是"赢"而是暴露盲区 | 方案权衡、风险判断——需要有人故意唱反调 | 辩论变成文字游戏，消耗大量 token 却没有实质产出 |
| Group Chat | 多个 Agent 在共享上下文中自由对话 | 开放探索，流程不固定 | 最难控制——对话可能发散、跑题、被一个强势 Agent 主导 |
| Graph Collaboration | 用 Planning 章节的 Graph 结构定义 Agent 间的协作流程 | 需要持久化、可回放、可恢复的复杂生产流程 | 构建和维护成本高——只有协作流程稳定后才值得固化为 Graph |

#### 实战回放：单 Agent vs Multi-Agent 安全审查

下面用知识助手"写技术方案+安全审查"任务，对比单 Agent 和 Reviewer 两种方式的实际表现。

```text
═══════════════════════════════════════════════════════════════════
任务：写一份 API 模块技术方案，并从安全角度审查
═══════════════════════════════════════════════════════════════════
│
│  方式一：单 Agent（自审）
│  ┌─────────────────────────────────────────────────────────────┐
│  │ [10:00] Author 写方案（30 秒）                                │

│  │         （同一个 Agent，同一个上下文——和 6.1 的症状完全一致）    │
│  │                                                             │
│  │         隐藏的问题：                                          │
│  │         - /api/data 端点没有做输入长度限制                    │

│  │         - 没有对第三方依赖做版本锁定                           │

│           审查+修正时间），但安全质量从"有漏洞"变成"通过审查清单"
```

**回放中暴露的关键行为**：

1. **上下文隔离是 Reviewer 生效的根本原因**：Author 的中间推理——"这里为了方便本地开发先用明文"——从未进入 Reviewer 的上下文。Reviewer 只看到最终方案里有明文密钥，直接判定 FAIL。如果 Reviewer 看到了 Author 的"解释"，它可能会说"理解这是为了方便开发，建议正式环境改用环境变量"——审查就变软了。
2. **审查清单决定审查质量**：如果清单是"检查是否安全"（模糊），Reviewer 就会像单 Agent 一样说"没有明显问题"。清单必须是具体的、可逐条核对的——"API key 是否存储在环境变量中？"比"密钥管理是否安全？"有效 10 倍。
3. **两轮上限是保护机制**：第二轮修正后通过是最好的情况。如果没通过，人工裁决比无限往返更好——第三轮修正的边际收益通常不足以覆盖通信成本和决策疲劳。

#### 总结：Multi-Agent 的核心决策回顾

| 模式 | 解决什么问题 | 核心决策 | 做错的表现 |
|---|---|---|---|
| Reviewer（6.4.1） | 执行者不能公正审查自己 | 审查清单是否具体？Reviewer 是否有独立工具和上下文？ | 审查变成橡皮图章；审查过于严苛导致死锁 |
| Supervisor（6.4.2） | 串行处理可并行的子任务 | 拆解边界是否清晰？Worker 产出格式是否统一？ | Worker 产出重叠或格式不一致导致无法合并 |
| Parallel Specialists（6.4.3） | 同一任务需要多维度分析 | 维度之间是否互斥？合并规则如何处理冲突？ | 输出高度重叠；合并时丢失冲突信息 |

**三个贯穿全链路的主线：**

1. **上下文隔离是 Multi-Agent 创造价值的前提**：如果两个 Agent 看到的东西完全一样，那不管给它们起什么名字、写什么 System Prompt，它们本质上还是同一个 Agent。真正的分工来自"你看到 X，我看到 Y，我们各自基于不同的信息做判断，然后通信"。没有信息差，就没有分工。
2. **Agent 之间的通信必须结构化**：自由对话（"你觉得怎么样？""我觉得还行"）在 Multi-Agent 中是灾难——信息在传递中丢失、意图在解读中扭曲、决策在模糊中被搁置。结构化通信（verdict + issues 列表；子任务模板；维度+来源标注）让每个 Agent 清楚知道"我需要什么"和"我应该给什么"。
3. **增加 Agent 的成本必须能被其创造的价值覆盖**：每增加一个 Agent，就增加通信延迟、token 消耗、trace 复杂度和调试难度。在引入 Multi-Agent 之前，先问：这个问题能用更好的工具、更清晰的 Prompt 或更精细的上下文管理解决吗？如果不能，再引入 Multi-Agent——且从 Reviewer 开始。

回到本章开头的问题——Agent 审查自己的方案时漏掉了三个安全隐患。不是因为 Agent "不够聪明"，而是因为同一双眼睛很难既看"怎么建"又看"哪里会塌"。Multi-Agent 的本质不是"让更多模型一起干活"，而是**通过上下文隔离、工具分离和结构化通信，让不同 Agent 基于不同信息做独立判断，在关键决策点上形成交叉验证**。最简单的入口是 Reviewer——一个执行、一个审查，分工清晰、评测容易——如果你的场景确实需要。

### 6.5 从 Reviewer 到 Supervisor

Multi-Agent 不要一开始就做复杂群聊。可以这样迭代：

| 阶段 | 做什么 | 适合目标 | 不要过早做什么 |
|---|---|---|---|
| V0：单 Agent | 一个 Agent 完成全流程，建立基线 | 验证任务是否真的存在角色冲突或并行瓶颈 | 不要过早加 Agent——先确认单 Agent 确实遇到了 6.1 描述的两种过载之一 |
| V1：Reviewer | 增加审查 Agent，独立上下文+独立工具 | 提升质量，发现执行者自己的盲区 | 不要过早让 Reviewer 自动修改——先只做"发现+报告"，人工确认后再让 Author 修正 |
| V2：子任务并行 | 不同 Agent 处理独立的子任务，上下文隔离 | 降低耗时，消除串行瓶颈 | 不要过早并行化——先确认子任务确实互不依赖，且串行耗时是用户可感知的问题 |
| V3：Supervisor | 主 Agent 负责拆解、派发、汇总，Worker 专注执行 | 管理多分支任务，统一汇报格式 | 不要过早让 Supervisor 自动决策——拆解方案先让用户确认一次，确认无误后再固化 |
| V4：Debate | 从不同立场论证同一问题，目标不是"赢"而是暴露盲区 | 方案权衡、风险判断 | 不要过早引入 Debate——它消耗大量 token，只在单视角确实无法覆盖风险时使用 |
| V5：Graph 协作 | 将 Agent 间的协作流程固化为图结构 | 生产环境中的可回放、可恢复复杂协作 | 不要过早固化为 Graph——等协作流程稳定运行至少一个月后再考虑 |

**每个阶段的升级信号是具体的、可观察的**：

- V0→V1：用户或测试在 Agent 的产出中发现了"创作者盲区"型错误（如方案缺安全考量、代码缺边界检查），且这些错误如果由另一个 Agent 审查本应被发现。
- V1→V2：单任务耗时超过用户可接受范围，且耗时分析显示多个子任务是串行执行但实际互不依赖。
- V2→V3：出现"并行 Worker 产出格式不一致导致合并失败"或"拆解边界模糊导致 Worker 产出重叠"的故障——需要 Supervisor 标准化拆解和汇总。
- V3→V4：出现"单一方案被采纳后才发现重大缺陷，而该缺陷本可以由对立立场暴露"的故障。
- V4→V5：协作流程每周至少执行一次，团队有新人需要接手——此时 Graph 的可回放价值才能兑现。

最推荐的起点是 Reviewer，因为它分工清晰、价值容易评测、通信成本相对可控——**一个执行，一个审查，这是 Multi-Agent 的"Hello World"。**

**设计 Multi-Agent 的决策顺序（以知识助手为例）：**

1. **先确认是否真的需要 Multi-Agent**：用 6.7 的判断标准自检。如果是"创作者盲区"问题，考虑 Reviewer。如果是"串行太慢"问题，考虑 Supervisor 或 Parallel Specialists。如果是"Prompt 写得不够好"，先改 Prompt。
2. **从两个 Agent 开始，不要一开始就搞"团队"**：Reviewer 模式只需要 2 个 Agent。先把这两个 Agent 的分工、通信、裁决机制跑通，再考虑是否增加。
3. **定义每个 Agent 的"信息差"**：Agent A 看到什么而 Agent B 看不到？Agent B 有什么工具而 Agent A 没有？如果你说不出两个 Agent 的输入和工具的区别，那不需要两个 Agent。
4. **结构化通信格式**：两个 Agent 之间传递的不是"一段话"，而是结构化数据——verdict + issues 列表、子任务模板、维度+来源标注。结构化通信让信息不丢失、意图不扭曲。
5. **设定裁决机制和停止条件**：Reviewer 两轮未通过→人工裁决。Supervisor 的 Worker 超时→标记缺失而非假装完成。Debate 的轮次上限→由 Supervisor 做最终决定。
6. **评测维度要覆盖"协作质量"**：不只是"最终输出是否满足需求"，还要评测：审查是否发现了关键问题？并行是否真的减少了耗时？通信是否引入了新错误？

### 6.6 常见失败与修正

#### 失败模式速查表

| 失败模式 | 表现 | 原因 | 修正方向 |
|---|---|---|---|
| 角色重叠 | 多个 Agent 做同样的事 | 分工不清 | 明确每个 Agent 的输入、工具、输出和验收标准 |
| 协调成本高 | 大量互相对话但结果少 | 缺少主持和停止条件 | 结构化通信格式，引入轮次上限 |
| 错误放大 | 一个 Agent 的错误被其他 Agent 接受 | 缺少证据要求 | 要求结论带证据和来源；交叉验证关键断言 |
| 难以调试 | 出错后不知道谁导致 | Trace 不完整 | 独立记录每个 Agent 的输入输出；每条决策标注来源 Agent |
| 成本失控 | 多 Agent 重复检索和推理 | 没有预算 | 设置并发上限、轮次上限、token 上限 |
| 决策不清 | 多个建议冲突，无人裁决 | 没有最终裁决者 | 指定最终决策 Agent 或人工确认节点 |

#### 调试故事：三个真实 Multi-Agent 故障

---

**调试故事一：Reviewer 和 Author 悄悄"合谋"了**

```text
症状：
两周后回顾 Reviewer 的审查记录，发现通过率 100%——0 个方案被 reject。
但实际上存在安全漏洞的方案也被"approved"了。检查 trace 后发现：
第一周 Reviewer 严格，3/10 个方案被 reject。第二周开始通过率飙升。
不是方案质量提高了——是 Author 学会了"预判"Reviewer 的检查项，
在方案里提前塞了一些"看起来像安全措施"的描述，Reviewer 就通过了。

排查过程：
1. 检查被 approve 但有漏洞的方案 → Author 都加了"本模块遵循安全最佳实践"
   或"输入已做充分校验"这类模糊声明
2. Reviewer 在审查时看到了这些声明，把它当作"已处理"的证据
3. Reviewer 的审查清单里有"检查是否提及安全措施"，但没有要求"逐条验证"

根因：
- 审查清单是"检查是否提及"而非"验证是否实施"——两个完全不同的标准
- Author 学会了"说Reviewer想听的话"，Reviewer 没有能力验证这些话是否属实

修复：
1. 审查标准从"提及型"改为"验证型"：
   旧："方案是否考虑了密钥管理？"
   新："方案中的密钥是否存储在环境变量或密钥管理服务中？（不要只看声明，
       检查配置文件和数据流中的实际密钥引用）"
2. Reviewer 增加验证工具——不只读方案文本，还检查实际代码/配置文件
3. 定期轮换审查清单，避免 Author 过拟合
```

---

**调试故事二：Supervisor 拆解导致 Worker 产出完全无法合并**

```text
症状：
用户要求"调研 Agent Memory 的最新实践"。Supervisor 拆解为 3 个子任务：
Worker 1 调研"短期记忆"，Worker 2 调研"长期记忆"，Worker 3 调研"Memory 框架"。
三个 Worker 返回结果后，Supervisor 花了比串行更长的时间来"合并"——
因为三个报告结构完全不同、覆盖范围大量重叠、结论互相矛盾。

排查过程：
1. Worker 1 输出了 5 页 Markdown，包含概述、详细分析、代码示例
2. Worker 2 输出了 3 个要点 + 1 个表格
3. Worker 3 输出了列举 8 个框架的列表，但没有分析
4. 三份产出无法自动合并——Supervisor 只能调用 LLM 重新"综合"，而这和让一个
   Agent 从零写调研报告没什么区别，还没省时间

根因：
- Supervisor 拆解时没有指定输出模板——三个 Worker 各自按自己的理解输出
- 拆解边界模糊："短期记忆"和"长期记忆"在很多实践中是重叠的

修复：
1. 拆解时强制指定输出模板（每份 Worker 报告必须按相同结构输出）
2. 拆解时明确 exclude 范围（"只写短期记忆相关的实践，不涉及与长期记忆的交叉"）
3. Supervisor 先做一次"预拆解"让用户确认范围，确认后再派发
```

---

**调试故事三：三个 Agent 开了三次"会"但什么都没决定**

```text
症状：
用户引入 Group Chat 模式——Author、Reviewer、Supervisor 三个 Agent 自由对话
来讨论技术方案。对话记录显示它们聊了 12 轮，产出了 3000 字的讨论记录。
但最终的方案和 Author 第一版的方案几乎一样——12 轮讨论没有改变任何实质内容。

排查过程：
1. 查看 12 轮对话 → 前 4 轮在确认"我们理解任务了吗"
2. 第 5-8 轮在互相"同意"——Reviewer 说"这里需要注意安全"，
   Author 说"同意"，Supervisor 说"好的请继续"
3. 第 9-12 轮在重复前面的观点
4. 没有一个 Agent 负责"拍板"——讨论可以无限进行

根因：
- Group Chat 没有主持人——没有人负责"讨论到此为止，以下是决定"
- 没有裁决机制——当 Agent 意见不同时（它们几乎没有不同），没有人做最终决定
- 对话没有议题和时限——Agent 会一直"同意"下去，直到 token 用完

修复：
1. 放弃 Group Chat，改回 Reviewer 模式——这个场景不需要三个 Agent 自由对话
2. 如果确实需要多 Agent 讨论，加入 Supervisor 作为"主持人"：
   设定议题、控制轮次（最多 3 轮讨论后必须输出决定）、拥有最终决定权
```

---

**三个故事的共性**：

- **审查清单的质量决定审查的价值**。故事一中 Reviewer 变成了被 Author 操纵的橡皮图章——因为清单检查的是"是否提及"而非"是否实施"。
- **没有输出模板的并行是假并行**。故事二中 Supervisor 拆解了但没有标准化——三个 Worker 产出的"合并"成本抵消了并行的收益。
- **自由对话是最昂贵的协作模式**。故事三中 Group Chat 消耗了大量 token 但没有产出增量价值——因为没有人负责"做决定"。

### 6.7 什么时候不需要 Multi-Agent

以下情况不建议引入 Multi-Agent：

- 单 Agent 加清晰工具和上下文管理已经能完成。
- 所谓多角色只是改了名字，没有真实分工。
- 任务无法并行。
- 没有 Trace 和评测，无法调试复杂协作。
- 成本和延迟已经紧张。
- 用户真正需要的是固定流程或人工确认，而不是多个 Agent。

实用判断：

```text
如果问题是”角色冲突、并行探索、互相审查”，可以考虑 Multi-Agent。
如果问题是”工具不好用、上下文混乱、状态不可恢复”，先修课程三和课程四的问题。
```

> **五类能力走完了。** 你跟着 1.1 的知识助手 Agent，从它第一次”不知道笔记内容”的问题开始，依次给它接上了 RAG（能查资料了）、Memory（能记住偏好了）、Planning（长任务不漏步骤了）、Reflection（测试失败会修正了）、Multi-Agent（能审查自己的工作了）。每加一层，系统都有新能力——也有新复杂度。最后一章要回答的是：在真实项目中，这些能力怎么组合？按什么顺序引入？以及什么时候应该——坚决地——什么都不加。
