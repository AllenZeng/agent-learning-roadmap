# 课程一：初识 Agent

## 课程导言

如果你正在读这段话，你很可能已经用过 ChatGPT、Claude 或类似的产品。你问它问题，它回答你——有时候惊艳，有时候胡说八道。你对它的感觉可能是："这东西挺聪明的，但它能真的帮我做点什么吗？"

**本课程要回答的就是这个问题。**

这一课不讲原理，不讲实现，不讲架构。它的目标只有一个：**让你看见 Agent，建立直觉。**

什么是"看见"？打开 Claude Code，让它帮你重构一个文件；打开 Cursor，看它如何理解你的代码库；打开 Kimi，让它做一份行业调研。你不需要理解它内部是怎么工作的——就像你不需要理解发动机原理就能感受到一辆车的加速性能。

什么是"直觉"？用完之后，你能自然地判断：

- 这个产品是真正在做决策，还是只是在走固定的流程？
- 它的哪一步让你觉得"聪明"，哪一步让你觉得"笨"？
- 下一步到底是谁在决定——用户、固定流程，还是模型？
- 如果让你来设计，你会改什么？

这些直觉，是后面所有学习的燃料。对初学者来说，没有体验支撑的原理很容易变成空洞术语；带着直觉去学原理，每一句话你都能想起一个具体的画面。

**本课是感性认知层，不讲实现细节。** 它为课程二的范式理解和课程三的最小闭环提供产品观察样本。

准备好了吗？让我们开始第一次接触。

---

## 学习目标

学完本课，你将能够：

1. **说出 Agent 是什么** — 区分它与 Chatbot、Workflow 的本质边界，而不是被"用了 LLM"的表象迷惑
2. **用"谁决定下一步"判断 Agent 成色** — 一眼看穿产品是模型在做决策，还是预设规则在驱动
3. **用 7 维度模板拆解任何 Agent 产品** — 从感性体验走向结构化分析，不再停留在"好用 / 不好用"
4. **解释用户为什么信任或不信任 Agent** — 理解惊喜、失控、可靠感背后的设计原因

## 第一章：什么是 Agent？

### 1.1 先看一个场景

你让 ChatGPT 帮你订一张从北京飞上海的机票。

ChatGPT 会说："抱歉，我无法帮你订机票。你可以去携程或航空公司官网查询。"

你让一个真正的 Agent 做同样的事。

Agent 会自己打开浏览器，搜索航班信息，比较不同航空公司的价格和时间，找到合适的航班后填写订单页面——你在旁边看着就行。到了支付那一步，它会停下来问你："找到一趟下午 3 点、票价 850 元的航班，要我帮你下单吗？"

这两种反应的差异，就是 Agent 和 Chatbot 的本质区别。

再来一个更日常的场景——假设你要写一份竞品分析报告。注意，这里不要只看产品名字，因为同一个产品可能既有普通问答模式，也内置了搜索、文件、代码执行等 Agent 能力：

**普通问答模式的做法**：你问它，它给你列出竞品名字、分析框架、写作建议。你需要自己打开浏览器搜索、阅读文章、提取信息、组织成文。

**带 Agent 能力的做法**：你告诉它"帮我做一份关于 XX 领域的竞品分析"，它自己搜索各家官网和新闻报道，提取关键信息，对比产品差异，生成一份结构化报告，标注信息来源。你只需要在关键节点（比如最终发布前）确认。

**差别在哪？** 不在于它叫不叫 ChatGPT、Claude 或 Kimi，而在于这次交互里它只是告诉你怎么做，还是能够调用工具、观察结果，并根据中间信息继续决定下一步。

### 1.2 Agent 不是"更聪明的聊天机器人"

这是理解 Agent 的第一个关键认知：**Agent 和 Chatbot 不是"智能程度"的差别，而是"能力维度"的差别。**

Chatbot 工作在一个维度上：**语言空间**。它接收文字，处理文字，输出文字。它可以理解你、回应你、建议你——但所有这些都在对话框之内。

Agent 工作在两个维度上：**语言空间 + 行动空间**。它不只是理解你的意图，还能在真实世界中产生行动——搜索、读写文件、调用 API、执行代码、操作浏览器等。行动的结果反过来影响它的下一步决策。

用一句话概括：

> **Agent 是能够围绕目标持续决策和执行的系统，而不仅仅是回答问题。**

这句话里每个词都很重要：

- **围绕目标**：Agent 不是"有问必答"，而是始终朝向一个目标推进。它会判断自己离目标还有多远，是否走偏了。
- **持续决策**：不是一次判断就结束，而是在每一步之后根据新的信息重新判断"接下来做什么"。
- **执行**：不只是"建议你怎么做"，而是真的去做了——调用工具获取外部信息、处理数据，或在需要时改变外部世界的状态。

### 1.3 Agent vs Chatbot vs Workflow

理解了 Agent 是什么之后，你需要区分三个经常被混淆的概念：**Chatbot**、**Workflow**、**Agent**。

#### Chatbot：负责对话和回答

Chatbot 的核心能力是理解和生成语言。用户输入一段话，它返回一段话。即使接入了搜索引擎或知识库，Chatbot 仍然是"你说什么，我回答什么"——它不会主动制定计划，不会自己决定"接下来要做三件事"。

**核心特征**：下一步永远是"等待用户输入"。

#### Workflow：按固定流程执行

Workflow 是预定义的执行链。每一步做什么、数据如何传递、什么条件触发什么分支，在设计阶段就确定了。虽然 Workflow 的某些节点可能调用 LLM（比如生成文案、分类意图），但 LLM 在这里只是"一个功能模块"，不是"决策者"。决策权属于那个设计好的流程。

**核心特征**：下一步由"预设规则"决定。

典型的 Workflow 例子：智能客服系统
1. 识别用户意图（5 种预定义类型）
2. 根据意图调用对应的知识库
3. LLM 润色生成回答
4. 如果解决不了，转人工

整个流程中，LLM 参与了步骤 1 和步骤 3，但它没有决策权——什么时候转人工、用哪个知识库，都是预设规则决定的。

#### Agent：根据目标、状态和反馈动态决定下一步

Agent 和前面两者的根本差异在于：
**下一步不由用户决定，不由预设规则决定，而是由模型根据当前目标、任务状态和环境反馈动态决定。**

**核心特征**：下一步由"模型动态决策"。

典型的 Agent 例子：Claude Code。给定"帮我找出这个项目的所有 bug 并修复"
- 它自己决定先读哪些文件
- 读到代码后决定运行什么诊断命令
- 看到诊断结果后决定一个现象是不是 bug
- 修复后决定如何验证
- 验证失败后决定是重试还是换策略

这个链条中每一步的决策者都是模型本身，而不是预设流程。

#### 三者的核心差异

| 对比维度 | Chatbot | Workflow | Agent |
|---------|---------|----------|-------|
| 下一步由谁决定 | 用户 | 预设规则 | 模型动态决策 |
| 能否调用工具 | 通常不能，或仅有搜索增强 | 可以，但调用时机由规则决定 | 可以，调用时机由模型判断 |
| 任务范围 | 单轮或多轮对话 | 固定流程内的任务 | 开放性目标 |
| 容错方式 | 无法行动，无所谓"执行错误" | 异常分支由预设规则处理 | 根据反馈动态调整策略 |
| 典型产品 | ChatGPT 基础版 | 传统智能客服、RPA 流程 | Claude Code、Codex、Trae |

#### 企业常见落地形态：Agentic Workflow

现实中，很多企业系统不会直接做成"完全自主的 Agent"，而是采用更可控的混合形态——**Agentic Workflow**。

Agentic Workflow 的核心思路是：

**Workflow 负责主流程边界，Agent 负责局部任务决策。**

它不是简单地"在某个节点调用一次 LLM"。普通 Workflow 里的 LLM 节点通常只完成一个固定动作，比如分类、摘要、改写；输入、输出和下一步流向仍由流程规则决定。Agentic Workflow 的关键差异在于：某些节点内部拥有一个受控的小闭环，模型可以围绕局部目标选择工具、观察结果、调整策略，并把判断结果返回给主流程。

这是当前企业 AI 转型里很常见的实践方向。原因很简单：企业场景通常需要权限、审计、合规、稳定交付和人工确认，不能把整条业务链路都交给模型自由发挥；但在信息理解、材料筛选、异常判断、下一步建议等局部环节，模型又能提供比固定规则更强的适应性。

比如一个内容审核系统：
- 整体流程是 Workflow：接收内容 → 自动规则检查 → 模型初筛 → 人工复核
- 但"模型初筛"这一步不是简单分类，而是一个受控 Agent 子任务：模型自己决定从哪些角度审核、是否需要检索相关政策、证据不足时是否请求更多上下文，最后输出风险等级、证据和建议处理方式

这种设计的工程智慧在于：不需要把"确定性"和"灵活性"二元对立，而是在不同层级使用不同的决策机制。主流程、权限、审计和人工确认交给 Workflow；局部理解、工具选择、异常判断和策略调整交给 Agent。

### 1.4 核心观察点：谁决定下一步？

如果你只能记住一个概念来区分 Agent 和其他产品，记住这个：

> **看"谁决定下一步"。**

- Chatbot：用户决定下一步（用户不输入，系统就等着）
- Workflow：预设流程决定下一步（A 步骤完了必定到 B 步骤）
- Agent：模型根据目标、状态和反馈动态决定下一步

这就是为什么"是否用了 LLM"不是一个好的判断标准——很多 Workflow 也用了 LLM，但 LLM 只负责执行某个环节（如分类、生成），不负责决定流程走向。

**判断练习**：思考以下场景——

> 一个数据分析工具，用户上传 Excel，系统自动生成图表和洞察报告。在生成报告时，系统会调用 LLM 来撰写分析文字。

这是 Agent 吗？**不是。** 虽然调用了 LLM，但"什么时候调用、调用后做什么"都是预设流程决定的。LLM 只是"文案生成器"，不是"决策者"。

> 同一个工具，但用户可以输入"帮我找出这个销售数据中异常的趋势，分析可能的原因"。系统自己决定：先按时间维度切分数据、检查季节性波动、对比去年同期、搜索行业新闻解释异常点。

这才是 Agent 行为：**模型在决定"接下来该查什么"。**

### 1.5 一个可操作的判断框架

基于"谁决定下一步"这个核心视角，这里给出一个快速判断框架。当你遇到任何一款声称是 Agent 的产品时，问三个问题：

1. **目标来源**：用户输入的是开放目标（Agent 自主拆解执行），还是逐步指令（用户引导每一步）？
   - 用户输入开放目标，Agent 自主规划执行路径 → 更强的 Agent 特征
   - 用户逐步下达具体指令 → 更偏 Chatbot 或 Workflow

2. **行动发起**：行动是由模型自主发起的，还是由用户指令或预设规则触发的？
   - 模型自主发起 → Agent 特征
   - 用户指令触发 → Chatbot 特征
   - 预设规则触发 → Workflow 特征

3. **策略调整**：当遇到意外情况时，系统能自主改变策略吗？
   - 能自主换工具、换关键词、换路径 → 强 Agent 特征
   - 只能按预设异常分支处理 → Workflow 特征
   - 直接失败或请求用户重新输入 → Chatbot 特征

这三个问题叠在一起，就是一个产品的 Agent 成色光谱。

## 第二章：建立任务直觉

初识 Agent 时，不要先问"它用了什么技术"。先问一个更贴近体验的问题：

```text
这个任务，适不适合交给一个会自己判断下一步的系统？
```

同一个产品，在不同任务上可能表现完全不同。一个 Agent 在写代码时很强，在订机票时可能受权限和支付限制；一个搜索型 Agent 很适合做资料整理，但不一定适合修改你的本地文件。所以课程一的重点不是给产品排名，而是训练你判断任务和产品能力是否匹配。

### 2.1 什么任务适合交给 Agent

适合 Agent 的任务通常有这些特征：

| 任务特征 | 说明 | 例子 |
|---|---|---|
| 目标明确，但路径不固定 | 你知道想要什么结果，但不知道中间应该先做什么 | "帮我找出这个项目测试失败的原因" |
| 需要多步推进 | 不是一次回答就结束，而是要分解、执行、检查、调整 | "帮我整理一份竞品分析简报" |
| 需要使用外部能力 | 需要搜索网页、读取文件、运行代码、查询数据或调用工具 | "读这几份 PDF，提取核心观点并生成表格" |
| 中途可能遇到意外 | 搜索不到、文件格式不对、测试报错、信息冲突，需要换策略 | "根据报错修复代码，并验证测试是否通过" |
| 结果可以检查 | 你能判断输出是否有用、是否完整、是否可靠 | "生成会议纪要，并列出待办事项" |

一句话判断：**如果任务需要"看情况决定下一步"，它就值得用 Agent 体验。**

### 2.2 什么任务不适合交给 Agent

不是所有任务都需要 Agent。很多任务用普通 Chatbot、固定 Workflow，甚至手动操作更合适。

| 不适合的任务 | 为什么不适合 | 更合适的方式 |
|---|---|---|
| 单次问答 | 不需要多步决策，也不需要工具执行 | Chatbot |
| 规则非常固定 | 每一步都清楚，分支很少，追求稳定一致 | Workflow / 自动化脚本 |
| 高风险且不可逆 | 错一步代价很高，例如转账、删除生产数据、群发邮件 | 人工确认 + 严格流程 |
| 判断标准完全主观 | 没有可验证结果，容易变成"听起来不错" | 人类判断，Agent 只做辅助 |
| 任务边界不清 | 用户自己也没想清楚目标，Agent 很容易跑偏 | 先澄清目标，再交给 Agent |

这不是说 Agent 不能参与这些任务，而是说它不应该拥有过高的自主权。高风险任务可以让 Agent 做信息整理、方案建议、草稿生成，但关键动作要留给人确认。

### 2.3 任务和产品能力要匹配

判断一个 Agent 产品，不能只看它"聪不聪明"，还要看任务是否落在它的能力边界内。

同一个任务，交给不同类型的 Agent，得到的体验会完全不同：

| 任务类型 | 更适合的产品形态 | 不匹配时会发生什么 |
|---|---|---|
| 查资料、做综述、写简报 | 研究 / 搜索型 Agent | 编程 Agent 可能能做，但信息检索体验不一定好 |
| 修改代码、运行测试、排查报错 | 编程 Agent | 通用助手可能只能给建议，不能直接读写项目和验证结果 |
| 处理固定业务流程 | Agentic Workflow / 搭建平台 | 纯对话 Agent 可能灵活，但缺少稳定流程、权限和审计 |
| 分析长文档、提取结构化信息 | 通用助手或文档分析型 Agent | 搜索型产品可能更擅长找资料，不一定擅长处理你的私有文件 |
| 高风险操作前的方案准备 | 通用助手、研究型 Agent、工作流平台 | 不应该让 Agent 直接执行不可逆动作 |

所以，体验 Agent 时要同时观察两件事：

- **任务本身是否适合 Agent**：它是否需要多步判断、工具使用和反馈调整？
- **这个产品是否适合这个任务**：它是否拥有完成任务所需的工具、权限、上下文和确认机制？

如果任务和产品不匹配，不要急着得出"这个 Agent 不行"的结论。你应该先判断：是 Agent 能力不足，还是你把任务交给了不合适的产品形态。

## 第三章：体验主流 Agent 产品

接下来是本课程最核心的学习活动：**亲自用。**

你不需要安装所有产品。从中选 3 款，每款至少用 30 分钟，带着目的去体验。**只看文档和演示视频不算体验。**

### 3.1 产品清单与体验要点

不要按"国内 / 国外"孤立地看产品。更好的方式是先按 Agent 产品类型分类，再比较同一类产品在能力边界、过程透明度和用户控制权上的差异。

下面不是完整榜单，而是一份适合入门体验的代表清单。产品能力变化很快，体验时以当前版本为准。

| Agent 产品分类 | 国外代表产品 | 国内代表产品 | 适合观察什么 |
|---|---|---|---|
| 通用助手型 Agent | ChatGPT、Claude | Kimi、通义千问 / 豆包 | 它什么时候只是回答问题，什么时候会主动搜索、分析文件、调用工具或切换模式 |
| 研究 / 搜索型 Agent | ChatGPT Deep Research、Perplexity | Kimi、腾讯元宝 / 通义千问 | 它如何拆解研究问题、选择搜索关键词、筛选来源、生成带依据的结论 |
| 编程 Agent | Claude Code、Codex | Trae、通义灵码 | 它是否能理解代码库、跨文件修改、运行命令、验证结果，并在高风险操作前请求确认 |
| Agentic Workflow / 搭建平台 | Zapier Agents、Lindy | 扣子 Coze、Dify | 它如何把固定流程、插件、知识库和局部 Agent 决策组合起来 |

体验时不要只记录"它支持什么功能"，而要记录：

- **任务入口**：你输入的是开放目标、具体指令，还是普通问答？
- **工具调用**：工具是你手动选择的，还是模型自己判断调用的？
- **过程可见性**：你能否看到计划、搜索、文件读取、代码执行或中间结果？
- **用户控制权**：哪些动作需要你确认，哪些动作它会直接执行？
- **失败恢复**：搜索不到、代码跑错、信息冲突时，它会重试、换路径，还是直接失败？

### 3.2 体验方法论：带着问题去用

很多人体验新产品是"随便点点，凭感觉评价"。我们不这样做。

在体验每一款 Agent 时，请始终带着这四个问题：

1. **它什么时候自己做决定？**
   - 哪些操作它直接执行了，没有问你的意见？
   - 你认为这些"自作主张"合理吗？
   - 有哪些操作你觉得它应该自己做但反而问了你？

2. **它什么时候问我？**
   - 在什么类型的操作前，它会停下来征求你的确认？
   - 这个停止点设置得合理吗？会不会过于频繁（烦人）或过于宽松（危险）？

3. **它什么时候做错了？**
   - 记录一个具体的错误。它是怎么出错的？
   - 它自己发现了吗？如果没发现，你是怎么发现的？
   - 出错后它是怎么处理的？

4. **它什么时候让我惊喜？**
   - 记录一个让你觉得"哇，它竟然能做到这个"的瞬间。
   - 为什么这个瞬间让你印象深刻？
   - 这个惊喜是"一次性"的还是"可复现"的？

这四个问题对应的是 Agent 设计的四个核心维度：**自主性、可控性、容错性、能力边界**。你现在不需要记住这些术语——带着问题去用就行。在第四章中，我们会把这些体验系统化。

### 3.3 进阶：用同一个任务测试不同产品

最有启发性的学习方法之一，是用**完全相同的任务**去测试不同 Agent。它能让你看到：面对同一个目标，不同产品的"思维方式"有多么不同。

**选择一个合适的测试任务**，比如"帮我研究最近三个月 AI 编程工具领域的重要变化，整理成一份简报"。

好的测试任务满足四个条件：
- **需要多步推理**：不是一问一答能解决的
- **需要工具使用**：需要查询外部信息或操作外部系统
- **有一定的开放性**：没有唯一正确答案，允许多种策略
- **可验证**：有办法判断结果好坏

把这个任务分别交给 2-3 款产品，然后观察它们的策略差异：
- 谁先搜什么？搜索关键词选得好不好？
- 谁花更长的时间规划？
- 谁的结果更全面？谁的结果更精准？
- 谁的效率更高（更少的步骤获得相近的结果）？
- 谁在遇到死链接或信息不足时更灵活？

把对比结果记录下来——这本身就是一份高质量的产品分析素材。

## 第四章：产品拆解与分析

体验完产品之后，你需要一个框架来系统化你的感受。从"感觉好用/不好用"到"我知道它为什么好用/不好用"——这就是本章的目的。

### 4.1 产品体验观察框架与拆解模板

下面这个 7 维度观察框架参考。右侧附有 Claude Code 的填写示范，帮助你理解每个维度的记录方式。

| 维度 | 观察问题 | 记录方式 | 示例（Claude Code） |
|------|---------|---------|-------------------|
| **任务入口** | 用户输入的是开放目标、具体指令，还是普通问答？ | 写出原始用户任务 | 「帮我把 src/utils 下的工具函数加上错误处理」——具体指令，有一定开放性 |
| **下一步决策权** | 下一步由谁决定？ | 用户 / 固定流程 / 模型 | 模型主导。自己决定先列出目录、逐个读取文件、提出改造方案。仅在 git 等高风险动作时请求确认 |
| **行动能力** | 产品能调用哪些工具或外部能力？ | 列出工具、数据源或工作流节点 | 文件读写、Shell 命令、代码搜索、Git 操作、Web 搜索 |
| **过程可见性** | 用户能看到哪些步骤？ | 计划、工具调用、进度、中间结果 | 高。Thinking 块展示推理；工具调用前标注名称和参数；执行结果可见 |
| **用户控制权** | 用户能否确认、中断、修改或接管？ | 标出可介入节点 | 任何工具调用可选允许/拒绝/总是允许；可用自然语言实时干预；高风险操作默认需确认 |
| **失败与恢复** | 出错后如何处理？ | 重试、降级、澄清、失败退出 | 测试文件不存在时主动搜索目录、检查配置、换框架尝试，连续失败后向用户澄清 |
| **Agent 成色判断** | 它更像 Chatbot、Workflow，还是 Agent？ | 给出判断并说明理由 | 强 Agent。自主决定工具和执行顺序；遇意外反馈动态调整；高风险确认是安全设计，非能力限制 |

这七个维度围绕同一个核心问题：**这个产品在处理任务时，模型到底承担了多少"决策"的职责？**

---


**判断时的三个层次**：

1. **这个产品的大多数行为是由什么驱动的？** —— 用户输入、预设规则，还是模型动态决策？
2. **在哪些具体场景下它表现出了 Agent 特征？** —— 是"做搜索规划"时的自主性，还是"代码生成后自动运行测试验证"时的反馈闭环？
3. **哪些场景让它露出了 Chatbot 或 Workflow 的底？** —— 是"只能在预设选项里选"的规划步骤，还是"失败后除了道歉什么都做不了"的错误处理？

**做判断时的一条底线**：不要因为一个产品"用了 LLM"就说它是 Agent。判断标准始终是 **"下一步决策权在谁手里"。**
