## 6.1 Agent 明明错了，却继续往下走

回到 1.1。知识助手 Agent 现在有了 Planning，发布准备的四个步骤不再漏了。但执行到"运行测试"这一步时：

```text
TypeError: Cannot read properties of undefined (reading 'files')
```

Agent 看到了这个错误输出。然后它做了什么？它继续写 changelog，最后汇报"发布准备已完成"。

这不是工具问题，不是 Planning 问题，不是 Memory 问题。这是一个更根本的缺陷：**Agent 缺少"看到失败信号 → 停止 → 分析原因 → 修正 → 重试"的闭环。** 它看到了 TypeError，但它把那条输出当成了和"读取文件内容"一样的 Observation——一段需要被记录的信息——而不是一个需要被修正的错误。

打个比方：你开车时仪表盘亮起了红色警告灯。有 Reflection 的司机会靠边停车、打开引擎盖、检查问题、决定是自己修还是叫拖车。没有 Reflection 的司机——也就是裸 LLM——只是看了一眼警告灯，在日志里记下"14:32 红色灯亮了"，然后继续以 120 公里/小时的速度往前开。

换到知识问答场景。Agent 给了一段引用："根据《软件工程实践》第 3 章……"但你查看原文，发现那句话根本不在第 3 章。Agent 没有"引用校验"这一步，它生成引用时只是在生成"看起来像引用"的文本——就像一个人写论文时凭记忆写参考文献，格式正确、页码精确，但那篇文章压根不存在。

这类问题的本质是：**Agent 缺少被反馈信号触发的修正机制。** 它不会说"等等，这个测试失败了，让我看看为什么"。它只会继续生成下一个 Token。

## 6.2 Self-Refine 和 Reflexion

让 LLM "反思自己的输出"这个想法，经历了几个关键节点：

**2023 年 3 月，Self-Refine（Madaan et al.）。** 论文提出了一个简单但有效的流程：生成 → 反馈 → 修正。每一步都让同一个 LLM 完成——先输出初稿，再给自己提反馈，再根据反馈修改。这个流程在写作、代码生成等任务上取得了明显提升。但它的致命弱点是：**反馈完全来自模型自身，没有外部验证信号。** 一个错误的初稿可能导致模型给自己提一堆看似合理但方向全错的"反馈"。

**2023 年 6 月，Reflexion（Shinn et al.）。** 这个工作的突破在于引入了**外部验证信号**——不再让模型"自评"，而是让模型根据真实的执行结果（测试失败、编译器报错、API 返回）来判断哪里出错。Reflexion 把"反思"从生成策略升级为一种**基于环境反馈的强化学习式循环**：尝试 → 观察真实结果 → 分析失败原因 → 记录经验 → 重试。这比 Self-Refine 可靠得多，因为它有了外部锚点。

**2023 年底至今，Reflection 从学术概念走向工程实践。** 关键转变是：Reflection 不再被当成"模型的自省能力"，而是被设计成 Runtime 中的一个**带触发条件和停止条件的系统机制**。触发条件是什么信号（测试失败、Schema 校验失败、用户驳回），停止条件是什么情况（最大重试次数、成本上限、相同错误重复出现）。

## 6.3 反思必须依赖反馈信号

Reflection 容易被误解成"让模型自己检查一遍"。这就像让学生给自己的考卷打分——他可能会认真检查，也可能只是每道题都打个勾然后交卷。你不知道是哪种。

模型自己说"我检查过了"并不等于真的可靠。这句话本身也是一个 token 序列——模型在生成它时，和生成"引用来自第 3 章"时，用的是同一个机制。**自我评价的输出和原始输出共享同一个幻觉源。**

更好的 Reflection 应该依赖外部反馈信号——来自系统、来自工具、来自用户，唯独不来自模型自己的"判断"：

| 反馈信号 | 知识助手的具体例子 | 为什么比模型自评可靠 |
|---|---|---|
| JSON Schema 校验失败 | Agent 输出 `{"tool": "search"}` 但 schema 要求 `tool_name` 必填 | Schema 校验是确定性规则——字段存在/不存在，类型匹配/不匹配——没有模糊空间 |
| 工具返回参数错误 | `search_notes(query="", limit=500)` → `Error: limit must be 1-100` | 工具的真实返回值，不是模型"觉得"参数对不对 |
| 单元测试失败 | 修复了 `session_manager.py` 后运行测试，3/10 个测试用例断言失败 | 测试框架输出具体的预期值和实际值——这是可复现的、确定性的 |
| 引用校验失败 | Agent 说"笔记中写道：Memory 有三层守卫"，但检索原文后发现只有两层 | 原文是客观存在的文本——不是模型"记得"有，而是检索到了/没检索到 |
| 用户驳回 | 用户说"不对，我要 checklist 不是代码审查报告" | 用户知道自己的意图——模型猜不到用户在想什么 |
| 权限检查不通过 | `cat /etc/shadow` → `Permission denied` | 操作系统返回的权限错误——不是模型判断"我可能没有权限" |

也就是说，Reflection 不是一个固定步骤（"每步做完都反思一下"），而是一个**被外部信号触发的修正机制**。没有信号就不触发，信号到了才启动。

要回答四个问题——这些问题是 6.4 节展开的主线：

- 什么时候触发反思？→ 6.4.2 触发信号
- 反思要看哪些证据？→ 6.4.3 错误分类
- 修正后是否重试？→ 6.4.4 修正方案
- 重试几次后停止？→ 6.4.5 停止条件

## 6.4 触发、批判、修正、停止

一个可控的 Reflection 机制可以分成四步：

```text
触发信号
  -> 错误分类 / 批判
  -> 生成修正方案
  -> 重试或停止
```

但在深入每一步之前，先要有一个具体的参照场景。否则"错误分类"只是八个词的列表，"修正方案"只是七个短语——你不知道它们在什么情况下用。

### 6.4.1 场景设定：知识助手的 Reflection 需求

回到 1.1 的知识助手。它现在已经有了 RAG（能查笔记）、Memory（能记住偏好）、Planning（能组织多步骤任务）。但它在遇到失败时仍缺少闭环：

- **代码分析场景**：用户贴了一段代码让 Agent 找 bug。Agent 输出了分析结果，但其中引用的 API 方法名不存在——那是模型自己编造的。Agent 不会自己发现"这个 API 不存在"。
- **笔记问答场景**：Agent 回答"根据你的笔记，Memory 模块的写入逻辑在 `src/memory/write.py` 中实现了三层守卫"。用户去翻那个文件，发现只有两层。Agent 没有"引用逆向校验"——它生成引用时只是生成"看起来像路径"的文本。
- **发布准备场景**：Agent 执行 Planning 的发布流程，运行测试失败（TypeError）。它把错误输出记录在状态里，然后——继续写 changelog。它缺少"看到失败→停止→分析→修正→重试"的反射弧。

Reflection 系统的设计约束：

- 触发必须来自外部信号（测试失败、schema 校验不通过、用户驳回），不能来自模型自评。
- 修正后必须重新验证，不能"改了就算好了"。
- 必须有硬停止条件——重试次数、成本上限、相同错误重复出现。
- 修正过程对用户可见但不啰嗦——用户需要知道"Agent 发现了一个错误并正在修正"，但不需要看到 200 行 trace。

有了这个场景，下面逐环节展开。每一步都会用知识助手的具体故障来演示。

### 6.4.2 触发信号

Reflection 不是"每步做完都反思一遍"——那是浪费 token 且拖慢响应。触发应该是**被动的、基于外部信号的**。信号到了才触发，没到就不触发。

但这个原则说起来简单，落到代码上就全是细节。什么叫"信号到了"？TypeError 打在 stdout 里算不算？用户说了一句"不太满意"算不算？工具返回了 warning 但不是 error 算不算？这些问题没有标准答案——取决于你的场景、你的工具、你的用户。但在回答它们之前，你需要先建立一个**触发信号的分类框架**：哪些通道可能产生失败信号，每种信号应该触发什么动作，以及——同样重要的——每种信号的"例外"是什么（不该触发的情况）。

下面的表格以知识助手为例，覆盖了七种触发信号。注意"不应触发的例外"这一列——它和"触发条件"同样重要。误触发和漏触发一样危险。"

以知识助手为例，常见触发信号及其对应的反思动作：

| 触发信号 | 知识助手的具体例子 | 反思动作 | 不应触发的例外 |
|---|---|---|---|
| 输出格式不合法 | JSON 缺少必填字段，Schema 校验返回 `required field 'tool_name' missing` | 重新生成结构化输出，补全缺失字段 | 格式问题是因为工具定义本身就变了（如新版 API）——此时需要更新 schema 而非重新生成 |
| 工具参数错误 | 调用 `search_notes(query="", limit=500)` 返回 `limit must be 1-100` | 修正参数（limit=500→limit=50），重试 | 参数值本身是正确的但被限流拒绝——不应改参数，应等待后重试 |
| 工具执行失败 | `git log` 返回 `fatal: not a git repository` | 分析错误类型：这是环境问题（不可恢复），不是参数问题——跳过这一步，通知用户 | 网络超时类失败——应重试而非跳过 |
| 测试失败 | `test_memory_cleanup` 断言失败，expected `cleaned` got `pending` | 定位失败断言→分析代码逻辑→修正→重新运行测试 | 测试环境本身有问题（如测试数据库未启动）——不应修正代码 |
| 引用不匹配 | Agent 说"笔记中写道：Memory 有三层守卫"，但原文只有两层 | 重新检索原文→对比→修正答案中的不实断言→标注"该结论基于笔记 A 的第 2 节" | 检索结果为空不是因为引用错了，而是笔记确实被删了——应告知用户而非反复检索 |
| 用户驳回 | 用户说"不对，我要的是发布 checklist，不是代码审查报告" | 更新目标理解→重新生成符合用户意图的输出 | 用户只是说"不太满意"但没指出方向——应先追问而非猜测 |
| 连续失败 | 同一个测试连续 3 次失败，且错误信息完全相同 | 停止→记录失败上下文→请求人工介入 | — |

**触发信号的设计原则**：

- **外部优先**：测试失败 > Schema 校验失败 > 工具返回错误码 > 用户驳回 > 模型自评。模型自评作为最后手段，且只在有明确校验标准（如"检查输出是否包含三个要点"）时使用。
- **信号要具体**：不是说"出错了"，而是"TypeError at line 42: cannot read 'files' of undefined"。具体的信号才能产生有效的修正。
- **相同信号不重复触发**：如果同一个错误已经被"反思→修正→重试→再失败"了 2 次，第 3 次不应该再触发反思——应该触发"停止并上报"。

这一环节的失败集中在"该触发没触发"和"不该触发乱触发"：

| 失败表现 | 典型原因 | 修正方式 |
|---|---|---|
| Agent 看到了错误但继续执行 | 触发条件只绑定了"工具返回 error 字段"，但 TypeError 是 stdout 输出，没有 error 字段 | 触发条件覆盖多种信号通道：exit code、stderr、输出中的异常模式匹配、返回值 schema 校验 |
| 正常输出被误判为失败 | 工具返回 `{status: "ok", warnings: 3}`，系统只检查了 `status != "ok"` 但忽略了 warnings 不等于 error | 区分 error 和 warning；warning 不触发 Reflection，但累积到一定数量后提示用户关注 |
| 每步都触发反思导致延迟爆炸 | 四个步骤每个都加了"执行后反思"，一个发布准备跑了 12 次 LLM 调用（4 执行 + 4 反思 + 4 修正） | 只在有外部失败信号时触发；正常步骤不触发反思 |
| 用户驳回被当成格式错误 | 用户说"内容不对"，系统触发 JSON schema 重校验（通过了），然后汇报"输出格式正确"——完全没有处理用户的真实反馈 | 用户驳回是独立的触发信号，应触发目标重理解，不是格式重校验 |

### 6.4.3 错误分类

触发信号告诉 Reflection "有问题了"，但不告诉它"是什么问题"。在生成修正方案之前，必须先分类——不同错误类型需要完全不同的修正策略。把"工具参数错误"当成"目标理解错误"来修，只会让 Agent 在错误的方向上越走越远。

怎么分类？一个实用的切入点是问三个问题：**错误的根因在 Agent 内部还是外部？**（幻觉是内部的，网络超时是外部的）**错误是可修正的还是只能跳过的？**（参数错了可以改，权限不够只能上报）**修正需要改变什么——prompt、工具、还是上下文？**（改了 prompt 不会解决"用错了工具"的问题）

下面用这三个问题来审视知识助手遇到的八类错误。每类错误都给出了识别特征和对应的修正策略——但更重要的是，注意错误类型之间的**误判边界**（见后面的常见陷阱表），因为分类错误比不分类更危险：

| 错误类型 | 知识助手的具体例子 | 识别特征 | 修正策略 |
|---|---|---|---|
| 目标理解错误 | 用户说"帮我检查代码"，Agent 理解成了"帮我写代码审查报告"（含评分和格式模板），用户驳回 | 输出结构和用户预期不匹配；用户反馈"我不是要这个" | 重新理解用户意图，必要时追问确认 |
| 上下文缺失 | Agent 引用的 API 方法名 `memory.load()` 不存在——它在训练数据里见过类似方法，但那个方法是另一个框架的 | 幻觉型输出（编造的引用、不存在的 API、虚构的数据） | 补充检索——去查知识库中的实际 API 文档或笔记原文 |
| 工具选择错误 | 用户问"我的笔记里关于 Memory 的论述有哪些"，Agent 用了 Web Search 而不是本地笔记检索 | 使用了错误的工具类别；工具返回的结果与用户预期领域不匹配 | 根据任务意图重新选择工具；在工具注册表中标注适用场景 |
| 工具参数错误 | `search_notes(query="", limit=500)` → `limit must be 1-100` | 工具返回明确的参数校验错误 | 修正参数值；如果参数规则复杂，在工具定义中声明约束 |
| 外部环境错误 | `git log` → `fatal: not a git repository`；网络超时；磁盘满 | 错误来自外部系统而非 Agent 自身；重试不能解决 | 不可恢复→跳过并通知用户；可恢复（网络超时）→等待后重试 |
| 输出格式错误 | JSON 缺少必填字段；生成的 checklist 不是 Markdown 格式 | Schema 校验失败；输出结构不符合预定义格式 | 重生成结构化输出；在 prompt 中强化格式约束 |
| 权限不足 | `cat /etc/shadow` → `Permission denied` | 操作系统或 API 返回权限错误 | 不重试——告知用户缺少权限，请求授权或降级 |
| 结果质量不足 | Agent 生成的 changelog 只有 3 行，覆盖了 2/5 个 commit | 输出通过了格式校验但内容不完整；用户反馈"太简单了" | 补充内容；增加验收标准（如"changelog 必须覆盖所有 feat 和 fix commit"） |

**分类不是一次性的**。一轮修正后如果仍然失败，需要重新分类——可能第一次分类错了，也可能根因在上游（比如"工具参数错误"实际上是因为"目标理解错误"导致选了不该选的工具）。

**分类的常见陷阱**：

| 失败表现 | 典型原因 | 修正方式 |
|---|---|---|
| 把幻觉当成工具错误 | Agent 引用了一个不存在的 API，归类为"工具参数错误"，反复调整参数重试——但问题不是参数，是这个 API 根本不存在 | 先检查引用的实体是否存在（API 名、文件路径、函数名），不存在→分类为"上下文缺失"→补充检索 |
| 把环境错误当成代码错误 | 测试失败因为数据库没启动，Agent 归类为"结果质量不足"，开始改代码 | 先区分"外部不可控"和"内部可修正"——环境问题不修正代码 |
| 分类太粗导致修正无效 | 所有失败都归类为"工具执行失败"，统一用"重试"处理 | 至少区分可重试（网络超时）、可修正（参数错误）、不可恢复（权限不足、环境缺失）三类 |

### 6.4.4 修正方案

分类完成后，下一步是生成修正方案。修正不是"重新做一遍"——那只会得到同样的错误。修正需要针对错误类型，改变输入或策略。

但在看具体方案之前，先理解"修正"的三种失败方式。第一种：**什么都没改**——只是重新生成了相同的 prompt，唯一变化的是随机种子，结果大概率还是错的。第二种：**改错了方向**——把幻觉当成参数错误，反复调整检索参数，但问题根本不在参数上。第三种：**改多了**——参数错了却连工具和 prompt 一起换，引入新的变量，让问题更难定位。

好的修正方案就像外科手术：切口精准、只动该动的地方、动了之后能验证效果。下面看看八类错误各自需要什么样的"手术"：

| 错误类型 | 修正方案 | 知识助手的操作 | 修正后如何验证 |
|---|---|---|---|
| 目标理解错误 | 改写提示词，补充约束 | 用户驳回后，在 prompt 中增加"不要生成代码审查报告格式，只输出问题和建议" | 重新生成后让用户确认 |
| 上下文缺失 | 补充检索 | Agent 编造的 API 方法→在笔记中检索该模块的实际方法名→用检索结果替换幻觉引用 | 逐条检查引用是否能在笔记中找到原文 |
| 工具选择错误 | 换工具 | Web Search 换成 `search_local_notes` | 检查工具返回结果是否来自正确的知识源 |
| 工具参数错误 | 调整工具参数 | `limit=500`→`limit=50`，`query=""`→补全查询词 | 工具返回成功状态码且结果非空 |
| 外部环境错误 | 不可恢复→降级 | `git log` 失败→改为手动指定 commit 范围→通知用户"因为不在 git 仓库中，已改为手动模式" | N/A（跳过后由用户接管） |
| 输出格式错误 | 重生成 + 强化格式约束 | JSON 缺字段→在 prompt 中追加 `Required fields: tool_name, args, reason` | Schema 校验通过 |
| 权限不足 | 请求用户授权或降级 | `Permission denied`→向用户申请 `sudo` 权限或建议替代方案 | 用户确认授权后重新执行 |
| 结果质量不足 | 补充内容 + 强化验收标准 | changelog 只覆盖 2/5 commit→在 prompt 中追加 `必须检查 git log --oneline 中的所有 feat 和 fix commit` | 逐条检查验收标准 |

**修正方案的设计原则**：

- **修正必须改变输入**：如果只是"再试一次"，唯一变化的是随机种子。修正方案需要改变 prompt、参数、工具选择或上下文——让第二次尝试和第一次有实质不同。
- **修正的范围要匹配错误的根因**：参数错了改参数，不要连工具一起换。工具选错了换工具，不要重新理解用户意图。过度的修正会引入新的变量，让问题更难定位。
- **修正方案本身可能失败**：如果修正后的验证仍然不通过，不要继续"修正修正方案"——这很容易变成无限循环。应该重新分类（可能第一次分类错了），或者停止并请求人工介入。

### 6.4.5 停止条件

停止条件是 Reflection 的安全阀。没有它，Reflection 不是"智能修正"而是"静默死循环"。停止条件必须硬编码，不能由模型自己决定"差不多了"。

| 停止条件 | 阈值建议 | 触发时的行为 | 为什么必须硬编码 |
|---|---|---|---|
| 最大重试次数 | 3 次（同一错误的修正） | 停止修正，保留最后一次失败结果，请求人工介入 | 模型会在第 4、5 次修正时仍声称"这次应该对了"——它没有"该停了"的直觉 |
| 最大成本 | $1.00（单次任务的 Reflection 总成本） | 停止修正，返回当前最优结果 | 模型不知道 Reflection 花了多少钱 |
| 最大耗时 | 60 秒（从第一次触发 Reflection 开始计时） | 停止修正，超时前的最后一次结果作为输出 | 用户已经在等了，不能让一个测试失败把发布准备拖成 5 分钟 |
| 相同错误重复出现 | 2 次（修正后错误信息完全一致） | 停止修正，"修正方案未改变失败结果，可能根因分类错误" | 如果修正后仍出现相同的错误，说明修正方案没有命中根因——继续修正就是随机游走 |
| 错误升级 | 可恢复错误→不可恢复错误（如网络超时变成磁盘满） | 停止修正，"错误类型已升级，不适合继续自动修正" | 错误升级意味着环境在退化，继续自动操作可能造成更大损害 |
| 缺少有效反馈信号 | 外部验证器返回的信息不足以定位错误（如只有 "failed" 没有具体原因） | 停止修正，"反馈信号不足以生成有效修正方案，请求更详细的错误信息" | 模型会"编造"一个听起来合理的原因然后去修正——但不能基于猜测做修正 |

**停止条件之间的关系**：它们是 OR 关系——任意一个条件满足就停止。最先触发的通常是"相同错误重复出现"（第 2-3 次修正就出现了），其次是"最大重试次数"（3 次），成本和时间限制在正常任务中很少触发——它们是防止极端情况的兜底。

这一环节的失败集中在"该停不停"：

| 失败表现 | 典型原因 | 修正方式 |
|---|---|---|
| 相同错误连续修正 5 次 | 修正方案没有改变输入（只是重新生成了相同 prompt），但每次 LLM 的随机输出略有不同，系统认为"在进步" | 检测修正方案是否与上次相同；如果相同的 prompt+参数产生了相同的错误 2 次，第 3 次必须改变策略或停止 |
| 成本悄悄超过预算 | 每次 Reflection 调用 LLM 进行"分类"和"生成修正方案"，单次 $0.02，但一个任务触发了 15 次 Reflection | 累计成本计数器在每次 LLM 调用后更新，超过 $1 硬停止 |
| 修正方向被模型带偏 | 模型在"分析错误"时编造了一个合理但错误的根因（如"测试失败可能是因为时区问题"），然后围绕这个错误根因生成了修正方案 | 分析错误时只允许引用外部信号中的具体事实（报错行、错误码、返回值），不允许模型推测"可能是什么原因" |

Reflection 循环的核心骨架——关键是触发条件必须来自外部信号，停止条件必须硬编码：

```python
# Reflection 循环：只在有外部失败信号时触发，永远设置停止边界
def reflection_loop(
    action: callable,       # 执行动作（生成代码、回答问题等）
    validate: callable,     # 外部验证器（运行测试、检查 schema、校验引用）
    max_retries: int = 3,   # 硬上限
    cost_budget: float = 1.0  # 成本上限（美元）
) -> dict:
    """带 Reflection 的执行循环"""
    cost_spent = 0
    last_error = None
    
    for attempt in range(max_retries):
        # 1. 执行动作
        result = action(
            previous_error=last_error,  # 把上次失败原因注入上下文
            attempt=attempt
        )
        cost_spent += result.cost
        
        # 2. 外部验证（不是模型自评！）
        validation = validate(result.output)
        
        if validation.passed:
            return {"status": "success", "output": result.output, 
                    "attempts": attempt + 1, "cost": cost_spent}
        
        # 3. 分类失败，准备修正上下文
        last_error = {
            "type": classify_error(validation),  # schema_error | test_failure | tool_error | ...
            "message": validation.message,
            "evidence": validation.evidence,      # 具体报错行、引用原文等
            "suggested_fix": validation.suggestion
        }
        
        # 4. 停止条件检查
        if cost_spent > cost_budget:
            return {"status": "stopped", "reason": "cost_limit", 
                    "last_error": last_error}
        if last_error["type"] == "same_error" and attempt >= 2:
            return {"status": "stopped", "reason": "repeated_failure",
                    "last_error": last_error}
    
    return {"status": "stopped", "reason": "max_retries_exceeded",
            "last_error": last_error}

# 使用示例：代码修改 + 测试驱动的 Reflection
result = reflection_loop(
    action=lambda prev_err, attempt: llm_generate_code(
        task="fix the bug in user_service.py",
        test_failure=prev_err["evidence"] if prev_err else None
    ),
    validate=lambda code: run_tests(code),  # 外部测试框架，不是 LLM
    max_retries=3
)
```

> **设计要点**：`validate` 必须是外部验证器——跑测试、检查 schema、对比引用原文，不能是"让模型再看看"。`action` 接收 `previous_error` 作为修正信号，但只注入事实（报错内容、失败测试名），不注入模型的自我评价。

### 6.4.6 实战回放：一次完整的 Reflection 生命周期

下面把触发→分类→修正→停止串起来，用知识助手的一次代码分析任务做端到端回放。

```text
═══════════════════════════════════════════════════════════════════
场景：用户让 Agent 分析一段代码中的 bug
═══════════════════════════════════════════════════════════════════
│
│  [10:00] 用户贴了一段代码：
│  "这段代码在处理 session 清理时好像有问题，帮我看看。"
│
│  Agent（第 1 次尝试）：
│    分析结果：
│    "问题在 session_manager.py 的 cleanup 方法中。该方法调用了
│     memory.clear_expired()，但这个 API 在 Memory 模块中不存在——
│     正确的 API 应该是 memory.remove_expired_sessions()。"
│
│  → 触发 Reflection：
│    验证器（引用校验）检查 Agent 声称的 API 名是否存在于笔记中。

│    {
│      "passed": false,
│      "type": "context_missing",

│
│  → 错误分类：
│    类型：上下文缺失（Agent 编造了不存在的 API 名）
│    不是工具参数错误，不是目标理解错误，是幻觉。
│
│  → 修正方案：
│    补充检索：在笔记中搜索 Memory 模块的 session 清理相关方法。
│    检索结果：笔记 agent-memory-mechanism.md §写入决策与遗忘机制 中提到
│    "expired session 数据由 decay() 在每次 start_session 时清理，
│     底层调用 _purge_old_records()。"
│
│  [10:01] Agent（第 2 次尝试）：
│    分析结果（修正后）：
│    "问题在 session_manager.py 的 cleanup 方法中。笔记显示 Memory
│     模块的过期清理由 decay() → _purge_old_records() 完成 [来源: S4]。
│     但 session_manager 的 cleanup 直接操作了 session 存储而没有调用
│     decay()，导致过期记录被跳过。"
│
│  → 验证通过：
│    validate() 检查：所有引用的 API 名都在笔记中找到了原文匹配。
│    {"passed": true}

│    带引用标注的修正后分析。
│    尝试次数：2，Reflection 额外成本：$0.03
```

对比没有 Reflection 的流程——Agent 第一次输出就返回了，用户看到 `memory.clear_expired()` 这个不存在的 API，要么被误导，要么需要自己发现错误后再问一遍。

**回放中暴露的关键行为**：

1. **触发是被动的**：Agent 不会主动"反思一下自己写得对不对"。是 `validate()` 返回了 `passed: false`，Reflection 才被触发。
2. **分类决定了修正方向**：如果错误被误判为"工具参数错误"，Agent 就会反复调整检索参数——但问题不是参数，是它编造了 API 名。正确分类为"上下文缺失"→补充检索→用检索结果替换幻觉引用。
3. **修正改变了输入**：第 2 次尝试时 Agent 的 prompt 中多了一段检索到的原文，这是第 1 次没有的——修正不是简单重跑。
4. **验证在修正后重新执行**：不是"改了就算好了"，必须重新通过 `validate()`。

### 6.4.7 总结：Reflection 的核心决策回顾

| 环节 | 核心决策（知识助手场景） | 做错的表现 |
|---|---|---|
| 触发信号（6.4.2） | 哪些外部信号触发 Reflection？验证器检查什么？ | TypeError 被当成普通 Observation 忽略；正常 warning 被误触发 Reflection |
| 错误分类（6.4.3） | 错误属于哪一类？分类粒度够不够指导修正？ | 幻觉被当成工具参数错误，反复调整参数但 API 根本不存在 |
| 修正方案（6.4.4） | 修正策略是否改变了输入？是否匹配错误类型？ | 只是重跑相同 prompt，唯一变化是随机种子；修正范围过大，连工具也换了 |
| 停止条件（6.4.5） | 硬上限设在哪里？相同错误出现几次停止？ | 相同错误修正 5 次，每次 LLM 随机输出略有不同，系统认为"在进步" |

**三个贯穿全链路的主线：**

1. **验证必须外部化**：Reflection 的可信度完全取决于验证器的可信度。如果 `validate()` 是"让 LLM 自己检查"，那整个 Reflection 循环就是 LLM 在和自己下棋——裁判和棋手是同一个人。测试框架、Schema 校验、引用原文对比、用户驳回——这些外部锚点是 Reflection 不变成自我欺骗的基础。
2. **分类先于修正，根因先于症状**：看到"测试失败"就重新跑一遍——这不是 Reflection，这是祈祷。有效的修正需要知道"为什么失败"：是代码逻辑错了还是测试环境坏了？是引用了不存在的 API 还是 API 存在但参数错了？分类错一次，后面的修正就是浪费。
3. **停止条件是安全阀，不能交给模型决定**：模型没有"该停了"的直觉——它在第 5 次修正时仍会自信地说"这次应该对了"。最大重试次数、成本上限、相同错误检测——这些必须硬编码，必须在运行时强制执行。

回到本章开头的问题——Agent 看到 TypeError 却继续写 changelog。以知识助手为例：不是因为 Agent "不够聪明"，而是因为它缺少一个被外部信号触发的修正闭环。Reflection 的本质不是"让模型反思"，而是**在 Runtime 中建立一个"检测失败→分类→修正→验证→重试或停止"的系统机制**。触发器来自外部信号，验证器来自外部系统，停止条件来自硬编码——模型只负责"根据事实修正"，不负责"判断自己是否该修正"。

## 6.5 从格式校验到基于测试的修正

Reflection 可以按阶段迭代：

| 阶段 | 做什么 | 解决什么问题 |
|---|---|---|
| V0：无反思 | 失败直接返回 | 保持简单，验证任务是否真的需要修正闭环 | 不要过早加 Reflection——如果任务允许失败后人工修正，V0 是正确的起点 |
| V1：格式修复 | JSON / schema 不合法时重校验并重生成 | 结构化输出稳定性 | 不要过早加内容反思——先修格式问题，内容质量让用户判断 |
| V2：工具错误修正 | 参数错误、超时、权限不足分类处理 | 工具调用稳定性 | 不要过早统一"重试"——先区分可重试（网络超时）、可修正（参数错误）、不可恢复（权限不足） |
| V3：测试驱动修正 | 根据测试失败的具体信息（报错行、断言值）调整代码 | 代码类任务质量 | 不要过早信任修正结果——修正后必须重新跑测试，不是"改了就算好了" |
| V4：引用和事实校验 | 逐条检查生成内容中的引用是否能在知识库中找到原文 | 知识问答可信度 | 不要过早让模型自评引用——引用校验是字符串匹配+检索，不是"你觉得引用对吗" |
| V5：用户反馈学习 | 把用户"不对"/"不是这个"/"应该是 X"纳入回归集，同类错误不再犯 | 持续改进 | 不要过早自动化学习——先积累至少 20 条用户驳回，分析模式后再决定学习策略 |

**每个阶段的升级信号是具体的、可观察的**：

- V0→V1：用户说"输出格式不对，我解析不了"或 Schema 校验失败率超过 10%。
- V1→V2：出现过"工具调用失败但 Agent 没有修正"的故障——同一个参数错误连续出现。
- V2→V3：代码类任务中，用户需要自己跑测试才能发现 Agent 的代码有 bug——Agent 不会主动验证。
- V3→V4：用户指出"这段话引用错了/这个数据是你编的"——且不止一次。
- V4→V5：用户在同类型任务中重复给出相同的驳回反馈 ≥3 次。

不要为了"看起来更智能"每步都加自我批判。Reflection 应该服务具体问题，不是装饰品。

**设计 Reflection 的决策顺序（以知识助手为例）：**

1. **先确认反馈信号是否存在**：你的场景中有哪些可自动获取的外部验证信号？测试框架？Schema 校验？引用原文对比？如果答案是"几乎没有"，Reflection 不值得引入——模型自评不可靠。
2. **从格式校验开始（最低成本、最高确定性）**：Schema 校验不通过→重生成的闭环实现简单、验证明确、不会引入新错误。这是 Reflection 最安全的起点。
3. **再处理工具错误（需要分类）**：工具失败后，先根据错误码/返回值分类——可重试（网络超时）、可修正（参数错误）、不可恢复（环境缺失、权限不足）——再决定策略。不要所有工具错误都用"重试"。
4. **测试驱动修正是最高杠杆但需基础设施**：需要可运行的测试套件、清晰的错误输出、CI 集成。如果这些没有，不要引入 V3。
5. **引用校验是可信度的最后防线**：如果你的 Agent 需要给用户引用来源，引用校验不是可选——不校验的引用不如没有引用。校验方式不是"让模型看看引用对不对"，而是用检索反向验证：生成引用→检索原文→逐条对比。
6. **永远硬编码停止条件**：最大重试 3 次、相同错误 2 次即停、成本上限 $1。这些不是"建议"，是 Runtime 强制检查。没有停止条件的 Reflection 循环比没有 Reflection 更危险。

## 6.6 常见失败与修正

### 6.6.1 失败模式速查表

| 失败模式 | 表现 | 原因 | 修正方向 |
|---|---|---|---|
| 错误反思 | 模型把错因分析错 | 缺少外部证据 | 引入测试、schema、工具错误码 |
| 无限重试 | 一直修正但不收敛 | 没有停止条件 | 设置重试、成本、耗时上限 |
| 成本过高 | 每步都反思 | 触发条件过宽 | 只在失败或低置信度触发 |
| 掩盖失败 | 反思后输出更自信但仍错误 | 自评替代验证 | 修正后必须重新验证 |
| 用户体验差 | Agent 反复解释过程，任务很慢 | 反思内容暴露过多 | 对用户展示摘要和关键选择 |

### 6.6.2 调试故事：三个真实 Reflection 故障

速查表告诉你"可能是什么问题"，但实际调试时，症状和根因之间往往隔着几层。下面用三个知识助手场景中发生的 Reflection 故障，展示如何从症状定位到根因并修复。

---

**调试故事一：验证器太弱，Reflection 变成自我欺骗**

```text
症状：
用户让 Agent 分析代码中的 bug，Agent 输出的分析引用了不存在的 API 名
"memory.clear_expired()"。Reflection 被触发，Agent 的 validate() 是
"让 LLM 检查自己的输出是否有事实错误"。LLM 检查后说"引用来源真实，
分析逻辑正确"。Reflection 通过，用户收到了带幻觉的分析。

排查过程：
1. 检查 validate() 实现 → 它调用的是 `llm.check_factual_accuracy(output)`
2. 回放 validate 的输入 → 输出中包含"memory.clear_expired()"，
   但 LLM 在 validate 时没有去检索知识库——它只是"凭感觉"判断引用是否正确
3. LLM 在"生成"时可能编造 API，在"检查"时同样可能编造判断——
   同一个人不可能当自己的裁判

根因：
- validate() 不是外部验证器，是模型自评
- 模型在"创作模式"下编造的内容，在"检查模式"下会被自己的训练数据
  认定为"听起来合理"

修复：
1. validate() 改为确定性操作：对每个引用进行检索原文→字符串匹配
2. 不要求精确字符串匹配，但要求引用中的关键实体名（API 名、文件名、
   函数名）在原文中出现
3. 如果 validate() 本身依赖 LLM（如"判断回答是否完整"），加上人工确认环节
```

---

**调试故事二：修正方向被模型"猜测"带偏**

```text
症状：
Agent 执行发布准备流程，运行测试后 `test_memory_cleanup` 失败。
Reflection 被触发。Agent 分析错误原因："测试失败可能是时区问题导致的，
因为 session 清理依赖时间戳比较。" → 修正：在 cleanup 方法中加了时区转换逻辑。
重新运行测试→仍然失败。Agent 又分析："可能是 Python 版本差异。"
→ 修正：改了语法。仍然失败。3 次重试耗尽，最终报告："无法自动修正。"

用户看了一眼：测试失败是因为 `memory_store` 目录不存在——测试环境没初始化。
这不是时区问题，不是版本问题，是一个 `mkdir -p` 能解决的环境问题。

排查过程：
1. 查看 Reflection trace → Agent 在错误分析阶段没有引用测试输出的具体内容
2. 测试输出明确写了：
   "FileNotFoundError: [Errno 2] No such file or directory: './memory_store/'"
3. 但 Agent 在"分析原因"时忽略了具体的错误信息，而是基于"测试失败的常见原因"
   进行推测——时区、版本差异、并发问题，全是猜的

根因：
- 错误分析 prompt 没有约束 Agent 只能基于外部证据中的具体事实做分析
- Agent 被允许"推测可能的原因"，而推测本质上就是编造

修复：
1. 错误分析 prompt 严格约束："只允许引用 validation.evidence 中的具体报错行、
   错误码和返回值。不允许推测可能的原因。如果不确定，给出'需要更多信息，
   以下是已知事实：……'"
2. validate() 在返回时把具体报错行作为 evidence 原样传递，不做摘要
3. 如果 evidence 不足以确定根因，不生成修正方案→停止并请求更多信息
```

---

**调试故事三：停止条件全都没触发**

```text
症状：
用户等了 2 分钟，Agent 还在"修正中"。查看 trace：同一个 JSON schema 校验失败
被修正了 7 次。第 1 次：缺 tool_name 字段。第 2 次：缺 tool_name 字段。
第 3 次：tool_name 是 null。第 4 次：tool_name 是 ""。第 5-7 次：继续缺。

排查过程：
1. 检查停止条件配置 → max_retries=3（但为什么跑了 7 次？）
2. 检查代码 → max_retries 被设置为 3，但重试计数器在每次修正成功后重置了——
   而"修正成功"的定义是"LLM 成功返回了修正结果"，不是"验证通过"
3. 所以流程是：失败→LLM返回修正→计数器归零→验证再失败→LLM再返回修正→归零……
4. "相同错误检测"也没触发，因为每次修正后 LLM 的输出文本略有不同
   （"tool_name: null" 和 "tool_name: \"\"" 被认为是"不同的失败"）

根因：
- 重试计数器在"修正方案生成成功"时重置，应该在"验证通过"时重置
- 相同错误检测对比的是完整的错误信息字符串，而不是错误的"类型"
  （schema_error with missing field 'tool_name' 应该被识别为相同错误）

修复：
1. 重试计数器只在 validate() 返回 passed=true 时重置
2. 相同错误检测不再对比完整错误字符串，而是对比 (error_type, failed_field)
   元组：("schema_error", "tool_name") 相同→判定为相同错误
3. 增加"无进展检测"：如果最近 2 次修正后的验证错误信息相似度 > 90%，
   判定修正没有产生有效进展→停止
```

---

**三个故事的共性**：

- **验证器不外部化，Reflection 就是表演**。故事一中 LLM 给自己当裁判，幻觉从生成阶段流入了验证阶段——整个 Reflection 闭环是假的。
- **修正必须基于事实，不能基于推测**。故事二中 Agent 忽略了 `FileNotFoundError` 而去猜"时区问题"——因为 prompt 允许它推测，而推测不需要事实。
- **停止条件必须精确、必须测试**。故事三中 max_retries=3 被轻易绕过，因为"重试"的定义在实现中被偷换了。停止条件是代码逻辑，不是 prompt 描述——需要像业务逻辑一样被测试。

## 6.7 什么时候不需要 Reflection

以下场景不需要复杂 Reflection：

- 输出可以用确定性规则直接修正。
- 失败成本很低，简单重跑更便宜。
- 没有外部反馈信号，模型无法判断自己是否出错。
- 任务对延迟非常敏感。
- 高风险任务需要人工或规则校验，模型自评不能替代。

实用判断：

```text
如果有明确反馈信号，并且修正成本低于失败成本，Reflection 值得引入。
如果只是让模型”再想想”，通常不值得。
```

> **这个故事还没完。** Reflection 让 Agent 学会了从错误中修正，测试失败后会分析原因并重试。但 Reflection 的停止条件中有一条是”请求人工介入”——当自动修正已经无法收敛时，人的判断是最后的安全阀。
>
> 不过，”请求人工介入”这四个字说起来简单，做起来全是坑：什么情况该问人、什么情况该自己处理？问得太频繁用户会烦，问得太少用户会怕。问的时候给用户看什么信息，才能让他在 3 秒内做出正确判断？这就是下一章 Human-in-the-loop 要解决的问题——当 Agent 不该自己决定时，如何设计人与 Agent 的交互边界。
>
> 再往后看，你会发现还有一个更深层的问题：有些任务，不是”让一个人和一个 Agent 配合”能解决的。你让 Agent 写出一个安全方案，再让它自己审查——它说”没有明显问题”。你作为人类看了一眼，发现三个安全隐患。为什么？因为”创作者”和”审查者”需要不同的视角、不同的上下文、不同的判断标准。这就是最后一类增强能力 Multi-Agent 要解决的：一个人写代码，另一个人盯着。

### 本章速记

Reflection 的本质不是”让模型反思”，而是在 Runtime 中建立一个”检测失败→分类→修正→验证→重试或停止”的系统机制。三个贯穿全章的主线：

**验证必须外部化。** Reflection 的可信度完全取决于验证器的可信度。如果 `validate()` 是”让 LLM 自己检查”，那整个 Reflection 循环就是 LLM 在和自己下棋——裁判和棋手是同一个人。测试框架、Schema 校验、引用原文对比、用户驳回——这些外部锚点是 Reflection 不变成自我欺骗的基础。模型自我评价的输出和原始输出共享同一个幻觉源。

**分类先于修正，根因先于症状。** 看到”测试失败”就重新跑一遍——这不是 Reflection，这是祈祷。有效的修正需要知道”为什么失败”：是代码逻辑错了还是测试环境坏了？是引用了不存在的 API 还是 API 存在但参数错了？把幻觉当成工具参数错误来修，只会让 Agent 在错误的方向上越走越远。修正方案必须改变输入（prompt、参数、工具选择或上下文），而不是简单重跑。

**停止条件是安全阀，不能交给模型决定。** 模型没有”该停了”的直觉——它在第 5 次修正时仍会自信地说”这次应该对了”。最大重试 3 次、相同错误 2 次即停、成本上限 $1——这些必须硬编码，必须在运行时强制执行。停止条件之间是 OR 关系，任意一个满足就停止。没有停止条件的 Reflection 循环比没有 Reflection 更危险。
