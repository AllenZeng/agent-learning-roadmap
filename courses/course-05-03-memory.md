### 3.1 Agent 总是忘记、重复或带错历史

回到 1.1 那个知识助手。你花了 20 分钟告诉它：

```text
以后写技术文章，先给我大纲确认，再展开正文。语气直接，不要营销化。
```

第二天你打开新会话："帮我写一篇 Agent Memory 的技术文章。"它输出了一篇以"在当今这个 AI 时代……"开头的营销风文章，没有先给你大纲。你盯着屏幕：昨天说的全白费了。

这是会话级别的遗忘。还有任务级别的遗忘：你让 Agent 整理一个 50 篇文档的目录，整理到一半你关掉了电脑。下次打开，Agent 不知道读过哪些文件、哪些已整理、草稿在哪、你确认过哪些结构——它从零开始，把前一半的活重做了一遍。

这两类遗忘根因相同：**LLM 的"记忆"完全依赖上下文窗口中的文本，会话关闭后上下文清零，一切归零。** 课程三的 State 管理解决了"单次任务内的状态延续"，但跨会话、跨任务的状态延续需要新的机制——Memory。

但要小心：Memory 是双刃剑。错误记忆、过期偏好、敏感信息如果被不加区分地长期保存，Memory 会从"个性化助手"退化为"长期污染源"。所以在动手之前，必须先想清楚什么值得记住、什么必须遗忘。

### 3.2 ChatGPT 的记忆之路

2022 年 11 月 ChatGPT 发布时，它有一个让所有用户"哇"的瞬间——它能记住当前对话里的上下文。你说"我叫张三"，后面问"我叫什么"，它能答出来。

但关掉浏览器标签页，重新打开——它什么都不记得。

这不是 bug，这是设计如此。当时的 LLM API 是无状态的，每次请求是独立的。对话历史由前端应用保存在客户端，服务端不知道你是谁、上次聊了什么。

2023 年以后，随着 ChatGPT 这类对话产品进入高频使用场景，"跨会话记住我说过什么"逐渐变成用户的核心期待之一。OpenAI 在 2024 年开始公开推出 ChatGPT Memory 相关能力，并配套提供查看、关闭、删除等控制项。这个节奏本身说明了一件事：Memory 比看起来危险得多。

- 如果模型自作主张记住了用户的健康信息，隐私怎么办？
- 如果记住了用户随口说的一个错误观点，未来任务都被带偏怎么办？
- 如果用户不知道系统"记住"了什么，信任从何而来？

这里值得借鉴的不是某个具体开关状态，而是设计原则：用户应该知道系统记住了什么，并且能管理它。这个设计哲学——**"宁可少记，不可乱记"**——值得每个 Memory 系统设计者参考。

### 3.3 不是所有历史都应该变成记忆

#### 3.3.1 四类信息：只有一种属于 Memory

Memory 经常被误解成"把聊天记录存起来"。这不够准确，也很危险。

你需要先区分四类东西：

| 概念 | 作用范围 | 示例 | 是否属于 Memory |
|---|---|---|---|
| 当前上下文 | 本轮模型调用 | 用户刚输入的问题、当前工具结果 | 不一定 |
| 任务状态 | 当前任务运行期间 | 当前步骤、已完成事项、错误信息 | 工作记忆的一部分 |
| 会话历史 | 当前会话内 | 多轮对话内容 | 可摘要为短期记忆 |
| 长期记忆 | 跨会话、跨任务 | 用户偏好、稳定事实、项目习惯 | 是 |

**关键区分：不是所有"存下来"的东西都是 Memory。** 任务状态（"整理到第 23 篇了"）只在任务存活期间有意义——任务结束后它就是一条死数据。会话历史（"上一轮我问了 X，你回答了 Y"）只在当前会话窗口内有效——新会话不需要知道上个月某次对话的第三轮说了什么。

长期记忆是第四列：**跨会话仍然有意义的信息**——用户偏好、稳定事实、项目约定。这些是真正值得跨时间保留的东西。

#### 3.3.2 从真实对话中分类：哪些该记住？

下面以知识助手的一次真实会话为例，逐条标注哪些该进入 Memory、哪些不该。这个标注过程本身就是 Memory 设计的核心——**分类错了，后续所有机制都是白费**。

```text
会话片段：2026-06-23 周二上午

用户：以后写技术文章，先给我大纲确认，再展开正文。语气直接，不要营销化。       ← ①
Agent：好的，我记住了。以后写技术文章先出大纲，语气直接不营销。

用户：我最近在用 TypeScript 写一个 Agent 框架，帮我看看这段代码。              ← ②
（粘贴了 80 行 TS 代码）
Agent：（分析代码，给出了重构建议）

用户：对了，我的 Notion API key 是 secret_abc123，帮我连一下。                 ← ③

用户：重构建议不错。帮我整理一下今天讨论的要点。                                ← ④
Agent：（整理了当天讨论的三个要点）

用户：我们公司下个月的发布计划是……（省略 200 字）                              ← ⑤
Agent：明白了，这是一个临时计划。
```

现在逐条判断：

| # | 原始内容 | 分类判断 | 是否写入 Memory | 理由 |
|---|---|---|---|---|
| ① | 写文章先大纲、语气直接 | **用户偏好，跨会话有效** | ✅ 写入长期记忆 | 用户明确表达的写作风格偏好，下次写文章时仍然需要 |
| ② | 在用 TypeScript 写 Agent 框架 | **可能偏好，需要推断** | ⚠️ 候选，待确认 | 用户可能偏好 TS 示例代码。但这是单次对话中提到的，也可能是临时项目。写入候选记忆，后续出现多次再确认 |
| ③ | Notion API key: secret_abc123 | **敏感信息** | ❌ 不写入 | API key 是敏感凭证，不应进入 Memory 存储。应引导用户使用环境变量或密钥管理工具 |
| ④ | 整理了三个要点 | **任务输出，非偏好** | ❌ 不写入 | 这是当次任务的输出结果，跨会话没有复用价值 |
| ⑤ | 公司下个月的发布计划 | **临时信息，会过期** | ❌ 不写入 | 下个月之后这个信息就过期了，且是临时告知，不是持久事实 |

**这里有一个容易忽略的细节**：第②条"用户可能偏好 TypeScript"——它既不是明确的偏好声明（不像①那样直接说"以后用 TS"），也不是完全没有信号（用户确实在用 TS 写框架）。这类"推断性偏好"的正确处理方式是：**写入候选记忆，但不自动生效。等用户在不同会话中多次表现出相同偏好，或者用户明确确认后，再升级为长期偏好。**

如果我们把每一条都无差别写入 Memory，来看看一个月后的污染效果：

```text
# 如果全量写入，一个月后的 Memory 存储：
{
  "preferences": {
    "writing_style": "先大纲，语气直接不营销",       // ✅ 有用
    "language": "TypeScript",                        // ⚠️ 可能是临时项目
    "notion_api_key": "secret_abc123",               // ❌ 安全漏洞
    "company_release_plan": "下个月发布……"           // ❌ 已过期但仍在召回
  }
}
```

一个月后用户问"帮我写一篇 Python 数据分析的文章"，Memory 系统召回 `language: TypeScript` 和 `writing_style: 先大纲`——然后 Agent 用 TypeScript 示例写 Python 文章。这就是**未分类全量写入的后果：噪声记忆污染当前任务。**

#### 3.3.3 Memory 的七个关键问题

分类只是第一步。Memory 的关键问题不是"存什么"，而是一整套生命周期管理：

- 什么时候写入？
- 谁决定写入？
- 写入后如何读取？
- 如何更新旧记忆？
- 如何遗忘？
- 用户如何查看和纠正？
- 错误记忆如何避免污染后续任务？

如果没有这些问题的答案，Memory 会从"个性化能力"变成"长期污染源"。下面 3.4 节将围绕一个具体场景，逐环节回答这些问题。

### 3.4 Memory 的读写更新机制

Memory 设计可以拆成五个环节：

```text
识别候选记忆
  -> 写入决策
  -> 存储
  -> 召回
  -> 更新 / 遗忘
```

但说"五个环节"太抽象。下面我们跟着知识助手走一遍——从真实对话中识别候选记忆、做出写入决策、存储、在新会话中召回、最后处理更新和遗忘。每一步都用具体数据演示。

#### 3.4.0 场景设定：知识助手的 Memory 需求

在深入各环节之前，先明确场景。这个知识助手的基本情况：

- **用户**：一位 Agent 开发者，维护着 200+ 篇技术笔记（见课程 05-02 的 RAG 系统）。
- **使用频率**：几乎每天使用，每次会话持续 20-60 分钟。
- **典型任务**：写技术文章、分析代码、查阅笔记、整理知识。
- **Memory 需求**：记住写作偏好、代码风格偏好、项目约定、常用工具配置。

Memory 系统的设计约束：

- 用户偏好应该跨会话保持（下次打开新会话仍然生效）。
- 敏感信息绝对不能自动写入（API key、密码、内部数据）。
- 临时约束应该在会话结束后自动失效（"这次先按简化版写"）。
- 用户可以随时查看、修改、删除任何记忆。
- 过期的偏好应该自动降权或提示用户确认。

有了这个场景设定，下面逐环节展开。

#### 3.4.1 识别候选记忆

不是所有对话都值得记住。回到 3.3.2 的会话片段，系统需要在对话过程中持续识别候选记忆。以下是接近真实实现的判别逻辑，省略了少量工程函数（如 `now()`、`count_similar_memories()` 和审计日志写入）；完整可运行版本见本章末尾的示例项目。

```python
def identify_memory_candidates(
    user_message: str,
    session_context: dict,
    existing_memories: list[dict]
) -> list[dict]:
    """
    从用户消息中识别候选记忆。
    返回候选列表，每条候选包含：内容、类型、置信度、来源、敏感标记。
    """
    candidates = []

    # ── 规则 1：显式偏好声明 ──
    # 匹配模式："以后……""每次都……""我习惯……""默认……"
    explicit_patterns = [
        r"以后[^，。]*",
        r"每次[^，。]*",
        r"我习惯[^，。]*",
        r"默认[^，。]*",
        r"不要[^，。]*",      # "不要营销化"
        r"优先[^，。]*",
    ]
    for pattern in explicit_patterns:
        matches = re.findall(pattern, user_message)
        for match in matches:
            candidates.append({
                "type": "preference",
                "content": match,
                "source": "user_explicit",
                "confidence": 0.95,      # 用户明确说出来的，高置信度
                "sensitive": False,
                "expires_at": None,       # 偏好通常不自动过期
            })

    # ── 规则 2：推断性偏好 ──
    # 用户没有明确说"以后都这样"，但行为表现出模式
    # 例："我最近在用 TypeScript 写……" → 可能偏好 TS
    infer_patterns = [
        (r"我最近在用 (\w+)", "language_preference"),
        (r"帮我(\w+)一下", "task_pattern"),
    ]
    for pattern, pref_type in infer_patterns:
        matches = re.findall(pattern, user_message)
        for match in matches:
            # 检查是否已有类似记忆——出现 3 次以上才升级为长期偏好
            similar_count = count_similar_memories(
                existing_memories, pref_type, match
            )
            candidates.append({
                "type": "preference",
                "content": f"用户可能偏好使用 {match}",
                "source": "inferred",
                "confidence": 0.3 + (0.2 * min(similar_count, 3)),
                # ↑ 出现 1 次 0.5，2 次 0.7，3 次 0.9
                "sensitive": False,
                "expires_at": now() + timedelta(days=30),
                "needs_confirmation": True,   # 推断的偏好需要确认
            })

    # ── 规则 3：敏感信息检测 ──
    # 绝对不自动写入 Memory
    sensitive_patterns = [
        r'(?:api[_\s]?key|secret|token|password|密钥|密码)\s*[：:=]\s*\S+',
        r'\b[A-Za-z0-9]{32,}\b',  # 看起来像 token 的长字符串
        r'(?:身份证|手机号|银行卡)\s*[：:=]\s*\d+',
    ]
    for pattern in sensitive_patterns:
        if re.search(pattern, user_message, re.IGNORECASE):
            # 标记但不写入——记录到审计日志供用户查看
            log_sensitive_detection(user_message)
            # 不添加到 candidates

    # ── 规则 4：临时约束检测 ──
    # "这次""这回""当前这个项目"开头的往往是临时约束
    temp_patterns = [
        r'这次[^，。]*',
        r'这回[^，。]*',
        r'当前这个[^，。]*',
        r'今天[^，。]*',
    ]
    for pattern in temp_patterns:
        matches = re.findall(pattern, user_message)
        for match in matches:
            candidates.append({
                "type": "temporary",
                "content": match,
                "source": "user_explicit",
                "confidence": 0.9,
                "sensitive": False,
                "expires_at": now() + timedelta(hours=24),
                # ↑ 临时约束 24 小时后自动失效
            })

    return candidates
```

注意这里识别出了 3 条候选——一句话里包含了三个独立的偏好信号。"写文章先大纲"和"语气直接"和"不要营销化"虽然是同一句话，但它们不一定要用同一种存储粒度。产品级系统可以把它们合并成一条复合偏好，课程示例为了让冲突检测和 category 粒度更容易观察，会把它们拆成多条细粒度偏好。

这一模块的失败通常不是"完全没有识别"，而是"识别错了类型"或"漏掉了关键信号"：

| 失败表现 | 典型原因 | 修正方式 |
|---|---|---|
| 敏感信息被标记为普通偏好 | 敏感检测正则未覆盖该模式（如中文全角字符、换行分隔的 key） | 扩充敏感模式库，增加熵检测（高熵长字符串即使不命中正则也标记为待审核） |
| 临时约束被识别为长期偏好 | "这次"等临时标记词在长句中未被正则捕获（如"这次我想试试……"后面跟了 50 字的补充说明） | 临时标记词匹配后检查句子边界，整句标记为 temporary |
| 推断偏好漏识别 | 用户没有用"以后""每次都"等显式标记，但连续多次表现出相同模式 | 不只依赖单条消息的正则匹配；结合会话历史中的行为模式做统计推断 |
| 一句多信号被错误合并 | "以后用 Python 写示例，语气直接"被识别为一条 candidate，后续冲突检测无法分别处理"语言偏好"和"语气偏好" | 按语义维度拆分：语言偏好 → code_style，语气偏好 → writing_tone |
| 否定式偏好被忽略 | "不要营销化"在正则中能匹配"不要"，但"别写成营销文""少用感叹号"等变体漏识别 | 增加否定偏好的变体模式库，不只依赖"不要"关键词 |

#### 3.4.2 写入决策

识别出候选记忆后，不是全部写入——每条候选都要经过写入守卫（should_remember gate）。这是整个 Memory 系统可靠性的基石。

继续上面的例子。候选进入写入决策：

```python
def should_remember(candidate: dict, context: dict) -> tuple[bool, str]:
    """
    写入决策：判断一条候选记忆是否应该写入长期存储。
    返回 (是否写入, 决策理由)——理由用于审计和调试。
    """

    # ── 绝对不写入的条件（硬守卫）──
    if candidate.get("sensitive"):
        return False, "敏感信息不自动写入"

    if candidate.get("type") == "temporary":
        return False, "临时约束不进入长期记忆"

    # ── 置信度门槛（软守卫）──
    conf = candidate.get("confidence", 0)
    if candidate.get("source") == "inferred" and conf < 0.7:
        return False, f"推断置信度不足 ({conf:.2f} < 0.7)，保持为 pending_candidate"

    if conf < 0.5:
        return False, f"置信度过低 ({conf:.2f} < 0.5)"

    # ── 冲突检测 ──
    if context.get("existing_memories"):
        for existing in context["existing_memories"]:
            if is_contradictory(candidate, existing):
                return (
                    True,
                    f"与已有记忆冲突（旧：'{existing['content']}'），"
                    f"写入后将标记旧记忆为 superseded"
                )

    return True, "通过所有守卫"
```

**用 3.4.1 的 3 条候选逐一过决策**：

```text
候选 1: "以后写技术文章，先给我大纲确认，再展开正文"
  → sensitive? False → 通过
  → temporary? False → 通过
  → confidence 0.95 > 0.5 → 通过
  → 无冲突 → 通过
  → 决策: ✅ 写入

候选 2: "语气直接"
  → sensitive? False → 通过
  → temporary? False → 通过
  → confidence 0.95 > 0.5 → 通过
  → 与候选 1 可以同时生效，不是冲突
  → 决策: ✅ 写入，category = writing_tone

候选 3: "不要营销化"
  → 与候选 1、2 可以同时生效，不是冲突
  → 决策: ✅ 写入，category = writing_tone 或 writing_style
```

最终写入几条，不是由"同一句话"决定，而是由后续召回和冲突管理决定。经验法则是：如果两条偏好会一起生效，拆开存储更容易管理；如果它们总是作为一个整体出现，合并成复合偏好更简洁。课程示例选择拆开写入，让你能看到 category 粒度如何影响冲突替换。

推断出来的候选记忆还要再分一层状态：

| 状态 | 是否持久化 | 是否默认召回 | 适合内容 |
|---|---:|---:|---|
| `rejected` | 否 | 否 | 敏感信息、临时约束、置信度太低的信息 |
| `pending_candidate` | 可选 | 否 | "用户可能偏好 TS" 这类推断信号 |
| `active_memory` | 是 | 是 | 用户明确确认的长期偏好或稳定事实 |

这个区分很重要：候选记忆可以被记录下来用于后续确认，但不应该在默认情况下直接影响模型行为。课程示例为了演示冲突替换，会把 TypeScript 推断偏好写入 preference 存储，并在演示流程中提示它是"候选记忆"；生产系统更建议使用独立的 `pending_candidate` 状态，并让它不参与默认召回。

**写入策略分级**是一个工程决策，不是教条：

| 写入方式 | 适合内容 | 知识助手场景的实例 | 风险 |
|---|---|---|---|
| 用户显式确认后写入 | 偏好、长期事实、敏感设置 | 用户说"记住我习惯用 TS 写示例"后直接写入 | 交互成本高，但安全 |
| 低风险自动写入 | 当前任务进度、非敏感状态 | "整理到第 23 篇了"自动保存为任务状态（非长期记忆） | 需要过期机制 |
| 候选记忆待确认 | 模型推断出的偏好 | "用户最近 3 次都用了 TS"→ 候选，提示用户确认 | 需要用户审核 |
| 不写入 | 敏感、短期、模糊、冲突内容 | API key、单次临时计划、情绪化表达 | 可能少一些个性化 |

**一个常被忽略的细节**：写入决策不仅要看"这条该不该写"，还要看"是否与已有记忆重复或矛盾"。如果用户上次说"示例代码用 TypeScript"，这次说"改用 Python 写示例"——新的写入应该**替换**旧记忆，而不是新增一条。两条矛盾的偏好并存会导致召回时模型收到冲突信号，表现比没有 Memory 更差。

**冲突检测的 category 粒度陷阱**：如果 category 太粗（比如只用 `writing_style`），会导致"写文章先大纲"和"不要营销化"这两个互补偏好被判定为冲突——它们 category 相同、内容不同，冲突检测会把其中一个标记为 superseded。正确的做法是使用更细粒度的 category（如 `writing_workflow` vs `writing_tone`），让冲突检测只在真正互斥的维度上生效。**经验法则：如果两条记忆可以在同一任务中同时生效，它们就不应该共享同一个 category。**

写入决策环节的常见失败集中在守卫失效和冲突漏检：

| 失败表现 | 典型原因 | 修正方式 |
|---|---|---|
| 低置信推断偏好被直接激活 | `pending_candidate` 状态未生效，推断偏好绕过了置信度门槛直接进入 active | 在 recall 阶段检查 status 字段，pending_candidate 不参与默认召回；只有用户确认后才转为 active |
| 矛盾偏好并存于 active 状态 | 新旧偏好的 category 不一致导致冲突检测未触发（如旧偏好 category=null，新偏好 category="code_style"） | 写入偏好时强制填写 category 字段；冲突检测不只依赖 category，增加 content 相似度判断作为兜底 |
| 临时约束进入长期存储 | type=temporary 但在写入 pipeline 中丢失了 type 字段，默认按 preference 处理 | write() 增加 type 字段校验；禁止绕过 should_remember 直接调用 write() |
| 敏感信息被写入审计日志 | should_remember 正确拒绝了写入，但 audit log 在记录 write_rejected 时把完整 content 写入了明文日志 | 审计日志的 content 字段脱敏：sensitive 类型的拒绝记录用 "[sensitive content]" 替代原始内容 |
| 用户更正被当成新增偏好 | 用户说"不要用 TS 了，改用 Python"，系统新增了一条 Python 偏好，但没有 supersede 旧的 TS 偏好 | 冲突检测需要识别"否定旧偏好 + 声明新偏好"的模式，将旧偏好标记为 superseded |

#### 3.4.3 存储结构

Memory 不一定都放向量库。选择存储方式时看两个问题：

- 读取时需要精确匹配，还是语义召回？
- 记忆是否需要用户查看、编辑和删除？

以知识助手为例，不同记忆类型适合不同存储：

```text
用户偏好 "写文章先大纲、语气直接" → 精确 key-value，用户要能编辑
项目约定 "API 路径统一用 /api/v1/"    → 语义召回（用户不会精确记住这个表述）
任务经验 "上次整理 50 篇文档时先建索引再逐篇处理，效率高 3 倍" → 语义召回
当前状态 "正在整理文档目录，完成 23/50" → 精确 key-value，任务结束后清理
```

下面用一个精简的伪代码骨架来串联这些设计决策。**完整的可运行代码见本章末尾的示例项目**（`examples/course-05-03-memory/`）——课程正文聚焦于理解设计思路，不必陷在几百行实现细节里。

```python
# ═══════════════════════════════════════════════
# Memory 核心接口（伪代码）
# ═══════════════════════════════════════════════

class AgentMemory:

    分层存储：
    - preferences:   JSON 文件，精确 key-value
    - facts:         JSON 文件，语义召回
    - task_history:  JSON 文件，任务经验（保留最近 500 条）
    - session_state: 内存 dict，会话结束后清理
    - audit_log:     JSONL 文件，所有操作的审计记录
    """

    # ── 写入守卫（整个系统可靠性的基石）──
    def should_remember(candidate) -> (bool, reason):

        if candidate.sensitive:    return False, "敏感信息"
        if candidate.is_expired:   return False, "已过期"
        if candidate.type == "temporary": return False, "临时约束"
        # 软守卫：置信度门槛
        if candidate.source == "inferred" and confidence < 0.7:
            return False, "推断置信度不足"
        # 冲突检测
        if existing := find_contradiction(candidate):
            return True, f"与已有记忆冲突，将替换旧记忆"
        return True, "ok"

    # ── 写入（分类存储）──
    def write(entry):
        ok, reason = should_remember(entry)
        if not ok: audit("write_rejected", reason); return
        # 按类型分存
        match entry.type:
            case "preference" -> preferences[key] = entry
            case "fact"       -> facts.append(entry)
            case "task_result" -> task_history.append(entry)
        audit("write_accepted", entry)

    # ── 召回（相关性过滤，不全量注入）──
    def recall(current_task, limit=5):
        relevant = []
        # 关键词匹配：偏好用精确匹配
        for pref in preferences:
            if keyword_overlap(current_task, pref.content) >= 2:
                relevant.append(pref, score=1.0)
        # 语义召回：事实和任务历史用向量相似度
        task_vec = embed(current_task)
        for fact in facts:
            score = cosine_sim(task_vec, embed(fact.content))
            if score > 0.6: relevant.append(fact, score=score)
        # 排序：相似度 × 时间衰减 + 访问频率加成

        return deduplicate(relevant)[:limit]
```

**核心设计决策速查：**

| 决策 | 选择 | 原因 |
|---|---|---|
| 偏好用精确 key-value 存储 | ✅ "写文章先大纲"这类确定偏好 | 语义召回对精确偏好反而不准 |
| 事实和经验用语义召回 | ✅ 用户不会精确记得"上次说的是哪句话" | 向量相似度能找回相关内容 |
| 写入前必须过守卫 | ✅ 敏感/临时/低置信推断不写入 | 一旦写入错误记忆，污染是长期的 |
| 审计日志记录所有操作 | ✅ 用户想知道"系统记住了我什么" | 无审计 = 不透明 = 用户不信任 |
| 会话结束清理状态 | ✅ 会话状态不跨会话保留 | 任务完成后状态数据就是死数据 |

上表的决策落到存储里，一条完整的 Memory 记录长这样（以知识助手的写作偏好为例）：

```json

  "type": "preference",
  "category": "writing_workflow",
  "content": "写技术文章时先给大纲确认，再展开正文",
  "source": "user_explicit",
  "confidence": 0.95,

  "sensitive": false,
  "expires_at": null,

  "updated_at": "2026-06-23T10:30:00",
  "last_accessed_at": "2026-06-25T09:05:00",
  "access_count": 9,
  "supersedes": null,
  "superseded_by": null,
  "audit_trail": [
    {"action": "write", "timestamp": "2026-06-23T10:30:00", "reason": "user_explicit, confidence 0.95"},
    {"action": "recall", "timestamp": "2026-06-24T09:05:00", "task": "写 RAG 最佳实践文章"},

```

注意几个字段的用途：`category` 决定冲突检测的粒度（同 category 的新记忆会触发替换检查）；`status` 控制是否参与召回（pending_candidate/superseded 不参与）；`last_accessed_at` 和 `access_count` 驱动衰减和排序；`supersedes`/`superseded_by` 维护版本链，保证旧偏好可审计；`audit_trail` 记录关键操作，是调试和用户可见性的基础。

> **设计要点**：`should_remember` 的保守策略是整个 Memory 系统可靠性的基石——宁可漏记无害信息，不可写入一条有害信息。五个设计决策对应五类问题，抓住它们就抓住了 Memory 设计的骨架。具体实现细节（文件读写、向量计算、冲突合并等）在示例项目中有完整演示。

#### 3.4.4 召回

Memory 召回应该按任务相关性，而不是全部注入。这是最容易出问题的环节——**如果召回策略太宽，模型会被无关历史淹没；太窄，有用的记忆进不来。**

回到知识助手。用户在周二打开新会话，说要"帮我写一篇 Agent Memory 的技术文章"。这是 Memory 系统在后台做的事情：

```python
# 初始化
memory = AgentMemory("./memory_store")
memory.start_session(user_id="user-001")

# 用户说："帮我写一篇 Agent Memory 的技术文章"
current_task = "帮我写一篇 Agent Memory 的技术文章"

# 召回相关记忆
recalled = memory.recall(current_task, limit=5)

# 召回结果（产品级设计示意；课程示例主要演示偏好召回）：
# ┌──────────────────────────────────────────────────────────────────┐
# │ #1 (score: 0.92, match: keyword)                                  │
# │   type: preference                                               │
# │   content: "写技术文章时：先给大纲确认，再展开正文。               │
# │             语气直接，不要营销化。"                                │
# │   created_at: 2026-06-23T10:30   access_count: 4                 │
# │   影响：Agent 会先输出大纲给用户确认，语气直接不营销               │
# ├──────────────────────────────────────────────────────────────────┤
# │ #2 (score: 0.78, match: semantic)                                 │
# │   type: preference                                               │
# │   content: "用户偏好 TypeScript 示例代码"                          │
# │   created_at: 2026-06-23T14:20   access_count: 2                 │
# │   source: inferred, confidence: 0.7, status: pending_candidate    │
# │   影响：默认不直接生效；可提示用户确认是否记住                    │
# ├──────────────────────────────────────────────────────────────────┤
# │ #3 (score: 0.65, match: semantic)                                 │
# │   type: task_result                                              │
# │   content: "上次写技术文章时，用户要求先写大纲被确认后再展开，     │
# │             最终版本经历了 3 轮迭代。先大纲再正文的方式效率高。"     │
# │   created_at: 2026-06-23T11:00                                   │
# │   影响：Agent 知道这个工作流程过去有效                             │
# └──────────────────────────────────────────────────────────────────┘
```

现在 Agent 收到的上下文包含了这些记忆。对比没有 Memory 时的 prompt：

```text
# 没有 Memory 的 prompt：
"帮我写一篇 Agent Memory 的技术文章。"
→ Agent 自由发挥，可能输出营销风、没有大纲

# 有 Memory 的 prompt：
"[Memory 上下文]
 用户写作偏好：先给大纲确认再展开，语气直接不营销。
 上次同类任务：先大纲再正文，3 轮迭代完成，效率高。

 [待确认候选，不默认约束回答]
 用户可能偏好 TypeScript 示例代码（推断，置信度 0.7）。

 [用户问题]
 帮我写一篇 Agent Memory 的技术文章。"
→ Agent 先输出大纲，语气直接，等待确认
```

**召回质量取决于三个因素**：

- **相关性过滤**：不是所有记忆都和当前任务相关。用户问"写文章"时，"API key 是 xxx"这条敏感信息（假设被错误写入了）不应该被召回。
- **置信度传递**：推断的记忆（如"可能偏好 TS"）不应该伪装成确定偏好；如果要进入上下文，必须附带置信度和 `pending_candidate` 标注。
- **数量截断**：limit=5 是一个经验值。太多记忆会让模型注意力分散；太少可能漏掉关键信息。

**召回时的额外考虑**：

- 这条记忆是否与当前任务相关？
- 是否过期？
- 是否与新信息冲突？
- 是否有权限问题？
- 是否需要告诉用户当前使用了哪些记忆？

**Memory 召回与 RAG 检索的关系**：

Memory 的语义召回在技术上复用了课程 05-02 中 RAG 的检索能力——用户偏好和经验记忆被向量化后存入索引，查询时通过向量相似度匹配。但 Memory 召回有几个不同于文档检索的特点：

| 维度 | RAG 文档检索（05-02） | Memory 召回（本章） |
|---|---|---|
| 语料规模 | 可能上万条 chunk | 通常几十到几百条记忆 |
| 检索精度要求 | 高召回优先（宁可多召回再 rerank） | 高精度优先（无关记忆的污染代价更高） |
| 匹配方式 | 以语义向量匹配为主 | 偏好用精确关键词匹配，事实/经验用语义匹配 |
| 时效权重 | 新旧文档按更新时间排序 | 记忆需时间衰减（90天未访问降权），且访问频率影响排序 |
| 状态过滤 | 按 status、tags、时间过滤 | 按 status（active/pending/superseded）、过期时间、置信度过滤 |
| 用户可见性 | 引用来源展示 | 用户需要能查看、编辑、删除记忆 |

这意味着 Memory 召回不适合直接复用 RAG 的"大召回 + rerank"模式。更务实的做法是：偏好类记忆用精确关键词匹配（用户说"写文章"→命中"写作偏好"），事实和经验类记忆用向量相似度做语义召回，两者按加权融合。由于 Memory 语料规模远小于文档库，混合检索的计算成本很低，不需要引入 HNSW 等近似最近邻算法——暴力精确搜索通常就够用了。

这一模块的失败集中在"该召回没召回"和"召回了不该召回的"：

| 失败表现 | 典型原因 | 修正方式 |
|---|---|---|
| 新偏好被旧偏好压过 | 旧记忆 access_count 过高（47次），访问频率加成让它在排序中压倒仅1次的新记忆 | 访问频率加成设上限（如 0.05）；当新旧偏好 category 相同且冲突时，新偏好直接替代旧偏好进入 active |
| 推断偏好被当成确定偏好注入上下文 | recall 时未检查 status 字段，pending_candidate 伪装成 active_memory 参与召回 | recall 阶段过滤 `status != "active"` 的记忆；pending_candidate 只在用户主动询问"你对我有什么推断"时展示 |
| 过期临时约束仍然被召回 | expires_at 已过但 decay() 未及时执行，或临时约束被误标为 preference | 每次 recall 前检查 expires_at；在 end_session() 中加入临时约束清理 |
| 无关记忆淹没当前任务 | limit 设置过高或相关性过滤太松，"写文章"召回了"API key 是 xxx"等敏感或无关记忆 | 按任务类型做第一级过滤（写文章→只看 writing_* category）；敏感记忆不参与自动召回 |
| 用户不知道当前使用了哪些记忆 | 召回结果直接注入 prompt 但没有告知用户 | 在回答中显式标注"基于你之前的偏好：……"，让用户感知到 Memory 在起作用并有纠正机会 |

#### 3.4.5 更新与遗忘

Memory 是活的——偏好会变，事实会过时，错误需要纠正。没有更新和遗忘机制的 Memory 系统会越来越不可信。

**场景一：偏好变更**

用户用了两周 TypeScript 示例后，项目切换到 Python：

```text
用户（周三）：以后示例代码改用 Python。

Memory 系统处理流程：
1. identify_memory_candidates → "以后示例代码改用 Python" (confidence: 0.95)
2. should_remember → 检测到冲突：已有 "用户偏好 TypeScript 示例"
   → 返回 "conflict_replacing: 与已有记忆冲突，将替换"
3. write → 新偏好写入（preference_python_abc123），旧偏好被标记 superseded
4. 旧偏好不删除，降级为历史版本（可审计但不召回）
```

```python
# 实际执行

    "type": "preference",
    "category": "code_style",
    "content": "示例代码使用 Python",
    "source": "user_explicit",
```

**场景二：过期清理**

```python

# stats: {
#   "expired_deleted": 3,    # 3 条临时约束已过期，被删除
#   "stale_flagged": 1       # 1 条偏好 90 天未访问，标记为 stale
# }

# stale 标记不删除记忆，只是在召回时降低分数或提示用户确认
# 用户下次会话时看到提示：
# "你有一条写作偏好久未使用（'文章开头用故事引入'），是否仍然需要？"
```

**遗忘机制的三个层次**：

| 层次 | 操作 | 触发条件 | 知识助手示例 |
|---|---|---|---|
| 软遗忘 | 降权，不删除 | 90 天未访问 | 用户 3 个月前设的某个偏好，最近从未使用 |
| 硬遗忘 | 删除记忆 | 用户主动删除 or 明确过期 | "这个项目已经结束了，相关的记忆删掉吧" |
| 覆盖遗忘 | 新记忆替代旧记忆 | 冲突检测 | "改用 Python" 替代 "用 TypeScript" |

**关键原则：永远不要静默删除。** 删除操作应该：记录审计日志 → 通知用户 → 提供恢复手段。如果用户不知道系统"忘记了"什么，Memory 就是不透明的。

**回放中暴露的几个关键行为**：

1. **跨会话延续**：Session 2 中 Agent 不需要用户重复"先给大纲"——Memory 让偏好跨会话生效。
2. **冲突替换**：Session 3 中"用 Python"覆盖了"用 TS"——不是新增一条冲突记忆，而是替换。
3. **审计可见**：被 superseded 的记忆不删除，用户可以查看"系统曾记住我偏好 TS"。
4. **推断 vs 确认**："可能偏好 TS"是推断的（confidence 0.5），生产系统应把它当作 pending candidate；"示例代码用 Python"是用户明确的（confidence 0.95），优先级更高。
5. **时间衰减**：长期不用的记忆自动降权，不永久占据存储和召回位置。

**每个环节的核心决策回顾：**

| 环节 | 核心决策（知识助手场景） | 做错的表现 |
|---|---|---|
| 识别候选（3.4.1） | 用哪些规则区分偏好/临时/敏感，如何处理一句多信号 | 敏感信息被标记为普通偏好；临时约束被识别为长期偏好 |
| 写入决策（3.4.2） | 守卫链的严格程度、冲突检测的 category 粒度、推断偏好的处理 | 矛盾偏好并存；低置信推断直接激活；临时约束绕过守卫 |
| 存储结构（3.4.3） | 精确 key-value vs 语义向量 vs 混合，用户是否可管理 | 偏好用语义召回的精度反而不如精确匹配；用户无法查看/编辑记忆 |
| 召回（3.4.4） | 按任务类型做一级过滤、状态过滤、数量截断、置信度标注 | 旧偏好访问次数多压倒新偏好；pending_candidate 伪装成确定偏好 |
| 更新与遗忘（3.4.5） | 冲突替换 vs 新增、软遗忘 vs 硬遗忘、衰减策略 | 用户更正的偏好被当成新增；过期记忆永不清理 |

**三个贯穿全链路的主线：**

1. **守卫链不能有旁路**：从 3.4.1 的敏感检测→3.4.2 的 should_remember 硬守卫→3.4.4 的召回状态过滤，任何一环的旁路都可能导致错误记忆进入长期存储并影响后续决策。审计日志是发现旁路的最后一道防线——但日志自身也需要脱敏。
2. **用户可见性决定信任**：Memory 系统如果不能让用户直观地看到"系统记住了我什么"，就会从个性化能力退化为不透明黑箱。从 3.4.3 的存储结构（用户可编辑的 key-value）到 3.4.5 的遗忘机制（静默删除是禁忌）到 3.4.4 的召回（告知用户当前使用了哪些记忆），可见性是贯穿全链路的要求。
3. **分类是一切的前提**：3.3 的四类信息区分、3.4.1 的 candidate type（preference/temporary/sensitive）、3.4.2 的 status（active/pending_candidate/rejected/superseded）、3.4.3 的存储选择（key-value vs 向量）——所有这些决策都取决于"这条信息到底是什么类型"。分类错了一次，后续所有机制都在错误的基础上运行。

回到本章开头的问题——Agent 总是忘记、重复或带错历史。以知识助手为例：用户周一花了 20 分钟告诉 Agent"写文章先大纲、语气直接"，周二打开新会话时 Agent 不应该从零开始。Memory 的本质就是在会话关闭、上下文清零后，把真正需要跨时间保留的信息（偏好、事实、经验）独立存储，在新会话中按需召回，同时确保错误信息、过期偏好和敏感内容不会污染未来的决策。

### 3.5 从会话状态到长期记忆

Memory 建议按阶段迭代：

| 阶段 | 做什么 | 解决什么问题 | 何时进入下一阶段 |
|---|---|---|---|
| V0：无长期记忆 | 只依赖当前上下文 | 验证任务本身是否成立 | 用户开始抱怨"每次都要重新说一遍" |
| V1：任务状态 | 保存当前步骤、工具结果、待办 | 支持长任务继续执行 | 出现过任务执行到一半会话中断的故障 |
| V2：会话摘要 | 把多轮对话压缩成当前会话状态 | 支持长对话不丢重点 | 单次会话超过 20 轮，模型开始忘记开头 |
| V3：显式用户偏好 | 用户确认后保存长期偏好 | 减少重复说明 | 用户在不同会话中重复表达相同偏好 ≥3 次 |
| V4：经验记忆 | 保存失败和成功策略 | 改善同类任务表现 | 同类任务执行 ≥5 次，出现了可复用的模式 |
| V5：可管理记忆 | 用户可查看、编辑、删除、导出 | 支持可信长期使用 | 记忆数量 >50 条，用户需要管理能力 |

**每个阶段的升级信号是具体的、可观察的**，不是"系统应该更好"这种模糊感觉：

- V0→V1：用户报告"关掉浏览器后任务进度丢失"。
- V1→V2：会话超过 20 轮后，模型开始重复提问或忽略早期的关键信息。
- V2→V3：用户在不同会话中说了 3 次以上相同的话（如"以后都用先大纲的方式"）。
- V3→V4：同类任务（如"写技术文章"）执行超过 5 次，每次用户都给出相似的修改意见。
- V4→V5：用户问"你到底记住了我什么？"——说明记忆数量已经超出用户能直觉感知的范围。

**不要过早优化**。V0 阶段就上自动长期记忆，结果是："系统记住了我随口说的一个错误观点，后面所有任务都被带偏了。"先做任务状态和显式偏好，风险更低，也更容易评测。

**设计 Memory 的决策顺序（以知识助手为例）：**

1. **先分类，再设计**：盘点你的场景中哪些信息需要跨会话保留——用户偏好？项目约定？任务经验？把 3.3.1 的四类区分应用到你的具体场景，明确每一类的边界。分类错一次，后续机制全在错误基础上运行。
2. **定写入策略分级**：确定哪些信息用户显式确认后写入、哪些可以低风险自动写入、哪些只能作为候选待确认。关键判断标准是"这条信息如果记错了，代价有多大"。写作偏好记错了只是体验差；API key 记错了是安全事件。
3. **选择存储方式**：偏好类用精确 key-value（用户要能编辑），事实/经验类用语义向量（用户不会精确记得表述），会话状态用内存 dict（会话结束即清理）。不要所有记忆都进向量库——精确匹配对于确定偏好反而更准。
4. **设定召回相关性与截断**：按任务类型做第一级过滤（写文章→writing_* category），状态过滤（只看 active），然后按相似度×时间衰减+访问频率排序，limit=5 截断。不要让 47 次访问的旧偏好压倒 1 次的新偏好。
5. **建立遗忘与衰减机制**：确定软遗忘（90 天未访问降权）、硬遗忘（用户主动删除或明确过期）、覆盖遗忘（冲突替换）的触发条件。关键原则：永远不要静默删除——记录审计日志、通知用户、提供恢复手段。
6. **用真实会话评测**：收集 5-10 个跨会话的真实场景（如"周一设偏好→周二开新会话→验证 Agent 是否按偏好行事"），而不是靠感觉判断"召回够不够准"。

### 3.6 常见失败与修正

#### 3.6.1 失败模式速查表

| 失败模式 | 表现 | 可能原因 | 修正方向 |
|---|---|---|---|
| 记忆污染 | 错误信息长期影响后续任务 | 未确认就写入长期记忆 | 写入确认、来源记录、可删除 |
| 召回噪声 | 不相关记忆进入上下文 | 召回条件太宽 | 按任务类型和相关性过滤 |
| 偏好过期 | 用户新偏好被旧偏好覆盖 | 没有更新时间和冲突处理 | 新记忆优先、冲突提示 |
| 隐私风险 | 敏感信息被长期保存 | 缺少数据分级 | 敏感内容默认不写入 |
| 上下文膨胀 | 记忆太多导致模型忽略当前目标 | 全量注入 | 只注入相关记忆，做摘要 |
| 用户不信任 | 用户不知道系统记住了什么 | 没有可见管理界面 | 提供查看、编辑、删除 |

#### 3.6.2 调试故事：三个真实 Memory 故障

速查表告诉你"可能是什么问题"，但实际调试时，症状和根因之间往往隔着几层。下面用三个知识助手场景中真实发生的故障，展示如何从症状定位到根因并修复。

**调试故事一：Agent 坚持用 TypeScript，尽管用户已经说过改用 Python**

```text
症状：
用户："帮我写一篇 FastAPI 的技术文章"
Agent：（输出了 TypeScript 示例代码）
用户："我说了改用 Python！"
Agent：（下一轮还是用了 TypeScript 示例）

排查过程：
1. 检查 Memory 存储 → preferences 中有两条：
   mem_12: "用户偏好 TypeScript" (active, access_count: 47)
   mem_89: "示例代码使用 Python" (active, access_count: 1)
2. 为什么 TypeScript 偏好还在 active 状态？→ 写入 "改用 Python" 时，
   冲突检测按 category 匹配，但 "用户偏好 TS" 的 category 是 null，
   "示例代码使用 Python" 的 category 是 "code_style"——没匹配上。
3. 召回时两条都被返回，但 TypeScript 的 access_count=47，
   访问频率加成让它排在 Python 前面。

根因：
- 冲突检测的 category 字段不一致。一条有 category，一条没有。
- 访问频率加成不应该权重过高（47 次访问的旧偏好压过了 1 次的新偏好）。

修复：
1. 写入偏好时强制填写 category 字段。
2. 冲突检测增加 content 相似度判断（不只是 category 匹配）。
3. 当新旧偏好冲突时，新偏好直接替代旧偏好进入 active，
   旧偏好标记 superseded，不再参与召回排序。
4. 访问频率加成上限从 0.2 降到 0.05（访问次数说明的是"过去使用多"，
   不代表"当前仍然有效"）。
```

---

**调试故事二：临时约束污染了长期行为**

```text
症状：
用户周二说"这次的文章写短一点，控制在 1000 字"，然后接下来两周，
Agent 写的所有文章都在 1000 字左右。用户终于忍不住问"你为什么每篇都这么短？"

排查过程：
1. 检查 preferences → 有一条：
   mem_34: "文章控制在 1000 字左右" (active, source: user_explicit, confidence: 0.9)
2. 回溯 audit log → 写入时间是周二 14:30，用户原话是"这次的文章写短一点"。
3. identify_candidates 阶段：正则 "这次[^，。]*" 被识别为临时约束，
   但正则只捕获了 "这次的文章写短一点"。
4. should_remember 阶段：临时约束的 type 是 "temporary"，
   应该被硬守卫拦截。但实际写入时 type 是 "preference"。

根因：
- identify 阶段正确识别了 "这次"，但写入时丢失了 type 字段，
  write() 默认按 preference 处理。
- 调用的代码路径是 write() 而不是通过完整的 identify → should_remember → write 管道。

修复：
1. write() 增加 type 字段校验：如果 type 是 "temporary" 但没有 expires_at，
   自动补充 24 小时过期时间。
2. 禁止绕过 should_remember 直接调用 write()——改为只有 MemoryManager
   这个高层接口可以调用 write，它强制在写入前执行完整的守卫链。
3. 在会话结束时的 end_session() 中加入检查：如果有 type=temporary
   的记忆没有在 expires_at 后清理，主动清理。
```

**调试故事三：敏感信息从会话泄露到 Memory**

```text
症状：
用户进行安全审计时，在 memory_store/audit.jsonl 中发现了 API key 的明文记录。
虽然不是直接写入 preferences，但审计日志里留下了痕迹。

排查过程：
1. 搜索 audit.jsonl → 有三条记录包含敏感模式：
   - "我的 OpenAI API key 是 sk-abc123..." (write_rejected, 但 content 记录在日志里)
   - "数据库密码是 db_pass_789" (write_rejected, 同样内容在日志里)
2. identify 阶段正确识别为 sensitive，should_remember 正确拒绝了写入。
   但 audit log 在记录 write_rejected 时，把完整的 candidate.content 写入了日志。
3. audit log 是明文存储的 JSONL 文件，没有加密。

根因：
- 审计日志在记录拒绝写入的原因时，包含了完整的候选内容。
- sensitive 标记只阻止了写入 Memory 存储，但没有阻止内容进入审计日志。

修复：
1. 审计日志的 content 字段脱敏：只保留前 20 个字符 + "...[redacted]"。
2. sensitive 类型的 write_rejected 审计记录中，content 用 "[sensitive content]" 替代。
3. audit.jsonl 文件权限设为 600（仅 owner 可读写）。
4. 增加配置项：生产环境审计日志加密存储。
```

---

**三个故事的共性**：

- 失败不是某个大错误导致的，而是**多个小决策的连锁反应**——category 字段缺失 → 冲突检测失败 → 旧偏好压倒新偏好。
- **审计日志是调试 Memory 系统的第一工具**。没有 audit log，你只能猜测"系统可能记住了什么"。
- **守卫链不能有旁路**。故事一中 write() 绕过了 should_remember 的 category 校验，故事二中 type 字段在传递中丢失——任何一条旁路都可能导致错误记忆进入长期存储。

### 3.7 什么时候不需要 Memory

以下场景不建议引入长期 Memory：

- 每次任务都是一次性请求。
- 用户明确不希望被记住。
- 错误记忆的风险高于个性化收益。
- 任务状态已经由确定性业务系统保存。
- 当前只是原型阶段，尚无稳定任务类型。
- 你还没有记忆查看、纠错和删除机制。

一个实用判断：

```text
如果系统只需要完成当前任务，优先保存任务状态。
如果系统需要跨任务理解用户或项目，才考虑长期 Memory。
```

**还有一个容易被忽略的判断维度**：Memory 的用户体验成本。如果用户不知道系统记住了什么、不知道为什么 Agent 这样回答、不知道如何纠正错误记忆——那 Memory 带来的困惑可能超过它带来的便利。在引入 Memory 之前，先问自己：**用户能直观地看到、理解和管理这些记忆吗？**

> **这个故事还没完。** Memory 让 Agent 记住了用户偏好，但新的问题出现了：用户给了一个多步骤的复杂任务——"做发布准备：检查 README、跑测试、整理 changelog、生成 checklist"。Agent 在第 4 步之后开始漂移，偏离原始目标。Memory 解决的是"记住什么"，但它不管"任务怎么组织"。这就是下一章要讲的 Planning。

### 本章速记

Memory 的本质是在会话关闭、上下文清零后，把真正需要跨时间保留的信息独立存储，在新会话中按需召回。三个贯穿全章的主线构成了 Memory 设计的骨架：

**分类是一切的前提。** 四类信息（当前上下文、任务状态、会话历史、长期记忆）只有最后一种属于 Memory。把临时约束当成长期偏好、把敏感信息当成普通偏好——分类错一次，后续所有机制都在错误的基础上运行。写入前先问：这是偏好、事实、临时约束，还是敏感信息？

**守卫链不能有旁路。** 从候选识别（3.4.1）的敏感检测 → 写入决策（3.4.2）的硬守卫 → 召回（3.4.4）的状态过滤，任何一环的旁路都可能导致错误记忆进入长期存储并影响后续决策。禁止绕过 `should_remember` 直接调用 `write()`。审计日志是发现旁路的最后一道防线——但日志自身也需要脱敏。

**用户可见性决定信任。** Memory 系统如果不能让用户直观地看到"系统记住了我什么"，就会从个性化能力退化为不透明黑箱。存储结构要支持用户编辑（偏好用精确 key-value）、召回要告知用户当前使用了哪些记忆、遗忘不能静默删除。宁可少记，不可乱记。如果用户问"你到底记住了我什么"而你只能给他看一行 JSON，Memory 的信任就已经透支了。
