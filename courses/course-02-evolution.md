# 课程二：Agent 范式演进

## 课程导言

课程一让你从产品体验出发，看见了 Agent 和 Chatbot、Workflow 的区别。你已经知道，一个产品是否具备 Agent 特征，关键不在于它是否用了 LLM，而在于：

```text
谁决定下一步？
```

课程二要回答更深一层的问题：

```text
为什么 Agent 会演进成今天这种形态？
```

学习 Agent 最容易踩的坑，是一上来就问"某个框架怎么用"。这样会很快接触到一堆名词：Tool Use、Function Calling、RAG、Memory、Planning、Graph、MCP、Multi-Agent。每个词看起来都重要，但你很难判断它们之间的关系，也不知道什么时候该用、什么时候不该用。

更好的学习方式是反过来：

```text
先问系统遇到了什么问题，再看新能力为什么出现。
```

举个例子：

- 工具调用出现，不是因为"模型应该能调用 API"，而是因为应用场景从"回答问题"走向"完成任务"时，模型只能说却做不了。
- ReAct 重要，不是因为它提出了 Thought/Action/Observation 三个词，而是因为它把此前常被分开讨论的推理和行动放进同一个循环：让判断影响行动，让行动结果反过来修正判断。
- 纠错与反思的出现，不是因为"应该让模型检查自己的错误"，而是因为多步任务中错误会沿着依赖链传播放大，而模型自身的推理无法替代外部验证。
- ...

本课是"范式理解层"。它不要求你立刻实现完整 Agent，也不要求你记住每篇论文的细节。它的目标是帮你建立判断力：当你看到一个新 Agent 技术或产品时，能问出正确的问题：

- 它解决的是哪个旧问题？
- 它把什么能力放进了系统？
- 它让哪些任务变得更容易？
- 它又带来了哪些新的问题？

带着这套问题意识进入课程三，你会更容易理解为什么一个最小 Agent 需要 LLM 决策、工具交互、状态管理和循环控制，而不是只需要一个"更聪明的模型"。

本课会提到一些代表论文、研究团队和产品节点，但它们不是用来堆时间线的。每个节点都会放回当时的问题里讲：为什么这个问题在那时变得尖锐，新的方案到底推进了什么，又把什么新麻烦留给了后来的 Agent Runtime。

这条主线主要覆盖到 2024 年底前形成的 Agent 工程范式：LLM 决策核心、工具调用、ReAct 循环、反思纠错、Planning 和 Multi-Agent。2025 年之后，reasoning models、coding agent、deep research、computer-use agent 等方向继续把这些能力推向产品级运行时，本课先把底层范式讲清楚，后续课程会在运行时架构和产品化实践中继续展开。

---

## 学习目标

学完本课，你将能够：

1. **解释传统 Agent 为什么没有大规模进入普通应用**：规则建模与开放环境之间的矛盾是根本瓶颈。
2. **说明 LLM 如何改变 Agent 的决策核心**：从"人工写规则"到"模型理解目标并动态决策"的范式转变。
3. **讲清工具调用为什么出现**：LLM "会说但不会做"→ Toolformer / Plugins / Function Calling → 让模型连接外部世界。
4. **解释 ReAct 为什么是理解 Agent 循环的经典框架**：Thought → Action → Observation 反馈循环，让 Agent 从"一次性回答"走向"持续执行任务"。
5. **理解纠错与反思的演进逻辑**：区分模型内部推理与系统外部反馈，理解为什么纠错不能只靠模型自评。
6. **说明 Planning 和 Multi-Agent 的出现背景**：复杂任务需要拆解与分工，但每种范式都带来新的工程问题。

## 第一章：传统 Agent 的历史背景

"Agent"不是 LLM 时代才造出来的词。早在人工智能和软件工程中，人们就在追问同一个问题：一个系统如何感知环境、做出决策、采取行动，并根据结果调整自己？

如果把 Agent 的历史拉长来看，它不像一个突然冒出来的新概念，更像是一条不断换发动机的技术路线。早期 Agent 依赖规则、状态机、强化学习策略或 BDI 心智模型；LLM 出现后的变化，不是"终于有人想到了 Agent 这个概念"，而是系统第一次有机会把"理解自然语言目标"这件事交给一个通用模型来做。

理解这条演进线很重要，因为它回答了很多人会问的一个问题：**如果 Agent 的基本问题早就被提出来了，为什么到现在才爆发？** 答案藏在传统 Agent 的根本局限里。

### 1.1 没有 LLM 时，怎么做一个"会办事"的系统

先回到一个没有 LLM 的时代。

你想做一个"自动出差助理"。它需要根据你的日程、预算、偏好和公司政策，帮你安排航班和酒店。

这个系统至少要处理：

- 你什么时候出发？
- 哪些航班符合预算？
- 酒店离会议地点多远？
- 公司报销政策允许什么价位？
- 如果航班延误，是否要重新安排酒店入住时间？
- 是否需要在付款前让用户确认？

如果没有 LLM，这个系统只能靠工程师提前写规则：

```text
如果会议在上午 10 点之前，优先选择前一天晚上到达。
如果航班价格超过预算 20%，需要用户确认。
如果酒店距离会议地点超过 3 公里，降低推荐优先级。
...
```

规则少的时候，这种系统可以工作。但你能感觉到它有多脆弱——一旦需求变复杂，规则会迅速膨胀。用户说一句"我不想太折腾，但也别太贵"，系统就很难处理，因为"太折腾"和"太贵"不是简单的固定字段能捕获的。

这就是传统 Agent 面临的核心困境：

```text
系统需要像人一样围绕目标做判断，但工程上又必须把世界拆成明确的规则和状态。
```

这两件事天然矛盾。人的目标表达是模糊的、依赖语境的、随时可能变化的；而规则系统要求每一条路径都被预先定义。这个矛盾不是某个工程师偷懒造成的，它是"规则驱动系统"和"开放语言目标"之间不可调和的张力。

### 1.2 Agent 的基本问题从来没有变

无论有没有 LLM，Agent 要解决的基本问题都很稳定。你可以把它概括成一个循环：

```text
感知环境 → 判断状态 → 选择行动 → 执行动作 → 接收反馈 → 调整下一步
```

用出差助理的例子走一遍：

- **感知环境**：读取日程、航班、酒店、预算。
- **判断状态**：知道当前还没有订票，会议时间固定，预算有限。
- **选择行动**：先查航班，还是先查酒店？
- **执行动作**：调用航班搜索 API。
- **接收反馈**：发现下午航班便宜但到达太晚。
- **调整下一步**：改查前一天晚上的航班。

这个循环听起来很简单，但把它放进开放现实世界，每一步都会变得异常困难：

- 环境状态太复杂（哪些信息与当前目标相关？）
- 用户目标太模糊（"别太折腾"到底意味着什么？）
- 可能行动太多（先查这个还是先查那个？）
- 反馈信号不总是清晰（航班便宜但到达晚——这算好消息还是坏消息？）

传统 Agent 方法的努力方向，就是想办法把这些东西**形式化**——把模糊的世界翻译成系统能处理的结构。不同年代的研究者给出了不同的答案，但这些答案都面对同一个天花板。

### 1.3 强化学习 Agent、软件 Agent、BDI 和经典 Agent 分类

传统 Agent 里有几类重要思路。它们不是互相孤立的概念，而是不同年代对同一个问题的回答：

```text
如果系统要自己做决定，决策依据从哪里来？
```

#### 强化学习 Agent

强化学习给出的答案是：让 Agent 在环境中试错，用奖励信号改进策略。它的核心循环是：**观察状态 → 选择动作 → 接收奖励 → 更新策略**。

这条线索很早就开始了：1957 年 Bellman 提出动态规划中的最优性原理，1989 年 Watkins 提出 Q-learning。到了 2010 年代，DeepMind 用 DQN 玩 Atari 游戏（2013 年），后来又用 AlphaGo 击败李世石（2016 年），大众第一次强烈地感受到"一个系统可以通过和环境反复交互学会复杂策略"。

比如一个游戏 AI：
- 它看到当前画面和分数（状态）。
- 它决定向左、向右、攻击还是防守（动作）。
- 如果得分变高，说明这一步可能有用（正奖励）。
- 如果失败或被击败，说明这一步可能不合适（负奖励）。
- 经过大量尝试后，它逐渐学会什么状态下该做什么。

这类 Agent 在游戏、机器人控制、推荐策略等场景中非常重要，因为这些场景的共性很明显：**状态可以被定义，动作空间可以被枚举，奖励信号可以被明确计算**。

但把它放到普通用户任务里，问题就来了：
- 需要大量试错——你不能让一个出差助理订错 100 次航班来学习。
- 奖励函数很难设计——"用户满意度"不像游戏分数那样能被简单计算。
- 开放语言任务不容易转成明确状态和动作——"帮我整理一下这个项目"包含多少种可能的状态和动作？
- 用户目标经常不是一个简单分数能表达的——有时候你需要的是"还行"的答案，而不是"最优"的答案。

强化学习 Agent 的故事告诉我们一个很重要的道理：**如果反馈信号清晰，Agent 可以通过反复交互学会复杂策略。但现实世界的很多任务，最难的不是"学策略"，而是"定义什么是好的结果"。**

#### 软件 Agent

软件 Agent 给出的答案是：把任务环境做窄，把触发条件和动作写清楚。

在 1990 年代，Wooldridge 和 Jennings 对软件 Agent 做过经典定义，提出了两个层次的特征：
- **弱定义**：自治性、社交能力、反应性、主动性。
- **强定义**：在弱定义基础上，增加了信念、愿望、意图等心智状态（即 BDI 模型）。

MIT 媒体实验室的 Pattie Maes 在 1995 年开发的自主 Agent 已经很像"软件助理"——会议调度、邮件过滤、信息检索。这些 Agent 能代表用户执行任务，但前提是环境足够封闭、规则足够明确。

例如：
- 邮件过滤器自动分类邮件。
- 监控系统发现服务异常后自动重启。
- 数据同步程序定时把一个系统的数据写入另一个系统。

这些系统能"自动做事"，但边界非常稳定。一旦用户目标变成自然语言，比如"帮我把这个项目整理一下"，它们就不知道该把这句话映射成哪些状态、规则和动作。不是它们不够好，而是它们的"理解"机制从根本上不支持开放语言输入。

#### BDI 模型

BDI 给出的答案是：把人的行动结构拆成 **Belief（信念）、Desire（愿望）、Intention（意图）**。它试图把"一个靠谱助理如何做事"翻译成工程结构。

假设你让助理安排一次出差：
- **Belief**：他知道你下周三有会议、预算有限、晚上不想太晚到酒店。
- **Desire**：他的目标是让你顺利参加会议，并尽量减少路上折腾。
- **Intention**：他决定先查航班，再比较酒店，最后把方案发给你确认。

BDI 的价值在于，它为 Agent 提供了一个**围绕目标组织行为的认知架构**。它提醒我们：Agent 不只是被动响应输入，还应该围绕目标形成计划。这个思路和今天的 LLM Agent 非常接近——实际上，很多现代 Agent 框架的设计，内核仍然是"理解世界 → 持有目标 → 承诺执行 → 根据反馈调整"。

但 BDI 的工程实现极为困难。系统需要明确定义世界状态、目标、规则和计划空间。在封闭领域（比如工业控制），这件事可以做；但当任务进入开放自然语言世界，建模成本会急剧上升，因为你需要为每一种可能的用户表达预先设计状态和规则。

#### Russell & Norvig 的 Agent 分类

经典人工智能教材《Artificial Intelligence: A Modern Approach》里把 Agent 分成几类，这个分类框架今天仍然非常有帮助：

| 类型 | 大致含义 | 核心局限 |
|---|---|---|
| 简单反射 Agent | 根据当前输入直接反应，遵循"条件-动作"规则 | 不处理复杂历史和目标，像一个恒温器 |
| 基于模型的反射 Agent | 维护对世界状态的内部模型，能处理部分可观测环境 | 世界模型难以完整构建 |
| 基于目标的 Agent | 不仅知道当前状态，还知道目标状态，据此选择行动 | 目标和行动空间需要被建模 |
| 基于效用的 Agent | 在多个方案中计算偏好，选择"最优"的那个 | 效用函数很难设计——"好"的标准是什么？ |
| 学习型 Agent | 根据经验改进行为 | 学习信号和泛化能力难以控制 |

这五种类型不是互相排斥的，而是代表了从简单到复杂、从反应到推理的演进路径。重要的是，它们描述的 Agent 系统的基本问题——感知、建模、决策、评估、学习——不会因为技术换代而消失。LLM 时代只是给了这些问题**新的解法**，而不是让问题本身消失了。

### 1.4 封闭环境里可以工作，开放任务里很难扩展

读到这里你可能会想：传统 Agent 是不是都失败了？

不是。它们在很多封闭环境中非常有效：

- **游戏环境**：状态、动作、奖励都比较明确。AlphaGo 能在 19×19 的棋盘上击败人类最强棋手。
- **工业控制**：传感器、控制器、反馈信号有清晰定义。一个温度控制系统不需要理解自然语言。
- **企业自动化**：流程稳定，规则可以提前写好。ERP 系统中的审批流程不需要临场发挥。
- **运维脚本**：触发条件和处理动作比较固定。"如果 CPU 超过 90%，扩容一台机器"。

这些场景的共同点很清晰：

```text
世界可以被建模，行动空间可以被枚举，反馈信号可以被定义。
```

但普通用户日常任务不是这样。

用户会说：

```text
帮我看看这个项目最近为什么变慢了。
```

这句话背后可能涉及代码、日志、数据库、网络请求、产品改动、部署环境、用户行为。你很难提前写完所有规则。更糟糕的是，用户自己可能也不完全知道"变慢"意味着什么——是响应时间长了？是某个页面卡了？是整体吞吐量下降了？

所以传统 Agent 的根本限制不是"没有智能"。它们在各自擅长的领域非常智能。它们的限制是：

```text
它们很难低成本地理解开放语言目标，并把目标转成可执行决策。
```

你可以把传统 Agent 想象成一个非常优秀的象棋程序——它可以在棋盘内下出人类难以企及的妙手，但它永远无法理解"下棋"这件事的意义，也无法在棋盘被掀翻时知道该怎么办。它的智能是被"硬编码"进规则和策略中的，无法迁移到任何其他任务上。

### 1.5 开放语言世界需要新的决策核心

传统 Agent 留下了一个重要问题，这个问题在 LLM 出现之前几乎无法回答：

```text
如果用户目标无法完全规则化，Agent 的决策核心应该是什么？
```

在 LLM 出现之前，工程师只有几条路：要么继续写更多规则，要么为特定领域训练专用模型，要么把任务限制在非常窄的范围里。每条路都在回避真正的问题——用户想要的是一个**通用的、能理解自然语言目标的决策者**。

LLM 出现后，情况发生了变化。它让系统第一次具备了较强的自然语言理解、常识推理和任务解释能力。这不是说 LLM 完美无缺——它有很多问题，我们后面会详细讨论。但它的出现让 Agent 的核心问题从：

```text
如何把所有规则写出来？
```

变成了：

```text
如何让模型根据目标和上下文判断下一步？
```

这个转变听起来只是措辞不同，但它的工程含义是根本性的。"写规则"要求工程师提前预判所有情况；"让模型判断"意味着系统可以在运行时处理工程师没有预见到的任务。前者是封闭的、确定性的、但对开放世界无能为力；后者是开放的、灵活的、但带来了全新的工程挑战。

这就是第二章要讲的范式变化。

## 第二章：LLM 带来的范式变化

LLM 没有让 Agent 的所有问题消失，但它改变了 Agent 最关键的一层：**决策核心**。

这个变化不是从某一天突然开始的。2020 年 OpenAI 发布 GPT-3 后，研究者先注意到的是少样本学习（Few-shot Learning）：模型不需要为每个任务重新训练，只要看几个例子，就能模仿任务格式。这本身已经很惊人——它意味着模型在预训练过程中**学到了某种通用的"任务理解"能力**，而不仅仅是记住了训练数据。

随后 Chain-of-Thought 等工作进一步说明，模型还可以通过"写出中间推理过程"显著提高复杂任务的表现。到这一步，大家才开始认真思考一个更激进的问题：如果模型能理解目标、拆解问题、选择下一步，它是不是可以成为 Agent 的**通用决策核心**？不是某个特定领域的决策者，而是一个能处理很多不同任务的决策者？

### 2.1 用户用自然语言表达目标，系统却只认识规则

想象你在一个代码项目里说：

```text
帮我看看为什么登录接口最近变慢了，尽量找出原因。
```

这不是一个固定按钮能表达的任务。系统需要理解：

- "登录接口"可能对应哪些文件、路由、服务或日志？
- "最近"意味着要看时间范围、提交记录或监控数据。
- "变慢"可能是数据库、缓存、网络、代码逻辑或第三方服务导致的。
- "找出原因"不是简单生成答案，而是需要逐步排查。

如果用传统 Workflow 做，它必须提前定义流程：

```text
读取指标 → 查询日志 → 查询数据库 → 分析最近提交 → 输出报告
```

这个流程覆盖一部分场景没问题。但如果用户下一次问的是"帮我把这个模块重构一下"或"帮我评估这个 PR 有没有风险"，固定流程又不够了。你不可能为每一种可能的用户请求预先写一条 Workflow。

问题的本质是：

```text
真实任务的目标表达是开放的，但传统系统的决策方式是封闭的。
```

这个矛盾不是细节问题，而是范式问题。就像用一套固定的模具去接形状各异的液体——不管你做了多少套模具，总会有漏出去的部分。

### 2.2 Agent 的决策核心发生了变化

LLM 带来的最大变化，是它可以在自然语言层面承担一部分决策工作。

过去系统要靠工程师写：

```text
如果用户问性能问题，就先查指标。
如果用户问测试失败，就先运行测试。
如果用户问代码重构，就先读相关文件。
```

每一条规则背后都是一次"用户意图 → 系统动作"的预判。问题是，工程师不可能预判所有用户意图。

LLM 时代，系统可以把当前目标、已有上下文、可用工具和历史反馈交给模型，让模型判断：

```text
下一步最有价值的动作是什么？
```

这个转变的意义怎么强调都不过分。不是说模型一定判断正确——恰恰相反，模型经常判断错误，这也是整个 Agent 工程要解决的核心问题。但变化在于**决策机制本身**：

| 维度 | 传统系统 | LLM Agent |
|---|---|---|
| 目标理解 | 依赖预定义字段和意图分类 | 直接理解自然语言目标 |
| 下一步选择 | 由流程或规则决定 | 由模型结合上下文判断 |
| 任务覆盖 | 窄而稳定 | 更开放、更通用 |
| 失败模式 | 规则覆盖不到（直接失败） | 模型误判、幻觉、不稳定（可能错但不直接拒绝） |
| 工程重点 | 写规则和流程 | 管理上下文、工具、状态和控制边界 |

所以 LLM 不是简单"增强了 Chatbot"。它让 Agent 有机会从**"规则驱动"走向"目标驱动"**。这两者的区别，就像自动售货机和人类助理的区别——前者只能响应预定义的按钮，后者能理解你口语化的请求，尽管后者偶尔也会理解错。

但这只解决了一半。模型可以判断"下一步应该查日志"，并不代表它真的能查日志；它可以判断"需要调用订单系统"，也不代表它有权限、有接口、有执行结果可回填。Agent 的下一步演进，正是从"会判断"走向"能做事"——也就是第三章要讲的工具调用。

### 2.3 让 LLM 承担目标理解和下一步判断

在 LLM Agent 中，模型通常承担几类职责。理解这些职责，能帮你更清楚地看到 LLM 在 Agent 系统中的确切位置。

#### 目标理解

用户不会总是给出清晰的任务说明。模型需要把模糊目标转成可操作的目标。

例如：

```text
用户说：这个项目有点乱，帮我整理一下。
```

模型要判断这可能意味着：
- 先查看目录结构。
- 找出重复或废弃文件。
- 理解主要模块。
- 给出整理建议。
- 在得到确认后再动手修改。

这个过程需要常识、需要理解"乱"在软件项目语境下可能指什么，还需要知道"整理"在文件系统层面对应哪些操作。模型不一定一次猜对，但它至少能朝着这个方向展开行动，而不是直接拒绝说"我不理解你的指令"。

#### 上下文解释

Agent 不只看用户一句话，还要看一整套上下文：当前文件、历史对话、工具返回结果、错误日志、任务进度。模型需要把这些信息合起来，形成对当前状态的判断。

这比单纯的"回答一个问题"复杂得多。在回答模式下，模型只需要关注用户的问题和它自己的知识；在 Agent 模式下，模型需要持续追踪一个动态变化的信息空间，并分辨哪些信息对当前决策最重要。

#### 下一步决策

Agent 最关键的问题永远是：

```text
现在应该做什么？
```

下一步可能是：回答用户、继续读取文件、调用搜索、执行测试、请求用户确认、或者停止任务。这个决策的质量直接决定了 Agent 能不能完成任务。更重要的是，它是"瞻前顾后"的——模型需要考虑当前看到了什么、之前做过什么、还差什么没做。

#### 结果解释

工具返回的数据通常不是最终答案。模型还要解释结果，判断它是否足够支持下一步。

例如测试失败后，模型需要从报错里判断：是语法错误？是断言失败？是环境依赖缺失？还是测试本身已经过期？不同类型的失败需要不同的后续动作，模型必须做出区分。

### 2.4 从固定流程走向动态决策

LLM 让很多过去很难产品化的任务变得可能。

例如：

```text
帮我调研某个行业的最新趋势。
```

Agent 可以：
1. 拆解问题——这个行业的关键维度是什么？
2. 搜索资料——哪些来源比较可信？
3. 判断资料质量——这篇分析是有数据支撑还是纯观点？
4. 阅读页面——提取关键事实。
5. 提炼观点——跨来源比较。
6. 发现遗漏——补充关键信息。
7. 生成报告。

这个过程不是一个完全固定的流程。搜索结果不同，下一步就会不同；发现资料不足，Agent 需要继续查；发现信息冲突，Agent 需要对比来源并判断可信度。

这就是 LLM Agent 和普通 Workflow 的核心差异：

```text
Workflow 适合步骤稳定的任务——你知道从 A 到 B 到 C 一定是正确的。
Agent 适合目标明确但路径不完全确定的任务——你知道要去哪，但中间可能需要绕路。
```

选择哪种方式，不应该取决于"Agent 是不是更先进"，而应该取决于任务本身的特点。如果一个任务的步骤是可预测的，Workflow 往往更可靠、更快、更便宜。Agent 的优势不在"它看起来更智能"，而在它可以处理路径不确定的任务。

### 2.5 会判断不等于能可靠执行

LLM 成为决策核心之后，新问题马上出现——而且每一个都是硬问题：

| 新问题 | 表现 | 后续需要的能力 |
|---|---|---|
| 事实不足 | 模型不知道实时信息、私有数据或当前系统状态 | 工具调用、RAG、数据库连接 |
| 无法行动 | 模型只能输出文本，不能真实执行任务 | Tool Use、Function Calling |
| 状态不稳 | 多步任务中忘记前面做过什么 | State、Memory、Trace |
| 循环失控 | 一直查资料、重复调用工具、迟迟不结束 | Loop Control、Stop Condition |
| 错误传播 | 前一步错了，后面继续基于错误推进 | 纠错与反思、Evaluation、Guardrails |
| 风险动作 | 模型可能误删文件、误发消息、越权操作 | Permission、Human-in-the-loop |

这些问题的共同点在于：它们都不是"模型不够聪明"能概括的。一个再聪明的模型，如果无法接触实时数据、无法执行动作、无法记住前几步做了什么，它仍然不是一个有用的 Agent。

所以这里有一个非常重要的认识，它会贯穿整个课程：

```text
LLM 是 Agent 的决策核心，但不是完整 Agent 系统。
```

一个好的决策核心是必要条件，但远非充分条件。Agent Runtime 要解决的问题——工具连接、状态管理、循环控制、错误处理、安全边界——才是让决策核心真正有用的骨架。后面的内容，本质上都是在给这个骨架添砖加瓦。

> 读到这你可能在疑惑 "Runtime" 到底是什么。这里先给一个直觉：**你可以把 Runtime 理解为 Agent 的"操作系统"——模型负责做决策，Runtime 负责管执行。** 课程三会给出精确定义并展开。现在只需要带着这个印象往下读就行。

## 第三章：工具调用为什么出现

工具调用（Tool Use）是 Agent 演进中第一个被明确工程化的能力。它的出现背景非常直白：**模型很会说，但什么都做不了**。

这听起来是一个简单问题——接几个 API 不就好了？但仔细看，工具调用涉及其实际远比"接 API"深远。它牵涉到模型和外部世界的关系、决策和执行的边界、以及工具如何被发现、描述、调用和治理。这一章我们从研究到产品再到工程接口，把这条线完整地走一遍。

### 3.1 LLM 会说，但不会做

早期 LLM 很像一个聪明但"没有手"的人。

你说：

```text
帮我检查这个项目为什么测试失败。
```

它可能回答：

```text
你可以先运行测试命令，查看错误日志，再定位相关文件。
```

这段建议可能是对的，但它是"建议"，不是"行动"。模型没有真的运行测试，也没有读取日志。它只是在用语言告诉你什么是合理的做法——就像一个人告诉你"你应该去看医生"，但他不能替你看。

你说：

```text
帮我确认这个订单是否已经退款。
```

它可能回答：

```text
你可以去订单系统查询退款状态。
```

这仍然是建议。真正有用的系统应该能在权限允许的情况下调用订单 API，读取真实状态，再告诉你结果。

这种"思行分离"的状态，是限制 LLM 应用场景的最大瓶颈。企业想用它做客服，但它查不到订单状态；想用它做助手，但它不能帮你订机票；想用它做分析，但它无法读取实时数据。模型的知识（知道一个任务该怎么完成）和模型的行动力（真的去完成它）之间有一条巨大的鸿沟。

工具调用要解决的就是这条鸿沟：

```text
用户要的不是"告诉我怎么做"，而是"在可控边界内帮我做"。
```

### 3.2 回答能力必须连接外部能力

LLM 的参数知识有几个天然限制，而且这些限制不是"模型再大一点"就能解决的：

- **不实时**：模型训练完成后，世界继续变化。它不会自动知道最新事件、最新库存、最新订单状态。
- **不私有**：模型不知道你公司的数据库、项目文件、内部文档——这些数据从未进入过它的训练集。
- **不精确执行**：模型可以描述计算步骤，但它用"生成文本"的方式做计算，就像让人用心算代替计算器——大题很容易出错。
- **不改变世界**：模型输出文本不会自动发邮件、改文件、创建工单或调用业务系统。它在语言空间内活动，无法触及物理世界或数字世界的真实状态。

所以 Agent 系统必须把两类能力明确分开：

```text
模型负责理解目标和判断下一步（决策层）。
工具负责获取事实、执行动作和返回结果（执行层）。
```

这条边界非常重要。**模型不应该假装自己查过数据库，也不应该假装自己运行过代码。** 它应该生成工具调用请求，由 Runtime 执行真实动作，再把结果返回给模型。这不是信任问题，而是架构问题——让模型做模型擅长的事（语义理解和决策），让确定性系统做它擅长的事（精确执行和权限控制）。

### 3.3 Toolformer：让模型自己学会使用工具

当我们意识到 LLM 必须连接外部能力后，最直接的工程做法是：在系统外部写规则，告诉模型有哪些工具、什么时候应该调用。但这会带来一个新问题：

```text
工具调用到底是外部系统硬塞给模型的规则，还是模型自己可以学会的一种行为？
```

这个问题不是哲学游戏——它有重要的工程含义。如果模型只有在外部提示词的"强迫"下才能正确调用工具，那么工具调用就是脆弱的：换一套提示词、换一个场景、换一个模型版本，调用行为就可能变化。如果模型自己能学会"什么时候该调用工具"，那么工具调用就是一种更稳定的、模型内化的能力。

Meta AI 在 2023 年 2 月发表的 Toolformer，就是沿着这个问题往前走了一步。它不是先做一个复杂的工具平台，而是先研究：模型能不能把"什么时候调用 API、调用后如何继续写"学进自己的生成模式里。

#### 核心洞察：工具使用是一种语言模式

Toolformer 团队的关键洞察非常优雅：**"使用工具"本质上是一种语言模式。** 既然模型是通过学习语言模式来获得各种能力的，那为什么不让"使用工具"也成为语言模式的一部分呢？

这个思路的精妙之处在于，它完全避免了对模型架构的任何修改。不需要给模型加新的网络结构，不需要设计新的训练目标——纯粹通过数据驱动的方式，让模型在它的"母语"（文本序列）中学会何时插入 API 调用、如何构造参数、以及如何使用返回结果继续生成。

#### 实现方式

Toolformer 的做法不需要人工标注工具调用数据，而是用了一套自监督流程，分三步走：

**第一步：生成候选 API 调用位置**

给定一个文本语料（比如从 Common Crawl 中采样的网页文本），先用已有的语言模型在上面跑一遍。在文本的每个位置，用一个包含少量示例（few-shot）的提示词问模型：这里是否应该调用某个 API？如果是，API 的名称和参数应该是什么？

例如，原文是一段话 "The population of Berlin is..."，在省略号的位置，模型可能会被提示 "我应该在这里插入一个对 Wikipedia Search API 的调用，参数是 'Berlin population'"。

更具体地说，对于语料中的每个位置 i，Toolformer 会构造一个提示，其中包含 k 个随机挑选的上下文示例（每个示例都展示了在什么位置插入什么 API 调用），然后让模型以概率 p 决定是否在位置 i 插入 `[API_name(parameters)]` 标记。为了防止过度生成，并不是每个位置都候选——只有模型给出足够自信的位置才会进入候选池。

**第二步：用困惑度筛选有价值的调用**

这是整个方法最精妙的一步。有了候选 API 调用后，Toolformer 会实际执行这些 API 调用，获取真实的返回结果（比如 Wikipedia 返回了 "Berlin's population in 2021 was 3.64 million"）。

然后，把返回结果填入原文，得到 "增强版" 的文本。现在比较两个版本：
- 不使用 API 返回结果时，模型对后续文本的预测损失（困惑度）是多少？
- 使用 API 返回结果后，模型对后续文本的预测损失降低了多少？

如果插入 API 结果后，模型对后面几个 token 的预测明显更好了（困惑度显著下降），说明这个 API 调用确实提供了有用的信息。如果困惑度反而上升（比如 API 返回了无关内容），说明这个调用没有价值，会被丢弃。

这个筛选策略是自监督的——不需要人来标注 "这个调用对不对"，模型的预测质量本身就是信号。论文用一个加权交叉熵损失来量化这种改善：只计算调用结果之后的 token 的损失，并且给予离调用位置更近的 token 更高权重。

**第三步：用高质量样本微调模型**

筛选后保留下来的 "原始文本 + API 调用标记 + API 返回结果 + 后续文本" 序列，被拼成新的训练数据。用这些数据对模型做 SFT（监督微调），模型就学会了：
- 在生成过程中，如果遇到需要外部信息的位置，插入 `[API_name(parameters)]` 标记；
- 外部系统执行 API 后，将返回结果插入到 `[API_name(parameters) → result]` 位置；
- 基于这个结果继续生成后续文本。

**推理时的工作方式**

微调后的模型在推理时，如果生成过程中输出 `→` 符号，系统会暂停生成，解析前面的 API 调用、执行它、将结果填入，然后让模型从断点继续生成。整个过程对用户透明——用户看到的是模型"自己知道什么时候查资料"。

论文中使用的工具包括计算器、问答系统、维基百科搜索、翻译和日历等，覆盖了不同类型的 API 场景。

#### 产生的效果和局限

Toolformer 证明了一件非常重要的事：

```text
工具使用不只是外部系统给模型加的能力，也可以成为模型学习到的行为模式。
```

这对 Agent 很关键。因为 Agent 的工具调用不能只靠硬编码规则——模型本身也需要能判断"我现在需要外部信息"和"我现在应该直接回答"之间的区别。

但 Toolformer 更偏研究范式，不是普通开发者最直接接触的接口。它没有完整解决：工具权限如何控制、工具执行失败如何处理、工具结果是否可信、高风险动作是否需要用户确认。这些问题会在产品和工程层面继续展开。

### 3.4 ChatGPT Plugins：把"AI 可以做事"变成产品体验

Toolformer 回答的是研究问题：模型能不能学会使用工具。产品侧很快遇到另一个更紧迫的问题：

```text
普通用户如何真正感受到 AI 不只是会回答，而是能做事？
```

用户对 LLM 的期待迅速从"回答问题"膨胀到"帮我完成任务"。查实时新闻、查机票和酒店、订餐、读取公司知识库、运行计算、调用第三方服务——这些需求在 ChatGPT 爆红后的几个月内集中涌现。

2023 年 3 月，OpenAI 发布 ChatGPT Plugins。它的历史意义不在于技术有多复杂，而在于：**它让主流聊天产品中的一批早期用户和开发者第一次在同一个聊天入口里看到，模型可以读网页、跑代码、查外部服务，甚至连接第三方业务系统。** 这在心理层面打破了"LLM 只是聊天工具"的认知边界。

#### 产品探索

一个插件需要向模型暴露两类信息：
- **面向模型的能力描述**：这个插件能做什么，什么时候适合用。
- **面向系统的接口描述**：有哪些 API、参数是什么、认证方式是什么。

这让模型可以在对话中"看到"可用工具，并在需要时发起调用。早期插件形态包括浏览器（访问最新网页）、代码解释器（运行代码和处理文件）、检索插件（查询私有或组织知识）、以及各种第三方服务插件。

#### 产生的效果和暴露的问题

Plugins 改变了大众对 LLM 的直觉：

```text
AI 不再只是生成文本，而是可以连接外部世界。
```

这一步为整个 Agent 领域打开了一扇门。早期用户和开发者在通用聊天入口里看到"模型 + 工具"的组合能力，产品经理和工程师开始认真构想"AI 能帮我做 X"的场景。

但 Plugins 也暴露了大量工程问题：工具需要被准确描述给模型；工具调用需要权限和安全边界；用户需要知道模型什么时候在调用外部服务；第三方工具可能带来隐私、越权和供应链风险；工具返回的内容可能诱导模型执行错误动作。

Plugins 的意义不在于它是唯一形态——事实上它后来被 GPTs 等形态取代了——而在于它把"LLM 连接外部世界"这件事推向了大规模产品体验，让整个行业看到了可能性和问题。

### 3.5 Function Calling：工具调用的工程化接口

Plugins 证明了"AI 可以做事"的产品体验，但它更偏 ChatGPT 自身的插件生态。开发者真正需要的是一种更底层、更稳定的接口，把自己的函数、数据库、内部系统和外部 API 接入任何使用 LLM 的应用。

例如你在自己的应用里希望模型能：根据用户请求创建订单、从文本中抽取结构化字段、查询数据库、调用 CRM 系统、触发内部工作流。这需要比"插件商店"更底层、更可控的接口。

2023 年 6 月，OpenAI 在 API 中推出 Function Calling。相比 Plugins，它更像一次**接口的标准化**：开发者不再只是把工具接到 ChatGPT 产品里，而是可以在自己的应用中声明函数、参数 Schema 和执行逻辑，让模型生成结构化调用请求。

#### 设计哲学：模型决策，环境执行

Function Calling 的基本流程是：

```text
开发者定义工具名称、描述和参数 Schema
    → 模型根据用户目标决定是否调用工具
    → 模型生成结构化参数（JSON）
    → Runtime 校验参数
    → Runtime 执行真实函数或 API
    → 工具结果返回给模型
    → 模型继续判断下一步或生成最终回答
```

这里有一个非常关键的设计选择：**模型不是直接执行函数，它只是生成一个结构化的调用意图。** 真正执行发生在开发者的系统里。

为什么这样设计？原因在于**真实性和安全性**：
- 模型的输出是不可靠的——它可能算错参数、编造不存在的订单号、或者给出一个看起来合理但实际不存在的函数名。
- 让模型"假装"执行等于放弃了所有外部验证的机会。
- 把执行权留在开发者手中意味着：数据库查询返回的是真实结果；危险操作（转账、删除）可以有权限控制；每次工具调用都在开发者的监控和审计之下。

这个设计体现了一个根本性的工程哲学：

```text
模型负责判断 what to do，Runtime 负责执行 how to do。
```

这种分工让 LLM 专注于它擅长的语义理解和决策，把精确执行交给确定性系统。这个原则后来被几乎所有 Agent 框架继承，是理解 Agent 架构的一条核心线索。

#### 产生的效果和新的工程问题

Function Calling 让很多应用可以稳定地把自然语言连接到业务系统。它把工具调用从"产品能力"变成了"开发者接口"，这意味着任何一个应用都可以让 LLM 调用它的内部函数，不仅仅是 ChatGPT 的插件生态。

但它没有消除复杂度。开发者仍然要处理：Schema 设计是否准确、参数校验和重试逻辑、工具结果是否可信、权限如何控制、高风险动作是否需要用户确认、模型版本变化是否影响调用行为、以及 Trace 和 Evaluation 如何覆盖工具链路。

### 3.6 MCP：工具连接的标准化尝试

当 Function Calling 解决了"模型如何表达工具调用"后，工程界很快撞上下一层问题：

```text
每个模型提供商、每个应用、每个工具服务器都有自己的连接方式。
```

这就像一个世界里的每个电器都有自己的电源插头标准——对生态发展是巨大的阻力。开发者如果同时使用多个模型提供商，就需要维护多套工具描述格式；如果想让 Agent 连接不同的工具和数据源，就需要为每一个写一套集成代码。

2024 年底，Anthropic 发布了 **MCP（Model Context Protocol，模型上下文协议）**，试图解决一个更根本的问题：如何让 Agent 连接到任意数据源和工具，而不需要每次都重新集成？

MCP 的设计灵感来自 **LSP（Language Server Protocol）**。LSP 让任意编辑器都能支持任意编程语言的智能提示——编辑器不需要为每种语言写一个插件，只需要实现 LSP 客户端，语言服务器只需要遵循 LSP 协议。MCP 的目标是让任意 Agent 都能使用任意工具和数据源，通过一个标准化的客户端-服务器架构。

MCP 定义了标准化的数据模型：
- **Resources**：数据（文件、数据库记录、API 响应等）。
- **Prompts**：提示模板。
- **Tools**：可执行函数。

这意味着工具和数据源连接开始出现一个**跨宿主、跨工具生态的开放标准化方向**。它代表了一个重要变化：工具调用的演进已经从"模型能不能调用函数"，进一步走向：

```text
工具如何被发现、描述、授权、调用、记录和治理？
```

### 3.7 工具调用是一整套系统机制

工具调用让 LLM 从"会说"走向"会做"，但它马上带来新的工程问题。这些问题不是"接一个 API"能解决的：

| 新问题 | 表现 | 后续课程 |
|---|---|---|
| 工具定义问题 | 模型不知道工具能做什么，或误解工具边界 | 课程四 |
| 工具选择问题 | 该用不用，不该用乱用，或选错工具 | 课程四 |
| 参数生成问题 | 参数类型错误、遗漏必填项、编造不存在对象 | 课程四 |
| 执行失败问题 | API 超时、权限不足、结果太长、外部服务错误 | 课程四 / 课程六 |
| 安全问题 | 误删文件、越权查询、发送错误消息、触发真实世界影响 | 课程四 / 课程六 / 课程七 |
| 可观测问题 | 用户和开发者不知道模型为什么调用这个工具 | 课程六 / 课程七 |

把工具调用的三个子问题拆开看会更有帮助：

1. **工具选择**：当有 100 个工具可用时，Agent 如何判断该用哪个？这本质上是语义匹配和效率的平衡问题。
2. **参数填充**：Agent 需要从对话上下文和工具描述中"猜"出函数需要的参数。参数可能来自历史对话、可能需要多步推理才能确定、也可能缺失需要追问用户。
3. **工具组合**：复杂任务往往需要按顺序调用多个工具——先搜索信息，再分析，再生成报告，再发送邮件。这实际上已经进入规划问题的范畴。

工具调用解决了"LLM 不能行动"的问题，但还没有解决"什么时候行动、行动后如何继续"的问题。这就引出了 ReAct。

## 第四章：ReAct 的意义

如果要选一篇对 Agent 领域影响最深的论文，很多研究者会指向 ReAct。不是因为它提出了多么复杂的理论，而是因为它解决了一个此前被忽视的问题：**推理和行动被错误地分开了。**

ReAct 是理解 Agent 运行循环的经典框架。它的核心不是一个具体的 API 或一个框架的名字，而是一种运行方式：

```text
Reasoning + Acting
```

也就是把推理和行动交织在一起，让它们互相喂给对方。

2022 年 10 月，Google Research 和 Princeton 联合发表《ReAct: Synergizing Reasoning and Acting in Language Models》。这篇论文的重要性在于，它把两条当时已经分别出现的能力放到了一起：一条是 Chain-of-Thought 展示的逐步推理，另一条是模型调用外部环境或工具的行动能力。ReAct 的贡献不是发明"推理"或"行动"，而是让它们**在同一个循环里互相喂给对方**。

### 4.1 一次性回答无法处理变化中的任务

让我们用一个具体的例子来感受这个问题。

假设你问 Agent：

```text
帮我找出这个项目测试失败的原因。
```

一个只会一次性回答的模型可能说：

```text
你应该先运行测试，然后查看错误日志，再检查相关代码。
```

这是建议，不是解决问题。模型在猜测答案，而不是在**与真实状态交互**。

一个只会调用工具但不会推理的系统可能机械执行：

```text
运行测试 → 输出一大段日志 → 结束
```

它拿到了信息，但没有判断下一步该看哪个文件，也没有解释错误原因。它像是一个只会执行指令但不会思考的助手。

真实任务需要的是：
1. 先判断当前缺什么信息。
2. 采取行动获取信息。
3. 观察行动结果。
4. 根据新信息调整判断。
5. 继续这个循环，直到足够回答或应该停止。

这个过程不能是"先全部想好再全部行动"，也不能是"先全部行动再全部分析"。它必须是交替的、动态的、根据每一步的结果调整的。

### 4.2 推理和行动被错误地分开了

在 ReAct 之前，研究者对 LLM 能力的探索大致分成两条线：

- **推理线**：让模型在文本里分析问题，写出中间步骤，再给答案。代表工作是 Chain-of-Thought。
- **行动线**：让模型输出某个动作或调用某个工具，从环境中获取信息。

问题是，这两条线基本是分离的。

如果只有推理：模型困在自己的参数内部。它无法接触最新事实，容易基于错误假设继续推理，也不知道外部环境的真实状态。就像一个被关在房间里的人，不管逻辑多强，也无法知道外面是否在下雨。

如果只有行动：系统不知道为什么调用这个工具。工具结果不会自然转化为下一步策略，多步任务缺少连贯判断。就像一个只会按按钮的人，每个按钮都按了，但不知道为什么要按、按完之后该干什么。

人类解决复杂问题的方式从来不是单纯的"推理"或单纯的"行动"——而是两者的交织。你遇到一个问题，先想一下，然后查资料；看到新信息后改变想法；再采取下一步行动。推理和行动互为输入输出，形成一个动态循环。

ReAct 的关键洞察就是把这种人类式的问题解决节奏，翻译成了 LLM 可以执行的文本协议。

### 4.3 ReAct 把判断和行动放进同一个循环

一个简化的 ReAct 过程如下：

```text
Thought: 我需要知道测试具体失败在哪里。
Action: RunTests["npm test"]
Observation: 有 2 个测试失败，错误指向 auth/session.spec.ts。

Thought: 错误和 session 过期逻辑有关，我应该读取相关实现。
Action: ReadFile["src/auth/session.ts"]
Observation: 代码中 expiresAt 使用了秒级时间戳，但测试传入毫秒级时间戳。

Thought: 现在可以解释失败原因，并给出修复建议。
Final Answer: 测试失败是因为时间单位不一致...
```

这个循环有三个关键点：

- **Thought** 让模型表达当前判断——为什么选择这个动作、当前知道什么、还缺什么。
- **Action** 让模型影响外部环境或获取外部信息——不只是"说应该怎么做"，而是真的去做。
- **Observation** 让工具结果回到上下文，影响下一步判断——外部信息不再只是"被输出"然后丢掉，而是成为下一轮决策的输入。

实际产品不一定会把 Thought 展示给用户。重要的是系统内部要有类似结构：

```text
模型判断下一步 → 工具执行 → 结果回填 → 模型继续判断
```

这个模式的深层意义在于：**模型不是在"孤岛"上推理，而是在与外部世界的持续交互中推理。** 每一次观察都可能改变它的思考方向，每一次思考都可能触发新的行动。这让 LLM 从一个"回答机器"变成了一个"问题解决器"。

### 4.4 Agent 有了最小运行模式

ReAct 重要，不是因为所有产品都必须显式输出 Thought 文本，而是因为它清楚展示了 Agent 从 Chatbot 走向 Agent 的**最小运行模式**。

它把一次性回答变成了：

```text
判断下一步 → 调用工具或行动 → 接收观察 → 继续判断 → 停止
```

这正是课程三要实现的最小 Agent 闭环。

你可以把 ReAct 理解成 Agent 的**"最小行为语法"**。它不等于生产系统——生产系统需要更多的状态管理、错误处理、循环控制和用户交互——但它让我们看清生产系统要管理什么。就像学会了基本的语法规则不等于能写出好文章，但不知道语法规则一定写不出好文章。

ReAct 的意义可以概括为：
- 将推理、行动、观察放入同一个循环，让它们互相喂给对方。
- 让 Agent 可以根据外部反馈调整下一步，而不是一条路走到黑。
- 给出了最小运行时的雏形，后世的 Agent 框架几乎都建立在这个核心循环之上。
- 暴露出后续工程问题——循环本身需要管理，这正是下一章要讲的。

### 4.5 循环本身也需要被管理

ReAct 本身是范式，不是生产系统。它给出循环结构，但没有定义循环的边界。

它会引出很多后续问题：

| 问题 | 说明 | 后续课程 |
|---|---|---|
| 停止条件 | Agent 什么时候应该结束？它怎么知道"够了"？ | 课程三 / 课程六 |
| 循环控制 | 如果一直调用同一工具、在同一个问题上打转怎么办？ | 课程三 / 课程六 |
| 工具失败 | 工具超时、报错、返回无关结果——循环如何处理？ | 课程四 / 课程六 |
| 上下文膨胀 | Observation 越来越多，上下文窗口迟早放不下 | 课程六 |
| 状态管理 | 哪些信息进入下一轮决策？哪些只写入日志？ | 课程三 / 课程六 |
| 用户控制 | 高风险动作前是否需要确认？用户能不能中途改变方向？ | 课程四 / 课程七 |

在这些问题中，有一个贯穿性的挑战：**当循环中某一步出错时，系统如何发现并修正？** ReAct 给出了循环骨架，但骨架本身没有纠错能力——Thought 可以误判，Action 可以失败，Observation 可以不完整。这就需要下一章要讲的能力：纠错与反思。

## 第五章：纠错与反思 —— 从两层能力到自我修正

ReAct 给了 Agent 一个行动循环，但循环本身不保证正确。在多步任务中，错误是常态而非异常：模型会误解目标，工具会返回意外结果，外部信息会和内部知识冲突，用户需求也可能中途变化。

这一章回答的问题不是"模型能不能再想一遍"，而是一个更深层的追问：

```text
系统如何知道自己错了，并把这个发现转化成有效的修正？
```

我们会从"为什么模型内部推理不够"出发，建立两层能力的框架，然后展开 Reflection 的演进故事——从 Self-Refine 的"作者+编辑"模式到 Reflexion 的情景记忆，最后汇集到一个核心原则：纠错必须依赖真实信号，不能变成自我安慰。

### 5.1 让模型"多想想"为什么不够

假设 Agent 要修改一段代码。它先读取文件，然后给出修改方案，最后编辑代码。

如果模型在第一步误解了需求——比如把"优化性能"理解成"简化代码"——后面可能全部建立在错误理解上。模型越自信，走得越远，错得越深。

你可能会想：

```text
那让模型多思考几步不就行了吗？
```

这有帮助，但不够。原因在于，很多错误不是靠"想得更久"就能发现的：

- 代码是否真的能编译，需要运行编译器——模型"觉得"能编译不算。
- 测试是否通过，需要运行测试——模型"觉得"逻辑正确不算。
- API 是否可用，需要真实调用——模型"记得"的 API 文档可能已经过时。
- 数据库查询是否正确，需要连接真实数据库——模型"推测"的表结构可能是错的。
- 用户是否接受某个方案，需要用户确认——模型"猜测"的用户偏好可能不对。

模型内部推理再强，也不能替代外部验证。这不是模型能力的问题，而是信息获取的问题。**模型内部的推理永远基于它已有的信息，而有些信息只有通过和外部世界交互才能获得。** 你把一个人关在房间里，不管他逻辑多强，也无法知道外面是否在下雨。

### 5.2 两层能力：模型内部推理与系统外部反馈

把这两层能力区分清楚，是理解 Agent 可靠性的关键。这也是本课最重要的概念框架之一。

**模型内部推理**指的是模型在生成答案或决策前，能更好地处理复杂问题、拆分步骤、比较路径、发现局部矛盾。它发生在模型生成过程中，依据的是上下文和模型自身的知识。它的优点是快、灵活、泛化能力强。它的局限是可能在内部自洽但对外错误——逻辑上没毛病，但前提假设就错了。就像一个学生在试卷上写出了完美的推理过程，但题目就理解错了。

**系统外部反馈**指的是运行时从模型外部收集信号，判断当前结果是否真的可用。它发生在 Agent Runtime 中，依据的是工具结果、验证器、编译器、测试框架和用户反馈。它的优点是更接近真实世界，不依赖模型"觉得自己对不对"。它的局限是需要工程设计和成本投入——你需要搭建测试体系、校验逻辑、监控链路。

两者的区别可以概括为：

| 维度 | 模型内部推理 | 系统外部反馈 |
|---|---|---|
| 发生位置 | 模型生成过程中 | Agent Runtime 中 |
| 依据 | 上下文和模型能力 | 工具结果、验证器、测试、用户反馈 |
| 优点 | 快、灵活、泛化强 | 更接近真实世界，可验证、可复现 |
| 局限 | 可能自洽但错误——逻辑对但事实错 | 需要工程设计和成本投入 |
| 典型代表 | CoT、ToT、模型自评 | 编译器报错、测试失败、Schema 校验失败、用户驳回 |

一个可靠的 Agent 需要这两层能力配合。只靠模型内部推理，系统容易陷入"自信的错误"——每一步都推理得很漂亮，但方向一开始就偏了。只靠外部反馈，系统缺乏理解目标和判断取舍的能力——它可能机械地修了一个 bug，但不知道这个 bug 值不值得修。关键不是选一边，而是**让外部反馈在模型可能出错的地方设检查点，及时介入。**

### 5.3 错误如何在多步任务中传播和放大

理解了这两层能力的区别之后，再看多步任务中一个更危险的现象：错误不会静态地"待在那里"，而是会**沿着任务链传播和放大**。

假设 Agent 正在写一份竞品分析：

- 第一步，它把 A 公司和 B 公司的信息混淆了。这个错误本身不大——只是张冠李戴。
- 第二步，它基于混淆后的信息做功能对比。对比分析本身可能非常细致，逻辑无可挑剔，但对比的**基础**是错的。
- 第三步，它根据错误对比给出产品建议。建议看起来很合理、论证充分，但前提事实已经被污染。
- 最后报告看起来结构完整、论证严密，但核心事实是错的——而且错误被层层推理"洗白"了，变得非常难以发现。

这类问题不是简单的格式错误或语法错误，而是**"错误沿着依赖链传播，每一步的输出是下一步的输入"**。一步错，后面的步骤越认真越危险——因为它们会把错误越埋越深。

代码任务里也一样：读错需求 → 改错文件 → 测试没跑全 → 看到一个错误就以为是根因 → 修复 A 问题时引入 B 问题。每多一步，问题就多缠一层。

如果系统没有纠错机制，Agent 就像一个闭着眼睛走路的人——他可能在迈出第一步时就偏了方向，但自己完全不知道，直到撞到墙上。而且他走的每一步都在让"撞墙"的后果更严重。

### 5.4 把反馈机制放进运行时

要阻止错误传播，系统必须在运行时引入可验证的反馈机制。这些机制不依赖模型"觉得对不对"，而是用外部信号约束模型。

#### Schema 校验

如果模型需要生成结构化参数，就用 Schema 检查：

```text
字段是否存在？类型是否正确？枚举值是否合法？
```

这层校验虽然简单，但极为有效——它能在模型输出进入执行层之前拦截大部分格式错误。它便宜、快速、不消耗 Token，是性价比最高的纠错手段。

#### 工具执行反馈

工具执行后，把真实结果完整返回给模型，而不是只返回"成功"或"失败"：

```text
成功时的具体数据、失败时的错误码和原因、超时、权限不足、结果为空、结果过长。
```

模型需要看到真实结果才能做出合理的下一步判断。如果只告诉它"失败了"，它就无从知道应该换一个工具、调整参数、还是改变策略。

#### 测试和验证器

对于代码任务，测试比模型自评更可信。这是一个硬原则：

```text
模型说"修好了"不算，测试通过才算。
```

模型可以很有说服力地解释为什么某个修改是正确的，但如果测试不通过，解释再漂亮也没用。测试是 Agent 系统中最重要的外部反馈信号之一，因为它直接验证了"修改是否达到了可观测的效果"。

#### 用户确认

对于高风险动作，用户确认是重要的反馈源。例如删除文件、发送邮件、创建订单、修改生产配置——这些动作一旦执行就可能有真实世界影响，模型不应该单方面决定。把用户拉入循环，不仅是安全需要，也是最精确的反馈信号——最终是用户，而不是模型或测试，定义了"对"与"错"。

#### 模型自评

让模型检查自己的输出：是否遗漏需求？是否逻辑矛盾？是否与前面的信息冲突？

优点是成本低、适用范围广。缺点是模型可能自我确认——它倾向于为自己的答案辩护，而不是挑战它。一个错误的前提推导出的结论，在模型自己的逻辑框架里可能看起来非常合理。**模型自评应该被当作辅助信号，而非主要依据。**

#### 专门评估器

用另一个模型或规则系统评估输出。优点是可以分离生成和评价——生成者不会同时当裁判。缺点是评估器本身也可能判断错误，而且多一次调用就多一份成本。

### 5.5 Self-Refine 与 Reflexion：反思成为 Agent 的内置能力

上述反馈机制回答了"有哪些信号可以用"。但下一个问题是：**如何把这些信号组织成系统化的纠错流程，让它成为 Agent 运行循环的内置部分？**

2023 年对这个问题的探索产生了两个相互补充的重要工作。

#### Self-Refine：作者与编辑

2023 年 3 月，CMU 和 Allen AI 发表 Self-Refine。它的核心思想很朴素但极为有效：**用同一个 LLM 既当"作者"又当"编辑"。**

工作流程是一个迭代循环：
1. 生成初始输出（作者角色）。
2. 对初始输出进行批评，指出问题（评论家角色）。
3. 根据批评意见改进输出（作者+编辑角色）。
4. 重复 2-3，直到输出质量满足要求或达到最大迭代次数。

Self-Refine 的设计巧妙之处在于**角色切换**。同一个模型在 Prompt 的引导下，可以扮演完全不同的认知角色。"作为作者"时，它专注于生成内容，思维是发散的、建设性的；"作为评论家"时，它切换到批判性思维模式，寻找漏洞、矛盾和不完整之处。这种角色切换比模糊地说"请检查你的答案"更有效，因为它**明确分开了"创造"和"批判"这两种本质上不同的心理模式**——而这两种模式混在一起正是导致"自我审查"困难的原因。

**局限也很诚实**：如果模型的初始输出包含"系统性错误"（而非偶然失误），那么自我批评可能只是强化了同一种错误的逻辑——毕竟批评者和作者用的是同一个模型，共享同样的知识盲区。一个看不懂代码的模型，换了角色也看不懂代码。

#### Reflexion：把反思嵌入 Agent 循环

2023 年 3 月，Noah Shinn 等人发布 Reflexion 预印本。它把反思从一个"独立的迭代过程"升级为 **Agent 决策循环的内置组件**。

工作机制：在传统的"行动-观察"之间插入一个**反思步骤**：

```text
任务 → 推理 → 行动 → 观察 → 反思 → 记忆 → 推理 → 行动 → ...
```

其中"反思"步骤会分析上一次行动的结果：
- **成功了？**——为什么成功？有什么经验可以记住？
- **失败了？**——为什么失败？如果是幻觉导致的错误，下次应该怎么做？
- **部分成功？**——哪些部分做对了，哪些需要调整策略？

Reflexion 的关键创新在于：反思的结果不是一次性使用的，而是被存储到**情景记忆（Episodic Memory）**或**语义记忆（Semantic Memory）**中。这意味着 Agent 不是每次都从零开始——它会在当前任务中**积累经验**。之前犯过的错、走过的弯路、成功过的策略，都会被记录下来，供未来的决策参考。

**设计灵感**来自人类的"元认知（Metacognition）"研究。元认知的核心是"对自己的认知过程进行监控和调节"——你不会只是机械地完成任务，而是会不时退一步想："我做得怎么样？这个方法对吗？是不是该换个思路？"Reflexion 把这个能力给了 Agent。

#### 两者的关系

Self-Refine 和 Reflexion 不是竞争关系，而是互补的。Self-Refine 回答的是"如何让单次输出变得更好"（**微观迭代**）；Reflexion 回答的是"如何让整个任务执行过程变得更聪明"（**宏观学习**）。两者共同推动了一个变化：模型不再只是一次性生成结果，而是可以在运行过程中利用反馈持续修正自己。

### 5.6 从"一次生成"到"可迭代改进"

有了上述纠错和反思机制，Agent 可以处理更多真实任务中的不确定性。它不再只是"一次性输出"，而是能基于反馈持续迭代。

例如代码修复：

```text
修改代码 → 运行测试 → 测试失败 → 分析失败原因 → 调整策略 → 再次运行 → ...
```

例如报告写作：

```text
生成初稿 → 检查是否覆盖所有维度 → 发现缺少价格信息 → 补充检索 → 更新报告 → ...
```

这个转变很重要。它意味着 Agent 从"生成器"变成了"改进器"——它可以在运行过程中变得越来越好，而不是一成不变地交出第一版结果。每一步失败都在缩小问题空间，每一次修正都在逼近正确答案。系统不再只是相信模型，而是在用真实信号约束模型。

### 5.7 纠错不能变成自我安慰

纠错机制听起来很美好，但如果反馈信号不可靠，它也会出问题。以下是最常见的陷阱：

#### 模型自评过于乐观

模型倾向于确认自己的答案，而不是挑战它。让它"检查一下自己的答案"不如让它从另一个角度重新解题——角色切换比模糊的自我审视更有效。

#### 评价标准太模糊

如果判断"好不好"的标准是模糊的，纠错过程就会变成随机游走。每次修改都往不同方向走，永远不收敛。结构化评估（按固定维度逐项检查）比自由发挥式的反思更可靠。

#### 工具结果不完整

部分成功、部分失败的情况，如果只返回"失败了"三个字，模型无法准确判断哪里出了问题。工具反馈需要包含足够的细节来支撑下一步决策。

#### 测试覆盖不足

测试通过了不代表代码没问题——可能只是没测到出问题的部分。对测试结果的信任程度，取决于测试覆盖的质量。

#### 无限修正

Agent 一直觉得还能改，永远不停止——完美主义导致无法交付。"差不多就可以"的停止条件，在 Agent 系统中和"发现问题要修"同样重要。

#### 目标漂移

修着修着偏离了用户原始目标——"我本来只想改一个 bug，现在成重构了。"每次反思时都需要回看原始目标，检查是否还在正确的方向上。

| 典型陷阱 | 表现 | 对策 |
|---|---|---|
| 模型自评过于乐观 | 模型为自己的错误辩护，把错的解释成对的 | 用外部信号（测试、编译、工具返回）而非模型自评作为主要依据 |
| 评价标准太模糊 | 每次修改往不同方向走，永远不收敛 | 结构化评估：按固定维度逐项检查，而非自由发挥 |
| 工具结果不完整 | 只返回"失败"，无法定位具体问题 | 工具反馈需包含错误码、原因、上下文 |
| 测试覆盖不足 | 测试通过但问题仍在 | 测试质量决定了反馈信号的质量 |
| 无限修正 | 完美主义导致永远不交付 | 设置最大迭代次数和明确的停止条件 |
| 目标漂移 | 修着修着偏离了原始任务 | 每次反思时回看原始目标 |

关键不是"多反思"，而是**在正确的时间、对正确的问题、用正确的信号进行反思**。未来的 Agent Runtime 会更依赖可观测的组合信号：

```text
模型反思 + 工具验证 + 测试反馈 + 用户确认 + 可观测 Trace
```

任何单一信号都不足以支撑可靠的纠错。只有当多种信号交叉验证时，系统才能真正区分"需要修正的错误"和"不必要的自我怀疑"。

纠错让 Agent 循环的每一步更可靠，但它回答不了另一个问题：**当任务本身很长、步骤之间有复杂的依赖关系时，光靠"下一步判断"够吗？** ReAct 擅长根据当前观察选择下一步，但如果任务有十几个步骤，而且第三步的输出决定了第七步能不能做，单纯依赖"走一步看一步"就会漂移。这就是 Planning 要解决的问题——不是让每一步更正确，而是让整条路径有结构。

---

## 第六章：Planning —— 当任务太长，一步一判不够用

Planning 和上一章的纠错与反思，解决的是同一个核心循环的不同维度问题。纠错让单步决策更可靠，Planning 让多步路径更可控。它们共同回答一个追问：**ReAct 给出的行动循环，在大任务面前怎么才能不失控？**

这一章先不从论文讲起，而是先看长任务为什么会漂移。等你看见了"局部最优""遗漏约束""步骤错序"这些真实失败模式，再回头看 Chain-of-Thought、Tree of Thoughts 和 Plan-and-Execute，它们每一步在解决什么会清楚得多。

### 6.1 复杂任务不是一步回答，而是一串依赖关系

假设你让 Agent 做一件事：

```text
帮我调研 3 个竞品，比较它们的定位、功能、价格和增长策略，最后给出我们产品的建议。
```

如果 Agent 直接用 ReAct 模式开始搜索，很容易出现几种问题：

- 只查了第一个竞品就开始写结论——因为第一个竞品的信息最丰富，模型被"吸"进去了。
- 查了很多资料但没有统一比较维度——每个竞品的信息格式不同，直接堆在一起没有可比性。
- 忘记价格和增长策略——查着查着就只关注了功能和定位。
- 引用来源混乱——不记得哪条信息来自哪个来源。
- 结论和前面调研材料脱节——结论是基于"印象"而不是基于系统整理。

这不是因为模型不会写字，而是任务本身有结构：

```text
确定竞品 → 定义比较维度 → 搜集资料 → 整理事实 → 横向比较 → 形成建议
```

每一步都依赖前一步。如果比较维度没定义清楚就搜集资料，搜回来的东西就没法横向对比；如果事实没整理完就写结论，结论就不够扎实。

### 6.2 没有计划，Agent 容易局部最优和中途漂移

ReAct 擅长动态探索——它让 Agent 可以根据当前观察调整下一步。但如果任务很长、依赖关系复杂，单纯依赖"下一步判断"会有几种典型的失败模式：

#### 局部最优

Agent 看到一个容易处理的信息，就围绕它展开，忽略全局目标。比如只找到一个竞品的详细资料，就开始过度分析这个竞品，忘了还要看另外两个。

#### 遗漏约束

长任务通常有多个要求。模型在后续步骤中可能忘记某些约束。比如用户要求"定位、功能、价格、增长策略"四个维度，最后只写了功能和价格——因为这两个维度的信息最容易获取。

#### 步骤顺序错误

有些任务必须先定义标准，再搜集资料；先理解影响范围，再动手修改。如果顺序反了，前面做的工作可能白费。

#### 难以验收

没有计划，就很难判断任务做到哪一步、还差什么、什么时候算完成。Agent 可能一直做下去，也可能在不完整的状态下停住。

Planning 的核心价值就是解决这些问题：

```text
先形成任务结构，再推进执行。
```

这个"先"字就是全部关键。计划的成本远低于执行的成本——修改大纲花 10 分钟，修改已经写好的草稿可能要花几小时。在 Agent 场景下也一样：让模型先花一些 Token 规划结构，可以避免后续大量的无效执行和返工。

### 6.3 从 CoT、ToT 到 Plan-and-Execute

Planning 的演进大致可以看成三步：先让模型写出中间推理，再让模型比较多条推理路径，最后把计划和执行拆成运行时结构。每一步都在解决前一步留下的问题。

#### Chain of Thought：不要跳步

2022 年 1 月，Google Research 发表 Chain-of-Thought Prompting。它的核心发现出奇朴素：

```text
在复杂数学、常识和符号推理任务中，如果让模型先写出中间步骤，再给出最终答案，表现会显著变好。
```

为什么？因为 LLM 的本质是一个"下一个 Token 预测器"，它天然是逐字生成的，而不是整体规划的。直接让它跳到答案，等于没有给它任何"思考空间"。CoT 改变了模型生成答案时的上下文窗口——前面的推理步骤为后面的结论提供了"脚手架"。

后来 Zero-shot CoT 进一步发现，有时候甚至不需要给模型展示解题样例，只需要在 Prompt 末尾加上一句"Let's think step by step"，就能唤起这种逐步推理能力。这说明"分步思考"是模型在预训练过程中从海量文本中学到的行为模式，CoT 只是在"唤醒"它。

**局限**：CoT 让模型"一步步思考"，但每一步往哪个方向走，模型没有太多选择——它基本上是按"最可能的路径"一直走。如果第一步就走偏了，后面的步骤只会让错误越来越大。这就像一个学生列了一长串计算步骤，但第一步就算错了，后面全白费。

#### Tree of Thoughts：不只一条路

2023 年 5 月，Shunyu Yao（同时也是 ReAct 的作者之一）和 Princeton 团队提出 Tree of Thoughts。ToT 把思考从"一条链"扩展为"多条候选路径"。

工作机制：
1. 在每个推理节点，生成多个可能的"下一步"，而不是只选最可能的一个。
2. 对每个候选步骤进行评估，判断它是否有希望通向正确答案。
3. 选择最有希望的路径继续探索——可以用广度优先（BFS）或深度优先（DFS）。
4. 如果发现某条路径走不通，可以回溯，换一条路。

ToT 和 CoT 的差别，可以用"写草稿"和"搜索解法"来理解：CoT 沿着一条推理线往前走，像是一气呵成写完；ToT 会生成多个候选下一步，评估哪条路更有希望，再继续展开，像是边写边改、多版本对比。

这对开放任务很重要。比如写市场分析报告，可以从用户群、商业模式、产品功能、渠道策略等不同路径进入——每条路看到的东西不同，比较之后才能找到最好的切入角度。

**代价**：ToT 需要多次调用 LLM（每一个候选步骤都是一次调用），Token 消耗是 CoT 的 10 倍以上。在实际应用中，是否值得为这种"多路径探索"付出成本，需要根据任务难度来判断。

#### Plan-and-Execute：把计划和执行拆开

到了 Agent 工程里，Planning 不再只是提示词技巧，而会变成运行时结构。很多系统会引入一个关键分工：

```text
Planner: 先生成任务计划。
Executor: 按计划执行每一步，并把结果返回。
```

例如对于竞品调研任务：

```text
Plan:
1. 确定 3 个主要竞品。
2. 为每个竞品收集官网、价格页和新闻资料。
3. 按定位、功能、价格、增长策略四个维度整理表格。
4. 横向对比，标出差异和差距。
5. 给出针对性建议。
```

执行时，Planner 负责生成这个结构，Executor 逐步完成每一步，并在每一步后记录状态。如果某一步执行失败（比如某个竞品的信息太少），Planner 可以收到反馈后调整计划。

这种分工的好处是：**规划和执行有了不同的关注点。** Planner 思考的是全局结构和依赖关系；Executor 关注的是当前这一步怎么做好。两者不需要在同一个上下文里争抢注意力。

### 6.4 复杂任务有了结构化推进方式

Planning 带来的直接效果是让长任务变得可管理：

- 任务目标更清晰——每一步做什么、做到什么程度算完成。
- 步骤依赖更明确——知道这一步完成了才能开始下一步。
- 中间产物更容易检查——计划本身就是一个检查清单。
- 用户更容易知道 Agent 在做什么——计划提供了透明度。
- 系统更容易判断任务是否完成——对照计划逐项核对。

Planning 特别适合以下几类场景：

| 场景 | 为什么需要 Planning |
|---|---|
| 调研报告 | 需要统一维度和资料来源，否则横向比较无从谈起 |
| 代码重构 | 需要先理解影响范围，再分步修改，否则改了 A 坏了 B |
| 数据分析 | 需要先定义指标，再处理数据，否则分析没有聚焦点 |
| 项目迁移 | 需要顺序、回滚和验证，否则出错了不知道回到哪 |
| 多系统操作 | 需要明确依赖和风险点，否则一个系统失败影响全局 |

### 6.5 计划需要被执行、检查和重规划

Planning 也会带来新问题，而且这些问题在工程中非常真实：

#### 计划可能一开始就是错的

模型生成的计划看起来结构完整，但可能遗漏关键步骤。比如竞品分析计划可能忘了"定义目标用户群"这个前提。

#### 计划可能执行不下去

执行过程中工具失败、资料缺失、权限不足。原计划无法继续时，系统需要有重新规划的机制，而不是卡住。

#### 计划可能过度复杂

简单任务如果强行走 Planning，就像用大炮打蚊子——成本高、速度慢、效果未必更好。判断一个任务是否需要 Planning，是 Agent 工程中的重要决策。

#### 计划和执行可能脱节

Planner 写了一个漂亮计划，Executor 实际执行时却偏离了方向。这种脱节很常见，因为规划时的假设在执行时可能已经被推翻。

所以未来更可靠的 Planning 不是"一次性写完计划然后照做"，而是：

```text
计划 → 执行 → 检查 → 重规划 → 继续执行
```

这也解释了为什么后续课程会涉及 Chain、Router、ReAct Loop、Plan-Execute、Graph 等不同模式。不同任务需要不同的组织方式，没有一种模式适合所有场景。

Planning 让单个 Agent 能驾驭更复杂的任务结构。但它没有解决另一个维度的瓶颈：**当一个 Agent 被塞进太多不同质的角色——既是产品经理又是安全工程师，既要写代码又要写文档——上下文、工具和评价标准就会在内部互相冲突。** 这不是"计划不够好"的问题，而是"什么都管必然什么都管不好"的问题。这就引出了 Multi-Agent。

## 第七章：Multi-Agent —— 当什么都管，什么都管不好

上一章 Planning 解决的是"任务太复杂，一步一判会漂移"。这一章 Multi-Agent 解决的是同类问题的另一个维度：**当一个 Agent 被塞进太多不同质的角色，上下文、工具和目标就会在内部互相冲突。** 不是 Agent 不够强，而是"什么都管"本身就会导致"什么都管不好"——这个洞察来自软件工程最基础的原则：关注点分离。

当任务变复杂，一个 Agent 同时承担所有职责，会很快撞上几堵墙。Multi-Agent 的出现，不仅是技术选择，更是一种**组织哲学**——它把软件工程中"分工协作"的思想带进了 Agent 系统。

### 7.1 一个 Agent 很快会变成"什么都管"的巨型系统

假设你想让一个 Agent 帮你完成产品发布：

- 分析需求。
- 设计方案。
- 写代码。
- 写测试。
- 做安全检查。
- 写文档。
- 准备发布说明。
- 监控上线结果。

如果全部交给一个 Agent，它需要同时扮演：产品经理、架构师、前端工程师、后端工程师、测试工程师、安全工程师、技术写作者、运维工程师。

这会带来几个硬问题：

- **上下文太大**：把所有角色的信息塞进同一个上下文窗口，模型会开始"遗忘"。Agent 的上下文窗口不是无限的，每多一个角色就多一份专属信息。
- **工具太多**：产品经理需要的工具和运维工程师需要的工具完全不同。100 个工具堆在一起，模型选择工具的准确率会明显下降。
- **角色目标冲突**：产品经理追求用户价值，安全工程师追求风险最小化，这两个目标天然有张力。同一个模型在同一个上下文里处理这些冲突，很容易产生混乱。
- **难以并行**：单 Agent 一次只能做一件事。需求和代码可以并行推进，但一个 Agent 只能一件一件来。
- **输出标准不一致**：写代码要精准，写文档要易懂，写安全审计要面面俱到——一个模型在不同标准间切换，质量很难稳定。

这些问题不是"模型不够强"导致的，而是**"一个执行单元承担了太多不同质的职责"**导致的。这个洞察来自软件工程的基本原则——关注点分离。

### 7.2 复杂任务需要角色、上下文和工具边界

人类团队不会让一个人同时负责所有事情。不是因为一个人绝对做不到，而是因为分工可以降低每个角色的复杂度，让每个人在自己的领域内做到最好。

Agent 系统也适用同样的逻辑。

一个代码实现 Agent 只需要关注：当前文件怎么改、测试怎么跑、类型错误怎么修。它的上下文是紧凑的、工具是聚焦的。

一个代码审查 Agent 需要关注：有没有 bug、有没有安全风险、有没有缺测试、有没有架构退化。它的上下文是代码 diff，工具是静态分析和安全扫描。

一个文档 Agent 需要关注：用户能不能看懂、概念是否准确、示例是否完整。它不需要知道代码的内部实现细节，只需要知道怎么把功能解释清楚。

这些角色需要不同的上下文、不同的工具、不同的评价标准。把它们的职责分开，不是增加了复杂度，而是**通过边界降低了复杂度**。

Multi-Agent 的核心问题是：

```text
如何把一个复杂任务拆给多个有边界的 Agent 协作？
```

### 7.3 角色分工、管理者-执行者和专家协作

2023 年以来，Multi-Agent 的探索产生了多种形态，背后有共同的设计思路：**不要让一个模型在同一个上下文里同时扮演所有角色。** 就像真实团队不会让同一个人同时做产品、架构、开发、测试、审查和发布，Agent 系统也需要把角色边界显式化。

Stanford 和 Google Research 的 **Generative Agents**（2023 年 4 月）展示了多个具备记忆、反思和计划能力的模拟角色如何形成社会行为。微软研究院的 **AutoGen**（2023 年 8 月）把多个 LLM Agent 之间的对话协作框架化，核心设计是通过多 Agent 对话来协作完成任务。**CrewAI**（2024 年初）则进一步把"角色、任务、工具、流程"包装成更接近团队协作的开发体验。

几种常见形态如下：

#### 角色分工

给不同 Agent 定义不同角色，每个角色有自己的目标、上下文和工具集：

- Research Agent：收集资料，聚焦搜索和分析。
- Planning Agent：制定计划，聚焦任务拆解和依赖关系。
- Coding Agent：执行代码修改，聚焦实现细节。
- Review Agent：检查风险，聚焦问题和边界情况。
- Writer Agent：整理文档，聚焦可读性和准确性。

每个 Agent 的目标更窄，上下文更聚焦，工具更精准。不是"一个全知全能的 Agent 做所有事"，而是"多个专注的 Agent 各司其职"。

#### 管理者-执行者

一个 Manager Agent 负责拆解任务和分配工作，多个 Worker Agent 负责执行具体子任务：

```text
Manager: 这个需求可以拆成 API、前端、测试、文档四块。
Worker A: 修改 API 实现。
Worker B: 修改前端代码。
Worker C: 补充测试。
Worker D: 撰写文档。
Manager: 汇总结果并检查一致性。
```

这个模式很像软件团队的项目经理 + 工程师结构。Manager 不需要精通每个子任务的细节，但需要知道任务如何拆解、结果如何整合。

#### 专家协作

不同 Agent 从不同角度独立评估同一方案，然后汇集意见：

- 性能专家看性能风险。
- 安全专家看权限和数据风险。
- 产品专家看用户价值。
- 工程专家看实现复杂度。

这适合决策质量要求高的任务。多个视角的碰撞更容易发现单一视角的盲点。

### 7.4 能力可以扩展，任务可以并行

Multi-Agent 的好处不只是"人多力量大"，而是系统性的：

- **角色更清晰**：每个 Agent 知道自己的职责范围，不会越界也不会遗漏。
- **上下文更可控**：每个 Agent 只需要加载与己相关的信息，不会因为上下文过长而遗忘关键信息。
- **工具权限更容易隔离**：安全敏感的 Agent 拥有不同的权限级别，修改代码的 Agent 和发送邮件的 Agent 不应该有相同的权限。
- **多个子任务可以并行**：API 修改和前端修改可以同时进行，不需要等一个完成再开始另一个。
- **复杂任务可以分阶段验收**：每个角色的输出可以独立检查，而不是等到最后才发现问题。
- **审查和执行可以分离**：写代码的人和审查代码的人不是同一个 Agent，避免了"自己审自己"的盲区。

例如一个产品发布流程可以变成流水线：

```text
需求分析 Agent → 方案设计 Agent → 实现 Agent → Review Agent → 文档 Agent → 发布检查 Agent
```

每个阶段有明确的输入输出，出了问题时能清楚追溯到哪个环节。

### 7.5 协作本身也会制造复杂度

Multi-Agent 并不是万能解法。当 Agent 数量增加时，新的问题随之而来：

| 问题 | 表现 |
|---|---|
| 协调成本 | Agent 之间需要传递状态和结果。N 个 Agent 之间的通信链路是 O(N^2) 级别的，如果不加设计，会产生大量"闲聊" |
| 责任不清 | 最终输出出错时，不知道是哪个 Agent 的责任——是 Planner 拆错了，还是 Worker 执行错了，还是 Reviewer 漏审了？ |
| 信息丢失 | 子任务结果被汇总时丢掉关键细节——一份 50 页的分析被压缩成 3 句话传给下一个 Agent |
| 冲突处理 | 不同 Agent 给出相反建议——安全专家说不要做，产品专家说必须做，谁说了算？ |
| 成本上升 | 多个 Agent 各自调用模型和工具，总 Token 消耗可能远超单 Agent |
| 调试困难 | Trace 分散在多个执行单元里，排查问题时需要在不同 Agent 的日志之间跳转 |

所以未来更实用的 Multi-Agent 方向，不是"越多 Agent 越好"，而是：

```text
角色清晰、边界明确、通信可追踪、结果可验证。
```

很多生产系统会把 Multi-Agent 和 Graph、Workflow、Human Review 结合起来，而不是让一群 Agent 自由聊天。Agent 之间的协作需要被精心的工程约束，就像人类团队需要明确的职责划分和沟通规范一样。

## 第八章：范式演进总览

本课走完了截至 2024 年底前 Agent 范式演进的主干脉络。现在我们把所有线索收束在一起。

### 8.1 一张表看主干演进线

| 阶段 | 代表节点 | 问题代入 | 解决方案 | 产生的效果 | 新问题和未来方向 |
|---|---|---|---|---|---|
| 传统 Agent | Bellman、Q-learning、BDI、Russell & Norvig | 系统需要感知、决策和行动，但决策核心是什么？ | RL 试错学习、软件 Agent 规则、BDI 心智模型、经典 Agent 分类 | 封闭环境里可控执行，奠定了 Agent 的基本问题框架 | 开放语言任务建模成本高，无法低成本理解自然语言目标 |
| LLM 决策核心 | GPT-3（2020）、Chain-of-Thought（2022.1） | 用户用自然语言表达开放目标，传统系统只认识规则 | 让 LLM 理解目标和判断下一步，从少样本学习到涌现能力 | 从规则驱动走向目标驱动，Agent 获得通用决策核心 | 会判断不等于能可靠执行，模型缺乏外部信息获取和行动能力 |
| 工具调用 | Toolformer（2023.2）、ChatGPT Plugins（2023.3）、Function Calling（2023.6）、MCP（2024） | LLM 会说但不会做——用户要的不是建议，是执行 | 工具学习、插件体验、结构化函数调用、跨模型工具协议 | 模型连接外部信息和真实动作，实现了"模型决策、环境执行"的分工 | 工具选择、参数填充、权限、安全、Trace、工具治理 |
| ReAct | ReAct（2022.10，Google Research + Princeton） | 推理和行动被错误分开，一次性回答无法处理动态任务 | Thought → Action → Observation 循环，让推理和行动互相喂给对方 | Agent 有了最小运行模式，从"回答机器"变成"问题解决器" | 循环需要停止条件、循环控制、上下文管理 |
| 纠错与反思 | Self-Refine（2023.3）、Reflexion（2023.3） | 模型更会推理 ≠ 系统更可靠；多步错误会沿依赖链传播放大 | 建立"内部推理 vs 外部反馈"两层能力框架；Self-Refine 的"作者+编辑"迭代、Reflexion 的内置反思循环与情景记忆 | 系统从"信任模型"走向"验证模型"，从"一次生成"走向"可迭代改进"，具备任务内学习能力 | 不能只靠模型自评，需要多信号交叉验证；过度反思导致瘫痪、成本上升、目标漂移 |
| Planning | CoT（2022.1）、ToT（2023.5）、Plan-and-Execute | 长任务有依赖关系，没有结构容易漂移和遗漏 | 从单链推理到多路径搜索，再到 Planner-Executor 分离 | 复杂任务有了结构化推进方式，步骤依赖明确、中间产物可检查 | 计划可能一开始就错、执行可能脱节、需要重规划机制 |
| Multi-Agent | Generative Agents（2023.4）、AutoGen（2023.8）、CrewAI（2024） | 单 Agent 上下文过载、角色冲突、无法并行 | 角色分工、管理者-执行者、专家协作——通过边界降低复杂度 | 能力扩展和并行执行，关注点分离让每个 Agent 更聚焦 | 协调成本 O(N^2)、责任不清、信息丢失、共识达成 |

这张表不是 Agent 生态的终局图。2025 年之后，行业重心继续向三个方向推进：一是 reasoning models 把部分规划、检查和长链推理能力进一步内化到模型层；二是 coding agent、deep research、computer-use agent 等产品把工具、权限、状态、恢复和评测推向生产级运行时；三是 MCP、A2A、框架编排和评测平台继续尝试降低 Agent 与外部系统协作的集成成本。理解这些新方向时，仍然可以回到本课这张表：它们到底解决哪个旧问题，又把复杂度转移到了哪一层？

### 8.2 如何分析一个新的 Agent 技术

以后你看到一个新的 Agent 框架、论文或产品功能，可以用这组问题判断它的位置和价值：

#### 第一步：它解决什么旧问题？

不要先问"它是不是先进"，先问：
- 它解决工具调用问题吗？（模型能接触真实世界吗？）
- 它解决长任务规划问题吗？（模型知道从哪开始、到哪结束吗？）
- 它解决状态管理问题吗？（模型记得之前做过什么吗？）
- 它解决错误反馈问题吗？（系统能发现并修正错误吗？）
- 它解决多 Agent 协作问题吗？（角色边界和通信效率解决了吗？）

#### 第二步：它把能力放在哪一层？

Agent 系统至少有这几层，不同技术解决的是不同层的问题：
- **模型层**：模型本身更会推理或更会调用工具（如 CoT、ToT 影响的是模型推理模式）。
- **接口层**：模型如何表达工具调用和结构化输出（如 Function Calling、MCP 影响的是接口标准）。
- **Runtime 层**：系统如何执行、记录、控制、恢复（如 ReAct Loop、Plan-and-Execute、Reflexion 影响的是运行结构）。
- **产品层**：用户如何感知、确认和信任 Agent（如 Plugins 影响的是用户认知）。
- **组织层**：多个 Agent 或人类如何协作（如 AutoGen、CrewAI 影响的是协作模式）。

#### 第三步：它带来什么新问题？

任何能力增强都会带来新问题。**能看见新问题，才说明你真正理解了旧方案。**
- 工具越多，选择越难。
- 计划越复杂，执行越容易脱节。
- 反思越多，越可能陷入自我安慰或无限修正。
- Agent 越多，协调越困难。
- 自动化越强，安全边界越重要。

### 8.3 常见误区

这八个误区是本课反复强调的，列在这里作为"避坑指南"：

#### 误区一：把 Agent 当成更强的 Chatbot

Chatbot 的核心是回答。Agent 的核心是围绕目标持续决策和执行。两者的根本差异不是"能力强弱"，而是"系统的运行模式不同"。

#### 误区二：以为加了工具就是 Agent

工具调用只是 Agent 的一部分。没有状态管理、循环控制和反馈处理，工具调用很容易变成一次性 API 包装——看起来像 Agent，但没有 Agent 的核心能力。

#### 误区三：以为 Planning 越复杂越好

简单任务不需要复杂计划。过度 Planning 会增加成本和失败点。判断一个任务是否需要 Planning，是 Agent 工程中的重要决策。

#### 误区四：以为纠错就是让模型自我批评

真正有价值的纠错与反思应该尽量依赖外部真实信号——测试结果、编译结果、工具返回、用户确认。纯粹的模型自评容易变成自我确认。记住两层能力框架：模型内部推理是必要的但不是充分的，系统外部反馈才是可靠性的保障。

#### 误区五：以为 Multi-Agent 一定比 Single-Agent 强

Multi-Agent 只有在角色边界清晰、通信可追踪、结果可验证时才有价值。如果协作成本超过了分工收益，单 Agent 反而是更好的选择。

#### 误区六：以为 ReAct 只是一种 Prompt 写法

ReAct 的意义远不止"在提示词里加 Thought/Action/Observation"。它是一种运行范式——让推理和行动在同一循环中交织，让模型在与外部世界的交互中持续推理。很多不显式输出 Thought 的 Agent 系统，内在逻辑仍然是 ReAct。

#### 误区七：以为模型越大，Agent 越可靠

模型推理能力的提升有助于决策质量，但 Agent 的可靠性更多取决于系统设计——是否有外部反馈、是否有状态管理、是否有循环控制。一个用较弱模型但 Runtime 设计良好的 Agent，可能比一个用最强模型但缺乏反馈机制的 Agent 更可靠。

#### 误区八：以为 Agent 演进已经完成

Agent 的范式演进远未结束。工具调用还在解决"工具如何被发现和治理"的问题，Planning 在解决"计划如何在执行中动态调整"的问题，纠错与反思在解决"如何不依赖模型自评来验证正确性"的问题，Multi-Agent 在解决"协调成本和责任边界"的问题。理解这些问题的发展脉络，比记住当前的最优实践更重要。
