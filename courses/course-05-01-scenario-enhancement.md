# 课程五：场景增强能力

## 课程导言

课程三和课程四之后，你已经有了一个能跑的 Agent——LLM 决策 + 工具调用 + State 管理 + 循环控制。它处理简单任务很稳：读文件、做摘要、搜索替换，都没问题。

但一旦把它放到真实场景里用上几周，你就会撞上五类依次暴露的问题：知识不在模型里、关了会话就失忆、长任务开始漂移、看到错误却不修正、自己审自己说没问题。

这五类问题分别对应五类增强能力：RAG（外部知识接入）、Memory（状态延续）、Planning（任务组织）、Reflection（失败修正）、Multi-Agent（角色分工与协作）。它们不是 Agent 的"必选核心模块"——一个最小闭环不需要它们也能跑。它们是**按场景问题选择性引入的增强能力**。

本课不从概念名词开始，而从五个真实的问题案例出发。每一章都沿着同一条路径展开：

```text
问题代入（怎么出问题的）→ 技术背景（这个能力怎么来的）→ 思考（设计者当时在想什么）
→ 方案（核心链路 + 代码骨架）→ 迭代路径（从最小版本到生产级）→ 判断边界（什么时候别用）
```

学完后，你不需要把五类能力全部做深，但你面对一个新场景时应该能判断：

- 当前场景到底遇到了什么问题？
- 这个问题是否真的需要某个增强能力？
- 引入后会增加哪些工程复杂度？
- 如果要引入，应该从哪个最小版本开始？
- 什么情况下应该明确不引入？

---

## 学习目标

学完本课，你将能够：

1. **从场景问题出发判断增强能力**：不再把 RAG、Memory、Planning、Reflection、Multi-Agent 当成固定模块清单，而是根据实际问题模式选择对应的增强方向
2. **为每类能力做场景判断**：知道什么时候需要 RAG、Memory 该记什么不该记什么、复杂任务选 Chain/Router/ReAct/Plan-Execute/Graph 中的哪种、Reflection 必须有外部反馈信号和停止条件、Multi-Agent 只在角色冲突或需并行时才引入
3. **从最小版本开始迭代**：每种能力都有从 V0 到生产级的路径，不要一次性堆满架构
4. **写出能力引入说明**：用"问题、方案、代价、迭代、边界"解释为什么引入或不引入某个能力

## 第一章：为什么要从场景问题学习增强能力

### 1.1 最小 Agent 为什么开始不够用

你花了两周时间做了一个个人知识助手 Agent。它跑在课程三的最小闭环上——LLM 决策 + 三个工具（读文件、写文件、搜索）+ State 管理 + 循环控制。你日常在 Notion 上写技术笔记，半年积累了 200 多篇，涵盖 Agent 架构、工具设计、Memory 机制等话题。

第一个月，你用它很顺手：

```text
你："帮我写一段 Agent Tool Use 的介绍。"
Agent：基于训练数据中的知识 → 输出一段结构清晰的介绍。够用了。
```

但第二个月开始，问题一个接一个冒出来。

**第一个问题：你的笔记在 Notion 上，它不知道里面写了什么。**

你问："根据我的笔记，Tool Use 和 Memory 的设计哲学有什么根本不同？"

Agent 没有访问 Notion 的能力。它只能靠训练数据里学到的通用知识来回答。答案看起来头头是道，但那是它"猜"的——你的笔记里有一段关键论述，它根本没机会看到。

这个问题的本质是：**正确答案不在模型参数里，在你的笔记里。Agent 需要一套机制，在生成回答之前先去外部知识源查资料，然后基于查到的资料来组织回答，而不是凭参数记忆硬猜。** 这引出了第二章要讲的 RAG / 外部知识接入。

**第二个问题：它永远不记得我上回说过什么。**

有一天你花了 20 分钟告诉 Agent："以后写技术文章，先给大纲我确认，再展开正文。语气直接，不要营销化。"

第二天你打开新会话："帮我写一篇 Agent Memory 的技术文章。"

它给了你一篇充满"在当今这个 AI 时代……"开头的营销风文章，没有先给你大纲。

你意识到：**Agent 的状态在会话关闭后就蒸发了。** 用户的偏好、项目的约定、上回任务的进度——所有这些都需要某种"跨会话延续"的机制。这引出了第三章要讲的 Memory。

**第三个问题：任务一长，它就开始漂。**

你说："帮我做发布准备：检查 README 完整性、跑一遍测试、整理 changelog、生成 release checklist。"

Agent 开始执行。它读了 README，写了一段评论。然后……它发现 README 里提到的一个技术细节，开始去翻源码核实。接着又去搜了相关的 GitHub issue。你等了五分钟，发现它偏离原始目标，已经不知道在做什么了。你叫停它，检查 trace——它在第 4 步之后就没再碰过和"发布"相关的任何操作。

这个问题的本质不是工具不好用，而是**裸 ReAct 循环没有"任务结构"的概念**。它每步都在临场反应，但没有全局视图。这引出了第四章要讲的 Planning / Workflow Patterns。

**第四个问题：错了，但它继续往前走。**

你让 Agent 改一段代码，改完后跑测试。测试输出：

```text
TypeError: Cannot read properties of undefined (reading 'files')
```

Agent 看到了这个输出，然后说：

```text
修改已完成，代码可以正常运行。
```

你盯着屏幕，不知道该笑还是该气。它明明看到了错误信息，但它没有把"测试失败"当作一个需要修正的信号。它只是在"生成下一个 Token"。

这个问题的本质是：**Agent 缺少"失败→分析→修正→重试"的闭环机制。** 这引出了第五章要讲的 Reflection。

**第五个问题：你让它审查自己的工作，它说"没问题"。**

你让 Agent 写一份技术方案，然后从安全角度审一遍。它写完了，你让它审查。它输出了：

```text
经过审查，该方案在安全方面没有明显问题。
```

你作为人类看了一眼，立刻发现了三个安全隐患。Agent 不是不聪明——它只是在"写方案"和"审方案"时用了同一个视角、同一个上下文。自己审自己，就像让学生给自己的试卷打分。

这个问题的本质是：**单一 Agent 的角色开始过载，不同视角需要不同的上下文和判断标准。创作者和审查者的目标天然矛盾，共享同一个上下文意味着互相妥协。** 这引出了第六章要讲的 Multi-Agent。

---

这五个问题有一个共同特点：**都不是"最小闭环公式错了"，而是场景复杂度升高后，最小闭环需要被增强。**

课程三和课程四给了你一个能跑的骨架。课程五要解决的问题是：当这个骨架在真实场景中开始发出咯吱声时，你应该往哪个方向加固。

### 1.2 能力是对问题的响应

上面五个问题，分别击中了最小 Agent 的五块短板。每一项增强能力都不是”技术圈的新玩具”，而是对具体问题的直接回应：

| 问题场景 | 根因 | 增强能力 | 引入后新增复杂度 | 学习重点 |
|---|---|---|---|---|
| 笔记在外部系统，Agent 访问不到 | 模型参数里没有答案，外部知识进不来 | RAG / 外部知识接入 | 数据质量、检索质量、引用可信度、权限管控 | Chunking、Embedding、混合检索、Rerank |
| 关了会话就失忆，偏好反复说 | 状态随会话蒸发，无法跨轮延续 | Memory | 写入策略、隐私、遗忘机制、记忆污染 | 短期/长期记忆、用户偏好、记忆更新 |
| 多步骤任务偏离目标，漏步骤 | 裸 ReAct 没有全局任务结构 | Planning / Workflow Patterns | 计划可执行性、重规划、调试成本 | Chain、Router、ReAct Loop、Plan-Execute、Graph |
| 看到错误却不修正，继续往下走 | 缺少”失败→分析→修正→重试”闭环 | Reflection | 成本增加、延迟上升、错误反思、需停止条件 | 外部验证信号、触发条件、停止策略 |
| 自己审自己说”没问题” | 单 Agent 角色过载，创作者和审查者共享上下文导致互相妥协 | Multi-Agent | 通信成本、协调复杂度、错误放大、难调试、token 消耗 2-5 倍 | 独立上下文、结构化通信、裁决机制、成本边界 |

这张表是你在后续章节中的定位坐标——不需要全部学完再动手，遇到哪个问题就翻到对应章节。增强能力的引入顺序，应该由**当前项目实际遇到的问题**驱动，而不是由技术热度驱动。

### 1.3 能力引入的通用判断流程

每次想引入一个增强能力前，先问五个问题：

1. **当前问题是什么？**
   是知识不够、状态断裂、任务组织混乱、缺少纠错，还是角色过载？

2. **有没有更简单的解决方式？**
   例如用户直接提供上下文、用固定流程、增加工具校验、优化提示词，是否已经足够？

3. **引入该能力会增加什么复杂度？**
   例如 RAG 会引入索引和引用问题，Memory 会引入隐私和污染问题，Multi-Agent 会引入通信和调试问题。

4. **如何做最小版本？**
   先做一个可评测的最小增强，而不是一开始上完整架构。

5. **如何判断它真的变好了？**
   必须有任务集、对比指标或人工验收标准。

### 1.4 全局架构：每个增强能力在 Agent 应用中的位置

在进入各章细节之前，先用一张图看清楚五类增强能力在整个 Agent 应用中分别处于什么位置：

```
                          ┌─────────────┐
                          │   用户/任务  │
                          └──────┬──────┘
                                 │
     ┌───────────────────────────┼───────────────────────────┐
     │                           │                           │
     ▼                           ▼                           ▼
┌──────────┐              ┌──────────┐                ┌──────────┐
│   RAG    │              │  Memory  │                │ Planning │
│  (Ch2)   │              │  (Ch3)   │                │  (Ch4)   │
│          │              │          │                │          │
│ 检索知识  │              │ 记住偏好   │                │ 拆解任务  │
│ 注入上下文 │              │ 延续状态  │                │ 选择模式  │
│ 标注引用  │              │ 跨会话    │                │ 控制流程   │
└────┬─────┘              └────┬─────┘                └────┬─────┘
     │                         │                           │
     │ 外部资料+引用             │ 偏好/历史/状态              │ 任务计划
     │                         │                           │
     └─────────────────────────┼───────────────────────────┘
                               │
                     ┌─────────▼─────────┐
                     │  Agent Runtime    │
                     │  ┌─────────────┐  │
                     │  │ LLM ⇄ Tool  │  │
                     │  │ State ⇄ Loop│  │
                     │  └─────────────┘  │
                     └────────┬──────────┘
                              │
          ┌───────────────────┼───────────────────┐
          │                   │                   │
          ▼                   ▼                   ▼
   ┌──────────────┐   ┌──────────────┐   ┌──────────────┐
   │  Reflection  │   │ Multi-Agent  │   │              │
   │    (Ch5)     │   │    (Ch6)     │   │     ...      │
   │              │   │              │   │              │
   │ 失败→分析     │   │ Agent A(执行) │   │  更多增强能力  │
   │ 修正→重试     │   │ Agent B(审查) │   │              │
   │ 停止条件      │   │ 协调者(汇总)   │   │              │
   └──────────────┘   └──────────────┘   └──────────────┘
```

每一层的作用和它与 Runtime 的关系：

| 位置 | 能力 | 在架构中的角色 | 与 Runtime 的连接方式 |
|---|---|---|---|
| **上方**（执行前） | RAG (Ch2) | 知识接入层：从外部知识库检索相关资料 | 作为上下文的一部分注入 LLM 调用 |
| **上方**（执行前） | Memory (Ch3) | 状态延续层：跨会话记住用户偏好、任务历史和项目约定 | 在 Runtime 启动时召回相关记忆注入上下文 |
| **上方**（执行前） | Planning (Ch4) | 任务组织层：把用户意图拆成可执行的步骤，选择执行模式 | 将计划作为结构化输入注入 Runtime |
| **下方**（执行后/并行） | Reflection (Ch5) | 纠错反馈层：检测失败信号，触发分析和修正 | 拦截 Runtime 输出，外部验证后决定重试或停止 |
| **下方**（执行后/并行） | Multi-Agent (Ch6) | 规模扩展层：将单 Agent 扩展为多角色协作 | 启动多个 Runtime 实例，通过消息或共享状态协调 |

关键理解：这五类能力**不是 Runtime 本身的模块**，而是围绕 Runtime 的增强层。最小闭环可以独立运行，增强层按需接入——这正是"从场景问题出发"的核心含义。

## 课后练习

### 练习一：从问题案例判断能力

阅读下面五个问题案例，判断应该优先引入什么能力，并说明为什么：

| 问题案例 | 你的判断 |
|---|---|
| Agent 回答公司政策时引用了旧政策 |  |
| 用户每次都要重复说明输出风格 |  |
| Agent 做发布准备时漏掉测试步骤 |  |
| Agent 生成 JSON 经常字段缺失 |  |
| Agent 写完方案后无法从安全视角审查 |  |

要求每个案例写出：

- 当前问题是什么。
- 优先能力是什么。
- 是否有更简单方案。
- 引入后的复杂度。
- 如何评测。

### 练习二：写一份“不引入某能力”的说明

选择一个项目，写一份不引入某能力的说明。

例如：

- 为什么个人知识助手第一版不引入 Multi-Agent。
- 为什么代码审查 Agent 第一版不引入长期 Memory。
- 为什么简单摘要工具不引入 Planning。

说明必须包含：

- 当前任务边界。
- 不引入的原因。
- 未来什么条件下会重新考虑。

### 练习三：设计外部知识接入方案

为个人知识助手设计一个外部知识接入方案：

- 数据来源。
- Chunking 策略。
- 索引方式。
- 查询改写方式。
- 引用格式。
- 无答案处理。
- 权限和隐私边界。
- 评测集。

### 练习四：设计 Memory 策略

为同一个项目设计 Memory 策略：

- 哪些内容只属于任务状态。
- 哪些内容属于会话摘要。
- 哪些内容可以长期保存。
- 哪些内容永不保存。
- 用户如何查看和删除记忆。
- 错误记忆如何纠正。

### 练习五：选择任务组织模式

为以下任务选择 Chain、Router、ReAct Loop、Plan-Execute 或 Graph：

- 把一篇文章改写成小红书风格。
- 判断用户问题应该走文档问答还是代码审查。
- 修复一个测试失败的代码问题。
- 完成一次项目发布准备。
- 处理一个包含多阶段审批的企业流程。

说明你选择的理由和不选择其他模式的原因。

### 交付物

完成本课后，你应该产出：

- 一份增强能力判断表。
- 一份 RAG / 外部知识接入最小方案。
- 一份 Memory 读写策略。
- 一份 Planning / Workflow Pattern 选择说明。
- 一份 Reflection 触发和停止规则。
- 一份 Multi-Agent 是否引入的判断说明。
- 至少一份"为什么不引入某能力"的技术判断。

## 验收标准

完成本课后，你应该能够：

- 不把 RAG、Memory、Planning、Reflection、Multi-Agent 当成固定核心模块。
- 根据具体问题判断优先增强能力。
- 说明 RAG / 外部知识接入和普通工具调用的关系。
- 区分任务状态、会话历史和长期 Memory。
- 为复杂任务选择 Chain、Router、ReAct、Plan-Execute 或 Graph。
- 说明 Reflection 为什么必须有触发信号和停止条件。
- 判断 Multi-Agent 的真实价值和复杂度风险。
- 写出一份能力引入或不引入的技术说明。
