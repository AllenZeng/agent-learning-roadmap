# 课程五：场景增强能力

## 课程导言

课程三和课程四之后，你已经有了一个能跑的 Agent——LLM 决策 + 工具调用 + State 管理 + 循环控制。它处理简单任务很稳：读文件、做摘要、搜索替换，都没问题。

但一旦把它放到真实场景里用上几周，你就会撞上两类不同性质的问题：

**第一类：上下文问题——Agent 决策所依赖的信息，从哪来、怎么管？** 私有知识不在模型参数里，外部数据进不来；用户偏好和任务状态跨一次会话就蒸发；等外部知识和历史状态都接入了，多个信息源同时往上下文里灌，上下文从精心设计的输入变成不断膨胀的信息堆。这类问题靠**上下文增强**来解决：RAG（获取外部数据）、Memory（延续历史状态）、Context Engineering（组织多源信息）。

**第二类：行为问题——Agent 面对复杂任务时，如何组织执行、协作、恢复？** 裸 ReAct 循环没有任务结构，多步骤任务容易漂移；执行过程出错后缺乏检测和恢复机制；高风险操作缺少决策权限边界；单一角色无法多视角审视自己的工作。这类问题靠**行为模式增强**来解决：Planning（任务分解与编排）、Reflection（失败检测与恢复）、Human-in-the-loop（决策权限边界）、Multi-Agent（多角色分工协作）。

这两组能力回答的是两个根本问题。但它们有一个共同点：**都不是 Agent 的必选核心模块**——一个最小闭环不需要它们也能跑。它们是按场景问题选择性引入的增强能力。

本课不从概念名词开始，而从问题出发。每一章都沿着同一条路径展开：

```text
问题代入（怎么出问题的）→ 技术背景（这个能力怎么来的）→ 思考（设计者当时在想什么）
→ 方案（核心链路 + 代码骨架）→ 迭代路径（从最小版本到生产级）→ 判断边界（什么时候别用）
```

学完后，你不需要把七类能力全部做深，但你面对一个新场景时应该能判断：

- 当前场景是上下文问题还是行为问题？
- 这个问题是否真的需要某个增强能力？
- 引入后会增加哪些工程复杂度？
- 如果要引入，应该从哪个最小版本开始？
- 什么情况下应该明确不引入？

---

## 学习目标

学完本课，你将能够：

1. **从场景问题出发判断增强能力**：不再把 RAG、Memory、Context Engineering、Planning、Reflection、Human-in-the-loop、Multi-Agent 当成固定模块清单，而是根据实际问题模式选择对应的增强方向
2. **区分两类增强能力**：判断问题是属于"决策依赖的信息从哪来、怎么管"（上下文增强），还是"复杂任务的执行如何组织、协作、恢复"（行为模式增强）
3. **为每类能力做场景判断**：知道什么时候需要 RAG、Memory 该记什么不该记什么、上下文如何分层管理、复杂任务选哪种组织模式、Reflection 必须有外部反馈信号、什么操作必须人类确认、Multi-Agent 只在角色冲突或需并行时才引入
4. **从最小版本开始迭代**：每种能力都有从 V0 到生产级的路径，不要一次性堆满架构
5. **写出能力引入说明**：用"问题、方案、代价、迭代、边界"解释为什么引入或不引入某个能力

## 第一章：为什么要从场景问题学习增强能力

### 1.1 最小 Agent 的两类天花板

你花了两周时间做了一个个人知识助手 Agent。它跑在课程三的最小闭环上——LLM 决策 + 工具调用 + State 管理 + 循环控制。第一个月用得很顺手，但第二个月开始，问题一个接一个冒出来。

这些问题看似五花八门，但仔细看，它们击中的是两块不同的天花板。

**第一块天花板：上下文。Agent 决策所依赖的信息，从哪来、怎么管？**

最小闭环的"上下文"就是 System Prompt + 用户消息 + 工具返回。所有的信息都在当前会话里，会话结束就清零。这意味着三件事：

- 模型训练数据之外的私有知识、实时数据，Agent 拿不到。它只能凭参数记忆"猜"。
- 用户的偏好、项目的约定、上回任务的进度，跨一次会话就蒸发。每次都得从头来过。
- 等你把外部知识（RAG）和历史状态（Memory）都接入之后，新问题随之而来：多个信息源同时往上下文里灌，系统提示词、检索片段、Memory 召回、工具输出、历史消息混在一起——上下文从"精心设计的输入"变成了"不断膨胀的信息堆"。模型开始忽略关键约束，信息之间互相打架。

这三个层次——**获取外部数据、延续历史状态、组织多源信息**——共同决定了 Agent 决策的信息质量。对应的增强能力是 RAG（第二章）、Memory（第三章）、Context Engineering（第四章）。

**第二块天花板：行为。Agent 面对复杂任务时，如何组织执行、协作、恢复？**

最小闭环只有一个裸 ReAct 循环：每一步根据当前上下文决定下一步做什么。这个模式处理"查一下 X 是什么"没问题，但面对更复杂的任务形态时就不够了：

- 任务本身是多步骤、有依赖关系的——不是"下一步做什么"，而是"整体怎么拆、按什么顺序做、做完怎么验证"。
- 执行过程会出错——工具调用失败、返回结果异常、中间产出不符合预期。没有纠错机制的话，错误会被无声地携带到后续步骤中。
- 有些操作的影响不可逆——删除文件、发送消息、执行支付。Agent 在技术上有能力执行，但判断"该不该执行"需要人类的业务知识和风险意识。
- 有些任务需要多视角审视——创作者和审查者共用同一个上下文，自己审自己天然有盲区。

这四个层次——**任务分解与编排、失败检测与恢复、决策权限边界、多角色分工协作**——共同决定了 Agent 处理复杂任务的能力。对应的增强能力是 Planning（第五章）、Reflection（第六章）、Human-in-the-loop（第七章）、Multi-Agent（第八章）。

---

这两块天花板分别回答了 Agent 增强的两个根本问题：**决策依赖的信息从哪来、怎么管？复杂任务的执行如何组织、协作、恢复？** 课程三和课程四给了你一个能跑的骨架，课程五要解决的是当骨架遇到真实场景的复杂性时，往哪个方向加固。

### 1.2 两类增强：上下文 vs 行为模式

上面七个问题对应的七类能力，自然地分成两组。它们的差别不只是"管什么的"，而是回答的根本问题不同：

```text
┌─────────────────────────────────────────────────────────────┐
│                    上下文增强（信息维度）                       │
│  核心问题：Agent 的"视野"里有什么？                             │
│                                                             │
│  RAG          → 查什么：外部知识怎么进到上下文里                  │
│  Memory       → 记什么：跨会话状态怎么延续                      │
│  Context Eng  → 怎么管：多个信息源怎么组织成有效上下文            │
│                                                             │
│  三者关系：RAG 和 Memory 是信息生产者，Context Engineering      │
│  是信息组织者。没有组织者，生产者越多上下文越乱。                   │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                   行为模式增强（行动维度）                      │
│  核心问题：Agent 怎么"干活"？                                  │
│                                                             │
│  Planning     → 怎么拆：复杂任务怎么分解和组织                   │
│  Reflection   → 怎么改：执行失败了怎么检测和修正                 │
│  HITL         → 怎么停：什么时候不该自己决定                     │
│  Multi-Agent  → 怎么分：单一角色不够时怎么分工协作                │
│                                                             │
│  四者关系：Planning 规划行动、Reflection 修正行动、HITL          │
│  约束行动（边界）、Multi-Agent 扩展行动（规模）。                 │
└─────────────────────────────────────────────────────────────┘
```

为什么要做这个区分？两个原因：

**第一，引入顺序。** 真实项目中，你通常先解决信息问题，再解决行为问题。一个 Agent 连"视野"都不清晰，你给它再复杂的 Planning 模式也是白费——它在错误的信息上做正确的任务分解，结果还是错的。RAG 检索到的错误片段、Memory 召回的过时偏好、未经组织的混乱上下文——这些信息层面的问题，会被 Planning 和 Multi-Agent 的复杂编排逐层放大。

**第二，定位问题。** 当 Agent 表现不好时，先判断是信息问题还是行为问题。模型回答不准确——是检索结果不对（信息问题），还是任务拆解有误（行为问题）？Agent 删了不该删的文件——是没有 HITL 确认（行为问题），还是 Memory 里记了错误的文件分类（信息问题）？把问题归到正确的维度，才能找到正确的增强方向。

下表是后续章节的定位坐标——不需要全部学完再动手，遇到哪个问题就翻到对应章节：

| 维度 | 问题场景 | 根因 | 增强能力 | 引入后新增复杂度 |
|---|---|---|---|---|
| **上下文增强** | 外部知识进不来 | 模型参数里没有答案 | RAG (Ch2) | 索引维护、检索延迟、引用校验 |
| **上下文增强** | 关了会话就失忆 | 状态随会话蒸发 | Memory (Ch3) | 写入策略、隐私、记忆污染 |
| **上下文增强** | 信息源一多上下文变垃圾场 | 多信息源无序注入 | Context Engineering (Ch4) | 分层设计、预算调参、过度压缩 |
| **行为模式增强** | 多步骤任务漂移漏步骤 | 裸 ReAct 没有任务结构 | Planning (Ch5) | 计划可执行性、重规划成本 |
| **行为模式增强** | 看到错误却不修正 | 缺少纠错闭环 | Reflection (Ch6) | 多轮成本、需停止条件 |
| **行为模式增强** | 不该自己决定的也决定了 | 模型分不清"能做"和"该做" | Human-in-the-loop (Ch7) | 阻塞延迟、确认疲劳 |
| **行为模式增强** | 自己审自己说没问题 | 单 Agent 角色过载 | Multi-Agent (Ch8) | 协调开销、成本 2-5 倍 |

### 1.3 能力引入的通用判断流程

每次想引入一个增强能力前，先问五个问题：

1. **当前问题是什么？**
   是知识不够、状态断裂、上下文混乱、任务组织乱、缺少纠错、不该自主决定，还是角色过载？

2. **有没有更简单的解决方式？**
   例如用户直接提供上下文、用固定流程、增加工具校验、优化提示词、加一个确认步骤，是否已经足够？

3. **引入该能力会增加什么复杂度？**
   例如 RAG 会引入索引和引用问题，Memory 会引入隐私和污染问题，Context Engineering 会增加信息分层设计成本，HITL 会增加延迟和交互设计复杂度，Multi-Agent 会引入通信和调试问题。

4. **如何做最小版本？**
   先做一个可评测的最小增强，而不是一开始上完整架构。

5. **如何判断它真的变好了？**
   必须有任务集、对比指标或人工验收标准。
