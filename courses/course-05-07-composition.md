### 7.1 不要一次性堆满能力

1.1 里的知识助手 Agent，最终集齐了五种能力。但这不是一次加完的——它是在几个月的使用中，一个问题接一个问题地被逼出来的。

真实项目中，增强能力确实经常组合出现。个人知识助手需要 RAG + Memory；代码审查 Agent 需要 Tool Use + Reflection + Reviewer。但组合不等于一次性在设计文档里画完架构图。

更好的顺序是——让问题驱动引入节奏：

```text
先让最小闭环跑起来
  -> 观察第一个明确的问题模式
  -> 用最小版本的能力修复它
  -> 评测是否真的变好了
  -> 再观察下一个瓶颈
  -> 循环
```

不要因为某个能力在社区里很热门就加入系统。每个能力进入系统之前，都应该有问题证据和评测方式。如果你说不出"没有 RAG 时用户具体在哪一步遇到了什么问题"，那 RAG 就不该是你的第一个优先。

### 7.2 能力组合案例

#### 案例一：个人知识助手

起始问题：

- 用户资料很多，模型不知道资料内容。
- 用户需要引用来源。

优先能力：

- RAG / 外部知识接入。

后续问题：

- 用户希望记住当前研究主题。
- 多轮追问时需要延续上下文。

再引入：

- 会话摘要和轻量 Memory。

暂不引入：

- Multi-Agent，除非出现明确的并行调研或审查需求。

#### 案例二：代码审查 Agent

起始问题：

- Agent 只读 diff 后输出泛泛建议。
- 不能验证自己的判断。

优先能力：

- 工具调用：读文件、搜索代码、运行测试。
- Reflection：测试失败后修正判断。

后续问题：

- Agent 修改代码后需要审查。

再引入：

- Reviewer 模式的轻量 Multi-Agent。

暂不引入：

- 长期 Memory，除非需要记住项目约定、测试习惯或用户偏好。

#### 案例三：个人任务执行 Agent

起始问题：

- 任务多步骤，容易漏步骤。
- 用户希望在关键节点确认。

优先能力：

- Planning / Workflow Patterns。
- Human-in-the-loop。

后续问题：

- 长任务中断后需要继续。

再引入：

- 任务状态和 Checkpoint，这部分会在课程六深入。

暂不引入：

- 自动长期 Memory，除非用户明确需要跨任务偏好延续。

### 7.3 能力引入决策表

| 当前问题 | 优先考虑 | 不建议引入的信号 |
|---|---|---|
| 答案依赖外部资料、私有知识或实时数据 | RAG / 外部知识接入 | 用户输入已包含全部信息 |
| 需要跨轮、跨会话或跨任务延续 | Memory | 每次任务独立，隐私风险高 |
| 任务步骤多、依赖强、容易漂移 | Planning / Workflow Patterns | 两三步固定流程足够 |
| 有明确失败信号，需要自动修正 | Reflection | 没有外部反馈，只是让模型自评 |
| 单 Agent 角色冲突、需要并行或互审 | Multi-Agent | 单 Agent 已可完成，缺少 Trace 和评测 |

### 7.4 写一份能力引入说明

一个成熟的 Agent 设计说明，应该能解释为什么引入某个能力，也能解释为什么不引入某个能力。

模板如下：

```text
能力名称：
当前场景：
用户问题：
问题证据：
为什么现有最小闭环不够：
为什么选择该能力：
最小实现版本：
新增复杂度：
评测方式：
不引入的边界：
后续迭代：
```

示例：

```text
能力名称：RAG / 外部知识接入
当前场景：个人知识助手回答用户笔记问题
用户问题：答案必须基于本地笔记，并给出引用
问题证据：无检索时模型输出通用解释，无法引用来源
为什么现有最小闭环不够：模型没有访问用户资料
为什么选择该能力：需要把本地资料作为回答证据
最小实现版本：Markdown 文档切分 + 向量检索 + 引用输出
新增复杂度：文档解析、切分、引用校验、权限边界
评测方式：20 个问题，检查召回、答案、引用和拒答
不引入的边界：用户直接提供完整资料时，不使用检索
后续迭代：混合检索、rerank、引用后校验
```
