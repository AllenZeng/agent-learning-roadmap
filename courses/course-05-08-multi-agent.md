# 第六章：Multi-Agent —— 从"一个人干活"到"一支团队协作"

[返回课程五主文档](./course-05-01-scenario-enhancement.md) | [上一章](./course-05-05-reflection.md) | [下一章](./course-05-07-composition.md)

## 课程导言

第五章讲的是 Reflection —— Agent 看到测试失败后停下来修正。这是一个 Agent 对自己做的事：执行、检测、修正、重试。

但有一类问题 Reflection 解决不了：**同一个大脑无法公正地审视自己的作品。**

回到知识助手。你让它写一份 API 模块技术方案，然后从安全角度审查。它写完后，你说："现在从安全角度审查这个方案。"它回答：

```text
经过审查，该方案在安全方面没有明显问题。建议按计划实施。
```

你扫了一眼，立刻发现三个问题：输入没有长度校验、API 密钥明文写在配置文件里、权限模型缺少最小权限原则。Agent 不是不聪明——而是刚刚作为"作者"写完了方案，现在要让同一个大脑作为"审查者"审视自己的作品。同一个上下文、同一段推理痕迹、同一个目标函数——它怎么可能发现自己的盲区？

这就引入了 Agent 开发的最后一类增强能力：**Multi-Agent（角色分工与协作）**。

但 Multi-Agent 也是五类增强能力中最容易被误用、最容易被过度工程化的一项。2023 年底 AutoGen、CrewAI 等框架相继出现，"多智能体协作"成了热词。很多团队一上来就定义四五个 Agent 角色——研究员、工程师、审查员、协调员——然后发现系统不仅没有变好，反而更慢、更贵、更难调试。

本课不会教你"如何用某个框架搭建 Multi-Agent 系统"。框架会过时，API 会变化。本课要讲的是更底层的东西：**Multi-Agent 的本质是什么、什么时候真的需要它、如何用最少的 Agent 解决真实问题、以及它的隐藏代价是什么。**

整章围绕一条核心叙事展开：设计 Multi-Agent 系统本质上是在做"团队管理"——拆解工作、分配上下文、设计通信协议、设定裁决机制。不是在写 Prompt。

学完本课，你不会成为某个 Multi-Agent 框架的专家。但你应该能够独立判断：我这个场景要不要引入 Multi-Agent？从哪个模式开始？如何避免最常见的五个坑？

---

## 学习目标

学完本章，你将能够：

1. **诊断单 Agent 的三类硬天花板**：角色冲突、上下文挤压、串行瓶颈——以及为什么"换更强的模型"解决不了这些结构性缺陷
2. **判断 Multi-Agent 的真实必要性**：用"四个不同"标准（输入、工具、目标、验收标准）检验你的场景是否真的需要多 Agent
3. **独立实现三种核心协作模式**：Reviewer（执行+审查）、Supervisor（拆解+派发+汇总）、Parallel Specialists（同任务多维度分析）
4. **设计 Agent 间的结构化通信协议**：知道为什么自由对话是最昂贵的协作模式，以及如何用结构化格式替代它
5. **建立裁决、停止和兜底机制**：确保 Multi-Agent 系统是一个"决策系统"而非"讨论组"
6. **核算 Multi-Agent 的真实成本**：不止 token 账单，还有延迟放大、trace 复杂度和长期维护成本
7. **写出 Multi-Agent 引入或不引入的技术判断**：用决策流程和自查清单做出理性选择

### 6.1 单 Agent 的三类硬天花板

在讨论 Multi-Agent 怎么设计之前，先要搞清楚一个更根本的问题：**单 Agent 到底卡在哪？** 如果你说不清楚这个问题，那你引入 Multi-Agent 大概率是在"为架构而架构"。

单 Agent 的能力瓶颈不是"模型不够聪明"，而是结构性的。有三类硬天花板，分别对应三种不同的故障模式。

#### 6.1.1 角色冲突：创作者审查不了自己的作品

回到本章开头那个场景。知识助手写完 API 技术方案后，你让它从安全角度审查。它说"没有明显问题"——而你一眼看到了三个安全隐患。

这不是态度问题，是机制问题。同一个 Agent 在"创作模式"下产生了大量的中间推理——"这里为了方便本地开发先用明文密钥""权限模型先简化，后面再细化"——这些推理在创作时是合理的权衡，但它们留在了上下文里。当你让它切换到"审查模式"时，这些推理成了先入为主的"解释"：看到明文密钥，它想到的是"这是为了方便开发"，而不是"这是一个安全漏洞"。

打个比方：你花了两小时写完一份 3000 字的技术方案，然后立刻被要求"从安全角度严格审查这个方案"。即使你是一个有安全经验的人，你也会下意识地为自己的设计辩护。同一个大脑，先入为主不是态度问题，是认知机制问题。Agent 也一样——只不过它的"先入为主"表现为上下文里已经存在的那些 token。

**角色冲突的结构性根因**：创作者的优化目标是"完成方案"，审查者的优化目标是"找出问题"。这两个目标天然矛盾。让同一个上下文、同一个目标函数同时服务两个矛盾的目标——结果是目标之间互相妥协，审查变成了走过场。

#### 6.1.2 上下文挤压：太多中间推理变成了噪声

单 Agent 处理复杂任务时，上下文窗口里会堆积大量中间产物：第一版的草稿、失败尝试的 trace、探索性检索的结果、各种"先记下来后面可能用到"的临时记录。

这些东西在它们产生的那个时刻是有价值的。但当 Agent 需要基于最终产出做判断时，这些中间推理变成了**噪声**——它们占据了上下文空间、分散了注意力、引入了过时的假设。

上下文挤压的典型表现：

```text
Agent 上下文窗口（简化示意）：
┌────────────────────────────────────────────────────────────┐
│ System Prompt (500 tokens)                                 │
│ 任务：写技术方案 + 安全审查                                  │
├────────────────────────────────────────────────────────────┤
│ 用户消息 (100 tokens)                                       │
├────────────────────────────────────────────────────────────┤
│ 第 1 轮：尝试用方案 A，发现不合理，放弃 (800 tokens)         │  ← 噪声
│ 第 2 轮：尝试用方案 B，写了一部分 (1200 tokens)              │  ← 部分噪声
│ 第 3 轮：完成方案 B 的初稿 (2000 tokens)                     │  ← 有用
│ 检索中间结果：关于 API 设计的 5 篇笔记摘要 (1500 tokens)     │  ← 部分噪声
│ 第 4 轮：润色方案 (800 tokens)                              │  ← 噪声
├────────────────────────────────────────────────────────────┤
│ 用户："现在从安全角度审查"                                   │
│ Agent 需要在上述全部上下文中做安全判断                        │
│                                                             │
│ 问题：它看到了第 1 轮中"为了简化先用明文"的推理，              │
│ 这个推理会在审查时变成"这是有意为之，不是问题"                 │
└────────────────────────────────────────────────────────────┘
```

**上下文挤压的结构性根因**：单 Agent 的上下文窗口是所有信息的"公共池"。创作过程的中间产物和最终产出混在一起，Agent 无法区分"这是最终决定"和"这是一次被放弃的尝试"。审查时，这些历史信息污染了判断。

#### 6.1.3 串行瓶颈：互不依赖的任务在排队

前两类是质量问题，第三类是速度问题。

知识助手接到了调研任务："帮我调研 Agent 架构的四个主流方向——Tool Use、Memory、Planning、Multi-Agent——每个方向给我最新实践总结。"单 Agent 的处理方式：

```text
时间线（串行执行）：
├─ [0:00-1:30] 调研 Tool Use：检索笔记 + 搜索最新资料 + 整理输出
├─ [1:30-3:00] 调研 Memory：检索笔记 + 搜索最新资料 + 整理输出
├─ [3:00-4:30] 调研 Planning：检索笔记 + 搜索最新资料 + 整理输出
├─ [4:30-6:00] 调研 Multi-Agent：检索笔记 + 搜索最新资料 + 整理输出
└─ [6:00-7:00] 汇总四份调研，生成最终报告

总耗时：约 7 分钟
```

但仔细看这四个调研任务——它们互不依赖。调研 Memory 不需要等 Tool Use 的结果出来。它们完全可以同时进行。

**串行瓶颈的结构性根因**：单 Agent 只有一个执行线程。即使任务之间没有依赖关系，也只能一个一个来。这不是模型速度的问题——模型推理再快，四个任务排队执行的总时间也是四个任务之和。

#### 6.1.4 "换更强的模型"为什么不是答案

一个自然的想法是：等更强的模型出来，这些问题是不是就自动解决了？

不是。因为这三类天花板是**结构性问题**，不是**能力问题**。

- **角色冲突**不是模型不够聪明所以审不出来——是同一上下文中矛盾的目标互相妥协。更强的模型可能更善于在两个目标间切换，但只要创作和审查共享同一个上下文，先入为主的"解释"就会软化审查标准。
- **上下文挤压**不是模型上下文窗口不够大——是信息结构的问题。更大的窗口只是能塞更多噪声，不解决"中间推理污染最终判断"的结构性缺陷。
- **串行瓶颈**更和模型能力无关——单线程就是单线程。

这就是 Multi-Agent 存在的根本理由：当单 Agent 的**结构**（而非能力）成为瓶颈时，需要用**多实例、多上下文、多工具集**来突破这些结构限制。

### 6.2 拆分的是什么——Multi-Agent 的"四个不同"

认清了单 Agent 的天花板，下一步不是"定义 Agent 角色"，而是先搞清楚：**Multi-Agent 到底在拆分什么？**

#### 6.2.1 改个 Prompt 不叫 Multi-Agent

最容易犯的错误：把"写 System Prompt"当成"创建 Agent"。定义三个角色——研究员、工程师、审查员——给每个角色写一段 System Prompt，然后让同一个模型轮流扮演。这不叫 Multi-Agent，这叫**角色扮演**。

角色扮演和 Multi-Agent 的本质区别：

```text
"假" Multi-Agent（改 Prompt）：
┌─────────────────────────────────────────────────────┐
│ 同一个 LLM 实例                                      │
│ 同一个上下文窗口                                     │
│ 同一个工具集                                        │
│                                                     │
│ Round 1: System = "你是研究员" → 产出调研报告         │
│ Round 2: System = "你是工程师" → 看到调研报告 + 所有  │
│          中间推理 → 产出技术方案                      │
│ Round 3: System = "你是审查员" → 看到技术方案 + 调研  │
│          报告 + 所有中间推理 → "没有明显问题"          │
│                                                     │
│ 三个角色共享同一个大脑、同一段记忆、同一套工具。        │
│ "审查员"看到了"研究员"和"工程师"的所有思考过程——       │
│ 包括那些"这里简化一下后面再说"的妥协。                 │
└─────────────────────────────────────────────────────┘

"真" Multi-Agent（独立实例）：
┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐
│ Agent A: 研究员  │  │ Agent B: 工程师  │  │ Agent C: 审查员  │
│                 │  │                 │  │                 │
│ 上下文：         │  │ 上下文：         │  │ 上下文：         │
│ - 调研任务       │  │ - 技术方案任务    │  │ - 审查标准       │
│ - 检索结果       │  │ - 调研报告（最终） │  │ - 技术方案（最终）│
│                 │  │                 │  │ - 原始需求       │
│ 工具：           │  │ 工具：           │  │                 │
│ - 搜索          │  │ - 写文件         │  │ 工具：           │
│ - 检索笔记       │  │ - 读文件         │  │ - 读文件（只读）  │
│                 │  │                 │  │ - 安全扫描       │
│ 看不到：         │  │ 看不到：         │  │                 │
│ - 工程师的推理    │  │ - 研究员的中间推理 │  │ 看不到：         │
│ - 审查员的判断    │  │ - 审查员的判断    │  │ - 研究员的推理    │
│                 │  │                 │  │ - 工程师的妥协    │
└─────────────────┘  └─────────────────┘  └─────────────────┘
```

关键差异不在 Prompt 写得多好，而在**每个 Agent 看到的东西不同、能做的事不同**。

#### 6.2.2 四个不同：输入、工具、目标、验收标准

真正的 Multi-Agent 拆分的是四个维度。以知识助手的"写技术方案 + 安全审查"场景为例：

| 维度 | 单 Agent（自审） | 真 Multi-Agent（Author + Reviewer） |
|---|---|---|
| **输入不同** | Author 和 Reviewer 是同一个上下文，Reviewer "看到"了 Author 的中间推理和妥协过程 | Author 上下文有需求文档、笔记检索结果、写作工具。Reviewer 上下文**只有**最终方案 + 审查标准，看不到 Author 的草稿和"先简化后面再说" |
| **工具不同** | 同一个工具集——既能写文件又能做安全扫描 | Author 能写文件、检索笔记。Reviewer 只能读文件、跑安全扫描——**不能写**，确保审查者不会悄悄"修了再通过" |
| **目标不同** | "完成用户的任务"——目标模糊，各子目标互相妥协 | Author 目标：产出满足需求的技术方案。Reviewer 目标：找到所有不满足安全标准的问题。**目标不互相妥协** |
| **验收标准不同** | "用户觉得行就行"——没有客观标准 | Author 标准：方案完整覆盖需求。Reviewer 标准：审查清单逐条通过，任何一条 FAIL 则整体 REJECT |

**这四个维度中，至少有两个不同，Multi-Agent 才有存在的价值。** 如果两个 Agent 的输入相同、工具相同、目标相同、验收标准相同——那不管给它们起什么名字，它们本质上是同一个 Agent。你只是在浪费 token 和延迟。

这里有一个最简判断标准：**如果你说不出两个 Agent "看到的东西有什么不同"，那就不需要两个 Agent。**

#### 6.2.3 自检清单：你的场景满足几个

在引入 Multi-Agent 之前，逐条回答：

```text
□ 角色冲突：是否存在两个目标天然矛盾的角色？
   （例：创作者 vs 审查者、优化者 vs 安全评估者、推销者 vs 风险分析师）
   如果不存在天然矛盾，单 Agent + 清晰的 Prompt 通常就够了。

□ 独立上下文需求：一个角色的中间推理对另一个角色是噪声还是必要信息？
   （例：代码实现过程中的调试 trace 对安全审查是噪声；方案讨论中的妥协对最终验收是噪声）
   如果中间推理对所有角色都有用，那不需要拆分上下文。

□ 工具权限分离：是否有些操作不应该由同一个 Agent 负责？
   （例：批准部署 vs 执行部署、写代码 vs 合并到主分支、生成发票 vs 审批发票）
   如果所有操作可以被同一个 Agent 安全地完成，不需要拆分。

□ 并行可能性：是否存在互不依赖、可以同时执行的子任务？
   （例：调研四个独立方向、分析三个独立模块、从五个数据源并行拉取数据）
   如果任务有严格的顺序依赖，并行的价值为零。

□ 视角多样性：是否需要从不同立场、不同假设、不同风险偏好来审视同一个问题？
   （例：技术方案需要同时从成本、安全、可维护性三个角度评估）
   如果单一视角已经能覆盖所有关注点，不需要多视角。
```

**入门判断**：满足其中至少两项，才值得认真考虑 Multi-Agent。只满足一项时，通常有更简单的方法（更好的 Prompt、更细的工具权限、预定义的检查清单）能达到类似效果。

### 6.3 Reviewer 模式：最简单的 Multi-Agent 入口

Reviewer 是 Multi-Agent 的"Hello World"——只需要两个 Agent，通信模式最简单，价值最容易评测。如果连 Reviewer 都跑不通，就不要想 Supervisor、Debate 或 Graph Collaboration。

它只解决一个问题：**执行者不能公正地审查自己的产出。**

#### 6.3.1 模式骨架：一个执行，一个审查

```text
Executor Agent（执行者）                   Reviewer Agent（审查者）
┌──────────────────────────┐              ┌──────────────────────────┐
│ System: 你是技术方案撰写者 │              │ System: 你是安全审查员     │
│                          │              │                          │
│ 工具：                    │              │ 工具：                    │
│  - 写文件                 │   最终产出    │  - 读文件（只能读！）       │
│  - 检索笔记               │─────────────►│  - 检索安全规范            │
│  - Web 搜索               │              │  - 检查清单核对            │
│                          │   审查意见    │                          │
│ 上下文：                   │◄─────────────│ 上下文：                   │
│  - 需求文档               │              │  - 最终产出（只有这个！）    │
│  - 笔记检索结果            │              │  - 审查标准清单            │
│  - 创作过程中的草稿和推理   │              │  - 安全规范文档            │
│                          │              │                          │
│ 目标：完成技术方案          │              │ 目标：找出所有安全问题       │
│ 验收：方案覆盖需求          │              │ 验收：审查清单逐条通过       │
└──────────────────────────┘              └──────────────────────────┘
```

核心骨架代码：

```python
# Reviewer 模式：最简 Multi-Agent，2 个 Agent + 结构化审查
class ReviewerPattern:
    """Executor 产出 → Reviewer 审查 → 修正或上报"""

    def __init__(self, executor: Agent, reviewer: Agent):
        self.executor = executor
        self.reviewer = reviewer

    def run(self, task: str, criteria: list[dict]) -> dict:
        """
        task: 用户任务描述
        criteria: 审查清单，每项是可逐条核对的检查项
          [{"id": "C1", "check": "所有用户输入是否经过长度校验？",
            "how_to_verify": "检查 API 端点定义中的参数声明"},
           {"id": "C2", "check": "API 密钥是否存储在环境变量中？",
            "how_to_verify": "搜索配置文件中是否包含 key=或 secret= 字面量"},
           ...]
        """
        # Step 1: Executor 产出
        draft = self.executor.run(task)

        for round_num in range(2):  # 最多修正两轮
            # Step 2: Reviewer 审查
            # 关键：reviewer 只收到最终产出和审查标准，收不到 executor 的中间推理
            review = self.reviewer.run(
                task="对照审查清单，逐条检查以下产出。给出具体的通过/不通过判断和证据。",
                context={
                    "original_requirement": task,
                    "artifact": draft.output,
                    "criteria": criteria
                }
            )

            if review.verdict == "approved":
                return {
                    "status": "approved",
                    "output": draft.output,
                    "review_rounds": round_num + 1,
                    "review_trace": review
                }

            # Step 3: Executor 修正
            # 关键：只传具体的 issues，不传 reviewer 的主观评价
            draft = self.executor.run(
                task=task,
                fix_instructions=review.issues  # 具体问题列表，不是"整体质量较差"
            )

        # 两轮未通过：标记分歧，人工裁决
        return {
            "status": "disputed",
            "output": draft.output,
            "unresolved_issues": review.issues,
            "message": "两轮修正仍未通过审查，需要人工裁决"
        }
```

> **设计要点**：Executor 和 Reviewer 使用独立的 LLM 实例——不同的 System Prompt、不同的工具集、不同的上下文窗口。`fix_instructions` 只传 Reviewer 发现的具体问题（如"第 3 行缺少输入长度校验"），不传 Reviewer 的主观评价（如"整体质量较差"）。让 Executor 根据事实修正，而不是被另一个模型的主观意见带偏。两轮上限是保护机制——第三轮修正的边际收益通常不足以覆盖通信成本和决策疲劳。

#### 6.3.2 为什么独立上下文比"换个角色 Prompt"有效

Reviewer 模式最关键的机制是**独立上下文**。它不是"让模型换个角度思考"——同一个上下文里换个角度，先入为主的信息还在。独立上下文是让 Reviewer **物理上看不到** Executor 的中间推理。

具体来说，Reviewer 看不到这些东西：

| Executor 上下文中有但 Reviewer 收不到的信息 | 为什么不能让 Reviewer 看到 |
|---|---|
| "这里为了方便本地开发，先用明文存储密钥" | 这句话会让 Reviewer 把"明文密钥"从"安全漏洞"重新理解为"有意的临时方案"，审查就变软了 |
| "权限模型先简化为 admin 一种角色，后面再细化" | Reviewer 看到这个解释后，可能不再将"缺少最小权限"标记为 FAIL |
| 三版草稿中的前两版（被放弃的） | 噪声——和最终方案无关，但可能混淆 Reviewer 对方案结构的理解 |
| Executor 在写作过程中的不确定和犹豫（"这里不太确定，先这样写"） | 会让 Reviewer 产生不必要的怀疑或放过真正的缺陷 |

**独立上下文的本质是信息不对称**——而且是**有意设计的信息不对成**。Executor 知道的信息比 Reviewer 多，但有些信息 Reviewer 不应该知道——知道了就会影响判断。这和现实中的"盲审"是一个道理：审稿人不知道作者是谁，不是能力问题，是制度设计——屏蔽身份信息让判断更客观。

#### 6.3.4 Reviewer 的失效边界

Reviewer 模式有效，但它只在特定条件下有效。下面是它的四个失效边界：

**边界一：审查清单模糊时失效。** 如果审查标准是"检查方案是否安全"（一句话），Reviewer 的输出会和单 Agent 自审一样——"没有明显问题"。审查清单必须是具体的、可逐条核对的、有明确 verify 方法的。

```text
模糊的审查标准：
  "检查方案是否安全" → Reviewer 输出"整体安全，没有明显问题"

具体的审查标准：
  "1. 所有用户输入是否经过长度和类型校验？验证方法：检查 API schema 中每个参数声明。
   2. 密钥是否存储在环境变量或密钥管理服务中？验证方法：grep config 文件中的 key= 模式。
   3. 是否存在非 admin 的角色定义？验证方法：检查权限模块中定义的角色列表。
   4. 第三方依赖是否锁定了版本号？验证方法：检查 requirements.txt 中的版本声明格式。"
```

**边界二：Reviewer 没有独立验证工具时失效。** 如果 Reviewer 只能"肉眼审查"——读方案文本然后判断——它和单 Agent 自审没有本质区别。Reviewer 必须有独立于 Executor 的验证工具：读原始配置文件（而不是读 Executor 在方案中的描述）、跑 linter、检索安全规范原文。核心原则：**Reviewer 验证的是"实际的东西"，不是 Executor "声称的东西"**。

**边界三：Reviewer 过于严苛导致死锁。** 如果每条审查建议都是"必须修改"，Executor 改完一轮后 Reviewer 又发现了新的问题——注意，是新的问题，不是原来的问题没修好——然后 Executor 再改，Reviewer 再发现新问题……永远不会 approved。修正方式：区分"必须修改"和"建议优化"；必须项不超过 5 条，建议项不阻塞通过。

**边界四：Executor 学会了"预判"审查清单。** 这是最隐蔽的失效模式。经过多次 Review 后，Executor 学会了在方案中主动添加"看起来像安全措施"的描述——"本模块遵循安全最佳实践""所有输入已做充分校验"——但这些描述并没有对应实际的实现。Reviewer 看到这些声明后，在检查清单中标记"已提及安全措施"→ PASS。但实际上安全措施并没有被实施。修正方式：审查清单从"是否提及"改为"是否实施"——不要检查"方案是否讨论了密钥管理"，而是检查"方案中的密钥是否实际存储在环境变量中（grep 验证）"。

### 6.4 Supervisor 模式：拆解、派发、汇总

Reviewer 解决的是"质量"问题——执行者需要被独立审查。但当任务本身可以自然拆成多个互不依赖的子任务时，单 Agent 遇到了**速度**问题——串行瓶颈。

Supervisor 模式用一个调度 Agent 负责拆解和汇总，多个 Worker Agent 并行执行。

#### 6.4.1 模式骨架：一个调度者 + 多个执行者

```text
Supervisor Agent                             Worker Agents
┌─────────────────────────┐       ┌─────────────────────────────────┐
│ 接收用户任务              │       │ Worker 1: 调研 Tool Use         │
│                         │       │  - 独立上下文                    │
│ 拆解为 N 个子任务：       │       │  - 独立检索工具                  │
│  - 明确边界               │───┬───│  - 输出：结构化调研结果           │
│  - 指定输出格式           │   │   │                                 │
│  - 指派 Worker           │   │   │ Worker 2: 调研 Memory           │
│                         │   │   │  - 独立上下文                    │
│ 汇总 N 份结果：           │   │   │  - 独立检索工具                  │
│  - 去重                  │   │   │  - 输出：结构化调研结果           │
│  - 识别冲突               │◄──┴──│                                 │
│  - 标注缺失               │       │ Worker 3: 调研 Planning         │
│  - 合成最终输出           │       │  ...                            │
└─────────────────────────┘       │ Worker 4: 调研 Multi-Agent      │
                                  └─────────────────────────────────┘
```

核心骨架代码：

```python
class SupervisorPattern:
    """Supervisor 拆任务 → Workers 并行 → Supervisor 汇总"""

    def __init__(self, supervisor: Agent, workers: dict[str, Agent]):
        self.supervisor = supervisor
        self.workers = workers

    def run(self, task: str) -> dict:
        # Step 1: Supervisor 拆解任务
        # 拆解结果必须包含：边界、输出模板、Worker 指派
        plan = self.supervisor.decompose(task)
        # plan.subtasks = [
        #   {"id": "T1", "topic": "Tool Use 最新实践",
        #    "worker": "researcher_1",
        #    "scope": "设计模式、失败模式、框架对比",
        #    "exclude": "不包含与其他方向的交叉内容（交叉由汇总时处理）",
        #    "output_template": "## Tool Use\n### 关键发现\n- ...\n### 失败模式\n- ...\n### 来源\n- ..."},
        #   ...
        # ]

        # Step 2: 并行执行
        results = parallel_execute(
            plan.subtasks,
            lambda st: self.workers[st.worker].execute(
                task=f"调研 {st.topic}。范围：{st.scope}。排除：{st.exclude}。"
                     f"输出格式：{st.output_template}",
                tools=["search_notes", "web_search"]
            )
        )
        # results 中失败或超时的 Worker 返回 None，不阻塞其他 Worker

        # Step 3: Supervisor 汇总
        # 关键操作：去重、识别冲突、标注缺失、合成
        final = self.supervisor.synthesize(
            task=task,
            worker_results=results,
            instructions="""
            汇总规则：
            1. 如果两个 Worker 对同一话题给出矛盾结论 → 标注"存在分歧"而非自动选择
            2. 如果某个 Worker 超时或失败 → 在报告中标注"该方向数据缺失"
            3. 去重：相同发现合并，标注来自哪些 Worker
            4. 最终输出按统一结构组织，不要直接拼接 Worker 原文
            """
        )
        return final
```

#### 6.4.2 拆解质量决定了整个模式的价值

Supervisor 模式最容易被低估的一步是**拆解**。很多实现把拆解简化为"让 LLM 把任务分成几份"——然后发现 Worker 产出重叠严重、格式不一致、无法合并。

好的拆解需要四样东西：

**1. 明确的边界（含 exclude）**

不只是"你调研 A，你调研 B"，还要说清楚"你不要碰什么"。

```text
糟糕的拆解：
  Worker 1: 调研 Tool Use
  Worker 2: 调研 Multi-Agent
  → 两个 Worker 都在写关于"Tool Use 在 Multi-Agent 中的应用"
  → 30% 内容重叠

好的拆解：
  Worker 1: 调研 Tool Use 的设计模式、失败模式和框架实现
            exclude: 不涉及 Tool Use 在 Multi-Agent 协作中的角色（由 Worker 4 覆盖）
  Worker 2: 调研 Multi-Agent 的协作模式、通信协议和故障模式
            exclude: 不涉及单个 Agent 的 Tool Use 机制（由 Worker 1 覆盖）
```

**2. 统一的输出模板**

每个 Worker 必须按相同的结构输出，否则 Supervisor 无法自动合并。

```text
输出模板（所有 Worker 共用）：
## {调研方向}
### 关键发现
- 发现 1（1-2 句话 + 来源引用）
- 发现 2
### 失败模式
- 常见失败 1（表现 + 原因 + 修正方向）
### 推荐实践
- 实践 1（适用场景 + 不适用场景）
### 来源引用
- [来源 1](链接或笔记路径)
```

**3. Worker 能力匹配**

不是所有 Worker 都应该是同一个模型。调研型 Worker 可能需要强检索能力（联网搜索、长上下文），分析型 Worker 可能需要强推理能力。把合适的任务指派给合适的 Worker。

**4. 拆解的粒度控制**

拆得太细（10 个子任务），通信和汇总成本超过执行收益。拆得太粗（2 个子任务），并行度不足。一个经验法则：**子任务数量 = min(可并行的独立维度数, 可用 Worker 数, 5)**。超过 5 个子任务的边际收益通常不足以覆盖协调成本。

#### 6.4.3 合并的成本——"三个 Worker 跑完了，Supervisor 花了更长时间合并"

这是 Supervisor 模式最经典的翻车现场：

```text
场景：用户要求"调研 Agent Memory 的最新实践"
Supervisor 拆解为 3 个子任务 → 3 个 Worker 并行执行

时间线：
├─ Worker 1 调研"短期记忆"（45 秒）
├─ Worker 2 调研"长期记忆"（50 秒）
├─ Worker 3 调研"Memory 框架"（40 秒）
│  并行耗时：50 秒 ✓ 比串行快
│
├─ Supervisor 开始合并（60 秒）← 问题在这里
│  为什么？因为三个 Worker 的产出：
│  - Worker 1 输出了 3 页 Markdown（概述 + 详细分析 + 代码示例）
│  - Worker 2 输出了 5 个要点 + 1 个表格
│  - Worker 3 输出了 8 个框架的列表（没有分析）
│  格式完全不同，结构完全不同，覆盖范围有大量重叠。
│  Supervisor 无法"自动合并"——它实际上是在让 LLM 重新"综合"
│  三份报告，而这个综合工作如果从一开始就让一个 Agent 做，
│  可能只需要 70 秒。
│
├─ 总耗时：50 秒（并行执行）+ 60 秒（合并）= 110 秒
│  串行耗时：约 120 秒（三个方向 × 40 秒 + 自然汇总 10 秒）
│  并行收益：几乎为零。引入了 3 倍复杂度，省了 10 秒。
└─ 

根因：拆解时没有指定输出模板。三个 Worker 各自按自己的理解输出，
      合并成本抵消了并行收益。

修复：
1. 拆解时强制指定统一输出模板（结构、字段、长度限制）
2. 每个 Worker 的输出长度控制在 300 字内（禁止输出"概述"和"背景介绍"）
3. Worker 只输出"提取后的信息"，不做综合和总结——综合由 Supervisor 负责
```

**这个翻车的核心教训**：并行的价值不在于"Worker 跑得快"，而在于 **"Worker 的产出可以直接被组合，不需要 LLM 重新理解和综合"**。如果合并步骤需要 LLM 通读所有 Worker 产出并"重新写一遍"，那不如一开始就让一个 Agent 写。

#### 6.4.4 Supervisor 的失效边界

| 失效模式 | 表现 | 根因 | 修正 |
|---|---|---|---|
| 拆解边界模糊 | Worker 产出 30% 重叠 | 拆解时没有定义 exclude 范围 | 每个子任务附带明确的 exclude 声明 |
| 格式不统一 | Worker 产出无法自动合并，Supervisor 需要重新"综合" | 没有输出模板 | 拆解时强制指定输出模板（允许 Worker 在模板内自由发挥） |
| Worker 失败导致报告不完整 | Worker 2 超时，Supervisor 假装 4 个方向都完成了 | 汇总时没有标注"数据缺失" | 汇总规则强制：失败 Worker 对应的方向标注"数据缺失，原因：{超时/错误}" |
| 并行变串行 | 拆解本身消耗了 30 秒（LLM 多次调用调整拆解方案） | 拆解逻辑过于依赖 LLM 决策 | 预定义拆解模板：常见任务用固定拆解维度，只在异常时让 LLM 调整 |
| Supervisor 自身成为瓶颈 | 5 个 Worker 都在等 Supervisor 的拆解结果才能开始 | 拆解和派发是串行的 | 对于标准任务，使用预定义的拆解方案（缓存），跳过 LLM 拆解 |

### 6.5 Parallel Specialists：同一任务，多双眼睛

Supervisor 的每个 Worker 处理不同任务。Parallel Specialists 是一个变体：**同一个任务，多个专家从不同维度同时分析，然后合并。**

#### 6.5.1 与 Supervisor 的区别：同一个输入，不同维度

```text
Supervisor 模式：                    Parallel Specialists 模式：
                                    
任务 A → Worker 1                   同一个任务
任务 B → Worker 2                         │
任务 C → Worker 3               ┌─────────┼─────────┐
                                ▼         ▼         ▼
不同任务，不同 Worker           Specialist A  Specialist B  Specialist C
                                (正确性)     (安全性)     (性能)
                                    │         │         │
                                    └─────────┼─────────┘
                                              ▼
                                          合并结果
                                      
不同维度，同一输入
```

适用场景：一段代码需要同时检查正确性、安全性和性能。一个方案需要同时评估技术可行性、成本、和可维护性。一个回答需要同时检查事实准确性、逻辑完整性、和表述清晰度。

#### 6.5.2 维度互斥是并行的前提

这个模式成功的核心前提是**维度之间互不依赖**。如果"性能分析"需要先知道"正确性分析"的结论，就不能并行——必须先跑正确性，再跑性能。

维度设计需要满足两个条件：
1. **互斥**：每个维度的关注点不重叠。如果"正确性"和"安全性"都在分析输入校验，60% 内容会重复。
2. **独立**：每个维度可以只看输入 + 自己的关注点就得出结论，不需要等其他维度的结果。

```python
class ParallelSpecialists:
    """多个专家并行处理同一任务的不同维度"""

    def __init__(self, specialists: dict[str, Agent]):
        self.specialists = specialists

    def run(self, task: str, dimensions: list[dict]) -> dict:
        """
        dimensions = [
          {"name": "correctness", "agent": "code_reviewer",
           "focus": "逻辑错误、边界条件、异常处理、状态一致性",
           "exclude": "不分析安全漏洞和性能瓶颈"},
          {"name": "security", "agent": "security_auditor",
           "focus": "注入风险、密钥泄露、权限越界、敏感数据暴露",
           "exclude": "不分析逻辑错误（即使它可能导致不确定行为）"},
          {"name": "performance", "agent": "perf_analyzer",
           "focus": "时间复杂度、空间占用、I/O 瓶颈、缓存策略",
           "exclude": "不分析正确性和安全性影响"},
        ]
        """
        # 并行执行
        results = parallel_execute(
            dimensions,
            lambda d: self.specialists[d["agent"]].analyze(
                task=task,
                focus=d["focus"],
                exclude=d["exclude"]
            )
        )

        # 合并：去重 + 标注来源 + 冲突识别
        return self.merge(results, dimensions)

    def merge(self, results: list[dict], dimensions: list[dict]) -> dict:
        """合并多维度分析结果"""
        all_findings = []
        conflicts = []

        for i, result in enumerate(results):
            for finding in result.findings:
                finding["source_dimension"] = dimensions[i]["name"]
                all_findings.append(finding)

        # 去重：相同位置 + 相同问题描述 → 合并为一条，标注来自多个维度
        deduped = self._deduplicate(all_findings)

        # 冲突检测：如果两个维度对同一位置给出矛盾判断
        # 例：Specialist A 说"这里的设计是安全的"
        #     Specialist B 说"这里有注入风险"
        # → 不能自动消解，标注为"存在分歧"
        conflicts = self._detect_conflicts(deduped)

        return {
            "findings": deduped,
            "conflicts": conflicts,  # 标注但不自动解决
            "dimension_summary": {
                d["name"]: len(r.findings) for d, r in zip(dimensions, results)
            }
        }
```

#### 6.5.3 合并规则：冲突不自动消解

Parallel Specialists 最危险的时刻是**合并**。当两个专家给出矛盾判断时，最容易犯的错误是让 LLM 自动选择——比如"取多数意见"或"让 Supervisor 决定"。

但这种自动消解会掩盖真正的问题。如果一个专家说"安全"而另一个说"有漏洞"，这意味着至少有一个专家的分析存在问题——可能是其中一个的 focus 定义不够清晰，可能是其中一个缺少了关键上下文。自动选择"多数"只是把问题掩盖了。

**合并规则**：
1. **相同发现自动去重**：两个专家指出同一个问题（相同位置 + 相同问题类型）→ 合并为一条，标注来自两个维度。
2. **矛盾判断不自动消解**：标注为"安全维度分析存在分歧→请人工审查"，附带两个专家的具体依据。
3. **来源标注**：每条发现标注来自哪个维度，让读者知道这个发现是在什么视角下被捕捉的。

这个模式的常见失败：

| 失效模式 | 表现 | 修正 |
|---|---|---|
| 维度不互斥导致重复 | "正确性"和"安全性"两个专家 60% 输出重叠 | 每个维度定义明确的 focus 和 exclude 范围 |
| 合并丢失冲突 | 专家 A 说安全，专家 B 说有问题，合并时取了 A | 合并规则明确：冲突标注而非消解 |
| 并行数量过多 | 开了 8 个专家，API 并发限制触发限流 | 维度数 ≤ 5；超过 5 个维度时分组并行 |
| 某个专家过于"温和" | "性能"专家每次输出"无明显性能问题" | 审查专家维度的 focus 定义是否给了足够具体的检查项 |

### 6.6 通信协议——Agent 之间不能"你觉得怎么样"

前面讲了三种协作模式的结构。但 Agent 之间具体怎么传递信息？这是 Multi-Agent 中最被低估的问题。很多系统把协作模式设计得很好，但倒在了通信协议上。

#### 6.6.1 自由对话为什么是灾难

最直觉的通信方式是让 Agent 自由对话——就像人类开会一样，大家你一句我一句地讨论。Group Chat 模式就是这样：多个 Agent 在一个共享对话中自由发言。

但自由对话在 Multi-Agent 中是**最昂贵、最难调试、最容易失败**的通信方式。原因有三：

**1. 信息衰减。** 每经过一次 Agent 之间的消息传递，原始信息都会衰减。Agent A 的发现经过 Agent B 的复述，再经过 Agent C 的引用——到 Supervisor 那里时，原始的具体判断已经变成了模糊的印象。

```text
原始："config.yaml 第 8 行 api_key 字段为明文，存在泄露风险"
↓ Agent B 复述：
"A 提到了配置文件中有密钥管理问题"
↓ Agent C 引用：
"前面的讨论涉及了安全性方面的考虑"
↓ Supervisor 收到：
"团队讨论了安全性" ← 原始信息完全丢失
```

**2. 意图扭曲。** 一个 Agent 说"建议优化"，另一个 Agent 理解为"必须优化"。"建议"和"必须"在人类沟通中靠语境区分，在 Agent 之间的文字传递中，这个语境很容易丢失。

**3. 决策模糊。** 自由对话没有"决定点"。Agent 们可以一直讨论下去，互相"同意"、互相"补充"、互相"建议进一步考虑"——但没有一个人说"讨论到此为止，以下是决定"。最终产出的不是决策，而是一份讨论纪要。

一个真实的 Group Chat 翻车记录：

```text
场景：Author、Reviewer、Supervisor 讨论技术方案
对话轮次：12 轮
总 token 消耗：8,500
最终产出：和 Author 第一版的方案几乎一样

12 轮对话的结构分析：
├─ 第 1-4 轮：互相确认"我们理解任务了吗"
│   "让我确认一下，任务是……" "是的，你的理解正确。" "好的。"
│   "我也想确认一下范围……" "范围是明确的。" "同意。"
├─ 第 5-8 轮：互相"同意"
│   Reviewer："这里需要注意安全。"
│   Author："同意，我会注意。"
│   Supervisor："好的请继续。"
│   Reviewer："另外可维护性也需要关注。"
│   Author："同意。"
├─ 第 9-12 轮：重复前面的观点
│   "如前面提到的……" "是的，正如之前讨论的……"
│
└─ 没有一个 Agent 负责"拍板"
   没有议题、没有时限、没有裁决机制
   讨论可以无限进行下去，直到 token 用完
```

#### 6.6.2 结构化通信的三层设计

替代自由对话的方式是**结构化通信**。不同场景需要不同的结构化程度，但核心原则一致：**Agent 之间的每次通信必须有明确的格式、字段和预期响应。**

**第一层：命令-响应（最刚性，适用于 Reviewer）**

```python
# Agent 之间的每次通信都是结构化对象，不是自由文本
@dataclass
class ReviewRequest:
    """Reviewer 收到的审查请求"""
    artifact: str              # 待审查的产出
    criteria: list[CheckItem]  # 审查清单
    context: dict              # 相关上下文（原始需求、约束条件）
    # 注意：不包含 executor 的中间推理

@dataclass
class CheckItem:
    id: str                    # 审查项编号
    check: str                 # 要检查什么
    how_to_verify: str         # 怎么验证
    severity: str              # "must_fix" | "should_fix" | "nice_to_have"

@dataclass
class ReviewResponse:
    """Reviewer 必须按此结构返回，不接受自由文本"""
    verdict: str               # "approved" | "rejected"
    checks: list[CheckResult]  # 逐条审查结果
    issues: list[Issue]        # 具体问题列表
    # 没有"整体评价"字段——防止模糊判断

@dataclass
class CheckResult:
    check_id: str
    passed: bool
    evidence: str              # 通过或失败的证据（具体位置、具体值）
    suggestion: str | None     # 如果失败，修正建议（≤2 句话）

@dataclass
class Issue:
    id: str
    description: str           # 具体问题，不是主观评价
    location: str              # 文件:行号
    severity: str
    suggestion: str            # ≤2 句话
```

**第二层：模板-填充（适用于 Supervisor）**

```python
# Supervisor 拆解任务时，给每个 Worker 的指派是填充好的模板
@dataclass
class SubTaskAssignment:
    task_id: str
    topic: str
    scope: str                 # 包含什么
    exclude: str               # 不包含什么
    output_template: str       # 强制输出格式
    max_length: int = 300      # 输出字数上限
    # 模板示例：
    # "## {topic}\n### 关键发现（≤3 条）\n- \n### 失败模式（≤2 条）\n- \n### 来源\n- "

@dataclass
class WorkerReport:
    task_id: str
    status: str                # "completed" | "partial" | "failed"
    content: str               # 按模板填充的内容
    sources: list[str]         # 引用来源
    error: str | None = None   # 如果失败，失败原因
```

**第三层：维度-标注（适用于 Parallel Specialists）**

```python
@dataclass
class DimensionAnalysis:
    dimension: str             # "correctness" | "security" | "performance"
    findings: list[Finding]
    # 每条发现标注维度和具体位置，便于去重和冲突检测

@dataclass
class Finding:
    id: str
    dimension: str             # 来源维度
    location: str              # 代码位置
    description: str           # 发现内容
    severity: str
    evidence: str              # 具体证据

@dataclass
class MergeResult:
    findings: list[Finding]    # 去重后的发现列表
    conflicts: list[Conflict]  # 矛盾的判断（不自动消解）
    coverage: dict             # 每个维度贡献了多少发现

@dataclass
class Conflict:
    location: str
    dimension_a: str
    finding_a: str
    dimension_b: str
    finding_b: str
    resolution: str            # 始终为 "needs_human_review"
```

**结构化通信的核心原则**：

- **不接受"整体还行"**：审查结果必须是具体的、逐条的、附带证据的。
- **自由文本只在叶节点出现**：具体描述可以是自然语言，但通信的"骨架"（状态、类型、位置、严重程度）必须是结构化字段。
- **缺失字段优于编造字段**：如果 Reviewer 找不到证据，`evidence` 字段填 `"not_found"`，而不是编造一个听起来合理的描述。
- **人类可读的 trace**：结构化通信天然产生可查询的 trace——你可以 grep `"verdict": "rejected"` 找到所有被驳回的审查，统计每个审查项的通过率。

#### 6.6.3 调试故事：三个经典通信故障

---

**故事一："整体感觉还行"——当 Reviewer 的输出没有结构**

```text
症状：
引入 Reviewer 模式两周后，检查审查记录：通过率 100%，0 个方案被 reject。
但实际上存在安全漏洞的方案也被 approved 了。

排查过程：
1. 检查 Reviewer 的历史输出 → 大部分是：
   "经过审查，该方案整体质量良好，未发现明显问题。"
2. 这不是 Reviewer 在偷懒——是通信格式允许它这样输出
3. Reviewer 的 System Prompt 里有审查清单，但没有约束输出格式
4. LLM 在面对开放式的"请审查"时，默认倾向是"先肯定再轻描淡写提建议"

根因：审查请求没有强制结构化响应。Reviewer 可以输出自由文本，而自由文本
      在"没有发现明显问题时"会退化为客套话。

修复：
1. 审查请求从"请审查以下产出"改为结构化 ReviewRequest（含逐条 CheckItem）
2. 审查响应强制为结构化 ReviewResponse——verdict 必须是 approved/rejected，
   每个 CheckItem 必须是 PASS/FAIL，每项附带 evidence
3. 不允许 Reviewer 输出"整体评价"——没有这个字段
```

---

**故事二：三个 Worker 的报告拼不起来**

```text
症状：
Supervisor 把"调研 Agent Memory 最新实践"拆给三个 Worker。
三个 Worker 都在 45 秒内返回了结果。Supervisor 花了 70 秒"合并"——
比串行执行还慢。

排查过程：
1. Worker 1 输出：5 页 Markdown，含概述、背景、详细分析、代码示例、总结
2. Worker 2 输出：3 个要点 + 1 个对比表格
3. Worker 3 输出：8 个框架的列表（每条一行，没有分析）
4. 三份产出格式完全不同。Supervisor 无法通过字段拼接来合并——
   它只能让 LLM 把三份报告"重新综合"成一份。
5. 这个"重新综合"和让一个 Agent 从零写调研报告没有本质区别。

根因：Supervisor 拆解时没有指定输出模板。每个 Worker 按自己的理解输出，
      合并成本抵消了并行收益。

修复：
1. 拆解时强制指定输出模板（结构、字段、长度限制）
2. Worker 只输出"提取后的信息"，不写概述、背景、总结——
   那些是 Supervisor 的活
3. 合并时 Supervisor 做的是"拼接 + 去重 + 标注"，不是"重新综合"
```

---

**故事三：Agent 开会上了瘾**

```text
症状：
用户引入 Group Chat 模式——Author、Reviewer、Supervisor 自由对话讨论方案。
12 轮对话后，产出了 3000 字的讨论记录。但最终方案和 Author 第一版几乎一样。

排查过程：
1. 查看 12 轮对话记录：
   第 1-4 轮：互相确认"我们理解任务了吗"
   第 5-8 轮：Reviewer 提建议，Author 说"同意"，Supervisor 说"好的"
   第 9-12 轮：重复前面的观点
2. 没有一个 Agent 负责说"讨论到此为止，以下是决定"
3. 没有议题、没有时限、没有裁决机制

根因：自由对话是"讨论"机制，不是"决策"机制。Agent 擅长生成"下一句话"，
      但不知道什么时候该停——因为停止需要判断"讨论已经充分了"，
      这正是 LLM 不擅长的元认知。

修复：
1. 放弃 Group Chat，改回 Reviewer（这个场景不需要三个 Agent 自由对话）
2. 如果确实需要多 Agent 讨论：
   - 设定明确议题（不是"讨论方案"而是"回答三个具体问题"）
   - 限制轮次（最多 3 轮后必须输出决定）
   - 指定拍板人（Supervisor 拥有最终决定权，不接受"再讨论讨论"）
```

---

**三个故事的共同教训**：

- **没有结构的通信会被 LLM 的"客套"倾向填满**。"整体还行""同意""建议进一步考虑"——这些话 token 不便宜，信息量为零。
- **输出模板不是"限制创造力"，而是"确保可组合"**。三个 Worker 的产出的价值不在于各自多精彩，而在于能否被有效组合。
- **自由对话是决策最差的载体**。讨论可以无限进行，决定需要一个明确的"拍板点"。

### 6.7 裁决、停止与兜底——Multi-Agent 的"交通规则"

协作模式定义了 Agent 怎么分工，通信协议定义了 Agent 怎么传递信息。但还有第三层：**控制机制**——当事情不如预期时，谁来决定、什么时候停、怎么收场？

没有这一层的 Multi-Agent 系统不是"决策系统"，而是一个"讨论组"——可能讨论得很好，但没人拍板。

#### 6.7.1 裁决机制：分歧时谁说了算

Multi-Agent 系统中会出现三类分歧，每类需要不同的裁决方式：

| 分歧类型 | 典型场景 | 裁决方式 | 为什么不能自动消解 |
|---|---|---|---|
| Reviewer vs Executor | Reviewer 判定 FAIL，Executor 认为"这不是问题" | 两轮修正后仍不通过→人工裁决。Executor 不应有权限否决 Reviewer 的判断 | 让被审查者裁决审查者等于取消审查 |
| Worker vs Worker（事实冲突） | Worker 1 说"框架 A 支持流式输出"，Worker 2 说"不支持" | 检查来源引用。来源明确的→引用对抗（谁的引用更权威）。来源都不明确→标注"存在事实分歧" | 事实问题需要溯源，不是投票 |
| Worker vs Worker（判断冲突） | Specialist A 说"安全"，Specialist B 说"有漏洞" | 标注冲突→人工审查。不做自动投票或取"多数意见" | 判断冲突意味着至少一个人看漏了或看偏了，需要人类重新审视 |

**裁决机制的设计原则**：

1. **裁决者不能是当事方**。Executor 不能裁决 Reviewer 的审查意见是否合理。Worker 不能裁决自己的产出是否正确。
2. **人工裁决优于自动裁决**。当两个 Agent 给出矛盾判断时，更安全的做法是暂停并请求人类介入，而不是让第三个 Agent "投票"——第三个 Agent 同样可能犯错。
3. **裁决需要有"最终期限"**。不能因为等待人工裁决而无限阻塞任务。设置超时：人工裁决超过 N 分钟未响应→采用最保守的选择（如 Reviewer 的判断优先，或暂停任务并保留现场）。

#### 6.7.2 停止条件：不能无限往返

Multi-Agent 的停止条件和 Reflection 的停止条件类似，但多了协作特有的维度：

| 停止条件 | 阈值建议 | 触发时的行为 |
|---|---|---|
| Reviewer 往返次数 | 2 轮修正后仍 rejected | 标记"disputed"，人工裁决，不进入第 3 轮 |
| Supervisor 汇总次数 | 1 次拆解-汇总 | 如果汇总结果有缺失，不重新拆解——标注缺失并输出 |
| Worker 并发超时 | 最长 Worker 耗时 × 1.5 | 超时 Worker 的结果丢弃，报告中标注"该方向数据缺失" |
| 总 token 消耗 | 单任务 50K tokens | 停止所有 Agent，返回已完成的部分结果 |
| 相同消息循环 | 连续 3 轮通信内容相似度 > 90% | 判定为"对话死循环"，强制停止并输出当前状态 |
| 错误升级 | 可恢复错误→不可恢复错误（如网络超时变成磁盘满） | 停止全部 Agent，保留现场，通知用户 |

**停止条件必须硬编码，不能由 Agent 自己决定。** Agent 没有"该停了"的直觉——它在第 6 轮往返时仍会自信地开始新的一轮通信。停止条件是 Runtime 层的强制检查，和 Agent 的推理能力无关。

#### 6.7.3 兜底策略：Worker 挂了怎么办

Multi-Agent 系统运行时，Worker Agent 可能因为各种原因失败：API 限流、网络超时、输出不可解析、上下文溢出。系统必须为每个失败模式预设兜底行为：

```text
Worker 失败模式              兜底策略
─────────────────────────────────────────────────────
单个 Worker 超时          → 丢弃该 Worker 结果
                           在最终报告中标注"方向X: 数据缺失（超时）"
                           不重试，不阻塞其他 Worker 的产出

单个 Worker 输出不可解析   → 尝试重新生成一次（仅一次）
                           仍不可解析→同上，标注"数据缺失（格式错误）"

多个 Worker 同时失败       → 可能根因是上游问题（如 API 故障）
（≥50% Worker 失败）         停止全部执行，返回部分结果 + 错误诊断
                           不继续——继续可能只是消耗更多 token

Supervisor 拆解失败        → 如果拆解方案不满足最小要求（如边界重叠>30%）
                           降级为"单 Agent 直接执行"，跳过 Multi-Agent
                           通知用户："拆解方案质量不满足并行条件，已降级"

Supervisor 汇总失败        → 返回各 Worker 的原始产出（标注为"未汇总"）
                           不尝试让 LLM 重新汇总——第一次汇总失败的原因
                           可能仍然存在
```

**兜底的核心原则：降级但不静默。** 系统可以在 Multi-Agent 不可用时降级为单 Agent 执行——但不能假装 Multi-Agent 执行成功了。缺失的数据、失败的 Worker、跳过的检查项——这些都要在最终输出中明确标注。

### 6.8 成本的真相——不只是 Token 账单

Multi-Agent 的代价经常被低估，因为它的大头不在每月的 API 账单上。

#### 6.8.1 一张真实对比账单

以知识助手"写技术方案 + 安全审查"任务为例，对比单 Agent 和 Reviewer 模式的实际消耗：

```text
═══════════════════════════════════════════════════════════════════
任务：写一份 API 模块技术方案（约 2000 字），从安全角度审查

单 Agent（自审）：
┌─────────────────────────────────────────────────────────────────┐
│ 写作阶段：                                                       │
│   System Prompt: 800 tokens                                     │
│   用户输入 + 上下文: 500 tokens                                   │
│   模型输出（方案）: 2,500 tokens                                  │
│   小计: 3,800 tokens                                            │
│                                                                 │
│ 自审阶段（同一上下文，追加一轮）：                                  │
│   用户追加消息: 100 tokens                                       │
│   模型输出（审查）: 300 tokens                                    │
│   小计: 400 tokens                                              │
│                                                                 │
│ 总计: ~4,200 tokens                                             │
│ 耗时: ~40 秒                                                    │
│ 成本: ~$0.06（以 Claude Sonnet 定价估算）                         │
│ 结果: 漏了 4 个安全隐患                                          │
└─────────────────────────────────────────────────────────────────┘

Reviewer 模式：
┌─────────────────────────────────────────────────────────────────┐
│ Author Agent（独立实例）：                                        │
│   System Prompt: 400 tokens（只管创作，不含审查逻辑）              │

│                                                                 │

│   写入上下文（方案 + 审查清单）: 2,800 tokens                      │
│   模型输出（结构化审查结果）: 600 tokens                           │
│   小计: 3,700 tokens                                            │
│                                                                 │
│ Author 修正阶段（独立实例，只收到 issues）：                       │
│   写入上下文（方案 + issues）: 3,300 tokens                       │
│   模型输出（修正后方案）: 2,600 tokens                             │
│   小计: 5,900 tokens                                            │
│                                                                 │
│ Reviewer 第二轮审查：                                             │
│   写入上下文（修正后方案 + 审查清单）: 2,900 tokens                 │
│   模型输出（审查结果）: 300 tokens                                 │
│   小计: 3,200 tokens                                            │
│                                                                 │
│ 总计: ~16,200 tokens（单 Agent 的 3.9 倍）                        │
│ 耗时: ~80 秒（单 Agent 的 2 倍）                                  │
│ 成本: ~$0.22（单 Agent 的 3.7 倍）                                │
│ 结果: 发现并修正了全部 4 个安全隐患                                │
└─────────────────────────────────────────────────────────────────┘

成本-收益分析：
┌─────────────────────────────────────────────────────────────────┐
│ 额外成本: +$0.16, +40 秒                                        │
│ 收益: 从"有 4 个安全漏洞"变为"通过安全审查清单"                     │
│                                                                 │

│       在这个场景中，额外成本是值得的                                │
│                                                                 │
│ 但如果任务是一个低风险的内部备忘录？                                │
│ 额外成本可能不值得——不是所有场景都需要 Multi-Agent                 │

```

**关键洞察**：Multi-Agent 的 token 消耗通常在单 Agent 的 2-5 倍。这不是"贵不贵"的问题——而是"额外的 token 买来了什么"。如果买来的是"发现了单 Agent 会漏掉的 4 个安全漏洞"，那 $0.16 很便宜。如果买来的是"三个 Agent 讨论 12 轮但产出和单 Agent 一样"，那每一分钱都是浪费。

#### 6.8.2 延迟放大：通信往返的真实代价

Multi-Agent 的延迟不是简单的"模型推理时间 × Agent 数量"。真实延迟包含：

```text
Reviewer 模式延迟拆解：
┌──────────────────────────────────────────────────────────────┐
│ Author 推理: 30 秒                                            │
│ +                                                             │
│ Reviewer 推理: 25 秒                                          │
│ +                                                             │
│ 上下文构建和传递: 5 秒                                         │
│   （将 Author 产出 + 审查清单打包为 Reviewer 的输入上下文）       │
│ +                                                             │
│ Author 修正推理: 20 秒                                        │
│ +                                                             │
│ Reviewer 二轮推理: 10 秒                                      │
│ =                                                             │
│ 总计: ~90 秒                                                  │
│                                                                │
│ 用户感知延迟: 90 秒（从发起到拿到 approved 结果）               │
│ 单 Agent 自审: ~40 秒                                         │
│ 延迟放大: 2.25 倍                                             │
│                                                                │
│ Supervisor 模式延迟（4 个 Worker 并行）：                       │
│ Supervisor 拆解: 8 秒                                         │
│ +                                                             │
│ 并行 Worker（取最慢）: 45 秒                                   │
│ +                                                             │
│ Supervisor 汇总: 15 秒                                        │
│ =                                                             │
│ 总计: ~68 秒                                                  │
│ 串行（4 × 40 秒 + 10 秒汇总）: ~170 秒                         │
│ 加速比: 2.5 倍 ✓                                              │
└──────────────────────────────────────────────────────────────┘
```

**延迟优化的关键**：
- Reviewer 模式天生比单 Agent 慢——每次往返增加一次推理延迟。设计时尽量减少往返次数（两轮上限）。
- Supervisor 模式的延迟收益来自并行。如果 Worker 数量少或任务差异大（一个 Worker 45 秒，另一个 10 秒），加速效果会打折扣。
- 上下文构建和传递的耗时很容易被忽略——尤其是当传递的内容很长（如完整的方案文本）时。

#### 6.8.3 长期成本：Trace 复杂度和接手难度

Multi-Agent 最隐蔽的成本不在账单上，在**维护**上。

三个月后，当新人接手这个系统，ta 面对的是：

```text
单 Agent 系统：
├─ 1 个 System Prompt
├─ 1 个工具集
├─ 1 条 trace（线性执行记录）
└─ 调试：找到出问题的步骤 → 修正 Prompt 或工具

Multi-Agent 系统：
├─ 3 个 System Prompt（Author、Reviewer、Supervisor）
├─ 3 个工具集（权限各不相同）
├─ 多条交叉 trace（Agent A 的输出是 Agent B 的输入，找出"这个错误是谁引入的"
│   需要同时翻 3 条 trace 并交叉对照）
├─ 调试："为什么最终方案少了一个安全检查？"
│   → Reviewer 漏掉了（检查 Reviewer 的 trace）
│   → Author 在修正时忽略了（检查 Author 的 trace）
│   → Supervisor 在汇总时丢失了（检查 Supervisor 的 trace）
│   → 通信协议中某个字段被错误地解析了（检查通信层的序列化逻辑）
└─ 修改影响面：改了 Reviewer 的审查清单 → 可能影响 Author 的修正策略
                → 可能影响 Supervisor 的判断逻辑
```

这不是说 Multi-Agent 不应该用。而是说：**只有当你确信额外的维护成本能被 Multi-Agent 创造的价值覆盖时，才应该引入。** 如果单 Agent + 好的 Prompt 就能做到 90 分，引入 Multi-Agent 追求 95 分的代价可能是 3 倍的维护复杂度。

### 6.9 从两个 Agent 到完整协作系统——迭代路径

Multi-Agent 不要一开始就做复杂群聊。最推荐的路径是从最小模式开始，在有明确的升级信号时才增加复杂度。

#### 6.9.1 升级信号，不是升级计划

不要因为"架构图上有五个 Agent"就去实现五个 Agent。每个阶段的升级应该由**具体故障**触发，而不是由**设计文档**驱动：

| 阶段 | 做什么 | 升级信号（出现了哪种故障，当前模式修不了） | 不要过早做什么 |
|---|---|---|---|
| V0：单 Agent | 一个 Agent 完成全流程，建立基线 | —（这是基线） | 不要过早加 Agent——先确认单 Agent 确实遇到了 6.1 描述的三类天花板之一 |
| V1：Reviewer | 增加审查 Agent，独立上下文+独立工具 | 用户或测试在 Agent 产出中发现了"创作者盲区"型错误（方案缺安全考量、代码缺边界检查），且如果由另一个 Agent 审查本应被发现 | 不要过早让 Reviewer 自动修改——先只做"发现+报告"，人工确认后再让 Author 修正 |
| V2：Supervisor | 拆解任务为并行子任务，Worker 独立执行 | 单任务耗时超过用户可接受范围，且耗时分析显示多个子任务串行执行但实际互不依赖 | 不要过早并行化——先确认子任务确实互不依赖，且串行耗时是用户可感知的问题（用户确实在等） |
| V3：Parallel Specialists | 同一个任务从多个维度并行分析 | 出现"单视角遗漏了重要问题，而该问题本可以从另一个维度发现"的故障 | 不要过早引入多维度——先确认单一视角确实有盲区，且盲区导致了实际损失 |
| V4：Debate | 从对立立场论证同一问题 | 出现"单一方案被采纳后才发现重大缺陷，该缺陷本可以由对立立场暴露"的故障 | 不要过早引入 Debate——token 消耗极大，只在单视角确实无法覆盖风险时使用 |
| V5：Graph 协作 | 将协作流程固化为图结构 | 协作流程每周至少执行一次，团队有新人需要接手；流程稳定运行至少一个月 | 不要过早固化为 Graph——流程本身还在变化，固化意味着每次调整都要改图 |

#### 6.9.2 一个真实迭代故事

下面用一个知识助手从单 Agent 到 Multi-Agent 协作系统的迭代过程，展示每个阶段的具体触发信号和决策：

```text
═══════════════════════════════════════════════════════════════════
知识助手的 Multi-Agent 迭代故事
═══════════════════════════════════════════════════════════════════

V0：单 Agent 基线（第 1-3 周）
────────────────────────────────
状态：一个 Agent 完成所有任务——回答笔记问题、写技术文章、
      做代码审查、整理发布 checklist。

发现的问题：
第 2 周：Agent 写的技术方案中有一个明显的安全漏洞（明文密钥），
       用户自己发现后要求修正。Agent 的"自审"说"没有明显问题"。
        → 这是"创作者盲区"型故障。
        → 升级信号触发：V0 → V1

V1：引入 Reviewer（第 4-6 周）
────────────────────────────────
做法：Author Agent + Reviewer Agent。
     Reviewer 有独立上下文、独立工具（读文件+安全扫描）、
     结构化审查清单（5 项安全检查）。

第 4 周：第一个方案被 Reviewer reject（3/5 检查项 FAIL）。
        Author 修正后第二轮通过。用户确认方案质量明显提升。
        ✓ V1 验证成功。

第 5 周：Reviewer 连续 8 个方案都是 approved。检查后发现 Review
        清单中的"检查是否提及安全措施"过于模糊，Reviewer 在"看到
        方案中提到安全"就算 PASS。修正清单为"验证型"。
        ✓ 这是 6.3.4 描述的"边界四"，通过改进审查清单修复。

第 6 周：用户开始要求"帮我调研三个技术方向，每个写一份总结"。
        单 Agent 串行处理，三个方向耗时 8 分钟。用户说"太慢了"。
        → 升级信号触发：V1 → V2

V2：引入 Supervisor（第 7-10 周）
────────────────────────────────
做法：Supervisor + 3 个 Researcher Worker。
     预定义拆解模板：每个 Worker 负责一个方向，exclude 交叉内容，
     统一输出格式（发现 + 失败模式 + 来源）。

第 7 周：第一次运行，三个 Worker 并行返回，耗时从 8 分钟降到 3 分钟。
        ✓ V2 验证成功。

第 8 周：翻车——"三个 Worker 跑完了，Supervisor 花了比串行更长的
        时间来合并"（见 6.4.3）。根因：一个 Worker 没按模板输出。
        修正：输出模板从"建议"改为"强制"，格式不符自动要求重生成。
        
第 9 周：稳定运行。用户开始要求"不仅要调研，还要从技术可行性和
        成本两个角度对比分析"。
        → 这不是 Supervisor 能解决的。需要同一份调研从两个维度分析。
        → 升级信号触发：V2 → V3

V3：引入 Parallel Specialists（第 11-13 周）
────────────────────────────────
做法：为调研结果增加两个分析维度——技术可行性分析、成本分析。
     每个维度有独立的 focus 和 exclude 定义。

第 11 周：运行后发现"技术可行性"和"成本分析" 40% 内容重叠——
        因为两个维度都在讨论"这个方案落地需要多少资源"。
        修正：重新定义维度边界——
        "技术可行性"只关心"技术上能不能做"，
        "成本分析"只关心"做出来花多少钱"。

第 12 周：稳定运行。团队开始依赖这个系统的输出做决策。
        出现了两次"方案被采纳后发现重大缺陷"的故障——
        但缺陷属于"风险判断"类，不是技术分析类。
        用户问："能不能让两个 Agent 从不同立场辩论？"
        → 升级信号触发：V3 → V4

V4：引入 Debate（第 14 周，试验性）
────────────────────────────────
做法：两个 Agent 从"激进采纳"和"保守否决"两个立场论证方案风险。

第 14 周：第一次 Debate 消耗了 12K tokens，产出了有价值的对立分析。
        但第二次 Debate 变成了文字游戏——两个 Agent 开始讨论
        "风险"的定义是什么。

决定：Debate 只用于高风险决策场景（涉及数据安全、大额支出）。
     其他场景保持在 V3。
     ✓ 没有升级到 V5——流程还不够稳定，不值得固化为 Graph。
```

#### 6.9.3 每个阶段的"最该做的一件事"和"最常见的坑"

| 阶段 | 最该做的一件事 | 最常见的坑 |
|---|---|---|
| V0→V1 | 定义可逐条核对的审查清单（含 verify 方法） | 审查清单用"是否提及"替代"是否实施" |
| V1→V2 | 为 Worker 定义统一输出模板（含字段和长度限制） | 合并成本吃掉并行收益 |
| V2→V3 | 定义每个维度的 focus 和 exclude 范围 | 维度不互斥导致 40% 内容重叠 |
| V3→V4 | 明确 Debate 的具体议题和停止条件 | Debate 变成文字游戏，消耗 token 无产出 |
| V4→V5 | 等流程稳定运行至少一个月再固化 | 流程还在变化就固化，每次调整改图成本高 |

### 6.10 什么时候不该用 Multi-Agent

讲了这么多 Multi-Agent 怎么设计和迭代，最后要讲一个最关键的问题：**什么时候不该用。**

#### 6.10.1 决策流程图

```text
你的场景需要 Multi-Agent 吗？
│
├─ 问题一：是否存在"角色冲突"？
│   即两个角色目标天然矛盾，且共享上下文会导致互相妥协
│   ├─ 是 → 考虑 Reviewer 模式
│   └─ 否 → 继续
│
├─ 问题二：是否存在"并行瓶颈"？
│   即多个子任务互不依赖但被串行执行，且串行耗时用户可感知
│   ├─ 是 → 考虑 Supervisor 模式
│   └─ 否 → 继续
│
├─ 问题三：是否存在"视角盲区"？
│   即单一视角无法覆盖所有关键关注点，且遗漏已导致实际损失
│   ├─ 是 → 考虑 Parallel Specialists
│   └─ 否 → 继续
│
├─ 问题四：这些问题能用更简单的方法解决吗？
│   - 更好的 Prompt 设计？
│   - 更清晰的工具权限分离？
│   - 预定义的检查清单（不需要 LLM 执行）？
│   - 更精细的上下文管理（哪些信息应该/不应该进上下文）？
│   ├─ 能 → 先用简单方法，不要引入 Multi-Agent
│   └─ 不能 → 从 Reviewer 开始，两个 Agent 足矣
│
└─ 关键自问：如果你说不出两个 Agent "看到的东西有什么不同"，
   那就不需要两个 Agent。
```

#### 6.10.2 引入前的自查清单

逐条检查，全部通过后再动手：

```text
□ 1. 我能否用一句话描述每个 Agent 的"信息差"？
     （Agent A 看到 X 而 Agent B 看不到 Y；Agent B 有工具 Z 而 Agent A 没有）
     如果不能 → 不需要 Multi-Agent。

□ 2. 我能否为每个 Agent 定义不同的验收标准？
     （不是"都完成用户任务"，而是 Author 的"方案覆盖需求"vs Reviewer 的"清单逐条通过"）
     如果验收标准相同 → 不需要 Multi-Agent。

□ 3. 单 Agent 在当前任务上的失败是可复现的吗？
     （不是"偶尔一次没审出来"，而是"结构性审不出来"——因为自审机制问题）
     如果不是结构性的 → 先改进 Prompt，不要加 Agent。

□ 4. 我是否已经实现了单 Agent 的基线并测量了性能？
     （耗时、成功率、常见失败模式）
     如果没有基线 → 先建立基线，否则无法判断 Multi-Agent 是否真的改善了。

□ 5. 额外成本（token、延迟、维护复杂度）是否可以被预期收益覆盖？
     （预期收益 = 发现更多错误？减少用户等待时间？提升决策质量？）
     如果说不清预期收益 → 不要引入。

□ 6. 我是否准备好调试 Multi-Agent 的 trace？
     （每个 Agent 的输入输出是否被独立记录？是否能交叉对照？）
     如果没有 trace 基础设施 → 先搭建，再引入 Multi-Agent。

□ 7. 用户或业务是否确实在承受单 Agent 的限制？
     （不是"未来可能需要"，而是"现在已经在痛"）
     如果只是"未来可能需要" → 等到真的痛了再说。
```

#### 6.10.3 常见误判场景

以下是"看起来需要 Multi-Agent，但实际上不需要"的典型场景：

| 误判场景 | 为什么看起来需要 Multi-Agent | 实际需要的是什么 |
|---|---|---|
| "我需要 Agent 既写代码又写文档" | 看起来是两个角色 | 同一个 Agent 分两步执行：先写代码，再基于代码写文档。这不是角色冲突——写文档不需要"审查"代码 |
| "我的 Agent 输出质量不稳定，加个审查 Agent 把关" | 看起来需要 Reviewer | 先检查 Prompt 是否清晰、工具返回是否稳定、上下文是否有污染。90% 的"质量不稳定"是这些原因 |
| "任务很复杂，拆成多个子任务让多个 Agent 做" | 看起来需要 Supervisor | 先检查是否能用 Planning（课程四）的 Plan-Execute 模式串行完成。复杂不等于需要并行 |
| "我想用最新的 Multi-Agent 框架" | 框架看起来很强大 | 框架是工具，不是需求。先有需要 Multi-Agent 解决的问题，再选框架。而不是先选框架再找问题 |
| "竞品用了 Multi-Agent，我们也得有" | 竞争压力 | 竞品可能也在后悔。用"四个不同"标准检验你的场景——不是检验竞品的场景 |

**最后一条实用判断**：

```text
如果问题是"角色冲突、并行瓶颈、视角盲区" → 考虑 Multi-Agent。
如果问题是"工具不好用、上下文混乱、Prompt 不清晰、缺少测试" → 
先修课程三、四、五的问题，不要引入 Multi-Agent。

Multi-Agent 是五类增强能力中复杂度最高、代价最大的。
它应该是你最后考虑的选项，不是第一个。
```

### 本章速记

Multi-Agent 的本质不是"让更多模型一起干活"，而是**通过独立上下文、工具分离和结构化通信，让不同 Agent 基于不同信息做独立判断**。判断是否需要 Multi-Agent 的标准很简单：**如果你说不出两个 Agent "看到的东西有什么不同"，那就不需要两个 Agent。**

六个贯穿全章的主线：

**单 Agent 的三类天花板是结构性的，不是能力问题。** 角色冲突（创作者和审查者共享上下文导致互相妥协）、上下文挤压（中间推理污染最终判断）、串行瓶颈（互不依赖的任务排队执行）——换更强的模型解决不了这些结构性缺陷。Multi-Agent 通过多实例、多上下文、多工具集来突破这些结构限制。

**拆分的是四个维度，不是角色名称。** 真正的 Multi-Agent 拆分的是输入（看到的信息不同）、工具（能做的事不同）、目标（要达成的结果不同）、验收标准（怎么算完成不同）。给同一个 Agent 换三套 System Prompt 轮流扮演——这不叫 Multi-Agent，这叫角色扮演。至少有两个维度不同，Multi-Agent 才有存在的价值。

**从 Reviewer 开始——一个执行，一个审查。** Reviewer 是 Multi-Agent 的"Hello World"：两个 Agent，最简通信，价值最容易评测。核心机制不是"多一个人看"，而是 Reviewer 的独立上下文——它看不到 Executor 的中间推理和妥协，因此不会先入为主。审查清单必须是具体的、可逐条核对的、有明确 verify 方法的。"检查方案是否安全"不叫审查清单，"grep 配置文件中是否存在 key= 字面量"才叫。

**Agent 之间的通信必须结构化。** 自由对话（"你觉得怎么样？""我觉得还行"）是 Multi-Agent 中最昂贵、最难调试的通信方式——信息在传递中衰减、意图在解读中扭曲、决策在模糊中被搁置。结构化通信（verdict + issues 列表、子任务模板、维度 + 来源标注）让每个 Agent 清楚知道"我需要什么"和"我应该给什么"。审查结果不接受"整体还行"，必须是具体的、附带证据的结论。

**裁决、停止、兜底——控制机制必须硬编码。** 没有裁决机制的 Multi-Agent 是"讨论组"不是"决策系统"。Reviewer 两轮修正未通过→人工裁决。Worker 超时→标注缺失而非假装完成。并行收益被合并成本吃掉→检查输出模板是否统一。这些规则不能由 Agent 自己决定——Agent 没有"该停了"的直觉。

**增加 Agent 的成本必须能被其创造的价值覆盖。** Multi-Agent 的 token 消耗是单 Agent 的 2-5 倍，延迟通常翻倍，长期维护复杂度增长 3 倍以上。在引入 Multi-Agent 之前，先问：这个问题能用更好的 Prompt、更清晰的工具权限、更精细的上下文管理解决吗？如果不能，再从 Reviewer 开始——两个 Agent 足矣。Multi-Agent 是五类增强能力中最后考虑的选项，不是第一个。
