## 9.1 不要一次性堆满能力

1.1 里的知识助手 Agent，最终集齐了七种能力。但这不是一次加完的——它是在几个月的使用中，一个问题接一个问题地被逼出来的。

真实项目中，增强能力确实经常组合出现。个人知识助手需要 RAG + Memory；代码审查 Agent 需要 Tool Use + Reflection + Reviewer。但组合不等于一次性在设计文档里画完架构图。

更好的顺序是——让问题驱动引入节奏：

```text
先让最小闭环跑起来
  -> 观察第一个明确的问题模式
  -> 用最小版本的能力修复它
  -> 评测是否真的变好了
  -> 再观察下一个瓶颈
  -> 循环
```

不要因为某个能力在社区里很热门就加入系统。每个能力进入系统之前，都应该有问题证据和评测方式。如果你说不出"没有 RAG 时用户具体在哪一步遇到了什么问题"，那 RAG 就不该是你的第一个优先。

### 9.1.1 反面教材：大爆炸引入的灾难

来看一个真实风格的场景。某团队决定做一个"全能型代码助手 Agent"，在一个 sprint 内同时引入了七种能力：

```
Sprint 目标：构建 SuperCodeAgent v1.0

第一周：
- 接入 RAG：索引公司内部全部代码仓库文档
- 接入 Memory：保存用户所有编码习惯和偏好
- 接入 Planning：支持多步骤重构计划自动拆解

第二周：
- 接入 Reflection：代码生成后自动审查并修正
- 接入 Multi-Agent：一个写代码，一个审代码，一个写测试

Sprint 结束后上线，结果：
```

**第一天的用户反馈：**

| 问题 | 根因 | 用户感受 |
|------|------|----------|
| Agent 回复太慢（平均 25 秒） | 五个能力叠加导致单次请求链路极长 | "还不如我自己写快" |
| 检索结果经常不相关 | RAG 索引质量没来得及打磨 | "搜出来的东西跟我问的没关系" |
| Memory 记了很多无用偏好 | 没有做记忆筛选策略 | "它记了我临时测试的配置，正式用全乱了" |
| Plan 步骤经常跑偏 | Planning 没有配合场景做约束 | "拆了 10 步，第 3 步就开始做无关的事" |
| 两个 Agent 互相矛盾 | Multi-Agent 没有定义冲突解决规则 | "写代码的说没问题，审代码的说要重写" |

**核心教训：** 不是这些能力不好，而是一次性引入导致：

1. **调试困境**：当问题出现时，你无法快速定位是哪个能力导致的
2. **评测困难**：你没有一个"引入前"的基线来对比
3. **用户信任透支**：第一次体验差，用户可能不再给你第二次机会
4. **团队认知过载**：维护者需要同时理解五个复杂子系统

> **记住一句话：一个能力，一个问题，一次评测。三个"一"原则是引入节奏的底线。**

### 9.1.2 能力的隐藏代价

每个能力在解决问题时，也悄悄带来代价。在决定引入前，先看清账单：

| 能力 | 解决什么问题 | 带来的代价 |
|------|-------------|-----------|
| **RAG / 外部知识接入** | 模型不知道私有/实时信息 | 索引维护、检索延迟（+500ms-2s）、引用校验、文档更新同步 |
| **Memory** | 跨会话/跨轮次状态延续 | 存储膨胀、遗忘策略调参、隐私合规、错误记忆的持久化 |
| **Context Engineering** | 多信息源导致上下文混乱 | 分层设计成本、Token 预算需要持续调参、过度压缩导致信息失真 |
| **Planning / Workflow** | 多步骤任务组织 | Prompt 长度暴增、步骤漂移、错误级联、打断恢复 |
| **Reflection** | 自动发现和修正错误 | 多轮调用成本（×2-3 token）、无休止自循环、过度修正 |
| **Human-in-the-loop** | 高风险操作需要人类确认 | 阻塞等待、UX 中断感、确认疲劳、流程延迟 |
| **Multi-Agent** | 角色冲突、并行执行 | 协调开销、Agent 间冲突裁决、Trace 复杂度倍增、成本线性增长 |
| **Tool Use** | 与外部世界交互 | 工具调用失败处理、超时控制、权限安全、结果校验 |

**代价放大效应：** 当多个能力组合时，代价不是简单相加，而是会出现交互放大：

```text
单能力延迟：
  RAG 检索：800ms
  Planning 拆解：1.2s
  Reflection 修正：+1 轮 LLM 调用（1.5s）
  Multi-Agent 协调：+500ms

组合后延迟：
  RAG + Planning + Reflection + Multi-Agent
  = 800ms + 1.2s + (1.5s × Agent数) + 500ms + 协调等待
  ≈ 5-8 秒（用户感知的"卡顿"）
```

这就是为什么能力引入需要克制——每加一层，用户的等待时间、你的调试时间、系统的出错概率都在增长。

### 9.1.3 能力复杂度阶梯

与其把能力当成"有/没有"的二元开关，不如先看清它们带来的复杂度。下面这张图不是所有 Agent 都必须遵循的升级路线，而是一个**代价递增参考**：越往上，系统需要承担的延迟、调试、状态一致性和评测成本越高。

```text
                        ┌─────────────────────────┐
                        │  高复杂度：自适应协同     │
                        │  多 Agent 根据任务自动    │
                        │  组队、分工、互审         │
                        ├─────────────────────────┤
                        │  高复杂度：决策边界       │
                        │  具备 Human-in-the-loop   │
                        │  高风险操作人类确认       │
                        ├─────────────────────────┤
                        │  中高复杂度：自我修正     │
                        │  单 Agent 具备 Reflection │
                        │  检测失败并自动重试/修正  │
                        ├─────────────────────────┤
                        │  中高复杂度：复杂任务编排 │
                        │  单 Agent 具备 Planning   │
                        │  多步骤任务自动拆分执行   │
                        ├─────────────────────────┤
                        │  中复杂度：上下文管理     │
                        │  具备 Context Engineering │
                        │  分层、预算、优先级管理   │
                        ├─────────────────────────┤
                        │  中复杂度：状态感知       │
                        │  单 Agent 具备 RAG+Memory │
                        │  能访问外部知识、记住偏好 │
                        ├─────────────────────────┤
                        │  基线：最小闭环           │
                        │  LLM + Tool Use + Loop    │
                        │  处理简单、独立的任务     │
                        └─────────────────────────┘
```

**使用这个阶梯的正确方式：**

- **不要把它当路线图**：不是先 RAG、再 Memory、再 Planning、最后 Multi-Agent。真实顺序由问题决定。
- **每加一层都要有证据**：确认当前问题无法被更简单的方式解决，再引入更复杂的能力。
- **允许停在低复杂度组合**：不是所有场景都需要爬到高复杂度。很多场景停在 RAG + 轻量 Memory 就是最优解。
- **可以回退**：如果高复杂度能力带来更多问题，降回更简单的组合是成熟的决定。

**自查问题：**

- 当前场景的痛点对应哪类能力？
- 这个能力解决的问题是否有数据或用户反馈支撑？
- 已有能力的评测指标是否稳定？
- 新增复杂度、延迟和维护成本是否值得？

### 9.1.4 问题驱动的正确节奏

下面是知识助手 Agent 在三个月内的真实引入节奏（经过简化）：

```text
┌─────────────────────────────────────────────────────────┐
│  第 0 周：最小闭环上线                                    │
│  - LLM + 基础对话                                         │
│  - 用户手动粘贴资料内容                                    │
│  - 核心指标：回答是否合理（人工判断）                       │
├─────────────────────────────────────────────────────────┤
│  第 2 周：问题发现 — RAG 引入                              │
│  用户反馈："每次都要手动粘贴笔记，太麻烦了"                  │
│  问自己："这个问题有多频繁？" → 70% 的对话都需要查资料       │
│  引入：Markdown 文档索引 + 向量检索 + 引用输出              │
│  评测：20 个固定问题，检查召回率、答案准确性、引用正确性       │
│  结果：召回率 78% → 优化分块策略后 → 89%                   │
├─────────────────────────────────────────────────────────┤
│  第 6 周：问题发现 — Memory 引入                           │
│  用户反馈："每次开启新对话，它都忘了我在研究什么"            │
│  问自己："跨会话上下文对体验影响多大？" → 用户平均 3-5 次会话 │
│  引入：会话摘要 + 关键信息持久化                            │
│  评测：跨会话追问 10 个场景，检查上下文衔接准确性            │
│  结果：衔接准确率 82% → 调整摘要策略后 → 91%               │
├─────────────────────────────────────────────────────────┤
│  第 8 周：问题发现 — Context Engineering 引入              │
│  用户反馈："接了 RAG 和 Memory 后，上下文经常超限，         │
│           偶尔会忽略我之前设的规则"                           │
│  问自己："几个信息源了？组织方式合理吗？" → 3 个信息源       │
│  引入：上下文分层 + Token 预算 + 工具输出压缩                │
│  评测：10 个约束遗忘测试场景，检查约束遵守率                  │
│  结果：约束遵守率从 72% 提升到 94%                          │
├─────────────────────────────────────────────────────────┤
│  第 12 周：问题发现 — Planning 引入                        │
│  用户反馈："让它帮我整理一周的笔记，它经常漏掉某些文件夹"      │
│  问自己："任务是线性的还是需要动态规划？" → 需要分支处理     │
│  引入：简单的 ReAct 模式，允许 Agent 自己决定下一步           │
│  评测：10 个多步骤整理任务，检查步骤覆盖率和遗漏率            │
│  结果：遗漏率从 20% 降到 5%                                │
├─────────────────────────────────────────────────────────┤
│  第 16 周：问题发现 — HITL 引入                            │
│  用户反馈："它删了一个我不确定该不该删的文件，没问我"        │
│  问自己："哪些操作风险高到必须确认？" → 文件删除、配置修改   │
│  引入：风险分级 + 确认模式 + 批次确认                        │
│  评测：高风险操作确认率 100%，用户误确认率 < 5%             │
│  结果：无意外文件删除事件，用户信任度提升                    │
├─────────────────────────────────────────────────────────┤
│  第 18 周：暂不引入任何新能力                               │
│  原因：当前五个能力稳定，无明确问题信号                       │
│  Reflection 和 Multi-Agent 留到有证据时再说                  │
└─────────────────────────────────────────────────────────┘
```

**关键观察：**

1. **每个能力的引入都有具体的问题触发**，不是因为"别人都在用"
2. **每次引入都有量化的评测**，不是凭感觉判断"好像变好了"
3. **两次引入之间有足够的观察期**（2-4 周），等问题的模式真正清晰
4. **第 18 周选择了"不引入"**，这是最容易被忽略但同样重要的决策

### 9.1.5 什么是"最小闭环"

在讨论能力组合之前，先搞清楚什么是"最小闭环"——因为你引入任何能力之前，先得有一个能跑的东西。

```text
最小闭环 = LLM 决策 + 工具调用 + 循环控制 + 状态管理
```

**最小闭环能做什么：**

- 接收用户指令
- 调用工具获取信息/执行操作
- 基于结果做出下一步决策
- 循环直到任务完成
- 返回最终结果

**最小闭环的边界：**

```text
✅ 属于最小闭环：
  - "帮我搜索这个文件夹里的所有 .md 文件"
  - "把这段代码里的 foo 替换成 bar"
  - "读这个 CSV 文件，统计每个分类的数量"

❌ 需要增强能力：
  - "根据我的笔记回答这个问题" → 需要 RAG
  - "记住我上次说的偏好" → 需要 Memory
  - "信息源多了，上下文经常乱" → 需要 Context Engineering
  - "把这个需求拆成子任务逐一完成" → 需要 Planning
  - "检查你生成的代码有没有 bug" → 需要 Reflection
  - "这个操作太重要，不能让它自己决定" → 需要 Human-in-the-loop
  - "找人审查你的方案" → 需要 Multi-Agent
```

**最小闭环作为基线的重要性：**

在引入任何增强能力前，先用最小闭环回答这些问题：

1. **这个任务最小闭环能做吗？** → 如果能，为什么还要加能力？
2. **最小闭环做的时候，具体卡在哪里？** → 这是引入新能力的唯一合法理由
3. **新能力引入后，和最小闭环相比好了多少？** → 这是评测的基线

## 9.2 能力组合案例

以下六个案例覆盖了最常见的 Agent 场景。每个案例都从"起始问题"出发，依次引入能力，并且明确标注"暂不引入"——后者和前者同样重要。

#### 案例一：个人知识助手

**场景描述：** 用户有大量本地笔记（Markdown、PDF、网页剪藏），希望 Agent 能基于这些资料回答问题，并给出引用来源。

**起始问题：**

- 用户资料很多，模型不知道资料内容。
- 用户需要引用来源。

**优先能力：**

- **RAG / 外部知识接入**。这是核心能力——没有它，Agent 只能给出通用回答，无法满足"基于我的笔记"这个核心需求。

**最小版本实现：**

```text
文档目录 → 文档解析（Markdown/PDF → 纯文本）
         → 文本切分（按段落/标题，每块 500-1000 tokens）
         → 向量化（embedding model）
         → 存入向量数据库（Chroma / Milvus / Pinecone）

用户询问 → 查询向量化 → 检索 Top-K（K=5-10）
        → 拼接检索结果到 Prompt
        → LLM 生成回答（附带引用标记）
```

**评测维度：**

- 召回率：检索是否找到了相关文档块
- 答案准确性：生成的回答是否正确使用了检索到的信息
- 引用正确性：引用标记是否指向了正确的来源
- 拒答率：当资料中没有相关信息时，Agent 是否诚实表示不知道

**后续问题：**

- 用户希望记住当前研究主题，下次对话可以继续。
- 多轮追问时需要延续上下文。
- 同一主题的多次问答之间缺乏关联。

**再引入：**

- **会话摘要和轻量 Memory**。注意这里不是引入"完整的长期记忆系统"，而是针对"记住当前研究主题"这个具体问题的轻量解决方案。

**Memory 最小版本：**

```text
每次会话结束：
  → 用 LLM 生成会话摘要（研究主题、关键发现、待解决问题）
  → 持久化存储（JSON 文件或轻量数据库）

下次用户开启新会话：
  → 加载最近的会话摘要
  → 附加到系统 Prompt 中作为上下文
  → 用户可以随时要求"忘记之前的上下文"
```

**暂不引入：**

- **Multi-Agent**，除非出现明确的并行调研或审查需求。如果用户只是一个人使用，单 Agent 完全够用。
- **Reflection**，除非检索结果质量出现明显的系统性问题。RAG 的检索质量问题应优先通过优化分块策略、调整检索参数来解决，而不是引入 Reflection 让它"自己检查自己"。
- **Planning**，除非用户的任务变得明显多步骤化（如"帮我在三个资料源之间做交叉验证"）。

**引入路径总览：**

```text
最小闭环
  → RAG（第 2 周）
    → 检索质量迭代（第 3-5 周）
      → Memory（第 6 周）
        → 稳定运行（第 7-14 周）
          → Planning？（等待问题信号）
```

#### 案例二：代码审查 Agent

**场景描述：** 团队需要一个 Agent 来审查 Pull Request，不仅看 diff 表面，还能深入理解代码上下文，执行测试验证判断。

**起始问题：**

- Agent 只读 diff 后输出泛泛建议（如"考虑添加注释"、"检查空指针"）。
- 不能验证自己的判断——说"这里可能有性能问题"但无法实际验证。
- 建议缺乏上下文——不了解这个函数的调用方和被调用方。

**优先能力：**

- **工具调用：读文件、搜索代码、运行测试**。这是从"表面审查"变成"深入审查"的关键。
- **Reflection：测试失败后修正判断**。当工具调用结果与初始判断矛盾时，Agent 需要修正自己的结论。

**最小版本实现：**

```text
接收到 PR diff
  → 分析 diff 中的变更
  → 对每个可疑点，调用工具：
      - read_file: 读取相关文件的完整内容
      - search_code: 搜索函数/变量的所有引用位置
      - run_test: 运行相关测试
  → 基于工具结果形成审查意见
  → 如果测试失败，Reflection：
      - 分析失败原因
      - 确认是自己的判断有误还是代码确实有问题
      - 修正审查意见
  → 输出最终审查报告
```

**关键设计决策：为什么先引入 Reflection 而不是 Multi-Agent？**

对于代码审查场景，很容易第一反应就是"引入 Multi-Agent，一个写一个审"。但如果审查 Agent 没有工具调用能力，另一个 Agent 也只是"再读一遍 diff 给出泛泛建议"——两个盲人并不能一起看见。

正确的顺序是：**先让单个 Agent 通过工具调用"看见"，再通过 Reflection "修正"，最后才考虑 Multi-Agent 做分工。**

**后续问题：**

- Agent 审查到一定阶段后，需要对修改后的代码进行二次审查（审查修改后的代码是否引入了新问题）。
- 单一 Agent 同时扮演"审查者"和"修改建议者"的角色存在冲突——它可能对自己的修改建议过于宽容。

**再引入：**

- **Reviewer 模式的轻量 Multi-Agent**：一个 Agent 负责审查和建议，另一个 Agent 负责审查修改后的代码。注意这不是完整的 Multi-Agent 系统，只是"Reviewer 模式"——两个 Agent 串行工作，不涉及复杂的协调。

**Multi-Agent 最小版本：**

```text
Reviewer Agent（审查者）：
  输入：PR diff
  输出：审查意见列表
  工具：读文件、搜索代码、运行测试

Verifier Agent（验证者）：
  输入：修改后的代码 + Reviewer 的审查意见
  输出：验证报告
  行为：逐条检查 Reviewer 的建议是否被正确实现
       检查修改是否引入了新问题

冲突解决规则（简单版）：
  - 如果 Reviewer 和 Verifier 意见一致 → 采纳
  - 如果意见不一致 → 标记为"需人工审查"
  - 禁止两个 Agent 互相调用形成循环
```

**暂不引入：**

- **长期 Memory**，除非需要记住项目约定、测试习惯或用户偏好。如果项目规范和测试策略已经在项目文件中定义好了，Memory 的边际收益很低。
- **Planning**，除非审查任务变得明显多步骤化（如"先审查安全性，再审查性能，最后审查可读性"）。

**引入路径总览：**

```text
最小闭环
  → Tool Use（有工具才能深入分析）
    → Reflection（有外部信号才能修正）
      → Multi-Agent Reviewer（角色冲突时才分工）
        → 稳定运行
          → Memory？（等待需求证据）
```

#### 案例三：个人任务执行 Agent

**场景描述：** 用户希望 Agent 能帮忙执行复杂的多步骤任务（如"为这个开源项目配置 CI/CD"），在关键节点暂停确认，支持中断后继续。

**起始问题：**

- 任务多步骤，容易漏步骤。
- 用户希望在关键节点确认（如在"创建 GitHub Secret"之前确认）。
- 手动追踪任务进度很繁琐。

**优先能力：**

- **Planning / Workflow Patterns**。多步骤任务天然需要结构化组织。
- **Human-in-the-loop**。关键操作需要人工确认，这是安全需求而非体验优化。

**最小版本实现：**

```text
用户任务："为仓库配置 CI/CD"
  → Planning Agent 拆解：
      Step 1: 分析项目结构，确定 CI/CD 方案
      Step 2: 编写 CI 配置文件
      Step 3: 生成需要的 Secrets 列表
      Step 4: [HITL] 等待用户确认 Secrets 列表
      Step 5: 配置 GitHub Secrets
      Step 6: 触发首次 CI 运行
      Step 7: [HITL] 检查运行结果，等待用户确认
  → 每个步骤执行完后更新状态
  → 用户可随时查看进度
```

**Human-in-the-loop 设计要点：**

```text
✅ 好的 HITL 设计：
  - 在不可逆操作前暂停（删除、发布、权限变更）
  - 暂停时给出明确的选项："确认执行 / 跳过 / 修改参数"
  - 超时后的默认行为是"不执行"（safe default）

❌ 不好的 HITL 设计：
  - 每一步都暂停（确认疲劳）
  - 暂停时信息不足，用户无法判断
  - 超时后自动执行高危操作
```

**后续问题：**

- 长任务中断后需要继续。
- 任务执行历史需要可追溯。

**再引入：**

- **任务状态和 Checkpoint**，这部分会在课程六深入。核心思路是将任务状态持久化，支持从中断点恢复。

**Checkpoint 最小版本：**

```text
任务状态结构：
{
  "task_id": "xxx",
  "goal": "为仓库配置 CI/CD",
  "steps": [
    {"id": 1, "desc": "分析项目结构", "status": "done"},
    {"id": 2, "desc": "编写 CI 配置", "status": "done"},
    {"id": 3, "desc": "生成 Secrets 列表", "status": "in_progress"},
    ...
  ],
  "checkpoint_data": {
    "branch": "feature/ci-setup",
    "ci_file": ".github/workflows/ci.yml",
    ...
  },
  "last_updated": "2024-03-15T10:30:00Z"
}

恢复流程：
  用户重新连接 → 检测未完成任务 → 加载 Checkpoint → 从断点继续
```

**暂不引入：**

- **自动长期 Memory**，除非用户明确需要跨任务偏好延续。任务之间的独立性是特性而非缺陷。
- **RAG**，除非任务执行需要频繁查阅外部文档。
- **Multi-Agent**，单用户场景下不需要并行。

**引入路径总览：**

```text
最小闭环
  → Planning + HITL（任务结构化和安全需求）
    → Checkpoint（中断恢复需求）
      → 稳定运行
        → Memory？（跨任务偏好是否需要延续）
```

#### 案例四：智能客服 Agent

**场景描述：** 电商平台的智能客服，需要回答产品问题、处理退换货、查询订单状态。部分问题可以自动处理，敏感操作（如退款）需要人工确认，复杂问题需要转接人工客服。

**起始问题：**

- 用户咨询产品信息，Agent 需要基于产品库回答。
- 用户查询订单状态，Agent 需要访问订单系统。
- 退款/换货操作涉及资金和库存，不能自动执行。

**优先能力：**

- **RAG / 外部知识接入**：产品信息、政策文档、FAQ 等需要实时检索。
- **Tool Use**：调用订单查询 API、物流查询 API 等内部系统。
- **Human-in-the-loop**：退款、换货审批等敏感操作需要人工确认。

**为什么这三个能力同时引入？**

客服场景的特殊性在于：RAG、Tool Use、HITL 三者不是在解决"同一个问题的逐层深化"，而是在解决"三个独立且同时存在的硬需求"：

- 没有 RAG → 产品信息无法回答（核心功能缺失）
- 没有 Tool Use → 订单无法查询（核心功能缺失）
- 没有 HITL → 退款有资金风险（安全红线）

这属于"场景本身就需要多重能力才能形成最小可用产品"的情况。但即使如此，也建议分步验证——先确保 RAG 检索正确，再接入 Tool Use，最后加上 HITL。

**最小版本实现：**

```text
用户消息
  → 意图分类（产品咨询 / 订单查询 / 售后处理 / 闲聊）
  → 路由到对应处理流程：

产品咨询流程：
  → RAG 检索产品知识库
  → 拼接检索结果生成回答
  → 附带产品链接

订单查询流程：
  → 验证用户身份
  → 调用订单 API
  → 格式化订单信息返回

售后处理流程：
  → 查询订单状态
  → 判断是否符合退换货条件
  → [HITL] 如需退款 → 生成退款申请 → 等待人工审批
  → [HITL] 审批通过 → 执行退款 → 通知用户
```

**后续问题：**

- 用户多次联系客服，每次都要重新说明问题。
- 回头客的偏好和历史问题需要被记住。
- 复杂问题转接人工后，人工客服需要看到对话上下文。

**再引入：**

- **会话 Memory**：记住当前会话上下文和近期联系历史。注意是"会话级"而非"用户级长期档案"——前者解决重复说明问题，后者涉及隐私合规。

**Memory 设计（客服场景特化）：**

```text
会话级记忆（7 天有效）：
  - 本次会话的对话历史
  - 本次会话涉及的产品/订单
  - 本次会话的未解决问题

用户级记忆（需要用户授权）：
  - 常用收货地址
  - 最近 3 次购买记录
  - 语言偏好

记忆清理策略：
  - 会话级记忆：会话结束后 7 天自动清除
  - 用户级记忆：用户可随时查看和删除
  - 敏感信息（支付、密码）永不记录
```

**暂不引入：**

- **Planning**：客服对话通常是线性流程（if-else 路由），不需要动态规划
- **Reflection**：客服场景的反馈来自用户满意度评分，而非 Agent 自我修正
- **Multi-Agent**：除非需要同时处理售前和售后两种不同角色，但通常可以通过路由解决

**引入路径总览：**

```text
最小闭环
  → RAG + Tool Use + HITL（场景硬需求，但分步验证）
    → Memory（减少重复说明，注意隐私边界）
      → 稳定运行
        → 根据用户满意度数据决定下一步
```

#### 案例五：数据分析助手

**场景描述：** 数据分析师需要 Agent 帮忙写 SQL、解析数据、生成可视化报告。任务通常是多步骤的：理解需求 → 写查询 → 验证结果 → 解释发现 → 生成图表。

**起始问题：**

- 用户用自然语言描述分析需求，Agent 需要翻译成 SQL。
- SQL 执行后，Agent 需要检查结果是否符合预期。
- 如果结果不对，Agent 需要自行修正查询。

**优先能力：**

- **Tool Use**：执行 SQL、读取表结构、查询数据字典。
- **Reflection**：SQL 执行失败或结果异常时自动检查并修正。

**为什么先是 Tool Use + Reflection，而不是一上来就 Planning？**

对单表查询、一次性指标计算这类任务来说，用户的自然语言已经给出了主要分析目标——"帮我看看上个月的销售趋势，按地区拆分，找出增长最快的三个品类"。此时最先要解决的不是"会不会规划"，而是"能不能正确读取 Schema、生成 SQL、执行查询，并在错误信息出现时修正"。

但这不是说数据分析不需要 Planning。当任务升级为跨表探索、分阶段验证假设、先查 A 再基于结果查 B、最后合并可视化时，Planning 才有明确价值。顺序应该是：先把执行和验证闭环跑稳，再根据多步骤分析需求引入 Planning。

**最小版本实现：**

```text
用户需求 → Agent 分析：
  1. 读取数据库 Schema，了解表结构和字段
  2. 生成 SQL 查询
  3. 执行 SQL
  4. 检查结果：
     - 执行是否成功？→ 如果失败，Reflection：分析错误信息，修正 SQL
     - 结果是否合理？→ 如果行数为 0 或数值异常，Reflection：检查查询逻辑
     - 结果是否符合用户意图？→ 对结果做合理性检查
  5. 解释发现
  6. 可选：生成可视化代码（matplotlib / eCharts）
```

**Reflection 在数据分析场景的具体应用：**

```text
Reflection 触发条件：
  ✅ SQL 语法错误 → 读取错误信息，修正语法
  ✅ 结果集为空 → 检查 WHERE 条件是否过于严格
  ✅ 数值明显异常 → 检查 JOIN 是否正确
  ✅ 结果与常识矛盾 → 提醒用户可能存在数据质量问题

Reflection 停止条件：
  ❌ 修正 3 次后仍然失败 → 向用户说明情况，请求指导
  ❌ 不确定是否正确 → 标注"需要人工验证"
  ❌ 发现数据源本身有问题 → 报告问题，不继续尝试修复
```

**后续问题：**

- 复杂分析需要多步骤（先查 A 表，基于 A 的结果查 B 表，合并分析）。
- 用户希望 Agent 记住常用的分析模板和偏好指标。
- 跨多个数据源做分析时，需要有条理地组织步骤。

**再引入：**

- **Planning**：当分析任务从"单次查询"升级为"多步骤分析流程"时引入。
- **Memory**：记住用户的分析习惯（常用指标、偏好图表类型、命名规范）。

**暂不引入：**

- **RAG**：除非分析需要频繁参考外部方法论文档或行业标准
- **Multi-Agent**：单人分析场景，没有角色冲突

**引入路径总览：**

```text
最小闭环
  → Tool Use + Reflection（翻译+修正，分析场景核心）
    → Planning（多步骤分析需求明确后）
      → Memory（分析习惯复用需求明确后）
        → 稳定运行
```

#### 案例六：写作协作 Agent

**场景描述：** 用户和 Agent 协作写技术博客。Agent 需要理解用户的写作风格、记住之前讨论的要点、检索参考资料，并在每次修改后基于外部证据做质量检查。

**起始问题：**

- 用户希望 Agent 保持一致的写作风格。
- 用户需要 Agent 引用技术参考资料。
- Agent 写完一段后，用户希望它能检查问题（引用是否可追溯、代码示例是否可运行、术语是否符合约定、用户反馈是否已处理）。

**优先能力：**

- **RAG**：检索技术参考资料、相关文章、API 文档。
- **Memory**：记住用户的写作风格偏好、术语选择、结构习惯。

**为什么这个场景 RAG + Memory 同时引入？**

写作场景中，RAG 和 Memory 解决的是不同维度的问题：
- RAG 解决"写什么"——提供准确的技术内容
- Memory 解决"怎么写"——保持一致的风格和术语

两者互相独立但缺一不可。没有 RAG 的内容不准确，没有 Memory 的风格不连贯。

**最小版本实现：**

```text
写作协作流程：
  1. 用户提出写作主题
  2. Agent 检索参考资料（RAG）
  3. Agent 回忆用户风格偏好（Memory）
  4. Agent 生成初稿
  5. [HITL] 用户审阅，提出修改意见
  6. Agent 根据反馈修改
  7. Agent 质量检查：
     - 引用校验（检索原文，确认引用存在）
     - 代码示例测试（运行或静态检查）
     - 术语/风格清单检查（对照 Memory 中的明确偏好）
     - 用户反馈核对（逐条确认修改意见已处理）
  8. 输出终稿

Memory 存储结构（写作场景）：
{
  "style_preferences": {
    "tone": "专业但亲切",
    "sentence_length": "中短句为主",
    "code_style": "Python, type hints 必加",
    "preferred_terms": {
      "use": ["数据库", "查询"],
      "avoid": ["DB", "query"]
    }
  },
  "structural_habits": {
    "opening": "故事/问题引入",
    "body": "概念 → 原理 → 示例 → 注意事项",
    "closing": "总结 + 延伸阅读"
  },
  "common_mistakes_to_avoid": [
    "不要过度使用被动语态",
    "代码示例需要可运行",
    "技术术语首次出现需要解释"
  ]
}
```

**后续问题：**

- 修改多轮后，Agent 需要基于明确检查项验证文章质量（而非依赖用户逐条指出）。
- 长文章结构复杂，需要分节管理。

**再引入：**

- **Reflection**：只用于有外部反馈信号的质量检查——引用能否在原文中找到、代码示例能否运行、用户修改意见是否逐条覆盖、术语是否符合已确认的风格清单。
- **Planning**：长篇内容的结构规划和大纲管理。

**Reflection 在写作场景的设计：**

```text
触发信号：
  1. 引用校验失败：生成内容中的来源无法在 RAG 原文中找到
  2. 代码检查失败：代码示例无法运行、类型检查失败或输出不符合预期
  3. 术语清单不匹配：使用了用户明确要求避免的术语
  4. 用户反馈未覆盖：修改意见清单中仍有未处理项

修正规则：
  - 每完成一个章节后自动运行
  - 发现问题后只修正对应问题，不重写整篇
  - 同类问题修正超过 2 次仍失败 → 标记并请用户介入
  - 纯主观表达质量（"够不够好看"、"语气是否高级"）交给用户或 Reviewer 清单，不用模型自评冒充 Reflection
```

**暂不引入：**

- **Multi-Agent**：写作是创造性工作，一个 Agent 配合用户审阅已足够。引入"一个写一个审"的 Multi-Agent 可能导致风格不统一。
- **Human-in-the-loop** 的强化版：写作协作天然是 HITL 模式（用户每次审阅就是 HITL），不需要额外的系统级 HITL 机制。

**引入路径总览：**

```text
最小闭环
  → RAG + Memory（内容准确 + 风格一致）
    → Reflection（基于外部信号做质量检查）
      → Planning（长文结构管理）
        → 稳定运行
```

| 当前问题 | 问题信号（你能观察到的） | Ready 信号（你准备好了吗） | 优先考虑 | 不建议引入的信号 |
|----------|------------------------|--------------------------|---------|-----------------|
| 答案依赖外部资料、私有知识或实时数据 | 用户频繁粘贴资料；Agent 说"根据我的训练数据"；回答过时 | 你有稳定的文档源；你定义了评测集；你能接受 500ms-2s 额外延迟 | RAG / 外部知识接入 | 用户输入已包含全部信息；文档源不稳定或质量差 |
| 需要跨轮、跨会话或跨任务延续 | 用户反复说"我刚才说过"；用户抱怨"又要重新来"；多轮后上下文丢失 | 你能定义什么该记什么不该记；你有遗忘/清理策略；隐私合规已确认 | Memory | 每次任务独立；用户更看重隐私；记忆质量无法保证 |
| 多信息源导致上下文混乱，关键约束被淹没 | 模型违反 System Prompt 中的约束；工具输出引导模型跑偏；上下文长度频繁超限 | 你能列出所有信息源；你能为每类信息定义优先级；你愿意持续调参预算 | Context Engineering | 信息源不超过 2 个；上下文远低于模型上限；单步简单任务 |
| 任务步骤多（>5步）、依赖强、容易漂移 | Agent 中途跑偏；遗漏关键步骤；执行顺序混乱；用户需频繁纠正 | 你能定义任务的步骤结构；任务确实有依赖关系；两三步的固定流程不需要 | Planning / Workflow Patterns | 任务步骤固定且已知；两步就能完成；用户自己拆解更高效 |
| 有明确失败信号，需要自动修正 | 同类错误反复出现；错误可被程序检测（语法错误、类型不匹配、测试失败）；用户频繁做同一类修正 | 你有明确的验证方式（测试、lint、API 返回码）；你有停止条件；你能评测修正后是否更好 | Reflection | 没有外部反馈（只让模型自评）；没有停止条件会导致无限循环；错误是主观的（如风格偏好） |
| 单 Agent 角色冲突、需要并行或互审 | Agent 同时做"执行"和"审查"；用户不信任单 Agent 的判断；任务可以独立并行 | 你能定义角色的边界；你有冲突解决规则；你愿意承担额外的协调成本和延迟 | Multi-Agent | 单 Agent 已可完成；缺少 Trace 和评测（你不知道哪个 Agent 在做什么）；角色边界模糊 |
| 需要与外部系统交互（数据库、API、文件系统） | Agent 说"我无法访问"；用户需要手动执行中间步骤；数据需要在多个系统间流转 | 你能定义工具的输入输出；你有错误处理策略；你有权限和安全控制 | Tool Use | 所有信息可通过对话传递；工具调用有安全隐患；外部系统不稳定 |
| 回答需要合规审查或安全边界 | Agent 生成了不安全的代码；Agent 给出了有法律风险的建议；内容需要符合公司政策 | 你有明确的规则清单；规则可以被程序化检查；违规的后果你愿意承担 | Guardrails / 安全护栏 | 规则模糊无法形式化；审查成本高于风险本身；过度限制导致 Agent 无法正常工作 |
| 关键操作不可逆，需要人工确认 | Agent 执行了删除/发布/支付操作；用户说"我只是想看看，没想到它真做了" | 你能定义"不可逆操作"的清单；你有确认 UI；默认行为是"不执行" | Human-in-the-loop | 每一步都需确认（导致确认疲劳）；操作可安全回滚；延迟敏感的场景 |

### 9.3.2 能力引入决策流程图

当你面对一个具体问题时，按以下流程判断：

```text
问题出现
  │
  ├── 这个问题能用最小闭环解决吗？
  │     ├── 能 → 不引入新能力，优化现有实现
  │     └── 不能 → 继续
  │
  ├── 这个问题反复出现吗？（频率 > 20% 的用户交互）
  │     ├── 否 → 记录问题，等待模式清晰
  │     └── 是 → 继续
  │
  ├── 这个问题有明确的"变好"的评测标准吗？
  │     ├── 否 → 先定义评测标准，再考虑引入
  │     └── 是 → 继续
  │
  ├── 匹配决策表：这个问题对应哪个能力？
  │     ├── 匹配到 1 个 → 继续
  │     ├── 匹配到多个 → 选择最轻量、最独立的能力先引入
  │     └── 匹配不到 → 可能不需要增强能力，优化 Prompt 或工具配置
  │
  ├── 引入该能力的最小版本
  │     ├── 定义最小实现（能验证效果的最简版本）
  │     ├── 定义停止条件（什么情况下认为引入失败）
  │     └── 定义评测方式
  │
  ├── 评测：引入后是否真的变好了？
  │     ├── 是 → 记录基线，进入观察期
  │     └── 否 → 回退，重新分析问题
  │
  └── 观察期（至少 2 周）
        ├── 新问题出现 → 回到流程开头
        └── 系统稳定 → 考虑是否需要迭代当前能力
```

### 9.3.3 常见错误判断对照表

| 常见想法 | 为什么可能是错的 | 正确做法 |
|---------|----------------|---------|
| "加上 RAG，Agent 就更智能了" | RAG 提升的是知识覆盖度，不是推理能力。把 RAG 当成通用智能增强是误解 | 只在"模型不知道内容"时引入 RAG，不要期望它提升推理质量 |
| "加个 Memory，Agent 就能了解用户了" | Memory 的质量取决于你记了什么、忘了什么。不加筛选的 Memory 是噪音积累 | 先定义"什么值得记"，再实现记忆。遗忘策略比记忆策略更重要 |
| "上下文长点没关系，模型支持 128K" | 上下文越长注意力越衰减，"Lost in the Middle"效应让模型忽略中间的关键信息 | 上下文工程先行：分层、预算、优先级。信息密度比信息量更重要 |
| "用 Planning 拆一下，成功率就高了" | Planning 的步骤越多，每个步骤的累积误差越大。步骤拆解本身也在消耗可靠性 | 能不用 Planning 就不用。能用固定流程（Chain）就不用动态规划（ReAct/Plan-Execute） |
| "让 Agent 自己反思一下，答案就准了" | 没有外部反馈信号的 Reflection 是"自己证明自己是对的"——置信度可能变高，但准确性不会 | 必须有外部验证信号（测试、lint、API 结果）才能引入 Reflection |
| "拆成多个 Agent 并行，效率就高了" | Multi-Agent 的协调开销和结果合并成本经常超过并行带来的收益 | 先测量单 Agent 的瓶颈在哪里。如果瓶颈是 I/O，并行可能有效；如果瓶颈是推理质量，多 Agent 不一定更好 |
| "这个能力大家都用了，我们也应该有" | 社区热点 ≠ 你的场景需要。每个能力都有适配成本 | 从问题出发，不要从能力清单出发 |
| "先全加上，不好用再删" | 删除能力比添加能力更难——用户已经依赖了某些行为，系统之间产生了耦合 | 加一个容易，删一个难。宁可慢加，不要快删 |

## 9.4 写一份能力引入说明

一个成熟的 Agent 设计说明，应该能解释为什么引入某个能力，也能解释为什么不引入某个能力。

### 9.4.1 模板详解

```text
能力名称：
  → 使用统一的能力命名，如"RAG / 外部知识接入"、"会话摘要 Memory"

当前场景：
  → 用 2-3 句话描述：谁、在什么环境下、用 Agent 做什么
  → 示例："个人用户在本地使用知识助手，查询个人笔记中的技术资料"

用户问题：
  → 用户具体遇到了什么问题？用用户的原话或近原话描述
  → 示例："用户说：'每次我都要手动粘贴笔记内容，太麻烦了'"

问题证据：
  → 量化证据：频率、影响范围、严重程度
  → 示例："70% 的会话中用户需要粘贴资料；平均每次粘贴耗时 30 秒；
          20 个测试问题中 18 个无法在无检索情况下正确回答"

为什么现有最小闭环不够：
  → 说明当前系统的局限，重点讲"为什么无法通过优化 Prompt 或已有工具解决"
  → 示例："模型训练数据截止于 2023 年，用户的笔记是 2024-2025 年的内容。
          优化 Prompt 无法改变模型的知识边界。"

为什么选择该能力：
  → 是否考虑过替代方案？为什么选这个？
  → 示例："考虑过让用户手动上传文件（太麻烦）、使用联网搜索（用户笔记不在公网）。
          选择 RAG 是因为需要检索私有文档并输出可验证的引用。"

最小实现版本：
  → 具体描述：用什么工具/库、数据流是什么、最少需要多少代码
  → 示例："Markdown 文档切分（langchain text splitter，chunk_size=500）
          + OpenAI embeddings + Chroma 向量库 + Top-5 检索
          + Prompt 拼接引用。核心代码约 80 行。"

新增复杂度：
  → 运维、调试、成本、延迟等方面的增加
  → 示例："文档解析（需要支持 Markdown/PDF/HTML）~200 行
          切分策略调优（需要持续调整 chunk_size 和 overlap）
          向量库运维（需要定期重建索引）
          引用校验（需要验证引用是否真的来自原文）
          额外延迟：平均 +800ms"

评测方式：
  → 具体的评测方法、指标、基线
  → 示例："准备 20 个固定问题（覆盖有答案/无答案/多文档三种情况），
          人工标注标准答案和引用来源。
          指标：召回率 > 85%、答案准确率 > 80%、引用正确率 > 90%、拒答正确率 > 95%"

不引入的边界：
  → 什么情况下不使用这个能力
  → 示例："当用户直接提供完整资料时，跳过检索直接基于输入回答。
          当用户的问题与笔记内容无关时（如通用编程问题），使用模型自身知识。"

后续迭代：
  → 如果当前最小版本效果不够好，下一步往哪个方向优化
  → 示例："混合检索（关键词 + 向量）、rerank 精排、多文档引用交叉验证、
          支持更多文档格式（PDF 表格提取、网页剪藏清洗）"
```

**示例二：会话摘要 Memory**

```text
能力名称：会话摘要 Memory

当前场景：
个人开发者使用 Agent 作为编程助手，通常一个功能跨越 3-7 次独立会话完成。
每次会话间隔 1-3 天。

用户问题：
用户说："每次开新会话，它完全不知道我昨天在做什么。我得花 5 分钟重新
描述上下文，包括我在哪个分支、遇到的问题是什么、上次试了哪些方案。"

问题证据：
- 统计 30 天内 45 次会话：72% 的会话开头用户都在重新描述上下文
- 平均每次"上下文重建"耗时 4.2 分钟
- 跨会话任务的中断率 40%（用户忘了上次试到哪一步，导致重复或遗漏）

为什么现有最小闭环不够：
最小闭环的会话状态在会话结束时丢失。系统 Prompt 只能包含当前会话的历史，
无法跨会话延续。"让用户自己记录上下文"违背了 Agent 帮助用户的初衷。

为什么选择该能力：
- 考虑过要求用户在会话开始前手写 summary（用户觉得麻烦，且容易遗漏）
- 考虑过全量保存对话历史（隐私风险 + 上下文膨胀）
- 选择会话摘要：用户可控、存储量小、隐私友好

最小实现版本：
- 会话结束时，用 LLM 生成结构化摘要（JSON）：
  {
    "project": "项目名",
    "branch": "当前工作分支",
    "goal": "本次会话的目标",
    "progress": "完成了什么",
    "blockers": "卡在什么地方",
    "decisions": "做了哪些技术决策及原因",
    "next_steps": "下一步计划"
  }
- 存储为本地 JSON 文件：~/.agent/sessions/{project}.json
- 下次会话开始时，读取最近 3 次摘要，注入系统 Prompt
- "忘记"指令：用户说"忘记之前的上下文"时清除摘要文件

新增复杂度：
- 摘要质量依赖 LLM 能力（可能有遗漏或偏差）
- 摘要结构需要随项目类型调整
- 需要处理"用户同时进行多个项目"的情况
- 需要定期清理旧摘要，避免过期信息干扰

评测方式：
- 准备 10 个跨会话测试场景，每个场景 3-5 次会话
- 指标：上下文衔接准确率（Agent 是否正确理解了上次的状态）
- 方法：人工判断 Agent 在新会话中是否需要用户重新解释
- 基线（无 Memory）：用户需要重新解释的比例 72%
- 目标：降至 20% 以下

不引入的边界：
- 用户明确表示不需要跨会话延续时，不自动保存摘要
- 涉及敏感信息（密码、Token、个人隐私）的会话不保存内容摘要
- 单次会话能完成的任务不使用 Memory

后续迭代：
- 摘要质量自动评估（检查是否遗漏关键决策）
- 支持用户编辑和补充摘要
- 基于摘要自动生成项目进度报告
- 支持"记忆搜索"：用户查询历史摘要中的信息
```

**示例三：Planning / Workflow Patterns**

```text
能力名称：Planning / Workflow Patterns（ReAct 模式）

当前场景：
DevOps Agent 帮助用户处理运维任务。典型任务如"为新微服务配置部署流水线"，
涉及检查代码仓库、创建 Dockerfile、配置 K8s、设置 CI、验证部署。

用户问题：
用户说："我让它配置部署，它跳过了 Dockerfile 直接去配 K8s 了。"
或者"它做了 3 步就以为完成了，实际还有 4 步没做。"

问题证据：
- 测试 15 个多步骤运维任务（每个 6-10 步）
- 步骤遗漏率：35%（平均每个任务遗漏 2-3 个步骤）
- 步骤顺序错误率：20%（在不该做某步的时候做了）
- 用户需要手动检查和补全的比例：60%

为什么现有最小闭环不够：
最小闭环使用"接收指令 → 线性执行 → 返回结果"的模式。
步骤少（1-3 步）时可行，但运维任务通常 6-10 步，且步骤间有依赖关系。
简单增加 Prompt 中的步骤清单无法解决动态调整需求。

为什么选择该能力：
- 考虑过 Chain（固定流程）：不够灵活，不同项目的部署流程不同
- 考虑过 Router（路由到固定模板）：模板维护成本高，且无法覆盖边界情况
- 选择 ReAct：Agent 根据当前状态动态决定下一步，灵活且可追溯

最小实现版本：
- 使用 ReAct 循环：
  Thought: 分析当前状态，决定下一步
  Action: 执行具体操作（调用工具）
  Observation: 记录操作结果
  [循环直到目标达成]
- 在系统 Prompt 中定义任务完成的明确标准
- 最大步数限制：15 步（防止无限循环）
- 每一步输出当前进度（已完成/总步骤）

新增复杂度：
- 步骤漂移：Agent 可能在中途偏离原始目标
- 步骤爆炸：复杂的 Thought 可能导致单步耗时过长
- 恢复困难：某步失败后，从哪一步重试需要单独设计
- 调试难度：15 步的 ReAct Trace 比 1 步的 Prompt 难调试 10 倍

评测方式：
- 20 个标准化运维任务（覆盖简单、中等、复杂三个难度）
- 指标：任务完成率、步骤遗漏率、步骤顺序正确率、平均步骤数
- 基线（无 Planning）：完成率 65%，遗漏率 35%
- 目标（ReAct）：完成率 90%，遗漏率 < 10%

不引入的边界：
- 任务步骤 ≤ 3 步 → 使用简单的线性执行，不引入 ReAct
- 任务是标准化的、可模板化的 → 使用固定流程模板
- 用户明确给出了步骤清单 → 跟随用户清单，不做动态规划

后续迭代：
- 从 ReAct 升级到 Plan-Execute（先生成完整计划，用户确认后再执行）
- 增加 Checkpoint 机制（支持中断后从断点恢复）
- 引入人工确认节点（在不可逆操作前暂停）
- 计划质量评估（自动评估生成的计划是否合理）
```

**示例四：Multi-Agent（Reviewer 模式）**

```text
能力名称：Multi-Agent（Reviewer 模式）

当前场景：
团队使用 Agent 生成代码后直接提交 PR。团队成员反馈 Agent 生成的代码经常
有低级错误（未使用的导入、类型不匹配、边界条件遗漏），审查负担反而加重了。

用户问题：
代码审查者说："Agent 生成的代码看似正确，但仔细检查总有几个问题。
我得花和审查人类代码一样的时间，有时候还更多，因为它的错误模式很怪异。"

问题证据：
- 统计 50 个 Agent 生成的 PR：
  - 30% 包含至少 1 个语法/类型错误（IDE 能直接标出的那种）
  - 25% 遗漏了明显的边界条件（空值、空列表）
  - 15% 有不一致的问题（函数签名改了但调用方没改）
- 人类审查者的额外时间投入：平均每个 PR +12 分钟

为什么现有最小闭环不够：
单 Agent 在生成代码后，用同样的上下文检查自己的代码——"自己审自己"，
很难发现自己的盲区。增加 Reflection 可能改善部分问题，但 Reflection 的
"外部信号"（lint、测试）对语义和风格问题覆盖有限。

为什么选择该能力：
- 考虑过加强 Reflection（更多检查维度）：
  问题：Reflection 仍然使用同一个模型、相近的上下文，盲区相似
- 考虑过增加更多 lint 规则：
  问题：lint 能捕获语法问题，但边界条件和一致性问题需要"另一个视角"
- 选择轻量 Multi-Agent（Reviewer 模式）：
  Generator Agent 专注生成，Reviewer Agent 专注审查
  两个 Agent 使用不同的系统 Prompt，模拟"另一个人看代码"

最小实现版本：
- Generator Agent：
  系统 Prompt 强调"生成可用、完整的代码"
- Reviewer Agent：
  系统 Prompt 强调"你是一只挑剔的代码审查者，你的任务是找出所有问题"
  审查清单：语法、类型、边界条件、一致性、安全性、可读性
- 工作流：
  Generator 生成代码 → Reviewer 审查 → 输出审查意见
  → Generator 根据意见修改 → Reviewer 再次审查
  → 最多 3 轮 → 仍未通过则标记"需人工审查"
- 冲突解决：
  Generator 和 Reviewer 意见不一致时 → 标记"需人工决策"

新增复杂度：
- 两个 Agent 的调用成本翻倍（每个 PR 的 token 消耗 ×2-3）
- Reviewer 可能过于挑剔（"为了找问题而找问题"），导致无意义的修改循环
- 审查意见的质量取决于 Reviewer 的系统 Prompt 设计
- 需要 Trace 记录两个 Agent 的完整交互，调试复杂度显著增加

评测方式：
- 对比 30 个 PR（15 个单 Agent，15 个 Multi-Agent）
- 指标：语法错误率、边界条件遗漏率、人类审查者额外时间
- 基线（单 Agent）：语法错误率 30%，审查额外时间 12 分钟
- 目标（Multi-Agent）：语法错误率 < 5%，审查额外时间 < 5 分钟

不引入的边界：
- 简单改动（< 20 行、单文件、纯配置变更）→ 单 Agent 足够
- 原型阶段或探索性代码 → 审查标准不适用
- 已经有 CI 能覆盖的错误类型 → 不应由 Agent 重复检查

后续迭代：
- 引入更多专门化的 Reviewer（安全检查、性能检查）
- Reviewer 学习团队代码规范，自动适配审查标准
- 建立"常见问题库"，提升审查效率
- 引入可选的第三个 Agent（仲裁者），在两个 Agent 意见冲突时做决策
```

### 9.4.3 填写常见问题

**问题一：说不清问题证据**

很多人在"问题证据"字段写的是感觉，不是证据：

```text
❌ 差："用户觉得回答不够好"
✅ 好："测试 20 个问题，12 个答案缺少引用来源，用户需要手动验证"

❌ 差："有时候 Agent 会忘记上下文"
✅ 好："跨会话场景中，45% 的会话需要用户重新说明背景（统计 30 天数据）"
```

**原则：如果一个问题的严重程度你无法量化，那你就无法判断引入能力后是否真的变好了。**

**问题二：跳过"为什么现有最小闭环不够"**

这个字段被跳过的原因通常是"显而易见"。但正是这个"显而易见"需要被写下来——因为在写的过程中，你可能会发现有些问题其实可以通过优化 Prompt 解决，不需要引入新能力。

```text
自检清单：
☐ 我是否尝试过优化 Prompt 来解决这个问题？
☐ 我是否尝试过调整工具配置来解决这个问题？
☐ 我是否尝试过改变使用流程来解决这个问题？
☐ 如果以上都试过且无效，我是否记录了具体的失败证据？
```

**问题三：最小实现版本不够"最小"**

很多人写的"最小实现版本"其实已经是生产级方案了：

```text
❌ 不够最小："使用 Elasticsearch 集群 + 自训练 embedding + 多层 rerank +
           混合检索 + 实时索引更新 + 分布式部署"
✅ 足够小："Chroma 单机 + OpenAI embedding + Top-5 检索"

❌ 不够最小："完整的多 Agent 协作框架，包含任务分配、进度同步、冲突仲裁、
           结果合并、Trace 可视化"
✅ 足够小："两个 Agent 串行工作，冲突时标记人工决策"
```

**原则：最小版本的目标是"验证这个能力是否解决了问题"，不是"构建一个完美的能力系统"。**

**问题四：评测方式过于模糊**

```text
❌ 差："看看效果好不好"
✅ 好："20 个固定测试问题，人工标注标准答案，检查召回率 > 85%"

❌ 差："让用户反馈"
✅ 好："统计用户主动重新说明上下文的频率，目标从 72% 降到 20%"
```

**原则：评测方式必须可重复、可量化、有基线对比。**

## 9.5 能力组合的反模式

前四小节讲了"怎么做"，这一小节讲"别怎么做"。以下五个反模式来自真实项目中的教训。

#### 反模式一：能力堆砌症（Capability Hoarding）

**症状：**

```text
"我们的 Agent 用了 RAG + Memory + Planning + Reflection + Multi-Agent，
  还有 Tool Use、Guardrails、Human-in-the-loop..."
```

**问题诊断：** 说得出"有什么"但说不出"为什么有"。每个能力都是"万一有用呢"加进去的。结果系统复杂到没人能完整理解，出了问题不知道谁负责。

**解药：**

- 每个能力必须有对应的"问题票"（issue/ticket），记录引入的原因
- 定期做"能力盘点"：这个能力还在用吗？还解决当初的问题吗？
- 如果没人能说清一个能力为什么存在，考虑移除它

---

#### 反模式二："等完美了再上线"

**症状：**

```text
"RAG 的召回率才 78%，等优化到 90% 再上线"
"Memory 的遗忘策略还没调好，先不上"
"Planning 有时候会漂移，等我们解决了再发布"
```

**问题诊断：** 等待完美的最小版本等于永不发布。用户宁愿要一个有缺点的 Assistant 也不愿意等一个不存在的完美 Assistant。而且，不发布就无法获得真实使用反馈，没有反馈就无法真正优化。

**解药：**

- 定义"足够好"的阈值，而不是"完美"的标准
- 低于阈值的能力可以上线但标注为"实验性功能"
- 设置用户反馈通道，让真实使用驱动优化

**判断标准：**

```text
能上线的最低标准：
☐ 核心功能是否可用？（不能有阻塞性 bug）
☐ 安全问题是否处理？（不能泄露数据、执行危险操作）
☐ 失败模式是否优雅？（出问题时告诉用户，而不是悄悄出错）
☐ 用户是否可以选择不使用？（能力应该是可选的）

不需要等到的标准：
☐ 性能是否最优？（可以后续优化）
☐ 边界情况是否全覆盖？（有些边界只有用了才会发现）
☐ 所有评测指标是否达标？（真实数据可能和评测集不同）
```

---

#### 反模式三：盲目跟风

**症状：**

```text
"Hacker News 上那篇文章说了，Multi-Agent 是 2024 年最重要的趋势"
"大家都在用 LangGraph 做 Agent 工作流，我们也应该用"
"XXX 公司的技术博客说他们的 Agent 用了 Reflection，效果很好"
```

**问题诊断：** 别人的场景、数据、约束、用户和你都不一样。适合他们的方案不一定适合你。用社区热点代替自己的问题分析，是引入不必要复杂度的最快路径。

**解药：**

- 读别人的方案时，关注"他们遇到了什么问题"而不是"他们用了什么技术"
- 如果不能用一句话说清楚你的场景需要这个能力，就先不引入
- 把社区文章当成"可选方案菜单"而非"必做清单"

**转换练习：**

```text
读到："我们用 Multi-Agent 提升了代码审查质量"
不要想："我也应该加 Multi-Agent"
应该想："他们的代码审查遇到了什么问题？这个问题我也有吗？
        他们用 Multi-Agent 解决了什么？为什么单 Agent 不够？
        我的代码审查流程中，单 Agent 的瓶颈是什么？"
```

---

#### 反模式四：引入后不评测、不迭代

**症状：**

```text
"我们两周前加了 RAG，用户说好像好了一点...其实我也不确定"
"Memory 加上去了，没看效果，反正应该有用"
"Planning 的步骤有时候不太对，但大部分时候还行吧"
```

**问题诊断：** 引入能力只是开始，不是结束。不加评测的能力引入等于在系统中加入了"不知道影响力"的变量。久而久之，你不知道系统的质量是在上升还是下降。

**解药：**

- 每个能力引入时必须定义评测方式（参见 9.4.1）
- 每周/每月查看评测指标，确认没有退化
- 引入新能力时重新跑所有已有能力的评测，检查是否互相影响

**评测节奏建议：**

```text
引入前：建立基线评测（当前系统在这些指标上的表现）
引入后第 1 周：每日评测（快速确认方向是否正确）
引入后第 2-4 周：每周评测（观察使用模式稳定后的表现）
引入后第 2 个月起：每月评测（持续监控，防止退化）
```

---

#### 反模式五：能力之间缺乏协调

**症状：**

```text
RAG 检索到的内容，Memory 也记了一份（重复存储）
Planning 生成了 5 步计划，但 Reflection 认为第 2 步已经完成（信息不一致）
Multi-Agent 中的 Agent A 修改了文件，Agent B 不知道（状态不同步）
```

**问题诊断：** 能力不是独立运行的插件，它们共享状态、影响彼此的行为。当多个能力对同一份数据有不同的理解时，系统的行为就不可预测了。

**解药：**

- 定义能力的职责边界：谁负责什么数据
- 统一的状态管理：所有能力通过同一个 State 层读写
- 能力间的信息传递要有明确的契约（schema/格式）
- 定期检查能力间的一致性

**能力协调清单：**

```text
☐ RAG 检索结果是否进入了 Memory 系统？如果是，是否需要？
☐ Memory 中的信息是否影响了 Planning 的步骤生成？
☐ Reflection 的修正是否更新了 Planning 的状态？
☐ Multi-Agent 中的各 Agent 是否共享同一个 Memory 视图？
☐ 一个能力的输出变更是否会破坏另一个能力的输入假设？
```

当能力开始组合时，它们不应该靠 Prompt 临时约定互相配合，而应该交给运行时承载：统一的 State、Trace、Checkpoint、Eval 回归集和权限边界会在课程六的 Harness 架构中展开。本章先解决"该不该组合"，课程六再解决"组合后如何稳定运行"。

## 9.6 渐进式引入的节奏与信号

#### 什么时候该加速引入

以下信号说明用户的需求已经明显超出了当前系统的能力边界，加速引入是合理的：

```text
加速信号：
✅ 同一个问题的用户反馈在 2 周内出现了 5+ 次
✅ 当前能力的评测指标持续下降（说明需求在增长但能力没跟上）
✅ 竞争对手已经提供了这个能力，用户开始流失
✅ 问题的解决方案明确且风险可控
✅ 用户明确表示"如果不支持 X 我就没法继续用了"
```

**加速不代表操之过急。** 即使加速，也要保持"引入 → 评测 → 观察"的流程。加速缩短的是观察周期（比如从 4 周缩短到 2 周），不是跳过评测。

#### 什么时候该暂停、巩固

以下信号说明你需要停下来消化，而不是继续加新能力：

```text
暂停信号：
🛑 上一次引入的能力的评测指标还不稳定
🛑 有已知 bug 还没修（超过 3 个 open issue）
🛑 团队成员反映"系统太复杂了，新加入的人看不懂"
🛑 用户反馈中关于"慢"和"不可靠"的比例在上升
🛑 你发现自己说不清当前系统有多少个能力在工作
```

**暂停不是失败，是负责任。** 暂停期间可以做：
- 修复已知 bug
- 完善文档和 Trace
- 优化评测体系
- 清理不再需要的能力（参见 9.7）
- 降低系统复杂度

#### 节奏控制的三原则

**原则一：一次一个新变量**

任何时候只引入一个新能力。如果一个能力引入后效果不好，你才能确定是这个能力的问题。同时引入两个能力，你永远分不清是哪个起了作用、哪个带来了问题。

```text
❌ 同时引入 RAG + Memory：
  "检索变好了，但跨会话上下文好像也有改善...不确定是哪个原因"

✅ 先引入 RAG，稳定 2 周后引入 Memory：
  "RAG 引入后召回率从 0% 提升到 85%（确定是 RAG 的效果）"
  "2 周后引入 Memory，跨会话衔接准确率从 28% 提升到 80%（确定是 Memory 的效果）"
```

**原则二：评测指标永远在新能力之前**

如果你没有一个可运行的评测流程，就不要引入新能力。没有评测的引入等于闭着眼睛走路——你不知道走对了吗，还是离目标越来越远。

**原则三：两周一复盘**

无论系统是否稳定，每两周至少做一次能力复盘：

```text
复盘检查清单：
☐ 每个已引入的能力：评测指标是否稳定或改善？
☐ 每个暂未引入的能力：不引入的理由是否仍然成立？
☐ 是否有新的问题模式出现？
☐ 是否有能力可以被降级或移除？
☐ 系统整体复杂度是否可以降低？
☐ 团队对系统的理解是否充分（新成员能否快速上手）？
```

#### 真实项目时间线参考

以下是三个不同规模团队的能力引入节奏参考（基于真实案例调整）：

```text
个人项目（1 人，业余时间）：
  月 1：最小闭环上线
  月 2-3：引入第一个增强能力
  月 4-5：迭代优化第一个能力
  月 6-7：引入第二个增强能力
  → 节奏：每 2-3 个月一个新能力

小型团队（3-5 人，全职）：
  月 1：最小闭环上线
  月 1-2：引入第一个增强能力
  月 2-3：引入第二个增强能力
  月 4-6：稳定运行，优化评测
  → 节奏：每 4-6 周一个新能力

中型团队（10-20 人，多个 Agent 项目并行）：
  月 1：最小闭环上线
  月 1-2：RAG 引入并稳定
  月 2-3：Memory 引入
  月 3-4：Planning 引入
  月 4-5：Reflection 引入
  月 5-6：Multi-Agent 引入（仅对需要并行/互审的项目）
  → 节奏：每 3-4 周一个新能力，但每个项目的能力组合不同
```

> **注意：** 以上时间线是参考，不是 KPI。如果你的项目不需要第五个能力，停在第三个就是最优解。

## 9.7 能力的降级与移除

大部分教程只讲"怎么加"，不讲"怎么减"。但成熟的系统设计和成熟的工程师，都要能判断什么时候该移除一个能力。

#### 什么时候该考虑移除

| 信号 | 说明 | 示例 |
|------|------|------|
| **问题已消失** | 当初引入这个能力要解决的问题已经不存在了 | 用户换了工作流，不再需要跨会话 Memory |
| **能力从未被使用** | 加了一直没用，或使用率极低（< 5%） | Planning 系统很完善，但 95% 的任务都是单步 |
| **代价大于收益** | 能力的维护成本/延迟/错误率超过它带来的价值 | Memory 经常记错导致用户困扰，维护遗忘策略的时间超过省下的时间 |
| **有更简单的方案** | 发现不需要这个能力也能解决问题 | Prompt 优化 + 更好的工具描述达到了和 Planning 相近的效果 |
| **与其他能力冲突** | 两个能力的行为互相矛盾，用户感到困惑 | Memory 记了一种风格，但 RAG 检索到的示例用了另一种风格 |
| **评测指标持续下降** | 能力的质量在持续退化，且优化成本过高 | RAG 的索引更新越来越慢，召回率持续下降 |

#### 如何安全地降级或移除

移除能力需要比引入能力更谨慎，因为用户可能已经依赖了某些行为。

**降级移除流程：**

```text
第一步：确认影响范围
  - 统计该能力的使用频率
  - 列出依赖该能力的用户场景
  - 评估移除后的用户体验变化

第二步：寻找替代方案
  - 能否用更简单的方式覆盖核心需求？
  - 能否把能力从"自动"改为"按需触发"？
  - 能否把能力限制到更小的范围？

第三步：渐进降级
  方案 A：从"自动"改为"手动"
    原：每次对话自动执行 Memory 摘要
    改：用户输入 /summarize 时才执行

  方案 B：缩小范围
    原：Memory 记录所有对话
    改：Memory 只记录用户明确标记为"重要"的对话

  方案 C：用更轻量的替代
    原：完整的 ReAct Planning
    改：固定的 Chain 流程（如果大部分任务已标准化）

第四步：灰度移除
  - 对 10% 用户关闭该能力，观察反馈
  - 如果无负面反馈，扩大到 50%
  - 如果持续无问题，完全移除

第五步：记录决策
  - 为什么移除？当初为什么引入？学到了什么？
  - 这段经验对未来的能力引入决策有什么帮助？
```

**降级优于直接移除：**

在很多情况下，把能力从"主动"降为"被动"是更好的选择：

```text
原状态：Memory 自动记录每次会话
降级后：Memory 只在用户明确要求时记录

原状态：RAG 自动检索每次查询
降级后：RAG 只在用户提及"查一下"时触发

原状态：Planning 自动拆解所有任务
降级后：Planning 只在任务超过 5 步时触发

原状态：Multi-Agent 并行处理
降级后：默认单 Agent 处理，用户添加 --review 标记时启动 Reviewer
```

#### 移除后的效果验证

移除一个能力后，需要确认：
- 之前被该能力解决的问题是否重新出现？
- 系统延迟是否改善？
- 用户反馈是否变化？
- 维护负担是否减轻？

如果一个能力被移除后，上述指标都没有变化——那说明这个能力本来就是不必要的，移除是正确的。

## 9.8 实战演练：为新场景做能力引入决策

以下是三个新场景。请为每个场景：
1. 判断应该优先引入哪些能力
2. 说明暂不引入哪些能力及理由
3. 画出能力引入路径

#### 练习一：法律文书审查 Agent

**场景：** 律师事务所需要一个 Agent 帮助初级律师审查合同。Agent 需要检查合同条款是否合规、是否存在风险条款、是否遗漏关键条款。审查结果需要人工律师最终确认。

**约束：**
- 合同内容高度敏感，不能上传到外部服务
- 不同行业的合同模板不同（科技、金融、房地产）
- 审查出错可能造成法律风险
- 律所有内部知识库（过往判例、合同模板库）

**请做出你的判断：**
- 优先能力：_______
- 暂不引入：_______
- 引入路径：_______

<details>
<summary>参考答案（点击展开）</summary>

**优先能力：**
- **RAG**：需要检索律所内部知识库（过往判例、合同模板），且必须本地部署（隐私约束）
- **Human-in-the-loop**：法律审查的最终决策必须由人类律师确认，这是硬需求

**暂不引入：**
- **Memory**：每个合同是独立的，跨合同记忆可能导致信息混淆
- **Planning**：合同审查是结构化的（逐条检查），用固定的审查清单即可，不需要动态规划
- **Reflection**：Agent 自查合同质量不如人类律师审查，且自查缺少法律判断的外部信号
- **Multi-Agent**：单人审查场景，一个 Agent + 人类律师确认已足够

**引入路径：**
```text
最小闭环（LLM + 合同读取工具）
  → RAG（检索知识库，对比合同条款）
    → HITL（人工确认节点）
      → 稳定运行
```

</details>

#### 练习二：智能家居中控 Agent

**场景：** 用户通过语音/文字与智能家居中控 Agent 交互。Agent 需要控制灯光、空调、窗帘、音响等设备，理解用户的日常习惯，并根据场景自动调整（如"我要睡觉了"触发关灯、关窗帘、调低空调）。

**约束：**
- 设备控制需要低延迟（< 500ms）
- 用户习惯因人而异（有人睡觉开空调 26°C，有人 24°C）
- 场景触发需要用户确认（防止误操作）
- 支持多用户（家庭成员各有偏好）

**请做出你的判断：**
- 优先能力：_______
- 暂不引入：_______
- 引入路径：_______

<details>
<summary>参考答案（点击展开）</summary>

**优先能力：**
- **Tool Use**：控制设备的基础能力（调用设备 API）
- **Memory**：记住各用户的偏好（温度、灯光亮度、常用场景）
- **Human-in-the-loop**：场景触发时向用户确认（安全需求）

**暂不引入：**
- **RAG**：智能家居控制不依赖外部知识
- **Planning**：场景通常是预定义的（"睡觉"→ 关灯+关窗帘+调温度），不需要动态规划
- **Reflection**：设备控制结果可通过传感器反馈确认（灯是不是关了），不需要 Agent 反思
- **Multi-Agent**：单用户服务，不需要并行

**引入路径：**
```text
最小闭环 + Tool Use（设备控制）
  → Memory（记住用户偏好）
    → HITL（场景确认）
      → 稳定运行
```

</details>

#### 练习三：技术文档自动维护 Agent

**场景：** 开源项目维护者需要一个 Agent 自动维护技术文档。当代码变更时，Agent 需要检测哪些文档需要更新，自动生成文档 PR，并在 PR 中解释变更内容。

**约束：**
- 文档和代码的一致性至关重要
- 需要跨多个文件理解变更的影响
- 文档质量影响新用户的 onboarding 体验
- 文档变更需要人类维护者最终审批

**请做出你的判断：**
- 优先能力：_______
- 暂不引入：_______
- 引入路径：_______

<details>
<summary>参考答案（点击展开）</summary>

**优先能力：**
- **Tool Use**：读取代码 diff、搜索受影响的文档、创建 PR
- **Reflection**：生成文档后，对比代码和文档的一致性（如检查 API 参数名是否匹配）
- **Planning**：复杂变更可能影响多份文档，需要有条理地处理

**暂不引入：**
- **RAG**：文档内容来源于代码本身，不需要外部知识库
- **Memory**：每次文档更新独立处理，不需要跨会话状态
- **Multi-Agent**：单 Agent 可以完成"读取变更 → 更新文档 → 验证一致性"的完整流程。PR 审批由人类维护者完成，不需要 Agent Reviewer

**引入路径：**
```text
最小闭环 + Tool Use（读代码、搜文档、创 PR）
  → Reflection（验证文档-代码一致性）
    → Planning（多文件变更的文档更新编排）
      → 稳定运行
```

</details>

### 本章速记

#### 核心原则回顾

课程五的前八章分别讲了七类增强能力的原理、边界和最小实现。本章（第九章）不是再加一种新能力，而是教你如何把多个能力组合起来，以及更重要的是——**如何有纪律地决定什么时候引入、什么时候暂缓、什么时候移除。**

```text
┌────────────────────────────────────────────────────────────┐
│                    能力组合的核心原则                        │
├────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 问题先行，能力响应                                        │
│     不是从能力清单出发，而是从用户遇到的具体问题出发              │
│                                                             │
│  2. 一个一个加，一次一个变量                                    │
│     同时引入多个能力 = 同时引入多个未知变量                      │
│                                                             │
│  3. 每个能力都要有评测                                         │
│     没有基线和评测指标的能力引入是闭眼开车                       │
│                                                             │
│  4. 最小版本验证，生产版本迭代                                   │
│     先用最简实现验证方向，确认有效后再投入工程资源                 │
│                                                             │
│  5. 知道什么时候不引入、什么时候移除                              │
│     "不做什么"和"做什么"是同等重要的工程决策                     │
│                                                             │
│  6. 每个能力都有代价                                           │
│     延迟、复杂度、调试难度、token 成本——这些代价在组合时放大      │
│                                                             │
└────────────────────────────────────────────────────────────┘
```

#### 一张图总结

```text
                    你的 Agent 从这里开始
                            │
                    ┌───────▼───────┐
                    │  基线：       │
                    │  最小闭环     │
                    │  LLM + Tool   │
                    │  + Loop       │
                    └───────┬───────┘
                            │
      ┌─────────────┬─────────────┬─────────────┐
      │ 问题驱动     │ 问题驱动     │ 问题驱动     │
      ▼              ▼              ▼              ▼
┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
│  RAG     │  │  Memory  │  │ Context  │  │ Planning │
│ 外部知识 │  │ 状态延续 │  │ 上下文管理│  │ 任务组织 │
└────┬─────┘  └────┬─────┘  └────┬─────┘  └────┬─────┘
     │              │              │              │
     └──────────────┴──────────────┴──────────────┘
                            │ 问题驱动
          ┌─────────────────┼─────────────────┐
          ▼                 ▼                 ▼
   ┌────────────┐    ┌────────────┐    ┌────────────┐
   │ Reflection │    │   HITL     │    │Multi-Agent │
   │ 异常处理   │    │ 决策边界   │    │ 角色分工   │
   └────────────┘    └────────────┘    └────────────┘

    每个能力的引入都是一个决策点：
    - 有明确的问题证据吗？
    - 有量化的评测方式吗？
    - 现有最小闭环真的不够吗？
    - 代价可以接受吗？

    如果任何一个答案是"否"，就暂不引入。
    停在当前能力组合可能已经是最优解。
```

#### 与其他课程的关系

- **课程三（最小 Agent）**：你从这里出发，有了一个能跑的最小闭环
- **课程四（工具机制）**：你学会了如何让 Agent 与外部世界交互
- **课程五第二章~第八章**：你学会了七类增强能力各自是什么、怎么实现、什么时候不用
- **本章（第九章）**：你学会了如何决策——什么时候加、什么时候不加、什么时候减
- **课程六（生产化）**：你将学会如何把稳定运行的 Agent 部署到生产环境，包括监控、Trace、持续评测、成本控制等

#### 值得记住的三句话

> **1. "没有证据，就不引入。"**
> 如果你说不出"用户在哪一步遇到了什么问题"，就不要引入那个能力。

> **2. "加一个能力容易，删一个能力困难。"**
> 用户会依赖你的系统行为。加之前三思，加之后负责。

> **3. "停在低复杂度组合可能是最优解。"**
> 不是每个 Agent 都需要爬满所有能力。对很多场景来说，RAG + 轻量 Memory 已经足够好。
