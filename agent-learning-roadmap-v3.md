# Agent 学习路线图（V3）：从最小闭环到产品化系统

## 课程基本介绍

这份路线图面向希望系统学习 Agent 产品开发的人，目标是从"第一次接触 Agent"逐步走到"能够设计、实现、评估并上线 Agent 产品"。

这份路线图将 Agent 学习内容分为不同层级：最小闭环、工具系统、场景增强能力、运行时架构和产品化能力。

课程主线：

```text
建立直觉 -> 理解范式演进 -> 掌握最小 Agent 闭环
    -> 深入工具机制 -> 学习场景增强能力
    -> 设计 Harness 运行时 -> 完成产品化实践
    -> 项目实战与生态跟进
```

核心观点是：

```text
Agent = LLM 驱动的目标导向运行时 + 可调用工具 + 可管理上下文 + 可验证执行过程
```

RAG、Memory、Planning、Reflection、Multi-Agent、MCP、Skill 都很重要，但它们不是同一层级的东西，也不是所有 Agent 应用的必选项。它们应该围绕最小 Agent 闭环展开：有些属于工具机制，有些属于场景增强，有些属于运行时架构，有些属于产品化能力。

## 设计原则

- **先感性，再结构化**：先体验真实产品，再建立抽象概念，避免一开始陷入术语堆砌。
- **先最小闭环，再扩展能力**：先实现一个能多步决策、调用工具、观察结果并停止的 Agent，再引入 RAG、Memory、Planning 等增强能力。
- **把 Tool Use 作为独立主线**：工具调用是 Agent 从"会说"到"会做"的关键，应该完整讲清工具定义、工具选择、工具执行、权限、MCP、Skill 和 Human-in-the-loop。
- **把 RAG / Memory 放回场景中**：它们不是所有 Agent 的核心模块，而是根据业务场景选择的能力。
- **从运行时理解架构**：Harness、Context Engineering、Evaluation、Guardrails、Observability 都应该围绕 Agent 运行时展开。
- **最终回到产品**：学习目标不是写出 Demo，而是理解如何让 Agent 可靠、可控、低成本、安全，并真正被用户使用。

## 课程一：初识 Agent

**定位**：建立感性认知，先看见 Agent 和普通 Chatbot / Workflow 的区别。

### 核心问题

- Agent 到底是什么？
- 它和 Chatbot、Workflow 的区别在哪里？
- 一个产品在什么地方体现出 Agent 特征？
- 用户为什么会觉得 Agent 有用、失控、惊喜或不可靠？

### 核心内容

- **Agent 基本理解**：Agent 不是"更聪明的聊天机器人"，而是能够围绕目标持续决策和执行的系统。
- **Agent vs Chatbot vs Workflow**：
  - Chatbot 主要负责对话和回答。
  - Workflow 按固定流程执行。
  - Agent 会根据目标、状态和反馈决定下一步。
- **观察重点**：不要只看产品是否用了 LLM，而要看"谁决定下一步"。
- **主流产品体验**：
  - 国外产品：ChatGPT、Claude / Claude Code、Cursor、Perplexity、Replit Agent。
  - 国内产品：Kimi、豆包、通义千问、腾讯元宝、扣子 Coze。
  - 观察重点：它是否会自己拆解任务、调用工具、生成计划、读写文件、检索资料、跨应用行动；还是主要停留在问答、搜索或固定工作流。
- **产品体验观察框架**：
  - **任务入口**：用户输入的是开放目标、具体指令，还是普通问答？
  - **下一步决策权**：下一步由用户决定、固定流程决定，还是模型根据状态动态决定？
  - **行动能力**：产品能调用哪些工具或外部能力，例如搜索、代码、文件、浏览器、数据分析、工作流节点？
  - **过程可见性**：用户能否看到计划、步骤、工具调用、进度和中间结果？
  - **用户控制权**：产品在哪些节点请求确认，是否支持中断、修改计划、回退或接管？
  - **失败与恢复**：出错后是直接失败、重新尝试、换路径，还是请求用户帮助？
  - **Agent 成色判断**：它更像 Chatbot、Workflow，还是 Agent？哪些地方体现出 Agent 特征？

### 实践里程碑

完成 3 款 Agent 产品拆解，每份拆解至少回答：

- 它解决了什么用户任务？
- 它的下一步决策权主要在用户、固定流程，还是模型？
- 它具备哪些行动能力和工具能力？
- 它如何让用户看到过程并保持控制？
- 它在哪里失败，失败后如何恢复？
- 它更像 Chatbot、Workflow，还是 Agent？
- 如果产品化，最大风险是什么？

## 课程二：Agent 范式演进

**定位**：理解 Agent 为什么会出现，以及它解决了 LLM 的什么问题。

这门课按问题演进来理解 Agent：每一次新方法的出现，都是为了解决前一阶段暴露出来的问题。

### 核心问题

- 传统 Agent 为什么很少进入普通用户的日常应用场景？
- LLM 出现后，Agent 的决策核心发生了什么变化？
- Function Calling / Toolformer / Plugins 解决了什么问题？
- ReAct 为什么适合作为理解 Agent 运行循环的经典框架？
- Planning、Reflection、Multi-Agent 分别是在什么背景下出现的？

### 核心内容

- **传统 Agent 的历史背景**：
  - 强化学习 Agent
  - 软件 Agent
  - BDI 模型
  - Russell & Norvig 的 Agent 类型
  - 传统 Agent 的根本限制：封闭规则空间、泛化能力弱、工程成本高
- **LLM 带来的范式变化**：
  - 从"人工写规则"变成"模型理解目标并生成下一步"
  - 从"固定流程"变成"动态决策"
  - 从"只回答"走向"能调用外部能力"
- **工具调用为什么出现**：
  - 让 LLM 能连接实时信息、私有数据、计算能力和外部系统
  - 让工具调用成为模型能力的一部分
  - 让工具定义、参数生成和结果返回变得工程化
- **ReAct 的意义**：
  - 将推理、行动、观察放入同一个循环，清晰展示了 Agent 如何从"一次性回答"走向"持续执行任务"。
  - 给出了 Agent 最小运行时雏形：判断下一步 → 调用工具 → 接收结果 → 继续或停止。
  - 引出后续工程问题：停止条件、循环控制、工具失败处理、上下文和状态管理。
- **Planning 的出现背景**：
  - 复杂任务无法一步完成
  - 需要任务拆解、依赖排序和阶段性目标
- **Reflection 的出现背景**：
  - 模型和工具调用都会失败
  - 系统需要自检、重试、纠错和复盘
- **Multi-Agent 的出现背景**：
  - 单 Agent 面临角色混淆、上下文拥挤、工具数量过多、复杂任务难以并行等问题
  - 多 Agent 通过角色分工、互相审查和并行处理解决部分瓶颈

### 实践里程碑

能用以下框架讲清一个 Agent 技术或产品公告：

```text
之前有什么问题 -> 新方法解决了什么 -> 新方法带来了什么新问题
```

## 课程三：最小 Agent 闭环

**定位**：实现一个最小但完整的 Agent。

这门课是整条路线的技术起点。学习者必须先理解 Agent 最小闭环，后面所有能力都是围绕这个闭环扩展。

### 核心问题

- 一个最小 Agent 为什么不能只有 LLM？
- LLM 如何根据上下文和状态做下一步决策？
- 工具/环境交互如何产生反馈，并影响下一轮决策？
- 状态管理如何支撑多步任务，而不是只处理单轮问答？
- 循环控制如何判断继续、停止、重试或失败退出？

### 核心内容

#### 3.1 最小 Agent 组成

```text
Agent = LLM 决策 + 工具/环境交互 + 状态管理 + 循环控制
```

| 组成 | 学什么 | 为什么重要 |
|---|---|---|
| LLM 决策 | 模型选择、System Prompt、结构化输出、下一步判断 | 决定 Agent 如何理解目标并选择下一步 |
| 工具/环境交互 | Tool Use、Function Calling、API / 文件 / 数据库 / 浏览器 / 代码执行 | 决定 Agent 如何获取外部反馈或影响外部环境 |
| 状态管理 | 当前任务、历史消息、工具结果/环境反馈、错误信息 | 决定 Agent 是否能连续工作 |
| 循环控制 | 继续/停止判断、最大步数、超时、重复动作检测、失败退出 | 决定 Agent 如何多步执行且保持可控 |

#### 3.2 最小运行链路

```text
User Goal
    -> Context Assembly
    -> LLM Decision
    -> Tool / Environment Interaction
    -> Observation / Feedback
    -> State Update
    -> Continue or Stop
```

核心公式描述的是 Agent 的组成能力；运行链路描述的是这些能力在每一步如何协作。因此 `Context Assembly` 和 `Observation / Feedback` 不是额外组件，而是状态管理、LLM 决策、工具/环境交互在运行时的连接点。

#### 3.3 运行时循环总览

理解 Agent 时需要一张运行时地图。可以把最小 Agent 看成一个不断推进的控制循环：

```text
User Goal -> Context Assembly -> LLM Decision -> Interaction -> Execution
                ^                                               |
                | read                                          v
       +-----------------------+                    Observation / Feedback
       | State / Memory Store  |                                |
       | task state, history,  |                                v
       | optional persistence  | <----------------------   State Update
       +-----------------------+          write                 |
                                                                v
                                                         Continue or Stop
```

这张图的作用是展示主链路、状态读写和反馈闭环之间的关系。各环节的学习重点在 3.4 展开。

这里要特别区分：最小 Agent 必须有任务状态和循环控制，但不一定需要长期 Memory；RAG 也不是循环的必需节点，而是可以作为一种检索工具参与 Context Assembly 或 Interaction。

#### 3.4 沿运行链路展开学习

基于最小运行链路，后续学习不再按概念堆砌，而是逐步回答每个运行环节需要解决的问题。

| 运行环节 | 作用 | 学习内容 |
|---|---|---|
| State / Memory Store | 状态存储位置，不是主链路上的独立步骤；Context Assembly 从中读取，State Update 向其中写入 | 任务状态、历史记录、可选长期记忆、状态读写边界 |
| Context Assembly | 决定哪些信息进入模型上下文 | 用户目标、系统约束、历史消息、任务状态、工具结果/环境反馈、检索结果和记忆召回 |
| LLM Decision | 模型根据当前上下文判断下一步 | System Prompt、结构化输出、下一步判断、工具选择、低置信度处理 |
| Tool / Environment Interaction | 模型选择工具或环境动作，并生成必要参数 | 工具定义、参数生成、环境动作选择、权限判断、用户确认 |
| Execution | 运行时执行工具或环境动作 | 超时、重试、沙箱、异常处理、执行结果规范化 |
| Observation / Feedback | 把工具结果或环境反馈转成模型可理解的信息 | 结果格式化、错误归一化、摘要压缩、关键信息提取 |
| State Update | 更新任务状态、历史记录、错误信息和中间结果 | 状态写入时机、错误记录、中间结果保存、可选长期 Memory 写入 |
| Continue or Stop | 判断继续、停止、重试或失败退出 | 停止条件、最大步数、重复动作检测、失败退出、人工接管 |

### 实践里程碑

实现一个最小 ReAct Agent，要求：

- 支持至少 3 个工具或环境动作。
- 能完成多步任务。
- 能记录每一步的 LLM 决策、工具/环境交互、观察结果和停止原因。
- 有基础错误处理：工具不存在、环境不可用、参数不合法、执行失败、循环超限。
- 用 10 条测试任务验证，成功率达到 70% 以上。

## 课程四：工具机制

**定位**：理解 Agent 在运行时如何调用工具，以及如何把工具调用设计成可选择、可执行、可控、可复用、可审计的系统机制。

课程三已经说明了最小 Agent 闭环，这门课继续深入其中的工具调用部分。重点不是"接入更多能力"，而是理解工具调用为什么容易失败，以及系统如何通过工具定义、选择、执行、权限、协议、Skill 和 Human-in-the-loop 来解决这些问题。

### 核心问题

- Agent 在运行时如何决定是否调用工具？
- 工具调用为什么经常失败？
- 如何让模型知道工具能做什么、什么时候该用？
- 如何让工具执行结果可靠地回到下一轮决策？
- 如何控制高风险工具调用？
- 如何把重复任务沉淀成可复用能力？

### 核心内容

#### 4.1 Agent 如何调用工具

工具调用不是一次孤立的 API 请求，而是 Agent Runtime 中的一段可控流程：

```text
LLM Decision
    -> Tool Selection
    -> Parameter Generation
    -> Permission Check
    -> Tool Execution
    -> Observation / Feedback
    -> State Update
```

每一步都可能失败：模型可能不知道该用哪个工具，参数可能填错，工具可能超时，结果可能太长，高风险动作可能需要用户确认。

| 工具调用中的问题 | 对应机制 | 学什么 |
|---|---|---|
| 模型不知道有哪些工具、什么时候该用 | 工具定义 | 名称、描述、参数 Schema、返回值、工具边界 |
| 模型选错工具或乱填参数 | 工具选择 | 工具路由、参数生成、强制/禁止工具调用、选择失败分类 |
| 工具执行可能失败或结果太长 | 执行与回填 | 参数校验、超时、重试、错误结构化、结果摘要 |
| 工具可能造成真实世界影响 | 权限与安全 | 风险分级、最小权限、审批、审计、回滚 |
| 工具接入方式不统一 | MCP | 工具发现、协议化接入、Client / Server、Tools / Resources / Prompts |
| 多个工具经常组合完成同类任务 | Skill | 工具组合、步骤经验、默认流程、失败处理、复用边界 |
| 模型不应独自决定高风险动作 | Human-in-the-loop | 确认、澄清、审批、接管、回滚、审计记录 |

#### 4.2 工具定义：让模型知道工具是什么

工具定义的目标是让模型知道"这个工具能做什么、什么时候该用、什么时候不该用"。

学习内容：

- 工具名、description、参数 Schema、返回值结构。
- 工具描述如何影响模型选择。
- 参数设计：类型、枚举、必填、默认值、约束。
- 工具粒度：原子工具 vs 组合工具。
- 常见错误：描述太泛、参数模糊、返回值太长、多个工具边界重叠。

#### 4.3 工具选择：让模型知道什么时候用哪个工具

工具选择的核心问题是三件事：什么时候用、用哪个、参数怎么填。

学习内容：

- 模型自主选择、规则路由、混合路由。
- `tool_choice`、强制工具调用、禁止工具调用。
- 多工具候选时的冲突处理。
- 参数生成和参数修正。
- 选择失败类型：该用不用、不该用乱用、选错工具、参数猜错。

#### 4.4 执行与回填：让工具结果进入下一轮决策

工具执行不是模型完成的，而是 Runtime 完成的。模型生成 tool call，Runtime 校验、执行并把结果回填成 Observation / Feedback。

学习内容：

- Runtime 如何接收 tool call 并执行。
- 参数校验、超时、重试、幂等。
- 错误结构化：参数错误、权限错误、网络错误、业务错误。
- 结果压缩：长结果摘要、分页、截断、转资源。
- Observation / Feedback 如何影响下一轮决策。

#### 4.5 权限与安全：让工具调用可控

工具调用可能读写文件、查询数据库、发送消息、修改配置或触发外部动作，因此必须有权限和安全设计。

学习内容：

- 工具风险分级：只读、低风险写入、高风险写入、危险操作。
- 权限模型：用户授权、最小权限、作用域。
- 执行前校验：参数、目标对象、影响范围。
- 审批策略：自动执行、确认后执行、禁止执行。
- 审计日志：谁发起、模型为什么调用、参数是什么、结果是什么。

#### 4.6 MCP：让工具接入标准化

MCP 是工具接入协议，不是 Agent 架构本身。它解决的是工具如何被发现、描述、调用和复用。

学习内容：

- MCP 解决什么问题：工具发现、参数暴露、资源访问、工具服务复用。
- MCP 和 Function Calling 的关系。
- MCP 的基本对象：Tools / Resources / Prompts。
- MCP Client / Server 的职责。
- 什么时候需要 MCP，什么时候普通函数工具就够了。

#### 4.7 Skill：把重复任务沉淀成能力包

Tool 是原子动作，Skill 是可复用能力包。Skill 可以组合多个工具，也可以封装某类任务的步骤经验、默认流程和失败处理。

学习内容：

- Skill 通常包含：说明、适用场景、工具组合、步骤经验、失败处理。
- Skill vs Tool：Tool 偏原子动作，Skill 偏任务能力。
- Skill vs Workflow：Skill 可被 Agent 灵活调用，Workflow 更偏固定流程。
- 好 Skill 的标准：边界清晰、输入输出明确、可组合、可回退。
- 示例：代码审查 Skill、数据分析 Skill、文档生成 Skill。

#### 4.8 Human-in-the-loop：让人参与高风险决策

Human-in-the-loop 不是单独的 Agent 能力，而是工具机制中的控制点。它用于处理高风险、低置信度、权限不足或连续失败的场景。

学习内容：

- 触发条件：高风险动作、低置信度决策、权限不足、连续失败。
- 介入方式：确认、驳回、修改参数、编辑计划、接管、回滚。
- 确认界面应该展示什么：动作、对象、影响范围、可撤销性。
- 人类反馈如何回到 Agent 状态里。
- 审计记录：确认人、确认时间、确认内容、执行结果。

### 实践里程碑

在课程三的 Agent 基础上扩展工具系统：

- 支持普通 API 工具。
- 支持文件或数据库工具。
- 至少接入一种 MCP 或 Skill 形态的能力。
- 为不同工具设置不同权限策略。
- 对高风险工具调用加入用户确认、审批或人工接管。
- 能记录工具调用日志和失败原因。

## 课程五：场景增强能力

**定位**：讲 RAG / 外部知识接入、Memory、Planning / Workflow Patterns、Reflection、Multi-Agent，但明确它们不是所有 Agent 都必须具备的"核心模块"，而是按场景选择的增强能力。

### 核心问题

- 什么情况下需要 RAG / 外部知识接入？
- 什么情况下需要 Memory？
- 什么情况下需要 Planning / Workflow Patterns？
- Reflection 是模型能力还是系统设计？
- Multi-Agent 是高级能力，还是复杂度陷阱？

### 能力地图

| 能力 | 什么时候需要 | 学什么 |
|---|---|---|
| RAG / 外部知识接入 | 需要外部知识、私有知识库、业务系统数据、长文档问答时 | Embedding、Chunking、Hybrid Search、Rerank、Web Search、数据库/API 查询、引用与事实性 |
| Memory | 需要跨轮、跨会话、跨任务延续时 | 短期记忆、长期记忆、用户偏好、任务状态、记忆更新 |
| Planning / Workflow Patterns | 任务复杂、步骤多、依赖关系强，或流程稳定但局部需要模型决策时 | Plan-and-Execute、任务拆解、动态重规划、Chain、Router、固定流程 + 局部 Agent |
| Reflection | 容易失败、需要自检和重试时 | Self-check、Critique、Retry、错误分类 |
| Multi-Agent | 需要角色分工、并行处理、互相审查时 | Supervisor、Debate、Group Chat、Graph 协作 |

### 5.1 RAG / 外部知识接入

RAG 不是 Agent 必需组件，而是外部知识接入的一种方式。外部知识接入还包括 Web Search、数据库查询、业务系统 API、Knowledge Graph 等。

适合引入外部知识接入的场景：

- 问答依赖私有文档。
- 需要引用来源。
- 模型训练数据不覆盖相关知识。
- 知识变化快，需要外部更新。
- 需要查询业务系统、数据库或实时网页。
- 需要降低幻觉。

主要内容：

- Embedding
- 向量数据库
- Chunking
- Metadata
- Hybrid Search
- Rerank
- Query Rewrite
- Web Search / 数据库查询 / 业务 API 查询
- 引用和溯源
- 失败模式：检索不到、检索错、查询错、引用错、模型忽略外部结果

### 5.2 Memory：状态延续能力

Memory 不是简单地"存聊天记录"。它解决的是 Agent 在不同时间尺度上的状态延续问题。

常见类型：

- 短期记忆：当前会话上下文。
- 工作记忆：当前任务状态、中间结果、待办事项。
- 长期记忆：用户偏好、历史任务、稳定事实。
- 经验记忆：曾经失败或成功的策略。

需要关注：

- 什么时候写入
- 写入什么
- 如何更新
- 如何遗忘
- 如何避免错误记忆污染后续任务

### 5.3 Planning / Workflow Patterns：复杂任务组织能力

Planning 适合多步骤、强依赖、需要长期目标管理的任务。Workflow Patterns 适合流程相对稳定，但某些步骤需要模型判断、工具选择或动态处理的任务。

常见模式：

- 一次性计划，然后执行。
- 边执行边重规划。
- 先粗计划，再逐步细化。
- 人类确认计划后执行。
- Chain：固定顺序执行。
- Router：按输入类型或状态分流。
- 固定流程 + 局部 Agent：主流程稳定，局部步骤交给模型动态决策。
- Graph：复杂状态机和多路径流程。

典型风险：

- 计划看起来合理，但不可执行。
- 计划过细，浪费 token。
- 执行和计划脱节。
- 环境变化后不重规划。
- 固定流程过硬，无法处理异常分支。
- Graph 过复杂，调试和评测成本上升。

### 5.4 Reflection：失败修正能力

Reflection 不是模型天然自带的可靠能力，而是系统设计出来的纠错机制。

常见用法：

- 执行后自检。
- 失败后分类错误。
- 对输出做批判。
- 根据测试结果重试。
- 对工具调用参数进行修正。

典型风险：

- 模型自信地反思错误。
- 反思增加成本但没有提升质量。
- 无限反思导致任务变慢。

### 5.5 Multi-Agent：角色分工能力

Multi-Agent 不应该被过早引入。它适合在单 Agent 遇到明确瓶颈时使用。

适合场景：

- 需要多个专业角色。
- 需要并行处理多个子任务。
- 需要互相审查。
- 需要模拟讨论或辩论。

常见架构：

- Supervisor
- Group Chat
- Debate
- Graph
- 发布订阅

典型风险：

- 通信成本高。
- 角色边界不清。
- 多个 Agent 互相放大错误。
- Debug 难度上升。

### 实践里程碑

在同一个基础 Agent 上分别接入外部知识、Memory、Planning / Workflow Patterns：

- 对比接入前后的任务完成率。
- 分析引入能力后增加了哪些复杂度。
- 能判断一个业务场景是否真的需要该能力。
- 至少写一份"为什么不引入某个能力"的技术判断说明。

## 课程六：Harness 运行时架构深入

**定位**：从"写一个 Agent"进入"设计 Agent Runtime"。

课程三关注最小闭环，课程四关注工具机制，课程五关注场景增强。课程六开始关注工程架构：如何让 Agent 稳定运行、可调试、可评测、可恢复、可治理。

课程三的运行时循环是概念模型，课程六的 Harness 是工程实现。换句话说，Harness 要把 `Context Assembly -> LLM Decision -> Interaction -> Execution -> Observation / Feedback -> State Update -> Stop` 这条循环变成可配置、可追踪、可恢复、可评测的系统。

### 核心问题

- Agent Runtime 由哪些工程环节组成？
- 每个环节最容易出现什么问题？
- 如何通过 Context Engineering、Orchestration、Checkpoint、Evaluation、Guardrails、Observability 解决这些问题？
- Harness 和 Orchestration 的边界是什么？
- 业界常见框架和平台分别在解决哪类工程问题？

### 核心内容

#### 6.1 运行时工程架构总览

可以把 Harness 理解为 Agent 的运行时引擎。它不是某一个具体框架，而是一组工程职责：驱动模型和工具、控制循环和状态、管理评测和观测。

| 层级 | 职责 |
|---|---|
| 驱动层 | 调用模型、执行工具、访问外部服务 |
| 控制层 | 管理循环、停止条件、重试、分支、状态迁移 |
| 管理层 | 会话、权限、配置、日志、评测、观测 |

对应到运行时工程环节，问题和解决机制可以这样看：

| 工程环节 | 常见问题 | 解决机制 / 业界实践 |
|---|---|---|
| Context Assembly | 上下文太长、遗漏约束、工具结果污染下一轮决策 | Context Engineering、上下文预算、信息分层、结果压缩、按需召回 |
| Decision / Orchestration | 步骤组织混乱、循环失控、规划和执行脱节 | Chain、Router、ReAct Loop、Plan-Execute、Graph，常见框架如 LangGraph |
| Execution / State | 工具失败后无法恢复，中断后不能继续，状态不可回放 | Checkpoint、状态机、幂等执行、重试、分支执行 |
| Evaluation | 只看最终答案，无法判断过程是否正确 | 端到端评测、步骤级评测、工具调用评测、LLM-as-Judge，平台如 Braintrust、LangSmith、Ragas |
| Guardrails | 越权调用、Prompt Injection、危险参数、低置信度自动执行 | 输入/输出校验、工具参数校验、权限控制、Human-in-the-loop |
| Observability | 出错后不知道为什么、成本和延迟不可解释 | Trace、Token / Latency / Tool Error 指标、决策归因，平台如 Langfuse、LangSmith、Phoenix |

#### 6.2 Context Engineering：解决上下文组装问题

Context Engineering 不是简单写 Prompt，而是决定哪些信息在什么时候进入上下文。

常见问题：

- 历史消息、工具结果、RAG、Memory 全部塞进上下文，导致噪声过多。
- 关键约束被长上下文淹没。
- 工具结果太长，影响下一轮 LLM Decision。
- 记忆召回不准，反而污染当前任务。

解决思路和实践：

- 按信息类型分层：用户目标、系统约束、任务状态、工具结果、检索结果、错误信息。
- 设定上下文预算，优先保留高密度、高约束、高时效信息。
- 对长工具结果做摘要、截断、分页或转资源。
- RAG 和 Memory 按需召回，不默认注入全部内容。
- 将稳定约束放在稳定位置，避免被后续内容稀释。

#### 6.3 Orchestration：解决任务组织问题

Orchestration 解决的是任务如何被组织；Harness 负责 Agent 如何稳定运行。两者相关，但不等价。

常见问题：

- 所有任务都交给 ReAct Loop，导致步骤不稳定。
- 复杂任务没有计划，执行过程容易漂移。
- 固定 Workflow 太硬，遇到异常分支无法处理。
- 多路径任务状态复杂，难以调试。

解决思路和实践：

- Chain：适合固定步骤。
- Router：适合按输入类型或状态分流。
- ReAct Loop：适合需要动态工具选择和反馈修正的任务。
- Plan-Execute：适合先规划、再执行、再修正的复杂任务。
- Graph：适合状态机、多路径流程和可回放执行，典型框架是 LangGraph。

#### 6.4 Checkpoint 与恢复：解决中断和失败恢复问题

Agent 任务经常是多步骤、长耗时、可中断的。没有 Checkpoint，失败后只能重跑，用户也很难接管。

常见问题：

- 工具执行到一半失败，任务状态丢失。
- 用户中断后无法恢复。
- 重试导致重复写入、重复发送、重复扣费。
- 无法回放某次失败过程。

解决思路和实践：

- 保存关键状态：当前目标、计划、已完成步骤、工具结果、错误信息、用户确认记录。
- 对写操作做幂等设计。
- 支持任务恢复、回放和分支执行。
- 在关键节点设置 Checkpoint，而不是每个 token 都保存。

#### 6.5 Evaluation：解决效果不可验证问题

Agent 评测不能只看最终回答，还要看过程是否正确。

常见问题：

- 最终答案看起来对，但工具选择和中间步骤是错的。
- 没有固定任务集，无法判断改动是否变好。
- LLM-as-Judge 不稳定，评测结果难以复现。
- 只评测单轮回答，忽略多步任务完成率。

解决思路和实践：

- 构建任务集：覆盖成功路径、失败路径、边界条件和权限场景。
- 同时评测最终结果和步骤过程。
- 记录工具选择、参数正确性、步骤数、权限遵守、失败恢复。
- LLM-as-Judge 要配合规则、人工抽检和回归样例。
- 使用 Braintrust、LangSmith、Ragas 等平台管理评测和回归。

#### 6.6 Guardrails：解决越界和风险控制问题

Guardrails 应该嵌入运行时，而不是上线前补一个过滤器。

常见问题：

- 用户输入包含 Prompt Injection。
- 模型计划越权或调用危险工具。
- 工具参数指向错误对象或影响范围过大。
- 低置信度情况下继续自动执行。
- 输出包含隐私、合规或事实性问题。

解决思路和实践：

- 输入阶段：防 Prompt Injection、识别恶意请求。
- 决策阶段：限制越权计划。
- 工具调用前：校验参数、权限、风险等级。
- 工具调用后：校验结果是否可信。
- 输出阶段：校验事实性、隐私、合规。
- Human-in-the-loop：高风险动作前确认，低置信度决策时请求澄清，异常状态下允许人工接管。

#### 6.7 Observability：解决不可调试问题

Agent 的可观测性要能解释"为什么这么做"，而不只是记录最终输出。

常见问题：

- 出错后不知道是哪一步失败。
- 成本和延迟无法拆解。
- 工具错误、模型错误、上下文错误混在一起。
- 用户在哪些节点频繁打断无法分析。

解决思路和实践：

- 记录完整 Trace：输入、上下文、模型输出、工具调用、Observation / Feedback、状态更新。
- 拆解核心指标：Token、Latency、Tool Error、Retry、Step Count、User Intervention。
- 对失败进行分类：工具选择、参数、权限、上下文、模型判断、外部服务。
- 用 Langfuse、LangSmith、Phoenix 等工具做 trace、调试和决策归因。

### 实践里程碑

把课程三的裸 ReAct 循环重构成 Harness：

- 每一步都有 trace。
- 支持 checkpoint。
- 支持中断和恢复。
- 有一组评测集。
- 有失败分类。
- 至少有一个 Guardrail 成功拦截越界行为。

## 课程七：Agent 产品化实践

**定位**：解决"技术能跑"和"用户愿意用"之间的差距。

Agent 产品化的难点不只是模型能力，而是可靠性、交互、成本、安全和组织流程。

### 核心问题

- 用户为什么觉得 Agent 慢？
- 用户为什么不信任 Agent？
- Agent 失败时如何让用户继续掌控局面？
- 如何控制 token 和模型成本？
- 如何证明 Agent 比人工或传统 Workflow 更好？
- 如何让 Agent 的风险可审计？

### 核心内容

### 7.1 交互设计

- 流式输出。
- 任务进度展示。
- 当前步骤解释。
- 中断与恢复。
- Human-in-the-loop：确认、驳回、编辑计划、接管、回滚。
- 可编辑计划。
- 失败后的下一步建议。

重点是让用户知道：

- Agent 正在做什么。
- 为什么要这么做。
- 下一步会影响什么。
- 用户什么时候可以介入。

### 7.2 可靠性工程

- 重试策略。
- 指数退避。
- 工具降级。
- 模型降级。
- 超时控制。
- Rate Limit 管理。
- 幂等设计。
- 任务恢复。

### 7.3 成本控制

- 模型路由。
- 小模型分类，大模型推理。
- Prompt Caching。
- 上下文裁剪。
- 工具结果压缩。
- RAG 检索数量控制。
- 失败重试成本上限。

### 7.4 安全合规

- Prompt Injection 防御。
- 数据脱敏。
- 权限隔离。
- 审计日志。
- 用户授权。
- 高风险动作审批。
- 人工确认记录、审批链路和回滚记录。
- 企业环境中的 GDPR、SOC2、数据驻留要求。

### 7.5 产品指标

Agent 产品不能只看 DAU 或调用次数。

关键指标：

- 任务完成率。
- 用户干预率。
- 平均步骤数。
- 工具调用成功率。
- Token 成本。
- 平均延迟。
- 用户满意度。
- 留存率。
- 人工替代或增强效果。

### 7.6 失败模式清单

常见失败类型：

- 工具选择错误。
- 工具参数错误。
- 工具执行失败。
- 死循环。
- 过早停止。
- 上下文遗漏。
- RAG 检索误导。
- Memory 污染。
- 权限越界。
- 输出无法执行。
- 用户意图理解错误。

### 实践里程碑

选择一个 Agent 项目做产品化改造：

- 有真实或模拟用户任务集。
- 有交互流程。
- 有评测报告。
- 有成本报告。
- 有失败复盘。
- 有安全和权限设计。
- 有下一轮迭代计划。

## 课程八：项目实战与生态跟进

**定位**：用完整项目整合知识，同时建立持续跟进生态的习惯。

项目实战和生态趋势可以放到最后，不应该在初学阶段打散主线。

### 8.1 项目实战

| 项目 | 重点训练 | 合格版 | 进阶版 | 产品版 |
|---|---|---|---|---|
| 个人知识助手 | RAG、Memory、引用、上下文管理 | 文档检索 + 回答 | 混合检索 + 多轮记忆 | 多用户、权限、评测、引用可信 |
| 代码审查 Agent | Tool Use、文件读取、测试调用、置信度、行内反馈 | 读取 diff 后输出 review | 调用测试和静态分析 | GitHub App、行内评论、误报反馈 |
| 领域 Agent 产品 | Harness、权限、评测、产品化、真实反馈 | 核心流程可用 | 覆盖 20+ 场景 | 有真实用户、监控、迭代闭环 |

### 8.2 每个项目的交付物

- 可运行代码。
- 设计文档。
- 工具清单。
- 权限设计。
- 评测集。
- Trace 示例。
- 失败复盘。
- 成本估算。
- 下一步产品计划。

### 8.3 生态跟进

生态内容是持续关注，不是初学阶段的主线必修。

建议关注：

- MCP / A2A。
- Code Agent。
- Multi-Agent 框架。
- Agent RL。
- 新模型的工具调用、长上下文、结构化输出能力。
- Agent 框架和 Runtime 的演进。

### 实践里程碑

完成至少一个产品版项目，并能回答：

- 为什么这个场景适合 Agent？
- 为什么不是传统 Workflow？
- 用了哪些工具？
- 哪些能力是必需的，哪些能力是可选增强？
- 如何评测？
- 如何上线？
- 如何持续改进？

## 推荐学习顺序

如果时间有限，可以按以下方式学习：

### 入门路径

```text
课程一 -> 课程三 -> 课程四 -> 做一个小 Agent
```

目标是先做出能跑的最小闭环。

### 工程路径

```text
课程三 -> 课程四 -> 课程六 -> 课程七
```

目标是理解如何把 Agent 做稳定、可评测、可上线。

### 产品路径

```text
课程一 -> 课程七 -> 课程八
```

目标是理解什么场景适合 Agent，以及如何判断 Agent 产品是否真的创造价值。

### 深入路径

```text
课程一 -> 课程二 -> 课程三 -> 课程四 -> 课程五 -> 课程六 -> 课程七 -> 课程八
```

目标是完整掌握 Agent 产品开发从认知到落地的全链路。
