# Agent 学习路线图（V3）：从最小闭环到产品化系统

## 课程基本介绍

这份路线图面向希望系统学习 Agent 产品开发的人，目标是从"第一次接触 Agent"逐步走到"能够设计、实现、评估并上线 Agent 产品"。

这份路线图将 Agent 学习内容分为不同层级：最小闭环、工具系统、场景增强能力、运行时架构和产品化能力。

课程主线：

```text
建立直觉 -> 理解范式演进 -> 掌握最小 Agent 闭环
    -> 深入工具与能力接入 -> 学习场景增强能力
    -> 设计 Harness 运行时 -> 完成产品化实践
    -> 项目实战与生态跟进
```

核心观点是：

```text
Agent = LLM 驱动的目标导向运行时 + 可调用工具 + 可管理上下文 + 可验证执行过程
```

RAG、Memory、Planning、Reflection、Multi-Agent、MCP、Skill 都很重要，但它们不是同一层级的东西，也不是所有 Agent 应用的必选项。它们应该围绕最小 Agent 闭环展开：有些属于工具接入，有些属于场景增强，有些属于运行时架构，有些属于产品化能力。

## 设计原则

- **先感性，再结构化**：先体验真实产品，再建立抽象概念，避免一开始陷入术语堆砌。
- **先最小闭环，再扩展能力**：先实现一个能多步决策、调用工具、观察结果并停止的 Agent，再引入 RAG、Memory、Planning 等增强能力。
- **把 Tool Use 作为独立主线**：工具调用是 Agent 从"会说"到"会做"的关键，应该完整讲清工具定义、工具选择、工具执行、权限、MCP、Skill 和 Human-in-the-loop。
- **把 RAG / Memory 放回场景中**：它们不是所有 Agent 的核心模块，而是根据业务场景选择的能力。
- **从运行时理解架构**：Harness、Context Engineering、Evaluation、Guardrails、Observability 都应该围绕 Agent 运行时展开。
- **最终回到产品**：学习目标不是写出 Demo，而是理解如何让 Agent 可靠、可控、低成本、安全，并真正被用户使用。

## 课程一：初识 Agent

**定位**：建立感性认知，先看见 Agent 和普通 Chatbot / Workflow 的区别。

### 核心问题

- Agent 到底是什么？
- 它和 Chatbot、Workflow 的区别在哪里？
- 一个产品在什么地方体现出 Agent 特征？
- 用户为什么会觉得 Agent 有用、失控、惊喜或不可靠？

### 核心内容

- **Agent 基本理解**：Agent 不是"更聪明的聊天机器人"，而是能够围绕目标持续决策和执行的系统。
- **Agent vs Chatbot vs Workflow**：
  - Chatbot 主要负责对话和回答。
  - Workflow 按固定流程执行。
  - Agent 会根据目标、状态和反馈决定下一步。
- **观察重点**：不要只看产品是否用了 LLM，而要看"谁决定下一步"。
- **主流产品体验**：
  - 国外产品：Claude Code、Cursor、Devin/OpenHands、GPT-Researcher、浏览器自动化 Agent、数据分析 Agent 等。
  - 国内可直接体验的产品：Kimi、豆包、通义千问、腾讯元宝、智谱清言、文小言 / 文心一言、秘塔 AI 搜索、扣子 Coze、TRAE / MarsCode、腾讯云 CodeBuddy 等。
  - 观察重点：它是否会自己拆解任务、调用工具、生成计划、读写文件、检索资料、跨应用行动；还是主要停留在问答、搜索或固定工作流。
- **产品拆解维度**：
  - 用户如何输入目标
  - 系统是否会主动规划
  - 什么时候调用工具
  - 什么时候请求用户确认
  - 失败后如何恢复
  - 是否能判断任务类型
  - 哪些地方让人惊喜
  - 哪些地方让人失望
  - 如果重做，会如何改进

### 实践里程碑

完成 3 款 Agent 产品拆解，每份拆解至少回答：

- 它解决了什么用户任务？
- 它什么时候像 Agent，什么时候只是 Chatbot 或 Workflow？
- 它如何调用工具？
- 它在哪里失败？
- 如果产品化，最大风险是什么？

## 课程二：Agent 范式演进

**定位**：理解 Agent 为什么会出现，以及它解决了 LLM 的什么问题。

这门课按问题演进来理解 Agent：每一次新方法的出现，都是为了解决前一阶段暴露出来的问题。

### 核心问题

- 传统 Agent 为什么没有成为大众产品？
- LLM 出现后，Agent 的决策核心发生了什么变化？
- ReAct 为什么重要？
- Function Calling / Toolformer / Plugins 解决了什么问题？
- Planning、Reflection、Multi-Agent 分别是在什么背景下出现的？

### 核心内容

- **传统 Agent 的历史背景**：
  - 强化学习 Agent
  - 软件 Agent
  - BDI 模型
  - Russell & Norvig 的 Agent 类型
  - 传统 Agent 的根本限制：封闭规则空间、泛化能力弱、工程成本高
- **LLM 带来的范式变化**：
  - 从"人工写规则"变成"模型理解目标并生成下一步"
  - 从"固定流程"变成"动态决策"
  - 从"只回答"走向"能调用外部能力"
- **ReAct 的意义**：
  - 把推理、行动、观察放入同一个循环
  - 让模型可以边思考、边行动、边根据结果修正
- **Toolformer / Plugins / Function Calling**：
  - 让 LLM 不再只依赖训练数据
  - 让工具调用成为模型能力的一部分
  - 让工具定义、参数生成和结果返回变得工程化
- **Planning 的出现背景**：
  - 复杂任务无法一步完成
  - 需要任务拆解、依赖排序和阶段性目标
- **Reflection 的出现背景**：
  - 模型和工具调用都会失败
  - 系统需要自检、重试、纠错和复盘
- **Multi-Agent 的出现背景**：
  - 单 Agent 面临角色混淆、上下文拥挤、工具数量过多、复杂任务难以并行等问题
  - 多 Agent 通过角色分工、互相审查和并行处理解决部分瓶颈

### 实践里程碑

能用以下框架讲清一个 Agent 技术或产品公告：

```text
之前有什么问题 -> 新方法解决了什么 -> 新方法带来了什么新问题
```

## 课程三：最小 Agent 闭环

**定位**：不急着讲 RAG、Memory、MCP，先实现一个最小但完整的 Agent。

这门课是整条路线的技术起点。学习者必须先理解 Agent 最小闭环，后面所有能力都是围绕这个闭环扩展。

### 核心公式

```text
Agent = LLM 驱动的目标导向运行时 + 工具调用 + 上下文状态 + 循环控制
```

### 最小运行链路

```text
User Goal
    -> Context Assembly
    -> LLM Decision
    -> Tool Call
    -> Observation
    -> State Update
    -> Continue or Stop
```

### 运行时循环总览

理解 Agent 时需要一张运行时地图。可以把最小 Agent 看成一个不断推进的控制循环：

```text
                         read / write
                  +-----------------------+
                  | State / Memory Store  |
                  | task state, history,  |
                  | optional persistence  |
                  +----------^------------+
                             |
                             v
User Goal -> Context Assembly -> Thought -> Action -> Execution
                 ^                |          |          |
                 |                |          |          v
                 |                |          +---- Observation
                 |                |                    |
                 +---------- State Update <------------+
                             |
                             v
                      Continue or Stop
```

这张图的作用是给后续知识定位：

- **Context Assembly**：决定哪些信息进入模型上下文，包括用户目标、系统约束、历史消息、工具结果、检索结果和记忆召回。
- **Thought**：模型根据当前上下文判断下一步，可能是回答、调用工具、请求确认、修正计划或停止。
- **Action**：模型选择工具并生成参数。
- **Execution**：运行时执行工具，处理权限、超时、重试和错误。
- **Observation**：把工具结果转成模型可理解的信息，必要时摘要、截断或结构化。
- **State Update**：更新任务状态、历史记录、错误信息和中间结果；如果场景需要，再写入长期 Memory。
- **Continue or Stop**：根据任务完成度、最大步数、置信度、用户确认点和失败策略决定继续还是结束。

这里要特别区分：最小 Agent 必须有任务状态和循环控制，但不一定需要长期 Memory；RAG 也不是循环的必需节点，而是可以作为一种检索工具参与 Context Assembly 或 Action。

### 核心内容

| 组成 | 学什么 | 为什么重要 |
|---|---|---|
| 决策核心 | LLM、System Prompt、结构化输出 | 决定 Agent 如何理解目标和下一步 |
| 行动接口 | Tool Use、Function Calling、JSON Schema | 决定 Agent 能不能真正做事 |
| 运行循环 | Context Assembly / Thought / Action / Observation / State Update / Stop | 决定 Agent 如何多步执行 |
| 状态管理 | 当前任务、历史消息、工具结果、错误信息 | 决定 Agent 是否能连续工作 |
| 失败处理 | 参数错误、工具失败、死循环、提前停止 | 决定 Agent 是否可控 |

### 重点知识

- **LLM 作为决策核心**：
  - 模型不是 Agent 本身，只是 Agent 的决策引擎。
  - Agent 的行为由模型能力、上下文、工具集合、系统提示词和运行时控制共同决定。
- **System Prompt 的作用**：
  - 定义角色
  - 定义行为边界
  - 定义输出格式
  - 定义工具使用规则
- **结构化输出**：
  - JSON Mode
  - Structured Outputs
  - Function Calling 参数生成
  - 输出解析和错误恢复
- **ReAct Loop**：
  - Context Assembly：组装当前这一步需要的上下文。
  - Thought：分析当前状态，决定下一步。
  - Action：选择工具并生成参数。
  - Execution：执行工具并处理权限、超时和错误。
  - Observation：接收工具结果。
  - State Update：更新任务状态和中间结果。
  - Stop：判断任务是否完成。
- **循环控制**：
  - 最大步数
  - 最大耗时
  - 重复动作检测
  - 用户确认点
  - 失败退出策略

### 实践里程碑

实现一个最小 ReAct Agent，要求：

- 支持至少 3 个工具。
- 能完成多步任务。
- 能记录每一步的决策、工具调用、工具结果和停止原因。
- 有基础错误处理：工具不存在、参数不合法、工具执行失败、循环超限。
- 用 10 条测试任务验证，成功率达到 70% 以上。

## 课程四：工具与能力接入

**定位**：把 Tool Use 作为独立主线讲清楚，因为这是 Agent 从"会说"到"会做"的关键。

Function Calling、Tool Use、MCP、Skill、Computer Use、Human-in-the-loop 都属于工具与能力接入体系。这门课把它们放在同一条主线里，帮助学习者建立统一理解。

### 核心问题

- 工具应该怎么定义，模型才会正确使用？
- 模型什么时候该调用工具？
- 工具结果应该如何返回给模型？
- 高风险工具如何做权限控制？
- MCP 到底属于什么层级？
- Skill 和 Tool 有什么区别？
- GUI 操作和浏览器自动化是不是工具调用？

### 核心内容

| 主题 | 内容 |
|---|---|
| 工具定义 | 工具名、描述、参数 Schema、返回值设计 |
| 工具选择 | 什么时候用工具、用哪个工具、参数怎么填 |
| 工具结果 | Observation 如何返回给模型，如何压缩、裁剪、纠错 |
| 权限边界 | 只读工具、写入工具、危险工具、审批机制 |
| MCP | MCP 作为工具接入协议，不是新的 Agent 核心模块 |
| Skill | Skill 作为可复用能力包：工具 + 指令 + 工作流经验 |
| Computer Use | 当 API 不存在时，用浏览器或 GUI 操作作为特殊工具 |
| Human-in-the-loop | 高风险动作前的用户确认、审批、回滚 |

### 4.1 工具定义

- 工具名要短、明确、可区分。
- 工具描述要告诉模型：
  - 这个工具做什么
  - 什么时候应该用
  - 什么时候不应该用
  - 输入参数有什么约束
  - 返回结果代表什么
- 参数 Schema 要避免模糊字段。
- 返回值要适合模型继续推理，而不是只适合程序读取。

### 4.2 工具选择

工具选择的核心问题是三件事：

```text
什么时候该用工具？
应该用哪个工具？
参数应该怎么填？
```

常见策略：

- 模型自主选择。
- 规则路由。
- 小模型先分类，大模型再填参。
- 固定 Workflow 中嵌入局部 Agent 决策。
- 对高风险工具强制用户确认。

### 4.3 工具执行与结果回填

- 工具调用需要超时控制。
- 工具错误要结构化返回。
- 工具结果不能无脑塞回上下文。
- 长结果要摘要、截断或转成可检索资源。
- Observation 应该突出对下一步决策最有用的信息。

### 4.4 权限与安全

工具按风险分层：

| 类型 | 例子 | 策略 |
|---|---|---|
| 只读工具 | 搜索、读取文件、查询数据库 | 默认允许，记录日志 |
| 低风险写入 | 创建草稿、生成报告 | 可自动执行，可撤销 |
| 高风险写入 | 发邮件、提交代码、修改生产数据 | 用户确认 |
| 危险操作 | 删除数据、支付、权限变更 | 强审批、沙箱、审计 |

### 4.5 MCP 的位置

MCP 是工具接入协议，不是另一种 Agent 核心能力。

它解决的问题是：

- 工具如何被发现
- 工具如何暴露参数
- 工具如何返回结果
- Agent 和工具服务如何解耦
- 多个 Agent 产品如何复用同一套工具生态

### 4.6 Skill 的位置

Skill 可以理解成一种可复用能力包，通常包含：

- 指令
- 工具组合
- 默认流程
- 失败处理经验
- 示例
- 约束和边界

Tool 是原子动作，Skill 是可复用能力。Skill 可以调用多个 Tool，也可以封装一段成熟的任务经验。

### 实践里程碑

在课程三的 Agent 基础上扩展工具系统：

- 支持普通 API 工具。
- 支持文件或数据库工具。
- 至少接入一种浏览器、GUI、MCP 或 Skill 形态的能力。
- 为不同工具设置不同权限策略。
- 对高风险工具加入用户确认。
- 能记录工具调用日志和失败原因。

## 课程五：场景增强能力

**定位**：讲 RAG、Memory、Planning、Reflection、Multi-Agent，但明确它们不是所有 Agent 都必须具备的"核心模块"，而是按场景选择的增强能力。

### 核心问题

- 什么情况下需要 RAG？
- 什么情况下需要 Memory？
- 什么情况下需要 Planning？
- Reflection 是模型能力还是系统设计？
- Multi-Agent 是高级能力，还是复杂度陷阱？

### 能力地图

| 能力 | 什么时候需要 | 学什么 |
|---|---|---|
| RAG | 需要外部知识、私有知识库、长文档问答时 | Embedding、Chunking、Hybrid Search、Rerank、引用与事实性 |
| Memory | 需要跨轮、跨会话、跨任务延续时 | 短期记忆、长期记忆、用户偏好、任务状态、记忆更新 |
| Planning | 任务复杂、步骤多、依赖关系强时 | Plan-and-Execute、任务拆解、动态重规划 |
| Reflection | 容易失败、需要自检和重试时 | Self-check、Critique、Retry、错误分类 |
| Multi-Agent | 需要角色分工、并行处理、互相审查时 | Supervisor、Debate、Group Chat、Graph 协作 |

### 5.1 RAG：知识接入能力

RAG 不是 Agent 必需组件，而是知识接入的一种方式。

适合使用 RAG 的场景：

- 问答依赖私有文档。
- 需要引用来源。
- 模型训练数据不覆盖相关知识。
- 知识变化快，需要外部更新。
- 需要降低幻觉。

主要内容：

- Embedding
- 向量数据库
- Chunking
- Metadata
- Hybrid Search
- Rerank
- Query Rewrite
- 引用和溯源
- RAG 失败模式：检索不到、检索错、引用错、模型忽略检索结果

### 5.2 Memory：状态延续能力

Memory 不是简单地"存聊天记录"。它解决的是 Agent 在不同时间尺度上的状态延续问题。

常见类型：

- 短期记忆：当前会话上下文。
- 工作记忆：当前任务状态、中间结果、待办事项。
- 长期记忆：用户偏好、历史任务、稳定事实。
- 经验记忆：曾经失败或成功的策略。

需要关注：

- 什么时候写入
- 写入什么
- 如何更新
- 如何遗忘
- 如何避免错误记忆污染后续任务

### 5.3 Planning：复杂任务拆解能力

Planning 适合多步骤、强依赖、需要长期目标管理的任务。

常见模式：

- 一次性计划，然后执行。
- 边执行边重规划。
- 先粗计划，再逐步细化。
- 人类确认计划后执行。

典型风险：

- 计划看起来合理，但不可执行。
- 计划过细，浪费 token。
- 执行和计划脱节。
- 环境变化后不重规划。

### 5.4 Reflection：失败修正能力

Reflection 不是模型天然自带的可靠能力，而是系统设计出来的纠错机制。

常见用法：

- 执行后自检。
- 失败后分类错误。
- 对输出做批判。
- 根据测试结果重试。
- 对工具调用参数进行修正。

典型风险：

- 模型自信地反思错误。
- 反思增加成本但没有提升质量。
- 无限反思导致任务变慢。

### 5.5 Multi-Agent：角色分工能力

Multi-Agent 不应该被过早引入。它适合在单 Agent 遇到明确瓶颈时使用。

适合场景：

- 需要多个专业角色。
- 需要并行处理多个子任务。
- 需要互相审查。
- 需要模拟讨论或辩论。

常见架构：

- Supervisor
- Group Chat
- Debate
- Graph
- 发布订阅

典型风险：

- 通信成本高。
- 角色边界不清。
- 多个 Agent 互相放大错误。
- Debug 难度上升。

### 实践里程碑

在同一个基础 Agent 上分别接入 RAG、Memory、Planning：

- 对比接入前后的任务完成率。
- 分析引入能力后增加了哪些复杂度。
- 能判断一个业务场景是否真的需要该能力。
- 至少写一份"为什么不引入某个能力"的技术判断说明。

## 课程六：Harness 运行时架构深入

**定位**：从"写一个 Agent"进入"设计 Agent Runtime"。

课程三关注最小闭环，课程四关注工具接入，课程五关注场景增强。课程六开始关注工程架构：如何让 Agent 稳定运行、可调试、可评测、可恢复、可治理。

课程三的运行时循环是概念模型，课程六的 Harness 是工程实现。换句话说，Harness 要把 `Context Assembly -> Thought -> Action -> Execution -> Observation -> State Update -> Stop` 这条循环变成可配置、可追踪、可恢复、可评测的系统。

### 核心问题

- 一个 Agent Runtime 应该分成哪些层？
- Context 应该如何组装？
- Orchestration 和 Harness 有什么区别？
- 如何支持中断、恢复和回放？
- 如何评测一个 Agent？
- 如何让 Agent 可观测、可调试？

### 核心内容

| 主题 | 关键问题 |
|---|---|
| Harness 分层 | 驱动层、控制层、管理层分别负责什么 |
| Context Engineering | Prompt、历史消息、工具结果、RAG、Memory 如何放进同一个上下文窗口 |
| Orchestration | Chain、Router、ReAct Loop、Plan-Execute、Graph 的适用场景 |
| Checkpoint | 中断、恢复、回放、分支执行 |
| Evaluation | 端到端评测、步骤级评测、工具调用评测、LLM-as-Judge |
| Guardrails | 输入防御、输出校验、工具参数校验、权限控制 |
| Observability | Trace、Token、Latency、Tool Error、决策归因 |

### 6.1 Harness 分层

可以把 Harness 理解为 Agent 的运行时引擎。

推荐分层：

| 层级 | 职责 |
|---|---|
| 驱动层 | 调用模型、执行工具、访问外部服务 |
| 控制层 | 管理循环、停止条件、重试、分支、状态迁移 |
| 管理层 | 会话、权限、配置、日志、评测、观测 |

对应到课程三的运行时循环：

| 运行时环节 | Harness 中的工程责任 |
|---|---|
| Context Assembly | 上下文组装、预算控制、RAG/Memory 召回、Prompt 模板渲染 |
| Thought | 模型选择、结构化输出、推理策略、低置信度处理 |
| Action | 工具路由、参数校验、权限判断、用户确认 |
| Execution | 工具执行、沙箱、超时、重试、并发控制 |
| Observation | 结果摘要、错误归一化、返回格式控制、证据保留 |
| State Update | 会话状态、任务状态、Checkpoint、长期记忆写入策略 |
| Continue or Stop | 停止条件、循环检测、失败退出、人工接管 |

### 6.2 Context Engineering

Context Engineering 不是简单写 Prompt，而是决定哪些信息在什么时候进入上下文。

需要管理的信息包括：

- System Prompt
- 用户目标
- 历史消息
- 当前任务状态
- 工具结果
- RAG 检索结果
- Memory 召回内容
- 计划
- 错误信息
- Guardrails 反馈

关键原则：

- 信息密度优先。
- 最近信息不一定最重要。
- 约束信息要放在稳定位置。
- 工具结果要压缩后进入上下文。
- 长期记忆必须按需召回。

### 6.3 Orchestration

Orchestration 解决的是任务如何被组织。

常见模式：

- Chain：固定步骤。
- Router：按输入类型分流。
- ReAct Loop：动态决策。
- Plan-Execute：先规划再执行。
- Graph：复杂状态机和多路径流程。

### 6.4 Evaluation

Agent 评测不能只看最终回答，还要看过程。

评测维度：

- 任务完成率
- 工具选择是否正确
- 参数是否正确
- 步骤数是否合理
- 是否遵守权限
- 是否引用正确来源
- 是否能失败恢复
- 用户是否需要频繁干预

### 6.5 Guardrails

Guardrails 应该嵌入运行时，而不是上线前补一个过滤器。

常见位置：

- 输入阶段：防 Prompt Injection、识别恶意请求。
- 决策阶段：限制越权计划。
- 工具调用前：校验参数、权限、风险等级。
- 工具调用后：校验结果是否可信。
- 输出阶段：校验事实性、隐私、合规。

### 6.6 Observability

Agent 的可观测性要能回答：

- 它为什么这么做？
- 哪一步失败了？
- 花了多少钱？
- 哪个工具最慢？
- 哪个 Prompt 最容易出错？
- 用户在哪些节点频繁打断？

### 实践里程碑

把课程三的裸 ReAct 循环重构成 Harness：

- 每一步都有 trace。
- 支持 checkpoint。
- 支持中断和恢复。
- 有一组评测集。
- 有失败分类。
- 至少有一个 Guardrail 成功拦截越界行为。

## 课程七：Agent 产品化实践

**定位**：解决"技术能跑"和"用户愿意用"之间的差距。

Agent 产品化的难点不只是模型能力，而是可靠性、交互、成本、安全和组织流程。

### 核心问题

- 用户为什么觉得 Agent 慢？
- 用户为什么不信任 Agent？
- Agent 失败时如何让用户继续掌控局面？
- 如何控制 token 和模型成本？
- 如何证明 Agent 比人工或传统 Workflow 更好？
- 如何让 Agent 的风险可审计？

### 核心内容

### 7.1 交互设计

- 流式输出。
- 任务进度展示。
- 当前步骤解释。
- 中断与恢复。
- 用户确认节点。
- 可编辑计划。
- 失败后的下一步建议。

重点是让用户知道：

- Agent 正在做什么。
- 为什么要这么做。
- 下一步会影响什么。
- 用户什么时候可以介入。

### 7.2 可靠性工程

- 重试策略。
- 指数退避。
- 工具降级。
- 模型降级。
- 超时控制。
- Rate Limit 管理。
- 幂等设计。
- 任务恢复。

### 7.3 成本控制

- 模型路由。
- 小模型分类，大模型推理。
- Prompt Caching。
- 上下文裁剪。
- 工具结果压缩。
- RAG 检索数量控制。
- 失败重试成本上限。

### 7.4 安全合规

- Prompt Injection 防御。
- 数据脱敏。
- 权限隔离。
- 审计日志。
- 用户授权。
- 高风险动作审批。
- 企业环境中的 GDPR、SOC2、数据驻留要求。

### 7.5 产品指标

Agent 产品不能只看 DAU 或调用次数。

关键指标：

- 任务完成率。
- 用户干预率。
- 平均步骤数。
- 工具调用成功率。
- Token 成本。
- 平均延迟。
- 用户满意度。
- 留存率。
- 人工替代或增强效果。

### 7.6 失败模式清单

常见失败类型：

- 工具选择错误。
- 工具参数错误。
- 工具执行失败。
- 死循环。
- 过早停止。
- 上下文遗漏。
- RAG 检索误导。
- Memory 污染。
- 权限越界。
- 输出无法执行。
- 用户意图理解错误。

### 实践里程碑

选择一个 Agent 项目做产品化改造：

- 有真实或模拟用户任务集。
- 有交互流程。
- 有评测报告。
- 有成本报告。
- 有失败复盘。
- 有安全和权限设计。
- 有下一轮迭代计划。

## 课程八：项目实战与生态跟进

**定位**：用完整项目整合知识，同时建立持续跟进生态的习惯。

项目实战和生态趋势可以放到最后，不应该在初学阶段打散主线。

### 8.1 项目实战

| 项目 | 重点训练 | 合格版 | 进阶版 | 产品版 |
|---|---|---|---|---|
| 个人知识助手 | RAG、Memory、引用、上下文管理 | 文档检索 + 回答 | 混合检索 + 多轮记忆 | 多用户、权限、评测、引用可信 |
| 代码审查 Agent | Tool Use、文件读取、测试调用、置信度、行内反馈 | 读取 diff 后输出 review | 调用测试和静态分析 | GitHub App、行内评论、误报反馈 |
| 领域 Agent 产品 | Harness、权限、评测、产品化、真实反馈 | 核心流程可用 | 覆盖 20+ 场景 | 有真实用户、监控、迭代闭环 |

### 8.2 每个项目的交付物

- 可运行代码。
- 设计文档。
- 工具清单。
- 权限设计。
- 评测集。
- Trace 示例。
- 失败复盘。
- 成本估算。
- 下一步产品计划。

### 8.3 生态跟进

生态内容是持续关注，不是初学阶段的主线必修。

建议关注：

- MCP / A2A。
- Computer Use。
- Code Agent。
- Multi-Agent 框架。
- Agent RL。
- 新模型的工具调用、长上下文、结构化输出能力。
- Agent 框架和 Runtime 的演进。

### 实践里程碑

完成至少一个产品版项目，并能回答：

- 为什么这个场景适合 Agent？
- 为什么不是传统 Workflow？
- 用了哪些工具？
- 哪些能力是必需的，哪些能力是可选增强？
- 如何评测？
- 如何上线？
- 如何持续改进？
