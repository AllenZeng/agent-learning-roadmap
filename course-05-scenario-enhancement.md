# 课程五：场景增强能力

## 课程导言

课程三和课程四之后，你已经有了一个能跑的 Agent——LLM 决策 + 工具调用 + State 管理 + 循环控制。它处理简单任务很稳：读文件、做摘要、搜索替换，都没问题。

但一旦把它放到真实场景里用上几周，你就会撞上五种依次出现的崩溃：知识不在模型里、关了会话就失忆、长任务开始漂移、看到错误却不修正、自己审自己说没问题。

这五种崩溃分别对应五类增强能力：RAG（外部知识接入）、Memory（状态延续）、Planning（任务组织）、Reflection（失败修正）、Multi-Agent（角色分工）。它们不是 Agent 的"必选核心模块"——一个最小闭环不需要它们也能跑。它们是**按场景问题选择性引入的增强能力**。

本课不从概念名词开始，而从五个真实的崩溃故事出发。每一章都沿着同一条路径展开：

```text
问题代入（怎么崩溃的）→ 技术背景（这个能力怎么来的）→ 思考（设计者当时在想什么）
→ 方案（核心链路 + 代码骨架）→ 迭代路径（从最小版本到生产级）→ 判断边界（什么时候别用）
```

学完后，你不需要把五类能力全部做深，但你面对一个新场景时应该能判断：

- 当前场景到底遇到了什么问题？
- 这个问题是否真的需要某个增强能力？
- 引入后会增加哪些工程复杂度？
- 如果要引入，应该从哪个最小版本开始？
- 什么情况下应该明确不引入？

---

## 学习目标

学完本课，你将能够：

1. **从问题出发判断增强能力**：不再把 RAG、Memory、Planning、Reflection、Multi-Agent 当成固定模块清单
2. **区分外部知识接入与 Agent 核心闭环**：知道什么时候需要 RAG / Web Search / 数据库 / 业务 API，什么时候不需要
3. **设计 Memory 的读写边界**：区分任务状态、会话历史、长期记忆和用户偏好，避免记忆污染
4. **为复杂任务选择合适组织模式**：在 Chain、Router、ReAct Loop、Plan-Execute、Graph 之间做场景判断
5. **正确理解 Reflection**：把它看成带触发条件和停止条件的系统纠错机制，而不是模型天然可靠能力
6. **判断 Multi-Agent 是否值得引入**：识别角色分工、并行探索、互相审查的真实需求和复杂度陷阱
7. **写出能力引入说明**：用"问题、方案、代价、迭代、边界"说明为什么引入或不引入某个能力

## 本课在路线图中的位置

课程三讲最小闭环，课程四讲工具机制。课程五站在"场景问题"这一层，回答：

- 最小闭环在什么场景下会不够用？
- 不够用时应该引入哪类增强能力？
- 引入增强能力后，系统复杂度会增加在哪里？
- 如何从最小版本开始迭代，而不是一次性堆完整架构？

本课关注增强能力的判断和设计，不深入展开完整工程架构。运行时层面的 Context Engineering、Checkpoint、Evaluation、Guardrails、Observability 会在课程六继续讲。

| 本课关注 | 本课不关注 |
|---|---|
| 按场景判断是否需要增强能力 | 把每个能力都实现到商用深度 |
| 能力引入带来的收益和复杂度 | 具体框架或平台的完整教程 |
| 最小版本和迭代路径 | 完整 Runtime 架构治理 |
| 写出为什么引入或不引入的说明 | 产品指标、商业化和组织运营 |

## 第一章：为什么要从场景问题学习增强能力

### 1.1 问题代入：最小 Agent 为什么开始不够用

2024 年初，你花了两周时间做了一个个人知识助手 Agent。它跑在课程三的最小闭环上——LLM 决策 + 三个工具（读文件、写文件、搜索）+ State 管理 + 循环控制。你的 30 篇学习笔记放在 `notes/` 目录下，Agent 可以按需读取。

第一个月，你用它很顺手：

```text
你："总结一下 notes/agent-tool-use.md 的核心观点。"
Agent：读取文件 → 理解内容 → 输出摘要。干净利落。
```

但第二个月开始，问题一个接一个冒出来。

**第一个崩溃：它不知道我的笔记里写了什么。**

你问："根据我所有笔记，Tool Use 和 Memory 的设计哲学有什么根本不同？"

Agent 读了 `agent-tool-use.md`，但没读其他 29 篇。它不知道哪些笔记和这个问题相关，也没办法在所有笔记里搜索。它给你的回答……看起来合理，但那是它凭训练数据"猜"的，不是基于你的笔记。你翻出笔记原文对照，发现关键论据全对不上。

这个问题的本质是：**答案依赖外部资料，但 Agent 没有"接入外部知识"的能力。** 这引出了第二章要讲的 RAG / 外部知识接入。

**第二个崩溃：它永远不记得我上回说过什么。**

有一天你花了 20 分钟告诉 Agent："以后写技术文章，先给大纲我确认，再展开正文。语气直接，不要营销化。"

第二天你打开新会话："帮我写一篇 Agent Memory 的技术文章。"

它给了你一篇充满"在当今这个 AI 时代……"开头的营销风文章，没有先给你大纲。

你意识到：**Agent 的状态在会话关闭后就蒸发了。** 用户的偏好、项目的约定、上回任务的进度——所有这些都需要某种"跨会话延续"的机制。这引出了第三章要讲的 Memory。

**第三个崩溃：任务一长，它就开始漂。**

你说："帮我做发布准备：检查 README 完整性、跑一遍测试、整理 changelog、生成 release checklist。"

Agent 开始执行。它读了 README，写了一段评论。然后……它开始读 `notes/` 目录下的其他文件。然后又去搜了点什么。你等了五分钟，发现它偏离原始目标，已经不知道在做什么了。你叫停它，检查 trace——它在第 4 步之后就没再碰过和"发布"相关的任何操作。

这个问题的本质不是工具不好用，而是**裸 ReAct 循环没有"任务结构"的概念**。它每步都在临场反应，但没有全局视图。这引出了第四章要讲的 Planning / Workflow Patterns。

**第四个崩溃：错了，但它继续往前走。**

你让 Agent 改一段代码，改完后跑测试。测试输出：

```text
TypeError: Cannot read properties of undefined (reading 'files')
```

Agent 看到了这个输出，然后说：

```text
修改已完成，代码可以正常运行。
```

你盯着屏幕，不知道该笑还是该气。它明明看到了错误信息，但它没有把"测试失败"当作一个需要修正的信号。它只是在"生成下一个 Token"。

这个问题的本质是：**Agent 缺少"失败→分析→修正→重试"的闭环机制。** 这引出了第五章要讲的 Reflection。

**第五个崩溃：你让它审查自己的工作，它说"没问题"。**

你让 Agent 写一份技术方案，然后从安全角度审一遍。它写完了，你让它审查。它输出了：

```text
经过审查，该方案在安全方面没有明显问题。
```

你作为人类看了一眼，立刻发现了三个安全隐患。Agent 不是不聪明——它只是在"写方案"和"审方案"时用了同一个视角、同一个上下文。自己审自己，就像让学生给自己的试卷打分。

这个问题的本质是：**单一 Agent 的角色开始过载，不同视角需要不同的上下文和判断标准。** 这引出了第六章要讲的 Multi-Agent。

---

这五个崩溃不是虚构的——它们是 2023-2024 年间 Agent 开发者反复遇到的真问题。而且它们有一个共同特点：**都不是"最小闭环公式错了"，而是场景复杂度升高后，最小闭环需要被增强。**

课程三和课程四给了你一个能跑的骨架。课程五要解决的问题是：当这个骨架在真实场景中开始发出咯吱声时，你应该往哪个方向加固。

### 1.2 对问题的思考：能力是对问题的响应

上面五个崩溃，分别击中了最小 Agent 的五块短板。把它们整理成一张表，你会发现每一项增强能力都不是”技术圈的新玩具”——它们是对具体失败的直接回应：

| 崩溃场景 | 根因 | 直觉表现 | 对应的增强能力 |
|---|---|---|---|
| 笔记问题 | Agent 缺少外部事实来源 | 回答像猜的，缺引用，知识过期 | RAG / 外部知识接入 |
| 偏好问题 | Agent 无法跨会话延续状态 | 用户反复补充偏好，任务中断后无法继续 | Memory |
| 发布准备 | 长任务缺乏全局组织 | 漏步骤、重复、漂移、提前停止 | Planning / Workflow Patterns |
| 测试失败 | Agent 不会从失败中修正 | 格式错、测试失败、用户驳回后仍继续 | Reflection |
| 自我审查 | 单 Agent 角色过载 | 探索、执行、审查混在同一视角，质量不稳 | Multi-Agent |

注意这五个问题的出现顺序。在 1.1 的故事里，它们是**依次**被发现的——先发现知识不够，然后发现记不住偏好，然后长任务漂移，然后发现不会修正错误，最后发现角色过载。这不是巧合。真实项目中，增强能力的引入通常也遵循这个顺序，因为每个问题都是在解决了前一个问题之后才暴露出来的。

所以本课的核心判断句不是”我要不要学 X 技术”，而是：

```text
不是”我要不要学 RAG”，而是”我的场景是否存在外部知识问题”。
不是”我要不要学 Memory”，而是”我的场景是否存在跨会话状态延续问题”。
不是”我要不要学 Planning”，而是”裸 ReAct 在多步任务中是否开始漂移”。
不是”我要不要学 Reflection”，而是”Agent 是否反复忽略明确的失败信号”。
不是”我要不要学 Multi-Agent”，而是”单 Agent 是否真的出现角色或并行瓶颈”。
```

增强能力的引入顺序，应该由**你当前项目实际遇到的失败**驱动，而不是由技术热度驱动。

### 1.3 五类增强能力的问题地图

| 增强能力 | 主要解决的问题 | 引入后新增复杂度 | 典型学习重点 |
|---|---|---|---|
| RAG / 外部知识接入 | 知识不在模型里、需要引用、数据变化快 | 数据质量、检索质量、权限、引用可信度 | Chunking、Embedding、Hybrid Search、Rerank、Web Search、数据库/API 查询、Citation |
| Memory | 状态不能跨轮、跨会话、跨任务延续 | 写入策略、隐私、遗忘、冲突、污染 | 短期记忆、工作记忆、长期记忆、用户偏好、记忆更新 |
| Planning / Workflow Patterns | 长任务步骤多、依赖强、容易漂移 | 计划可执行性、重规划、状态机、调试 | Chain、Router、ReAct Loop、Plan-Execute、Graph |
| Reflection | 失败后缺少修正机制 | 成本、延迟、错误反思、停止条件 | Self-check、Critique、Retry、外部验证信号 |
| Multi-Agent | 单 Agent 角色过载、需要并行或互审 | 通信成本、协调复杂度、错误放大 | Supervisor、Reviewer、Debate、Group Chat、Graph 协作 |

这张表不是让你一次性全部学习，而是给你一个定位坐标。

### 1.4 能力引入的通用判断流程

每次想引入一个增强能力前，先问五个问题：

1. **当前失败是什么？**
   是知识不够、状态断裂、任务组织混乱、缺少纠错，还是角色过载？

2. **有没有更简单的解决方式？**
   例如用户直接提供上下文、用固定流程、增加工具校验、优化提示词，是否已经足够？

3. **引入该能力会增加什么复杂度？**
   例如 RAG 会引入索引和引用问题，Memory 会引入隐私和污染问题，Multi-Agent 会引入通信和调试问题。

4. **如何做最小版本？**
   先做一个可评测的最小增强，而不是一开始上完整架构。

5. **如何判断它真的变好了？**
   必须有任务集、对比指标或人工验收标准。

可以用这个模板：

```text
场景：
当前问题：
失败证据：
可选方案：
选择的增强能力：
最小版本：
新增复杂度：
评测方式：
不引入的边界：
```

### 1.5 本课的学习方法

每章都按同一结构展开：

- **问题代入**：用 1.1 中那个知识助手的故事作为引子，感受这类崩溃的真实面貌。
- **技术演进背景**：这项能力是怎么被提出的？在它出现之前，人们是怎么凑合着过的？
- **对问题的思考**：拆解问题本质，跟踪设计者的推理链——为什么简单方案不行、为什么选择了这个方向。
- **解决方案 + 代码骨架**：核心链路的关键机制，配上伪代码展示设计思路。
- **迭代路径**：从最小版本到生产级的演变——先做什么、后做什么、什么时候该停。
- **常见失败与判断边界**：这项能力在什么情况下会翻车、什么情况下根本不该引入。

这比直接讲概念花费更多篇幅，但它让你不仅知道"怎么做"，还知道"为什么这么做"和"什么时候别做"——后者在真实项目中往往比前者更重要。

## 第二章：RAG / 外部知识接入

### 2.1 问题代入：模型不知道、记错或不该凭空回答

还记得 1.1 里那个知识助手 Agent 吗？用户把 30 篇学习笔记放进 `notes/` 目录，然后问：

```text
根据我所有笔记，总结 Tool Use 和 Memory 的设计哲学有什么根本不同？
```

Agent 读了 `agent-tool-use.md`，但不知道其他 29 篇笔记里有没有相关内容。它没有"在所有笔记里搜索"的能力。它给出的回答看起来头头是道，但你对照笔记原文一看——关键论据全是模型自己编的。

这不是模型"不够聪明"。这是模型根本没有**看到**那些笔记。

换个场景。一个企业客服 Agent，用户问"我们公司最新的退款政策是什么？"模型训练数据停留在 2023 年，而公司的退款政策上个月刚改过。模型如果直接回答，轻则给出过时信息，重则造成业务纠纷。

这类场景的共性是：**正确答案不在模型参数里，在外部资料里。** 模型需要一种机制，在回答问题之前先去"查资料"，然后基于查到的资料来组织回答。

### 2.2 技术演进背景：从"开卷考试"到 RAG

在 RAG 这个词被发明之前，人们已经意识到 LLM 的知识困境。2020 年之前，常见的应对方案有三种：

**方案 A：微调。** 把新知识通过 Fine-tuning 写进模型参数。问题是成本高、周期长，而且容易灾难性遗忘——学会了新知识，忘了旧能力。更麻烦的是，知识会过时，你总不能每个月微调一次模型。

**方案 B：把文档全塞进 prompt。** 直接把所有相关资料贴在问题前面。GPT-3 刚出来时很多人这么做。但当时上下文窗口只有 2048 tokens，一篇中等长度的文档就塞满了。而且塞得越多，模型越容易忽略中间的信息（"Lost in the Middle"）。

**方案 C：让用户自己筛选。** 用户自己决定哪些资料相关，贴进输入框。这在个人使用场景里勉强可行，但在自动化 Agent 场景中行不通——Agent 需要自己判断什么资料相关。

2020 年，Meta AI（当时还叫 Facebook AI）的 Lewis 等人发表了一篇论文：《Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks》。论文的核心思想可以用一句话概括：

> **与其把所有知识塞进模型参数，不如在需要的时候去查。**

这本质上是一种"开卷考试"策略：不要求模型背下所有知识，但要求它能在拿到参考资料后正确理解和回答问题。这个想法虽然简单，却解决了一个根本性问题：**模型的推理能力和知识存储可以解耦。** 推理能力留在模型参数里，知识放在外部存储中，两者通过"检索"这个动作在运行时动态连接。

2023 年后，RAG 迅速从学术概念走向工程实践。推动力来自三个方向：ChatGPT 引爆了 LLM 应用热潮（大量私有知识问答需求出现）、Embedding 模型质量大幅提升（text-embedding-ada-002 等）、向量数据库生态成熟（Pinecone、Weaviate、Chroma 等）。到今天，RAG 已经成为 Agent 的外部知识接入的事实标准——但它不等于"向量数据库 + 检索 + 回答"，下面会展开讲。

### 2.3 对问题的思考：知识从哪里来，如何进入决策

RAG 经常被理解成"向量数据库 + 检索 + 回答"。这个理解太窄。

更准确地说，本章讨论的是**外部知识接入**：当模型自身不知道、不能确定或不应该凭空回答时，系统如何把可信资料接入到 Agent 的决策上下文中。

外部知识不只来自向量库：

| 知识来源 | 适合场景 | 主要风险 |
|---|---|---|
| 用户上传文档 | 私有资料问答、个人知识库 | 文档解析、权限、引用准确性 |
| 向量检索 | 语义相似召回 | 相关但不事实匹配 |
| 关键词搜索 | 专有名词、ID、精确表达 | 不擅长语义变体 |
| Web Search | 最新公开信息 | 来源可信度不稳定 |
| 数据库查询 | 结构化业务数据 | 查询权限、SQL 安全、数据解释 |
| 业务 API | 订单、库存、工单、CRM | 接口权限、实时性、错误处理 |
| Knowledge Graph | 强关系查询 | 构建和维护成本高 |

所以要先问：

- 答案依赖哪类知识？
- 知识是否需要实时更新？
- 用户是否需要引用来源？
- 是否有权限边界？
- 知识进入模型前是否需要过滤、压缩或重排序？
- 回答后是否需要验证引用？

如果这些问题不清楚，直接上 RAG 只会得到一个"能检索但不可信"的系统。

### 2.4 解决方案：外部知识接入链路

一个完整的外部知识接入链路通常包含九步：

```text
数据接入
  -> 清洗与切分
  -> 索引
  -> 查询理解 / 改写
  -> 召回
  -> 重排序
  -> 上下文组装
  -> 生成回答
  -> 引用和反馈
```

#### 数据接入

先确定知识从哪里来：

- Markdown、PDF、Word、网页、代码仓库。
- 数据库、业务系统 API、搜索引擎。
- 用户当前上传的资料。
- 已审核的知识库。

数据接入阶段要保留元数据：

- 来源路径或 URL。
- 作者、时间、版本。
- 文档标题、章节、页码。
- 权限标签。
- 数据更新时间。

没有元数据，后面就很难做引用、权限过滤和知识过期判断。

#### 清洗与切分

Chunking 的核心问题是：如何把长资料切成适合检索和引用的小片段。

切得太碎：

- 语义不完整。
- 引用片段缺少上下文。
- 模型容易误解。

切得太大：

- 召回噪声高。
- token 成本高。
- 多个主题混在一起。

切分策略要根据资料类型选择：

| 资料类型 | 建议切分方式 |
|---|---|
| Markdown 文档 | 按标题层级、段落和列表切分 |
| PDF 报告 | 按页码、章节、表格附近语义切分 |
| API 文档 | 按接口、参数、错误码切分 |
| 代码文件 | 按函数、类、模块边界切分 |
| 会议记录 | 按议题、发言段落、决议切分 |

一个实用的递归分块函数，核心思路是"从大到小尝试分隔符，兜底按字符截断"：

```python
# 递归分块：优先按段落切，段落太长按句子切，句子太长按字符截断
def chunk_document(
    text: str,
    chunk_size: int = 512,       # tokens，不是字符数
    overlap: int = 64,           # 块之间保留重叠，避免边界截断
    separators: list[str] = ["\n\n", "\n", "。", ". ", " "]
) -> list[dict]:
    """返回带元数据的 chunk 列表，每个 chunk 记录来源位置"""
    chunks = []
    _recursive_split(text, separators, chunk_size, overlap, chunks, 
                     start_pos=0)
    return chunks

def _recursive_split(text, separators, size, overlap, result, start_pos):
    if estimate_tokens(text) <= size:
        if text.strip():
            result.append({
                "content": text.strip(),
                "char_start": start_pos,
                "char_end": start_pos + len(text),
                "token_estimate": estimate_tokens(text)
            })
        return
    
    # 尝试当前层级的分隔符
    sep = separators[0] if separators else ""
    if sep:
        segments = text.split(sep)
    else:
        segments = [text[i:i+size*4] for i in range(0, len(text), size*4)]
    
    for segment in segments:
        if estimate_tokens(segment) > size:
            # 本段仍超限，降级到下一级分隔符
            _recursive_split(segment, separators[1:], size, overlap, 
                           result, start_pos)
        else:
            if segment.strip():
                result.append({
                    "content": segment.strip(),
                    "char_start": start_pos,
                    "char_end": start_pos + len(segment),
                    "token_estimate": estimate_tokens(segment)
                })
        start_pos += len(segment) + (len(sep) if sep else 0)
```

> **代码说明**：这不是一个可运行的完整实现（`estimate_tokens` 需要接入具体 tokenizer），但它展示了递归分块的核心设计——分层分隔符、重叠保留、元数据追踪。每条 chunk 带 `char_start` / `char_end` 是为了引用时能定位回原文位置。

#### 索引

索引不等于只做 embedding。常见方式包括：

- 向量索引：适合语义相似。
- 关键词索引：适合精确匹配。
- 元数据索引：适合按时间、权限、文档类型过滤。
- 结构化索引：适合表格、数据库、实体关系。

很多场景会使用混合检索：先用关键词和元数据过滤，再用向量召回，最后用 rerank 重排序。

#### 查询理解 / 改写

用户问题可能并不适合直接检索。例如：

```text
这个方案和之前那个有什么区别？
```

系统需要根据当前会话状态把问题改写为：

```text
比较“Agent 工具调用方案”和“Agent Memory 方案”在目标、实现机制、风险和适用场景上的区别。
```

查询改写要注意不要改变用户意图。它应该让检索更准确，而不是替用户重新定义问题。

#### 召回与重排序

召回解决"找一批可能相关的资料"，重排序解决"从这批资料里选最有用的"。

召回通常追求覆盖，重排序追求精度。

如果召回太少，容易漏答案；如果召回太多，模型上下文会被噪声污染。

混合检索 + 重排序的关键接口：

```python
# 检索管线的三个核心步骤

# Step 1: 多路召回 —— 并行执行，各取所长
def multi_stage_retrieve(query: str, top_k: int = 20) -> list[Chunk]:
    """混合召回：向量语义匹配 + 关键词精确匹配"""
    # 查询改写（填充分辨率低的指代词）
    rewritten = rewrite_query(query)  # "这个和那个有什么区别" → 具体查询
    
    # 向量召回：捕捉语义相似
    query_vec = embedding_model.encode(rewritten)
    vector_hits = vector_db.search(query_vec, limit=top_k)
    
    # 关键词召回：专有名词、ID、精确表达
    keyword_hits = bm25_index.search(rewritten, limit=top_k)
    
    # 元数据过滤（如果有时间/权限/来源约束）
    hits = [h for h in vector_hits + keyword_hits 
            if metadata_filter.allows(h)]
    
    return deduplicate(hits)

# Step 2: 重排序 —— 用更强的模型做精细排序
def rerank(query: str, candidates: list[Chunk], top_k: int = 5) -> list[Chunk]:
    """用 cross-encoder 对候选做精排"""
    pairs = [(query, c.content) for c in candidates]
    scores = reranker_model.score(pairs)  # 比 embedding 相似度更准确
    ranked = sorted(zip(candidates, scores), 
                    key=lambda x: x[1], reverse=True)
    return [c for c, _ in ranked[:top_k]]

# Step 3: 上下文组装 —— 决定最终进入模型的内容和顺序
def assemble_context(query: str, ranked_chunks: list[Chunk]) -> str:
    """将检索结果编排为 prompt 可用的上下文"""
    parts = []
    for i, chunk in enumerate(ranked_chunks):
        parts.append(
            f"[来源 {i+1}] {chunk.metadata.get('title', '未知')}\n"
            f"{chunk.content}\n"
            f"—— 出处：{chunk.metadata.get('source', '未知')}，"
            f"更新时间：{chunk.metadata.get('updated', '未知')}"
        )
    return "\n\n---\n\n".join(parts)
```

> **设计要点**：多路召回在覆盖面和精度之间做权衡——向量保证不漏（高召回），关键词保证不错（高精度），rerank 做最终筛选。上下文组装时每个 chunk 标注来源编号，是为了生成阶段做引用对齐。

#### 上下文组装

上下文组装决定哪些资料进入模型。

要考虑：

- 是否保留原文。
- 是否先摘要。
- 是否按来源分组。
- 是否把互相冲突的资料同时给模型。
- 是否标注时间和权限。
- 是否把引用 id 绑定到片段。

外部知识不是越多越好。模型看到的内容越多，越可能忽略关键约束。

#### 生成回答与引用

生成阶段要明确要求：

- 只基于给定资料回答。
- 对关键断言给出引用。
- 资料不足时说明不知道。
- 如果资料冲突，要指出冲突来源。
- 不要把外部资料中的指令当成系统指令。

引用不是装饰，而是可信度机制。引用必须能对齐到原文片段。

### 2.5 迭代路径：从手动上下文到可信知识系统

外部知识接入建议按阶段迭代：

| 阶段 | 做什么 | 适合目标 | 不要过早做什么 |
|---|---|---|---|
| V0：手动上下文 | 用户把资料直接贴进输入 | 验证任务是否需要外部知识 | 不做索引系统 |
| V1：文件读取 | Agent 能按需读取指定文件 | 验证资料结构和引用需求 | 不做复杂向量库 |
| V2：基础检索 | 文档切分、向量召回、回答引用 | 跑通知识问答主路径 | 不追求多路检索 |
| V3：混合检索 | 关键词 + 向量 + 元数据过滤 | 提升召回和精确匹配 | 不无脑增加 top-k |
| V4：重排序与引用校验 | Rerank、引用对齐、冲突识别 | 提升可信度 | 不只看回答流畅度 |
| V5：权限与评测 | 权限过滤、回归集、来源质量 | 进入可持续使用 | 不把所有资料默认可见 |

学习时建议从 V0 到 V2。只有当你看到明确失败，再进入 V3 或 V4。

### 2.6 常见失败与修正

| 失败模式 | 表现 | 可能原因 | 修正方向 |
|---|---|---|---|
| 检索不到 | 文档里有答案，但没有召回 | Chunking 不合理、查询表达不同 | 查询改写、混合检索、调整切分 |
| 检索错 | 召回内容看似相关但事实无关 | 向量相似不等于事实匹配 | Rerank、关键词约束、元数据过滤 |
| 引用错 | 回答和引用片段对不上 | 生成阶段没有强引用约束 | 逐句引用、引用后校验 |
| 编造答案 | 资料不足但仍然回答 | 没有证据阈值和拒答规则 | 增加“不知道”路径 |
| 权限泄露 | 用户看到不该看的资料 | 检索前未做权限过滤 | 先过滤权限，再检索或召回 |
| 知识过期 | 使用旧政策或旧文档 | 缺少版本和更新时间 | 元数据排序、过期提示 |
| 外部注入 | 文档中指令影响 Agent | 把外部内容当指令 | 内容隔离、工具权限校验 |

外部知识接入的评测不能只问"答案对不对"，还要问：

- 正确资料是否被召回？
- 引用是否对应关键断言？
- 无答案时是否拒答？
- 权限过滤是否生效？
- 资料冲突时是否指出冲突？

### 2.7 判断边界：什么时候不需要外部知识接入

以下场景不一定需要 RAG / 外部知识接入：

- 用户输入本身已经包含全部资料。
- 任务是格式转换、改写、分类、摘要，不需要外部事实。
- 资料质量很差，接入后只会增加噪声。
- 业务风险不允许模型自由引用未经审核来源。
- 问题更适合结构化数据库查询，而不是语义检索。
- 外部资料变化太快，但你没有更新和过期机制。

一个实用判断：

```text
如果回答正确性依赖”模型之外的信息”，就考虑外部知识接入。
如果只是让模型处理”用户已经提供的信息”，先不要引入 RAG。
```

> **这个故事还没完。** 你给知识助手接上了 RAG，它现在能搜笔记、能给引用了。但用户第二天打开新会话，发现 Agent 完全不记得昨天花了 20 分钟教它的”先大纲、再正文、语气直接”——你得重新说一遍。知识问题解决了，但状态延续问题暴露了。这就是下一章要讲的 Memory。

## 第三章：Memory：状态延续能力

### 3.1 问题代入：Agent 总是忘记、重复或带错历史

回到 1.1 那个知识助手。你花了 20 分钟告诉它：

```text
以后写技术文章，先给我大纲确认，再展开正文。语气直接，不要营销化。
```

第二天你打开新会话："帮我写一篇 Agent Memory 的技术文章。"它输出了一篇以"在当今这个 AI 时代……"开头的营销风文章，没有先给你大纲。你盯着屏幕：昨天说的全白费了。

这是会话级别的遗忘。还有任务级别的遗忘：你让 Agent 整理一个 50 篇文档的目录，整理到一半你关掉了电脑。下次打开，Agent 不知道读过哪些文件、哪些已整理、草稿在哪、你确认过哪些结构——它从零开始，把前一半的活重做了一遍。

这两类遗忘根因相同：**LLM 的"记忆"完全依赖上下文窗口中的文本，会话关闭后上下文清零，一切归零。** 课程三的 State 管理解决了"单次任务内的状态延续"，但跨会话、跨任务的状态延续需要新的机制——Memory。

但要小心：Memory 是双刃剑。错误记忆、过期偏好、敏感信息如果被不加区分地长期保存，Memory 会从"个性化助手"退化为"长期污染源"。所以在动手之前，必须先想清楚什么值得记住、什么必须遗忘。

### 3.2 技术演进背景：ChatGPT 的记忆之路

2022 年 11 月 ChatGPT 发布时，它有一个让所有用户"哇"的瞬间——它能记住当前对话里的上下文。你说"我叫张三"，后面问"我叫什么"，它能答出来。

但关掉浏览器标签页，重新打开——它什么都不记得。

这不是 bug，这是设计如此。当时的 LLM API 是无状态的，每次请求是独立的。对话历史由前端应用保存在客户端，服务端不知道你是谁、上次聊了什么。

2023 年，随着 ChatGPT 用户突破 1 亿，"记忆"成为排名前三的用户诉求。但 OpenAI 花了整整一年才在 2024 年 2 月推出 Memory 功能。为什么这么慢？因为 Memory 比看起来危险得多：

- 如果模型自作主张记住了用户的健康信息，隐私怎么办？
- 如果记住了用户随口说的一个错误观点，未来任务都被带偏怎么办？
- 如果用户不知道系统"记住"了什么，信任从何而来？

OpenAI 的解决方案很克制：Memory 默认关闭（后来改为用户可管理），每条记忆可查看可删除，敏感内容自动过滤不写入。这个设计哲学——"宁可少记，不可乱记"——值得每个 Memory 系统设计者参考。

### 3.3 对问题的思考：不是所有历史都应该变成记忆

Memory 经常被误解成"把聊天记录存起来"。这不够准确，也很危险。

你需要先区分四类东西：

| 概念 | 作用范围 | 示例 | 是否属于 Memory |
|---|---|---|---|
| 当前上下文 | 本轮模型调用 | 用户刚输入的问题、当前工具结果 | 不一定 |
| 任务状态 | 当前任务运行期间 | 当前步骤、已完成事项、错误信息 | 工作记忆的一部分 |
| 会话历史 | 当前会话内 | 多轮对话内容 | 可摘要为短期记忆 |
| 长期记忆 | 跨会话、跨任务 | 用户偏好、稳定事实、项目习惯 | 是 |

Memory 的关键问题不是"存什么"，而是：

- 什么时候写入？
- 谁决定写入？
- 写入后如何读取？
- 如何更新旧记忆？
- 如何遗忘？
- 用户如何查看和纠正？
- 错误记忆如何避免污染后续任务？

如果没有这些问题的答案，Memory 会从"个性化能力"变成"长期污染源"。

### 3.4 解决方案：Memory 的读写更新机制

Memory 设计可以拆成五个环节：

```text
识别候选记忆
  -> 写入决策
  -> 存储
  -> 召回
  -> 更新 / 遗忘
```

#### 识别候选记忆

不是所有对话都值得记住。候选记忆通常包括：

- 用户明确要求记住的偏好。
- 长期稳定的用户背景。
- 项目中反复使用的约定。
- 某类任务的成功经验或失败教训。
- 当前任务的关键状态。

不应该自动记住：

- 敏感信息。
- 一次性临时约束。
- 未确认的推测。
- 情绪化表达。
- 可能过期的事实。

#### 写入决策

写入策略可以分级：

| 写入方式 | 适合内容 | 风险 |
|---|---|---|
| 用户显式确认后写入 | 偏好、长期事实、敏感设置 | 交互成本高，但安全 |
| 低风险自动写入 | 当前任务进度、非敏感状态 | 需要过期机制 |
| 候选记忆待确认 | 模型推断出的偏好 | 需要用户审核 |
| 不写入 | 敏感、短期、模糊、冲突内容 | 可能少一些个性化 |

#### 存储结构

Memory 不一定都放向量库。常见存储包括：

- 会话状态对象。
- 数据库记录。
- 文件系统中的项目配置。
- 向量库中的语义记忆。
- 用户画像或偏好表。
- 任务日志和经验库。

选择存储方式时看两个问题：

- 读取时需要精确匹配，还是语义召回？
- 记忆是否需要用户查看、编辑和删除？

Memory 核心接口骨架——注意写入前的决策比存储本身更重要：

```python
class AgentMemory:
    """Memory 的核心不是存储，是写入决策和召回过滤"""
    
    def __init__(self):
        self.preferences = {}        # 用户偏好：精确 key-value
        self.facts = []              # 长期事实：带时间戳和来源
        self.task_log = []           # 任务历史：成功/失败经验
        self.session_state = {}      # 会话状态：当前会话内有效
    
    # ── 写入前必须经过决策 ──
    def should_remember(self, candidate: dict) -> bool:
        """不是所有信息都值得写入长期记忆"""
        if candidate.get("sensitive"):
            return False            # 敏感信息不自动写入
        if candidate.get("expires_at") and is_expired(candidate):
            return False            # 已过期
        if candidate.get("confidence", 0) < 0.8:
            return False            # 低置信度的推断不写入
        if candidate.get("type") == "temporary":
            return False            # 一次性临时约束不写入
        return True
    
    # ── 写入（分类存储）──
    def write(self, entry: dict):
        if not self.should_remember(entry):
            return
        store = {
            "preference": self.preferences,
            "fact": self.facts,
            "task_result": self.task_log
        }.get(entry["type"])
        if store is not None:
            store.append({**entry, "recorded_at": now()})
    
    # ── 召回：相关性过滤，不全量注入 ──
    def recall(self, current_task: str, limit: int = 5) -> list[dict]:
        """只召回与当前任务相关的记忆"""
        relevant = []
        # 精确匹配：用户偏好中与当前任务关键词相关的
        for key, val in self.preferences.items():
            if any(kw in current_task for kw in key.split()):
                relevant.append({"key": key, "value": val, "score": 1.0})
        # 语义召回：长期事实和任务历史
        task_vec = embed(current_task)
        for fact in self.facts:
            score = cosine_sim(task_vec, embed(fact["content"]))
            if score > 0.6:
                relevant.append({**fact, "score": score})
        # 排序、截断、去重
        relevant.sort(key=lambda x: x["score"], reverse=True)
        return deduplicate(relevant)[:limit]
    
    # ── 过期与遗忘 ──
    def decay(self):
        """定期清理：过期记忆降权或删除"""
        cutoff = now() - timedelta(days=90)
        self.facts = [f for f in self.facts 
                      if f.get("recorded_at", cutoff) > cutoff]
        self.task_log = self.task_log[-500:]  # 只保留最近 500 条
```

> **设计要点**：`should_remember` 的保守策略是整个 Memory 系统可靠性的基石。宁可漏记一些无害信息，不可写入一条有害信息。召回时的相关性过滤避免把所有记忆全量塞进上下文——那会让模型注意力被无关历史淹没。

#### 召回

Memory 召回应该按任务相关性，而不是全部注入。

召回时要考虑：

- 这条记忆是否与当前任务相关？
- 是否过期？
- 是否与新信息冲突？
- 是否有权限问题？
- 是否需要告诉用户当前使用了哪些记忆？

#### 更新与遗忘

长期系统必须支持：

- 用户修改偏好。
- 新事实覆盖旧事实。
- 过期记忆自动降权。
- 错误记忆删除。
- 敏感记忆彻底清除。

Memory 没有遗忘机制，就会越来越不可信。

### 3.5 迭代路径：从会话状态到长期记忆

Memory 建议按阶段迭代：

| 阶段 | 做什么 | 解决什么问题 |
|---|---|---|
| V0：无长期记忆 | 只依赖当前上下文 | 验证任务本身是否成立 |
| V1：任务状态 | 保存当前步骤、工具结果、待办 | 支持长任务继续执行 |
| V2：会话摘要 | 把多轮对话压缩成当前会话状态 | 支持长对话不丢重点 |
| V3：显式用户偏好 | 用户确认后保存长期偏好 | 减少重复说明 |
| V4：经验记忆 | 保存失败和成功策略 | 改善同类任务表现 |
| V5：可管理记忆 | 用户可查看、编辑、删除、导出 | 支持可信长期使用 |

建议不要一开始就做自动长期记忆。先做任务状态和显式偏好，风险更低，也更容易评测。

### 3.6 常见失败与修正

| 失败模式 | 表现 | 可能原因 | 修正方向 |
|---|---|---|---|
| 记忆污染 | 错误信息长期影响后续任务 | 未确认就写入长期记忆 | 写入确认、来源记录、可删除 |
| 召回噪声 | 不相关记忆进入上下文 | 召回条件太宽 | 按任务类型和相关性过滤 |
| 偏好过期 | 用户新偏好被旧偏好覆盖 | 没有更新时间和冲突处理 | 新记忆优先、冲突提示 |
| 隐私风险 | 敏感信息被长期保存 | 缺少数据分级 | 敏感内容默认不写入 |
| 上下文膨胀 | 记忆太多导致模型忽略当前目标 | 全量注入 | 只注入相关记忆，做摘要 |
| 用户不信任 | 用户不知道系统记住了什么 | 没有可见管理界面 | 提供查看、编辑、删除 |

### 3.7 判断边界：什么时候不需要 Memory

以下场景不建议引入长期 Memory：

- 每次任务都是一次性请求。
- 用户明确不希望被记住。
- 错误记忆的风险高于个性化收益。
- 任务状态已经由确定性业务系统保存。
- 当前只是原型阶段，尚无稳定任务类型。
- 你还没有记忆查看、纠错和删除机制。

一个实用判断：

```text
如果系统只需要完成当前任务，优先保存任务状态。
如果系统需要跨任务理解用户或项目，才考虑长期 Memory。
```

> **这个故事还没完。** Memory 让 Agent 记住了用户偏好，但新的问题出现了：用户给了一个多步骤的复杂任务——"做发布准备：检查 README、跑测试、整理 changelog、生成 checklist"。Agent 在第 4 步之后开始漂移，偏离原始目标。Memory 解决的是"记住什么"，但它不管"任务怎么组织"。这就是下一章要讲的 Planning。

## 第四章：Planning / Workflow Patterns

### 4.1 问题代入：任务变长后 Agent 开始漂移

回到 1.1 的知识助手。你给它一个多步骤任务：

```text
帮我做发布准备：检查 README 完整性、跑一遍测试、整理 changelog、生成 release checklist。
```

Agent 开始工作。第一步，它读了 README，写了一段评论。第二步，它……开始读 `notes/` 目录下的其他文件。第三步，它搜索了点什么。五分钟后你回来检查，发现它偏离原始目标，不知道在做什么。你翻 trace 才发现：它在第 4 步之后就再也没碰过任何和"发布"相关的操作。

根因不是工具不好用，不是 State 丢了，不是没记住偏好。根因是：**裸 ReAct 循环没有"全局任务结构"的概念。** 它每步都在做局部最优的决策，但局部最优之和不一定能到达目标。就像一个没有导航的司机，每个路口都选看起来对的方向，但可能永远到不了目的地。

任务越长，越需要一种机制来回答：

- 要做哪些步骤？步骤之间的依赖关系是什么？
- 哪些步骤可以并行执行？
- 哪些步骤执行前需要用户确认？
- 某一步失败后，是重试、跳过、重规划还是停止？

这就是 Planning / Workflow Patterns 要解决的核心问题。

### 4.2 技术演进背景：从 ReAct 到 LangGraph

Planning 在 Agent 领域不是一开始就有的。回顾一下演进：

**2022 年 10 月，ReAct 论文（Yao et al.）。** 论文的核心发现是：让 LLM 在"推理"和"行动"之间交替循环，能大幅提升复杂任务的完成率。但 ReAct 的模式是"每一步临场判断"——模型看到当前 Observation，决定下一步 Action。这对 3-5 步的任务有效，但步数一多就暴露了问题：模型没有全局视野，容易漂移。

**2023 年初，Plan-and-Execute 模式出现。** 思路很直接：让模型先把整个任务拆成执行计划，用户确认后再逐步执行。这解决了 ReAct 的"没有全局视图"问题，但引入了新问题——计划可能不可执行（模型列出的步骤在实际环境中无法完成），或者环境变化后原计划不再适用。

**2023 年下半年，图结构（Graph）思想兴起。** LangGraph、CrewAI 等框架开始把任务建模成节点和边的图——每个节点是一个动作或判断，边代表状态迁移，失败后可以跳回特定节点重试。图结构本质上是对 Plan-and-Execute 的泛化：计划不再是一维的步骤列表，而是一个可以有分支、回溯、并行路径的有向图。

**2024 年至今，Workflow Patterns 成为共识。** Anthropic 的"Building Effective Agents"文章确立了一个重要观点：大多数 Agent 场景不需要复杂的 Graph，简单的 Chain 或 Router 就能覆盖。关键不是"用不用 LangGraph"，而是"根据任务形态选择对应模式"。

### 4.3 对问题的思考：任务组织比单步决策更重要

ReAct Loop 擅长动态决策，但不擅长天然保证全局结构。

复杂任务通常有几种形态：

| 任务形态 | 例子 | 适合模式 |
|---|---|---|
| 步骤固定 | 读取文件 -> 摘要 -> 输出 | Chain |
| 输入类型差异大 | 问答、写作、代码审查走不同路径 | Router |
| 工具选择动态 | 需要边查边决定下一步 | ReAct Loop |
| 任务可先拆解 | 发布准备、调研报告、代码迁移 | Plan-Execute |
| 状态和分支复杂 | 工单处理、审批流、多阶段任务 | Graph |

所以不要问"Planning 好不好"，而要问：

- 任务是否足够复杂，需要先拆解？
- 步骤是否稳定，可以固化成流程？
- 是否需要中途重规划？
- 用户是否需要确认计划？
- 失败后是否需要回到某个节点？

### 4.4 解决方案：常见任务组织模式

#### Chain

Chain 是固定顺序执行。

```text
输入 -> 分类 -> 检索 -> 生成 -> 校验 -> 输出
```

适合步骤稳定、异常少的任务。优点是简单、可控、易调试。缺点是灵活性低。

#### Router

Router 根据输入类型或状态选择路径。

```text
用户请求
  -> 文档问答路径
  -> 代码审查路径
  -> 任务执行路径
```

适合多类型任务入口。关键是分类准确，以及分类失败后的兜底路径。

#### ReAct Loop

ReAct Loop 让模型根据当前状态选择工具并观察结果。

适合信息不完整、需要探索的任务。风险是步骤不稳定，需要停止条件、工具约束和 trace。

#### Plan-Execute

先生成计划，再按计划执行。

```text
目标 -> 计划 -> 用户确认 -> 执行步骤 -> 观察结果 -> 必要时重规划
```

适合中长任务。计划最好是结构化的，而不是一段漂亮文字。

#### Graph

Graph 把任务建模成节点和边：

- 节点表示动作、判断、工具调用、人工确认。
- 边表示状态迁移。
- 每个节点可以有输入、输出、失败路径。

Graph 适合复杂状态机、需要回放和恢复的任务。缺点是建模和调试成本高。

四种核心模式的关键接口骨架——核心区别在于"谁决定下一步"：

```python
# ── Chain：固定顺序，确定性最强 ──
def chain_execute(steps: list[Step], context: dict) -> dict:
    """每步的输出是下一步的输入，无分支"""
    for i, step in enumerate(steps):
        result = execute_step(step, context)
        if result.status == "error" and not step.allow_skip:
            return {"status": "failed", "failed_at": i, "error": result.error}
        context[step.name] = result.output
    return {"status": "completed", "context": context}

# ── Router：根据输入特征选择路径 ──
def router_execute(query: str, routes: dict[str, list[Step]]) -> dict:
    """分类器决定走哪条路径，失败时走 default 兜底"""
    category = classify(query, list(routes.keys()))
    selected = routes.get(category, routes["default"])
    return chain_execute(selected, {"query": query})

# ── Plan-Execute：先规划再执行，支持中途重规划 ──
def plan_execute(goal: str, tools: dict, max_retries: int = 2) -> dict:
    """生成计划 → 用户确认 → 逐步执行 → 失败时重规划"""
    plan = generate_plan(goal, tools)        # 结构化步骤列表
    if not user_confirm(plan):
        return {"status": "rejected"}
    
    for step in plan.steps:
        result = execute_step(step)
        if result.status == "error":
            if step.retries < max_retries:
                # 重规划：从当前状态生成新的后续步骤
                new_steps = replan(goal, plan.completed_steps, result.error)
                plan.replace_remaining(new_steps)
                continue
            else:
                return {"status": "failed", "step": step, "error": result.error}
        plan.mark_completed(step)
    
    return {"status": "completed", "result": plan.final_output}

# ── Graph：节点+边+状态机 ──
class WorkflowGraph:
    """节点是动作，边是条件跳转，每个节点有失败出口"""
    def __init__(self):
        self.nodes: dict[str, Node] = {}   # name → Node(action, next, on_error)
        self.edges: list[Edge] = []         # (from, to, condition)
    
    def add_node(self, name: str, action: callable, 
                 on_success: str = None, on_error: str = None):
        self.nodes[name] = Node(action, on_success, on_error)
    
    def run(self, entry: str, context: dict) -> dict:
        current = entry
        visited = set()
        while current and current not in visited:
            visited.add(current)
            node = self.nodes[current]
            result = node.action(context)
            context[node.name] = result
            current = node.on_error if result.error else node.on_success
        return {"status": "completed", "context": context}
```

> **选择指南**：任务步骤完全固定 → Chain；多类型入口 → Router；需要全局规划且有验证需求 → Plan-Execute；状态复杂、分支多、需要回放和恢复 → Graph。不要把简单任务硬塞进 Graph——那就像用 Kubernetes 部署一个静态网页。

### 4.5 迭代路径：从固定步骤到可回放图结构

Planning / Workflow Patterns 可以这样迭代：

| 阶段 | 做什么 | 适合目标 |
|---|---|---|
| V0：裸 ReAct | 模型逐步决定工具 | 探索任务形态 |
| V1：Checklist | 让模型先列出步骤清单 | 减少漏步骤 |
| V2：用户确认计划 | 执行前让用户修改计划 | 提升可控性 |
| V3：结构化 Plan | 每步包含目标、工具、输入、验收标准 | 支持执行和追踪 |
| V4：动态重规划 | 失败或环境变化时调整计划 | 支持复杂任务 |
| V5：Graph | 将任务节点、状态和分支显式化 | 支持恢复、评测、回放 |

初学阶段建议做到 V2 或 V3。Graph 等到状态和分支真的复杂时再引入。

### 4.6 常见失败与修正

| 失败模式 | 表现 | 可能原因 | 修正方向 |
|---|---|---|---|
| 计划不可执行 | 步骤看似合理但无法落地 | 没有工具和环境约束 | 计划中绑定工具和验收标准 |
| 计划过细 | 生成大量无意义步骤 | 追求形式完整 | 限制步骤数和粒度 |
| 执行偏离计划 | 执行时忘记原计划 | 状态没有记录当前步骤 | 每步执行后更新状态 |
| 不会重规划 | 工具失败后硬继续 | 没有失败分支 | 增加重规划触发条件 |
| 过度工程化 | 小任务也上 Graph | 技术驱动 | 从 Chain / Router 开始 |
| 用户不买账 | 计划不符合用户真实目标 | 缺少确认节点 | 计划执行前让用户确认 |

### 4.7 判断边界：什么时候不需要 Planning

以下场景不需要复杂 Planning：

- 任务一步就能完成。
- 两三步固定流程已经足够。
- 用户更想要快速结果，不需要先看计划。
- 计划无法验证，只是生成一段中间文本。
- 任务由确定性业务流程稳定承载。
- 引入计划后成本和延迟明显上升，但质量没有提升。

实用判断：

```text
如果失败主要是”漏步骤、走偏、无法恢复”，考虑 Planning。
如果失败只是”某一步工具调用错了”，先修工具机制。
```

> **这个故事还没完。** Planning 让 Agent 有了全局任务视图，发布准备不再漏步骤。但新问题来了：Agent 执行到”运行测试”这一步时测试失败了，TypeError 清清楚楚地打在日志里——Agent 看到了，却继续写 changelog，没有停下来分析为什么失败。它缺少”失败→分析→修正→重试”的能力。这就是下一章要讲的 Reflection。

## 第五章：Reflection：失败修正能力

### 5.1 问题代入：Agent 明明错了，却继续往下走

回到 1.1。知识助手 Agent 现在有了 Planning，发布准备的四个步骤不再漏了。但执行到"运行测试"这一步时：

```text
TypeError: Cannot read properties of undefined (reading 'files')
```

Agent 看到了这个错误输出。然后它做了什么？它继续写 changelog，最后汇报"发布准备已完成"。

这不是工具问题，不是 Planning 问题，不是 Memory 问题。这是一个更根本的缺陷：**Agent 缺少"看到失败信号 → 停止 → 分析原因 → 修正 → 重试"的闭环。** 它看到了 TypeError，但它把那条输出当成了和"读取文件内容"一样的 Observation——一段需要被记录的信息——而不是一个需要被修正的错误。

换到知识问答场景。Agent 给了一段引用："根据《软件工程实践》第 3 章……"但你查看原文，发现那句话根本不在第 3 章。Agent 没有"引用校验"这一步，它生成引用时只是在生成"看起来像引用"的文本。

这类问题的本质是：**Agent 缺少被反馈信号触发的修正机制。** 它不会说"等等，这个测试失败了，让我看看为什么"。它只会继续生成下一个 Token。

### 5.2 技术演进背景：Self-Refine 和 Reflexion

让 LLM "反思自己的输出"这个想法，经历了几个关键节点：

**2023 年 3 月，Self-Refine（Madaan et al.）。** 论文提出了一个简单但有效的流程：生成 → 反馈 → 修正。每一步都让同一个 LLM 完成——先输出初稿，再给自己提反馈，再根据反馈修改。这个流程在写作、代码生成等任务上取得了明显提升。但它的致命弱点是：**反馈完全来自模型自身，没有外部验证信号。** 一个错误的初稿可能导致模型给自己提一堆看似合理但方向全错的"反馈"。

**2023 年 6 月，Reflexion（Shinn et al.）。** 这个工作的突破在于引入了**外部验证信号**——不再让模型"自评"，而是让模型根据真实的执行结果（测试失败、编译器报错、API 返回）来判断哪里出错。Reflexion 把"反思"从生成策略升级为一种**基于环境反馈的强化学习式循环**：尝试 → 观察真实结果 → 分析失败原因 → 记录经验 → 重试。这比 Self-Refine 可靠得多，因为它有了外部锚点。

**2023 年底至今，Reflection 从学术概念走向工程实践。** 关键转变是：Reflection 不再被当成"模型的自省能力"，而是被设计成 Runtime 中的一个**带触发条件和停止条件的系统机制**。触发条件是什么信号（测试失败、Schema 校验失败、用户驳回），停止条件是什么情况（最大重试次数、成本上限、相同错误重复出现）。

### 5.3 对问题的思考：反思必须依赖反馈信号

Reflection 容易被误解成"让模型自己检查一遍"。这不可靠。

模型自己说"我检查过了"并不等于真的可靠。更好的 Reflection 应该依赖外部反馈信号：

- JSON Schema 校验失败。
- 工具返回参数错误。
- 单元测试失败。
- 引用校验失败。
- 用户驳回。
- 权限检查不通过。
- 结果和验收标准不一致。

也就是说，Reflection 不是一个固定步骤，而是一个被触发的修正机制。

要回答四个问题：

- 什么时候触发反思？
- 反思要看哪些证据？
- 修正后是否重试？
- 重试几次后停止？

### 5.4 解决方案：触发、批判、修正、停止

一个可控的 Reflection 机制可以分成四步：

```text
触发信号
  -> 错误分类 / 批判
  -> 生成修正方案
  -> 重试或停止
```

#### 触发信号

不要默认每一步都反思。常见触发信号包括：

| 触发信号 | 适合动作 |
|---|---|
| 输出格式不合法 | 重新生成结构化输出 |
| 工具参数错误 | 修正参数或请求用户补充 |
| 工具执行失败 | 分析错误类型，重试或降级 |
| 测试失败 | 定位失败原因并修改 |
| 引用不匹配 | 重新检索或修改回答 |
| 用户驳回 | 更新目标或约束 |
| 连续失败 | 停止并请求人工介入 |

#### 错误分类

Reflection 要先判断失败属于哪类：

- 目标理解错误。
- 上下文缺失。
- 工具选择错误。
- 工具参数错误。
- 外部环境错误。
- 输出格式错误。
- 权限不足。
- 结果质量不足。

不同错误需要不同修正方式。

#### 修正方案

修正可以是：

- 改写提示词。
- 调整工具参数。
- 换工具。
- 补充检索。
- 回到上一步。
- 请求用户澄清。
- 降级为人工确认。

#### 停止条件

必须设置停止条件：

- 最大重试次数。
- 最大成本。
- 最大耗时。
- 相同错误重复出现。
- 风险等级过高。
- 缺少足够反馈信号。

没有停止条件，Reflection 很容易变成无限循环。

Reflection 循环的核心骨架——关键是触发条件必须来自外部信号，停止条件必须硬编码：

```python
# Reflection 循环：只在有外部失败信号时触发，永远设置停止边界
def reflection_loop(
    action: callable,       # 执行动作（生成代码、回答问题等）
    validate: callable,     # 外部验证器（运行测试、检查 schema、校验引用）
    max_retries: int = 3,   # 硬上限
    cost_budget: float = 1.0  # 成本上限（美元）
) -> dict:
    """带 Reflection 的执行循环"""
    cost_spent = 0
    last_error = None
    
    for attempt in range(max_retries):
        # 1. 执行动作
        result = action(
            previous_error=last_error,  # 把上次失败原因注入上下文
            attempt=attempt
        )
        cost_spent += result.cost
        
        # 2. 外部验证（不是模型自评！）
        validation = validate(result.output)
        
        if validation.passed:
            return {"status": "success", "output": result.output, 
                    "attempts": attempt + 1, "cost": cost_spent}
        
        # 3. 分类失败，准备修正上下文
        last_error = {
            "type": classify_error(validation),  # schema_error | test_failure | tool_error | ...
            "message": validation.message,
            "evidence": validation.evidence,      # 具体报错行、引用原文等
            "suggested_fix": validation.suggestion
        }
        
        # 4. 停止条件检查
        if cost_spent > cost_budget:
            return {"status": "stopped", "reason": "cost_limit", 
                    "last_error": last_error}
        if last_error["type"] == "same_error" and attempt >= 2:
            return {"status": "stopped", "reason": "repeated_failure",
                    "last_error": last_error}
    
    return {"status": "stopped", "reason": "max_retries_exceeded",
            "last_error": last_error}

# 使用示例：代码修改 + 测试驱动的 Reflection
result = reflection_loop(
    action=lambda prev_err, attempt: llm_generate_code(
        task="fix the bug in user_service.py",
        test_failure=prev_err["evidence"] if prev_err else None
    ),
    validate=lambda code: run_tests(code),  # 外部测试框架，不是 LLM
    max_retries=3
)
```

> **设计要点**：`validate` 必须是外部验证器——跑测试、检查 schema、对比引用原文，不能是"让模型再看看"。`action` 接收 `previous_error` 作为修正信号，但只注入事实（报错内容、失败测试名），不注入模型的自我评价。

### 5.5 迭代路径：从格式校验到基于测试的修正

Reflection 可以按阶段迭代：

| 阶段 | 做什么 | 解决什么问题 |
|---|---|---|
| V0：无反思 | 失败直接返回 | 保持简单 |
| V1：格式修复 | JSON / schema 不合法时重试 | 结构化输出稳定性 |
| V2：工具错误修正 | 参数错误、超时、权限不足分类处理 | 工具调用稳定性 |
| V3：测试驱动修正 | 根据测试失败调整代码 | 代码类任务质量 |
| V4：引用和事实校验 | 检查回答是否有证据 | 知识问答可信度 |
| V5：用户反馈学习 | 把用户驳回纳入回归集 | 持续改进 |

不要为了"看起来更智能"每步都加自我批判。Reflection 应该服务具体失败。

### 5.6 常见失败与修正

| 失败模式 | 表现 | 原因 | 修正方向 |
|---|---|---|---|
| 错误反思 | 模型把错因分析错 | 缺少外部证据 | 引入测试、schema、工具错误码 |
| 无限重试 | 一直修正但不收敛 | 没有停止条件 | 设置重试、成本、耗时上限 |
| 成本过高 | 每步都反思 | 触发条件过宽 | 只在失败或低置信度触发 |
| 掩盖失败 | 反思后输出更自信但仍错误 | 自评替代验证 | 修正后必须重新验证 |
| 用户体验差 | Agent 反复解释过程，任务很慢 | 反思内容暴露过多 | 对用户展示摘要和关键选择 |

### 5.7 判断边界：什么时候不需要 Reflection

以下场景不需要复杂 Reflection：

- 输出可以用确定性规则直接修正。
- 失败成本很低，简单重跑更便宜。
- 没有外部反馈信号，模型无法判断自己是否错。
- 任务对延迟非常敏感。
- 高风险任务需要人工或规则校验，模型自评不能替代。

实用判断：

```text
如果有明确反馈信号，并且修正成本低于失败成本，Reflection 值得引入。
如果只是让模型”再想想”，通常不值得。
```

> **这个故事还没完。** Reflection 让 Agent 学会了从错误中修正，测试失败后会分析原因并重试。但还有一个根本性的问题：Agent 写完方案后，你让它从安全角度审查自己的方案——它说”没有明显问题”。你作为人类看了一眼，发现三个安全隐患。不是 Agent 不聪明，而是”创作者”和”审查者”需要不同的视角、不同的上下文、不同的判断标准。这就是最后一类增强能力要解决的：Multi-Agent。

## 第六章：Multi-Agent：角色分工能力

### 6.1 问题代入：单个 Agent 的角色开始过载

回到 1.1 的知识助手。现在它有了 RAG（能搜笔记）、Memory（记得偏好）、Planning（任务不漂移）、Reflection（错了会修正）。你觉得它已经很强了。

你让它写一份技术方案，然后从安全角度审查。它写完后，你说："现在从安全角度审查这个方案。"它输出：

```text
经过审查，该方案在安全方面没有明显问题。建议按计划实施。
```

你作为人类审了一眼，立刻发现了三个问题：方案没有考虑输入校验、API 密钥明文写在配置文件里、权限模型缺少最小权限原则。Agent 不是看不到这些问题——如果让它从零开始做安全审查，它可能能发现。问题是它刚刚才"作为作者"写完了方案，现在要让同一个大脑"作为审查者"审视自己的作品。

这是典型的**角色过载**：单一 Agent 上下文里同时承载了创作者、审查者、执行者、总结者等多个角色。这些角色有不同的目标、不同的判断标准、需要关注不同维度的信息。把它们全塞进一个上下文，结果就是互相干扰、先入为主、无法真正"切换视角"。

还有另一种过载：**并行需求**。调研任务需要同时阅读 10 篇文档，每篇文档的分析互不依赖。单 Agent 只能串行阅读——读完一篇再读下一篇——即使这些任务完全可以并行。

这就是 Multi-Agent 要解决的最后一类问题。

### 6.2 技术演进背景：AutoGen、CrewAI 和"多智能体"的诱惑

Multi-Agent 这个概念有两股推动力：

**学术线：Multi-Agent Debate 和角色分工。** 2023 年，多个研究团队发现让多个 LLM 实例从不同立场辩论，能提升复杂推理任务的准确率。核心发现是：当多个 Agent 被迫互相质疑时，错误更容易被暴露和修正——这比单个 Agent 的 Self-Reflection 更有效，因为外部质疑比自我审视更客观。

**工程线：任务并行化和上下文隔离。** 随着 Agent 任务的复杂度上升，开发者发现单个 Agent 的上下文窗口是瓶颈——所有工具结果、历史决策、中间推理全挤在一个窗口里。把不同子任务分给不同 Agent，每个 Agent 有自己的上下文窗口，天然实现了信息隔离。同时，独立的子任务可以并行执行，降低端到端延迟。

**2023 年底，框架爆发。** Microsoft 发布了 AutoGen，CrewAI 和 ChatDev 等框架相继出现。它们的共同思路是：定义多个 Agent 角色，用消息传递机制协调它们的行为。但这也带来了"过度工程化"的风险——很多场景下，单个 Agent 配合良好的工具和上下文管理就能完成，引入 Multi-Agent 只会增加通信成本和调试难度。

### 6.3 对问题的思考：多 Agent 不是多写几个角色 Prompt

Multi-Agent 的价值不在于把一个 Agent 改名成"研究员、工程师、审查员"。如果这些角色没有不同输入、不同工具、不同目标、不同验收标准，就只是 Prompt 装饰。

真正的 Multi-Agent 至少要有一个明确理由：

- **角色分工**：不同 Agent 负责不同目标。
- **并行处理**：多个子任务可以独立探索。
- **互相审查**：一个执行，一个审查。
- **视角对抗**：需要不同立场比较方案。
- **上下文隔离**：避免所有中间噪声污染主 Agent。

同时要意识到它的代价：

- 通信成本上升。
- Trace 更复杂。
- 多个 Agent 可能互相放大错误。
- 决策归属更难判断。
- 评测难度上升。

### 6.4 解决方案：分工、并行、互审和协调

常见 Multi-Agent 模式：

| 模式 | 工作方式 | 适合场景 |
|---|---|---|
| Reviewer | 一个 Agent 执行，另一个审查 | 代码审查、文档审查、方案审查 |
| Supervisor | 主管 Agent 拆任务，Worker 执行 | 子任务边界清晰 |
| Parallel Specialists | 多个专家并行处理不同部分 | 调研、资料整理、多文件分析 |
| Debate | 多个 Agent 从不同立场辩论 | 方案权衡、风险判断 |
| Group Chat | 多 Agent 共享上下文协作 | 开放探索，但难控制 |
| Graph Collaboration | 用图结构定义协作关系 | 复杂流程、可回放协作 |

设计 Multi-Agent 时，要明确：

- 每个 Agent 的目标是什么。
- 每个 Agent 能看到什么上下文。
- 每个 Agent 能使用哪些工具。
- 谁做最终决策。
- Agent 之间如何通信。
- 中间结果如何汇总。
- 失败如何定位。

最推荐入门的模式是 Reviewer——分工清晰、评测容易、通信成本可控：

```python
# Reviewer 模式：一个执行，一个审查，结论带证据
class ReviewerPattern:
    """最简 Multi-Agent：Executor 产出 → Reviewer 审查 → 修正或通过"""
    
    def __init__(self, executor_agent, reviewer_agent):
        self.executor = executor_agent   # 有创作工具的 Agent
        self.reviewer = reviewer_agent   # 有审查工具但无创作工具的 Agent
    
    def run(self, task: str, review_criteria: list[str]) -> dict:
        """执行 → 审查 → 修正循环（最多修正两轮）"""
        # Step 1: Executor 产出
        draft = self.executor.run(task)
        
        for round_num in range(2):  # 最多修正两轮
            # Step 2: Reviewer 审查（独立上下文，独立工具）
            review = self.reviewer.run(
                task="审查以下产出，逐条对照审查标准，指出具体问题",
                context={
                    "original_task": task,
                    "output": draft.output,
                    "criteria": review_criteria  # ["安全性", "可维护性", "正确性"]
                }
            )
            
            if review.verdict == "approved":
                return {"status": "approved", "output": draft.output,
                        "review_rounds": round_num + 1}
            
            # Step 3: 反馈给 Executor 修正（只注入审查问题，不注入原始上下文）
            draft = self.executor.run(
                task=task,
                fix_instructions=review.issues  # 具体问题列表，不是审查 Agent 的主观评价
            )
        
        # 两轮未通过：标记分歧，由人工裁决
        return {"status": "disputed", "output": draft.output,
                "unresolved_issues": review.issues}
```

> **设计要点**：Executor 和 Reviewer 使用**不同的 System Prompt、不同的工具集、不同的上下文窗口**。如果只是改个名字把同一个 LLM 调两次，那不是真正的 Multi-Agent。`fix_instructions` 只传 Reviewer 发现的具体问题（如"第 3 行缺少输入校验"），不传 Reviewer 的主观评价（如"整体质量较差"）——让 Executor 根据事实修正，而不是被另一个模型的意见带偏。

### 6.5 迭代路径：从 Reviewer 到 Supervisor

Multi-Agent 不要一开始就做复杂群聊。可以这样迭代：

| 阶段 | 做什么 | 适合目标 |
|---|---|---|
| V0：单 Agent | 一个 Agent 完成全流程 | 建立基线 |
| V1：Reviewer | 增加审查 Agent | 提升质量和发现错误 |
| V2：子任务并行 | 不同 Agent 处理独立部分 | 降低耗时，隔离上下文 |
| V3：Supervisor | 主 Agent 分配任务、汇总结果 | 管理多分支任务 |
| V4：Debate | 多视角比较和反驳 | 方案权衡 |
| V5：Graph 协作 | 显式状态和协作关系 | 复杂生产系统 |

最推荐的起点是 Reviewer，因为它分工清晰、价值容易评测、通信成本相对可控。

### 6.6 常见失败与修正

| 失败模式 | 表现 | 原因 | 修正方向 |
|---|---|---|---|
| 角色重叠 | 多个 Agent 做同样的事 | 分工不清 | 明确输入、输出和验收标准 |
| 协调成本高 | 大量互相对话但结果少 | 缺少主持和停止条件 | 引入 Supervisor 和轮次上限 |
| 错误放大 | 一个 Agent 的错误被其他 Agent 接受 | 缺少证据要求 | 要求结论带证据和来源 |
| 难以调试 | 出错后不知道谁导致 | Trace 不完整 | 独立记录每个 Agent 的输入输出 |
| 成本失控 | 多 Agent 重复检索和推理 | 没有预算 | 设置并发、轮次、token 上限 |
| 决策不清 | 多个建议冲突 | 没有最终裁决者 | 指定最终决策 Agent 或人工确认 |

### 6.7 判断边界：什么时候不需要 Multi-Agent

以下情况不建议引入 Multi-Agent：

- 单 Agent 加清晰工具和上下文管理已经能完成。
- 所谓多角色只是改了名字，没有真实分工。
- 任务无法并行。
- 没有 Trace 和评测，无法调试复杂协作。
- 成本和延迟已经紧张。
- 用户真正需要的是固定流程或人工确认，而不是多个 Agent。

实用判断：

```text
如果问题是”角色冲突、并行探索、互相审查”，可以考虑 Multi-Agent。
如果问题是”工具不好用、上下文混乱、状态不可恢复”，先修课程四和课程六的问题。
```

> **五类能力走完了。** 你跟着 1.1 的知识助手 Agent，从它第一次”不知道笔记内容”的崩溃开始，依次给它接上了 RAG（能查资料了）、Memory（能记住偏好了）、Planning（长任务不漏步骤了）、Reflection（测试失败会修正了）、Multi-Agent（能审查自己的工作了）。每加一层，系统都有新能力——也有新复杂度。最后一章要回答的是：在真实项目中，这些能力怎么组合？按什么顺序引入？以及什么时候应该——坚决地——什么都不加。

## 第七章：能力组合与引入顺序

### 7.1 不要一次性堆满能力

真实项目中，增强能力经常组合出现。例如个人知识助手可能需要外部知识接入，也可能需要 Memory；代码审查 Agent 可能需要工具调用、Reflection 和 Reviewer。

但组合不等于一次性堆满。

更好的顺序是：

```text
先明确任务失败在哪里
  -> 用最小能力修复主要失败
  -> 评测是否改善
  -> 再处理下一个瓶颈
```

不要因为某个能力热门就加入系统。每个能力都应该有失败证据和评测方式。

### 7.2 能力组合案例

#### 案例一：个人知识助手

起始问题：

- 用户资料很多，模型不知道资料内容。
- 用户需要引用来源。

优先能力：

- RAG / 外部知识接入。

后续问题：

- 用户希望记住当前研究主题。
- 多轮追问时需要延续上下文。

再引入：

- 会话摘要和轻量 Memory。

暂不引入：

- Multi-Agent，除非出现明确的并行调研或审查需求。

#### 案例二：代码审查 Agent

起始问题：

- Agent 只读 diff 后输出泛泛建议。
- 不能验证自己的判断。

优先能力：

- 工具调用：读文件、搜索代码、运行测试。
- Reflection：测试失败后修正判断。

后续问题：

- Agent 修改代码后需要审查。

再引入：

- Reviewer 模式的轻量 Multi-Agent。

暂不引入：

- 长期 Memory，除非需要记住项目约定、测试习惯或用户偏好。

#### 案例三：个人任务执行 Agent

起始问题：

- 任务多步骤，容易漏步骤。
- 用户希望在关键节点确认。

优先能力：

- Planning / Workflow Patterns。
- Human-in-the-loop。

后续问题：

- 长任务中断后需要继续。

再引入：

- 任务状态和 Checkpoint，这部分会在课程六深入。

暂不引入：

- 自动长期 Memory，除非用户明确需要跨任务偏好延续。

### 7.3 能力引入决策表

| 当前问题 | 优先考虑 | 不建议引入的信号 |
|---|---|---|
| 答案依赖外部资料、私有知识或实时数据 | RAG / 外部知识接入 | 用户输入已包含全部信息 |
| 需要跨轮、跨会话或跨任务延续 | Memory | 每次任务独立，隐私风险高 |
| 任务步骤多、依赖强、容易漂移 | Planning / Workflow Patterns | 两三步固定流程足够 |
| 有明确失败信号，需要自动修正 | Reflection | 没有外部反馈，只是让模型自评 |
| 单 Agent 角色冲突、需要并行或互审 | Multi-Agent | 单 Agent 已可完成，缺少 Trace 和评测 |

### 7.4 写一份能力引入说明

一个成熟的 Agent 设计说明，应该能解释为什么引入某个能力，也能解释为什么不引入某个能力。

模板如下：

```text
能力名称：
当前场景：
用户问题：
失败证据：
为什么现有最小闭环不够：
为什么选择该能力：
最小实现版本：
新增复杂度：
评测方式：
不引入的边界：
后续迭代：
```

示例：

```text
能力名称：RAG / 外部知识接入
当前场景：个人知识助手回答用户笔记问题
用户问题：答案必须基于本地笔记，并给出引用
失败证据：无检索时模型输出通用解释，无法引用来源
为什么现有最小闭环不够：模型没有访问用户资料
为什么选择该能力：需要把本地资料作为回答证据
最小实现版本：Markdown 文档切分 + 向量检索 + 引用输出
新增复杂度：文档解析、切分、引用校验、权限边界
评测方式：20 个问题，检查召回、答案、引用和拒答
不引入的边界：用户直接提供完整资料时，不使用检索
后续迭代：混合检索、rerank、引用后校验
```

### 练习二：写一份“不引入某能力”的说明

选择一个项目，写一份不引入某能力的说明。

例如：

- 为什么个人知识助手第一版不引入 Multi-Agent。
- 为什么代码审查 Agent 第一版不引入长期 Memory。
- 为什么简单摘要工具不引入 Planning。

说明必须包含：

- 当前任务边界。
- 不引入的原因。
- 未来什么条件下会重新考虑。

### 练习三：设计外部知识接入方案

为个人知识助手设计一个外部知识接入方案：

- 数据来源。
- Chunking 策略。
- 索引方式。
- 查询改写方式。
- 引用格式。
- 无答案处理。
- 权限和隐私边界。
- 评测集。

### 练习四：设计 Memory 策略

为同一个项目设计 Memory 策略：

- 哪些内容只属于任务状态。
- 哪些内容属于会话摘要。
- 哪些内容可以长期保存。
- 哪些内容永不保存。
- 用户如何查看和删除记忆。
- 错误记忆如何纠正。

### 练习五：选择任务组织模式

为以下任务选择 Chain、Router、ReAct Loop、Plan-Execute 或 Graph：

- 把一篇文章改写成小红书风格。
- 判断用户问题应该走文档问答还是代码审查。
- 修复一个测试失败的代码问题。
- 完成一次项目发布准备。
- 处理一个包含多阶段审批的企业流程。

说明你选择的理由和不选择其他模式的原因。

## 下一课衔接

课程五解决的是"什么时候引入哪些增强能力"。

但一旦你开始引入这些能力，新的工程问题会出现：

- 外部知识、Memory 和工具结果如何进入上下文？
- 长任务中断后如何恢复？
- 多步执行如何 Trace？
- 失败如何进入评测？
- 权限和 Prompt Injection 如何嵌入运行时？
- 多 Agent 或子任务委托如何调试？

课程六会进入 Harness 运行时架构，学习如何把最小 Agent 闭环和本课的增强能力，变成可配置、可追踪、可恢复、可评测、可治理的工程系统。
