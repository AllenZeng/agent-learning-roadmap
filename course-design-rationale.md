# 课程设计思考：支柱划分与体系逻辑

本文档记录课程体系设计中的关键决策和思考过程，说明"为什么这样划分"以及"边界在哪"。

---

## 一、V1→V2：为什么从六阶段调整为七阶段

### 1.1 V1 的问题

V1 的六课程结构（基础认知 → 技术基础 → 架构深入 → 产品化 → 前沿生态 → 作品集）在早期设计中是合理的，但存在几个问题：

- **课程一（基础认知）内容过重**：既讲历史渊源、又讲范式演进、又讲产品拆解，认知负荷太大
- **"历史演进"和"动手实践"混在一起**：学习者不清楚什么时候该理解、什么时候该动手
- **缺少一个纯粹的"入门引导"阶段**：应该先建立感性认识，再讲原理

### 1.2 V2 的改进

将 V1 课程一拆分为两个阶段：**初识Agent**（感性认知）→ **Agent演进**（理性理解），形成七课程结构：

```
课程一（初识Agent）    →  建立感性认知，先"看见"Agent
课程二（Agent演进）     →  理解范式从何而来，理论准备
课程三（底层核心）      →  动手起点，五大支柱
课程四（架构深入）      →  从能用到好用
课程五（走向生产环境）  →  从好用到能上线
课程六（项目实战）      →  整合验证
课程七（最新发展趋势）  →  保持敏锐，持续跟进
```

核心原则：
1. 每门课程有单一、明确的目标（认知 → 理解 → 动手 → 深入 → 产品化 → 实战 → 跟进）
2. 课程之间顺序依赖清晰，不会让学习者在"该做什么"上迷茫
3. 五大支柱作为课程的核心锚点，其余内容围绕它展开

### 1.3 V1→V2 映射速查

| V1（六阶段） | V2（七个课程） | 变更说明 |
|-------------|-------------|---------|
| 课程一：基础认知 | **课程一：初识Agent** + **课程二：Agent演进** | 拆分为二，认知先行，演进后置 |
| 课程二：技术基础 | **课程三：Agent底层核心** | 内容基本一致，五大支柱 |
| 课程三：架构深入 | **课程四：Agent架构深入** | 内容基本一致，Harness + 五大架构模块 |
| 课程四：产品化 | **课程五：走向生产环境** | 内容基本一致 |
| — | **课程六：项目实战** | 原课程六，位置不变 |
| 课程五：前沿生态 | **课程七：最新发展趋势** | 内容基本一致 |

## 二、核心设计原则

### 2.1 支柱的定义标准

一个问题或技术要被列为课程三的"支柱"，需要同时满足三个条件：

1. **解决的必须是 LLM 本身的根本局限**，而非工程优化问题
2. **不解决就根本不能用的级别**——不是"做不好"，是"做不到"
3. **有独立的技术体系和公认的解决方案**，不是横切关注点

### 2.2 课程分层的逻辑

```
课程三（五大支柱）    →  解决"能不能"的问题（能力扩展）
课程四（架构深入）    →  解决"好不好"的问题（质量保障）
课程五（走向生产环境）→  解决"能不能上线"的问题（工程落地）
课程七（最新发展趋势）→  了解"正在发生什么"（视野拓展）
课程六（项目实战）    →  验证"我到底会了没有"（能力整合）
```

## 三、五大支柱的确认过程

### 3.1 最终确定的五大支柱

| 支柱 | 解决的 LLM 局限 | 确立理由 |
|------|---------------|---------|
| **Tool Use** | 只能"说"不能"做" | 无争议。没有工具调用，Agent 连最基础的任务都无法完成 |
| **RAG** | 知识有截止日期、缺乏专业领域知识 | 无争议。LLM 的知识是静态的、训练时固化的 |
| **Memory** | 上下文窗口有限、跨会话无状态 | 最初放在课程四，经讨论确认为与上述同级别的根本局限，已调回课程三 |
| **Prompt Engineering** | 输出不可控、不稳定 | 无争议。这是人类与 LLM 交互的"编程语言" |
| **ReAct** | 单次调用无法处理复杂多步任务 | 有一定讨论空间（见下文 3.3），最终确认单列 |

### 3.2 Memory 的归属争议

**最初设计**：Memory 放在课程四（架构深入），与 Orchestration、Evaluation 等并列。

**质疑**：Memory 解决的"上下文窗口有限、跨会话无状态"是 LLM 的根本局限，和 Tool Use、RAG 同级。为什么把它放架构层？

**重新审视后的结论**：Memory 应该出现在课程三。理由：

- 上下文窗口管理（短期记忆）是每次 LLM 调用都涉及的基础问题，不是"架构优化"
- 即使是最简单的 Agent（没有向量数据库、没有编排），也需要"记住用户偏好"的能力
- RAG 和 Memory 共享技术栈（Embedding、向量检索），但解决的问题不同——RAG 解决"世界知识"，Memory 解决"个人经验"

**最终方案**：课程三讲 Memory 基础（概念分层 + 最简实现），课程四讲 Memory 架构深入（向量数据库、MemGPT、记忆衰减）。

### 3.3 ReAct 是否应该单列

**争议点**：ReAct 不是独立于其他支柱的技术——它依赖 Tool Use（工具调用）和 Prompt Engineering（推理提示），本质上是一种"编排模式"。是否应该归入 Tool Use 或编排？

**单列的理由**：

- ReAct 解决了一个独立的根本局限：**LLM 单次调用只能处理原子任务**。这是和"只能说不"同级别的问题
- ReAct 论文（2022）在 LLM Agent 发展史上有里程碑地位，它首次证明了"思考-行动-观察"循环的可行性
- 将 ReAct 单列，可以让学习者在一开始就建立"Agent ≠ 单次 LLM 调用"的心智模型

**不单列的理由**：

- ReAct 更多是一种"使用模式"而非独立的技术栈
- 复杂任务分解也可以通过 Plan-then-Execute、Tree-of-Thoughts 等其他方案解决

**当前取舍**：单列。因为从教学角度看，ReAct 是理解"Agent 为什么需要循环"的入口，这个认知应该在基础阶段建立。

## 四、未被列入支柱的候选

### 4.1 能力局限 vs 质量局限

关键在于区分两个层次的问题：

**能力局限**（不解决就做不到）：这构成了五大支柱。
**质量局限**（能做到但不够好）：这些在架构层（课程四/五）解决。

| 质量局限 | 表现 | 解决方案 | 课程位置 |
|---------|------|---------|---------|
| 会"胡说"（幻觉） | 一本正经地编造不存在的事实 | RAG（治标）、事实校验工具、citation 链条 | 课程三(RAG) + 课程四 |
| 会"越界"（安全性） | 被注入攻击、执行危险操作 | Guardrails、Human-in-the-Loop、权限分级 | 课程四 + 课程五 |
| 会"迷路"（规划弱） | 复杂任务走偏方向、遗漏步骤 | Orchestration、Plan-then-Execute、动态重规划 | 课程四 |
| 会"变慢变贵"（效率差） | Token 消耗大、延迟高 | 模型路由、Prompt Caching、上下文压缩 | 课程五 |
| 行为不可解释 | 不知道为什么要做某个决策 | Tracing、决策归因、可视化回放 | 课程四 |
| 行为不稳定 | 同样的输入，两次结果不同 | 评测体系、回归测试、Self-Consistency | 课程四 |
| 不会自我纠正 | 错了就一路错到底 | Reflection、Self-Refine、Reflexion | 课程二 + 课程四 |

### 4.2 Planning（规划）是否应列第六支柱

**候选理由**：
- CoT、ToT、Plan-and-Solve 等技术的体量足以独立成章
- "规划"是 Agent 的核心能力之一

**不列的理由**：
- 规划的"基础版"（CoT）已经包含在 Prompt Engineering 中
- 规划的"进阶版"（编排调度）在课程四的 Orchestration 中覆盖
- 单独成支柱会导致与 ReAct（怎么做）和编排（怎么组织）的边界模糊

**当前处理**：通过 ReAct + Orchestration 两条线覆盖规划的全景，不单列。

### 4.3 Safety/Alignment 是否应列第六支柱

**候选理由**：
- "LLM 缺乏安全边界意识"是和五大支柱同级别的根本问题
- Anthropic 的"Building effective agents"强调安全应内置而非外挂

**不列的理由**：
- Safety 是**横切关注点（cross-cutting concern）**，影响所有支柱而非独立存在
- 把它放在任何一个层级都不合适——它应该渗透到 Tool Use（工具权限）、Memory（记忆隐私）、Prompt Eng（指令隔离）等每个环节
- 当前课程将 Guardrails 放在课程四、安全合规放在课程五，正是因为它们更多是"工程实践"而非"能力扩展"

**这个决定值得持续反思**。如果未来 Safety 领域出现像 MCP 之于 Tool Use 那样的标准化方案，将其提升为支柱会更合理。

### 4.4 Harness（运行时引擎）

**候选理由**：
- 所有 Agent 系统都有 Harness，是通用抽象
- 在课程四中已经作为独立课程（第零课）出现

**不列的理由**：
- Harness 不是解决 LLM 局限的解决方案，而是**组织这些解决方案的运行框架**
- 类比：汽车引擎不是解决"如何跑得更快"的方案，而是"让所有零件协同工作的容器"
- 放在课程四的编排之前讲，作为架构的入口，位置更合适

## 五、课程的层次结构

### 5.1 完整的能力分层

```
┌─────────────────────────────────────────────────────────┐
│                    课程六：项目实战                        │
│                  验证"我到底会了什么"                       │
└──────────────────────────┬──────────────────────────────┘
                           │
┌──────────────────────────┴──────────────────────────────┐
│                    课程七：最新发展趋势                     │
│         MCP/A2A / Computer Use / Multi-Agent / RL        │
│                 了解"正在发生什么"                          │
└──────────────────────────┬──────────────────────────────┘
                           │
┌──────────────────────────┴──────────────────────────────┐
│                    课程五：走向生产环境                     │
│      交互设计 / 可靠性 / 成本控制 / 安全合规 / 评价体系       │
│                   解决"能不能上线"                          │
└──────────────────────────┬──────────────────────────────┘
                           │
┌──────────────────────────┴──────────────────────────────┐
│                课程四：架构深入（质量保障层）                 │
│   Harness / Orchestration / Memory架构 / Evaluation /   │
│              Guardrails / Observability                  │
│                   解决"好不好"的问题                         │
└──────────────────────────┬──────────────────────────────┘
                           │
┌──────────────────────────┴──────────────────────────────┐
│              课程三：五大支柱（能力扩展层）                   │
│   LLM原理 / Tool Use / RAG / Memory / Prompt Engineering │
│           / ReAct                                        │
│                 解决"能不能"的问题                          │
└──────────────────────────┬──────────────────────────────┘
                           │
┌──────────────────────────┴──────────────────────────────┐
│              课程一 + 课程二：认知与理解                     │
│         概念辨析 / 范式演进 / 产品拆解 / 历史脉络             │
│                建立"这是什么"的认知                          │
└─────────────────────────────────────────────────────────┘
```

### 5.2 横切关注点

以下概念不归属某一层，而是贯穿所有层级：

| 横切概念 | 贯穿方式 |
|---------|---------|
| **Safety** | 工具权限设计（课程三）→ Guardrails（课程四）→ 安全合规（课程五）→ MCP安全模型（课程七） |
| **评测** | 从课程三的"10条测试任务"开始，到课程四的系统化评测框架，到课程五的产品指标 |
| **成本意识** | 课程三（Token化理解）→ 课程四（观测Token消耗）→ 课程五（成本控制方案） |
| **可观测性** | 课程三（Trace日志）→ 课程四（Tracing架构）→ 课程五（产品监控） |

## 六、已知的取舍和待讨论的问题

### 6.1 已做出的取舍

| 决策 | 取舍 | 理由 |
|------|------|------|
| Memory 放课程三 | 增加了课程三的认知负荷，但逻辑更一致 | Memory 是根本局限，应与其他支柱同级 |
| Planning 不单列 | 损失了"规划"作为一个统一概念的表达力 | 通过 ReAct + 编排双线覆盖，避免了与邻接概念的边界模糊 |
| Safety 不列支柱 | Safety 的重要性未被充分突出 | Safety 是横切关注点，渗透式设计比独立成章更合理 |
| Harness 放课程四 | 基础层接触不到这个概念 | Harness 是"组织方案"而非"能力扩展"，放架构入口更合适 |
| V1 课程一拆分为二 | 增加了课程总数，但认知节奏更好 | 感性认知和理性理解不应混在一起，拆分后每门课程目标更单一 |

### 6.2 待持续思考的问题

1. **Safety 是否最终需要成为支柱**：如果 Agent 安全领域出现类似 ReAct 那样的范式级方案（不仅是最佳实践，而是技术上的根本性突破），重新考虑
2. **ReAct 是否应该合并到 Tool Use**：随着 Tool Use 概念逐渐包含"使用工具的多步推理"，ReAct 的独立性可能减弱
3. **课程三和课程四的边界**：随着领域发展，某些"架构深入"内容可能被证明是基础能力，需要上调到课程三
4. **是否有遗漏的支柱**：LLM 的"推理不可靠性"当前分散在多个支柱中处理（CoT、Reflection、Evaluation），未来可能需要一个统一的"Reasoning Engineering"支柱
