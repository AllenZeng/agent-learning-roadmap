# 课程七：Agent 产品化实践

## 课程导言

前面几课已经把 Agent 的技术主线讲完整了：课程三讲最小闭环，课程四讲工具机制，课程五讲场景增强能力，课程六讲运行时工程架构。到了这一课，问题会变成另一个层面：

**一个 Agent 能跑起来，不等于它能成为产品。**

技术 Demo 通常只需要证明"它能完成一次任务"。产品化则要回答更难的问题：用户愿不愿意等？用户敢不敢让它执行？失败时用户能不能接管？成本是否可控？风险是否可审计？团队能不能持续改进？

很多 Agent 项目不是因为模型不够强而失败，而是因为产品机制没有设计好：

- 用户不知道 Agent 正在做什么，于是觉得慢。
- Agent 偶尔做对一次，但用户无法判断什么时候可以信任它。
- 工具调用失败后，用户只看到一句泛泛的错误提示。
- 一次任务跑了十几步，成本和延迟上线后才暴露。
- 高风险动作没有确认、审批、回滚和审计。
- 团队看不到线上失败的结构化数据，下一轮迭代只能凭感觉。

本课要解决的不是"如何再加一个能力"，而是**如何把 Agent 的能力转化为用户可理解、业务可验收、组织可运营的产品机制**。

学完这一课，你应该能把一个可运行的 Agent 原型，改造成一个更接近真实产品的版本。

---

## 学习目标

学完本课，你将能够：

1. **解释 Agent 产品化的核心矛盾**：为什么"能完成任务"还不够，为什么用户会觉得慢、不可信、不可控
2. **设计交互与信任机制**：用进度展示、可编辑计划、确认节点、接管和回滚降低用户的不确定感
3. **设计可靠性与恢复机制**：用重试、降级、幂等、超时、任务恢复处理不可避免的失败
4. **拆解成本与延迟来源**：估算单次任务成本，识别 token、工具、检索、重试和排队带来的开销
5. **设计安全、权限与审计机制**：让 Agent 的高风险动作可授权、可追踪、可回滚、可复盘
6. **建立产品指标与反馈闭环**：用任务完成率、用户干预率、失败恢复率、成本和风险指标指导迭代
7. **完成一次 Agent 产品复盘**：把失败从"现象描述"转化为可复现、可归类、可改进的产品问题

## 本课在路线图中的位置

课程六解决的是运行时是否稳定，课程七解决的是产品是否可信、业务是否可衡量、风险是否可治理、团队是否能持续迭代。

你可以把前面几课和本课的关系理解为：

```text
最小闭环 -> 工具机制 -> 场景增强 -> 运行时架构 -> 产品化机制
```

本课不重复讲工具、RAG、Memory 或 Harness 的实现细节，而是回答：

- 这些能力如何被用户看见？
- 哪些步骤必须让用户确认？
- 失败后如何继续，而不是让任务中断？
- 线上数据如何进入评测和下一轮改进？
- 风险和成本如何在产品层面被约束？

如果课程三到课程六是在搭建 Agent 的"发动机、传动和仪表盘"，课程七就是在设计它能否被真实用户安全驾驶。

## 第一章：产品化问题总览

### 1.1 Demo 和产品的差异

Agent Demo 通常围绕一次成功案例展开：

1. 用户输入一个目标。
2. Agent 制定计划。
3. Agent 调用工具。
4. Agent 得到结果。
5. 用户看到一个看起来不错的输出。

这能证明技术可行，但不能证明产品成立。

真实产品会遇到更多约束：

- 用户输入不清楚，Agent 是否追问？
- 任务执行很慢，用户是否知道它卡在哪里？
- 工具返回错误，系统是否有恢复路径？
- Agent 想执行高风险动作，是否需要确认？
- 输出结果看起来合理，但用户如何判断是否可信？
- 同一个任务跑 100 次，成本、延迟和成功率是否稳定？
- 出现事故后，团队能否回放当时发生了什么？

所以，产品化的目标不是"让 Agent 看起来更聪明"，而是让它在真实使用中更可控、更可信、更可运营。

### 1.2 Agent 产品化问题地图

Agent 产品化可以从六类问题切入：

| 产品化问题 | 用户感受 | 对应机制 |
|---|---|---|
| Agent 慢 | 不知道它在干什么，等不下去 | 流式输出、进度展示、步骤解释、延迟拆解 |
| Agent 不可信 | 不敢让它自动执行 | Human-in-the-loop、可编辑计划、确认节点、回滚 |
| Agent 容易失败 | 出错后用户不知道怎么办 | 重试、降级、恢复、失败后的下一步建议 |
| Agent 太贵 | 单次任务成本不可控 | 模型路由、缓存、上下文裁剪、任务级成本上限 |
| Agent 有风险 | 可能泄露数据或越权操作 | 权限隔离、审批、审计、安全合规 |
| Agent 价值不清楚 | 不知道是否真的比人工或传统流程更好 | 任务完成率、用户干预率、满意度、留存、ROI |

这个表的作用是把"产品体验问题"和"系统机制设计"连接起来。

例如，用户说"这个 Agent 很慢"，这不一定只是模型延迟问题。可能是：

- 没有流式输出，用户只能看到空白等待。
- 没有展示当前步骤，用户不知道任务是否还在推进。
- 工具调用串行执行，等待时间被放大。
- 每一步都把全部历史上下文塞回模型，token 量过大。
- 失败后静默重试，用户看到的是长时间无反馈。

产品化分析要避免只听表面抱怨，而是把用户感受拆成可观察、可度量、可优化的系统环节。

### 1.3 产品化设计顺序

一个 Agent 项目的产品化设计，建议按下面的顺序推进：

1. **定义任务边界**：Agent 负责什么，不负责什么，什么情况下必须拒绝或转人工
2. **设计用户可见过程**：计划、当前步骤、工具调用、等待状态、失败原因如何展示
3. **定义确认节点**：哪些动作可以自动执行，哪些动作必须用户确认
4. **设计失败恢复**：不同失败如何重试、降级、恢复、接管
5. **约束成本和延迟**：为任务设定预算、超时、模型路由和缓存策略
6. **设计权限和审计**：谁授权、谁执行、执行了什么、能否回滚
7. **建立评测和反馈闭环**：把线上结果、失败案例、用户反馈进入下一轮改进

这个顺序很重要。不要一开始就讨论复杂指标，也不要先做漂亮界面。产品化首先要让任务边界、信任边界和风险边界清楚。

### 1.4 产品化验收视角

一个 Agent 产品是否进入可用阶段，可以从以下问题验收：

| 验收问题 | 合格表现 | 不合格表现 |
|---|---|---|
| 用户是否知道 Agent 在做什么 | 能看到计划、进度、当前步骤和等待原因 | 只看到转圈或长时间无反馈 |
| 用户是否能控制关键动作 | 高风险动作前有确认，用户能驳回或接管 | Agent 自动执行不可逆动作 |
| 失败是否可恢复 | 有重试、降级、恢复和下一步建议 | 失败后只返回笼统错误 |
| 成本是否可解释 | 能看到任务级成本估算和异常成本来源 | 上线后才通过账单发现问题 |
| 风险是否可审计 | 有权限、审批、日志和回滚记录 | 出事后无法还原执行链路 |
| 价值是否可衡量 | 有任务完成率、干预率、满意度、ROI | 只看调用量或演示效果 |

产品化不是一次性包装，而是一套持续迭代机制。

## 第二章：交互与信任

### 2.1 用户为什么不信任 Agent

用户不信任 Agent，通常不是因为他们完全不相信 AI，而是因为他们不知道 Agent 的边界在哪里。

典型的不信任来源包括：

- **过程不可见**：用户不知道 Agent 正在思考、等待工具，还是已经卡住。
- **决策不可解释**：用户不知道它为什么选择这个工具、这个计划、这个结论。
- **风险不可控**：用户担心 Agent 会误删文件、发错邮件、泄露数据或提交错误代码。
- **失败不可恢复**：一旦 Agent 出错，用户不知道如何继续。
- **结果不可验证**：输出看似合理，但缺少证据、引用或执行日志。

因此，交互设计的目标不是让 Agent 表演智能，而是让用户知道：

- 它现在在做什么。
- 它为什么这么做。
- 它下一步打算做什么。
- 哪些地方需要我确认。
- 如果它错了，我如何纠正或接管。

### 2.2 让过程可见

Agent 任务通常不是一次模型调用，而是多轮循环。用户如果只看到最终结果，很容易把中间等待理解为"卡住了"。

过程可见至少包括四层信息：

| 信息层 | 用户看到什么 | 产品价值 |
|---|---|---|
| 任务目标 | Agent 当前要完成的目标 | 防止目标漂移 |
| 执行计划 | 准备做哪些步骤 | 让用户提前发现方向错误 |
| 当前状态 | 正在检索、读取文件、调用工具、等待返回 | 降低等待焦虑 |
| 中间证据 | 已找到的资料、工具返回、测试结果 | 建立结果可信度 |

一个简单的代码审查 Agent，可以展示：

```text
目标：审查当前 Git diff
计划：
1. 读取变更文件
2. 判断风险区域
3. 运行相关测试
4. 输出按严重程度排序的 review

当前步骤：正在运行 npm test
结果摘要：测试失败 2 个，集中在 user-service 模块
```

这些信息不需要很花哨，但必须稳定、及时、可理解。

### 2.3 可编辑计划与确认节点

Agent 的计划不应该永远藏在模型内部。只要任务跨度较长、风险较高，就应该让用户看到并编辑计划。

可编辑计划通常适合以下场景：

- 任务涉及多个外部系统。
- 任务会修改文件、数据库、配置或线上资源。
- 用户目标含糊，需要先确认执行方向。
- 不同执行路径成本差异很大。
- 用户可能更关心过程控制，而不只是最终答案。

计划展示不只是 UI 细节，它会改变用户和 Agent 的协作关系：

- 用户可以在执行前修正方向。
- Agent 可以把不确定性暴露出来。
- 高风险步骤可以自然变成确认节点。
- 后续失败复盘可以对照计划和实际执行差异。

确认节点要按风险分级，而不是每一步都打断用户。

| 动作类型 | 示例 | 建议处理 |
|---|---|---|
| 只读动作 | 搜索、读取文档、查看 diff | 通常自动执行 |
| 可逆低风险写入 | 生成草稿、创建临时文件 | 可自动执行，但要可撤销 |
| 中风险动作 | 修改项目文件、发送内部消息 | 执行前确认或执行后明确回滚路径 |
| 高风险动作 | 支付、删除数据、发布上线、发送外部邮件 | 必须确认、记录审批、支持回滚或补救 |

### 2.4 Human-in-the-loop

Human-in-the-loop 不是"Agent 不够强，所以让人兜底"。它是产品设计中主动定义人机分工的机制。

适合加入人工节点的位置包括：

- **目标澄清**：用户目标不完整，继续执行会放大错误。
- **计划确认**：任务复杂或涉及多条路径。
- **高风险动作前**：写入、删除、发布、支付、外部发送。
- **低置信度判断后**：Agent 给出多个候选，但无法确定最佳选项。
- **失败恢复时**：自动恢复路径会带来额外风险。
- **最终交付前**：输出将进入正式场景。

设计 Human-in-the-loop 时，要避免两种极端：

- 所有动作都要确认，用户变成按钮操作员。
- 所有动作都自动执行，用户只能事后发现错误。

更好的方式是按风险和置信度动态决定是否需要人参与。

```text
是否需要人工确认 =
动作风险等级 + Agent 置信度 + 可逆性 + 用户历史授权范围
```

### 2.5 Progressive Trust

Progressive Trust 指的是：不要一开始就要求用户完全信任 Agent，而是让信任随着成功经验逐步增加。

例如一个代码审查 Agent 可以这样设计：

1. 初始阶段：只输出建议，不自动修改代码。
2. 稳定后：允许 Agent 生成 patch，但用户手动应用。
3. 进一步稳定后：允许 Agent 修改低风险文件，但提交前必须确认。
4. 更高信任阶段：对特定目录、特定规则、特定测试通过的改动自动提交到分支。

Progressive Trust 的关键是授权范围要具体：

- 哪类任务可以自动执行。
- 哪些工具可以自动调用。
- 哪些数据可以访问。
- 哪些动作必须确认。
- 授权什么时候过期。
- 用户如何撤销授权。

信任不是一句"AI 可以帮你完成任务"，而是一组明确的权限边界和成功记录。

### 2.6 中断、接管与回滚

Agent 产品必须允许用户中断和接管。原因很简单：Agent 的执行路径不是完全可预测的。

基础能力包括：

- 暂停当前任务。
- 查看当前状态。
- 修改目标或计划。
- 跳过某一步。
- 手动执行某一步后让 Agent 继续。
- 回滚 Agent 已执行的可逆动作。
- 从失败节点恢复，而不是从头开始。

中断和恢复需要依赖课程六讲过的 Checkpoint 和状态持久化。产品层面要把这些工程能力转化为用户可理解的操作：

| 工程能力 | 产品表达 |
|---|---|
| 保存状态 | 任务可暂停、可恢复 |
| 记录工具调用 | 用户能看到 Agent 做过什么 |
| 记录写入动作 | 用户能撤销或回滚 |
| 保存计划和进度 | 用户能从某一步继续 |
| 保存失败原因 | 用户能知道下一步可选方案 |

## 第三章：可靠性与恢复

### 3.1 Agent 失败为什么不可避免

Agent 的失败来源比传统应用更多，因为它同时依赖模型、工具、上下文、外部环境和运行时状态。

常见失败包括：

- 模型误解用户目标。
- 模型选择了错误工具。
- 工具参数不合法。
- 工具调用超时或返回错误。
- 外部系统数据变化。
- RAG 检索到错误信息。
- Memory 写入了污染信息。
- 多步任务中某一步失败导致后续状态不一致。
- Agent 陷入重复尝试或过早停止。

产品化的目标不是消除所有失败，而是让失败可分类、可恢复、可解释。

### 3.2 重试策略

重试是最常见的恢复方式，但不是所有失败都应该重试。

适合自动重试的情况：

- 网络抖动。
- 外部 API 临时超时。
- Rate Limit 后等待窗口可恢复。
- 模型输出格式不合法，但可通过结构化修复重试。
- 工具执行失败且没有产生副作用。

不适合盲目重试的情况：

- 用户目标本身不清楚。
- 权限不足。
- 工具参数来自错误理解。
- 已经产生不可逆副作用。
- 每次重试都会消耗较高成本。
- 失败原因是外部条件不满足，例如账号未登录、资源不存在。

重试策略至少要定义：

| 设计项 | 说明 |
|---|---|
| 最大重试次数 | 防止死循环和成本失控 |
| 重试间隔 | 固定间隔、指数退避或按服务端提示等待 |
| 重试范围 | 重试模型调用、工具调用，还是整个步骤 |
| 重试前修正 | 是否根据错误信息调整参数或提示词 |
| 重试可见性 | 是否向用户展示正在重试和重试原因 |

### 3.3 超时、Rate Limit 与排队

Agent 任务通常会串联多个慢环节。只要其中一个工具没有超时控制，整个任务就可能被拖住。

必须设计的超时包括：

- 模型调用超时。
- 工具调用超时。
- 检索超时。
- 文件读写超时。
- 用户确认等待超时。
- 整体任务超时。

Rate Limit 不应该只被当成异常处理。产品层面要考虑：

- 超过限制时是否排队。
- 是否降级到备用模型或备用工具。
- 是否提示用户预计等待时间。
- 是否允许用户稍后继续。
- 是否对不同用户或任务设置优先级。

对于长任务，排队体验也很重要。用户至少需要知道：

- 任务已进入队列。
- 当前排队原因。
- 预计开始时间或等待区间。
- 是否可以取消。
- 是否可以在后台完成后通知。

### 3.4 降级策略

降级不是失败后的随意补救，而是提前设计好的产品路径。

常见降级方式：

| 降级类型 | 示例 | 适用场景 |
|---|---|---|
| 模型降级 | 大模型不可用时切到小模型 | 任务低风险、可接受质量下降 |
| 工具降级 | 实时搜索失败时使用缓存结果 | 外部服务不稳定 |
| 功能降级 | 无法自动执行时只生成操作建议 | 高风险动作或权限不足 |
| 输出降级 | 无法生成完整报告时先给摘要和缺口 | 长任务部分完成 |
| 人工降级 | 自动恢复失败后转人工确认 | 风险或不确定性过高 |

降级必须向用户说明边界。不要在用户不知情的情况下用低质量路径替代高质量路径，否则会伤害信任。

### 3.5 幂等设计

幂等设计是 Agent 产品化里经常被忽视但非常重要的一点。

Agent 会重试，会恢复，会从中断点继续。如果一个动作重复执行会造成副作用，就必须有幂等保护。

高风险例子：

- 重复发送同一封邮件。
- 重复提交同一个表单。
- 重复创建订单。
- 重复删除或覆盖文件。
- 重复发布同一个版本。

幂等设计的常见方法：

- 为每个动作生成唯一 request id。
- 写入前检查是否已经执行过。
- 将写入分成草稿和确认两步。
- 对文件修改使用 patch 和 diff，而不是直接覆盖。
- 将外部动作记录到审计日志。
- 恢复任务时先检查上一步是否已完成。

如果系统没有幂等设计，就不要让 Agent 自动执行不可逆动作。

### 3.6 任务恢复

任务恢复解决的是：失败或中断后，如何继续完成任务。

恢复需要依赖四类信息：

| 信息 | 作用 |
|---|---|
| 原始目标 | 确认任务仍然朝向用户意图 |
| 当前计划 | 判断已经完成哪些步骤 |
| 状态快照 | 恢复上下文、工具结果、文件变更 |
| 失败原因 | 决定重试、降级、追问还是转人工 |

任务恢复不是简单地把历史消息塞回模型。恢复时要重建的是可执行状态：

- 哪些步骤已经完成。
- 哪些工具调用成功。
- 哪些外部副作用已经发生。
- 哪些信息仍然有效。
- 哪些信息需要重新确认。
- 下一步是否安全。

这也是为什么课程六强调 Checkpoint、Trace 和 append-only 状态记录。

### 3.7 失败处理策略表

设计 Agent 产品时，可以先建立一张失败处理表：

| 失败类型 | 典型原因 | 自动处理 | 是否提示用户 | 是否进入复盘 |
|---|---|---|---|---|
| 模型输出格式错误 | JSON 不合法、字段缺失 | 结构化修复或重试 | 通常不提示 | 频繁出现时进入 |
| 工具参数错误 | 模型理解错误、schema 不清 | 重新生成参数或追问用户 | 视风险提示 | 是 |
| 工具超时 | 网络、服务不可用 | 重试、降级、排队 | 是 | 是 |
| 权限不足 | 用户未授权、越权访问 | 请求授权或停止 | 是 | 是 |
| 外部知识错误 | 检索质量差、来源过期 | 更换来源、增加引用检查 | 是 | 是 |
| 高风险动作受阻 | 需要确认、审批未通过 | 暂停等待确认 | 是 | 是 |
| 死循环 | 计划不收敛、反馈判断错误 | 达到步数上限后停止 | 是 | 是 |
| 用户目标不清 | 输入缺少必要条件 | 追问 | 是 | 通常不需要 |

这张表不需要一开始覆盖所有情况，但至少要覆盖主路径上的高频失败和高风险失败。

## 第四章：成本与延迟

### 4.1 成本和延迟是产品问题

Agent 成本和延迟不能只交给工程团队上线后优化。它们会直接影响产品形态。

如果一次任务需要 2 分钟完成，就不能设计成普通聊天问答体验；它更像一个后台任务，需要进度、通知和恢复。

如果一次任务成本较高，就不能让用户无感地无限重试；需要预算、上限、确认和缓存。

如果不同任务成本差异很大，就不能只看平均值；需要看 P95、P99 和异常任务。

产品经理、设计师和工程师都应该理解成本和延迟来源，因为它们决定了：

- 哪些任务适合自动执行。
- 哪些任务适合异步执行。
- 哪些任务需要用户确认。
- 哪些任务应该限制次数。
- 哪些任务必须先做成本估算。

### 4.2 单次任务成本估算

最基础的成本估算公式是：

```text
单次任务成本 = 模型输入/输出 token 成本 + 工具/API 成本 + 重试成本
```

例如，一个 10 步任务，如果每步平均消耗 4000 token，总量就是约 40000 token。再乘以所选模型的单价，就能得到粗略成本。

但真实成本还要考虑：

- 系统提示词和工具定义重复进入上下文的成本。
- RAG 检索结果进入上下文的成本。
- 工具返回内容过长带来的成本。
- Reflection、重试和评测调用带来的额外成本。
- 多模型路由中不同模型的单价差异。
- LLM-as-Judge 或后台质量检查的成本。

成本估算不需要一开始非常精确，但必须能回答：

- 平均一次任务大概多少钱。
- 最贵的一类任务为什么贵。
- 失败重试会把成本放大多少。
- 哪些步骤可以缓存或裁剪。
- 是否需要对用户设置任务级预算。

### 4.3 上下文和 token 控制

Agent 成本最常见的失控来源是上下文不断膨胀。

常见问题：

- 每一轮都把完整历史放进模型。
- 工具返回结果不压缩直接进入上下文。
- RAG 一次塞入太多文档片段。
- Memory 无过滤地注入。
- 子任务之间共享了过多无关信息。
- 错误和日志内容反复进入模型。

控制方式包括：

| 控制点 | 方法 |
|---|---|
| 历史消息 | 摘要、裁剪、只保留关键决策 |
| 工具结果 | 结构化提取、摘要、分页读取 |
| RAG 内容 | 限制 top-k、重排序、去重、引用压缩 |
| Memory | 按任务相关性注入，而不是全部注入 |
| 工具定义 | 只暴露当前任务需要的工具 |
| 子任务结果 | 返回结论和证据，不返回全部过程 |

上下文控制不能只追求省钱。过度裁剪会损害质量，所以要结合评测验证。

### 4.4 模型路由

模型路由的核心思想是：不同任务使用不同模型，不要所有问题都用最贵、最强的模型。

常见策略：

- 小模型做意图分类、简单抽取、格式修复。
- 大模型做复杂推理、计划生成、综合判断。
- 低风险任务使用低成本模型。
- 高风险任务使用更强模型并增加人工确认。
- 连续失败后升级模型。
- 简单任务在达到置信度阈值后提前结束。

模型路由需要定义清楚：

| 维度 | 问题 |
|---|---|
| 任务类型 | 这个任务是分类、抽取、生成、推理还是执行？ |
| 风险等级 | 错了会造成什么影响？ |
| 上下文长度 | 是否需要长上下文模型？ |
| 延迟要求 | 用户是否正在等待？ |
| 成本预算 | 单次任务能接受多少成本？ |
| 失败处理 | 失败后是升级模型、追问用户还是停止？ |

不要为了省成本过早引入复杂路由。先用日志看清楚成本结构，再优化高频和高成本路径。

### 4.5 缓存与复用

缓存适合用在稳定、重复、可验证的环节。

常见缓存对象：

- 系统提示词或长上下文的 Prompt Caching。
- 检索结果。
- 文档解析结果。
- 工具 schema 或权限检查结果。
- 低频变化的外部知识。
- 已完成子任务的中间结果。

缓存设计要注意失效策略：

- 数据源变化后是否失效。
- 用户权限变化后是否失效。
- 模型版本变化后是否需要重算。
- 任务目标变化后是否还能复用。
- 缓存结果是否需要记录来源和时间。

缓存的风险是使用过期信息。对于高风险输出，缓存结果应该带有来源、时间和验证方式。

### 4.6 成本预算与上限

产品层面应该设置任务级预算，而不是只在平台账单里看总成本。

预算可以按以下维度设计：

- 单次任务最大 token。
- 单次任务最大费用。
- 最大工具调用次数。
- 最大重试次数。
- 最大执行步数。
- 最大运行时间。
- 单用户每日或每月额度。

达到预算上限时，Agent 不应该直接失败，而应该给出选择：

- 用当前结果生成部分交付。
- 降级为低成本模式继续。
- 请求用户确认是否继续消耗预算。
- 暂停任务并保存状态。
- 转为手动操作建议。

成本控制本质上也是信任机制。用户知道系统不会无限消耗资源，才更愿意使用长任务 Agent。

### 4.7 延迟拆解与体验设计

延迟要拆解到环节，而不是只看总耗时。

| 延迟来源 | 常见原因 | 优化方向 |
|---|---|---|
| 模型延迟 | 模型大、上下文长、输出长 | 模型路由、上下文裁剪、流式输出 |
| 工具延迟 | 外部 API 慢、文件多、浏览器操作慢 | 并行、超时、缓存、分页 |
| 检索延迟 | 文档量大、重排序慢 | 索引优化、分层检索、缓存 |
| 排队延迟 | 并发不足、Rate Limit | 队列、优先级、异步任务 |
| 重试延迟 | 自动重试过多 | 错误分类、重试上限、用户提示 |

体验设计要根据任务耗时选择形式：

| 任务耗时 | 建议体验 |
|---|---|
| 1-3 秒 | 普通同步反馈 |
| 3-10 秒 | 流式输出和明确等待状态 |
| 10-60 秒 | 展示步骤进度和中间结果 |
| 1 分钟以上 | 后台任务、通知、可暂停恢复 |
| 不确定长任务 | 先确认预算和计划，再执行 |

## 第五章：安全、权限与合规

### 5.1 Agent 风险的产品化表达

课程四已经讲过工具权限，课程六讲过 Guardrails。本章关注的是：这些安全机制如何变成产品可以承载的治理能力。

Agent 风险常见于三类动作：

- **读数据**：读取私人文件、客户数据、内部文档、代码库、数据库。
- **写数据**：修改文件、创建记录、更新配置、写入数据库。
- **外部动作**：发送邮件、提交代码、发布内容、支付、调用第三方服务。

产品层面要把风险说清楚：

- Agent 将访问哪些数据？
- Agent 将使用哪些工具？
- Agent 是否会产生外部影响？
- 哪些动作需要用户确认？
- 用户可以在哪里撤销授权？
- 出现问题后能否追溯和回滚？

### 5.2 数据分级

没有数据分级，就很难设计权限。

一个简化的数据分级可以是：

| 数据级别 | 示例 | Agent 处理方式 |
|---|---|---|
| 公开数据 | 官网、公开文档、公开仓库 | 可默认读取，但要记录来源 |
| 个人数据 | 个人笔记、日程、邮件草稿 | 需用户授权，限制用途 |
| 团队内部数据 | 项目文档、内部代码、工单 | 需组织权限，遵守访问控制 |
| 敏感数据 | 密钥、财务、客户隐私、医疗信息 | 默认不注入模型，需脱敏和审批 |
| 高风险凭证 | API Key、支付凭证、生产密钥 | 不应暴露给模型，只能通过受控工具使用 |

数据分级的重点不是制作复杂制度，而是让 Agent 的上下文组装、工具调用和日志记录有明确边界。

### 5.3 权限模型

Agent 权限模型至少要回答：

- 谁在授权？
- 授权给哪个 Agent 或哪个任务？
- 授权访问哪些数据？
- 授权使用哪些工具？
- 授权执行哪些动作？
- 授权有效期多久？
- 如何撤销？

一个实用的权限设计可以分为三层：

| 权限层 | 说明 | 示例 |
|---|---|---|
| 数据权限 | Agent 能读哪些信息 | 只能读取某个项目目录 |
| 工具权限 | Agent 能调用哪些工具 | 允许读文件和运行测试，不允许提交代码 |
| 动作权限 | Agent 能否产生外部影响 | 可以生成邮件草稿，发送前必须确认 |

权限不要只放在提示词里。提示词可以告诉模型规则，但真正的权限控制必须由系统执行。

### 5.4 审批、审计与回滚

高风险动作要有审批。审批记录应该包含：

- 用户确认了什么动作。
- Agent 给出的理由是什么。
- 当时的输入、计划和上下文摘要是什么。
- 工具参数是什么。
- 执行结果是什么。
- 是否产生了外部副作用。
- 是否可以回滚。

审计日志不应该只服务合规，也服务调试和复盘。

```text
用户目标 -> Agent 计划 -> 工具调用 -> 用户确认 -> 执行结果 -> 失败/成功 -> 回滚记录
```

回滚设计要尽量靠近动作本身：

- 修改文件时保存 patch 和原始版本。
- 发送前先创建草稿。
- 发布前先发布到预览环境。
- 删除前先移动到可恢复区域。
- 更新配置前保存旧配置。

如果一个动作无法回滚，就必须提高确认和审批等级。

### 5.5 Prompt Injection 防御

Prompt Injection 在 Agent 产品中更危险，因为 Agent 不只是生成文本，还可能调用工具和访问数据。

常见风险：

- 外部网页或文档中包含恶意指令。
- 用户上传的文件试图覆盖系统规则。
- 检索结果诱导 Agent 泄露隐私或密钥。
- 工具返回内容伪装成系统指令。
- 多轮对话中用户逐步诱导越权。

防御思路包括：

- 区分系统指令、用户指令、工具结果和外部内容的信任等级。
- 外部内容只作为数据，不作为指令。
- 高风险工具调用必须经过权限层验证。
- 敏感数据默认不进入模型上下文。
- 对工具参数做规则校验，而不是完全相信模型生成。
- 对外部输出做安全检查。

Prompt Injection 防御不能只靠一段提示词。它需要上下文隔离、权限控制、工具参数校验和审计日志共同完成。

### 5.6 企业合规视角

企业环境会额外关注：

- 数据驻留：数据是否离开特定地区或云环境。
- 数据保留：日志和对话保存多久。
- 访问控制：是否继承企业账号和角色权限。
- 审计要求：谁访问了什么、做了什么、什么时候做的。
- 安全认证：是否满足 SOC2、ISO 27001 等要求。
- 隐私法规：是否符合 GDPR、CCPA 或本地隐私法规。
- 供应商风险：模型和工具供应商如何处理数据。

初学阶段不需要深入每一项法规，但要建立一个原则：

**Agent 产品越接近真实业务数据和外部动作，越需要把安全、权限、合规作为产品能力，而不是上线前的补丁。**

## 第六章：产品指标与反馈闭环

### 6.1 为什么不能只看调用量

Agent 产品如果只看调用量，很容易得出错误结论。

调用量高可能说明用户喜欢，也可能说明用户反复重试、任务失败、找不到正确操作方式。

更适合 Agent 的指标，要围绕"任务是否完成"和"用户是否愿意放权"设计。

典型问题：

- 用户交给 Agent 的任务完成了吗？
- 中间需要用户干预多少次？
- 用户是否接受了结果？
- 用户是否进行了修改？
- Agent 的输出是否进入真实业务流程？
- 任务成本和延迟是否在可接受范围内？
- 失败后用户是否继续使用？

### 6.2 指标分层

Agent 产品指标可以分为五层：

| 指标层 | 关注问题 | 示例指标 |
|---|---|---|
| 任务价值 | Agent 是否真的帮用户完成工作 | 任务完成率、结果采纳率、节省时间 |
| 用户信任 | 用户是否愿意让 Agent 执行更多步骤 | 用户干预率、确认通过率、接管率 |
| 运行质量 | Agent 执行过程是否稳定 | 工具调用成功率、失败恢复率、平均步骤数 |
| 成本效率 | 任务是否经济可持续 | 单次任务成本、P95 成本、token 使用量 |
| 风险治理 | 高风险动作是否可控 | 越权拦截次数、审批通过率、回滚次数 |

不同阶段的重点不同：

- 原型阶段：看任务完成率和失败原因。
- 内测阶段：看用户干预率、恢复率和成本。
- 上线阶段：看留存、ROI、风险指标和长期质量。

### 6.3 任务评测集

课程六讲过 Evaluation。本课关注产品化评测集如何构建。

Agent 评测集不应该只包含"标准问答"，而应该包含真实任务：

| 任务类型 | 示例 | 评测重点 |
|---|---|---|
| 正常任务 | 资料总结、代码 review、生成计划 | 是否完成目标 |
| 边界任务 | 信息不足、权限不足、工具不可用 | 是否追问、拒绝或降级 |
| 高风险任务 | 删除文件、发布内容、发送邮件 | 是否确认和审计 |
| 对抗任务 | 文档中包含恶意指令 | 是否抵御 Prompt Injection |
| 长任务 | 多步执行、跨工具协作 | 是否保持状态和恢复 |

每个评测任务最好包含：

- 用户输入。
- 预期行为。
- 不允许的行为。
- 可接受的输出范围。
- 必须调用或不得调用的工具。
- 风险等级。
- 验收标准。

### 6.4 线上反馈如何进入迭代

线上反馈要进入三个地方：

1. **产品需求**：用户在哪里不理解、哪里想接管、哪里觉得慢。
2. **评测集**：真实失败案例转成可回归测试。
3. **运行时策略**：调整工具、上下文、路由、权限、重试和降级。

一个可执行闭环如下：

```text
线上任务 -> Trace -> 失败分类 -> 复盘 -> 回归用例 -> 策略调整 -> 灰度上线 -> 指标观察
```

用户反馈不应该只是一段自由文本。建议结构化采集：

- 结果是否有用。
- 哪一步出错。
- 用户是否修改了输出。
- 用户是否接管了任务。
- 用户认为缺少什么信息。
- 用户是否愿意下次让 Agent 自动执行。

### 6.5 实验、灰度与版本比较

Agent 产品迭代经常涉及提示词、模型、工具、检索、路由和 UI 的变化。每次变化都可能影响结果。

建议建立版本比较机制：

- Prompt 版本。
- 模型版本。
- 工具 schema 版本。
- RAG 索引版本。
- Memory 策略版本。
- Orchestration 策略版本。
- UI 和确认策略版本。

灰度上线时要观察：

- 任务完成率是否提升。
- 用户干预率是否下降。
- 工具失败率是否变化。
- 成本和延迟是否上升。
- 风险拦截是否异常。
- 用户投诉是否集中在某类任务。

不要只看离线评测。Agent 的真实表现往往受用户输入、外部环境和工具稳定性影响，必须结合线上数据。

### 6.6 ROI 和业务价值

Agent 产品最终要回答：它是否值得上线和持续投入。

ROI 可以从三类价值计算：

| 价值类型 | 示例 |
|---|---|
| 节省时间 | 原来人工 30 分钟的任务，Agent 5 分钟完成并由人确认 |
| 提升质量 | 减少漏检、提高一致性、降低人工疲劳 |
| 扩大覆盖 | 原来做不了或做不完的长尾任务可以被处理 |

同时也要计算成本：

- 模型成本。
- 工具和外部 API 成本。
- 工程维护成本。
- 人工审核成本。
- 错误带来的补救成本。
- 安全和合规成本。

一个 Agent 是否有价值，不只看它能不能替代人工，也看它能否增强人工：让人从重复执行转向判断、确认和决策。

## 第七章：失败模式与产品复盘

### 7.1 失败分类

Agent 失败模式可以从运行链路定位：

| 链路位置 | 失败类型 | 例子 |
|---|---|---|
| Goal | 目标理解错误 | 用户要总结，Agent 开始写营销文案 |
| Context | 上下文遗漏或污染 | 忘记用户限制，引用错误记忆 |
| LLM Decision | 决策错误 | 选择了不合适的工具或执行顺序 |
| Tool / Environment | 工具失败 | API 超时、文件路径错误、权限不足 |
| Observation | 反馈理解错误 | 把测试失败误判为通过 |
| State Update | 状态更新错误 | 已完成步骤未记录，导致重复执行 |
| Loop Control | 循环控制错误 | 死循环、过早停止、无限重试 |
| Product Layer | 用户无法接管 | 没有展示计划、没有恢复入口 |

这种分类能把"Agent 不好用"拆成具体问题。

### 7.2 产品复盘模板

一次有效复盘至少包含：

```text
1. 用户目标是什么？
2. Agent 实际做了什么？
3. 哪一步开始偏离？
4. 当时的上下文、工具结果和状态是什么？
5. 用户是否有机会发现并纠正？
6. 系统是否有重试、降级或恢复路径？
7. 这次失败影响了哪些指标？
8. 是否需要进入回归评测集？
9. 下一步改进是产品机制、运行时策略，还是模型/提示词？
```

复盘要避免只得出"模型不够好"这种结论。很多失败可以通过产品和系统机制解决：

- 目标不清：增加追问。
- 高风险：增加确认。
- 工具错误：加强 schema 和参数校验。
- 过程不可见：展示计划和状态。
- 成本失控：增加预算上限。
- 失败不可恢复：增加 Checkpoint 和恢复入口。

### 7.3 三类项目的常见失败

**个人知识助手**

常见失败：

- 检索结果相关但不准确。
- 回答没有引用或引用不对应。
- 多轮对话中忘记当前研究主题。
- 把用户笔记中的错误内容当成事实。

产品化改进：

- 强制引用来源。
- 对低置信度回答标记不确定。
- 支持用户纠正资料。
- 把纠正样本进入评测集。

**代码审查 Agent**

常见失败：

- 输出泛泛建议，没有具体文件和行号。
- 把风格问题误报成严重 bug。
- 不运行测试就做确定判断。
- 自动修改代码后引入新问题。

产品化改进：

- 按严重程度和置信度排序。
- 要求每条发现给出证据。
- 修改代码前先生成 patch。
- 测试失败时保留日志并给出恢复建议。

**个人任务执行 Agent**

常见失败：

- 计划过大，不适合一次执行。
- 执行过程中偏离目标。
- 不知道什么时候追问用户。
- 任务中断后无法恢复。

产品化改进：

- 强制生成可编辑计划。
- 长任务分阶段确认。
- 保存任务状态和产物。
- 每个阶段输出可验收结果。

### 7.4 从复盘到迭代 backlog

复盘最终要进入 backlog，而不是停留在会议记录。

建议把改进项分为四类：

| 改进类型 | 示例 |
|---|---|
| 产品交互 | 增加计划确认、失败恢复入口、进度展示 |
| 运行时策略 | 增加重试上限、模型路由、上下文裁剪 |
| 工具机制 | 调整工具 schema、增加权限校验、增加 dry-run |
| 评测治理 | 增加回归用例、失败分类、风险指标 |

每个改进项都要绑定一个验证方式：

- 哪个指标会变化？
- 哪个评测用例会通过？
- 哪类失败会减少？
- 用户体验会具体改善什么？

### 练习二：设计 Human-in-the-loop 节点

为一个任务执行 Agent 设计至少 5 个节点：

- 哪些节点自动执行。
- 哪些节点需要用户确认。
- 确认时展示什么信息。
- 用户可以选择哪些操作。
- 超时或拒绝后如何处理。

### 练习三：做一次成本和延迟估算

选择一个多步任务，估算：

- 平均步骤数。
- 每步输入 token。
- 每步输出 token。
- 工具调用次数。
- 可能的重试次数。
- 平均成本和 P95 成本。
- 平均延迟和 P95 延迟。

### 练习四：设计失败恢复策略

选择 5 类失败，为每类写出：

- 失败如何被检测。
- 是否自动重试。
- 是否降级。
- 是否提示用户。
- 是否进入复盘。
- 如何避免下次重复。

### 练习五：建立产品指标面板

为你的 Agent 项目定义：

- 1 个北极星指标。
- 3 个任务质量指标。
- 3 个运行质量指标。
- 2 个成本指标。
- 2 个风险指标。
- 1 个反馈进入评测集的流程。

- 目标用户和核心任务。

- 可靠性与恢复策略。

## 验收标准

完成本课后，你应该能够：

- 解释为什么 Agent 产品不能只用 Demo 成功来判断。
- 为一个 Agent 任务设计进度、计划、确认、接管和恢复机制。
- 判断哪些动作可以自动执行，哪些动作必须确认。
- 写出重试、降级、幂等和任务恢复策略。
- 估算单次任务成本和主要延迟来源。
- 设计基本权限模型、审批流程和审计日志。
- 定义适合 Agent 的任务指标、成本指标和风险指标。
- 把一次 Agent 失败转化为复盘、评测用例和迭代项。

### 术语速查

| 术语 | 简要解释 |
|---|---|
| Human-in-the-loop | 在关键节点引入用户确认、审批、接管或纠正 |
| Progressive Trust | 随着成功经验逐步扩大自动执行范围 |
| 幂等 | 同一动作重复执行不会产生额外副作用 |
| 降级 | 主路径不可用时切换到低风险、低能力或低成本路径 |
| 审计日志 | 记录目标、计划、工具调用、确认、结果和副作用 |
| 任务完成率 | Agent 是否真正完成用户目标的核心指标 |
| 用户干预率 | 用户在任务中接管、修改、驳回或补充信息的比例 |
