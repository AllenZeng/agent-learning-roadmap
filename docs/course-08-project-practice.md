# 课程八：项目实战与生态跟进

## 课程导言

学完前七课，你已经有了理解和设计 Agent 产品的完整框架：

- 课程一建立产品直觉。
- 课程二理解 Agent 范式如何从 Chatbot、工具调用、Planning、Reflection、Multi-Agent 演进而来。
- 课程三掌握最小运行闭环。
- 课程四理解工具机制。
- 课程五知道如何按场景引入 RAG / 外部知识接入、Memory、Planning / Workflow Patterns、Reflection、Multi-Agent。
- 课程六理解 Harness 运行时架构。
- 课程七掌握产品化机制。

课程八要做两件事。

第一，用项目把这些知识串起来。只看概念很容易觉得自己理解了 Agent，但一旦开始做项目，就会遇到具体问题：工具 schema 怎么写？状态怎么保存？检索结果怎么引用？失败后怎么恢复？用户什么时候确认？成本怎么估算？

第二，建立跟进生态的习惯。Agent 领域变化很快，但学习不应该被新名词打散。你需要知道如何把新技术放回已有分层中判断：它到底是工具机制、场景增强、运行时工程，还是产品化能力？

本课不是让你一次做完所有项目，而是给出三个递进项目。你可以选择一个做深，也可以按顺序逐步推进。

---

## 学习目标

学完本课，你将能够：

1. **选择合适的 Agent 实战项目**：根据学习目标、难度和时间选择个人知识助手、代码审查 Agent 或个人任务执行 Agent
2. **定义具体使用场景**：把模糊的"做一个 Agent"转化为用户、任务、输入、输出、边界和验收标准
3. **设计项目能力组合**：判断哪些能力是必需的，哪些只是场景增强，避免为了技术完整而堆模块
4. **完成项目交付物**：产出可运行代码、设计文档、工具清单、权限设计、评测集、Trace 示例和失败复盘
5. **把项目推进到产品版思维**：从合格版、进阶版走向多用户、权限、评测、可观测和指标报告
6. **建立生态跟进方法**：用分层框架理解 MCP / A2A、Code Agent、Multi-Agent 框架、Agent RL、模型能力和运行时框架演进
7. **完成最终复盘**：回答为什么这个场景适合 Agent、如何评测、如何上线、如何持续改进

## 本课在路线图中的位置

课程八位于整条路线的最后一层：项目整合和生态跟进。

它不是新增一批理论概念，而是把前面课程压到真实任务里：

```text
产品直觉 -> 范式理解 -> 最小闭环 -> 工具机制 -> 场景增强 -> 运行时架构 -> 产品化 -> 项目实战
```

本课的重点不是"做三个漂亮 Demo"，而是让你真正经历一次 Agent 项目的完整链路：

- 定义场景。
- 明确边界。
- 选择能力。
- 设计工具。
- 设计状态和循环。
- 加入评测。
- 加入产品化机制。
- 复盘失败。
- 跟进生态变化。

## 第一章：项目实战总览

### 1.1 为什么项目不是 Demo

Demo 的目标是展示一次成功路径。项目的目标是验证一类任务是否可以稳定完成。

一个 Demo 可以这样做：

- 写一个固定提示词。
- 接一两个工具。
- 用一个精心准备的输入。
- 展示一次看起来不错的输出。

一个项目至少要回答：

- 换一个用户输入还能工作吗？
- 工具失败后怎么办？
- 输出是否有证据？
- 是否有权限边界？
- 是否能复现失败？
- 是否能评测质量？
- 是否能控制成本和延迟？
- 用户能否接管？

所以，本课的项目会按"合格版、进阶版、产品版"递进，而不是停留在一次演示。

### 1.2 三个推荐项目

| 项目 | 重点训练 | 合格版 | 进阶版 | 产品版 |
|---|---|---|---|---|
| 个人知识助手 | RAG / 外部知识接入、Memory、引用、上下文管理 | 文档检索 + 回答 | 混合检索 + 多轮记忆 | 多用户、权限、评测、引用可信 |
| 代码审查 Agent | Tool Use、文件读取、测试调用、置信度、行内反馈 | 读取 diff 后输出 review | 调用测试和静态分析 | GitHub App、行内评论、误报反馈 |
| 个人任务执行 Agent | Planning / Workflow Patterns、工具机制、状态管理、Human-in-the-loop、产品化 | 支持学习计划、文档整理、项目发布准备 3 类任务，能拆解计划并调用本地工具 | 支持信息追问、分支流程、任务恢复、失败复盘 | 有任务看板、trace、成本/耗时统计、确认/接管机制和指标报告 |

这三个项目覆盖三类典型 Agent：

- **知识型 Agent**：重点是知识接入、引用和可信回答。
- **工具型 Agent**：重点是文件、命令、测试、外部系统和结构化反馈。
- **任务执行型 Agent**：重点是计划、状态、恢复、确认和产品化。

### 1.3 项目选择建议

| 项目 | 建议顺序 | 难度 | 建议时间 | 适合目标 |
|---|---|---|---|---|
| 个人知识助手 | 第一个项目 | 入门-中级 | 3-5 天 | 训练 RAG / 外部知识接入、引用、Memory |
| 代码审查 Agent | 第二个项目 | 中级 | 4-7 天 | 训练工具调用、文件读取、测试调用、行内反馈 |
| 个人任务执行 Agent | 第三个项目 | 中级-高级 | 7-10 天 | 训练 Planning / Workflow Patterns、状态管理、Human-in-the-loop、产品化 |

如果你是产品经理，建议优先做个人任务执行 Agent 的需求设计和产品化报告。

如果你是工程师，建议优先做代码审查 Agent，因为它最容易暴露工具、状态、测试、权限和评测问题。

如果你是学习者或内容创作者，建议优先做个人知识助手，因为资料容易准备，效果容易观察。

### 1.4 通用项目架构

三个项目都可以用同一个基础架构理解：

```text
用户目标
  -> 任务边界判断
  -> 上下文组装
  -> LLM 决策
  -> 工具 / 环境交互
  -> Observation
  -> 状态更新
  -> 循环控制
  -> 输出 / 确认 / 恢复
```

不同项目的差异在于：

- 上下文来源不同。
- 工具类型不同。
- 状态记录不同。
- 风险等级不同。
- 输出验收方式不同。
- 产品化机制不同。

### 1.5 里程碑推进方式

建议按四个里程碑推进：

| 里程碑 | 目标 | 主要交付 |
|---|---|---|
| M1：最小链路 | 跑通一次核心任务 | 最小代码、工具清单、一次成功 Trace |
| M2：稳定执行 | 处理常见失败和边界输入 | 失败分类、恢复策略、基础评测集 |
| M3：产品化 | 增加确认、权限、成本、指标 | HITL 节点、权限设计、成本报告 |
| M4：复盘迭代 | 根据评测和反馈改进 | 失败复盘、回归用例、下一轮计划 |

不要一开始就追求完整产品。先让最小链路清楚，再逐步加复杂度。

## 第二章：项目一：个人知识助手

### 2.1 使用场景定义

个人知识助手的用户是有大量个人资料的人，例如学习笔记、PDF、网页摘录、项目文档、会议记录。

它要解决的问题不是"模型知道多少"，而是：

- 用户自己的资料分散在不同地方。
- 用户记得自己看过某个内容，但找不到位置。
- 用户想让回答基于指定资料，而不是模型自由发挥。
- 用户希望答案带引用，方便核查。
- 用户希望多轮对话中保持当前研究主题。

典型任务：

- "根据我这 5 篇文章，总结 Agent 工具调用的关键设计点。"
- "我之前整理过 RAG 的笔记，帮我找出和 Memory 相关的部分。"
- "回答这个问题时必须引用来源，不确定就说不知道。"

这个项目适合训练 RAG / 外部知识接入、引用、上下文管理和 Memory。

### 2.2 功能范围

合格范围：

- 导入 Markdown、PDF 或网页摘录。
- 对文档做切分和索引。
- 根据用户问题检索相关片段。
- 基于检索结果回答。
- 给出引用来源。
- 在无法找到依据时明确说明不确定。

不建议一开始就做：

- 全网自动搜索。
- 大规模多用户知识库。
- 复杂权限系统。
- 自动长期记忆。
- 多 Agent 协作写报告。

最小功能越清晰，越容易评测。

### 2.3 最小运行链路

个人知识助手的最小链路可以表示为：

```text
用户问题
  -> 查询改写
  -> 检索文档片段
  -> 片段重排序
  -> 上下文组装
  -> LLM 生成回答
  -> 引用检查
  -> 用户反馈
  -> 状态 / Memory 更新
```

这里的 Agent 成分主要体现在：

- 是否需要改写查询。
- 是否需要追加检索。
- 是否判断证据不足并停止回答。
- 是否根据用户反馈调整下一轮上下文。
- 是否把当前研究主题写入短期状态。

注意：并不是所有知识助手都是 Agent。如果系统只是"检索一次 -> 回答一次"，它更像 RAG 应用。只有当系统能围绕目标持续决策、补充检索、处理反馈和管理状态时，才更接近 Agent。

### 2.4 数据、工具与权限

数据来源可以从简单开始：

| 数据类型 | 处理方式 |
|---|---|
| Markdown 笔记 | 直接解析标题、段落、代码块 |
| PDF | 提取文本并保留页码 |
| 网页摘录 | 保存 URL、标题、正文和时间 |
| 项目文档 | 保留路径、章节和更新时间 |

工具可以设计为：

| 工具 | 作用 |
|---|---|
| search_documents | 根据查询检索文档片段 |
| read_document | 按文档 id 读取原文片段 |
| list_sources | 列出当前知识库来源 |
| save_note | 保存用户确认后的总结 |
| update_memory | 更新当前研究主题或偏好 |

权限设计：

- 默认只读取用户显式导入的资料。
- 不自动读取用户电脑其他目录。
- 保存新笔记前确认位置。
- 引用中不要暴露敏感路径或隐私信息。
- 多用户版本必须按用户隔离知识库。

### 2.5 评测方法

个人知识助手的评测重点是"依据资料回答"，而不是"回答看起来流畅"。

评测维度：

| 维度 | 问题 |
|---|---|
| 相关性 | 检索结果是否覆盖用户问题 |
| 忠实性 | 回答是否只基于资料，不编造 |
| 引用准确性 | 引用是否对应回答中的关键断言 |
| 不确定性 | 资料不足时是否承认不知道 |
| 多轮连续性 | 是否记住当前研究主题和用户限制 |

评测任务示例：

- 资料中明确有答案，检查是否能找到并引用。
- 资料中没有答案，检查是否拒绝编造。
- 多篇资料互相冲突，检查是否指出差异。
- 用户要求"只基于 A 文档回答"，检查是否越界使用其他资料。
- 用户纠正一次回答后，检查下一轮是否吸收纠正。

### 2.6 合格版、进阶版、产品版

| 版本 | 能力范围 | 验收标准 |
|---|---|---|
| 合格版 | 文档检索 + 回答 + 引用 | 能基于本地资料回答 20 个测试问题 |
| 进阶版 | 混合检索 + 多轮记忆 + 证据不足判断 | 能处理追问、冲突资料和无答案问题 |
| 产品版 | 多用户、权限、评测、引用可信 | 有权限隔离、评测面板、引用核查和反馈闭环 |

产品版不一定意味着商业化，而是意味着它可以被持续使用和改进。

### 2.7 常见失败模式

| 失败模式 | 典型原因 | 改进方向 |
|---|---|---|
| 检索不到关键资料 | 查询表达和资料表达不一致 | 查询改写、混合检索、标签 |
| 引用不对应 | 生成阶段没有约束引用 | 引用后校验、逐句引用 |
| 编造答案 | 资料不足但模型强答 | 证据阈值、不确定性提示 |
| 多轮跑偏 | 状态没有更新或历史太长 | 当前主题摘要、上下文裁剪 |
| 记忆污染 | 错误信息被写入 Memory | 写入确认、记忆来源、可删除 |

## 第三章：项目二：代码审查 Agent

### 3.1 使用场景定义

代码审查 Agent 的用户是个人开发者或小团队。它要在提交前读取代码变更，发现潜在 bug、遗漏测试、边界条件、类型问题和安全风险。

典型任务：

- "审查当前 Git diff，指出潜在 bug 和缺少的测试。"
- "运行相关测试，根据失败结果判断问题可能在哪里。"
- "把 review 结果按严重程度排序，并给出行内评论。"

这个项目适合训练工具调用、文件读取、命令执行、结构化输出、置信度和行内反馈。

### 3.2 功能范围

合格范围：

- 读取当前 Git diff。
- 识别变更文件和影响范围。
- 读取相关上下文文件。
- 输出按严重程度排序的 review。
- 每条发现包含文件、行号、原因和建议。
- 能区分 bug、风险、测试缺口和风格建议。

进阶范围：

- 运行相关测试。
- 运行静态分析或类型检查。
- 根据失败日志调整判断。
- 对低置信度发现做标记。
- 支持生成修复建议或 patch。

暂时不建议一开始就做：

- 全自动合并 PR。
- 自动修改生产代码并提交。
- 复杂团队权限系统。
- 多仓库跨服务影响分析。

### 3.3 工具与运行循环

代码审查 Agent 的工具可以包括：

| 工具 | 作用 |
|---|---|
| get_diff | 读取当前 Git diff |
| read_file | 读取相关文件 |
| search_code | 搜索符号、函数或调用点 |
| run_tests | 运行指定测试 |
| run_lint | 运行静态检查 |
| get_blame_or_history | 查看历史上下文 |
| create_comment | 生成行内评论 |

运行循环：

```text
用户目标
  -> 读取 diff
  -> 判断变更类型
  -> 读取相关上下文
  -> 决定是否运行测试
  -> 观察测试 / lint 结果
  -> 生成 review
  -> 标注置信度
  -> 用户反馈
```

关键不是让 Agent 一次读完整个仓库，而是让它学会按需读取。

### 3.4 Review 输出结构

Review 输出要结构化。否则用户很难判断优先级。

建议结构：

```text
发现：
- 严重程度：P1 / P2 / P3
- 类型：bug / 测试缺口 / 安全风险 / 可维护性 / 风格
- 文件和行号：
- 证据：
- 影响：
- 建议：
- 置信度：
```

严重程度可以这样定义：

| 等级 | 含义 |
|---|---|
| P1 | 可能导致线上故障、安全问题或数据损坏 |
| P2 | 明确 bug、边界问题或缺少关键测试 |
| P3 | 可维护性、可读性、局部优化建议 |

置信度要避免假装确定：

- 高：有代码证据或测试失败支持。
- 中：逻辑上有风险，但缺少完整证据。
- 低：只是可疑点，需要人工判断。

### 3.5 测试和静态分析接入

代码审查 Agent 如果只读 diff，很容易产生泛泛建议。接入测试和静态分析可以显著提升可信度。

但测试接入也有风险：

- 测试很慢。
- 环境不完整。
- 命令可能有副作用。
- 失败日志很长。
- Agent 可能误读失败原因。

建议做法：

- 优先运行与变更文件相关的测试。
- 设置命令超时。
- 捕获退出码、stdout、stderr。
- 对长日志做摘要。
- 区分"测试本身失败"和"环境失败"。
- 不自动运行破坏性命令。

测试结果要进入 review 证据，而不是单独贴在最后。

### 3.6 置信度与行内反馈

代码 review 的产品价值来自具体性。

不好的输出：

```text
这里可能有问题，建议增加错误处理。
```

更好的输出：

```text
P2：user-service.ts:87
当 getUser 返回 null 时，后续直接读取 user.email 会抛出异常。
建议在调用 sendEmail 前增加空值分支，并补充 getUser 返回 null 的测试。
置信度：高。依据：当前 diff 删除了原有 null check。
```

行内反馈要注意：

- 评论必须落在相关行附近。
- 不要为同一问题生成多条重复评论。
- 不要把偏好型建议提升为严重问题。
- 对自动修复建议，要说明是否已验证。

### 3.7 合格版、进阶版、产品版

| 版本 | 能力范围 | 验收标准 |
|---|---|---|
| 合格版 | 读取 diff 后输出 review | 能对 10 个真实或模拟 diff 给出结构化发现 |
| 进阶版 | 调用测试和静态分析 | 能把测试失败转化为 review 证据 |
| 产品版 | GitHub App、行内评论、误报反馈 | 能在 PR 中评论、收集反馈、形成回归集 |

产品版可以先做本地命令行版本，再接 GitHub 或 GitLab。

### 3.8 常见失败模式

| 失败模式 | 典型原因 | 改进方向 |
|---|---|---|
| 泛泛而谈 | 只读 diff，没有读上下文 | 按需读取调用点和测试 |
| 误报严重问题 | 没有置信度和证据 | 强制证据字段和严重程度规则 |
| 漏掉影响范围 | 没有搜索调用链 | 增加 search_code 和依赖分析 |
| 测试误读 | 日志太长或环境失败 | 日志摘要、失败类型分类 |
| 自动修复引入新问题 | 缺少验证 | patch 后必须运行相关测试 |

## 第四章：项目三：个人任务执行 Agent

### 4.1 使用场景定义

个人任务执行 Agent 面向个人工作流。它不是客服，也不是企业工单系统，而是帮助个人完成一类可拆解、可验证、可交付的任务。

典型任务：

- "帮我制定一周 Agent 学习计划，并生成每天的任务清单。"
- "把这个目录下的零散笔记整理成一份 Markdown 文章草稿。"
- "为这个小项目做发布准备：检查 README、整理 changelog、生成发布 checklist。"

这个项目适合训练 Planning / Workflow Patterns、工具机制、状态管理、Human-in-the-loop 和产品化。

### 4.2 三类典型任务

建议先支持三类任务：

| 任务类型 | 输入 | 输出 | 典型工具 |
|---|---|---|---|
| 学习计划 | 学习目标、时间、基础水平 | 每日计划、任务清单、复盘问题 | 文档生成、日历草稿 |
| 文档整理 | 笔记目录、主题、目标读者 | Markdown 草稿、结构提纲、引用列表 | 文件读取、搜索、写文件 |
| 发布准备 | 项目目录、版本目标 | README 检查、changelog、发布 checklist | Git、测试、文件读取 |

这三类任务都适合个人练习，因为不依赖企业工单数据，也不需要复杂业务系统。

### 4.3 功能范围

合格范围：

- 接收一个目标。
- 判断任务类型。
- 必要时追问缺失信息。
- 生成可编辑计划。
- 调用本地工具执行。
- 在关键节点请求确认。
- 生成最终交付物。

进阶范围：

- 支持任务暂停和恢复。
- 支持分支流程。
- 支持失败复盘。
- 支持成本和耗时统计。
- 支持用户接管某一步后继续。

不建议一开始做：

- 任意网页自动操作。
- 任意桌面控制。
- 自动登录第三方账号。
- 自动执行不可逆操作。

### 4.4 最小运行链路

个人任务执行 Agent 的运行链路：

```text
用户目标
  -> 任务类型识别
  -> 缺失信息追问
  -> 计划生成
  -> 用户确认计划
  -> 工具执行
  -> Observation
  -> 状态更新
  -> 循环控制
  -> 阶段性交付
  -> 最终确认
```

这里最关键的是状态管理：

- 当前目标是什么。
- 当前计划有哪些步骤。
- 哪些步骤已完成。
- 每一步产生了什么产物。
- 哪些工具调用成功或失败。
- 哪些节点需要用户确认。
- 中断后从哪里恢复。

### 4.5 工具与权限

工具可以从本地安全工具开始：

| 工具 | 作用 | 风险等级 |
|---|---|---|
| list_files | 列出目录文件 | 低 |
| read_file | 读取文件内容 | 低-中 |
| search_files | 搜索关键词 | 低 |
| write_draft | 写入草稿文件 | 中 |
| run_tests | 运行测试 | 中 |
| git_diff | 查看变更 | 低 |
| generate_checklist | 生成 checklist | 低 |

权限原则：

- 默认只读。
- 写入前确认路径。
- 覆盖文件前必须确认。
- 运行命令必须有白名单。
- 不允许自动删除文件。
- 不允许自动发布或推送。

工具越贴近真实环境，越要重视权限和审计。

### 4.6 任务看板、Trace 与状态

个人任务执行 Agent 最适合加入任务看板。

看板可以包含：

- 目标。
- 当前计划。
- 步骤状态：待执行、进行中、已完成、失败、等待确认。
- 每一步的工具调用。
- 中间产物链接。
- 用户确认记录。
- 成本和耗时。

Trace 则用于调试和复盘：

```text
Step 1: 读取目录 -> 成功 -> 发现 12 个 Markdown 文件
Step 2: 生成结构提纲 -> 成功 -> 等待用户确认
Step 3: 写入草稿 -> 失败 -> 文件路径不存在
Step 4: 创建目录并重试 -> 成功
```

没有看板和 Trace，长任务 Agent 很容易变成黑箱。

### 4.7 Human-in-the-loop 与恢复

个人任务执行 Agent 的确认节点建议包括：

| 节点 | 为什么确认 |
|---|---|
| 任务目标不明确 | 防止一开始跑偏 |
| 计划生成后 | 用户可以调整执行方向 |
| 写入文件前 | 防止覆盖用户内容 |
| 运行耗时命令前 | 避免等待和副作用 |
| 最终交付前 | 用户确认是否满足目标 |

恢复能力至少包括：

- 从上一次完成步骤继续。
- 重新执行失败步骤。
- 跳过某一步。
- 修改计划后继续。
- 用户手动完成一步后继续。

### 4.8 合格版、进阶版、产品版

| 版本 | 能力范围 | 验收标准 |
|---|---|---|
| 合格版 | 支持学习计划、文档整理、项目发布准备 3 类任务，能拆解计划并调用本地工具 | 每类任务至少完成 3 个测试案例 |
| 进阶版 | 支持信息追问、分支流程、任务恢复、失败复盘 | 中断或工具失败后能继续 |
| 产品版 | 有任务看板、Trace、成本/耗时统计、确认/接管机制和指标报告 | 可以连续使用一周并形成迭代报告 |

这个项目最接近完整 Agent 产品练习，也最容易暴露课程三到课程七之间的连接问题。

## 第五章：项目交付方法

### 5.1 从任务集开始

不要先写框架，先写任务集。

每个项目至少准备：

- 10 个正常任务。
- 5 个边界任务。
- 3 个失败任务。
- 2 个高风险任务。
- 2 个多轮任务。

任务集决定你真正要解决什么问题，也决定评测是否有意义。

### 5.2 设计文档模板

建议每个项目写一份简短设计文档：

```text
1. 项目名称
2. 目标用户
3. 使用场景
4. 核心任务
5. 不做什么
6. 最小运行链路
7. 必需能力
8. 可选增强能力
9. 工具清单
10. 状态管理
11. Human-in-the-loop 节点
12. 权限和安全
13. 评测方法
14. 成本和延迟估算
15. 失败复盘方式
```

这份文档不需要很长，但必须具体。

### 5.3 工具清单和权限设计

每个工具都要说明：

| 字段 | 说明 |
|---|---|
| 工具名称 | Agent 如何识别和选择 |
| 输入参数 | 参数类型、必填项、约束 |
| 输出结构 | 返回给模型的 Observation |
| 风险等级 | 只读、可逆写入、不可逆动作 |
| 权限要求 | 是否需要用户授权 |
| 失败模式 | 常见错误和恢复方式 |
| 是否可重试 | 重试是否会产生副作用 |

工具清单是连接课程四和课程七的关键交付物。

### 5.4 评测集设计

评测集要覆盖：

- 正常完成。
- 信息不足。
- 工具失败。
- 权限不足。
- 外部知识缺失。
- 用户中途修改目标。
- 高风险动作确认。
- 成本或步骤上限触发。

每个评测用例至少包括：

```text
用户输入：
初始环境：
允许工具：
不允许行为：
预期行为：
验收标准：
风险等级：
```

### 5.5 Trace 和失败复盘

每个项目至少保存 3 条 Trace：

- 一条成功 Trace。
- 一条工具失败后恢复的 Trace。
- 一条用户接管或确认的 Trace。

失败复盘要回答：

- 失败发生在哪个链路环节？
- 是否因为目标、上下文、工具、状态或循环控制？
- 用户是否有机会发现并纠正？
- 是否需要新增评测用例？
- 是否需要调整产品机制？

### 5.6 产品化报告

产品化报告建议包含：

- 目标用户和核心任务。
- 任务完成率。
- 用户干预率。
- 平均步骤数。
- 工具调用成功率。
- 平均成本和耗时。
- 高风险动作数量。
- 失败分类。
- 用户反馈摘要。
- 下一轮迭代计划。

这份报告会帮助你从"我做了一个 Agent"走向"我知道这个 Agent 解决了什么问题，还差在哪里"。

## 第六章：生态跟进方法

### 6.1 为什么最后再讲生态

Agent 领域新概念很多。如果一开始就跟生态，很容易被框架、协议和论文打散。

放到最后讲生态，是因为你已经有了分层框架：

- 产品直觉。
- 范式演进。
- 最小闭环。
- 工具机制。
- 场景增强。
- 运行时架构。
- 产品化。

有了这些层次，你看到新概念时就能判断它解决什么问题，而不是被名字带着走。

### 6.2 重点跟进方向

| 方向 | 关注问题 | 放回本路线的位置 |
|---|---|---|
| MCP / A2A | 工具、上下文、Agent 之间如何标准化交互 | 工具机制、运行时架构 |
| Code Agent | Agent 如何理解代码库、调用测试、修改文件 | 工具机制、产品化项目 |
| Multi-Agent 框架 | 多角色如何协作、委托、评审和汇总 | 场景增强、运行时架构 |
| Agent RL | 如何通过环境反馈优化策略 | 运行时、评测、产品反馈 |
| 新模型能力 | 工具调用、长上下文、结构化输出、多模态 | 最小闭环、工具机制 |
| Agent Runtime / Framework | 状态、编排、Trace、评测、部署如何工程化 | Harness 运行时架构 |

### 6.3 如何判断新概念

遇到一个新框架、新论文或新产品，可以问七个问题：

1. 它解决的是 Agent 链路中的哪个环节？
2. 它是核心闭环必需，还是场景增强？
3. 它降低了什么复杂度，又引入了什么复杂度？
4. 它需要哪些工程前提？
5. 它对用户体验有什么影响？
6. 它如何评测？
7. 它是否适合你当前项目？

不要因为某个概念很热就立刻加入项目。先判断它解决的问题是否真实存在。

### 6.4 每月跟进模板

建议每月做一次生态复盘：

```text
本月关注：
1. 新模型能力：
2. 新框架 / Runtime：
3. 新协议 / 标准：
4. 新产品案例：
5. 新论文或技术文章：

分层判断：
- 属于哪一层？
- 解决什么问题？
- 是否影响当前项目？
- 是否需要实验？

行动：
- 需要阅读：
- 需要试用：
- 需要加入评测：
- 暂不跟进：
```

生态跟进的目标不是追热点，而是持续校准自己的路线图。

## 第七章：最终验收与复盘

### 7.1 最终交付物

完成课程八后，你至少应该交付一个项目，并包含：

- 可运行代码。
- 起始模板或最小骨架。
- 设计文档。
- 工具清单。
- 权限设计。
- 评测集。
- Trace 示例。
- 失败复盘。
- 成本估算。
- 下一步产品计划。

如果你选择做产品版，还应包含：

- 用户任务集。
- Human-in-the-loop 节点。
- 指标报告。
- 风险和审计说明。
- 线上或模拟用户反馈。

### 7.2 答辩问题

项目完成后，你应该能回答：

- 为什么这个场景适合 Agent？
- 为什么不是传统 Workflow？
- 用了哪些工具？
- 哪些能力是必需的，哪些能力是可选增强？
- 如何评测？
- 如何上线？
- 如何持续改进？
- 最大失败风险是什么？
- 用户什么时候应该接管？
- 成本和延迟是否可接受？

如果这些问题答不清楚，说明项目还停留在 Demo 阶段。

### 7.3 评分标准

| 维度 | 合格 | 优秀 |
|---|---|---|
| 场景定义 | 有明确用户和任务 | 有边界、不做什么和验收标准 |
| 运行链路 | 能跑通主路径 | 有状态、恢复和 Trace |
| 工具机制 | 工具可调用 | 有 schema、权限、失败处理 |
| 能力选择 | 使用必要能力 | 能说明为什么不引入某些能力 |
| 产品化 | 有基本交互 | 有确认、接管、成本和指标 |
| 评测 | 有少量测试任务 | 有分层评测和回归用例 |
| 复盘 | 能描述失败 | 能分类失败并进入迭代 |

- 判断项目需要哪些核心能力和增强能力。
- 写出工具清单、权限设计和最小运行链路。
- 构建覆盖正常、边界、失败和高风险任务的评测集。
- 保存 Trace 并完成失败复盘。
- 给出项目的成本、延迟、产品指标和下一轮计划。
- 用路线图分层判断一个新的 Agent 技术是否值得引入。

## 课程结束语

到这里，整条 Agent 产品开发学习路线已经闭环。

你不需要把所有框架都学一遍，也不需要把所有论文都复现。更重要的是建立一套判断系统：

- 先看产品直觉。
- 再看范式演进。
- 回到最小闭环。
- 按问题引入工具和增强能力。
- 用运行时工程稳定系统。
- 用产品化机制承接用户、业务和风险。
- 最后通过项目和生态跟进持续迭代。

这套路线的目标不是让你记住一堆术语，而是让你能独立判断：一个 Agent 产品为什么成立，为什么失败，以及下一步应该改哪里。
