# Agent 学习路线图（V2）：以运行时循环为主轴

## 课程基本介绍

这份路线图基于个人系统学习 Agent 的过程整理，是一套从"第一次接触 Agent"到"能设计、实现、评估并上线 Agent 产品"的系统性学习路径，共七个课程。

**核心思路**：以 **Agent 运行时的实际控制流** 为主轴，以 ReAct 循环（Thought → Action → Observation → Memory Update）串起全部知识点，每个知识点都能在循环中找到自己起作用的位置。

**学什么**：Agent 是一套由 LLM、工具调用、知识检索、记忆系统、规划反思、运行时控制、上下文工程、编排调度、评测体系、安全护栏和可观测性共同组成的系统。七个课程从认知建立开始，逐层深入到运行时机制、工程架构和产品落地，最后以持续跟进生态收尾。

**怎么学**：每门课程都有明确的目标和实践里程碑。先建立直觉、再理解能力演进、再深入运行时循环、再进入工程架构、最后回归产品。

**设计特点**：

- **以运行时循环组织知识，而非概念堆砌**：每个阶段回答"这个知识点在 Agent 运行时的哪个位置起作用"
- **工具系统完整统一**：从工具定义、工具选择、工具执行，到 MCP 协议标准化，再到 Skill 可复用能力单元，一条主线讲完，不再分散在多个课程中
- **Prompt Engineering 回归手段**：不再作为独立模块，而是融入需要它的地方（模型层讲推理增强、工具层讲描述规范、产出层讲格式化）
- **RAG 回归正确位置**：定位为"知识接入"的一种具体实现模式，明确标注非 Agent 必需，根据应用场景可选
- **从技术走向产品**：课程五专门覆盖交互设计、可靠性、成本控制、安全合规、失败模式
- **理论和动手并重**：每门课程都有验收标准，课程六用三个递进项目整合全部知识
- **生态跟进独立成章**：Multi-Agent、Computer Use、RL 等快速变化的内容放在课程七，不干扰主线

## 课程一：初识 Agent

**定位**：建立感性认知，先"看见"Agent，建立直觉。

- **Agent 概念辨析**：理解 Agent vs Chatbot vs Workflow 的本质区别——Chatbot 只能"说"，Workflow 是固定流水线，Agent 是自己决定怎么做。核心定义公式：`Agent = LLM + Tool Use + 循环决策`
- **为什么现在谈 Agent**：GPT-3.5/ChatGPT 让大众体验到了 LLM 的能力，但光"说"不够，需要"做"。Agent 是 LLM 能力从"对话"到"行动"的自然延伸
- **体验主流产品**：使用 Claude Code、Cursor、Devin/OpenHands、GPT-Researcher 等产品，带着问题去用——"它什么时候自己做决定？什么时候问我？什么时候做错了？"
- **产品拆解模板**：用 9 维度观察框架（用户输入 / 规划时机 / 工具调用 / 用户确认 / 失败恢复 / 类型判断 / 惊喜与失望 / 改进建议）分析至少 3 款产品

**实践里程碑**：用产品拆解模板完成 3 份分析报告，每份覆盖全部 9 个维度。

## 课程二：Agent 能力的演进脉络

**定位**：理解 Agent 关键能力的来龙去脉。不再强行划分"四大/五大核心能力"（这类分类本身缺乏行业共识），改为按时间线讲清每一步解决了什么关键矛盾。

- **传统 Agent 的困境**：从强化学习的 Agent（Bellman 1957 → DQN 2013 → AlphaGo 2016），到软件 Agent（BDI 模型），再到 Russell & Norvig 四种 Agent 类型。根本痛点：封闭规则空间，无法泛化
- **LLM 带来的突破**：GPT-3 涌现能力（2020）让模型具备了少量示例即可适应新任务的能力，Agent 的决策核心从"人工编写的规则"变成了"模型的理解和推理"
- **ReAct（2022.10）：推理与行动的融合**：首次将推理（Thought）和行动（Action）融合为统一循环。关键贡献：模型在推理过程中可以主动调用工具获取信息，而非仅依赖训练数据
- **Toolformer（2023.2）：模型自监督学会用工具**：模型通过自监督学习自主决定何时调用哪种 API，无需人工标注。核心意义：工具使用能力可以"训练进去"，而非全靠 prompt 驱动
- **Function Calling 产品化（2023.6）**：OpenAI 将工具调用标准化为 API 能力，此后 JSON Schema 定义工具、tool_choice 控制策略成为主流范式
- **规划与反思：同一闭环的两半**：CoT（2022）→ Tree-of-Thoughts（2023）→ Plan-and-Solve（2023）→ Self-Refine / Reflexion（2023）。核心洞察：规划和反思不是两个独立能力——规划决定"走向哪里"，反思决定"走错没有"，它们在同一个循环中交替推动任务前进
- **Multi-Agent 的出现动机**：单 Agent 面临能力上限（上下文窗口、工具数量、角色混淆），多 Agent 通过分工协作突破瓶颈——AutoGen（2023）→ CrewAI（2024）→ 动态 Agent 团队

**实践里程碑**：能用"之前有什么问题 → 提出者怎么想的 → 带来了什么变化"的框架讲清楚每种能力；能判断一篇 Agent 论文或产品公告属于哪条演进主线。

## 课程三：Agent 运行时——以决策循环串起全部模块

**定位**：课程三是整个路线图的核心。不再并列堆砌模块，而是以 ReAct 循环为主轴，每个环节引出相关知识。学习者始终知道"这个知识点在 Agent 运行时的哪个位置起作用"。

**运行时循环总览**：

```
        ┌──────────────────────────────────┐
        │          Memory（记忆系统）         │
        │   短期记忆 / 长期记忆 / 工作记忆     │
        └──────┬──────────────────┬────────┘
               │ 写入             │ 读取
               ▼                  ▼
┌──────────┐   ┌──────────┐   ┌──────────────┐
│ Thought  │←──│  Memory  │──→│ Observation  │
│ （推理决策）│   │  Update  │   │ （结果处理）   │
└─────┬────┘   └──────────┘   └──────▲───────┘
      │ 决策输出                     │ 执行结果
      ▼                              │
┌──────────┐                  ┌──────────────┐
│  Action  │─────────────────→│  Execution   │
│ （工具调用）│                  │ （工具执行）   │
└──────────┘                  └──────────────┘
```

### 3.1 决策循环本身

先让学习者建立整体画面——这是 Agent 的"心跳"，后续所有知识点都挂在循环的相应环节。

- 完整链路：Thought（分析现状 → 决定下一步）→ Action（选择工具 → 填充参数 → 发起调用）→ Observation（接收结果 → 判断是否满足需求）→ Memory Update（写入关键信息）→ 回到 Thought
- 停止条件设计：任务完成判定、最大步数限制、用户确认断点、置信度阈值
- 循环检测与防护：重复调用同一工具、输出高度相似、步数超标但无进展
- 超时与中断：分层超时（单步超时 / 任务超时 / 会话超时）、Checkpoint 恢复

### 3.2 模型与推理（循环的「大脑」）

- Transformer 关键机制：Self-Attention / Multi-Head Attention 如何让模型理解上下文关系
- Tokenization 与工具调用：BPE 原理、token 边界对 JSON 参数生成的影响
- 采样策略与可控性：Temperature / Top-p / Top-k 如何影响模型在决策中的"创造性 vs 确定性"
- **Prompt Engineering 回归手段**：
  - System Prompt：定义 Agent 角色、行为边界、输出规范——是 Agent 的"行为章程"而非独立模块
  - Few-shot Prompting：通过示例引导工具选择和参数格式
  - Chain-of-Thought：增强复杂推理的中间步骤，本质是帮模型在 Thought 环节走得更远
  - 结构化输出约束：JSON Mode / Structured Outputs 确保 Action 环节参数可解析

### 3.3 工具系统（循环的「双手」）⭐ 完整统一的主线

旧版将工具相关知识分散在课程二（历史）、课程三（定义）、课程四（编排）、课程七（MCP），新版在此统一为一条完整链路。

**3.3.1 工具定义**

- JSON Schema 定义工具：function name / description / parameters / required
- 工具描述的黄金法则：description 是模型选择工具的唯一依据，必须说清"这个工具做什么、什么时候用、什么时候不要用"
- 参数设计：参数名即文档、类型约束、枚举值声明、默认值设计
- 工具粒度：原子性 vs 组合性——一个工具做一件事还是做一组事？

**3.3.2 工具选择**

- 核心三问：什么时候该用工具（时机判断）、用哪个工具（工具路由）、参数怎么填（参数推断）
- 路由策略：模型自主选择 vs 规则路由 vs 混合路由（小模型分类 + 大模型填参）
- 并行调用：哪些工具可以同时调（无依赖关系），哪些必须串行
- 工具选择失败：选了不存在的工具、该用时不用、不该用时乱用

**3.3.3 工具执行**

- 执行环境：沙箱隔离、权限最小化
- 执行控制：超时设置、重试策略、结果截断
- 错误处理：工具不可用、参数格式错误、执行超时、结果异常——每种情况如何反馈给模型
- 结果格式化：将执行结果转化为模型可理解的 Observation，控制长度、突出关键信息

**3.3.4 工具协议标准化：MCP**

- 为什么要标准化：每个工具都是定制接入 → 工具方和 Agent 方耦合太紧 → 生态无法繁荣
- MCP 的核心设计：Client-Server 架构、Tool/Resource/Prompt 三种原语、JSON-RPC 传输
- MCP 不是"前沿趋势"，而是工具系统成熟化的必然方向——从课程三起就应建立这个意识
- 对比：Function Calling（单次调用）vs MCP（持续连接 + 资源发现 + 双向交互）

**3.3.5 可复用能力单元：Skill / Plugin**

- Skill 是什么：将常用工具组合 + 提示词模板封装为可复用的"能力包"
- 和单个 Tool 的区别：Tool 是原子操作，Skill 是包含判断逻辑和默认行为的工作流
- 设计原则：独立性强、边界清晰、可组合、有 fallback
- 典型例子：代码审查 Skill（读 diff + 静态分析 + 生成评论）、数据查询 Skill（写 SQL + 执行 + 可视化）

**3.3.6 工具生态**

- 内置工具：文件读写、代码执行、网络请求——几乎所有 Agent 都需要的基础能力
- 自定义工具：根据业务场景定义的工具
- 社区 / MCP 生态工具：通过标准协议接入的第三方工具

### 3.4 记忆系统（循环的「记忆」）

- **短期记忆（会话内）**：滑动窗口（简单但浪费上下文）、摘要压缩（省 token 但会丢细节）、重要性过滤（保留关键信息、遗忘噪声）
- **长期记忆（跨会话）**：用户偏好持久化、历史任务经验、知识积累——本质是"下一次启动时 Agent 还记得上次学了什么"
- **工作记忆（任务内）**：当前任务的中间结果、已尝试但失败的方案、待执行的子任务
- **三层上下文架构（热/温/冷）**：热数据（当前任务上下文，直接注入 prompt）、温数据（近期会话摘要，按需检索）、冷数据（历史归档，显式查询才加载）
- **记忆写入时机**：每步都写 vs 关键节点写 vs 任务结束写——写入频率影响检索质量

### 3.5 知识接入（可选增强）

> **定位说明**：知识接入不是 Agent 的必需组件。一个纯操作型 Agent（如自动化部署、定时任务调度）完全不需要知识接入。是否引入取决于应用场景是否需要"外部知识的支持"。旧版将 RAG 列为五大核心模块之一，这里纠正为可选增强。

- **RAG 基础**：Embedding 与向量检索、Chunking 策略（固定大小 / 语义切分 / 递归切分）、检索后重排序（Rerank）
- **混合检索**：关键词检索（BM25）+ 语义检索（向量相似度），互补各自的盲区
- **RAG 与工具系统的关系**：RAG 本质上是"通过检索工具为模型注入外部知识"，检索器就是一个特殊的 Tool，在 Thought 环节被调用，在 Observation 环节返回结果
- **其他知识接入方式**：Web Search（实时信息）、Browser（页面内容提取）、Knowledge Graph（结构化知识查询）
- **何时需要知识接入**：需要引用特定文档、需要事实核查、知识超出模型训练数据范围

**实践里程碑**：实现一个 ReAct Agent（Python/TypeScript），支持 3+ 个工具（至少包含一个知识检索类工具），具备跨会话记忆，内置循环检测与超时控制。10 条测试任务成功 >= 7 条，能按 tool selection / prompt / schema / 上下文管理 / memory / 知识检索 分类失败原因。

## 课程四：工程化架构

**定位**：让 Agent 从能用到好用。有了课程三的运行时基础，这些工程化话题会更容易理解——因为学习者已经知道每个环节在运行时是怎么运转的。

| 主题 | 关键概念 | 推荐框架/工具 |
|------|---------|------------|
| **Harness（运行时引擎）** | 三层架构：驱动层（模型调用、工具执行）/ 控制层（循环控制、停止判定、状态管理）/ 管理层（会话管理、资源配置）；Harness vs Orchestration 的区别——前者管"一个 Agent 怎么跑"，后者管"多个步骤怎么串" | 主流框架的 Harness 设计对比 |
| **Context Engineering** | 上下文窗口的工程化管理：信息密度优先、位置即优先级、分层管理、性价比思维；课程三中各环节产生的 Prompt / Tool 结果 / Memory 如何在同一个上下文窗口里高效编排 | — |
| **Orchestration** | 四种编排模式：Chain（固定顺序）/ Router（条件分发）/ Plan-Execute（先规划再执行）/ Graph（有向图编排）；与课程三决策循环的关系：循环是微观的步进逻辑，编排是宏观的任务分解 | LangGraph、CrewAI、AutoGen |
| **Memory 架构深入** | 向量数据库选型、MemGPT 虚拟内存思想、记忆衰减与更新策略、多 Agent 共享记忆 | Mem0、Letta |
| **Evaluation** | 端到端评测 vs 步骤级评测、LLM-as-Judge、评测集构建方法论 | Braintrust、LangSmith、Ragas |
| **Guardrails** | 输入护栏（Prompt Injection 防御）、输出护栏（工具参数校验、事实性校验）、Human-in-the-Loop 五级审批策略 | Guardrails-AI、自定义中间件 |
| **Observability** | Tracing 架构、关键指标（延迟分解、Token 消耗、错误分类）、调试与决策归因 | Langfuse、LangSmith、Phoenix |

**实践里程碑**：将课程三的决策循环重构为三层 Harness 架构；构建多步骤 Agent（规划→执行→反思→修正），配有记忆和可观测性；Guardrails 至少拦截 1 次真实的越界行为。

## 课程五：Agent 走向生产环境

**定位**：让 Agent 从好用到能上线。解决"用户觉得它慢、贵、不安全、不靠谱"的问题。

- **交互设计**：流式输出设计（SSE 原理）、中断与恢复（Checkpoint 机制）、渐进式信息披露、用户确认节点设计
- **可靠性工程**：重试策略（指数退避 + 智能重试）、降级机制（工具降级 / 模型降级 / 功能降级）、超时处理（分层超时）、Rate Limit 管理
- **成本控制**：模型选择策略（路由器模式 / 级联策略——小模型做分类、大模型做推理）、Prompt Caching 原理与实践、上下文窗口经济性
- **产品评价体系**：六大核心指标——任务完成率、平均步数、用户干预率、Token 效率、用户满意度、留存率。Agent A/B 测试的特殊挑战
- **安全合规**：Prompt Injection 防御体系、数据脱敏与审计日志、GDPR / SOC2 合规要点
- **失败模式清单**：工具调用类（选择错误 / 参数错误 / 死循环）、上下文管理类（遗漏约束 / 知识检索误导）、决策逻辑类（过早执行 / 权限不明）

**实践里程碑**：流式输出延迟感知 <= 500ms；常见故障可自动恢复；单任务平均 Token 消耗降低 >= 20%；失败模式清单覆盖 >= 5 种类型。

## 课程六：Agent 项目实战

**定位**：整合所有知识。用一个完整的端到端项目，将前五门课程的知识串联输出。

三个递进项目：

| 项目 | 难度 | 合格版 | 进阶版 | 产品版 |
|------|------|--------|--------|--------|
| **个人知识助手** | 入门 | 检索+回答，2 工具，正确率>=60% | 混合检索+Rerank，多轮记忆，流式输出 | Web 界面，多用户，正确率>=90% |
| **代码审查 Agent** | 中级 | 读取 diff → 输出 review | 调用测试/静态分析，置信度标注 | GitHub App 自动触发，行内评论，误报反馈 |
| **领域 Agent 产品** | 高级 | 核心功能可用，5 场景 | 20+ 场景，有评测报告 | 有真实用户反馈，监控告警，持续迭代 |

每个项目都附带设计文档模板、复盘模板和评测框架。

## 课程七：Agent 前沿趋势

**定位**：保持敏锐。了解正在发生什么，但主线精力不在此。定期浏览即可。

> ⚠️ **重要定位**：本课程是"跟进生态"，不是"主线必修"。建议在完成课程三到课程五之前，不要在这里投入过多时间。

| 方向 | 关注内容 | 与主线关联 |
|------|---------|----------|
| **Multi-Agent** | 四种架构模式深入（层级式 / 对话式 / 辩论式 / 发布-订阅式）、框架对比（AutoGen vs CrewAI vs LangGraph） | ⭐⭐⭐⭐ 单 Agent 准确率遇瓶颈时深入 |
| **Computer Use** | GUI Agent（截屏分析 + 元素定位 + 操作执行）、浏览器自动化（Playwright + LLM） | ⭐⭐⭐ 了解 + Demo |
| **Code Agent** | 里程碑：Copilot(2021) → SWE-bench(2023) → Devin(2024) → Claude Code(2025) | ⭐⭐⭐⭐⭐ 日常使用就是学习 |
| **RL for Agent** | RLHF → GRPO → DeepSeek-R1；Agent RL 的特殊挑战（奖励稀疏、探索空间大、环境非平稳） | ⭐⭐ 了解概念 |

## 新旧结构对照

| 维度 | V1（旧版） | V2（新版） |
|------|-----------|-----------|
| **组织原则** | 模块堆砌——五大核心模块并列（LLM / Tool Use / RAG / Memory / Prompt Engineering） | 运行时循环——以 ReAct（Thought → Action → Observation → Memory Update）为主轴串起全部知识点 |
| **RAG 定位** | 五大核心模块之一，与 Tool Use、Memory 并列必修 | 知识接入的一种具体实现，明确标注可选增强，Agent 不一定需要 |
| **Tool Use 组织** | 分散在课程二（历史）、课程三（定义）、课程四（编排）、课程七（MCP），知识碎片化 | 课程三统一主线：工具定义 → 工具选择 → 工具执行 → MCP 协议 → Skill → 工具生态 |
| **Prompt Engineering** | 独立模块，与 LLM、Tool Use 等其他模块并列 | 回归手段——融入模型层（推理增强）、工具层（描述规范）、产出层（结构化输出） |
| **MCP / Skill** | 课程七"发展趋势"，被视为前沿选修 | 课程三"工具系统"，作为工具标准化的自然延伸，核心知识 |
| **Memory** | 独立模块，课程三讲基础、课程四讲深入 | 挂在循环的记忆环节，和运行时紧密结合 |
| **课程二** | 划分四大核心能力（Tool Use / Planning / Reflection / Multi-Agent） | 按时间线讲演进脉络，不强行分类——规划和反思是同一闭环的两半 |

## 课程依赖关系

```
课程一（初识 Agent）
    │
    ▼
课程二（能力演进脉络）
    │
    ▼
课程三（运行时：决策循环串起全部模块）  ← 整个路线图的核心
    │
    ▼
课程四（工程化架构）          课程七（前沿趋势）← 可并行
    │                            ▲
    ▼                            │
课程五（走向生产环境）─────────────┘
    │
    ▼
课程六（项目实战）
```

- 课程一 → 课程二 → 课程三：顺序依赖，不可跳
- 课程三 → 课程四 → 课程五：顺序依赖，不可跳
- 课程七：可在课程三之后随时开始，与课程四/五/六并行
- 课程六：依赖课程一至课程五的学习，但可将项目拆分为阶段性交付
