# 课程一：初识 Agent —— 先看见，再理解

---

## 课程导言

如果你正在读这段话，你很可能已经用过 ChatGPT 或类似的产品。你问它问题，它回答你——有时候惊艳，有时候胡说八道。你对它的感觉可能是："这东西挺聪明的，但它能真的帮我做点什么吗？"

本门课程要回答的就是这个问题。

本课程的目标很明确：**看见 Agent，建立直觉**。

什么是"看见"？打开 Claude Code，让它帮你重构一个文件；打开 Cursor，看它如何理解你的代码库；打开 GPT-Researcher，让它做一份行业调研。你不需要理解它内部是怎么工作的——就像你不需要理解发动机原理就能感受到一辆车的加速性能。

什么是"直觉"？用完之后，你能自然地判断：这个产品是真正在做决策，还是只是在走固定的流程；它的哪一步让你觉得"聪明"，哪一步让你觉得"笨"；如果让你来设计，你会改什么。

这些直觉，是后面所有学习的燃料。没有直觉支撑的原理，是空洞的；带着直觉去学原理，每一句话你都能想起一个具体的画面。

准备好了吗？让我们开始第一次接触。

完成本课程后，你将能够：

## 第一章：什么是 Agent？

### 1.1 先看一个场景

你让 ChatGPT 帮你订一张从北京飞上海的机票。

ChatGPT 会说："抱歉，我无法帮你订机票。你可以去携程或航空公司官网查询。"

你让 Agent 做同样的事。

Agent 会自己打开浏览器，搜索航班信息，比较不同航空公司的价格和时间，找到合适的航班后填写订单页面——你在旁边看着就行。到了支付那一步，它会停下来问你："找到一趟下午 3 点、票价 850 元的航班，要我帮你下单吗？"

这两种反应的差异，就是 Agent 和 Chatbot 的本质区别。

### 1.2 一句话定义

**Agent 是能自主使用工具完成任务的 AI 系统。**

这句话里每个词都很重要：

- **自主**：不需要人类逐步指导。你告诉它"做什么"，它自己决定"怎么做"。
- **使用工具**：不只是在语言空间里推理，而是能操作外部世界——搜索、读写文件、调用 API、执行代码。
- **完成任务**：有清晰的目标，能判断自己是否完成了、是否需要调整策略。

### 1.3 Agent 和 Chatbot 的区别

Chatbot 只能"说"，Agent 能"做"。

这不是修辞上的区别，而是能力边界上的根本不同。来看几个具体场景：

| 场景 | Chatbot 能做什么 | Agent 能做什么 |
|------|-----------------|---------------|
| 帮你做旅行攻略 | 生成一份文字攻略 | 搜索航班、比价、查酒店空房、生成可预订的方案 |
| 帮你分析数据 | 告诉你分析方法 | 读取文件、写 Python 代码、运行、根据结果调整分析方向 |
| 帮你写代码 | 给你一段代码 | 理解你的项目结构、写代码、运行测试、根据报错修改 |
| 帮你做竞品调研 | 列出一些竞品名字 | 搜索官网、阅读新闻、提取关键信息、输出对比表格 |

你可能会问：Chatbot 后面接一个搜索引擎，不也能查东西吗？

对，但"谁来决定查什么、什么时候查、查到之后下一步做什么"——这才是分水岭。在 Agent 中，这些决策是 LLM 自己做出的；在 Chatbot + 搜索中，这些决策是由预设的流程控制的。

### 1.4 最小定义公式

如果用最少的词定义 Agent，就是：

> **Agent = LLM + Tool Use + 循环决策**

拆解一下：

- **LLM**：大脑，负责理解任务、推理、判断。
- **Tool Use**：手，负责操作外部世界。
- **循环决策**：心，在"思考→行动→观察结果→再思考"之间不断循环，直到任务完成。

这三者缺一不可。有 LLM 但不会用工具，是 Chatbot；有工具但 LLM 不参与决策，是传统软件；有决策但只执行一步就停止，是单次问答。

循环决策的循环长这样：

```
思考：我需要知道最近一周某公司股价的变化
  → 行动：调用股价查询工具
  → 观察：获得了数据，但发现波动很大
  → 思考：波动可能和新闻有关，我需要查相关新闻
  → 行动：调用新闻搜索工具
  → 观察：找到了三条相关新闻
  → 思考：现在我有股价数据和新闻背景，可以综合分析了
  → 输出：生成分析报告
```

每一轮"观察"都可能改变 Agent 的下一步决策。这就是循环的意义——它不是按照预设剧本走，而是根据实际情况动态调整。

### 1.5 Agent 和 Workflow 的区别

理解了 Agent 是什么之后，你需要知道另一个经常被混淆的概念：**Workflow（工作流）**。

想象两家餐厅。

**Workflow 餐厅**：后厨是一条装配流水线。第一个工位洗菜，第二个工位切菜，第三个工位炒菜，第四个工位装盘。每个工位做什么是规定好的，不需要思考。

**Agent 餐厅**：你告诉主厨"我今天想吃清淡的"，主厨自己决定：先看看冰箱里有什么食材（查工具），判断哪些搭配适合清淡口味（推理），然后决定烹饪顺序和方法（决策），中间尝一尝味道再调整火候（反馈循环）。做成什么样，取决于主厨的判断。

在技术术语中：

- **Workflow**：预定义的固定执行流程。每一步做什么、什么时候调用 LLM、如何传递数据——在设计阶段就确定了。
- **Agent**：LLM 自主决定执行策略。给定目标后，Agent 自己决定：现在该做什么？用哪个工具？结果不理想怎么办？

典型的 Workflow 例子：一个智能客服系统——识别用户意图（5 种预定义类型）→ 根据意图调用对应的知识库 → LLM 润色生成回答。虽然其中有 LLM，但它没有决策权，只是负责"润色"。

典型的 Agent 例子：Claude Code。给定"帮我找出这个项目的所有 bug 并修复"，它自己决定：先读哪些文件、运行什么诊断命令、如何判断一个现象是不是 bug、修复后如何验证。

**为什么这个区分很重要？**

因为 Agent 和 Workflow 在成本和可控性上处于光谱两端：

Workflow 的优势：
- 可预测：输入相同，输出一定相同
- 高效：不会做"多余"动作
- 低成本：不会因"反复思考"浪费 Token
- 易调试：出问题能精确定位到哪一步

Agent 的优势：
- 灵活：能处理设计者未曾预料的情况
- 通用：同一个 Agent 可以处理各种不同类型的任务
- 自主：不需要人类逐步指导

**Trade-off 的本质**：Agent 的灵活性来自 LLM 的自主决策，而自主决策意味着不确定性——你无法完全预测 Agent 会做什么、会不会陷入循环。这也是为什么在实际工程中，更常见的做法不是纯 Agent，而是 **Agentic Workflow**——在整体流程固定的前提下，在关键决策节点让 Agent 自主发挥。用 Workflow 保证下限（不出大错），用 Agent 提升上限（做得更好）。

## 第二章：为什么现在谈 Agent？

### 2.1 那一刻，所有人都在和 AI 聊天

2022 年 11 月，ChatGPT 上线。两个月内，月活用户突破 1 亿。

这不是 AI 领域第一次有"爆款"，但它第一次让**普通人**真切地感受到：机器能听懂我说的话了。

你问它什么，它能回答。虽然有时出错，但那种"它在理解我"的感觉是前所未有的。人们用它写邮件、翻译、头脑风暴、学外语、写诗——这是"对话式 AI"的全民启蒙。

### 2.2 但人们很快发现了一件事

ChatGPT 很聪明。但它的聪明被困在对话框里。

它会告诉你应该怎么分析数据，但它不能帮你分析数据。它会建议你去哪里旅行，但它不能帮你订机票。它能描述一个 bug 的可能原因，但它不能帮你修。

这就产生了一种微妙的挫败感：**你已经看到了它的能力，却用不到实处。**

其实，这个问题在 2023 年上半年就已经在业界被意识到了。2023 年 3 月，OpenAI 发布了 ChatGPT Plugins——让 ChatGPT 能够调用外部工具。这是大众用户第一次看到 AI 不仅可以"说"，还可以"做"：订餐、查机票、搜索论文、生成图表——都在同一个聊天窗口里完成。

Plugins 在工程上不完美（依赖手动安装、Plugin 之间难以协作），但它在心理层面至关重要：它打破了"LLM 只是聊天工具"的认知边界。

### 2.3 不是替代，是延伸

理解 Agent 的一个关键认知是：**Agent 不是要替代 LLM，而是 LLM 能力的自然延伸。**

这个延伸的方向是从"对话"到"行动"。就像一个聪明的人，光能说话是不够的——你得给他手，让他能做事；给他眼，让他能看到做事的结果；给他判断力，让他能根据结果调整下一步。

Agent 就是在做这件事：让 LLM 长出"手"和"眼"。

你可能会问：那这和传统软件自动化有什么区别？传统软件的自动化是"我知道所有可能的情况，我把规则都写好"。Agent 的自动化是"我只有目标，过程由 LLM 临场判断"。前者像地铁——轨道是铺好的，列车只能沿着轨道走；后者像出租车——目的地是确定的，但走哪条路可以灵活选择。

这就是为什么现在谈 Agent：因为 LLM 的推理能力已经到了一个临界点，让它能够承担"临场判断"这个角色。这在三年前是不可想象的。

## 第三章：体验 3-5 款 Agent 产品

理论说到这儿就够了。接下来是本章最核心的学习活动：**亲自用。**

你不需要安装所有产品。从中选 3 款，每款至少用 30 分钟，带着目的去体验。

### 3.1 推荐体验清单

**1. Claude Code —— 终端里的编程搭档**

Claude Code 是 Anthropic 推出的命令行 Agent，运行在你的终端里。你告诉它想做什么——"帮我把这个模块重构一下"、"给这个函数写单元测试"、"帮我定位一个 bug"——它会自己读代码、分析结构、做出修改、运行测试验证。

**观察要点**：
- 它什么时候会"思考"（Thinking），思考了什么？
- 它在执行危险操作（如删除文件、git commit）前会问你吗？
- 它一次能做对的事情和反复修改才能做对的事情分别是什么？

**2. Cursor —— IDE 里的 AI 编程搭档**

Cursor 是基于 VS Code 的 AI 编辑器。它的 Agent 模式不只是补全代码，而是能理解你的整个项目结构，跨文件做修改。

**观察要点**：
- Apply 模式：它修改代码后，你需要确认哪些改动——为什么是这个粒度？
- 它的上下文感知能力：它是否知道你的项目里有哪些文件、它们之间的关系？
- 当你连续提出多个需求时，它能否保持上下文？

**3. GPT-Researcher —— 自主研究助手**

GPT-Researcher 是一个开源的研究 Agent。你给它一个研究主题，它会自己规划搜索关键词、爬取网页、分析内容、综合成一份研究报告。

**观察要点**：
- 从你输入主题到它输出报告之间，它做了哪些步骤？
- 它的搜索关键词选得好不好？有没有搜了很多但都是不相关的内容？
- 最终报告的质量和覆盖面如何？有没有重要的信息来源被遗漏？

**4. Devin / OpenHands —— 自主软件工程师**

Devin（Cognition 出品）和 OpenHands（开源）都是"自主软件工程师"类型的 Agent。你给它一个开发任务（如"写一个带用户认证的 Todo 应用"），它从零开始：规划架构、写代码、调试、部署。

**观察要点**：
- 它是如何分解一个复杂开发任务的？
- 它会不会在某个步骤卡住？卡住后是怎么办的？
- 它的哪些决策让你觉得"合理"，哪些让你觉得"它在兜圈子"？

### 3.2 体验方法论：带着问题去用

很多人体验新产品是"随便点点，凭感觉评价"。我们不这样做。

在体验每一款 Agent 时，请始终带着这四个问题：

1. **它什么时候自己做决定？** 哪些操作它直接执行了，没有问你的意见？你认为这些"自作主张"合理吗？

2. **它什么时候问我？** 在什么类型的操作前，它会停下来征求你的确认？这个停止点的设置合理吗？会不会过于频繁（烦人）或过于宽松（危险）？

3. **它什么时候做错了？** 记录一个具体的错误。它是怎么出错的？它自己发现了吗？如果没发现，你是怎么发现的？

4. **它什么时候让我惊喜？** 记录一个让你觉得"哇，它竟然能做到这个"的瞬间。为什么这个瞬间让你印象深刻？

这四个问题对应的是 Agent 设计的四个核心维度：自主性、可控性、容错性、能力边界。你现在不需要记住这些术语——带着问题去用就行。在第四章中，我们会把这些体验系统化。

## 第四章：产品拆解与分析

体验完产品之后，你需要一个框架来系统化你的感受。从"感觉好用/不好用"到"我知道它为什么好用/不好用"——这就是本章的目的。

### 4.1 产品拆解模板

下面是一个固定的 9 维度观察框架。用这个模板去回顾你体验的每款产品，填写每个维度的观察结果。

---

**产品拆解模板**

| 维度 | 观察要点 | 你的记录 |
|------|---------|---------|
| **产品信息** | 产品名、版本/日期、使用场景 | |
| **用户输入** | 用户以什么方式输入？自然语言？表单？代码？是否支持多轮对话？ | |
| **规划时机** | Agent 什么时候开始规划？是在第一次收到用户输入时？还是每轮都重新规划？规划结果是否可见？ | |
| **工具调用** | Agent 使用了哪些工具？调用的触发条件是什么？工具调用的过程用户能看到吗？ | |
| **用户确认** | 哪些操作需要用户确认？哪些自动执行？确认节点是显式的（弹窗/按钮）还是隐式的？ | |
| **失败恢复** | Agent 出错后如何表现？是自动重试？是切换策略？还是直接告知用户？用户有没有手动纠错的入口？ | |
| **类型判断** | 这个产品的哪些行为是 Workflow（固定流程）？哪些是真 Agent（自主决策）？整体上它更偏向哪一端？ | |
| **惊喜与失望** | 哪一次行为让你觉得"它真聪明"？哪一次让你觉得"它怎么会犯这种错"？ | |
| **改进建议** | 如果你是这个产品的 PM，你最想改什么？ | |

---

**如何填写？**

以 Claude Code 为例的前 3 个维度示范：

| 维度 | 观察要点 | 你的记录 |
|------|---------|---------|
| **产品信息** | 产品名、版本/日期、使用场景 | Claude Code，终端 CLI 工具，用于代码开发和维护 |
| **用户输入** | 用户以什么方式输入？ | 用户在终端中以自然语言输入需求，支持多轮对话，Claude 可以请求用户确认 |
| **规划时机** | Agent 什么时候开始规划？ | 收到用户输入后，Claude 先思考（Thinking），然后决定行动方案。复杂任务会分解为多个子任务，每次执行前重新评估 |

填写时最重要的原则：**记录具体的交互细节，而非笼统感受。** "在 XXX 操作时，Agent 做了 YYY"比"Agent 很智能"有价值十倍。

### 4.2 从四个维度分析 Agent 产品

除了 9 维度的结构化模板，还有四个相互关联的分析维度值得关注。这四个维度不需要你逐条填写，而是作为体验时的"思考清单"存在。

**维度一：决策透明度**

**用户能否理解 Agent 为什么这样做？**

高透明度：Agent 展示它的每一步推理——"我决定先搜索关键词 X，因为……"、"搜索结果不够精确，我改用关键词 Y……"。你能看到它的思考链条。

低透明度：Agent 只扔给你一个结果，中间怎么想的一概不知。

透明度关乎信任。Agent 一定会犯错，而一个"黑箱"Agent 一旦犯错，你会迅速失去信心——因为你不知道还有没有更多隐藏的错误。好的透明设计不是把技术细节全暴露出来（你看不懂，也不想看），而是在关键决策点提供可理解的人类语言解释。

**维度二：可控性**

**用户在多大程度上可以影响 Agent 的行为？**

可控性有几个层次：
- **目标层**：你可以调整目标——"不用太详细，给我概要就行"
- **策略层**：你可以改变偏好——"优先用官方文档，别用论坛结果"
- **操作层**：你可以随时中断、查看状态、选择继续/换方向/重来
- **纠错层**：你可以实时纠正——"不对，那家公司已经被收购了，重新搜"

一个重要的设计原则：**操作的不可逆性越高，人类的介入门槛就应该越低。** Agent 自主决定用哪个搜索引擎——可以。Agent 要发一封邮件给 100 个客户——最好先让你看一眼。

**维度三：容错能力**

**当某个步骤出错时，Agent 能否恢复？**

观察这几个层次：
- **检测**：它知道自己出错了吗？（API 返回错误，它能识别吗？）
- **诊断**：它知道是什么类型的错误吗？（网络超时？权限不足？参数格式错？）
- **恢复**：它有应对策略吗？（换工具？换参数重试？降级给出部分结果？）
- **求助**：它在什么时候放弃并告诉你"我需要帮助"？（好的 Agent 知道什么时候该求助，而不是无限重试）

**维度四：效率**

**Agent 完成任务花了多少资源？**

效率不是单纯的"越快越好"。有时多花一点代价做 double-check 是值得的——如果错误代价很高的话。

- 推荐餐厅的任务：几轮快速搜索就够了，错了也没关系
- 计算财务报表的任务：多花几步交叉验证是合理的——错的代价远大于慢的代价

### 4.3 对比体验：用同一个任务测试不同产品

最有效的学习方法之一，是用**完全相同的任务**去测试不同 Agent。

选一个任务，比如"帮我研究最近三个月 AI 编程工具领域的重要变化，整理成一份简报"。把这个任务分别交给 Claude Code、GPT-Researcher 和你选的另一款产品。

观察它们的策略差异：谁先搜什么？谁花更长的时间规划？谁的结果更全面？谁的效率更高？

好的测试任务满足四个条件：
- **需要多步推理**：不是一问一答能解决的
- **需要工具使用**：需要查询外部信息或操作外部系统
- **有一定的开放性**：没有唯一正确答案，允许多种策略
- **可验证**：有办法判断结果好坏
