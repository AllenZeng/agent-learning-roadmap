# 课程一：初识 Agent

---

## 课程导言

如果你正在读这段话，你很可能已经用过 ChatGPT、Claude 或类似的产品。你问它问题，它回答你——有时候惊艳，有时候胡说八道。你对它的感觉可能是："这东西挺聪明的，但它能真的帮我做点什么吗？"

**本课程要回答的就是这个问题。**

这一课不讲原理，不讲实现，不讲架构。它的目标只有一个：**让你看见 Agent，建立直觉。**

什么是"看见"？打开 Claude Code，让它帮你重构一个文件；打开 Cursor，看它如何理解你的代码库；打开 Kimi，让它做一份行业调研。你不需要理解它内部是怎么工作的——就像你不需要理解发动机原理就能感受到一辆车的加速性能。

什么是"直觉"？用完之后，你能自然地判断：

- 这个产品是真正在做决策，还是只是在走固定的流程？
- 它的哪一步让你觉得"聪明"，哪一步让你觉得"笨"？
- 下一步到底是谁在决定——用户、固定流程，还是模型？
- 如果让你来设计，你会改什么？

这些直觉，是后面所有学习的燃料。没有直觉支撑的原理，是空洞的；带着直觉去学原理，每一句话你都能想起一个具体的画面。

**本课是感性认知层，不讲实现细节。** 它为课程二的范式理解和课程三的最小闭环提供产品观察样本。

准备好了吗？让我们开始第一次接触。

---

## 学习目标

学完本课，你将能够：

1. **说出 Agent 是什么** — 区分它与 Chatbot、Workflow 的本质边界，而不是被"用了 LLM"的表象迷惑
2. **用"谁决定下一步"判断 Agent 成色** — 一眼看穿产品是模型在做决策，还是预设规则在驱动
3. **用 7 维度模板拆解任何 Agent 产品** — 从感性体验走向结构化分析，不再停留在"好用 / 不好用"
4. **解释用户为什么信任或不信任 Agent** — 理解惊喜、失控、可靠感背后的设计原因

## 第一章：什么是 Agent？

### 1.1 先看一个场景

你让 ChatGPT 帮你订一张从北京飞上海的机票。

ChatGPT 会说："抱歉，我无法帮你订机票。你可以去携程或航空公司官网查询。"

你让一个真正的 Agent 做同样的事。

Agent 会自己打开浏览器，搜索航班信息，比较不同航空公司的价格和时间，找到合适的航班后填写订单页面——你在旁边看着就行。到了支付那一步，它会停下来问你："找到一趟下午 3 点、票价 850 元的航班，要我帮你下单吗？"

这两种反应的差异，就是 Agent 和 Chatbot 的本质区别。

再来一个更日常的场景——假设你要写一份竞品分析报告：

**Chatbot 的做法**：你问它，它给你列出竞品名字、分析框架、写作建议。你需要自己打开浏览器搜索、阅读文章、提取信息、组织成文。

**Agent 的做法**：你告诉它"帮我做一份关于 XX 领域的竞品分析"，它自己搜索各家官网和新闻报道，提取关键信息，对比产品差异，生成一份结构化报告，标注信息来源。你只需要在关键节点（比如最终发布前）确认。

**差别在哪？** Chatbot 告诉你怎么做，Agent 自己去做。

### 1.2 Agent 不是"更聪明的聊天机器人"

这是理解 Agent 的第一个关键认知：**Agent 和 Chatbot 不是"智能程度"的差别，而是"能力维度"的差别。**

Chatbot 工作在一个维度上：**语言空间**。它接收文字，处理文字，输出文字。它可以理解你、回应你、建议你——但所有这些都在对话框之内。

Agent 工作在两个维度上：**语言空间 + 行动空间**。它不只是理解你的意图，还能在真实世界中产生行动——搜索、读写文件、调用 API、执行代码、操作浏览器。行动的结果反过来影响它的下一步决策。

用一句话概括：

> **Agent 是能够围绕目标持续决策和执行的系统，而不仅仅是回答问题。**

这句话里每个词都很重要：

- **围绕目标**：Agent 不是"有问必答"，而是始终朝向一个目标推进。它会判断自己离目标还有多远，是否走偏了。
- **持续决策**：不是一次判断就结束，而是在每一步之后根据新的信息重新判断"接下来做什么"。
- **执行**：不只是"建议你怎么做"，而是真的去做了——调用了工具，改变了外部世界的状态。

### 1.3 Agent vs Chatbot vs Workflow

理解了 Agent 是什么之后，你需要区分三个经常被混淆的概念：**Chatbot**、**Workflow**、**Agent**。

#### Chatbot：负责对话和回答

Chatbot 的核心能力是理解和生成语言。用户输入一段话，它返回一段话。即使接入了搜索引擎或知识库，Chatbot 仍然是"你说什么，我回答什么"——它不会主动制定计划，不会自己决定"接下来要做三件事"。

**核心特征**：下一步永远是"等待用户输入"。

#### Workflow：按固定流程执行

Workflow 是预定义的执行链。每一步做什么、数据如何传递、什么条件触发什么分支，在设计阶段就确定了。虽然 Workflow 的某些节点可能调用 LLM（比如生成文案、分类意图），但 LLM 在这里只是"一个功能模块"，不是"决策者"。决策权属于那个设计好的流程。

**核心特征**：下一步由"预设规则"决定。

典型的 Workflow 例子：智能客服系统——
1. 识别用户意图（5 种预定义类型）
2. 根据意图调用对应的知识库
3. LLM 润色生成回答
4. 如果解决不了，转人工

整个流程中，LLM 参与了步骤 1 和步骤 3，但它没有决策权——什么时候转人工、用哪个知识库，都是预设规则决定的。

#### Agent：根据目标、状态和反馈动态决定下一步

Agent 和前面两者的根本差异在于：**下一步不由用户决定，不由预设规则决定，而是由模型根据当前目标、任务状态和环境反馈动态决定。**

**核心特征**：下一步由"模型动态决策"。

典型的 Agent 例子：Claude Code。给定"帮我找出这个项目的所有 bug 并修复"——

- 它自己决定先读哪些文件
- 读到代码后决定运行什么诊断命令
- 看到诊断结果后决定一个现象是不是 bug
- 修复后决定如何验证
- 验证失败后决定是重试还是换策略

这个链条中每一步的决策者都是模型本身，而不是预设流程。

#### 三者的核心差异

| 对比维度 | Chatbot | Workflow | Agent |
|---------|---------|----------|-------|
| 下一步由谁决定 | 用户 | 预设规则 | 模型动态决策 |
| 能否调用工具 | 通常不能，或仅有搜索增强 | 可以，但调用时机由规则决定 | 可以，调用时机由模型判断 |
| 任务范围 | 单轮或多轮对话 | 固定流程内的任务 | 开放性目标 |
| 容错方式 | 无法行动，无所谓"执行错误" | 异常分支由预设规则处理 | 根据反馈动态调整策略 |
| 典型产品 | ChatGPT 基础版 | 传统智能客服、RPA 流程 | Claude Code、Cursor Agent、Devin |

#### 一个重要的灰色地带：Agentic Workflow

现实中，很多产品不是纯 Workflow，也不是纯 Agent，而是两者的混合——**Agentic Workflow**。

Agentic Workflow 的核心思路是：**在整体框架中嵌入 Agent 决策节点，用 Workflow 保证下限（不出大错），用 Agent 提升上限（做得更好）。**

比如一个内容审核系统：
- 整体流程是 Workflow：接收内容 → 自动规则检查 → 模型初筛 → 人工复核
- 但"模型初筛"这一步是 Agent 行为：模型自己决定从哪些角度审核、是否需要检索相关政策、证据不足时是否请求更多上下文

这种设计的工程智慧在于：不需要把"确定性"和"灵活性"二元对立，而是在不同环节使用不同的决策机制。

### 1.4 核心观察点：谁决定下一步？

如果你只能记住一个概念来区分 Agent 和其他产品，记住这个：

> **看"谁决定下一步"。**

- Chatbot：用户决定下一步（用户不输入，系统就等着）
- Workflow：预设流程决定下一步（A 步骤完了必定到 B 步骤）
- Agent：模型根据目标、状态和反馈动态决定下一步

这就是为什么"是否用了 LLM"不是一个好的判断标准——很多 Workflow 也用了 LLM，但 LLM 只负责执行某个环节（如分类、生成），不负责决定流程走向。

**判断练习**：思考以下场景——

> 一个数据分析工具，用户上传 Excel，系统自动生成图表和洞察报告。在生成报告时，系统会调用 LLM 来撰写分析文字。

这是 Agent 吗？**不是。** 虽然调用了 LLM，但"什么时候调用、调用后做什么"都是预设流程决定的。LLM 只是"文案生成器"，不是"决策者"。

> 同一个工具，但用户可以输入"帮我找出这个销售数据中异常的趋势，分析可能的原因"。系统自己决定：先按时间维度切分数据、检查季节性波动、对比去年同期、搜索行业新闻解释异常点。

这才是 Agent 行为：**模型在决定"接下来该查什么"。**

### 1.5 一个可操作的判断框架

基于"谁决定下一步"这个核心视角，这里给出一个快速判断框架。当你遇到任何一款声称是 Agent 的产品时，问三个问题：

1. **目标来源**：任务目标是用户明确输入的，还是系统从对话中推断的？
   - 用户明确输入 → 更偏 Agentic Chatbot 或简单 Agent
   - 系统推断目标并主动推进 → 更强的 Agent 特征

2. **行动发起**：行动是由模型自主发起的，还是由用户指令或预设规则触发的？
   - 模型自主发起 → Agent 特征
   - 用户指令触发 → Chatbot 特征
   - 预设规则触发 → Workflow 特征

3. **策略调整**：当遇到意外情况时，系统能自主改变策略吗？
   - 能自主换工具、换关键词、换路径 → 强 Agent 特征
   - 只能按预设异常分支处理 → Workflow 特征
   - 直接失败或请求用户重新输入 → Chatbot 特征

这三个问题叠在一起，就是一个产品的 Agent 成色光谱。

## 第二章：为什么现在谈 Agent？

### 2.1 LLM 改变了什么

在 LLM 出现之前，Agent 这个概念在 AI 领域已经存在了几十年。传统 Agent（如强化学习 Agent、BDI Agent）在某些特定领域（游戏、机器人控制）取得了成功，但它们有一个根本的限制：**只能在封闭的规则空间内运行。** 构建一个传统 Agent 需要人工定义所有可能的状态、动作和规则——这在开放性的真实任务中几乎不可行。

LLM 的出现改变了这个局面：

- **从"人工写规则"变成"模型理解目标并生成下一步"**：不需要穷举所有可能的情况，模型可以理解自然语言描述的目标并自己决定策略。
- **从"封闭规则空间"变成"开放语义空间"**：模型可以处理它在训练中从未精确见过但语义上相似的任务。
- **从"只回答"走向"能调用外部能力"**：模型可以通过工具调用连接实时信息、私有数据、计算能力和外部系统。

换句话说，LLM 让 Agent 有了一个"通用的大脑"。这个大脑不一定在每个具体任务上比专用系统强，但它的**泛化能力**是传统 Agent 无法企及的。

### 2.2 从"对话"到"行动"的自然延伸

理解 Agent 为什么会出现，可以沿着一条清晰的演化线：

```text
LLM 能理解和生成语言（Chatbot）
  → 但语言只能"建议"，不能"执行"
  → 给 LLM 接上工具（Function Calling / Tool Use）
  → LLM 可以根据任务需要自己决定调用什么工具
  → Agent 诞生
```

这不是"突然发明了一个新东西"，而是 LLM 能力从"语言空间"向"行动空间"的自然溢出。

一个帮助你理解的类比：

> Chatbot 就像一个非常博学的顾问——你可以问它任何问题，它能给你很好的建议，但它不能帮你做任何事。
>
> Agent 就像给这个顾问配了一个执行助理——顾问（LLM）做出判断，助理（工具系统）去执行，然后把结果汇报给顾问，顾问再根据结果做下一步判断。

### 2.3 Agent 解决的是 LLM 的什么局限

LLM 有三个众所周知的局限，Agent 架构在很大程度上是为了突破这些局限：

| LLM 的局限 | Agent 如何突破 |
|-----------|---------------|
| **知识截止日期**：训练数据停在某个时间点 | 工具调用让模型可以实时搜索、查询数据库、访问最新信息 |
| **无法行动**：只能在对话框里输出文字 | 工具调用让模型可以操作外部世界——读写文件、发邮件、执行代码 |
| **单次推理不可靠**：复杂任务一步到位容易出错 | 循环决策让模型可以分步执行、观察反馈、修正错误 |

但这里有一个重要的提醒：**Agent 突破这些局限的方式，本身也引入了新的问题。** 能调用工具意味着可能越权；能多步决策意味着可能陷入循环；能自主行动意味着可能做出用户意料之外的事。这些问题将在课程四、六、七中深入讨论。现在，你只需要知道：Agent 不是 LLM 的"升级版"，而是在 LLM 之上构建的一套决策和执行系统。

## 第三章：体验主流 Agent 产品

理论说到这儿就够了。接下来是本课程最核心的学习活动：**亲自用。**

你不需要安装所有产品。从中选 3 款，每款至少用 30 分钟，带着目的去体验。**只看文档和演示视频不算体验。**

### 3.1 产品清单与体验要点

#### 国际产品

**1. ChatGPT（OpenAI）**

当前最具普遍性的 Agent 产品入口。在基础对话之上，ChatGPT 可以搜索网页、分析上传的文件、生成图片、执行代码（数据分析模式）。Plus/Pro 订阅用户可以体验更深度的工具调用和分析能力。

**观察要点**：
- 它在什么时候主动搜索网页，什么时候不搜？
- 数据分析模式下，它是如何决定"写什么代码 → 执行 → 看结果 → 调整"的？
- 它的决策过程你能看到多少？

**2. Claude / Claude Code（Anthropic）**

Claude 的对话产品展现了很强的推理和长文本处理能力。Claude Code 是其终端 CLI 形态的编程 Agent，在命令行中运行，可以读取项目文件、编写代码、执行命令、管理 git。

**观察要点**：
- Claude Code 的 Thinking 过程展示了什么？哪些信息对你理解它的决策有帮助？
- 它在执行危险操作（如删除文件、git commit、执行外部命令）前如何征求你的确认？
- 它一次能做对的事情和反复修改才能做对的事情分别是什么？

**3. Cursor（Cursor Inc.）**

基于 VS Code 的 AI 编辑器，其 Agent 模式不只是补全代码，而是能理解整个项目结构、跨文件做修改、运行终端命令。

**观察要点**：
- Agent 模式和普通 Chat 模式的行为差异在哪里？
- Apply 时它修改了哪些文件？修改的粒度合理吗？
- 当你连续提出多个需求时，它能否保持上下文？

**4. Perplexity（Perplexity AI）**

以搜索为核心的 AI 产品，但近年来越来越向 Agent 方向演进——可以规划搜索策略、深入特定页面、对比不同来源。

**观察要点**：
- 它的搜索是"一次性搜索"还是"多轮深入搜索"？
- 搜索关键词的选择质量如何？
- Pro Search 模式下它的行为有什么不同？

**5. Devin / Replit Agent**

Devin 是"自主软件工程师"，Replit Agent 帮助用户从自然语言描述构建完整应用。两者都展现了从需求理解到代码交付的完整 Agent 链路。

**观察要点**：
- 它是如何拆解一个复杂开发任务的？
- 它的哪些步骤让你觉得"合理"，哪些让你觉得"在兜圈子"？
- 遇到错误后它是如何恢复的？

#### 国内产品

**6. Kimi（月之暗面）**

以长上下文能力著称，支持 200 万字上下文窗口。Kimi 在搜索、文件处理和长文档分析方面有独特优势，其浏览器插件支持实时网页内容获取。

**观察要点**：
- 长上下文是否真的让它做复杂任务时表现不同？
- 它在搜索网页时的策略是怎样的？
- 它的回复结构和国际产品有什么不同？

**7. 豆包 / 扣子 Coze（字节跳动）**

豆包面向普通用户，扣子 Coze 面向开发者和 Bot 搭建者。Coze 的 Bot 搭建平台让用户可以配置工作流、插件、知识库，是观察 Workflow 到 Agent 过渡的绝佳样本。

**观察要点**：
- Coze 中的"工作流"和"Agent 模式"的区别是什么？
- 插件的触发是由规则决定的还是模型决定的？
- 搭建一个简单 Bot 的过程中，哪些是你在决策，哪些是模型在决策？

**8. 通义千问 / 腾讯元宝**

通义千问（阿里）和腾讯元宝（腾讯）都提供了文件分析、网页搜索、代码执行等能力，是国内观察大厂 Agent 产品化思路的重要窗口。

**观察要点**：
- 功能边界在哪里？它能做什么、不能做什么？
- 不同产品在类似功能上的体验差异是什么？

### 3.2 体验方法论：带着问题去用

很多人体验新产品是"随便点点，凭感觉评价"。我们不这样做。

在体验每一款 Agent 时，请始终带着这四个问题：

1. **它什么时候自己做决定？**
   - 哪些操作它直接执行了，没有问你的意见？
   - 你认为这些"自作主张"合理吗？
   - 有哪些操作你觉得它应该自己做但反而问了你？

2. **它什么时候问我？**
   - 在什么类型的操作前，它会停下来征求你的确认？
   - 这个停止点设置得合理吗？会不会过于频繁（烦人）或过于宽松（危险）？

3. **它什么时候做错了？**
   - 记录一个具体的错误。它是怎么出错的？
   - 它自己发现了吗？如果没发现，你是怎么发现的？
   - 出错后它是怎么处理的？

4. **它什么时候让我惊喜？**
   - 记录一个让你觉得"哇，它竟然能做到这个"的瞬间。
   - 为什么这个瞬间让你印象深刻？
   - 这个惊喜是"一次性"的还是"可复现"的？

这四个问题对应的是 Agent 设计的四个核心维度：**自主性、可控性、容错性、能力边界**。你现在不需要记住这些术语——带着问题去用就行。在第四章中，我们会把这些体验系统化。

### 3.3 进阶：用同一个任务测试不同产品

最有启发性的学习方法之一，是用**完全相同的任务**去测试不同 Agent。它能让你看到：面对同一个目标，不同产品的"思维方式"有多么不同。

**选择一个合适的测试任务**，比如"帮我研究最近三个月 AI 编程工具领域的重要变化，整理成一份简报"。

好的测试任务满足四个条件：
- **需要多步推理**：不是一问一答能解决的
- **需要工具使用**：需要查询外部信息或操作外部系统
- **有一定的开放性**：没有唯一正确答案，允许多种策略
- **可验证**：有办法判断结果好坏

把这个任务分别交给 2-3 款产品，然后观察它们的策略差异：
- 谁先搜什么？搜索关键词选得好不好？
- 谁花更长的时间规划？
- 谁的结果更全面？谁的结果更精准？
- 谁的效率更高（更少的步骤获得相近的结果）？
- 谁在遇到死链接或信息不足时更灵活？

把对比结果记录下来——这本身就是一份高质量的产品分析素材。

## 第四章：产品拆解与分析

体验完产品之后，你需要一个框架来系统化你的感受。从"感觉好用/不好用"到"我知道它为什么好用/不好用"——这就是本章的目的。

### 4.1 产品体验观察框架（7 维度模板）

下面这个 7 维度观察框架，是本课程的核心分析工具。在体验每一款产品后，用这个模板做结构化拆解。

| 维度 | 观察问题 | 记录方式 |
|------|---------|---------|
| **任务入口** | 用户输入的是开放目标、具体指令，还是普通问答？ | 写出原始用户任务 |
| **下一步决策权** | 下一步由谁决定？ | 用户 / 固定流程 / 模型 |
| **行动能力** | 产品能调用哪些工具或外部能力？ | 列出工具、数据源或工作流节点 |
| **过程可见性** | 用户能看到哪些步骤？ | 计划、工具调用、进度、中间结果 |
| **用户控制权** | 用户能否确认、中断、修改或接管？ | 标出可介入节点 |
| **失败与恢复** | 出错后如何处理？ | 重试、降级、澄清、失败退出 |
| **Agent 成色判断** | 它更像 Chatbot、Workflow，还是 Agent？哪些地方体现出 Agent 特征？ | 给出判断并说明理由 |

这七个维度不是割裂的——它们围绕同一个核心问题：**这个产品在处理任务时，模型到底承担了多少"决策"的职责？**

### 4.2 拆解模板与填写示范

以下是完整的拆解模板。以 Claude Code 体验为例，展示如何填写：

---

**产品拆解报告**

| 维度 | 观察记录 |
|------|---------|
| **产品信息** | 产品名、版本/日期、使用场景 |
| **任务入口** | 用户输入的原始任务是什么？属于开放目标、具体指令还是普通问答？ |
| **下一步决策权** | 任务推进过程中，下一步主要由谁决定？（用户 / 固定流程 / 模型）标注具体的决策转变时刻。 |
| **行动能力** | 产品能调用哪些工具或外部能力？（搜索、代码、文件、浏览器、数据分析、工作流节点等） |
| **过程可见性** | 用户能看到哪些环节？（计划/思考过程、工具调用的名称和参数、执行结果、进度条、中间产物） |
| **用户控制权** | 用户可以在哪些节点介入？（确认、中断、修改计划、驳回单步操作、接管、回退） |
| **失败与恢复** | 记录至少一个失败场景：发生了什么？产品如何响应？（重试、换策略、降级、请用户帮助、直接失败） |
| **Agent 成色判断** | 整体判断：它更像 Chatbot、Workflow，还是 Agent？哪些具体行为支撑你的判断？哪些行为让你觉得 Agent 成色不足？ |

---

**填写示范（以 Claude Code 为例）**

| 维度 | 观察记录 |
|------|---------|
| **产品信息** | Claude Code, 2026 年 6 月使用，终端 CLI 工具，用于代码开发和维护 |
| **任务入口** | "帮我把 src/utils 下的工具函数加上错误处理"——这是一个具体指令，但有一定开放性（哪些函数、加什么错误处理由 Agent 判断） |
| **下一步决策权** | 模型主导。Claude 自己决定先列出目录结构、逐个读取文件、对每个函数提出改造方案。只在 git 操作和删除文件等高风险动作时请求确认。 |
| **行动能力** | 文件读写（Read/Write/Edit）、Shell 命令执行（Bash）、代码搜索（Grep）、Git 操作、Web 搜索 |
| **过程可见性** | 高。Thinking 块展示推理过程；每个工具调用前标注名称和参数；工具调用结果可见；当前步骤目标有文字说明。 |
| **用户控制权** | 用户可以在任何工具调用弹窗选择允许/拒绝/总是允许；可以用自然语言实时干预（"不对，用 try-catch 而不是 .catch()"）；某些高风险操作默认需要确认。 |
| **失败与恢复** | 示例：运行测试时发现测试文件不存在，它主动搜索测试目录、检查测试配置、尝试用不同框架运行——连续三次尝试失败后，说明情况并询问用户项目的测试设置方式。属于"重试→降级→澄清"路径。 |
| **Agent 成色判断** | 强 Agent。自主决定工具选择和执行顺序；遇到意外反馈后动态调整策略；复杂任务能自动拆解为子任务。不是纯 Agent 的点：高风险操作的确认是强制性的（不可跳过），这是产品化的安全设计而非 Agent 能力的限制。 |

---

填写时最重要的原则：**记录具体的交互细节，而非笼统感受。** "在 XXX 操作时，Agent 做了 YYY，参数是 ZZZ"比"Agent 很智能"有价值十倍。

### 4.3 深入理解四个关键维度

除了结构化的 7 维度模板，有四个相互关联的分析维度值得你深入思考。这些不需要逐条填写在模板中，而是作为你体验和反思时的"思维清单"。

#### 维度一：决策透明度

**用户能否理解 Agent 为什么这样做？**

- **高透明度**：Agent 展示它的每一步推理——"我决定先搜索关键词 X，因为……""搜索结果不够精确，我改用关键词 Y……""根据以上信息，我的结论是……"你能看到它的思考链条。
- **低透明度**：Agent 只扔给你一个最终结果，中间怎么想的一概不知。

透明度关乎信任。Agent 一定会犯错。一个"黑箱"Agent 一旦犯错，你的信心会迅速崩掉——因为你不确定还有多少隐藏的错误。好的透明设计不是把技术细节全暴露出来（你看不懂，也不想看），而是在**关键决策点**提供可理解的人类语言解释。

**体验时问自己**：当 Agent 做了什么让我意外的事，我能从界面上找到它为什么那样做的线索吗？

#### 维度二：可控性

**用户在多大程度上可以影响 Agent 的行为？**

可控性有几个层次：
- **目标层**：你可以调整目标——"不用太详细，给我概要就行""重点看性能，别的可以忽略"
- **策略层**：你可以改变偏好——"优先用官方文档，别用论坛结果""遇到不确定的情况先问我"
- **操作层**：你可以随时中断、查看状态、选择继续/换方向/重来
- **纠错层**：你可以实时纠正——"不对，那家公司已经被收购了，重新搜"

一个重要的设计原则：**操作的不可逆性越高，人类的介入门槛就应该越低。**

- Agent 自主决定用哪个搜索引擎 → 可以
- Agent 要发送一封邮件给 100 个客户 → 最好先让你看一眼
- Agent 要删除生产环境的数据 → 必须经你明确确认

**体验时问自己**：当我想改变 Agent 的行为时，我的操作成本有多高？我能在不刷新页面的情况下实时干预吗？

#### 维度三：容错能力

**当某个步骤出错时，Agent 能否恢复？**

观察这几个层次：
- **检测**：它知道自己出错了吗？（API 返回 500，它能识别吗？搜索结果全是无关内容，它意识到了吗？）
- **诊断**：它知道是什么类型的错误吗？（网络超时？权限不足？参数格式错？搜得太窄？）
- **恢复**：它有应对策略吗？（换参数重试？换一个工具？换搜索关键词？降级给出部分结果？）
- **求助**：它在什么时候放弃并告诉你"我需要帮助"？（好的 Agent 知道什么时候该求助，而不是无限重试）

**体验时问自己**：Agent 在失败时的行为，是让我觉得"它在努力解决问题"，还是"它在浪费时间兜圈子"？

#### 维度四：效率

**Agent 完成任务花了多少资源？**

效率不是单纯的"越快越好"。有时多花一点代价做 double-check 是值得的——如果错误代价很高的话。

- 推荐餐厅的任务：几轮快速搜索就够了，错了也没关系
- 计算财务报表的任务：多花几步交叉验证是合理的——错的代价远大于慢的代价
- 发送批量客户邮件：宁可慢一些，每一封都要确认内容是否正确

**体验时问自己**：Agent 的步骤数合理吗？有没有明显多余的步骤？有没有为了"看起来很聪明"而过度分析？

### 4.4 如何判断一个产品的 Agent 成色

做完拆解后，你需要在 7 个维度的观察之上做出综合判断。这不是一个精确的科学，但有一个有用的心智模型：

把产品放在一个光谱上——

```text
Chatbot ◀──────────────────────────────▶ Agent
(用户决定下一步)              (模型动态决定下一步)
```

判断的关键信号：

| 信号类型 | 更偏 Chatbot | 更偏 Agent |
|---------|-------------|-----------|
| 任务复杂度 | 只能处理单轮问答 | 能处理多步、有依赖的复杂任务 |
| 工具调用 | 工具调用由用户触发或预设规则触发 | 工具调用由模型根据任务需要自主决定 |
| 策略灵活性 | 固定响应模式 | 遇到意外时改变方法 |
| 错误恢复 | 失败后只能请求用户重新输入 | 失败后尝试不同方法 |
| 过程主动性 | 等待用户告知每一步 | 主动推进，只在关键节点确认 |

**判断时的三个层次**：

1. **这个产品的大多数行为是由什么驱动的？** —— 用户输入、预设规则，还是模型动态决策？
2. **在哪些具体场景下它表现出了 Agent 特征？** —— 是"做搜索规划"时的自主性，还是"代码生成后自动运行测试验证"时的反馈闭环？
3. **哪些场景让它露出了 Chatbot 或 Workflow 的底？** —— 是"只能在预设选项里选"的规划步骤，还是"失败后除了道歉什么都做不了"的错误处理？

**做判断时的一条底线**：不要因为一个产品"用了 LLM"就说它是 Agent。判断标准始终是**"下一步决策权在谁手里"。**
