# Human-in-the-Loop 设计

## 为什么需要人类参与

模型不理解后果。它看到的不是"这个操作会删掉用户三年的数据"，而是"根据概率分布，下一个 Token 可能是 DELETE"。当动作的后果不可逆、影响真实用户、或者模型自身置信度不够高时，应该把人类拉入循环。

Human-in-the-loop 不是"Agent 能力不够时的临时补丁"，而是工具机制中的结构性控制点。

## 触发条件

- **高风险动作**：删除、支付、发布、发送消息
- **模型不确定**：低置信度决策
- **权限不足**：用户可能愿意授权
- **不可撤销**：动作一旦执行无法回滚
- **参数存在歧义**：模型在"猜"参数值

## 介入方式

不只是"同意/拒绝"二选一。完整的介入方式包括：
- 确认执行
- 驳回动作
- 修改参数
- 编辑计划
- 要求解释
- 接管执行
- 回滚结果

## 反馈闭环

用户反馈必须写回 Agent 状态。如果反馈不进状态，Agent 下一轮会基于旧上下文做出同样的决策，再次触发同一个确认——用户会觉得在和一个没有记忆的系统反复拉扯。

## 风险分级

```
L1 只读低风险（搜索网页）       → auto
L2 只读中风险（读用户文件）     → auto
L3 写入低风险（生成本地草稿）   → confirm
L4 写入高风险（修改数据库）     → confirm
L5 危险操作（删除、支付、发布） → deny by default
```
