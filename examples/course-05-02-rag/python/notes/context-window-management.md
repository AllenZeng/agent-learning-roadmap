# 上下文窗口管理

## 上下文是有限的

无论 128K 还是 1M tokens，上下文窗口总有一个硬上限。而且更长的上下文意味着更高的延迟、更高的成本、以及模型注意力被稀释后"忘了中间说了什么"的"迷失在中间"问题。

## 上下文组装策略

RAG 场景下，检索回来的 chunk 如何放入上下文：

### 排序

最相关的 chunk 放最前面和最后面——模型对开头和结尾关注度最高。

### 来源标注

每个 chunk 标注出处，让模型能生成可追溯的引用：
```
[来源 1] agent-tool-use-design.md - Tool Use 与 Memory 的关系
更新于 2026-05-20

...内容...

[来源 2] agent-memory-mechanism.md - 写入决策与遗忘机制
更新于 2026-06-10

...内容...
```

### Token 预算

给检索结果分配固定的 token 预算（如 2000 tokens），按分数从高到低填充，剩余空间留给历史对话和系统指令。

## 工具结果的处理

工具返回结果也可能撑爆上下文。四种处理方式：
- **摘要**：结果长但只需关键信息（搜索、API 响应）
- **截断**：只留头部和尾部（日志、长文件）
- **分页**：结果列表长，逐步查看（数据库查询）
- **转资源**：原始结果写入临时文件，模型拿引用路径

详见 [[rag-retrieval-practice]] 了解检索阶段的上下文控制。
