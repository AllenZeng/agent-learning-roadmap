# RAG 检索实践

## RAG 的核心思路

RAG（Retrieval-Augmented Generation）解决一个根本问题：模型的知识有时效性，训练数据有截止日期，私有知识不在参数里。RAG 的思路是"开卷考试"——回答前先去知识库里找相关资料，把资料和问题一起给模型。

## 完整链路

RAG 分两个阶段：

**离线建库**：原始文档 → 解析清洗 → Chunking → Embedding → 索引入库

**在线查询**：用户问题 → 查询改写 → 多路召回 → 重排序 → 上下文组装 → 生成

## 混合检索

单靠向量检索不够。向量擅长语义匹配但容易漏掉精确关键词，BM25 擅长关键词但不懂同义词。混合检索取两者之长：

- **向量召回 top-k**：捕捉语义相似，泛化能力强
- **关键词召回 top-k**：精确匹配专有名词、术语
- **元数据过滤**：只检索 `status=published`、时间范围符合的文档
- **RRF 融合**：合并两路结果并去重

## Chunking 策略

Chunking 在检索精度和上下文完整性之间做权衡：
- 切太小 → 信息被打散，关键论述被截断
- 切太大 → 噪声混入，检索精度下降

对于 Markdown 笔记，按 `##` 标题层级切分是一个实用的默认策略——每个 chunk 通常是 500-1500 字符，自包含一个主题。

## 重排序的重要性

召回追求覆盖（宁可多召不能漏），重排序追求精度（把真正有用的排前面）。Cross-encoder 比 embedding 相似度更准确，但计算成本更高，所以只在候选集上使用。分数断崖（#4 0.68 → #5 0.45）是截断的自然信号。
